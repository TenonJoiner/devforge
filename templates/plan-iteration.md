## 迭代概览

> **写什么**：迭代元数据表，含里程碑编号、迭代主题、容量预算（来自 milestone-plan.md 迭代总览）、实际选入量、团队规模、迭代周期、前置依赖。表中数据为后续 Wave 分组和合理性检查提供基准。
> **不写什么**：不列出具体 proposal 清单（在 Wave 中展开），不做风险管理（在风险与缓冲中处理），不分析关键路径（独立小节处理）。
> **最低深度**：表格 8 要素（里程碑/编号/主题/容量/选入/团队/周期/前置依赖）齐全，容量预算与实际选入对比显式。

## Wave 0 — 无依赖，立即启动

> **写什么**：无依赖、可立即启动的 proposal 列表。每行含名称、故事点、大小、类型、描述。这些 proposal 不依赖本迭代内其他 proposal 的产出，可立即并行启动。
> **不写什么**：不包含有依赖的 proposal（放入后续 Wave），不包含依赖未在本迭代内的外部产出的 proposal，不展开设计细节。
> **最低深度**：至少 1 个 proposal，所有依赖列均为"无"。

| P02 | `<name>` | X SP | M | 基础设施 | <一句话> | <可选> |

## Wave 1 — 依赖 Wave 0

> **写什么**：依赖 Wave 0 产出的 proposal 列表，每个 proposal 标注具体的依赖 proposal 编号。依赖必须是 Wave 0 中已列出的 proposal。
> **不写什么**：不包含无依赖的 proposal（应在 Wave 0），不包含依赖未在 Wave 0 中的 proposal，不包含跨 Wave 的间接依赖（只标注直接前驱）。
> **最低深度**：至少 1 个 proposal，每个依赖项具体到 proposal 编号，依赖在 Wave 0 中存在对应项。

| # | Proposal | 故事点 | 大小 | 类型 | 依赖 | 描述 | 执行人 |
|---|----------|--------|------|------|------|------|--------|
| P05 | `<name>` | X SP | L | Spike | P01, P02 | <一句话> | <可选> |

## Wave N — 依赖 Wave 0..N-1

> **写什么**：按需添加更多 Wave，每个 Wave 包含依赖前序 Wave 产出的 proposal 列表。每个 proposal 标注具体的依赖 proposal 编号，依赖项必须在前序 Wave 中存在。
> **不写什么**：不创建只在拓扑上必需但无实际 proposal 的占位 Wave，不包含无依赖的 proposal。
> **最低深度**：如有内容（≥1 个 proposal），每个依赖项具体到 proposal 编号且可追溯到前序 Wave。

## 关键路径

> **写什么**：从 Wave 0 到最后一个 Wave 的最长依赖链（迭代内部），标注链上 proposal 序列、总故事点、估算周期。瓶颈分析指出关键路径上的高风险节点和缓解措施。
> **不写什么**：不分析跨迭代的关键路径（属于里程碑级），不重复风险评估（仅在此标注瓶颈节点），不展开缓解措施的设计细节。
> **最低深度**：至少 1 条关键路径链（含完整 proposal 序列），标注总故事点和估算周期，瓶颈分析至少指出 1 个高风险节点。

## 风险与缓冲

> **写什么**：迭代级风险表，每行含风险描述、风险标记（🔴技术可行性 / 🟡外部依赖）、影响范围、缓解策略。只列出本迭代特有的风险，不重复里程碑级风险清单。
> **不写什么**：不重复已在 milestone-plan.md 中列出的里程碑级风险，不展开缓解方案的具体实施步骤。
> **最低深度**：至少 1 个风险，每个含标记和缓解策略，技术可行性风险的缓解方案必须含 Spike 验证或缓冲。

| # | 风险 | 标记 | 影响 | 缓解 |
|---|------|------|------|------|
| 1 | <风险描述> | 🔴 | <影响范围> | <缓解策略> |
| 2 | <风险描述> | 🟡 | <影响范围> | <缓解策略> |

**风险标记说明**：
- 🔴 技术可行性风险（需要 Spike + 缓冲）
- 🟡 外部依赖风险（需要 fallback 方案）

## 并行就绪检查（可选）

> **写什么**：当迭代内存在多个可并行开发的 proposal 时，检查并行前置条件（接口契约是否已文档化、Mock/桩机制是否就绪、外部依赖是否已登记交付时间）。如迭代内无并行需求，可删除本节。
> **不写什么**：不在此重复 Wave 分组中的依赖关系、不在此展开具体接口契约内容。
> **最低深度**：如填写，至少覆盖接口契约、Mock 机制、外部依赖三项检查。

> 当迭代内有多个 proposal 需要并行开发时，检查并行前置条件。

- [ ] 接口契约已文档化（相关接口头文件已冻结或已有草案）
- [ ] Mock/桩机制已就绪（并行开发的 proposal 可基于接口契约独立编码）
- [ ] 外部依赖已登记交付时间

## 合理性检查清单

> **写什么**：逐项验证的 checkbox 清单，覆盖容量预算、风险占比、接口冻结、集成测试占比、Wave 分组依赖正确性、关键路径优先安排、proposal 来源合法性。每项对应迭代计划的一个关键质量维度。
> **不写什么**：不包含里程碑级质量检查（属于 milestone plan 模板），不重复其他章节的具体内容。
> **最低深度**：≥ 7 项，每项可客观判断（是/否），覆盖容量、风险、接口、集成测试、依赖、关键路径、来源七个维度。

- [ ] 总点数 ≤ 迭代容量预算
- [ ] 🔴 高风险任务占比 ≤ 40-50%
- [ ] 跨子系统接口已冻结或有明确草案
- [ ] 集成测试 proposal 占比 ≥ 15%（如本迭代包含集成测试）
- [ ] Wave 分组正确反映 proposal 间依赖关系
- [ ] 关键路径上的 proposal 已优先安排
- [ ] 所有 proposal 均来自 milestone-plan.md 的 Backlog

## 评审状态

> 本章节由合并 agent 在评审修正循环通过后写入，不属人工写作范围。

```markdown
**评审状态**: ✅ PASS（缺陷密度 X/评估对象，无 CRITICAL，循环 N/3）
```

- 仅当无 CRITICAL + 缺陷密度达标时写入 `✅ PASS`
- 评审中或未通过写入 `❌ FAIL` 或不写
- 接受带债放行写入 `⚠️ ACCEPTED-WITH-DEBT`（不视为 PASS，但允许阶段过渡）

## 评审记录

> **写什么**：至少 3 个评审角色（架构师/产品经理/项目经理）的独立评审结论和问题分级清单。每角色含通过结论、CRITICAL/HIGH/MEDIUM/LOW 问题的简要标题。每角色 ≤ 200 字。
> **不写什么**：不展开详细评审意见（留在对话上下文），不讨论修正过程，不重复文档已有的内容。
> **最低深度**：至少 3 个评审角色，每个有明确结论（✅ 通过 / ⚠️ 带债通过 / ❌ 不通过），问题按四级分类。

> 每个评审角色限 200 字以内，只记录结论和 CRITICAL 问题。
