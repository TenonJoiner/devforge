# 系统架构总纲

## 设计目标

> **写什么**：描述系统要解决的核心痛点、目标用户（Actor）、主力场景、预期数据/访问规模、部署约束和关键质量属性排序
> **不写什么**：不涉及具体技术选型、不展开子系统内部设计、不描述实现细节
> **最低深度**：核心痛点、目标用户、主力场景、预期规模、部署约束、关键约束、明确边界七个子章节全部必填，关键约束中的 SLA 目标必须量化

> 所有子章节必填，信息来源为前置调研的交互问答。

## 全局设计约束

> 量化定义系统的全局约束，作为架构设计和技术选型的基准。

### 一致性模型

<明确系统在整体上对外提供的一致性保证>

> **写什么**：至少覆盖读写一致性模型（强/顺序/因果/最终）、会话语义（read-your-writes 等）、故障场景一致性保证（CP/AP 选择与脑裂恢复校验策略）、跨副本同步方式（同步/异步复制及 quorum 规则）四种一致性维度
> **不写什么**：不在此展开具体一致性协议实现细节（如 Raft 选主流程），不在此定义各子系统内部的一致性机制
> **最低深度**：四种一致性维度全部覆盖，每项需明确选择并说明选择理由，涉及数据不可丢失的场景需标注持久化保证级别（fsync 策略等）

- **读写一致性模型**：`□ 强一致性（Linearizability） / □ 顺序一致性 / □ 因果一致性 / □ 最终一致性`
- **会话语义**：`□  read-your-writes / □ monotonic-reads / □ monotonic-writes / □ writes-follow-reads`
- **故障场景一致性保证**：<网络分区时选择 CP 还是 AP，脑裂恢复后的数据校验策略>
- **跨副本同步方式**：`□ 同步复制 / □ 异步复制`，法定副本数（quorum）规则：`<如 N=3 时 W=2, R=2>`

### 故障假设

> **写什么**：明确定义系统必须容忍的故障类型、每种故障的假设范围和系统级应对策略
> **不写什么**：不假设故障不会同时发生、不省略拜占庭故障的假设声明（须明确假设存在或不假设存在及阈值）
> **最低深度**：至少覆盖 crash-stop、network partition、disk corruption、silent data corruption、Byzantine fault 五种故障类型

<明确定义系统在设计和测试时必须容忍的故障类型与范围>

| 故障类型 | 假设 | 系统级应对策略 |
|----------|------|---------------|
| Crash-stop | 任意进程随时崩溃，重启后数据可能丢失或损坏 | WAL + 定期 checkpoint；重启后从 WAL 恢复状态 |
| Network partition | 网络延迟、丢包、分区不可避免，无法区分节点崩溃与网络中断 | 基于心跳的超时检测 + 租约机制；分区时遵循预设的 CP/AP 策略 |
| Disk corruption | 磁盘可能因硬件故障产生位翻转或扇区损坏 | 端到端校验和（CRC32C / xxHash64）；定期 scrub 扫描 |
| Silent data corruption | 数据在传输或存储过程中可能静默损坏而不触发 IO 错误 | 每次网络传输和落盘必须携带校验和；读取时验证校验和 |
| Byzantine fault | `□ 假设存在 / □ 不假设存在`，若存在需说明阈值（如 f < N/3） | <若考虑拜占庭故障，说明对应的容错机制，如 BFT 变体> |

### 性能基线

> **写什么**：量化定义吞吐、延迟、扩展性、容量等性能目标，作为所有子系统设计和测试的基准线
> **不写什么**：不使用"HIGH/LOW"等模糊表述代替量化值，不省略测试条件（数据大小/并发数/节点规模）
> **最低深度**：至少覆盖写吞吐、读吞吐、P50 延迟、P99 延迟、扩展性目标和容量上限共 6 个量化指标

<量化定义系统在不同规模下的性能目标，作为架构设计和技术选型的约束>

| 指标 | 目标值 | 测试条件 |
|------|--------|----------|
| 吞吐（写） | <X MB/s 或 Y OPS> | <数据大小、并发客户端数、副本数> |
| 吞吐（读） | <X MB/s 或 Y OPS> | <缓存命中/未命中场景> |
| 延迟 P50 | <X ms> | <同集群、同可用区、标准负载> |
| 延迟 P99 | <X ms> | <同集群、同可用区、高负载> |
| 延迟 P99.9 | <X ms> | <尾延迟敏感场景> |
| 扩展性目标 | <随节点数线性/次线性提升的指标> | <节点范围，如 3~64 节点> |
| 单节点容量上限 | <X TB> | <典型硬件配置> |
| 集群总容量上限 | <X PB> | <规划中的最大规模> |

### 安全边界（可选）

> **写什么**：本系统在认证、授权、加密、审计、隔离五个安全维度的具体约束。无安全要求时可跳过本节。
> **不写什么**：不在此展开安全实现方案（如证书管理流程、密钥轮换机制），不在此定义各子系统的安全策略。
> **最低深度**：如填写，五个维度至少覆盖与本系统相关的维度，每项给出具体约束（不能仅写"需要 RBAC"等笼统描述）。

| 维度 | 约束 |
|------|------|
| 认证 | <是否要求双向 TLS、X.509 证书、Token 认证等> |
| 授权 | <RBAC / ACL / 其他访问控制模型，最小权限原则落实情况> |
| 加密 | <传输层 TLS 版本 / 静态数据加密算法和密钥管理方案> |
| 审计 | <关键操作是否必须记录审计日志，保留时长> |
| 隔离 | <多租户场景下的命名空间/资源隔离策略> |

### 可运维性约束（可选）

> **写什么**：本系统在部署方式、升级策略、监控要求、日志要求、配置管理五个运维维度的具体约束。非运维密集型系统可跳过本节。
> **不写什么**：不在此展开运维平台选型、不写具体监控面板配置、不定义告警规则细节。
> **最低深度**：如填写，五个维度至少覆盖与本系统相关的维度，每项给出可实施的具体约束（不能仅写"支持滚动升级"等笼统描述）。

| 维度 | 约束 |
|------|------|
| 部署方式 | <裸机 / 容器 / K8s / 混合，是否支持一键部署脚本> |
| 升级策略 | <滚动升级 / 蓝绿部署 / 金丝雀发布，是否支持在线升级和版本回滚> |
| 监控要求 | <必须暴露的指标类型（RED/USE/custom），采样频率，告警收敛策略> |
| 日志要求 | <日志级别规范、结构化日志格式、日志轮转策略、保留时长> |
| 配置管理 | <配置热加载能力 / 是否需要重启生效 / 配置校验机制> |

## 风险与缓解策略

> **写什么**：识别系统级的技术风险和实施风险，每项风险需给出可能性、影响程度和具体的缓解策略
> **不写什么**：不列出无缓解策略的风险、不回避高影响风险、不重复 ADR 中已有详细分析的单点技术风险
