# SKILL.md 合规性检查清单

> 每次评审 SKILL.md 前，必须先读取本清单并逐项打勾。
> 禁止以"通读觉得没问题"替代逐项确认。

---

## 一、术语规范

每项都必须通过，禁止自定义同义替换。

- [ ] **启动检测**：所有 `/df:*` 命令启动时的文件系统扫描判断一律叫"启动检测"，不叫"状态检测""相位检测"等
- [ ] **文档写入铁律**：SKILL.md 中必有独立章节叫"文档写入铁律"，且内容包含"agent 产出后立刻写入文件 + 写入后验证文件存在 + 禁止仅在对话输出"
- [ ] **问题分级四级制**：使用 CRITICAL/HIGH/MEDIUM/LOW，且定义与以下一致：
  - CRITICAL = 阻塞性，必须修正且阻塞推进
  - HIGH = 重要缺陷，默认接受修正
  - MEDIUM = 一般问题，评估后决定
  - LOW = 轻微问题，可延后
- [ ] **评审动作叫"独立评审"**：评审 agent 产出的动作不叫"评审""检查""审阅"，统一叫"独立评审"
- [ ] **修正流程叫"评审修正循环"**：发现问题→修正→复核的迭代不叫"修改""迭代""修正"，统一叫"评审修正循环"
- [ ] **通过门槛叫"出口标准"**：每阶段的通过条件不叫"通过标准""完成标准""检查清单"，统一叫"出口标准"
- [ ] **质量评分叫"置信度"**：使用 0.00-1.00 小数制，配合维度权重表，不叫"评分""质量分""信心指数"
- [ ] **协作节奏三态**：
  - 全自动 = "自动推进"（阶段内产出→评审→修正，无需用户确认）
  - 半自动 = "确认后推进"（阶段/轮次间汇报后等用户确认）
  - 阻塞等待 = "等待人工决策"（置信度低于阈值或信息缺失时停止）
- [ ] **维护/扩展机制术语**：
  - "维护模式" = 主体流程完成后进入的按需调用状态
  - "快速通道" = 小修小补，不走完整轮次
  - "回退机制" = 大变更回退到对应子流程重新执行
- [ ] **独占词未混用**：
  - "阶段"不与"轮/层/步/Phase"混用
  - "Step"不与"子步骤/环节/步/Phase"混用

### 新 Skill 术语自检清单（开发期快速自查）

开发新 skill 时，SKILL.md 定稿前逐项确认。以下为上述术语规范的紧凑版，聚焦最高频的 10 项：

- [ ] 启动检测是否叫"启动检测"
- [ ] 文档写入章节是否叫"文档写入铁律"
- [ ] 问题分级是否使用 CRITICAL/HIGH/MEDIUM/LOW
- [ ] 评审动作是否叫"独立评审"，修正流程是否叫"评审修正循环"
- [ ] 通过门槛是否叫"出口标准"，质量评分是否用"置信度"（0.00-1.00）
- [ ] 主流程大单元是否叫"第 X 阶段"
- [ ] 阶段内子单元是否叫"步骤 X"
- [ ] 是否存在"轮/层/步/Phase/Step"与"阶段/步骤"混用
- [ ] 是否存在小数编号（如步骤 2.1）
- [ ] 多 agent 协作模式是否用"评审修正循环"内联描述，而非拆分子层级

## 二、层级命名规范

- [ ] **主流程阶段用"第 X 阶段"**：所有主流程大阶段统一叫"第 X 阶段"（如第 0 阶段、第 1 阶段），禁止混用"轮""层""步""Phase"
- [ ] **阶段内子单元用"Step X"**：阶段内的子步骤统一叫"Step 1""Step 2"等，禁止用小数编号（如 Step 2.1）、禁止混用中文数字
- [ ] **无第三层**：层级只有两层（阶段 → Step），不设第三层
- [ ] **Step 内部用评审修正循环术语内联描述**：Step 内部的多 agent 协作模式使用基础设施层的"评审修正循环"术语内联描述，不再拆分子层级

## 三、多 Agent 协作设计

### 3.1 复杂度评估与 Agent 配置

- [ ] **每个产出阶段已评估复杂度**：基于"内容类型矩阵 + 决策属性上调法"评估，非基于篇幅
- [ ] **复杂度判定基于语义属性**：复杂度判定依据是文档承载的决策权重和技术交叉度，而非篇幅行数
- [ ] **篇幅异常检测红旗**：检查是否存在以下异常（需复核）：
  - 高/极高复杂度文档篇幅过短（<100 行）
  - 低复杂度文档篇幅过长（>400 行）
  - 篇幅与同类文档历史均值偏差 >50%
- [ ] **Agent 数量与复杂度匹配**：
  - 极高复杂度 → 最小并行产出 agent ≥5，最小独立评审 agent ≥3
  - 高复杂度 → 最小并行产出 agent ≥3，最小独立评审 agent ≥2
  - 中等复杂度 → 1-2 个产出 agent，1-2 个评审 agent
  - 低复杂度 → 1 个产出 agent，1 个评审 agent
- [ ] **极高复杂度按上限配置不压减**：在 token 成本不敏感场景（如架构设计、核心需求定义），极高复杂度按上限配置 agent，不刻意压减数量
- [ ] **评审不足优先补评审不压缩产出**：评审 agent 视角需覆盖所有产出 agent 的维度，数量不足时优先补评审而非压缩产出
- [ ] **极高/高复杂度有 Plan 阶段**：Plan 阶段先读取对应 Template 识别决策权重分布，再决定"谁分析什么"，不产出分析结论
- [ ] **Plan 阶段形式正确区分**：
  - 极高复杂度：按任务类型派遣对应角色 agent（架构/技术类用 architect，需求/产品类用 product-manager，计划/调度类用 project-manager）
  - 高复杂度：主会话基于上下文自评估，或派遣 1 个与任务类型匹配的 agent 快速确认
- [ ] **Plan 阶段边界完整**：Plan 阶段只决定"谁分析什么"，包含以下允许事项和禁止事项：
  - ✅ 读取对应 Template，识别各章节的决策权重分布
  - ✅ 确定切分维度（按子系统/分析类型/职能侧重/范围深度）
  - ✅ 分配每个并行 agent 的视角约束（不同角色、不同分析范式、不同约束侧重）
  - ✅ 列出需要被显式质疑的关键假设
  - ✅ 识别产出 agent 未覆盖的视角，指导评审 agent 配置
  - ❌ 不产出深度分析内容
  - ❌ 不预设定最终结论或倾向性方案

### 3.2 视角独立性

- [ ] **禁止同构思维**：多个并行 agent 不是"一个 agent 内部发散多个方案"的替代
- [ ] **禁止同一 prompt 模板**：并行 agent 不能同一分析框架、同一输入源、同一 prompt
- [ ] **禁止润色替代产出**：不能一个 agent 产出后其他 agent 只做"润色""格式调整"
- [ ] **同角色多实例已区分**：同一预定义角色的多个实例，prompt 中明确指定了不同关注点（按范围/约束侧重/分析范式区分）

### 3.3 预定义角色

- [ ] **无临时构造角色**：所有子 agent 使用 `.claude/agents/` 中已定义的角色名，SKILL.md 中无"你是一个擅长...的助手"之类临时角色描述
- [ ] **无单次任务新类型名**：未为单次任务创造新的 agent 类型名

### 3.4 产出-评审对称

- [ ] **评审视角 ≥ 产出视角**：评审 agent 的视角丰富度覆盖或超过产出 agent
- [ ] **禁止"1 产出 + 丰富评审"**：产出阶段的结构性缺陷不能靠评审阶段的丰富度来弥补
- [ ] **评审 agent 不与产出 agent 角色重叠**：评审 agent 的角色名与产出 agent 的角色名不同（如产出用 architect，评审用 architect-reviewer）

### 3.5 并行与拆分

- [ ] **无依赖任务已并行**：文档中明确标注了无依赖任务并行执行，未因"流程描述简洁"而串行化
- [ ] **依赖判断方法明确**：无依赖任务的判断基于"如果两个任务的输出互不引用对方的结论，即可并行"
- [ ] **拆分维度正确**：复杂任务按"领域/子系统""分析类型""职能侧重""范围深度"拆分，未按"前半/后半"机械拆分
- [ ] **拆分禁止清单**：
  - 禁止按"前半部分/后半部分"机械拆分（agent 失去上下文）
  - 禁止按"写文档/改格式"拆分（后者不是 agent 工作）
  - 禁止过度拆分导致整合成本超过并行收益（>3 个并行 agent 产出同一份文档时需谨慎）
- [ ] **修正工作量分级匹配**：
  - 少量修正（<5 处）→ 1 个对应角色 agent
  - 中量修正（5-15 处）→ 2 个 agent 分块并行修正 + 主会话整合
  - 大量修正（>15 处或接近重写）→ 回退到产出阶段重新多 agent 并行发散

### 3.6 主会话边界

- [ ] **主会话只干四件事**：调度、搬运（写入文件）、整合（拼接去重不改结论）、门禁（检查标准）
- [ ] **主会话不直接产出深度分析内容**
- [ ] **主会话不对 agent 产出润色后写入**：只能原样搬运或明确标注修改
- [ ] **主会话不声称无执行记录的阶段已完成**

## 四、出口标准与评审修正循环

### 4.1 出口标准（每个有产出的阶段/Step）

- [ ] **结构性 checklist**：用 checkbox 列明通过条件，含 CRITICAL 问题处理规则 + 评审 agent 判定标准 + 文档版本/状态要求
- [ ] **量化置信度评分**：至少 3 个评分维度，每个配权重和评分标准；一般门槛 ≥0.80，不可逆决策/一致性关键路径 ≥0.85
- [ ] **低于门槛处理规则明确**：低于置信度门槛时的处理规则明确定义（自动再修正/等待人工决策/降级标注）
- [ ] **自动推进规则**：明确什么条件下自动进入下一阶段，什么条件下等待用户确认
- [ ] **评审修正循环定义**：每个产出阶段定义了"独立评审 → 修正 → 复核"循环，含：
  - 至少 2 个不同角色的 reviewer agent 并行
  - 修正工作量分级与 agent 配置
  - 循环上限（最多 3 次），3 次后降级或等人工决策
  - 终止条件：所有 CRITICAL **和 HIGH** 问题已修正 + 所有评审 agent 给出"通过"或"有条件通过"
- [ ] **产出写入后立即进入评审**：产出写入文件后，必须立即进入"独立评审 → 修正 → 复核"循环

### 4.2 豁免场景（仅限以下两种）

- [ ] **纯交互式信息收集阶段（第 0 阶段等）**：无正式文档产出时，有轻量级"准入检查清单"确认信息充分性，无评审修正循环
- [ ] **维护模式单点修正**：走"快速通道"（读取→定位→修正→落盘），修正后至少 1 个 reviewer 快速复核
- [ ] **无其他豁免**：任何有正式文档产出的阶段/Step，都有完整的出口标准和评审修正循环

## 五、自包含性

- [ ] **通用术语内联完整定义**：通用术语（评审结论三态、修正工作量分级、标准评审修正循环骨架等）在 SKILL.md 内联完整定义，禁止简化为"详见 CLAUDE.md"的引用
- [ ] **模板路径完整可解析**：引用 Template 时给出完整相对路径（如 `.claude/templates/arch-system.md`），确保任何项目上下文可定位
- [ ] **无外部行为依据**：所有影响流程编排的决策依据（复杂度判定、agent 配置、出口标准阈值）在 SKILL.md 内部定义

## 六、四文件分工

- [ ] **四类文件存放路径符合规范**：
  - SKILL.md 在 `.claude/skills/<category>/<skill-name>/SKILL.md`
  - Command 在 `.claude/commands/df/<skill-name>.md`
  - Agent 在 `.claude/agents/<role-name>.md`
  - Template 在 `.claude/templates/<template-name>.md`
- [ ] **SKILL.md 只含编排逻辑**：阶段结构、agent 调度、质量门禁、文件 I/O 规则；不含 agent 人设描述、文档具体章节结构、模板内容复述
- [ ] **Command 只含启动引导**：一句话说明 + 用法示例 + 产出物清单 + 调用 Skill 工具的指令；上限 30 行；不含流程细节、agent 调度、质量标准
- [ ] **Agent.md 只含独立人格**：身份定义、工作模式、输出风格、协作边界、通用质量准则；不含具体阶段流程、文档模板
- [ ] **Template 只含内容契约**：必填章节、内容要求、自检清单、示例片段；不含流程逻辑、agent 调度、质量门禁
- [ ] **上下文池隔离清晰**：主会话上下文 = Command + SKILL.md；子 agent 上下文 = `.claude/agents/<role>.md` + 按需读取的 Template；两个上下文池互不干扰
- [ ] **去重检查**：无内容在多个文件中重复出现。若一段话需要两处看到，说明抽象层次未分清
- [ ] **Skill 元数据约束**：Skill 相关文件（SKILL.md/Command/Agent/Template）不保留 version 和 changelog 信息，禁止在 YAML frontmatter 或正文中添加 version、changelog、变更记录等字段

## 七、核心信条体现

- [ ] **产出阶段质量上限由 agent 数量与独立性决定**：SKILL.md 的 agent 配置体现"交付件质量的上限，由产出阶段的 agent 数量与独立性决定；评审只能发现已有框架内的缺陷，无法弥补框架本身的盲区"这一核心信条

## 八、执行行为规范（CLAUDE.md 新增章节）

### 8.1 阶段顺序与启动检测

- [ ] **启动检测存在**：SKILL.md 中定义了启动检测逻辑，通过文件系统扫描判断当前所处阶段
- [ ] **主流程推进时唯一下一步**：主流程尚未完成全部规定阶段时，根据文档状态直接判定并提示**唯一的下一步动作**，未罗列多个选项让用户选择
- [ ] **维护模式允许多选项**：主体流程已完成、进入维护/持续改进模式时，允许给出多个改进方向供用户选择（如新增/完善/重构）
- [ ] **启动检测不跳过**：未跳过启动检测直接假设从零开始

### 8.2 交付件闭环后人工确认

- [ ] **交付件内部自动推进**：产出 → 评审 → 修正循环全自动闭环，未中途询问用户"是否继续"
- [ ] **交付件闭环后汇报确认**：每个重要交付件产出或更新，并经过自动化评审优化后，向用户汇报概况（产出文件、置信度、关键决策点），等待用户确认后再进入下一项工作
- [ ] **交付件内部中断条件明确**：仅在置信度低于阈值、CRITICAL 或 HIGH 未修正、或需用户做不可逆决策时，才在交付件内部中断等待人工决策
- [ ] **未混淆"自动推进"与"确认后推进"的边界**：不是简单按"阶段内/阶段间"划分，而是按"交付件是否完成闭环"划分

### 8.3 文档内评审记录长度约束

- [ ] **评审记录 ≤200 字**：产出文档正文中的评审记录章节总篇幅不超过 200 字
- [ ] **评审记录聚焦结论**：仅保留评审 agent 名称、通过结论、遗留问题数及简要分类，未展开详细评审意见
- [ ] **详细评审意见不写入文档**：详细评审意见留存于对话上下文，未写入正式交付文档
- [ ] **修正记录同样简略**：修正记录聚焦"改了什么"，控制在 200 字以内
