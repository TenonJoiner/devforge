# SKILL.md 合规性检查清单

> **双阶段使用**：
> - **设计阶段**：了解检查大类分布，确保 skill 设计不偏离规范方向
> - **评审时**：严格执行本清单，逐项打勾，禁止以"通读觉得没问题"替代逐项确认
>
> 检查覆盖：术语规范 / 层级命名 / 多 Agent 协作设计 / 出口标准与评审修正循环 / 自包含性 / 四文件分工 / 核心信条体现 / 执行行为规范 / 领域内容层不干预

---

## 零、领域适配设计规范

CLAUDE.md 明确限定了 teamskills 的适用领域和普适性要求。以下检查确认 skill 设计符合领域边界：

- [ ] **适用领域正向匹配**：Skill 明确服务于以下至少一种大型复杂基础软件领域——分布式系统（存储/计算/消息队列/数据库）、操作系统/内核、编译器/运行时、网络协议栈、虚拟化/容器、高性能计算、复杂嵌入式系统
- [ ] **适用领域反向排除**：Skill 未越界到明确不适用领域——简单的 Web CRUD 应用、小型移动 App、原型项目
- [ ] **跨领域普适性**：Skill 设计不绑定特定系统类型或编程语言，具备跨领域普适性
- [ ] **交互式领域确定**：使用前通过交互式方式确定具体系统类型（分布式系统/OS/编译器/网络栈等）和编程语言（C/C++/Rust/Go/Python/Java），而非硬编码或假设
- [ ] **自动适配机制**：后续流程根据确定的领域信息自动适配——选择对应的编码规范、测试框架、质量标准
- [ ] **禁止单一领域设计**：Skill 未为单一领域或单一语言设计，必须通过适配机制支持多领域

## 一、术语规范

每项都必须通过，禁止自定义同义替换。

### 1.1 三层分离策略

- [ ] **三层分离清晰**：SKILL.md 的术语体系明确区分"基础设施层（强制统一）""流程骨架层（规范命名规则）""领域内容层（不干预）"三个层次，未混淆层次边界
- [ ] **基础设施层术语强制统一**：所有 skill 共通的机制（启动检测、文档写入铁律、问题分级、独立评审、评审修正循环、出口标准、置信度、自动推进/确认后推进/等待人工决策）术语完全一致，无自创同义替换
- [ ] **流程骨架层遵守命名规则**：主流程用"第 X 阶段"、阶段内子单元用"步骤 X"，不强制统一各 skill 的阶段数量与内容
- [ ] **领域内容层保留差异**：产品级文档类型（ADR/Feature/Scenario/milestone-plan）、代码级概念（RED/GREEN/REFACTOR）、Agent 角色名等专业术语由各 skill 自行定义

### 1.2 基础设施层术语

- [ ] **启动检测**：所有 `/df:*` 命令启动时的文件系统扫描判断一律叫"启动检测"，不叫"状态检测""相位检测"等
- [ ] **启动检测含完整行为规则**：SKILL.md 的启动检测章节包含以下 3 条行为规则：
- [ ] **启动检测——主流程未完成行为**：主流程未完成时，根据文档状态直接判定并提示唯一的下一步动作，不罗列多个选项让用户选择
- [ ] **启动检测——维护模式行为**：主流程已完成后（维护模式），允许给出多个改进方向供用户选择（如新增/完善/重构）
- [ ] **启动检测——禁止跳过**：禁止跳过启动检测直接假设从零开始
- [ ] **文档写入铁律——章节存在**：SKILL.md 中必有独立章节叫"文档写入铁律"
- [ ] **文档写入铁律——立即写入**：Agent 产出后，主会话必须立即写入文件，不批量积累多个 agent 产出后再统一写入
- [ ] **文档写入铁律——写入验证**：写入后必须验证文件存在（`ls`/`wc` 确认），未仅凭写入操作返回值假定成功
- [ ] **文档写入铁律——禁止仅对话输出**：每轮迭代产出禁止仅在对话中输出，必须写入文件系统落盘
- [ ] **问题分级四级制**：使用 CRITICAL/HIGH/MEDIUM/LOW，且定义与以下一致：
  - CRITICAL = 阻塞性，必须修正且阻塞推进
  - HIGH = 重要缺陷，默认接受修正
  - MEDIUM = 一般问题，评估后决定
  - LOW = 轻微问题，可延后
- [ ] **问题分级核心原则**：问题级别基于对整个产品系统级的影响判定，而非对单个文档所描述的局部影响判定
- [ ] **CRITICAL 判定标准（必须同时满足 3 条）**：(1) 影响整个产品系统的核心决策或整体方向；(2) 不修正会导致后续工作推倒重来；(3) 涉及不可逆或高成本变更
- [ ] **HIGH 判定标准明确**：HIGH 的判定定义为"影响局部设计或实现质量，但不改变系统整体方向"（非仅"重要缺陷，默认接受修正"），明确它是局部质量缺陷而非改善建议
- [ ] **MEDIUM 判定标准明确**：MEDIUM 的判定定义为"优化建议，不影响功能正确性"，代表改善机会而非缺陷修复
- [ ] **LOW 判定标准明确**：LOW 的判定定义为"格式、命名、注释等非功能性问题"，限于非功能性表面问题
- [ ] **HIGH 含义精确**：HIGH 的定义包含"显著影响局部质量"（区分 HIGH 与 MEDIUM 的关键——前者影响质量/可用性，后者是改善性质）
- [ ] **MEDIUM 含义精确**：MEDIUM 的定义包含"有改善空间"（明确它代表质量提升机会而非缺陷）
- [ ] **问题分级判定基准明确**：CRITICAL = 系统级影响，HIGH = 文档/模块级影响，MEDIUM = 优化建议，LOW = 非功能性
- [ ] **判定时考虑系统上下文 1（文档位置与重要性）**：评审 agent prompt 中包含"文档在整个产品系统中的位置和重要性"的判定维度
- [ ] **判定时考虑系统上下文 2（组件可替换性）**：评审 agent prompt 中包含"所描述组件/模块的可替换性"的判定维度
- [ ] **判定时考虑系统上下文 3（影响范围）**：评审 agent prompt 中包含"错误的影响范围（仅限当前模块 vs 影响多个子系统 vs 影响整体架构）"的判定维度
- [ ] **判定时考虑系统上下文 4（修正成本）**：评审 agent prompt 中包含"修正成本（局部调整 vs 推倒重来）"的判定维度
- [ ] **常见误判纠正示例 1（文档结构≠系统级）**：子系统设计文档内部结构混乱 → HIGH（非 CRITICAL，该子系统可独立重构，不影响其他子系统）
- [ ] **常见误判纠正示例 2（内容不完整≠方向性错误）**：Feature Scenario 定义不完整 → HIGH（非 CRITICAL，该 Feature 可后续补充，不影响产品整体方向）
- [ ] **常见误判纠正示例 3（协议选型=系统级方向性）**：核心一致性协议选型错误 → CRITICAL（系统级方向性决策，影响多个子系统且难以回退）
- [ ] **常见误判纠正示例 4（架构分层=影响全局）**：整体架构分层错误 → CRITICAL（影响所有子系统，不修正导致后续工作推倒重来）
- [ ] **评审动作叫"独立评审"**：评审 agent 产出的动作不叫"评审""检查""审阅"，统一叫"独立评审"
- [ ] **"独立评审"定义完整**：SKILL.md 中"独立评审"的定义包含"一个或多个 Agent 独立产出评审意见"（强调多 agent 并行、独立产出，而非一个 agent 产出后另一个复核）
- [ ] **修正流程叫"评审修正循环"**：发现问题→回退判断→修正→复核的迭代不叫"修改""迭代""修正"，统一叫"评审修正循环"
- [ ] **"评审修正循环"定义完整**：SKILL.md 中"评审修正循环"的定义包含"发现问题 → 修正 → 复核"的完整迭代闭环，非单次评审动作
- [ ] **通过门槛叫"出口标准"**：每阶段的通过条件不叫"通过标准""完成标准""检查清单"，统一叫"出口标准"
- [ ] **"出口标准"定义完整**：SKILL.md 中"出口标准"的定义包含"用 checkbox 列表明确定义通过条件"的结构化要求（非自然语言描述）
- [ ] **质量评分叫"置信度"**：使用 0.00-1.00 小数制，配合维度权重表，不叫"评分""质量分""信心指数"
- [ ] **协作节奏三态**：
  - 全自动 = "自动推进"（阶段内产出→评审→修正，无需用户确认）
  - 半自动 = "确认后推进"（阶段/轮次间汇报后等用户确认）
  - 阻塞等待 = "等待人工决策"（置信度低于阈值或信息缺失时停止）
- [ ] **核心边界定义存在**：SKILL.md 中明确定义了用户协作节奏的核心边界——"自动推进与人工确认的切换点按交付件是否完成闭环划分，不按'阶段内/阶段间'简单切分"
- [ ] **维护/扩展机制术语**：
  - "维护模式" = 主体流程完成后进入的按需调用状态
  - "快速通道" = 小修小补，不走完整轮次
  - "回退机制" = 大变更回退到对应子流程重新执行
- [ ] **独占词未混用**：
  - "阶段"不与"轮/层/步/Phase"混用
  - "步骤"不与"子步骤/环节/步/Step/Phase"混用
- [ ] **独占词语义正确**：
  - "阶段"仅用于指代主流程大单元（第 X 阶段），不用于其他含义（如不用于子步骤的指代）
  - "步骤"仅用于指代阶段内的子单元（步骤 X），不用于其他含义（如不用于主流程大单元的指代）

## 二、层级命名规范

- [ ] **主流程阶段用"第 X 阶段"**：所有主流程大阶段统一叫"第 X 阶段"（如第 0 阶段、第 1 阶段），禁止混用"轮""层""步""Phase"
- [ ] **阶段内子单元用"步骤 X"**：阶段内的子步骤统一叫"步骤 X""步骤 1""步骤 2"等，禁止用小数编号（如步骤 2.1）、禁止混用中文数字
- [ ] **禁止混用中文数字和阿拉伯数字**：阶段编号统一使用阿拉伯数字（"第 1 阶段"），禁止同一文档中出现"第一阶段"与"第 2 阶段"混用
- [ ] **与废弃术语明确区分**：主流程大单元已废弃"第 X 步"（统一为"第 X 阶段"），仅阶段内子单元使用"步骤 X"
- [ ] **无第三层**：层级只有两层（阶段 → 步骤），不设第三层
- [ ] **步骤内部用评审修正循环术语内联描述**：步骤内部的多 agent 协作模式使用基础设施层的"评审修正循环"术语内联描述，不再拆分子层级

## 三、多 Agent 协作设计

### 3.1 复杂度评估与 Agent 配置

- [ ] **每个产出阶段已评估复杂度**：基于"内容类型矩阵 + 决策属性上调法"评估，非基于篇幅。Agent 数量由输出件复杂度决定，**禁止按阶段数量推导 agent 数量**（每个阶段复杂度需独立评估，不因"有N个阶段所以每阶段1个agent"）
- [ ] **复杂度判定基于语义属性**：复杂度判定依据是文档承载的**决策权重**和**技术交叉度**（语义属性），而非篇幅行数（语法属性）。一份 80 行的 ADR（如一致性协议选型）复杂度远高于一份 500 行的接口清单罗列——评审时确认此对比示例被正确理解，防止潜意识按篇幅判定
- [ ] **评估前已读取对应 Template**：主会话在评估文档复杂度前，必须先读取对应的 template，了解文档的章节结构和内容要求，基于 template 的具体要求结合矩阵和上调法做出判定
- [ ] **Template 各章节决策权重已识别**：Template 各章节的决策权重已被识别，复杂度判定时可追溯至 template 特定章节的要求（非泛泛引用 template 名称）。不同项目的同类文档 template 差异可能很大，严格基于当前项目 template 的具体要求判定
- [ ] **篇幅异常检测红旗完整**：检查是否存在以下三种异常，且每种异常已触发对应的调查动作：
  - [ ] 高/极高复杂度文档篇幅过短（<100 行）→ 已检查是否遗漏关键决策依据，分析深度不足
  - [ ] 低复杂度文档篇幅过长（>400 行）→ 已检查是否信息密度过低、存在冗余或模板化填充
  - [ ] 篇幅与同类文档历史均值偏差 >50% → 已复核确认是合理差异还是质量问题
- [ ] **篇幅仅用于异常检测**：篇幅不作为复杂度判定依据，仅用于发现异常（高/极高复杂度过短=分析深度不足，低复杂度过长=信息密度过低）。复杂度判定依据始终是语义属性（决策权重+技术交叉度）
- [ ] **Template 差异影响复杂度**：不同项目的同类文档 template 可能差异很大（如同是"子系统架构设计"，10 章节 vs 3 章节的 template 决定复杂度为"高"vs"中等"），复杂度评估严格基于当前项目 template 的具体要求，不跨项目类推
- [ ] **内容类型矩阵逐项核对**：每个产出文档的类型已正确判定，基准复杂度与以下矩阵一致：
  - 架构决策（ADR 级别）→ 极高
  - 系统级架构设计 → 极高
  - 需求核心 Feature / Actor 定义 → 极高
  - 子系统级设计 → 高（特别复杂的子系统如分布式存储引擎/OS 内存管理，同时涉及多个上调属性时→极高）
  - 接口 / 契约设计 → 高
  - Scenario / 用例设计 → 中等
  - 调研报告 → 中等
  - 模板化文档（如迭代计划格式填充）→ 低
- [ ] **内容类型判定标准已逐项核对**：每个产出文档不仅基准复杂度正确，其内容类型归类也符合以下判定标准：
  - 架构决策：是否涉及"影响多个子系统 + 一旦确定难以回退 + 涉及核心技术选型"
  - 系统级架构设计：是否"定义子系统边界、交互契约、全局数据流"
  - 需求核心 Feature/Actor 定义：是否"定义产品边界、Actor 权责、验收标准"
  - 子系统级设计：是否涉及"单一子系统内多技术域交叉（如 WAL + 索引 + 缓存）"
  - 接口/契约设计：是否"跨模块边界 + 变更成本高 + 需兼容演进"
  - Scenario/用例设计：是否"基于已确定 Feature 的展开 + 有明确输入输出"
  - 调研报告：是否"信息整理 + 对比分析 + 不产生新的设计决策"
  - 模板化文档：是否"结构固定 + 内容基于已有信息组合"
- [ ] **决策属性上调法逐项核对**：文档若命中以下属性，复杂度已在基准上逐项上调（每个属性 +1 级）：
  - 跨子系统边界 → +1 级
  - 不可逆决策 → +1 级
  - 性能 / 一致性关键路径 → +1 级
  - 无现成方案（需原创设计）→ +1 级
  - 安全 / 合规敏感 → +1 级
- [ ] **上调法上限效应已确认**：复杂度上调已确认以下边界规则：
  - 基准为"极高"的文档，即使命中上调属性，复杂度仍保持"极高"（不存在超极高），上调属性仅作为对极高复杂度的佐证
  - 基准为"低/中等"的文档，命中上调属性后，已正确从基准上调

### 3.2 视角独立性

- [ ] **禁止同构思维**：多个并行 agent 不是"一个 agent 内部发散多个方案"的替代
- [ ] **禁止同一 prompt 模板**：并行 agent 不能同一分析框架、同一输入源、同一 prompt
- [ ] **禁止润色替代产出**：不能一个 agent 产出后其他 agent 只做"润色""格式调整"
- [ ] **并行 agent 输入源差异化**：每个并行 agent 收到的输入材料（读取的文件/代码/上下文）与其分配的视角和约束侧重匹配，不同 agent 的输入材料有实质性差异（非同一文件集的不同排列）
- [ ] **正确做法——多维度出发**：每个并行 agent 从不同维度出发（不同专业角色、不同分析范式、不同约束侧重、不同范围切分）。四种维度中至少使用两种以上，不得仅依赖单一维度区分。各维度的具体差异方式已按以下示例参考确认：
  - 不同专业角色示例：architect vs product vs researcher
  - 不同分析范式示例：自顶向下 vs 自底向上
  - 不同约束侧重示例：性能优先 vs 可维护性优先 vs 交付速度优先（区分同角色多实例时需覆盖）
  - 不同范围切分示例：按子系统 vs 按特性域 vs 按场景类型
- [ ] **禁止同角色完全相同 prompt**：同一预定义角色的多个实例，即使关注点不同，prompt 内容也不能完全相同。完全相同 prompt 的多个实例退化为"一个 agent 内部发散"的等价形式，违反视角独立性原则
- [ ] **同角色多实例已区分**：同一预定义角色的多个实例，prompt 中明确指定了不同关注点（按范围/约束侧重/分析范式区分），禁止完全相同 prompt
- [ ] **同角色区分方式已显式标注**：同角色多实例的 prompt 中显式标明了区分方式（如"按范围区分：关注 X 模块 vs 关注 Y 模块"或"按约束侧重区分：性能优先 vs 可维护性优先"），区分方式与 CLAUDE.md 第 182-184 行列出的三种方式之一或组合对应

### 3.3 预定义角色

- [ ] **无临时构造角色**：所有子 agent 使用 `.claude/agents/` 中已定义的角色名，SKILL.md 中无"你是一个擅长...的助手"之类临时角色描述
- [ ] **无单次任务新类型名**：未为单次任务创造新的 agent 类型名

### 3.4 产出-评审对称

- [ ] **评审视角 ≥ 产出视角**：评审 agent 的视角丰富度覆盖或超过产出 agent
- [ ] **禁止"1 产出 + 丰富评审"**：产出阶段的结构性缺陷不能靠评审阶段的丰富度来弥补
- [ ] **评审 agent 不与产出 agent 角色重叠**：评审 agent 的角色名与产出 agent 的角色名不同（如产出用 architect，评审用 architect-reviewer）
- [ ] **评审 agent 之间角色互不相同**：评审阶段的多个 agent 不仅与产出 agent 角色不同，彼此之间角色也互不相同（如极高复杂度下 ≥3 个不同角色的 reviewer：architect-reviewer + product-reviewer + researcher-reviewer），确保多视角评审覆盖

### 3.5 并行与拆分

- [ ] **无依赖任务已并行**：文档中明确标注了无依赖任务并行执行，未因"流程描述简洁"而串行化
- [ ] **依赖判断方法明确**：无依赖任务的判断基于"如果两个任务的输出互不引用对方的结论，即可并行"
- [ ] **必须并行的三类场景已保障**：(1) 多个独立文档的产出；(2) 多个独立维度的调研；(3) 多个独立评审
- [ ] **拆分维度正确且含示例参照**：复杂任务按以下四个维度之一拆分，每个维度有明确适用场景和示例：

  - 按职能侧重（同一文档不同章节需不同专业能力时）— 如"结构规划 agent + 量化估算 agent + 风险分析 agent"

- [ ] **拆分主导维度明确且唯一**：拆分以四种维度之一作为主导逻辑，若确实需要组合使用（如按子系统拆分后每个子系统再按分析类型区分），Plan 中已说明组合逻辑和整合方式
- [ ] **拆分后子任务完整性**：拆分后的每个子任务包含独立完整的上下文——agent 仅凭分配的材料和 prompt 即可理解任务全貌，无需依赖其他 agent 的产出才能形成判断
- [ ] **拆分禁止清单**：
  - 禁止按"前半部分/后半部分"机械拆分（agent 失去上下文）

- [ ] **多 agent 产出同一文档的整合成本已评估**：若 >3 个并行 agent 产出同一文档 → 已评估整合成本可控，并行收益 > 整合开销（附评估依据）。非绝对禁止，但需要审慎评估（某些极高复杂度场景下 >3 agent 产出同一文档是合理的）
- [ ] **修正工作量分级匹配**（总问题数 = CRITICAL + HIGH + MEDIUM，LOW 不计入）：
  - <10 个问题 → 1 个对应角色 agent
  - 10-30 个问题 → 2 个 agent 分块并行修正 + 主会话整合
  - >30 个问题 → 评估：分批修正 or 回退重做

### 3.6 主会话边界

- [ ] **主会话应做七件事**：
  1. 读取 template 进行复杂度评估
  2. 交互式调研（前置）：通过启发式提问收集产品背景、系统定位、业务目标、技术约束等信息
  3. 调度编排：决定派遣哪些 agent、并行还是串行、何时推进下一阶段；极高/高复杂度时派遣 plan agent 并原样执行方案
  4. 信息搬运：将 agent 产出写入文件（原样搬运，不改写核心内容）
  5. 结果整合：合并多个 agent 产出（拼接/去重/选择，不改写核心结论）
  6. 质量门禁：检查出口标准、置信度阈值、agent 执行记录完整性

- [ ] **主会话不直接产出深度分析内容**（含"内容创作"和"深度专业分析"两项禁止）：不直接编写需要深度分析的文档章节（应派遣 agent 产出），不进行架构设计/需求分解/技术选型等专业思维工作（应派遣对应 agent）。简单元数据填写、文件管理、信息搬运不在此列
- [ ] **主会话不对 agent 产出润色后写入**：只能原样搬运或明确标注修改。以下行为属于允许的格式调整（非改写）：统一编号、修正拼写、补全链接路径。以下行为属于禁止的改写：调整论点结构、删除或修改分析结论、插入主会话自己的判断、改变技术建议的方向
- [ ] **主会话冲突处理遵循 Plan 规则**：多 agent 产出有冲突时，遵循 Plan 中的"冲突处理"规则合并；若 Plan 未定义冲突处理规则或冲突超出规则范围，升级至用户决策而非主会话自行选择
- [ ] **主会话不声称无执行记录的阶段已完成**
- [ ] **主会话不修改 plan agent 的方案**：如对方案有疑问，与用户确认或重新派遣 plan agent

## 四、出口标准与评审修正循环

### 4.1 出口标准（每个有产出的阶段/步骤）

- [ ] **出口标准与评审修正循环缺一不可**：每个有产出的阶段/Step，必须同时定义明确的出口标准和评审修正循环机制。任何由 agent 进行深度分析产出的阶段/Step，若缺少出口标准或评审修正循环定义，视为 SKILL.md 定稿不合格，禁止投入使用
- [ ] **结构性 checklist**：用 checkbox 列明通过条件，含以下三个子项，每个子项均有独立展开定义：
  - [ ] **子项 1——CRITICAL 问题处理规则**：已定义 CRITICAL 问题的处理规则——区分"必须修正且阻塞推进"和"可延后处理"的边界
  - [ ] **子项 2——评审 agent 判定标准**：已明确定义"通过"和"有条件通过"的判定条件，两者界限清晰（非仅罗列术语名称）
  - [ ] **子项 3——文档版本/状态要求**：已定义文档版本标识或状态标记要求（如"草稿""已评审""已定稿"等状态的判定标准和标记方式）
- [ ] **量化置信度评分**：至少 3 个评分维度，每个配权重和评分标准；一般门槛 ≥0.80，不可逆决策/一致性关键路径 ≥0.85
- [ ] **低于门槛处理规则明确**：低于置信度门槛时的处理规则明确定义（自动再修正/等待人工决策/降级标注）
- [ ] **自动推进规则**：明确什么条件下自动进入下一阶段，什么条件下等待用户确认
- [ ] **评审修正循环完整定义了五步骤**：每个产出阶段包含以下五个步骤的完整定义：
  1. **独立评审**：至少与复杂度匹配的 reviewer agent 并行（极高≥3、高≥2、中等/低≥1），不同角色并行执行，视角差异化；同时满足产出-评审对称原则（评审视角丰富度 ≥ 产出视角丰富度，禁止"1 产出 + 丰富评审"模式）
  2. **回退判断**：根据 CRITICAL 数量和框架性错误判断是否回退重做（决策表见 4.2）
  3. **修正工作量分级**：根据总问题数（CRITICAL+HIGH+MEDIUM，LOW 不计入，可延期处理）配置修正 agent——<10 个→1 个 agent，10-30 个→2 个 agent 分块并行，>30 个→评估分批修正或回退重做
  4. **循环上限与降级**：最多 3 次评审修正循环；3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或等待人工决策
  5. **终止条件**：所有 CRITICAL 问题已修正 + 所有评审 agent 给出"通过"或"有条件通过"
- [ ] **产出写入后立即进入评审**：产出写入文件后，必须立即进入"独立评审 → 回退判断 → 修正 → 复核"循环

### 4.2 回退判断

- [ ] **回退决策表完整**：
  - CRITICAL ≥3 或 (CRITICAL 1-2 且为框架性错误) → **回退重做**
  - CRITICAL 1-2 且非框架性错误 → 进入修正
  - CRITICAL = 0 → 进入修正
- [ ] **"框架性错误"定义明确**：整体方向错误、核心假设错误、架构选型根本不适用等

### 4.3 CRITICAL 控制要求

- [ ] **评审 agent 按三条判定标准识别 CRITICAL**：评审 agent 必须严格按照"问题分级"的三条判定标准（系统级影响 + 推倒重来风险 + 不可逆/高成本）识别 CRITICAL，不得仅凭单文档内部问题判定
- [ ] **CRITICAL 基于系统级影响判定**：某个被评审文档内的结构性错误 ≠ 系统级 CRITICAL。例如：子系统设计文档内部结构混乱 → HIGH（非 CRITICAL）；核心一致性协议选型错误 → CRITICAL
- [ ] **主会话调度评审时说明系统上下文**：主会话在调度评审 agent 时，在 prompt 中说明文档的系统上下文（在系统中的位置、重要性、可替换性），帮助评审 agent 正确进行系统级影响判定

### 4.4 豁免场景（仅限以下两种）

- [ ] **纯交互式信息收集阶段（第 0 阶段等）**：无正式文档产出时，有轻量级"准入检查清单"确认信息充分性，无评审修正循环
- [ ] **维护模式单点修正**：走"快速通道"（读取→定位→修正→落盘），修正后至少 1 个 reviewer 快速复核
- [ ] **无其他豁免**：任何有正式文档产出的阶段/步骤，都有完整的出口标准和评审修正循环

### 4.5 评审记录写入约束

CLAUDE.md 明确要求评审与修正记录写入正式交付文档时必须保持简略，避免喧宾夺主：

- [ ] **评审记录 ≤200 字**：产出文档正文中的评审记录章节总篇幅不超过 200 字，仅保留评审 agent 名称、通过结论、遗留问题数及简要分类，不展开详细评审意见
- [ ] **修正记录 ≤200 字**：修正记录聚焦"改了什么"，控制在 200 字以内，不展开论证过程
- [ ] **详细评审意见不写入文档**：详细评审意见留存于对话上下文，不写入正式交付文档

### 6.1 去重原则

总则：如果一段话同时需要两个地方看到，说明抽象层次没分清——要么提升到更通用的表述放一处，要么拆成两个不同粒度的版本各放各处。

#### Command 与 SKILL.md 去重

- [ ] Command 不描述阶段划分、执行逻辑、质量门禁
- [ ] Command 只列出产出物清单（文件路径），不描述如何产出
- [ ] SKILL.md 不重复 Command 中的"一句话说明"和用法示例

#### SKILL.md 与 Template 去重

- [ ] SKILL.md 不列举文档的章节结构
- [ ] SKILL.md 不描述每个章节应该写什么内容
- [ ] SKILL.md 只引用 template（说"读取 template X"或"按 template 要求"），不展开 template 内容
- [ ] Template 不描述何时产出、由谁产出、如何评审

#### SKILL.md 与 Agent.md 去重

- [ ] SKILL.md 不描述 agent 的角色定位、专业领域、工作模式
- [ ] SKILL.md 不描述 agent 的输出格式、深度、长度要求
- [ ] SKILL.md 调度 prompt 只说"派遣 architect 分析 X"，不描述 architect 是什么角色
- [ ] Agent.md 不描述何时被派遣、在哪个阶段、派遣条件
- [ ] Agent.md 不描述整体流程编排逻辑

#### Agent.md 与 Template 去重

- [ ] Agent.md 不列举文档的章节划分
- [ ] Agent.md 只说"按 template 要求写作"，不复述 template 的章节结构
- [ ] Template 不描述由哪个 agent 写作、agent 的能力要求

#### 跨文件通用去重原则

- [ ] **权威定义唯一**：信息在 A 文件是权威定义，B 文件需要时应该引用而不是复制
- [ ] **禁止"方便"复制**：禁止"为了方便"而在多个文件中重复同一信息
- [ ] **重复即设计问题**：发现需要在多个文件中重复，说明抽象层次有问题，应该重新设计

## 十一、项目核心原则检查

CLAUDE.md 第 7-11 行定义了项目级三大核心原则，决定 Skill 的整体设计方向和架构定位：

### 团队协作优先

- [ ] **团队协作优先已体现**：Skill 设计以团队整体提效和高效协作为目标，通过 git worktree 隔离、Agents 并行分工等机制减少并行开发冲突，未以个人开发者效率为唯一设计导向
- [ ] **多成员并行安全性**：Skill 产出的文件路径包含隔离机制（如 worktree/分支名称），状态文件支持多成员独立会话，不会因多人同时使用同一 skill 产生文件冲突

### 四层分层驱动

- [ ] **层级定位明确**：Skill 清晰定位自身属于产品级 / 特性级 / 代码级 / 测试验证级中的某一层，设计遵循该层的职责边界
- [ ] **上游约束可追溯**：Skill 的输入来源（上游层的产出物）已明确定义，且流程中校验了上游产出的存在性和完整性
- [ ] **下游反馈机制**：Skill 定义了当下游层发现问题时如何向上反馈的机制（如代码级发现架构问题如何反馈到产品级）
