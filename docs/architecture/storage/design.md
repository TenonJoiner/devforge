## 1. 设计目标

### 1.1 核心问题

Storage 子系统是分布式 KV Cache Offloading 系统中负责数据物理生命周期管理的核心层。它回答的关键问题是：在多级异构存储介质（GPU HBM / CPU DRAM / NVMe SSD / 远程节点）之间，KV Cache 数据如何在正确的时间出现在正确的位置，并且以可接受的延迟和成本完成写入、读取、晋升、淘汰与回收。

如果没有 Storage 子系统，或该子系统设计不当，将直接导致：
- **TTFT 抖动**：SSD 层在竞争场景下 P99.9 延迟可能暴涨 2.9 倍，首 token 无法在规定窗口内生成；
- **HBM 利用率低下**：细粒度冷热分离缺失，导致宝贵的 GPU 显存被一次性数据污染；
- **带宽浪费**：异构介质间的数据搬运缺少策略控制，SSD 写放大和网络带宽被无效预取消耗；
- **扩展性受限**：单节点容量耗尽后无法平滑接入远程层，整个集群吞吐受限于单机存储上限。

### 1.2 核心职责边界

Storage 子系统的职责聚焦于「数据的在哪里」和「数据怎么搬」，不负责：
- **前缀匹配与逻辑寻址**：由 Metadata 子系统负责维护前缀到物理位置的映射；
- **请求调度与算子执行**：由 Connector/推理引擎负责决定何时发起 KV Cache 的分配和加载请求；
- **跨节点传输协议细节**：由 Transport 子系统负责封装 RDMA/TCP 等底层网络操作。

Storage 从 Metadata 接收「位置提示」，但会对提示做实际可用性校验；迁移完成后异步通知 Metadata 更新。Storage 接收 Connector 的「分配/加载」请求，但具体的容量配额和热点晋升策略由 Storage 内部自治。

### 1.3 质量目标

| 维度 | 目标 | 说明 |
|------|------|------|
| 延迟 | L2 命中 < 10μs，L3 命中 < 1ms | 跳层直达减少中间层 staging |
| 吞吐 | PCIe/NVMe 带宽利用率 > 60% | 通过异构粒度匹配 IO 最佳块大小 |
| 内存效率 | HBM 内部碎片率 < 4% | 与 vLLM 16-token 粒度对齐 |
| 可扩展性 | 单节点满后可通过 L4 无缝扩展 | L4 作为容量扩展层而非延迟敏感层 |
| 可靠性 | SSD 写竞争下 P99.9 增量 < 50ms | 基于基线 15ms，write buffer + 异步写回缓解 |

**P99.9 增量 < 50ms 的量化推导**：
- 反标杆研究（SolidAttention，USENIX FAST '26）显示，三星 990 PRO NVMe SSD 在 800k IOPS 后台随机读竞争下，decode 阶段 P99.9 延迟暴涨 2.9 倍。
- 该论文未给出绝对延迟值，但提供了平均延迟基线：空载 SSD 读取延迟约 100-200μs， decode 阶段涉及多层缓存命中时平均延迟约 5-15ms。
- 我们选取 **L3 命中基线延迟 = 15ms** 作为保守估计（包含数据反序列化、PCIe 传输、内核态调度等综合开销）。
- 2.9 倍意味着竞争下可能达到 43.5ms，增量约 28.5ms。
- 然而 SolidAttention 的测试环境是单consumer PC，无 write buffer 隔离；我们的设计引入 32KB/layer DRAM write buffer 和 IO 并发限速（最大并发=4），参考 LMCache 实测可将尾延迟增量压缩约 40-50%。
- 因此保守估计：28.5ms × 0.6 ≈ 17ms（理想情况），但考虑到多实例共享节点、SSD 固件 GC 等不可控因素，留足 3x 安全余量，设定 **P99.9 增量 < 50ms** 为可达成目标。
- **关键前提**：该目标仅在 DRAM write buffer 生效、同步 IO 并发受限、且 SSD 本身健康（无严重磨损降级）时成立。若基线测量值偏离 15ms 超过 ±30%，此目标需重新校准。

### 2.2 模块职责与协作

#### API Router（请求入口）

- **职责**：统一接收来自 Connector 和 Metadata 的请求，进行参数校验、权限检查、请求格式转换。
- **输入**：Connector 的 `allocate` 请求、`load` 请求；Metadata 的 `location hint` 查询和 `post-migration notify`。
- **输出**：经路由后的内部请求对象，分发至 Request Dispatcher。
- **协作**：与 Metadata 保持弱一致——Metadata 返回的位置提示可能过期，API Router 将提示透传，但由 SpaceManager/TierManager 做最终可用性确认。

#### Request Dispatcher（同步请求 vs 异步任务路由）

- **职责**：区分「用户请求关键路径」（同步）与「后台搬运任务」（异步），防止后台晋升/淘汰阻塞同步 IO。
- **输入**：API Router 转发的请求；HotnessTracker 触发的自动迁移信号；PrefetchEngine 触发的预取信号。
- **输出**：同步请求进入 SpaceManager 快速路径；异步任务进入 DataMover 和 TierManager 后台队列。
- **协作**：与 DataMover 通过优先队列协作，同步请求持有更高调度权重。
- **调度算法**：采用两级反馈队列（MLFQ）变体。
  - **Q0（同步请求队列）**：严格 FIFO，时间片 2ms，超时未完成的请求降级到 Q1；
  - **Q1（高优先级异步）**：预取任务、晋升任务，加权公平队列（WFQ），权重 0.6；
  - **Q2（低优先级异步）**：淘汰任务、回收任务、大批量 L3→L4 迁移，WFQ 权重 0.4。
  - **抢占规则**：Q0 中的请求到达时，若当前处理的是 Q1/Q2 中的异步任务，当且仅当该异步任务处于「可安全暂停点」（如数据拷贝间隙、非临界区）时允许抢占，避免数据损坏。

#### SpaceManager（容量/配额管理）

- **职责**：维护各层的实时剩余容量、总配额上限、预留缓冲。决定「新数据能不能放进来」，不负责「数据放到哪个具体位置」。
- **输入**：分配请求（大小、期望最低层级）、回收通知。
- **输出**：分配结果（成功/失败，若失败则返回建议层级）、配额告警。
- **协作**：与各 TierManager 保持反向心跳，TierManager 上报层内可用空间变化；SpaceManager 在 L1 不足时触发 L2→L3 的异步淘汰信号。
- **心跳机制**：TierManager 每 100ms 向 SpaceManager 推送一次容量变化事件（增量）；若 SpaceManager 连续 3 个周期（300ms）未收到某层心跳，则将该层标记为 **DEGRADED**，暂停向该层分配新数据，并向监控告警。

#### DataMover（层级间数据搬运引擎）

- **职责**：执行实际的跨层数据拷贝、合并、拆分、校验。屏蔽不同介质的 IO 差异。
- **输入**：Dispatcher 下发的迁移任务（源层级、目标层级、数据描述符、优先级）。
- **输出**：迁移完成/失败事件、异步通知 Metadata 的变更摘要。
- **协作**：调用 TierManagerFactory 获取源/目标层的读写接口；调用 Remote Data Mover 处理 L4 远程搬运；迁移完成后回调 HotnessTracker 更新状态。

#### PrefetchEngine（异步预取引擎）

- **职责**：基于请求特征和历史模式，提前将数据从深层搬运到较浅层，将延迟转化为带宽问题。
- **输入**：工作负载模式信号（如 Connector 提供的请求批次提示）、命中 L3/L4 的记录、预测模型输出。
- **输出**：预取任务队列，提交给 Request Dispatcher 的异步通道。
- **协作**：受 SpaceManager 的缓冲配额约束，预取带宽上限设为总带宽的 30%，不可配置覆盖；预取任务设置 5 秒硬超时，超时后自动取消并标记为「低优先级淘汰」。

#### TierManagerFactory & 各层 TierManager

- **职责**：为每一层级提供统一的存储管理抽象。L1Mgr 到 L4Mgr 分别处理 HBM、DRAM、SSD、Remote 的介质特性。
- **输入**：读写请求、分配请求、淘汰信号。
- **输出**：IO 结果、空间变化事件、层级内统计指标。
- **协作**：
  - L1Mgr 与 L2Mgr 之间处理 16-token → 256-token 的合并/拆分；
  - L3Mgr 内聚 SSD write buffer（32KB/layer），合并随机写入为顺序写；
  - L4Mgr 通过 Remote Data Mover 发起远程操作。

#### HotnessTracker（热度追踪器）

- **职责**：近似统计 KV Cache 块的访问频率和 recency，为跳层直达提供决策依据。
- **输入**：每次命中/写入的位置和时间戳（采样率 1/10）。
- **输出**：热度评分、晋升建议（到 HBM）、淘汰建议（到 SSD/Remote）。
- **协作**：采用 Count-Min Sketch（宽度 2048，深度 4，内存约 32KB/层）+ 指数衰减滑动窗口（半衰期 5 分钟）；故障或误判时支持保守降级（回退至严格 LRU）。
- **更新公式**：每次采样访问时，CMS 计数器递增 1，同时应用时间衰减因子 `decay = 2^(-Δt/300s)`，其中 Δt 为距离上次更新的秒数。热度评分 `score = CMS_estimate × decay_factor × recency_bonus`，recency_bonus 在最近 60 秒内访问过则 ×1.5。

#### RecycleManager（回收管理器）

- **职责**：管理被删除、过期或损坏数据的物理回收，防止不同层级出现幽灵数据。
- **输入**：元数据变更通知（数据失效）、各层周期性扫描结果。
- **输出**：异步回收任务、空间释放确认。
- **协作**：与各 TierManager 调用回收接口；回收完成后通知 SpaceManager 更新容量视图。

#### Remote Data Mover（远程数据搬运语义接口）

- **职责**：向 Transport 子系统发起**语义化的远程数据搬运请求**，仅表达「从节点 A 的某地址范围读取/写入到节点 B 的某地址范围」的意图，不感知底层协议细节。
- **输入**：远程读写请求（源节点ID、目标节点ID、地址范围、数据长度、优先级、超时）。
- **输出**：完成/失败结果（成功、超时、对端不可达、校验失败），**不暴露连接状态、重试次数、底层协议类型**。
- **协作**：与外部 Transport 子系统对接；MVP 阶段要求同时支持 RDMA 路径和 TCP 降级路径，但具体选路、连接生命周期管理、超时重试策略均由 Transport 子系统负责。

## 3. 核心概念与抽象

### 3.1 数据组织单元

Storage 子系统内部使用四层异构粒度管理数据单元。本表格已与 ADR-004 修正后的粒度决策保持一致：

| 层级 | 粒度 | 数据单元名称 | 典型大小（70B 模型，FP16，4096 维 hidden size） |
|------|------|-------------|-----------------------------------------------|
| L1 HBM | 16 tokens | Block | ~45 MB |
| L2 DRAM | 256 tokens | Chunk | ~720 MB |
| L3 SSD | 256 tokens | Segment | ~2 MB（头部元数据按 4KB 对齐，数据体顺序写入） |
| L4 Remote | 65536 tokens | Object | ~185 GB |

**粒度映射关系**：
- L1 → L2：16 Blocks 合并为 1 Chunk，在 DataMover 后台线程异步执行；
- L2 → L3：粒度相同（均为 256 tokens），无需合并/拆分，L3 的价值体现在 DRAM write buffer 的异步刷盘、TTL 管理、以及跨 Chunk 的顺序写聚合；
- L3 → L4：256 个 Segments 批量聚合为 1 Object，最大化网络带宽利用率。

**L2/L3 同粒度下的分层差异化价值**：
虽然 L2（DRAM）与 L3（SSD）的数据粒度相同，但两层的价值定位截然不同：
1. **访问延迟**：L2 命中延迟 < 10μs，L3 命中延迟约 0.2-1ms，相差 2 个数量级；
2. **持久化保障**：L3 提供非易失性存储，允许节点重启后数据可恢复；L2 在重启后必须重建；
3. **写缓冲隔离**：L3 配备 32KB/layer 的 DRAM write buffer，将细粒度随机写合并为顺序写，降低 SSD 写放大；L2 无此缓冲，直接映射到 DRAM；
4. **容量与成本**：L2 受限于单机 DRAM（~1TB），L3 可扩展至单机 NVMe SSD（~60TB），$/GB 相差约 30 倍。

**为什么 L2 不需要 write buffer，而 L3 需要 32KB/layer？**
- L2 直接映射到 CPU DRAM，DRAM 的随机写延迟约为 50-100ns，且不存在写放大或磨损问题。因此 L2 可以直接接受细粒度写入，无需额外的 write buffer 聚合。
- L3 面对的是 NAND Flash SSD，其随机 4KB 写延迟约为 50-200μs，且固态硬盘的 FTL（Flash Translation Layer）会将随机写转化为内部的写放大。如果不做聚合，256-token Segment（约 2MB）的写入可能被拆分为数百个 4KB 随机写，导致严重的写放大和控制器压力。
- **32KB/layer 的推导依据**：SolidAttention 论文 §7 中明确使用了 "32KB per-layer write buffers" 来缓解 SSD 磨损。我们选择 32KB 的原因：
  1. 它是 PCIe Gen4/Gen5 TLP（Transaction Layer Packet）最大 payload 大小（256B）的整数倍，能够高效利用 DMA 传输；
  2. 32KB 约等于 16 个 2KB 的 KV Cache 层切片，对于常见的大模型（70B，80-128 层）来说，这是一个小到可以控制延迟、大到可以有效聚合的 sweet spot；
  3. 对于 NVMe SSD，32KB 已经能够触发顺序写模式，充分发挥 NAND 的 page program 效率。
- **统一 DRAM-SSD 缓存层对比**：为什么不把 L2 和 L3 合并为一个统一的 DRAM-SSD 缓存层？因为统一缓存层会模糊「延迟敏感区」和「持久化扩展区」的边界。在统一缓存层中，所有写入都会先进入 DRAM，然后再异步下沉到 SSD，这意味着即使是只需要在 DRAM 中短暂停留的热数据，也会经历一次不必要的 SSD 写路径（增加写放大）。而 L2/L3 分层允许热数据直接停留在 L2，完全绕过 SSD 写路径。

### 3.2 核心元数据结构（高层描述）

Storage 子系统内部使用统一的**数据对象描述符**来追踪跨层数据。描述符包含以下核心信息：
- **全局标识**：唯一标识该 KV Cache 数据对象；
- **Token 范围**：覆盖的 `[start_token, end_token)`；
- **层位图**：标记该对象当前在哪些层级存在完整副本；
- **主位置**：当前 PRIMARY 层级（L1/L2/L3/L4）及可选的副本位置列表；
- **迁移状态**：当前是否处于 STABLE、PROMOTING、DEMOTING 或 PREFETCHING 状态，以及乐观锁版本号；
- **热度指标**：滑动窗口访问次数、最近一次访问时间戳、预测复用时间（由 PrefetchEngine 给出）。

> 注：具体的字段定义、序列化格式、接口契约属于接口规格范畴，详见 `docs/interfaces/storage-metadata.md` 与 `docs/interfaces/storage-internal.md`。

### 3.3 跳层直达判定逻辑

跳层直达的核心判定条件基于三重信号：

1. **频次信号**：滑动窗口内访问次数 >= 3 次，直接触发 L3/L4 → L1 晋升；
2. **时间信号**：相邻两次访问间隔 < 1 分钟，触发 L2 ↔ L1 快速晋升/保留；
3. **负面信号**：重用距离估计 > 阈值（如未来 5 分钟内预计无访问），触发 L1 → L3/L4 直接淘汰（跳过 L2）。

判定流程：

```
当收到读取请求时：
  1. 查询 HotnessTracker，获取目标对象的热度评分
  2. 若评分 >= 高频阈值 AND 当前位于 L3/L4：
       -> 发起异步 PROMOTE 到 L1（带优先级标记）
  3. 若评分处于中频区间 AND 当前位于 L2：
       -> 保持 L2 不变，命中返回
  4. 若评分 < 低频阈值 AND 当前位于 L1/L2：
       -> 标记为候选淘汰对象，由 SpaceManager 触发异步 DEMOTE
  5. 当 SpaceManager 感受到 L1 容量压力时：
       -> 优先淘汰标记为"将死"的对象，直接落到 L3
```

**阈值调参方法**：
- 高频阈值初始设为 CMS 估计值的 TOP 15%（保守策略），通过线上 A/B 逐步收敛到理论最优（TOP 10%）；
- 低频阈值设为滑动窗口内访问次数 < 1；
- 阈值调整周期为每天凌晨低峰期，基于过去 24 小时的命中率和延迟反馈自动微调，单次调整幅度不超过 ±5%。

### 3.4 异步预取触发条件

PrefetchEngine 维护一个轻量级的请求模式窗口：

- **session_prefix**：当前会话的前缀特征标识；
- **historical_pattern**：同一前缀在过去 N 次请求中的 token 扩展模式；
- **engine_hint**：Connector 提供的下批次请求范围（可选）。

当收到请求时，若 `engine_hint` 存在且前缀与历史模式匹配度 > 70%，则提前将后续 Segment/Object 从 L3/L4 加载到 L2，预取窗口需保证预取完成时间早于预测使用时间至少一个 L3 读取延迟单位（约 200-1000 μs）。

## 4. 方案对比与推导过程

### 4.1 核心约束条件

在设计 Storage 子系统时，以下约束是不可妥协的：

**约束 A：延迟可预测性**
- TTFT 是核心 SLO。SSD 在后台竞争下尾延迟可能暴涨 2.9 倍，Storage 必须能在竞争环境下保证 P99.9 增量 < 50ms。

**约束 B：HBM 利用率**
- GPU 显存是最昂贵的资源，内部碎片率必须 < 4%，且不能被一次性数据长期占用。

**约束 C：带宽匹配**
- PCIe Gen5 ~50 GB/s，NVMe SSD 顺序 ~14 GB/s，但小粒度随机 IO 下利用率可能 < 5%。必须匹配各介质的最佳块大小。

**约束 D：渐进扩展**
- 系统需要从单机分层演进到集群分离式架构。Storage 必须支持 L4 远程层作为容量扩展，而非一次性引入复杂的全局调度。

### 4.2 候选方案对比

#### 方案 X：严格分层 LRU

- **原理**：HBM → DRAM → SSD → Remote 逐层降级，命中后逐层回填，每层独立 LRU。
- **在约束下的适配度**：
  - 约束 A：不满足。SSD 命中后需经过 DRAM staging，累计延迟 200+ μs；尾延迟抖动无法预测。
  - 约束 B：部分满足。LRU 机制简单，但中间层容易被一次性数据污染，降低 HBM 有效利用率。
  - 约束 C：不满足。逐层回填导致同一份数据在多层间反复拷贝，浪费带宽。
  - 约束 D：满足。实现简单，易于渐进扩展。

#### 方案 Y：跳层直达 + 热度分类

- **原理**：引入全局热度统计，热数据可直接从深层晋升到 HBM，冷块可直接从 HBM 淘汰到 SSD/Remote。
- **在约束下的适配度**：
  - 约束 A：满足。跳过中间层 staging，L3 命中端到端延迟可从 200-500 μs 降至 50-200 μs；结合 write buffer 吸收抖动。
  - 约束 B：满足。热数据精准驻留 HBM，冷数据不入 DRAM，减少污染。
  - 约束 C：满足。减少无效层间拷贝，DRAM 带宽用于有效数据和预取。
  - 约束 D：部分满足。实现复杂度高于严格分层，但仍在单节点可控范围内。

#### 方案 Z：Tiering + 预取（组合主架构）

- **原理**：将预取引擎与严格分层深度融合，预取决策直接影响数据在分层中的位置，而非作为独立增强层。
- **在 ADR-003 评估矩阵中的表现**：性能 10/10，实现复杂度 4/10，内存效率 8/10，适用性 7/10，综合评分最高。
- **为何未选作主架构**：
  1. **实现复杂度 4/10 的风险被严重低估**：Tiering+预取要求预取决策与 TierManager 的状态机深度耦合。预取失败时不仅要取消任务，还要回滚 TierManager 中已预留的容量分配，这对并发一致性要求极高；
  2. **与推理引擎强耦合**：该方案若要发挥 10/10 性能，PrefetchEngine 必须深度侵入 Connector/推理引擎的调度循环，获取细粒度的解码计划。这与 ADR-006 中「Library 模式为主、与 vLLM 深度耦合风险高」的决策相冲突；
  3. **预取失效的级联影响不可控**：当预取预测失败时，不仅浪费带宽，还会因 Tiering 的预留机制导致热数据被挤出 HBM，形成「预取失败 → 热数据降级 → 命中率下降 → 更激进预取」的死亡螺旋；
  4. **退化路径不清晰**：跳层直达（方案 Y）在 PrefetchEngine 完全失效时，系统仍然是一个功能完整、延迟可预测的分层存储系统；而 Tiering+预取在预取失效时，TierManager 的状态机中存在大量「预取预留但未命中」的悬空状态，难以快速退化为可用模式。

#### 方案 W：异步预取主导

- **原理**：基于预测模型提前将深层数据搬到浅层，理论上隐藏全部传输延迟。
- **在约束下的适配度**：
  - 约束 A：若预测准确则最优；但预测失败时浪费带宽且污染缓存，反而恶化延迟。
  - 约束 B：不确定。预取数据若预测失败，将占据 HBM/DRAM 空间，降低有效利用率。
  - 约束 C：预测失败时直接违反。无效预取占用总带宽，可能导致同步请求被挤占。
  - 约束 D：满足。预取引擎本身可渐进增强。

### 4.3 推导结论

**选择：跳层直达 + 异步预取组合**

推导逻辑：
1. 单独使用严格分层 LRU（方案 X）无法同时满足延迟可预测性和带宽效率；
2. 单独使用异步预取（方案 W）风险过高，预测准确率不可控；
3. **跳层直达（方案 Y）是解决核心延迟和利用率问题的基础架构**——它在预取引擎完全关闭时仍能独立运行，提供了清晰的退化基线；
4. **异步预取作为正交增强层叠加**，在有高置信度预测信号时发挥作用，无信号时不盲目预取；
5. **Tiering+预取（方案 Z）评分更高但未被选作主架构**，是因为其高评分依赖于「预取与推理引擎深度耦合」这一不可接受的假设，且实现复杂度和失效退化路径均劣于「跳层直达+预取」组合。

为了更严谨地论证选择，我们用以下不等式描述预取增强的有效性边界：
- 设预取命中率为 `p_hit`，预取带宽占比为 `b_prefetch`（上限 30%），同步请求带宽需求为 `b_sync`；
- 预取有效的条件是：`p_hit × (L3_latency - L2_latency) > (1 - p_hit) × b_prefetch / (b_sync + b_prefetch) × sync_degradation`。
- 代入典型值：p_hit = 80%，L3-L2 延迟差 ≈ 500μs，sync_degradation ≈ 50μs，则 LHS ≈ 400μs，RHS ≈ 30μs，净收益显著为正。
- 当 p_hit 降至 50% 时，LHS ≈ 250μs，RHS ≈ 75μs，仍然有收益，但群内收益已收窄至约 175μs；低于此阈值则触发 PrefetchEngine 保守降级（缩小预取窗口或直接关闭）。

**牺牲的代价**：
- 热度统计本身引入额外状态维护开销（通过 Count-Min Sketch + 采样降低至可忽略）；
- 跳层误判时（冷数据被预测为热），性能会短暂低于严格分层 LRU，但通过保守阈值和快速回退机制控制影响范围；
- 未选择方案 Z 意味着理论性能上限（10/10）无法完全达到，但换取了架构稳定性和渐进演进能力，这是以 TTFT 可预测性为核心的系统中更可接受的 trade-off。

## 5. 动态行为描述

### 5.1 系统启动初始化序列

Storage 子系统的启动遵循严格的依赖顺序，确保各模块在对外暴露接口前完成内部状态恢复：

```
Phase 1 (0-50ms):   加载持久化配置与元数据快照
  └─► ConfigLoader 读取上次正常关闭时保存的 TierManager 容量状态
      └─► 若发现上次为非正常关闭，标记 L3 为 NEED_SCRUB（需一致性校验）

Phase 2 (50-200ms): 初始化 TierManagerFactory 与各层介质管理器
  └─► L1Mgr: 向 GPU 驱动注册显存池
  └─► L2Mgr: 分配 pinned DRAM 区域
  └─► L3Mgr: 打开 SSD 数据文件，校验头部 magic number 和校验和
      └─► 若校验失败，报告 CORRUPTED，进入只读降级模式
  └─► L4Mgr: 初始化 Remote Data Mover，但此时不建立连接

Phase 3 (200-300ms): 启动后台服务模块
  └─► HotnessTracker: 从 checkpoint 恢复 CMS 结构（若存在）
  └─► SpaceManager: 汇总各 TierManager 上报的容量，计算全局可用空间
  └─► RecycleManager: 启动后台扫描线程，标记待回收区域

Phase 4 (300-500ms): ready 验证与对外暴露
  └─► API Router 启动 health check endpoint
  └─► 若全部 TierManager 返回 HEALTHY，则向外部发布 READY 事件
      └─► 若任一 TierManager 返回 DEGRADED，Storage 仍可服务，但向监控告警
```

**故障恢复分支**：
- **L3 非正常关闭**：启动时触发 `NEED_SCRUB`，RecycleManager 在后台逐 Segment 校验校验和，预计 1TB 数据需 30-60 秒；期间 L3 可读但写操作暂存于 DRAM write buffer 队列，scrub 完成后批量刷盘。
- **L4 连接不可达**：L4Mgr 启动时不阻塞等待连接，而是在首次收到 L4 相关请求时 lazy 初始化。若连续 3 次连接失败，标记 L4 为 UNAVAILABLE，新分配请求回退到 L3 并触发容量告警。

### 5.2 并发控制模型

Storage 子系统采用**事件循环 + 工作线程池**的混合并发模型：

```
┌─────────────────────────────────────────────┐
│             同步请求事件循环 (1 per GPU)      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ API Router│─►│Dispatcher│─►│SpaceManager│─►│TierManager│
│  │  (无锁)   │  │ (原子操作)│  │ (读写锁)  │  │ (per-layer锁)│
│  └─────────┘  └─────────┘  └─────────┘     │
│         关键路径：单线程事件循环，避免锁竞争    │
└─────────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│           异步任务工作线程池 (N=CPU cores)     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │DataMover │  │PrefetchEngine│  │RecycleMgr│
│  │ (任务队列)│  │ (预测计算)   │  │ (后台扫描) │
│  └─────────┘  └─────────┘  └─────────┘     │
│         非关键路径：并行处理，通过消息传递交互  │
└─────────────────────────────────────────────┘
```

**锁粒度与策略**：
- **API Router / Dispatcher**：运行在单个事件循环线程中，内部状态变更无需加锁，通过无锁队列（MPSC）与外部交互；
- **SpaceManager**：使用读写锁（read-heavy）。容量查询走读锁，分配/释放走写锁；写锁持有时间必须 < 10μs；
- **TierManager**：每层独立 mutex。L1Mgr 的锁竞争最激烈，因此采用细粒度 per-Block 锁（256 个锁桶，按 Block ID 取模），将锁冲突概率降低至 1/256；
- **HotnessTracker**：CMS 更新使用原子操作，无需显式锁；滑动窗口裁剪在独立后台线程每 5 秒执行一次；
- **DataMover / PrefetchEngine / RecycleManager**：任务队列使用无锁 MPMC 队列，任务之间无共享可变状态。

### 5.3 故障恢复与降级流程（L3Mgr / DataMover 故障场景）

#### 场景 A：L3Mgr SSD 不可读（硬件故障或文件损坏）

```
[L3Mgr 检测到 read() 返回 EIO]
        │
        ▼
[通知 SpaceManager: L3 标记为 DEGRADED]
        │
        ▼
[SpaceManager 触发容量重算]
  ├─► 新分配请求：回退到 L2（若 L2 空间不足则直接拒写或降级到 L4）
  ├─► 已在 L3 中的数据：标记为 ORPHAN，命中的请求回退到 MISS（触发重算）
  └─► 后台任务：暂停所有 L3↔L2/L4 的迁移
        │
        ▼
[DataMover 报告迁移失败给 Metadata]
        │
        ▼
[Metadata 更新位置索引，将受影响的 prefix 标记为 unavailable]
        │
        ▼
[告警：触发 ops 介入，判断是否需要更换 SSD 或恢复备份]
```

**降级 SLA**：L3 故障检测到完成降级（DEGRADED 标记传播到所有相关模块）的总时间 < 500ms。此期间命中的请求可能体验到 1-2 次额外 MISS，但不会影响已加载到 L1/L2 的数据。

#### 场景 B：DataMover 后台任务崩溃

```
[DataMover 线程异常退出]
        │
        ▼
[Watchdog 检测心跳丢失，3 秒内重启 DataMover]
        │
        ▼
[重启后从任务队列恢复未完成任务]
  ├─► 迁移中的任务：检查源和目标层的数据一致性校验和
  │    ├── 一致：任务已完成，跳过
  │    └── 不一致：重新执行迁移
  └─► 新任务：正常调度
        │
        ▼
[若 3 秒内无法重启，则暂停异步迁移，同步请求继续服务]
```

    │              │                │  [Async Batch]   │           │        │           │        │         │
    │              │                │──merge_task─────►│           │────────┼─────────►│        │         │
    │              │                │                  │           │        │           │        │         │
    │              │                │                  │           │        │           │──copy─►│         │
    │              │                │                  │           │        │           │◄─done──│         │
    │              │                │                  │           │        │           │        │         │
    │              │                │──notify_metadata()─────────────────────────────────────────────────────►│
    │              │                │                  │           │        │           │        │         │
```

**写入路径的并发冲突处理**：
- **写入 vs 晋升冲突**：当 Connector 正在向 L1 写入某 Block 时，若 HotnessTracker 同时触发该 Block 的晋升任务，DataMover 会检查该 Block 的写入锁。若写入尚未完成，晋升任务延迟到写入完成后执行；DataMover 的迁移任务队列按 Block ID 分片，确保同一 Block 的写入和晋升不会并发执行。
- **锁持有区间**：L1Mgr 的 per-Block 锁仅在数据拷贝期间持有（约 5-20μs）；SpaceManager 的写锁仅在分配/释放间隙持有（<10μs）。DataMover 不会在持锁期间调用其他模块，避免死锁。

### 5.5 数据晋升/淘汰状态转换图

**状态规则**：
- 跳层直达允许 L4 → L1、L3 → L1、L1 → L3 等跨层转换；
- `MIGRATING` 状态期间，前台查询可访问源层数据，目标层数据对查询不可见；
- `PREFETCH` 状态的数据优先被淘汰，不会触发普通数据的淘汰链式反应。

### 5.6 DataMover 任务状态机与幂等性保证

DataMover 是跨层数据一致性的核心执行者。每个迁移任务都有明确的生命周期状态机：

```
                    +-----------+
                    |  PENDING  |
                    | (任务入队) |
                    +-----+-----+
                          |
                          v
                    +-----------+
                    |IN_PROGRESS|
                    |(拷贝执行中) |
                    +-----+-----+
                          |
            +-------------+-------------+
            |                           |
            v                           v
    +-------------+            +------------------+
    |  COMMITTED  |            |    ABORTED       |
    |(目标层+元数据|(           |(失败/超时/校验  |
    |  均已更新)  |            |   不一致，回滚)  |
    +-------------+            +------------------+
```

**幂等性保证**：
- 每个 DataMover 任务携带全局唯一的 `task_id`（由源层 ID + 目标层 ID + 数据对象描述符的 hash 生成）。
- SpaceManager 维护一个 `recent_task_log`（最近 60 秒已完成任务的 task_id 集合）。当同一 task_id 被重复提交时：
  - 若任务已在 `COMMITTED` 状态，直接返回成功；
  - 若任务在 `IN_PROGRESS` 状态，等待其完成；
  - 若任务在 `ABORTED` 状态，允许重新调度一次（重试上限 = 3）。
- **避免重复拷贝**：在写入目标层前，DataMover 先查询目标层是否已存在相同校验和的数据对象。若存在，跳过实际拷贝，直接进入元数据更新阶段。

**"写后崩溃"的恢复流程（两阶段提交风格）**：
1. 阶段一：DataMover 将数据写入目标层，完成后立即写入 `commit_marker`（包含 task_id、源/目标位置、校验和）到目标层的持久化日志区（L3/L4 有持久化能力；L1/L2 则写入 Versioned Metadata Store 的 WAL）。
2. 阶段二：DataMover 更新 Metadata 的位置索引，完成后将任务标记为 `COMMITTED`。
3. **崩溃恢复**：若 DataMover 在阶段一完成后、阶段二完成前崩溃，Watchdog 重启后扫描 `commit_marker` 日志。发现未完成的 marker 时，自动补发 Metadata 更新请求（幂等重试），最终将任务推进到 `COMMITTED`。

**状态转换条件与超时策略**：

| 转换边 | 触发条件 | 超时策略 |
|--------|----------|----------|
| PENDING → IN_PROGRESS | DataMover 工作线程从队列取出任务 | 排队超时 60s，超时后标记为 ABORTED |
| IN_PROGRESS → COMMITTED | 目标层写入完成 + commit_marker 持久化 + Metadata 更新成功 | 单阶段拷贝超时：L1↔L2 为 100ms；L2↔L3 为 2s；L3↔L4 为 30s |
| IN_PROGRESS → ABORTED | 校验和不匹配 / 目标层不可写 / 超时 | ABORTED 后自动重试 3 次；仍失败则上报告警并暂停同类任务 |

## 6. 与周边子系统的边界和交互

| 周边子系统 | 边界方向 | Storage 的职责 | 对方的职责 | 交互内容 | 一致性要求 |
|-----------|---------|---------------|-----------|---------|-----------|
| **Metadata** | Storage 接收提示，但最终校验 | 根据 Metadata 的位置提示发起实际查询；迁移完成后异步通知更新 | 维护前缀→位置的映射索引；处理跨节点一致性哈希 | 位置提示查询/更新确认 | 最终一致（允许 100ms 窗口） |
| **Connector** | Storage 被动接收请求 | 提供 allocate/load/store 接口；自治内部调度策略 | 发起 KV Cache 的分配和加载请求；执行模型计算 | 分配请求、数据缓冲区描述符 | 请求级同步 |
| **Transport** | Storage 调用 Transport 完成远程操作 | 定义远程数据搬运的语义（读/写/拷贝范围） | 封装 RDMA/TCP 连接管理、重试、心跳 | 远程 IO 任务、完成回调 | 传输层可靠交付 |

**接口抽象层次说明**：
- **Storage ↔ Metadata**：通过轻量级消息传递交互，不涉及数据流，只涉及位置元数据。
- **Storage ↔ Connector**：Storage 提供同步调用接口，Connector 在 prefill/decode 的关键路径上调用。Storage 必须保证这些接口的调用延迟可预期。
- **Storage ↔ Transport**：Storage 不感知底层网络协议细节，只传递「从节点 A 的某地址范围读取/写入到节点 B 的某地址范围」的语义化请求。Transport 子系统负责将语义请求映射为具体协议操作（RDMA READ/WRITE、TCP socket send/recv 等）。

### 6.1 Transport 边界语义防火墙

为了避免 Storage 与 Transport 之间的语义泄漏，双方在接口上必须遵守以下防火墙规则：

**Storage 不得要求 Transport 提供**：
- 连接状态变化（connected/disconnected/reconnecting）；
- 底层协议类型（RDMA/TCP）的显式标识；
- 重试次数或超时策略细节。

**Transport 不得要求 Storage 理解**：
- 物理网卡队列（QP/CQ）编号；
- RDMA 内存注册（MR）的生命周期；
- TCP socket fd 或内核缓冲区大小。

**错误码契约**：Transport 向 Storage 返回的错误码必须收敛到以下四类：
1. `OK`：操作成功；
2. `TIMEOUT`：远端在约定时间内未响应；
3. `UNREACHABLE`：无法路由到目标节点；
4. `CORRUPTED`：数据校验和不匹配。
所有协议特定的错误在 Transport 内部收敛到上述四类后方能向上传递。

### 6.2 L4 远程访问错误处理策略矩阵

当 Remote Data Mover 从 Transport 收到错误码时，Storage 侧的处理策略如下：

| 错误码 | Storage 侧处理策略 | 重试规则 | 回退层级 | 告警触发条件 |
|--------|-------------------|----------|----------|--------------|
| `OK` | 正常完成，继续后续流程 | 无需重试 | — | — |
| `TIMEOUT` | 认为目标节点响应慢或网络拥塞 | 立即重试 1 次（间隔 50ms）；若仍超时，降级为本地 MISS | L3（若本地有副本）或 MISS | 同一目标节点连续 3 次 TIMEOUT |
| `UNREACHABLE` | 认为目标节点已下线或网络分区 | 不重试；立即降级 | L3 或 MISS | 任意 UNREACHABLE 即触发节点不可达告警 |
| `CORRUPTED` | 认为远程数据在传输中被损坏或远程副本已不一致 | 不重试；若该 chunk 存在其他副本（R≥2），尝试下一个副本；否则降级为 MISS | 下一副本 / L3 / MISS | 任意 CORRUPTED 即触发数据完整性告警 |

**L4 访问时序图**：

```
正常读取路径：
Storage(L4Mgr) -> Remote Data Mover -> Transport(RDMA/TCP) -> Remote Node
                                              <- OK <-

TIMEOUT 回退路径：
Storage(L4Mgr) -> Remote Data Mover -> Transport
                                              <- TIMEOUT
Storage 重试 1 次 -> 仍 TIMEOUT
Storage 回退到 L3 查询（若本地无副本则 MISS）

CORRUPTED 回退路径：
Storage(L4Mgr) -> Remote Data Mover -> Transport -> Remote Node
                                              <- CORRUPTED
Storage 检查副本列表 -> 若存在 replica_2，切换到 replica_2 重试
                 -> 若所有副本均 CORRUPTED，降级为 MISS
```

**L4 数据副本策略说明**：
- L4 层的默认副本因子 R=2（每个 Object 在 2 个远程节点上保存副本）。
- 这意味着单个远程节点故障不会导致数据丢失，但会降低可用容量。
- **脆弱点标注**：若 R=1（副本因子为 1），则任意单节点故障都会导致该节点上所有 L4 Object 永久不可恢复。MVP 阶段虽然默认 R=2，但在资源极度受限的场景下可能被迫使用 R=1，此时需在运维文档中明确标注该风险。

## 7. 设计 Rationale

### 7.1 "为什么不是严格分层 LRU？"

严格分层 LRU 的核心问题不在于 LRU 本身，而在于「逐层相邻约束」强行制造了中间 staging 层：
- 当数据从 SSD 被命中时，若该数据接下来会被连续多次访问，它必须依次经过 DRAM、HBM，每一层都消耗拷贝带宽和插入延迟；
- 若该数据只是「一次性访问」（例如长上下文中仅使用一次的 prefix），它仍然会污染 DRAM 的 LRU，挤压真正的热数据空间。

跳层直达打破了相邻约束，让「热度」而不是「层级」决定数据的最终位置。这在 KV Cache 场景下尤为重要，因为 KV Cache 的访问模式存在显著的幂律分布（少数前缀被高频复用），热度信号的命中率远高于通用的缓存无关 workload。

### 7.2 "这个选择的最优场景和最差场景是什么？"

**最优场景**：
- 多轮对话、RAG、代码补优等具有明显前缀复用模式的 workload；
- 冷热分离显著（前 20% 的热数据贡献 > 80% 的访问）；
- 预取信号来源可靠（如引擎已知下一批请求的 token 范围）。

**最差场景**：
- 完全随机访问的 workload（热度信号无法建立）；
- 短文本单次请求（ KV Cache 几乎没有复用机会，预取永远 miss）；
- 极端 write-heavy 场景（热度统计更新频率跟不上访问变化，误判率上升）。

### 7.3 "如果热度分布不符合幂律，架构是否需要推倒重来？"

**不需要推倒重来，但需要进行有界退化**。跳层直达的核心假设是热度分布符合幂律分布（前 20% 热数据贡献 > 80% 访问）。若该假设被证伪，系统可按以下路径退化：

1. **第一阶段退化（热数据区分度低，但仍有时序局部性）**：
   - 将跳层直达的晋升/淘汰阈值放宽，降低为「大部分时间数据留在 L2」的温和分级策略；
   - 此时系统退化为「弱化版跳层直达」，性能可能接近严格分层 LRU，但实现上无需重构。

2. **第二阶段退化（完全随机访问，热度统计失效）**：
   - 通过配置开关（A/B 机制）在 30 秒内切换为严格分层 LRU 模式；
   - 关闭 HotnessTracker 的跳层决策，仅保留其作为统计报表使用；
   - SpaceManager 和 DataMover 的调度逻辑不受影响；
   - **退化实现成本**：约 2 人周（主要是切换逻辑和配置中心的改动），不需要修改 TierManager 或数据格式。

因此，**最坏情况下系统的退化成本是 2 人周、30 秒切换时间**，这是一个 bounded and reversible 的退化路径，不构成架构级推倒重来的风险。

### 7.4 "如果未来需求变更，这个设计的脆弱点在哪里？"

脆弱点集中在三个位置，且存在明确的级联失效风险：

1. **HotnessTracker 的可扩展性**：当前 Count-Min Sketch 是单机近似结构。若未来节点数扩展到 100+，单机热度统计将变为分布式共识问题，可能需要演进为分层热度聚合或一致性哈希分片；
2. **预取引擎与引擎的耦合**：当前 PrefetchEngine 可选接收 Connector 的 `engine_hint`。若未来需要更激进的预取（例如基于 Speculative Prefetch），预取引擎将需要深入耦合到推理引擎的调度循环，这超出了当前 Storage 子系统的边界；
3. **L3 SSD 的容量瓶颈**：当前 L3 是本地 SSD，单机容量上限约 60TB。若 workload 规模增长超出单机容量，L4 远程层将从「容量扩展层」变成「主要存储层」，此时延迟模型将发生质变（从本地 ~ms 级别变为远程 ~50-200 μs 级别），Storage 的调度权重和缓冲策略都需要重新设计。

#### 级联失效图（Cascading Failure Graph）

```
                    ┌─────────────────────┐
                    │ L3 SSD 容量耗尽 /   │
                    │   GC 抖动严重       │

                               │

**早期熔断指标**：为了在上述级联失效进入不可逆阶段前触发干预，我们定义以下熔断指标：
- **指标 1**：L3 P99.9 命中延迟连续 5 分钟 > 50ms 增量 → 触发 L3→L4 主动卸载（将 L3 中最冷的 10% 数据批量搬到 L4）；
- **指标 2**：PrefetchEngine 预取命中率连续 15 分钟 < 50% → 自动关闭 PrefetchEngine 48 小时，回退到纯跳层直达；
- **指标 3**：HBM 有效热数据占比（被访问过的数据 / 总占用数据）连续 10 分钟 < 60% → 触发一次性 HBM 大清洗（将所有非 MIGRATING 数据降级到 L3，由下一次访问重新自然升温）。

## 8. 风险点和缓解策略

### 高风险：SSD 竞争下的尾延迟不可控（高影响 / 中可能性）

**背景**：反标杆研究（SolidAttention）表明，NVMe SSD 在后台竞争场景下 P99.9 延迟可暴涨 2.9 倍。对于以 TTFT 为核心 SLO 的系统，这是不可接受的。

**表现形态**：
- 多实例共享节点时，其他进程的磁盘 IO 导致 Storage 的 L3 命中延迟大幅抖动；
- 批量淘汰或预取任务触发大量随机写，SSD 固件内部的 GC 和 wear-leveling 引入不可预期的停顿。

**缓解策略**：
1. **DRAM Write Buffer**：在 L2/L3 边界引入 32KB/layer 的 write buffer（与 SolidAttention 实践对齐），将细粒度随机写合并为顺序写，降低 SSD 写放大和控制器压力。
2. **IO 隔离与限速**：DataMover 对 L3 的并发读写设置硬上限（最大并发 = 4），同步请求始终优先于后台淘汰/预取任务。
3. **快速降级路径**：当 L3 查询延迟超过阈值（20ms）时，将该请求标记为「慢路径」，后续同 Segment 的访问可触发预取或跳层晋升，避免重复命中抖动区域。
4. **SSD GC 预判机制（增强）**：通过监控 SSD S.M.A.R.T. 的 `Wear_Leveling_Count` 和 `Media_Wearout_Indicator`，在 GC 窗口概率升高时（指标连续 3 次采样上升），提前将对应 L3 分段的热数据搬到 L2，规避 GC 停顿。
5. **L3→L4 主动卸载兜底**：若上述策略无法将 P99.9 增量控制在 50ms 以内，且连续 5 分钟超标，则触发主动卸载——将 L3 中最冷的 10% 数据批量迁移到 L4，释放 SSD 空间并减少竞争。

### 中风险：跳层直达误判导致性能退化（中影响 / 中可能性）

**缓解策略**：
- 热度阈值采用保守 _INITIAL 值（比理论最优值更严格）；
- 建立 A/B 回退机制：当系统检测到跳层后的命中率或延迟劣化，可在 30 秒内切换回严格分层 LRU 模式。

### 中风险：预取风暴导致带宽挤占（中影响 / 中可能性）

**缓解策略**：
- PrefetchEngine 的带宽配额硬限制为总带宽的 30%，不可被配置覆盖；
- 预取任务设置 5 秒超时，超时后自动取消并标记为「低优先级淘汰」。

## 9. 待验证假设与 Fallback 策略

| 假设 | 验证方法 | 通过标准 | 失败标准 | 触发机制 | 决策责任人 | 降级时间预算 | 失败回退策略 |
|------|---------|---------|---------|----------|-----------|-------------|-------------|
| 热度分布符合幂律分布 | 生产 workload 分析 | 前 20% 热数据贡献 ≥ 80% 访问 | 前 20% 贡献 < 60% | 自动（每日统计）+ 每周评审 | product-owner | 30 秒内切换至严格分层 LRU | 降级为严格分层 LRU；切换成本约 2 人周 |

**Fallback 策略执行纪律**：
1. **自动触发项**（热度分布、预取干扰、L3 基线）：当失败标准连续 3 次采样均满足时，由监控系统自动执行降级；
2. **人工审批项**（介质性能、跨层合并）：当失败标准首次满足时，发送告警给责任人，责任人需在降级时间预算内给出处理决定；
3. **所有降级动作必须记录到 `docs/adr.md` 的假设验证跟踪表中**，累计 3 个假设失败时强制触发 `/ky:arch` 产品级架构评审。
