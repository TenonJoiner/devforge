# 系统架构总纲

## 设计目标

### 核心痛点

现有分布式对象存储（Ceph、MinIO 等）在 AI 推理场景下**P99 延迟过高、吞吐量不足**，无法满足以下核心需求：

1. **KV Cache Offloading**：HBM 成本极高，需要将推理过程中的 KV Cache 实时卸载到外置存储，释放显存以支持更大 batch size 或更长上下文窗口
2. **KV Cache 跨请求复用**：相同 prompt prefix 的 KV Cache 可在不同请求间复用，避免重复计算，显著降低首 token 延迟（TTFT）

如果没有本系统，推理引擎只能：
- 将 KV Cache 保留在 GPU HBM 中 → 显存成为瓶颈，batch size 受限，长上下文无法支持
- 或使用现有对象存储 Offload → 延迟不可接受（P99 通常在 10ms+），吞吐量不足

### 子系统职责边界与 Phase 范围

**S1 存储引擎**
- Phase 1：SPDK Blobstore + KVCache 对象层；DRAM 缓存 + NVMe SSD 持久化；3 副本异步复制；LRU 本地淘汰
- Phase 2+：自研日志结构引擎替换 Blobstore；热 3 副本 + 冷纠删码；Connector API hints 指导分层
- 合并 rationale：① 架构层面——KV Cache WORM 场景下，LRU 淘汰既是分层决策（ADR-008）也是存储空间管理（ADR-002），两者职责边界天然模糊；② 实现层面——共享 SPDK reactor 资源和 NVMe 设备句柄，拆分会引入不必要的跨线程通信；③ 演进保障——S1 内部定义三个模块边界（存储模块 / 分层模块 / 冗余模块），Phase 2 存储引擎替换时分层模块接口契约不变

**S2 元数据服务**
- Phase 1：内存哈希表（uthash）+ 异步 WAL；Prefix Hash Chain 索引；分区 Raft（3 副本）；Block Table 分片
- Phase 2+：跳表二级索引优化范围查询；LMDB 降级路径；元数据规模超阈值时自动分片扩展
- 独立性：可独立开发测试，仅依赖传输层提供的 RPC 通道

**S3 传输层**
- Phase 1：RDMA 双边操作（服务端主导）；SPDK Reactor 原生 CQ 处理；TCP 控制平面
- Phase 2+：独立 RDMA 线程（忙轮询 CQ）；TCP 数据平面降级路径；WR 批处理动态调优
- 独立性：纯基础设施层，可独立开发和压测，不依赖业务逻辑

**S4 调度器**
- Phase 1：轻量级启发式路由（本地优先 + 一致性哈希）；节点状态心跳；基础 PD 路由
- Phase 2+：PD 语义增强调度（Prefix Cache 全局优化）；负载感知放置；多租户 QoS
- 独立性：无状态决策服务，可水平扩展，仅依赖元数据服务提供的 Block Table 查询

**S5 连接器**
- Phase 1：vLLM Connector SDK（C 库 + Python binding）；基础 put/get/delete API
- Phase 2+：SGLang 适配；Connector API hints（pin/unpin/valloc）；批量预取接口
- 独立性：客户端库，可独立开发测试，通过传输层与存储节点通信

## 全局设计约束

> **轮次归属**：第 0 步（初稿）→ 第 3 轮 Step 2（定稿）
> **当前状态**：✅ 终稿

### 一致性模型

基于 ADR-004（分层一致性）决策：

- **读写一致性模型**：分层一致性
  - 元数据层：强一致性（Linearizability），分区 Raft，写等 majority 确认
  - 数据层：最终一致性，本地 NVMe 写入即确认，异步 3 副本复制
- **会话语义**：read-your-writes（元数据层 Raft 保证）+ monotonic-reads（本地缓存保证）
- **故障场景一致性保证**：
  - 网络分区时选择 CP（元数据层）：少数派分区拒绝写入，多数派分区继续服务
  - 数据层分区时：本地写入继续，副本同步延迟，分区恢复后自动追赶
  - 脑裂恢复：Raft 日志回放 + 数据层校验和比对 + 定期 scrub
- **跨副本同步方式**：
  - 元数据：同步复制（Raft majority，N=3, W=2, R=2）
  - 数据：异步复制（本地确认即返回，后台 RDMA 传输到 2 个副本节点）

### 故障假设

| 故障类型 | 假设 | 系统级应对策略 |
|----------|------|---------------|
| Crash-stop | 任意进程随时崩溃，重启后内存状态丢失 | 元数据：Raft 日志回放恢复；本地状态：WAL + NVMe SSD 扫描重建（ADR-009）；数据：3 副本容忍单节点丢失 |
| Network partition | 节点间网络随时可能分区，无法区分崩溃与网络中断 | 心跳超时检测（默认 3s）；元数据层 Raft 租约；数据层异步复制容忍短暂分区；调度器将不可达节点标记为不可用 |
| Disk corruption | NVMe SSD 可能产生位翻转或扇区损坏 | 端到端校验和（CRC32C）；每次读取验证校验和；定期 scrub 扫描（每 24h）；损坏数据从副本恢复 |
| Silent data corruption | RDMA 传输或 NVMe 写入可能静默损坏 | 网络传输携带校验和；落盘前后校验和验证；RDMA 启动时 selftest；定期 ping 检测静默降质（ADR-007） |
| Byzantine fault | 不假设存在 | 本系统面向可信内部集群，不考虑拜占庭故障 |
| RDMA 静默降质 | RoCE 配置错误导致 RDMA 性能降级但不报错 | 启动时 RDMA selftest + 定期 ping 验证；检测到降质触发告警并强制降级到 TCP（ADR-007） |
| 元数据服务暂时不可用 | Raft leader 选举（1-5s）、网络分区导致无 majority、滚动升级期间 | 写入路径：I4 超时后返回错误，S1 本地数据标记为 UNCOMMITTED（孤儿数据），推理引擎收到错误后可重试或降级；读取路径：I5 优先命中本地内存缓存（缓存命中率 > 99%，ADR-004 H1），缓存未命中时返回 UNAVAILABLE；Prefix 匹配：完全不可用，推理引擎降级为全量 Prefill 计算。推理引擎需容忍 1-5s 写入中断（Raft 选举窗口） |

### 数据流 1：KV Cache 写入（Prefill → 存储）

> 回应 Reviewer 质疑 #1：明确区分控制面（RPC）和数据面（RDMA），补充 ack 回调路径。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  推理引擎     │     │  连接器(S5)   │     │  调度器(S4)   │
│  (Prefill)   │     │  Connector   │     │  Scheduler   │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                     │                     │
       │ ① put(key, block)   │                     │
       │────────────────────>│                     │
       │                     │ ② I3: route_request  │
       │                     │   (key, PREFILL_WRITE)│
       │                     │────────────────────>│
       │                     │   ③ {target_node}   │
       │                     │<────────────────────│
       │                     │                     │
       │                     │ ④ I1: kv_put(key, block, hints)  [控制面 RPC]
       │                     │──────────────────────────────────>┌──────────────┐
       │                     │                                   │  存储引擎(S1) │
       │                     │                                   │  (目标节点)   │
       │                     │                                   └──────┬───────┘
       │                     │                                          │
       │                     │              ⑤ S1 分配空间 + I7: RDMA 拉取 block [数据面]
       │                     │              (S1 作为 RDMA 服务端，主导传输)
       │                     │                                          │
       │                     │              ⑥ NVMe 写入 [持久化边界]     │
       │                     │              ════════════════════════════ │
       │                     │                                          │
       │                     │              ⑦ I4: update_block_table    │
       │                     │                (key, location, COMMITTED) │
       │                     │                     ┌──────────────┐     │
       │                     │                     │  元数据(S2)   │<────│
       │                     │                     │  Metadata    │     │
       │                     │                     └──────┬───────┘     │
       │                     │              ⑧ Raft majority 确认        │
       │                     │              [一致性边界]                 │
       │                     │                     │────────────────────>│
       │                     │                                          │
       │                     │  ⑨ I1 回调: ack(success)                 │
       │                     │<─────────────────────────────────────────│
       │  ⑩ put 返回成功      │                                          │
       │<────────────────────│                                          │
       │                     │              ⑪ I9: replicate_async [后台] │
       │                     │              (异步 RDMA 传输到 2 副本节点)  │
```

**控制面 vs 数据面分离**：
- 步骤 ④ 是控制面 RPC（I1 接口），携带 key、hints 等元信息
- 步骤 ⑤ 是数据面 RDMA 传输（I7 接口），由存储引擎(S1)作为 RDMA 服务端主导拉取数据
- 步骤 ⑨ 是 I1 的异步回调，S1 在完成 NVMe 写入(⑥) + 元数据 COMMITTED(⑧) 后触发

**关键边界**：
- 持久化边界：⑥ NVMe 写入完成（本地持久化）
- 一致性边界：⑧ Raft majority 确认（元数据强一致）
- 返回时机：⑨ S1 回调连接器，连接器返回推理引擎，不等副本传输(⑪)

### 数据流 2：KV Cache 读取（Decode ← 存储）

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  推理引擎     │     │  连接器(S5)   │     │  调度器(S4)   │     │  元数据(S2)   │
│  (Decode)    │     │  Connector   │     │  Scheduler   │     │  Metadata    │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │ ① get(key)         │                     │                     │
       │────────────────────>│                     │                     │
       │                     │ ② lookup_block()    │                     │
       │                     │─────────────────────────────────────────>│
       │                     │   ③ {node, offset}  │                     │
       │                     │<─────────────────────────────────────────│
       │                     │                     │                     │
       │                     │ ④ RDMA read(node, offset) ──────────────────────>┌──────────────┐
       │                     │                     │                     │       │  存储引擎(S1) │
       │                     │                     │                     │       │  (数据节点)   │
       │                     │                     │                     │       └──────┬───────┘
       │                     │                     │                     │              │
       │                     │                     │                     │  ⑤ DRAM/SSD  │
       │                     │                     │                     │  读取数据     │
       │                     │  ⑥ RDMA 返回 kv_block │                    │              │
       │                     │<────────────────────────────────────────────────────────│
       │  ⑦ kv_block         │                     │                     │              │
       │<────────────────────│                     │                     │              │
```

**关键边界**：
- 读取路径不经过调度器（直接查元数据获取位置，RDMA 直连数据节点）
- DRAM 命中：P50 < 500μs；SSD 读取：P50 < 2ms（估算，待原型验证）

### 数据流 3：Prefix Cache 命中

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  推理引擎     │     │  连接器(S5)   │     │  元数据(S2)   │     │  调度器(S4)   │
│  (新请求)    │     │  Connector   │     │  Metadata    │     │  Scheduler   │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │ ① prefix_match     │                     │                     │
       │   (prompt_tokens)   │                     │                     │
       │────────────────────>│                     │                     │
       │                     │ ② batch_prefix_     │                     │
       │                     │   lookup(hashes)    │                     │
       │                     │────────────────────>│                     │
       │                     │                     │ ③ Prefix Hash Chain │
       │                     │                     │ 遍历匹配             │
       │                     │   ④ matched_chunks  │                     │
       │                     │   + locations       │                     │
       │                     │<────────────────────│                     │
       │                     │                     │                     │
       │  ⑤ {match_len,      │                     │                     │
       │     chunk_locations} │                     │                     │
       │<────────────────────│                     │                     │
       │                     │                     │                     │
       │ ⑥ 仅计算未命中部分   │                     │                     │
       │   的 KV Cache        │                     │                     │
       │ ⑦ 从 chunk_locations │                     │                     │
       │   RDMA 读取已命中块   │                     │                     │
```

**关键边界**：
- Prefix 匹配在元数据服务内完成（内存哈希表 O(1) + 链式遍历）
- 并行批量查找：将 O(N × RTT) 降至 O(1 × RTT + N 本地比较)
- 命中后直接 RDMA 读取数据，不经过调度器

## 关键接口索引

> **轮次归属**：第 3 轮 Step 2（定稿）
> **当前状态**：✅ 终稿

| # | 接口名 | 调用方 | 被调用方 | 同步/异步 | 关键契约 |
|---|--------|--------|----------|-----------|----------|
| I1 | `kv_put(key, block, hints)` | 连接器(S5) | 存储引擎(S1) | 异步（RDMA 写入后回调） | 前置：key 非空，block ≤ 100MB；返回：写入确认（本地持久化 + 元数据 COMMITTED）；超时：5s；幂等：同 key 重复写入覆盖 |
| I2 | `kv_get(key)` | 连接器(S5) | 存储引擎(S1) | 异步（RDMA 读取后回调） | 前置：key 已 COMMITTED；返回：KV 块数据 + 校验和；超时：10s；未命中返回 NOT_FOUND |
| I3 | `route_request(key, op_type)` | 连接器(S5) | 调度器(S4) | 同步（< 2.5ms） | 前置：无；返回：目标节点 ID + 端点地址；op_type 区分 prefill_write / decode_read；无状态，可重试 |
| I4 | `update_block_table(key, location, status)` | 存储引擎(S1) | 元数据(S2) | 同步（Raft majority） | 前置：本地写入已完成；返回：COMMITTED 确认；超时：1s；失败时存储引擎重试 3 次后报错 |
| I5 | `lookup_block(key)` | 连接器(S5) | 元数据(S2) | 同步（< 1ms） | 前置：无；返回：{node_id, offset, status}；未命中返回 NOT_FOUND；读取内存哈希表，无 I/O |
| I6 | `batch_prefix_lookup(hash_chain)` | 连接器(S5) | 元数据(S2) | 同步（< 5ms） | 前置：hash_chain 为 token 序列的 Prefix Hash Chain；返回：匹配的 chunk 列表 + 位置；批量查找��单次 RTT |
| I7 | `rdma_transfer(src, dst, size)` | 存储引擎(S1) / 连接器(S5) | 传输层(S3) | 异步（RDMA 完成后回调） | 前置：src/dst 已注册为 RDMA MR；返回：传输完成 + 校验和验证；超时：100ms（降级触发）；双边操作，服务端主导 |
| I8 | `heartbeat(node_status)` | 存储引擎(S1) | 调度器(S4) | 异步（周期性，1s 间隔） | 上报：节点 ID、DRAM/SSD 使用率、活跃连接数、健康状态；3 次心跳超时标记节点不可用 |
| I9 | `replicate_async(key, block, targets)` | 存储引擎(S1) | 传输层(S3) | 异步（后台） | 前置：本地写入已完成；目标：2 个副本节点；失败重试 3 次后标记副本降级；不阻塞前台写入 |
| I10 | `query_data_distribution(key_range)` | 调度器(S4) | 元数据(S2) | 同步（< 5ms） | 前置：无；返回：key_range 内的数据分布（节点→块数映射）；Phase 1 复用 I5 逐 key 查询，Phase 2 批量接口；用于负载感知放置决策 |
| I11 | `raft_send(msg, target_node)` | 元数据(S2) | 传输层(S3) | 异步（RDMA/TCP） | Raft 协议消息传输（AppendEntries/RequestVote/Heartbeat）；元数据服务通过传输层进行节点间 Raft 通信；超时：500ms |
| I12 | `replica_receive(key, block)` | 传输层(S3) | 存储引擎(S1) | 异步（RDMA 接收后回调） | 远端副本接收接口；S1 收到副本数据后执行本地 NVMe 写入 + 本地元数据更新（不走 Raft，仅本地 WAL）；失败返回 NACK 触发源节点重试 |

## 技术演进路线图

> **轮次归属**：第 3 轮 Step 2（ADR 收敛后定稿）
> **当前状态**：✅ 终稿

### Phase 1：MVP（0-3 月）

核心目标：端到端 KV Cache 写入/读取/Prefix 命中，单集群 4 节点可运行。

| 子系统 | Phase 1 交付 |
|--------|-------------|
| S1 存储引擎 | SPDK Blobstore + KVCache 对象层；DRAM 缓存 + NVMe SSD 持久化；3 副本异步复制；LRU 本地淘汰 |
| S2 元数据 | 内存哈希表（uthash）+ 异步 WAL；Prefix Hash Chain；分区 Raft（3 副本）；Block Table 基础分片 |
| S3 传输层 | RDMA 双边操作（SPDK Reactor 原生 CQ）；TCP 控制平面；降级到错误日志（非 TCP 数据面） |
| S4 调度器 | 轻量级启发式路由（本地优先 + 一致性哈希）；节点心跳；基础 PD 路由 |
| S5 连接器 | vLLM Connector SDK（C 库 + Python binding）；put/get/delete/prefix_match API |

已知技术债务：
- Blobstore GC 策略不透明，可能在高写入负载下产生延迟抖动
- TCP 降级路径未实现（仅记录错误日志）
- 无多租户隔离

### Phase 2：增强（3-6 月）

核心目标：性能优化 + PD 调度增强 + 生产可用性。

| 子系统 | Phase 2 交付 |
|--------|-------------|
| S1 存储引擎 | 自研日志结构引擎（验证后替换 Blobstore）；热 3 副本 + 冷纠删码（节点 ≥ 6 且利用率 > 60%） |
| S2 元数据 | 跳表二级索引；LMDB 降级路径；元数据分片自动扩展 |
| S3 传输层 | 独立 RDMA 线程（忙轮询 CQ）；TCP 数据面降级路径；WR 批处理动态调优 |
| S4 调度器 | PD 语义增强调度（Prefix Cache 全局优化）；负载感知放置 |
| S5 连接器 | SGLang 适配；Connector API hints（pin/unpin/valloc）；批量预取 |

触发条件（Phase 1 → Phase 2）：
- S1 存储引擎：写入 P99 > 2ms 或 GC 碎片率 > 30% 持续 5 分钟
- S2 元数据：单节点元数据内存占用 > 5GB（ADR-009 H1 阈值）触发 LMDB 降级评估；Prefix 查询 P99 > 5ms 触发跳表索引优化
- S3 传输层：RDMA 单次 KV 操作端到端延迟 > 1ms 持续 5 分钟
- S4 调度器：Prefix Cache 全局命中率 < 60%（表明本地优先路由不足以利用跨节点缓存）触发 PD 语义增强
- S5 连接器：vLLM 集成稳定运行 2 周且通过集成测试后，启动 SGLang 适配；Connector API hints 在 S1 分层模块就绪后启动
- 冗余策略：节点数 ≥ 6 且存储利用率 > 60%

触发条件（Phase 2 → Phase 3）：
- 功能里程碑：所有子系统 Phase 2 特性交付且通过集成测试
- 性能里程碑：4~16 节点规模下所有性能基线指标达标
- 稳定性里程碑：连续运行 7 天无 P0 故障

### Phase 3：成熟（6-12 月）

核心目标：企业级特性 + 大规模验证。

- 多租户隔离（命名空间 + QoS 资源配额）
- 静态数据加密（AES-256-GCM）
- K8s Operator 部署
- 100+ 节点规模验证
- 跨版本滚动升级和灰度发布
- 完整的审计日志和合规认证
