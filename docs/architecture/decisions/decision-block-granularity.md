## 核心技术深度剖析

### 方案1：固定小 Block（16/32 tokens）— vLLM PagedAttention 模式

#### 1. 问题背景与必要性

**解决什么问题**：
- 传统 LLM 推理系统为每个请求预分配连续内存，导致 60%-80% 的 KV Cache 内存浪费（vLLM 论文实测数据）
- 请求长度差异大，静态分配造成严重的内部碎片和外部碎片
- 无法支持动态增长的序列（如多轮对话）

**不解决的后果**：
- GPU 内存利用率仅 20-38%
- batch size 受限，吞吐量下降 2-4 倍
- 长序列请求容易 OOM

**替代方案对比**：
| 方案 | 内存利用率 | 复杂度 | 适用场景 |
|------|-----------|--------|----------|
| 静态连续分配 | 20-38% | 低 | 固定长度推理 |
| 动态重分配 | 50-60% | 中 | 中等长度变化 |
| **PagedAttention** | **>96%** | 高 | 通用场景 |

#### 2. 核心原理

**算法思想**：
借鉴操作系统虚拟内存的页表机制，将 KV Cache 划分为固定大小的 Block，通过 Block Table 实现逻辑地址到物理地址的映射。

**关键数据结构**：
```
Block Table (per sequence):
  logical_block_0 -> physical_block_127
  logical_block_1 -> physical_block_45
  logical_block_2 -> physical_block_89
  ...

Physical Block:
  - size: block_size tokens (default 16)
  - data: [num_layers, 2, num_heads, head_dim]
```

**理论复杂度分析**：
| 操作 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| 分配 Block | O(1) | O(num_blocks) |
| 释放 Block | O(1) | O(1) |
| 前缀匹配 | O(num_blocks) | O(num_blocks) |
| 内存碎片 | 外部碎片 ≈ 0 | 内部碎片 ≤ B-1 |

**内部碎片计算公式**：
```
最大内部碎片 = B - 1 = 15 tokens (B=16 时)
平均内部碎片 ≈ B/2 = 8 tokens
内部碎片率 < 4% (当序列长度 >> B 时)
```

#### 3. 设计机制

**系统架构**：
```
┌─────────────────────────────────────────┐
│           Scheduler/Worker              │
│  ┌─────────────┐    ┌──────────────┐   │
│  │ Block Table │───▶│ Block Allocator│  │
│  │  (per seq)  │    │  (global)      │  │
│  └─────────────┘    └──────────────┘   │
│         │                    │          │
│         ▼                    ▼          │
│  ┌──────────────────────────────────┐   │
│  │      PagedAttention Kernel       │   │
│  │  (block_sparse_attention)        │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**关键设计机制**：
1. **Block Table 间接寻址**：逻辑 Block 号通过查表映射到物理 Block 地址
2. **Copy-on-Write**：共享 Block 在写入时复制，支持 Beam Search 等场景
3. **前缀缓存（APC）**：基于 Block 级 Hash 链实现前缀匹配

**数据流**：
```
Request → Tokenize → Block Table 构建 → GPU Kernel 执行
              ↓
         Block Hash 计算 → Prefix Cache 查询 → 命中则复用
```

#### 4. 架构决策分析

**关键决策点1：Block Size = 16 tokens 的选择依据**

| 可选方案 | 内部碎片 | 元数据开销 | GPU 并行度 | 选择理由 |
|----------|----------|-----------|-----------|----------|
| 8 tokens | 更低 | 2× | 较低 | 碎片收益有限 |
| **16 tokens** | **<4%** | **基准** | **良好** | **经验最优** |
| 32 tokens | 8% | 0.5× | 更好 | 碎片率上升 |
| 64 tokens | 16% | 0.25× | 最佳 | 碎片不可接受 |

**决策依据**：
- vLLM 作者测试多种配置，16 tokens 是内存效率与 GPU 利用率的平衡点
- 13B 模型下，16 tokens Block 大小约 12.8KB，适合 GPU cache line
- 更大的 Block 会增加内部碎片，更小的 Block 增加元数据开销

**关键决策点2：Block-level Hash vs Token-level 匹配**

| 方案 | 匹配精度 | 实现复杂度 | 内存开销 | 选择 |
|------|----------|-----------|----------|------|
| Token-level | 100% | 高（Radix Tree） | 高 | SGLang |
| **Block-level** | ~95% | **低（Hash Table）** | **低** | **vLLM** |

**Block-level Hash 链原理**：
```
Block_Hash[i] = Hash(Block_Hash[i-1], Token_Hash[i])
                = Hash(Prefix, Current_Block_Tokens)
```
- 每个 Block 的 Hash 依赖其父 Block 的 Hash，形成链式结构
- 前缀匹配时只需比较 Block Hash，O(1) 查询
- 缺点：匹配必须对齐 Block 边界，可能产生"假阴性"（实际前缀相同但 Block 边界不对齐）

#### 5. 性能剖析

**瓶颈点识别**：
1. **Block Table 查询开销**：每个 token 需要查询 Block Table 获取物理地址
2. **Kernel 启动开销**：小 Block 导致更多的 kernel 调用次数
3. **Prefix Cache 假阴性**：Block 边界不对齐导致命中率下降

**关键性能路径**：
```
Prefill: 计算 KV → 写入 Block → 计算 Hash → 更新 Block Table
Decode:  查询 Block Table → 加载 KV → Attention 计算 → 写入新 Block
```

**量化数据**：
| 指标 | 数值 | 来源 |
|------|------|------|
| 内存利用率 | >96% | vLLM 论文 |
| 内部碎片率 | <4% | vLLM 论文 |
| 外部碎片率 | ≈0% | 设计特性 |
| Kernel 延迟开销 | +20-26% | 单请求 |
| 端到端吞吐提升 | 2-4× | batch 提升 |

#### 6. Trade-off 与限制

| 维度 | 选择A | 选择B | 取舍分析 | 本方案选择 |
|------|-------|-------|----------|-----------|
| **时间 vs 空间** | 小 Block（低碎片） | 大 Block（高并行） | 16 tokens 平衡 | 16 tokens |
| **一致性 vs 性能** | Token-level（精确） | Block-level（近似） | 接受 5% 假阴性 | Block-level |
| **复杂度 vs 收益** | 简单 Hash | 复杂 Radix Tree | 收益/复杂度比低 | Hash |
| **通用性 vs 专用性** | 通用 Block 大小 | 模型专用 | 通用性优先 | 16 tokens |

**适用场景边界**：
- **最佳场景**：中等长度序列（1K-32K tokens），多请求共享前缀
- **次优场景**：超长序列（>100K），Block Table 本身成为瓶颈
- **不适用**：单请求、无共享前缀、序列长度极短（<B）

**扩展性天花板**：
- Block Table 大小与序列长度成正比，超长序列需要分层管理
- Prefix Cache 的 Hash 表大小受限于 CPU 内存

### 方案2：固定大 Chunk（256/512 tokens）— LMCache 模式

#### 1. 问题背景与必要性

**解决什么问题**：
- 小 Block（16 tokens）在 Offloading 场景下传输效率极低
- PCIe/NVLink 带宽利用率不足 22%（小传输无法摊平 DMA 启动开销）
- 频繁的细粒度传输导致 CPU 开销剧增

**不解决的后果**：
- Offloading 带宽利用率 <5%（Grace-Hopper 平台实测）
- CPU-GPU 同步开销成为瓶颈
- 无法充分利用高带宽网络（RDMA 400Gbps）

**替代方案对比**：
| 方案 | 带宽利用率 | 延迟 | 适用场景 |
|------|-----------|------|----------|
| 16 tokens/block | ~5-22% | 高 | GPU 内管理 |
| **256 tokens/chunk** | **~60-80%** | **中** | **Offloading** |
| 1024 tokens/chunk | ~85% | 低 | 冷存储 |

#### 2. 核心原理

**算法思想**：
将多个小 Block 聚合成大 Chunk 进行批量传输，通过 Streaming GPU Buffer 实现聚合-传输-拆分流程。

**关键数据结构**：
```
Streaming Buffer (GPU):
  - size: chunk_size tokens × num_layers × 2 × num_heads × head_dim
  - 作用: 聚合分散的 Block，形成连续内存

Chunk Metadata:
  - token_range: [start, end)
  - layer_bitmap: 哪些 layer 的数据包含在此 chunk 中
  - location: {GPU, CPU, SSD, Remote}
```

**理论复杂度分析**：
| 操作 | 小 Block (16) | 大 Chunk (256) | 提升 |
|------|--------------|----------------|------|
| 传输次数 | 16× | 1× | 16× |
| DMA 启动开销 | 16× | 1× | 16× |
| 元数据查询 | 16× | 1× | 16× |
| 有效带宽利用率 | ~20% | ~70% | 3.5× |

**传输效率公式**：
```
Effective_Bandwidth = Theoretical_Bandwidth × (Transfer_Size / (Transfer_Size + Overhead))

小 Block:  Effective = 32GB/s × (12KB / (12KB + 50μs×32GB/s)) ≈ 7%
大 Chunk:  Effective = 32GB/s × (200KB / (200KB + 50μs×32GB/s)) ≈ 55%
```

#### 3. 设计机制

**系统架构**：
```
┌─────────────────────────────────────────────────────────┐
│                      LMCache Layer                      │
│  ┌─────────────┐    ┌──────────────┐    ┌───────────┐  │
│  │ vLLM Engine │───▶│ GPU Buffer   │───▶│ Chunker   │  │
│  │ (16t/block) │    │ (aggregate)  │    │ (256t/chk)│  │
│  └─────────────┘    └──────────────┘    └─────┬─────┘  │
│                                                │        │
│  ┌─────────────────────────────────────────────┘        │
│  │                                                      │
│  ▼                                                      │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐    │
│  │  GPU HBM   │    │  CPU DRAM  │    │  SSD/Remote│    │
│  │  (Hot)     │◄──►│  (Warm)    │◄──►│  (Cold)    │    │
│  └────────────┘    └────────────┘    └────────────┘    │
└─────────────────────────────────────────────────────────┘
```

**关键设计机制**：
1. **Streaming Buffer 聚合**：CUDA kernel 将分散的 Block 复制到连续 Buffer
2. **Delayed Write**：缓存新 token 的 KV，满 Chunk 后批量写入
3. **Layer-wise Pipelining**：传输与计算重叠，隐藏延迟

**数据流**：
```
Store:  vLLM Blocks → GPU Buffer (aggregate) → DMA → CPU/SSD
Load:   CPU/SSD → DMA → GPU Buffer → Scatter → vLLM Blocks
```

#### 4. 架构决策分析

**关键决策点：Chunk Size = 256 tokens 的选择依据**

| Chunk Size | 带宽利用率 | 延迟 | 内存碎片 | 选择理由 |
|------------|-----------|------|----------|----------|
| 64 tokens | ~35% | 较高 | 低 | 利用率不足 |
| **256 tokens** | **~70%** | **中等** | **中** | **经验最优** |
| 512 tokens | ~80% | 较低 | 高 | 碎片增加 |
| 1024 tokens | ~85% | 低 | 很高 | 延迟敏感场景不适用 |

**决策依据**：
- LMCache 论文：256 tokens 在 32-128 Gbps 网络下达到延迟-带宽平衡
- 小于 256 时带宽利用率不足，大于 256 时延迟增加明显
- 与 vLLM 16 tokens 对齐：256 = 16 × 16，便于整数倍映射

**关键决策点2：Chunk 与 Block 的双层管理**

| 层级 | 粒度 | 作用 | 管理组件 |
|------|------|------|----------|
| Block (vLLM) | 16 tokens | GPU 内计算 | vLLM BlockManager |
| Chunk (LMCache) | 256 tokens | Offloading 传输 | LMCache Controller |

**Trade-off**：
- 保留 Block 层：兼容 vLLM，支持细粒度 GPU 内存管理
- 增加 Chunk 层：优化传输效率，但增加一次内存拷贝

#### 5. 性能剖析

**瓶颈点识别**：
1. **Scatter/Gather 开销**：Chunk 与 Block 之间的转换需要 CUDA kernel
2. **延迟写回**：Delayed Write 策略在故障时可能丢失数据
3. **Chunk 粒度与 Prefix 匹配的冲突**：粗粒度降低前缀匹配精度

**关键性能路径**：
```
TTFT (Time To First Token):
  无缓存: Prefill 计算
  有缓存: Chunk 加载 → Scatter → Decode 启动

延迟对比 (256K context, 64Gbps):
  无缓存 Prefill: ~10s
  LMCache 加载: ~2s (5× 提升)
```

**量化数据**：
| 指标 | 数值 | 来源 |
|------|------|------|
| TTFT 降低 | 3-10× | LMCache 论文 |
| 带宽利用率 | 60-80% | 实测 |
| 延迟写回批处理 | 16× 减少写入次数 | 设计特性 |
| 多轮对话加速 | 3-10× | 实测 |

#### 6. Trade-off 与限制

| 维度 | 选择A | 选择B | 取舍分析 | 本方案选择 |
|------|-------|-------|----------|-----------|
| **时间 vs 空间** | 小 Chunk（低延迟） | 大 Chunk（高吞吐） | 256 平衡 | 256 tokens |
| **一致性 vs 性能** | 立即写回（安全） | 延迟写回（高性能） | 接受一定丢失风险 | 延迟写回 |
| **复杂度 vs 收益** | 单层管理 | 双层管理 | 收益显著 | 双层 |
| **通用性 vs 专用性** | 固定 Chunk | 自适应 Chunk | 简单优先 | 固定 |

**适用场景边界**：
- **最佳场景**：多轮对话、RAG、长上下文复用
- **次优场景**：单次请求、无共享上下文
- **不适用**：延迟极敏感场景（延迟写回增加风险）

**扩展性天花板**：
- Chunk 大小固定，无法适应不同网络带宽
- 超长序列（>1M tokens）需要多级分层

### 方案3：分层异构粒度 — HBM/DRAM/SSD 差异化管理

#### 1. 问题背景与必要性

**解决什么问题**：
- 单一粒度无法满足多级存储的差异化需求
- HBM 需要细粒度管理以最大化利用率
- SSD 需要粗粒度以优化顺序 IO
- 不同层级的访问频率差异巨大（热数据 1%，温数据 10%，冷数据 89%）

**不解决的后果**：
- HBM 碎片率高，无法支持大 batch
- SSD 随机 IO 性能差，Offloading 效果不佳
- 无法根据数据热度动态调整存储位置

**替代方案对比**：
| 方案 | HBM 效率 | SSD 效率 | 实现复杂度 |
|------|----------|----------|-----------|
| 统一细粒度 | 高 | 极低 | 低 |
| 统一粗粒度 | 低 | 高 | 低 |
| **分层异构** | **高** | **高** | **高** |

#### 2. 核心原理

**算法思想**：
借鉴 CPU 多级缓存（L1/L2/L3）的设计思想，为不同存储层级配置不同管理粒度，热数据用小块精细管理，冷数据用大块批量传输。

**关键数据结构**：
```
Hierarchical Block Table:
  HBM (L1):  Block Size = 16 tokens
  DRAM (L2): Chunk Size = 256 tokens
  SSD (L3):  Segment Size = 4096 tokens

Promotion/Demotion Policy:
  - Access count > 3 in 1000 steps → Promote to HBM
  - No access for 5000 steps → Demote to SSD
```

**理论复杂度分析**：
| 层级 | 粒度 | 访问延迟 | 管理开销 | 容量比例 |
|------|------|----------|----------|----------|
| HBM | 16t | <1μs | 高 | 1% |
| DRAM | 256t | ~10μs | 中 | 10% |
| SSD | 4096t | ~1ms | 低 | 89% |

#### 3. 设计机制

**系统架构**：
```
┌─────────────────────────────────────────────────────────┐
│                   Hierarchical Manager                  │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────┐  │
│  │  Hot    │   │  Warm   │   │  Cold   │   │  Ice   │  │
│  │ (HBM)   │◄─►│ (DRAM)  │◄─►│ (SSD)   │◄─►│(Remote)│  │
│  │ 16t/blk │   │256t/chk │   │4Kt/seg  │   │64Kt/obj│  │
│  └────┬────┘   └────┬────┘   └────┬────┘   └───┬────┘  │
│       │             │             │            │       │
│       └─────────────┴─────────────┴────────────┘       │
│                         │                              │
│                    ┌────┴────┐                         │
│                    │Migrator │                         │
│                    │(动态迁移) │                         │
│                    └─────────┘                         │
└─────────────────────────────────────────────────────────┘
```

**关键设计机制**：
1. **热度追踪**：基于访问频率和 recency 的评分机制
2. **异步迁移**：后台线程执行层级间数据迁移
3. **预取策略**：基于访问模式的预测性加载

**数据流**：
```
Access Pattern → Hotness Score → Migration Decision → Async Transfer
                     │
                     ▼
              ┌─────────────┐
              │  HBM Hit?   │──Yes──▶ Direct Access
              └──────┬──────┘
                     │ No
                     ▼
              ┌─────────────┐
              │  DRAM Hit?  │──Yes──▶ Promote to HBM
              └──────┬──────┘
                     │ No
                     ▼
              ┌─────────────┐
              │  SSD Hit?   │──Yes──▶ Load to DRAM
              └──────┬──────┘
                     │ No
                     ▼
              Compute from scratch
```

#### 4. 架构决策分析

**关键决策点1：各层粒度的选择依据**

| 层级 | 粒度 | 选择理由 |
|------|------|----------|
| HBM | 16 tokens | 与 vLLM 对齐，最大化 GPU 内存利用率 |
| DRAM | 256 tokens | 与 LMCache 对齐，优化 PCIe 传输 |
| SSD | 4096 tokens | 匹配 SSD 4KB 页大小，优化顺序 IO |

**关键决策点2：迁移策略**

| 策略 | 触发条件 | 优点 | 缺点 |
|------|----------|------|------|
| **访问计数** | 3次/1000步 | 简单有效 | 延迟敏感 |
| **预测预取** | 模式匹配 | 零延迟 | 预测准确率 |
| **应用提示** | API 调用 | 精确控制 | 需要应用配合 |

#### 5. 性能剖析

**瓶颈点识别**：
1. **迁移开销**：数据在层级间移动消耗带宽
2. **热度追踪开销**：维护统计信息需要额外内存和计算
3. **抖动**：频繁在边界访问的数据导致反复迁移

**关键性能路径**：
```
HBM Hit:  ~1μs
DRAM Hit: ~10μs + 可能的 Promotion
SSD Hit:  ~1ms + DRAM Promotion
Miss:     Compute time
```

**量化数据**：
| 指标 | 数值 | 来源 |
|------|------|------|
| HBM 命中率 | 78-82% | PAM 论文 |
| DRAM 命中率 | 15-18% | 估算 |
| SSD 命中率 | 2-5% | 估算 |
| 迁移开销 | <5% 总带宽 | 设计目标 |

#### 6. Trade-off 与限制

| 维度 | 选择A | 选择B | 取舍分析 | 本方案选择 |
|------|-------|-------|----------|-----------|
| **时间 vs 空间** | 激进迁移（低延迟） | 保守迁移（低开销） | 平衡策略 | 计数阈值 |
| **一致性 vs 性能** | 同步迁移（安全） | 异步迁移（高性能） | 异步优先 | 异步 |
| **复杂度 vs 收益** | 单层 | 多层 | 收益显著 | 3-4层 |
| **通用性 vs 专用性** | 通用策略 | 工作负载定制 | 通用优先 | 通用 |

**适用场景边界**：
- **最佳场景**：长上下文、多轮对话、有明显访问模式的工作负载
- **次优场景**：随机访问、无规律的工作负载
- **不适用**：访问模式完全随机的场景

### 方案4：可变长 Chunk — 语义边界切分

#### 1. 问题背景与必要性

**解决什么问题**：
- 固定粒度切分会破坏语义完整性（如句子被截断）
- 语义边界处的 Attention 模式通常更重要
- 固定粒度导致重要 token 被均匀压缩，而非重要 token 浪费空间

**不解决的后果**：
- 压缩后准确率下降（ChunkKV 论文：固定方法下降 5.5-55.1%）
- 语义不连贯导致生成质量下降
- 无法利用自然语言的层次结构

**替代方案对比**：
| 方案 | 语义保持 | 复杂度 | 压缩率 |
|------|----------|--------|--------|
| 固定大小 | 差 | 低 | 高 |
| **语义切分** | **好** | **高** | **中** |
| 动态预算 | 较好 | 很高 | 中 |

#### 2. 核心原理

**算法思想**：
基于自然语言的层次结构（段落/句子/短语）进行切分，保持语义单元的完整性，在压缩时优先保留语义边界处的信息。

**关键数据结构**：
```
Semantic Chunk:
  - boundary_type: {paragraph, sentence, phrase}
  - tokens: variable length
  - importance_score: computed by attention analysis
  - compression_budget: dynamic allocation

Delimiter Detection:
  - 标点符号: 。！？. ! ?
  - 特殊 token: [SEP], [EOS]
  - 注意力突变点: attention score 剧烈变化位置
```

**理论复杂度分析**：
| 操作 | 固定大小 | 语义切分 | 开销增加 |
|------|----------|----------|----------|
| 切分 | O(1) | O(L) | L 倍 |
| 匹配 | O(1) | O(log N) | log N 倍 |
| 存储 | O(1) | O(1) | 相同 |

#### 3. 设计机制

**ChunkKV 设计**：
```
┌─────────────────────────────────────────┐
│           ChunkKV System                │
│  ┌─────────────┐    ┌──────────────┐   │
│  │ Attention   │───▶│ Chunk Score  │   │
│  │ Analysis    │    │ Computation  │   │
│  └─────────────┘    └──────┬───────┘   │
│                            │            │
│                            ▼            │
│  ┌──────────────────────────────────┐   │
│  │    Semantic Chunk Detection      │   │
│  │  (punctuation + attention pattern)│   │
│  └──────────────────────────────────┘   │
│                            │            │
│                            ▼            │
│  ┌──────────────────────────────────┐   │
│  │    Layer-wise Index Reuse        │   │
│  │  (跨层压缩索引复用)               │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**关键设计机制**：
1. **语义边界检测**：结合标点和注意力模式识别语义单元
2. **Chunk 级重要性评分**：基于 Attention 分布计算 Chunk 重要性
3. **跨层索引复用**：利用相邻层相似性减少计算

**DynSplit-KV 设计**：
```
┌─────────────────────────────────────────┐
│         DynSplit-KV System              │
│  ┌─────────────┐    ┌──────────────┐   │
│  │ Dynamic     │───▶│ Importance-  │   │
│  │ Delimiter   │    │ Aware Select │   │
│  │ Selection   │    │              │   │
│  └─────────────┘    └──────┬───────┘   │
│                            │            │
│                            ▼            │
│  ┌──────────────────────────────────┐   │
│  │    Uniform Mapping Strategy      │   │
│  │  (可变长 → 定长格式转换)          │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**关键设计机制**：
1. **动态定界符选择**：根据场景自适应选择切分点
2. **统一映射策略**：将可变长块映射为固定长度格式，降低推理开销

#### 4. 架构决策分析

**关键决策点1：定界符选择策略**

| 定界符类型 | 准确率影响 | 实现复杂度 | 适用场景 |
|------------|-----------|-----------|----------|
| 固定标点 | -28.6% | 低 | 通用 |
| **动态选择** | **-5%** | **高** | **自适应** |
| 注意力突变 | -15% | 中 | 长上下文 |

**关键决策点2：可变长到定长的映射**

| 方案 | 推理开销 | 存储效率 | 选择理由 |
|------|----------|----------|----------|
| 保持可变长 | +73.1% | 高 | 不适用 |
| **统一映射** | **+15%** | **中** | **开销可接受** |
| 填充对齐 | +5% | 低 | 简单 |

#### 5. 性能剖析

**瓶颈点识别**：
1. **语义分析开销**：需要额外的 Attention 计算
2. **可变长存储**：非对齐访问降低内存效率
3. **匹配复杂度**：Radix Tree 查询比 Hash 表慢

**关键性能路径**：
```
Prefill:  Full Attention → Chunk Scoring → Selective Retention
Decode:   Query → Semantic Matching → Chunk Loading
```

**量化数据**：
| 指标 | ChunkKV | 固定大小 | 提升 |
|------|---------|----------|------|
| LongBench 准确率 | 45.8% | 42.3-44.1% | +1.5-3.5% |
| Needle 检索率 | 100% | 95-98% | +2-5% |
| 吞吐量提升 | 26.5% | 基准 | +26.5% |
| 压缩率 | 4× | 4× | 相同 |

#### 6. Trade-off 与限制

| 维度 | 选择A | 选择B | 取舍分析 | 本方案选择 |

| **通用性 vs 专用性** | 通用 | 语言专用 | 通用优先 | 通用 |

**适用场景边界**：
- **最佳场景**：长文本理解、文档问答、需要保持语义完整性的场景
- **次优场景**：短文本、生成任务
- **不适用**：实时性要求极高的场景（语义分析增加延迟）

## 与 KV Cache 特性的关系

### Head 维度的影响

**Head Dimension (d_h) 对 Block 大小的影响**：
```
Block Size (bytes) = block_size_tokens × num_layers × 2 × num_kv_heads × d_h × sizeof(dtype)

Example (Llama-2-7B):
  = 16 × 32 × 2 × 32 × 128 × 2
  = 8,388,608 bytes ≈ 8 MB per block
```

| Head Dim | 16 tokens 大小 | 256 tokens 大小 | 影响 |
|----------|---------------|-----------------|------|
| 64 | 4 MB | 64 MB | 适合小 Block |
| 128 | 8 MB | 128 MB | 标准配置 |
| 256 | 16 MB | 256 MB | 考虑减小 Block |

**GQA/MQA 的影响**：
| 注意力类型 | KV Heads | Block 大小 | 元数据开销 |
|------------|----------|-----------|-----------|
| MHA | 32 | 100% | 基准 |
| GQA | 8 | 25% | 4× 降低 |
| MQA | 1 | 3.125% | 32× 降低 |

### Layer 维度的影响

**Layer 数量对传输策略的影响**：
```
Total KV Size = num_layers × 2 × num_heads × seq_len × d_h
```

| 模型 | Layers | 总 KV 大小 (4K seq) | 传输策略 |
|------|--------|---------------------|----------|
| Llama-7B | 32 | 512 MB | 全量传输 |
| Llama-70B | 80 | 2.5 GB | 分层传输 |
| GPT-4 (est) | 120 | 10+ GB | 选择性传输 |

**跨层共享（CLA）的影响**：
- 相邻层共享 KV Cache，减少 50% 存储
- 传输时可只传输奇数层或偶数层
- 对 Block 粒度选择无直接影响，但影响总传输量

### 决策结论

**推荐架构：分层异构粒度**

| 层级 | 粒度 | 选择理由 | 映射关系 |
|------|------|----------|----------|
| HBM (L1) | 16 tokens | 与vLLM对齐，最大化GPU内存利用率 | 基准粒度 |
| DRAM (L2) | 256 tokens | 优化PCIe传输，LMCache验证 | 16 × 16 |
| SSD (L3) | 4096 tokens | 匹配SSD 4KB页大小，优化顺序IO | 256 × 16 |
| Remote (L4) | 65536 tokens | 最大化网络带宽利用率 | 4096 × 16 |

**双层管理架构**：

```
vLLM Block Layer (16 tokens)
    ↕ 聚合/拆分
LMCache Chunk Layer (256 tokens)
    ↕ 传输
Storage Layer (SSD/Remote)
```

**关键假设（需验证）**：
1. 双层管理的聚合/拆分开销 < 5%总时间
2. 256 tokens Chunk在典型网络带宽下利用率 > 60%
3. 4096 tokens SSD写入匹配4KB页大小，随机IO性能可接受

**风险与缓解**：
| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 粒度映射开销过高 | 中 | 高 | 优化CUDA kernel，批量聚合 |
| 大Chunk前缀匹配精度下降 | 高 | 中 | 实现部分Chunk匹配机制 |
| 超长序列Block Table膨胀 | 中 | 中 | Block Table分层，超长序列特殊处理 |
