## 问题定义

KV Cache 存储系统的数据冗余策略需要解决的核心问题：

1. **数据耐久性**：单盘/单节点故障时，KV Cache 数据不丢失
2. **重建性能**：故障后重建数据的速度（影响 MTTR 和重建风暴风险）
3. **存储效率**：冗余开销（多副本的 2x/3x 空间 vs 纠删码的 1.2x~1.5x 空间）
4. **读写性能影响**：冗余策略对正常 I/O 路径的影响

**场景约束**：
- 数据特征：KV Cache 数据块大（32MB~100MB+），写入后只读，生命周期从分钟到小时/天
- 容量目标：1TB~100TB 的 KV Cache 数据
- 成本敏感：用户以"低成本外部存储替代 HBM"为核心价值主张
- 硬件：全 NVMe SSD，4~16 节点

## 标杆冗余方案提炼

| 标杆 | 冗余策略 | 关键特征 | 与本系统的关联 |
|------|---------|---------|--------------|
| **VAST Data** | **宽条带纠删码（146+4 LDC）** | 局部可解码；重建只需读取 1/4 幸存数据；开销 2.7%；MTTDL 4200 万年 | 纠删码的极致实现，但条带宽度过大不适合小集群 |
| **Ceph** | **多副本 / 纠删码可选** | 默认 3 副本；纠删码支持 k+m（如 4+2）；灵活配置 | 通用参考，但 Ceph 的纠删码实现复杂度高 |
| **MoonCake** | **未明确公开** | 生产环境必有冗余，但论文未详述 | 无法直接参考 |
| **LMCache** | **无冗余**（纯缓存层）| 作为缓存层，不保证持久性；丢失后重新计算 | 本系统需完整存储能力，不能仅做缓存 |
| **MinIO** | **纠删码（ Reed-Solomon）** | 默认 4+2 或 8+4；每对象独立编码；分布式校验 | 对象存储的纠删码参考 |

## 候选方案

### 方案 A：3 副本（多副本）

**核心思路**：每份 KV Cache 数据在 3 个不同节点上各存一份完整副本。写入时同步或异步复制到 2 个副本节点，读取时从最近的副本读取。

**参考来源**：HDFS 默认 3 副本 + Ceph 默认多副本 + Cassandra

**优点**：
1. **实现最简单**：无编码/解码计算，直接复制完整数据
2. **读性能最好**：可从任意副本读取，负载均衡；单副本故障不影响读取
3. **重建最快**：只需从幸存副本直接拷贝，无计算开销
4. **小文件友好**：无需考虑条带对齐，任意大小的数据块都可复制

**缺点**：
1. **存储开销最高**：200% 冗余开销（3x 原始数据），成本最高
2. **写放大**：写入需复制 3 份，网络带宽消耗大
3. **跨节点写入延迟**：同步复制时需等待 2 个副本确认

**适用场景**：数据量小（< 10TB）、对重建速度要求极高、实现资源有限的场景。

**量化指标**：
| 指标 | 数值 |
|------|------|
| 存储开销 | 200%（3x） |
| 可容忍故障 | 任意 2 节点 |
| 重建读取量 | 1x（单副本拷贝） |
| 重建计算量 | 0（无编码/解码） |
| 重建时间（1TB 数据，10Gbps）| ~15 分钟 |

---

### 方案 B：纠删码（Reed-Solomon，k+m）

**核心思路**：将数据分片为 k 个数据块，计算 m 个校验块，共 k+m 个块分布在不同节点。可容忍任意 m 个块故障。

**参考来源**：Ceph 纠删码 + MinIO Reed-Solomon + RAID-6

**优点**：
1. **存储效率最高**：冗余开销 = m/k，如 4+2 配置仅需 50% 冗余
2. **可配置灵活**：根据可靠性要求调整 k 和 m
3. **写入性能可控**：数据块和校验块可并行写入

**缺点**：
1. **重建性能差**：重建需读取 k 个幸存数据块并计算，重建流量大
2. **读取性能下降**：单节点故障时，读取需从多个节点收集数据块并解码
3. **实现复杂**：需要 Galois Field 运算库，编码/解码引入 CPU 开销
4. **条带对齐要求**：小对象需要填充到条带大小，空间浪费

**适用场景**：数据量大（> 10TB）、读多写少、可接受重建开销的场景。

**量化指标**（以 4+2 配置为例）：
| 指标 | 数值 |
|------|------|
| 存储开销 | 50%（1.5x） |
| 可容忍故障 | 任意 2 节点 |
| 重建读取量 | 4x（读取 4 个幸存数据块） |
| 重建计算量 | 高（Galois Field 解码） |
| 重建时间（1TB 数据，10Gbps）| ~60 分钟（含解码） |

---

### 方案 C：混合策略（热数据多副本 + 冷数据纠删码）

**核心思路**：根据数据热度动态选择冗余策略。热数据（高频访问、新写入）使用 3 副本，冷数据（低频访问、旧数据）后台迁移到纠删码存储。

**参考来源**：Facebook HDFS RAID（热数据多副本 → 冷数据纠删码）+ VAST Data 的热数据策略

**优点**：
1. **兼顾性能和效率**：热数据享受多副本的读取性能，冷数据享受纠删码的存储效率
2. **渐进式迁移**：后台自动迁移，对前台 I/O 无影响
3. **成本最优**：实际存储开销介于多副本和纠删码之间（取决于冷热比例）

**缺点**：
1. **实现最复杂**：需要维护两套存储路径、迁移逻辑、策略决策
2. **迁移开销**：后台迁移消耗网络和磁盘带宽
3. **策略调参复杂**：冷热判定阈值、迁移时机等参数需要调优

**适用场景**：冷热数据分布明显、有资源实现复杂策略的场景。

**量化指标**（假设 30% 热数据）：
| 指标 | 数值 |
|------|------|
| 有效存储开销 | ~120%（30% × 200% + 70% × 50%） |
| 可容忍故障 | 热数据：2 节点；冷数据：2 节点（4+2） |
| 平均重建时间 | ~30 分钟（介于多副本和纠删码之间） |

---

### 方案 D：局部可修复码 LRC（类似 VAST Data LDC 的简化版）

**核心思路**：使用局部可修复码（Locally Repairable Codes），重建时只需读取部分幸存块而非全部。相比传统 RS 码，重建读取量显著减少。

**参考来源**：VAST Data LDC（局部可解码纠删码）+ Azure LRC + Facebook HDFS-Xorbas

**优点**：
1. **重建性能接近多副本**：只需读取少量局部校验块即可重建
2. **存储效率接近 RS 码**：开销略高于 RS，但远低于多副本
3. **局部性优化**：重建流量限制在局部节点组，减少网络冲击

**缺点**：
1. **实现复杂度高**：LRC 编码/解码算法比 RS 更复杂
2. **参数调优困难**：局部组大小、全局校验块数等参数需要仔细调优
3. **小集群不适用**：LRC 的优势在大盘/大集群中才能体现，4~16 节点下优势有限

**适用场景**：大规模集群（100+ 盘）、重建性能是关键瓶颈的场景。

**量化指标**（以 (4,2,1) LRC 为例：4 数据 + 2 局部校验 + 1 全局校验）：
| 指标 | 数值 |
|------|------|
| 存储开销 | 75%（1.75x） |
| 可容忍故障 | 任意 2 节点（局部）+ 1 全局 |
| 重建读取量 | 1x~2x（仅需局部组内数据） |
| 重建时间（1TB 数据，10Gbps）| ~25 分钟 |

## 关键 Trade-off 分析

### T1：存储成本 vs 重建性能
- **方案 A** 选择重建性能，接受最高存储成本
- **方案 B** 选择存储效率，接受最差重建性能
- **本系统选择**：偏向方案 C（混合策略），原因：
  - 用户以"低成本替代 HBM"为核心价值，存储成本是关键竞争力
  - 但 KV Cache 的热数据（新写入、高频复用）需要快速读取，多副本有优势
  - 冷数据（历史积累、低频访问）可用纠删码节省空间

### T2：实现复杂度 vs 灵活性
- **方案 C** 最灵活但最复杂
- **方案 A** 最简单但成本最高
- **本系统选择**：MVP 阶段采用方案 A（3 副本），后续迭代引入方案 C（混合策略），原因：
  - 4~16 节点规模下，3 副本的存储开销尚可接受（假设单节点 10TB NVMe，3 副本后有效容量 = 总容量 / 3）
  - MVP 优先保证正确性和可维护性，复杂策略后续引入

### T3：小集群下的纠删码有效性
- 4~16 节点下，宽条带纠删码（如 VAST 的 146+4）完全不可用
- 窄条带（如 4+2）需要至少 6 节点，在本系统的最小规模（4 节点）下不适用
- **本系统选择**：纠删码仅在 ≥6 节点时启用，4~5 节点使用 3 副本

## 初步倾向

**分阶段策略**：

### MVP 阶段（v1.0）：3 副本
- 所有数据使用 3 副本
- 简单、可靠、快速实现
- 支持的最小集群：3 节点（每个副本在不同节点）

### 演进阶段（v2.0）：混合策略
- 热数据（最近写入、高频访问）：3 副本
- 冷数据（低频访问、旧数据）：4+2 纠删码
- 后台自动迁移（基于访问频率和年龄）
- 支持的最小集群：6 节点（纠删码需要）

### 关键设计决策：
1. **写入路径**：无论冗余策略如何，写入先确认到本地节点，后台异步复制/编码
2. **读取路径**：优先从本地副本读取（若有），否则从最近副本读取
3. **重建限速**：重建带宽限制为总带宽的 30%，避免 rebuild storm
4. **副本放置策略**：跨机架/跨 NUMA 域放置副本（若硬件支持）
