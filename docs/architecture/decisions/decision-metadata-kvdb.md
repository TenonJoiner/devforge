- **状态**: 草稿
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中

---

## 背景

### 问题定义

KV Cache 分布式存储系统的元数据管理面临以下核心挑战：

1. **读多写少的极端访问模式**：元数据在 chunk 创建时写入一次，随后被高频读取（每次推理请求都需查询 chunk 位置），引用计数更新频率中等。估计读写比为 **50:1 ~ 500:1**（取决于 Prefix Cache 命中率和请求复用度）。

2. **范围查询需求**：需要支持按 prefix hash 的范围查询，以查找某段 token 序列对应的 KV Cache chunk 位置。这要求存储引擎支持高效的有序迭代。

3. **引用计数更新的写放大敏感**：chunk 的引用计数（ref_count）在每次请求开始/结束时增减。若使用 LSM-Tree，每次更新产生一条新记录，compaction 时需扫描合并大量版本，写放大极高。

4. **分布式扩展**：元数据需随存储节点水平扩展，单节点管理百万级~千万级条目，集群总规模可达数亿条。

5. **用户明确质疑**："RocksDB 的 LSM-Tree 布局方式对读场景不是最优的，在 KV Cache 下是否合适需要考虑"。

### 元数据条目结构估算

| 字段 | 大小 | 说明 |
|------|------|------|
| chunk_id | 16 B | UUID 或自增 ID |
| prefix_hash | 32 B | token 序列的哈希值 |
| location (node_id + offset + length) | 24 B | 物理位置 |
| ref_count | 4 B | 引用计数 |
| create_time | 8 B | 创建时间戳 |
| model_version | 32 B | 模型版本标识 |
| status_flags | 4 B | 状态标记 |
| **单条合计** | **~120 B** | 不含 padding |
| 含索引开销估算 | **~200 B** | 含 B+tree/LSM 索引开销 |

**规模估算**：
- 单节点 100 万条目：约 200 MB
- 单节点 1000 万条目：约 2 GB
- 16 节点集群：1.6 亿条目，约 32 GB 总元数据

## 候选方案

### 方案 A：RocksDB 作为元数据 KVDB

**核心思路**：使用成熟的 RocksDB 作为元数据存储引擎，通过配置优化（prefix_extractor、block_cache、Bloom Filter）缓解读放大问题。引用计数更新通过 RocksDB Merge Operator 合并，减少写放大。

**参考来源**：RocksDB (Facebook) + TiKV 实践经验

**优点**：
1. **生态成熟**：经过 Facebook、TiKV、MyRocks 等大规模生产验证，bug 少，社区活跃
2. **功能丰富**：内置压缩、备份、快照、WAL、事务等，减少自研工作量
3. **范围查询原生支持**：有序存储，迭代器接口成熟，prefix_extractor 可优化前缀查询
4. **C API 可用**：提供 `rocksdb/c.h`，可从 C 代码直接调用
5. **Merge Operator 优化引用计数**：自定义 merge 操作可将多次引用计数更新合并为一次，减少写放大

**缺点**：
1. **读放大固有缺陷**：点查询读放大 3-10x，范围查询读放大 10-100x（即使配置优化后，仍高于 B-tree）
2. **Compaction 延迟抖动**：后台 compaction 占用磁盘 IO，导致 P99 读延迟波动（实测可达 2-5x）
3. **C++ 依赖**：核心为 C++ 实现，编译依赖重，与项目纯 C 定位有摩擦
4. **内存占用不可控**：Block cache、Index/filter block、Memtable 多层缓存，总内存占用难以精确预估

**适用场景**：团队已有 RocksDB 经验、追求快速落地、可接受一定读延迟开销的场景。

**置信度**：中（读放大问题真实存在，但可通过配置缓解）

---

### 方案 B：LMDB 嵌入式 B+tree

**核心思路**：使用 LMDB（Lightning Memory-Mapped Database）作为元数据引擎。LMDB 基于 B+tree + Copy-on-Write + 内存映射，读性能极优，天然适合读多写少场景。

**参考来源**：LMDB (Symas) + OpenLDAP 生产实践

**优点**：
1. **读性能极优**：B+tree 点查询只需树高次访问（通常 3-4 层），无读放大；内存映射实现零拷贝读取
2. **纯 C 实现**：与项目技术栈完全匹配，无 C++ 依赖，代码量仅 ~10K LOC
3. **多读者并发**：MVCC 机制支持无锁多线程读，读并发线性扩展（实测 16 线程可达 10M ops/sec）
4. **范围查询高效**：B+tree 叶子节点链表支持顺序扫描，范围查询只需遍历相邻叶子
5. **内存占用确定**：内存映射文件，占用 = 文件大小 + 少量元数据，无隐藏缓存

**缺点**：
1. **单写者瓶颈**：所有写操作（包括引用计数更新）串行，写吞吐量受限（~100K ops/sec）
2. **文件大小需预分配**：数据库文件大小增长需要显式操作，不支持自动扩展
3. **无内置压缩**：元数据需自行压缩，或依赖文件系统压缩
4. **分布式扩展需自研**：LMDB 为单节点引擎，集群分片、复制、故障恢复需自行实现

**适用场景**：读极多写极少、单节点元数据规模可控、追求极致读性能的场景。

**置信度**：高（LMDB 在读多写少场景的优越性有充分理论和实践支撑）

---

### 方案 C：内存 Hash 表 + 异步 WAL 持久化

**核心思路**：元数据完全驻留内存（使用 uthash 或 Judy 数组），通过异步 WAL（Write-Ahead Log）保证持久化。重启时从 WAL 重放恢复。引用计数更新纯内存操作，无持久化开销。

**参考来源**：Redis AOF + LMCache 内存索引 + uthash/Judy

**优点**：
1. **读性能最优**：纯内存哈希表查找，O(1) 复杂度，单次访问 ~100ns，比磁盘引擎快 1000x
2. **引用计数零开销**：内存中原地更新 ref_count，无 WAL 同步开销（或异步批量刷盘）
3. **实现极简**：uthash（单头文件，~2000 行）或 Judy 数组，无需引入外部引擎
4. **内存占用可控**：单节点千万级条目约 2-5GB，现代服务器 DRAM 充裕（通常 256GB+）
5. **Prefix Hash 索引灵活**：可在内存中维护 trie 树或二级索引，不受存储引擎限制

**缺点**：
1. **数据量受内存限制**：单节点无法管理超过物理内存的元数据（但 2-5GB 在 256GB 服务器上占比很小）
2. **故障恢复时间**：重启时需重放 WAL，千万级条目可能需要数秒至数十秒
3. **无内置范围查询**：哈希表不支持有序遍历，需额外维护有序结构（如跳表或 B+tree 索引）
4. **持久化可靠性**：异步 WAL 存在窗口期内数据丢失风险（可配置 fsync 策略权衡）

**适用场景**：元数据规模明确可控、读性能要求极致、可接受内存存储约束的场景。

**置信度**：高（技术简单可靠，风险主要来自规模假设）

---

### 方案 D：自研轻量 B+tree 元数据引擎

**核心思路**：针对 KV Cache 元数据场景自研轻量 B+tree 引擎。核心设计：内存中缓存整棵 B+tree（或至少缓存所有内部节点），叶子节点按需加载；引用计数更新原地修改内存中的叶子节点，异步批量刷盘。

**参考来源**：SQLite B-tree + LMDB B+tree 设计 + Bw-Tree 论文

**优点**：
1. **完全场景定制**：可针对 KV Cache 元数据的访问模式优化（如固定大小记录、引用计数特殊处理）
2. **引用计数原地更新**：B+tree 支持原地修改，引用计数更新不产生版本累积
3. **范围查询原生支持**：B+tree 叶子节点链表天然支持 prefix hash 范围扫描
4. **无外部依赖**：纯 C 自研，与项目完全融合，无版本兼容问题
5. **内存/磁盘混合**：内部节点常驻内存，叶子节点按需加载，兼顾性能和容量

**缺点**：
1. **开发成本高**：需实现 B+tree 插入、删除、分裂、合并、并发控制等，代码量 ~5000-10000 行
2. **测试负担重**：存储引擎的正确性要求极高，需大量单元测试和故障注入验证
3. **功能缺失**：需自行实现 WAL、快照、压缩、备份等，工作量显著
4. **风险集中**：存储引擎 bug 可能导致数据丢失，团队需具备深度调试能力

**适用场景**：团队有存储引擎开发经验、元数据访问模式特殊、现有引擎无法满足需求的场景。

**置信度**：低（开发成本高，风险大，除非其他方案均不满足才考虑）

## 关键 Trade-off

### 1. 读性能 vs 工程复杂度

- **RocksDB**：读性能中等（50-200us），但工程复杂度最低，集成即可使用
- **LMDB/自研 B+tree/内存 Hash**：读性能更优（1-50us），但需要更多集成工作或自研投入

**权衡**：KV Cache 元数据查询在关键路径上（每次推理请求都需查询），读延迟直接影响端到端延迟。若 RocksDB 的 50-200us 读延迟在整体预算内（P99 < 10ms 含网络传输），则可接受；否则需选择更优读性能的引擎。

### 2. LSM-Tree 写优化 vs 读多写少场景错配

LSM-Tree 的设计假设是**写密集型 workload**（通过顺序写和批量 compaction 摊平写成本）。但 KV Cache 元数据是**极端读多写少**：

- 每次 chunk 创建产生 1 次写
- 每次推理请求产生 50-500 次读
- 每次请求产生 0-2 次引用计数更新

**读写比 50:1 ~ 500:1**，LSM-Tree 的写优化优势无法发挥，反而承受读放大的代价。

### 3. 引用计数更新模式 vs 存储引擎适配

引用计数更新的特征是**高频、小数据量、可合并**：

| 引擎 | 引用计数处理方式 | 写放大 |
|------|-----------------|--------|
| RocksDB | Merge Operator 合并多版本 | 中等（取决于合并频率） |
| LMDB | 原地更新 B+tree 叶子节点 | 1x |
| 内存 Hash | 内存中原地修改 | 0x |
| 自研 B+tree | 原地更新 | 1x |

RocksDB 的 Merge Operator 虽可减少写放大，但增加了读取时的合并开销（读一条记录需合并多个版本）。

### 4. 内存容量 vs 读性能

| 方案 | 单节点元数据容量 | 读性能 |
|------|-----------------|--------|
| RocksDB | 无限制（磁盘为主） | 中 |
| LMDB | 受磁盘容量限制 | 高 |
| 内存 Hash | 受物理内存限制 (~2-5GB/千万条) | 极高 |
| 自研 B+tree | 可设计为内存/磁盘混合 | 高 |

单节点 256GB DRAM 可轻松容纳千万级元数据（2-5GB），内存 Hash 方案的容量约束在实际中不构成瓶颈。

## 决策

**初步倾向：方案 C（内存 Hash 表 + 异步 WAL）作为首选，方案 B（LMDB）作为备选。**

### 选择方案 C 的核心理由

1. **场景高度匹配**：KV Cache 元数据读多写少的特征与纯内存方案完美契合。元数据规模（单节点 2-5GB）在现代服务器内存容量下完全可行。

2. **读性能决定性优势**：纯内存哈希表 O(1) 查找（~1-5us）比 RocksDB（50-200us）快 **10-100 倍**。元数据查询在推理关键路径上，这直接转化为端到端延迟优势。

3. **引用计数零开销**：内存中原地更新 ref_count，无需持久化同步，彻底解决引用计数更新的写放大问题。

4. **实现极简可控**：uthash（单头文件）或 Judy 数组即可满足，无外部引擎依赖，无 C++ 引入，与项目纯 C 定位一致。

5. **Prefix Hash 索引灵活**：内存中可自由设计索引结构（如 trie 树 + 哈希表组合），不受存储引擎的数据模型限制。

### 方案 B 作为备选的理由

若以下假设不成立，则退回到 LMDB：
- 元数据规模超过单节点内存容量（需精确评估）
- 需要持久化存储保证跨重启零丢失（LMDB 的 mmap + COW 更可靠）
- 团队对内存管理可靠性有顾虑

### 排除方案 A（RocksDB）的理由

1. **读放大与场景错配**：LSM-Tree 的读放大（3-10x 点查、10-100x 范围查）在读多写少场景下是固有缺陷，配置优化只能缓解不能消除。
2. **Compaction 延迟抖动**：后台 compaction 导致 P99 读延迟波动，与系统 P99 < 10ms 的目标冲突。
3. **C++ 依赖摩擦**：项目定位为纯 C 开发，引入 RocksDB（C++）增加编译链复杂性和二进制体积。

### 排除方案 D（自研 B+tree）的理由

1. **开发成本过高**：自研存储引擎需 5000-10000 行代码 + 大量测试，与"不重复造轮子"原则冲突。
2. **风险收益比不佳**：LMDB 已提供成熟的 B+tree 实现，自研的收益（定制化）不足以抵消风险。

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 元数据规模超过内存容量 | 中 | 高 | ① 精确建模单节点元数据上限；② 超过阈值时自动切换至 LMDB；③ 支持元数据分片到多进程 |
| 异步 WAL 导致故障窗口数据丢失 | 中 | 高 | ① 配置 `fsync` 策略（每秒/每请求）；② 关键操作（chunk 创建）强制同步 WAL；③ 引用计数可重建（从 chunk 引用关系重算） |
| 内存 Hash 表无内置范围查询 | 低 | 中 | ① 额外维护跳表或 B+tree 二级索引；② Prefix Hash 查询走独立索引结构 |
| 重启恢复时间过长 | 中 | 中 | ① 定期 Checkpoint（内存快照 + 增量 WAL）；② 并行重放 WAL；③ 热备节点快速接管 |
| 多线程哈希表并发瓶颈 | 低 | 中 | ① 分片哈希表（每个 CPU 核心一个分片）；② 使用 RCU 或细粒度锁；③ 参考 uthash 并发方案 |
| 方案 C 验证失败后回退成本 | 中 | 中 | ① 抽象元数据存储接口（`meta_store_t`），底层可替换；② 早期原型同时验证方案 C 和 B |

## 待验证假设

| # | 假设 | 验证方法 | 不成立的后果 |
|---|------|---------|------------|
| H1 | 单节点元数据峰值不超过 5GB（千万级条目） | 基于实际模型和 chunk 大小分布建模计算 | 需退回到 LMDB 或引入分片 |
| H2 | 异步 WAL（每秒 fsync）的丢失窗口可接受 | 故障注入测试：kill -9 后检查数据一致性 | 需改为每次写操作同步 fsync，或退回到 LMDB |
| H3 | 内存哈希表 + 跳表二级索引可满足 Prefix Hash 范围查询性能 | 原型测试：百万级条目下前缀查询 P99 延迟 | 需引入 B+tree 索引（LMDB 或自研） |
| H4 | 引用计数可从 chunk 引用关系重建（即使 WAL 丢失） | 设计引用计数重建算法并验证正确性 | 引用计数更新必须同步持久化 |
| H5 | 纯 C 的 uthash/Judy 可满足性能和功能需求 | 原型压测：对比 uthash/Judy/LMDB 的读写性能 | 需引入 C++ 引擎（RocksDB）或自研 |

---

## 替代方案排除理由

- **方案 A (RocksDB)**：LSM-Tree 的读放大和 compaction 抖动与 KV Cache 读多写少场景根本错配。虽然生态成熟，但性能特征不适合本场景。用户已明确质疑其适用性。
- **方案 D (自研 B+tree)**：开发成本高、测试负担重、风险集中。LMDB 已提供成熟的纯 C B+tree 实现，自研的收益不足以抵消风险。仅在方案 C 和 B 均不满足需求时才考虑。
