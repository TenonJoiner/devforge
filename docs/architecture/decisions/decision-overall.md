## 方案A：分层存储架构（LMCache风格）

### 核心思路

构建GPU显存→本地NVMe SSD→远程DRAM的三层分级存储体系，热数据驻留显存，温数据下沉SSD，冷数据按需从远程加载。

### 参考来源

- **LMCache**：Chunk-based存储单元、Connector抽象、分层存储策略
- **vLLM PagedAttention**：Block Table管理、按需加载机制

### 核心设计机制

**架构图**：

```
┌─────────────────────────────────────────┐
│           Inference Instance             │
│  ┌─────────┐    ┌──────────────────┐    │
│  │  vLLM   │◄──►│  KV Cache Manager │   │
│  │ Engine  │    │  (Block Table)    │   │
│  └─────────┘    └────────┬─────────┘    │
│                          │              │
│  ┌─────────┐    ┌────────┴────────┐     │
│  │  GPU    │◄──►│   Local SSD     │     │
│  │ Memory  │    │  (NVMe Cache)   │     │
│  └─────────┘    └────────┬────────┘     │
└──────────────────────────┼──────────────┘
                           │
              ┌────────────┴────────────┐
              │      Remote DRAM Pool    │
              │  (Cluster Memory Nodes)  │
              └─────────────────────────┘
```

**数据流**：
1. **写入路径**：KV Chunk → GPU Block → 异步刷入SSD → 条件上传远程
2. **读取路径**：本地GPU未命中 → 查询SSD索引 → SSD未命中则远程拉取 → 异步预取相邻Chunk
3. **淘汰策略**：显存LRU → SSD LRU+TTL → 远程持久化

**关键组件**：
- **Chunk Store**：固定大小Chunk（如256 tokens）作为最小存储单元，对齐LMCache设计
- **Location Index**：内存中的两级哈希表（ChunkID→{GPU/SSD/Remote}位置）
- **Async IO Engine**：基于io_uring的异步读写，避免阻塞推理
- **Connector Interface**：可插拔后端（本地文件/RDMA/对象存储）

### 优点

1. **成本效益最优**：复用节点闲置NVMe SSD，无需额外硬件投入，$/GB最低
2. **故障域隔离**：本地SSD作为缓冲，远程服务短暂不可用不影响本地推理
3. **渐进式部署**：可从单机SSD缓存开始，逐步扩展到远程集群

### 缺点

1. **性能抖动**：SSD带宽（~3GB/s）远低于显存（~900GB/s），大规模miss时延迟飙升
2. **容量碎片化**：Chunk大小固定，短序列造成内部碎片，长序列需跨Chunk管理
3. **一致性复杂**：三层之间的异步同步需处理脏数据、回源冲突等问题

### 适用场景边界

- **适合**：预算敏感、已有SSD资源、workload具有明显热点特征（如多轮对话）
- **不适合**：延迟敏感型应用（如实时交互）、SSD带宽成为瓶颈的大规模集群

## 方案B：分离式调度架构（MoonCake风格）

### 核心思路

将KV Cache生命周期从推理引擎完全剥离，构建独立的KV Pool服务集群，通过全局Conductor实现跨Prefill/Decode节点的KV动态调度。

### 参考来源

- **MoonCake**：Conductor全局调度、KVCache as a Service、分离式架构
- **NVIDIA CMX**：硬件加速分层、以存代算思想（用存储换计算）

### 核心设计机制

**架构图**：

```
┌─────────────────────────────────────────────────────┐
│                    Global Conductor                  │
│         (调度策略：局部性优先 / 负载均衡 / 成本)        │
└──────────────────────┬──────────────────────────────┘
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Prefill    │  │   Decode    │  │  KV Pool    │
│   Nodes     │  │    Nodes    │  │   Nodes     │
│ (Compute    │  │  (Compute   │  │ (Memory     │
│  Heavy)     │  │   Light)    │  │  Dense)     │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────────────┴────────────────┘
                          │
                    ┌─────┴─────┐
                    │  RDMA/    │
                    │  NVLink   │
                    │  Network  │
                    └───────────┘
```

**数据流**：
1. **Prefill阶段**：计算节点生成KV → 不本地缓存，直接通过RDMA写入KV Pool
2. **Decode阶段**：计算节点从KV Pool按需拉取所需KV块 → 异步预取后续块
3. **调度决策**：Conductor根据请求特征（序列长度、预估生存周期）决定KV放置策略

**关键组件**：
- **Conductor**：全局视图维护（各节点显存/内存容量、网络拓扑、负载状态），调度算法插件化
- **KV Pool Service**：专用内存节点，提供Put/Get/Delete API，内部按Instance组织
- **RDMA Transport**：零拷贝数据传输，双边操作保证低延迟
- **Token-based计费**：精确追踪每个Token的KV存储成本，支持多租户配额

### 优点

1. **资源解耦**：计算与存储独立扩缩容，Prefill节点无需大显存，KV节点无需计算能力
2. **全局最优调度**：跨实例KV共享，相同Prefix只需存储一份，重复计算彻底消除
3. **性能可预测**：专用KV服务网络隔离，不受推理计算抖动影响

### 缺点

1. **架构复杂度高**：引入独立服务集群，需解决Conductor单点、网络分区、脑裂等问题
2. **网络带宽敏感**：每次Decode都需网络拉取KV，RDMA带宽成为规模瓶颈
3. **冷启动延迟**：首次访问远程KV的RTT（~50μs RDMA vs ~1μs本地显存）影响TTFT

### 适用场景边界

- **适合**：大规模生产集群（100+节点）、多租户共享场景、Prefix重复率高的workload
- **不适合**：小规模部署（<10节点，overhead占比高）、网络基础设施不足的环境

## 方案C：统一Block池架构（vLLM PagedAttention扩展）

### 核心思路

在vLLM Block Table基础上扩展，构建跨实例的全局虚拟Block地址空间，实现Block级COW（Copy-on-Write）共享和细粒度迁移。

### 参考来源

- **vLLM PagedAttention**：Block Table、COW机制、Prefix Caching
- **LMCache**：Connector抽象（用于跨节点Block传输）

### 核心设计机制

**架构图**：

```
┌─────────────────────────────────────────────────────────┐
│                    Global Block Directory                │
│              (Virtual Block ID → Physical Location)      │
│                  ┌─────────────────┐                    │
│                  │  Block RefCount │                    │
│                  │  COW Pointer    │                    │
│                  │  Version Chain  │                    │
│                  └─────────────────┘                    │
└─────────────────────────┬───────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│  Instance A   │  │  Instance B   │  │  Instance C   │
│ ┌───────────┐ │  │ ┌───────────┐ │  │ ┌───────────┐ │
│ │ Block     │ │  │ │ Block     │ │  │ │ Block     │ │
│ │ Table     │◄┼──┼►│ Table     │◄┼──┼►│ Table     │ │
│ │ (Local)   │ │  │ │ (Local)   │ │  │ │ (Local)   │ │
│ └─────┬─────┘ │  │ └─────┬─────┘ │  │ └─────┬─────┘ │
│       │       │  │       │       │  │       │       │
│ ┌─────▼─────┐ │  │ ┌─────▼─────┐ │  │ ┌─────▼─────┐ │
│ │ Local GPU │ │  │ │ Local GPU │ │  │ │ Local GPU │ │
│ │  Blocks   │ │  │ │  Blocks   │ │  │ │  Blocks   │ │
│ └───────────┘ │  │ └───────────┘ │  │ └───────────┘ │
└───────────────┘  └───────────────┘  └───────────────┘
       ▲                  ▲                  ▲
       └──────────────────┴──────────────────┘
                    P2P Block Transfer
```

**数据流**：
1. **Block分配**：从全局空闲池分配Virtual Block ID，映射到本地物理Block
2. **COW共享**：相同Prefix的Block标记为只读共享，RefCount>1时写触发COW
3. **Block迁移**：实例A的Block被实例B请求时，通过P2P直接传输或经全局目录中转
4. **版本管理**：COW产生的多版本Block通过版本链管理，支持时间旅行回滚

**关键组件**：
- **Global Block Directory**：集中式元数据服务，维护Virtual→Physical映射、RefCount、版本链
- **Local Block Allocator**：每个实例内的物理Block管理，对接vLLM Block Allocator
- **P2P Transfer Engine**：GPU Direct RDMA或NVLink P2P，Block级细粒度传输
- **COW Handler**：写时复制触发、新Block分配、旧版本回收

### 优点

1. **细粒度共享**：Block级（通常128-256 tokens）比Chunk级更精细，Prefix匹配率更高
2. **零拷贝COW**：只读共享避免数据复制，写操作仅复制被修改的Block
3. **与vLLM生态兼容**：基于现有Block Table扩展，迁移成本低

### 缺点

1. **元数据开销**：全局Block Directory成为潜在瓶颈，大规模下需分片
2. **COW雪崩风险**：长序列频繁修改触发连锁COW，内存膨胀不可控
3. **碎片整理复杂**：Block粒度细，跨实例碎片化后难以有效合并

### 适用场景边界

- **适合**：已使用vLLM的集群、长序列多轮对话（COW收益大）、Block级热点明显的workload
- **不适合**：超短序列（Block管理overhead占比高）、写密集型workload（COW频繁触发）

### 关键Trade-off识别

| Trade-off | 方案A | 方案B | 取舍分析 | 推荐选择 |
|-----------|-------|-------|----------|----------|
| **性能 vs 成本** | 分层存储 | 分离式 | 分层存储成本低但性能受限，分离式性能好但成本高 | 分阶段：初期分层，规模扩大后分离式 |
| **延迟 vs 可用性** | 本地优先 | 网络优先 | 本地缓冲保证基础服务，网络故障不中断 | 本地缓冲为基础，网络为扩展 |
| **简单性 vs 扩展性** | 分层存储 | 分离式 | 分层存储简单但扩展性受限，分离式复杂但扩展性好 | 可演进架构，预留扩展接口 |
| **一致性 vs 性能** | 最终一致 | 最终一致 | KV Cache可重算，允许短暂不一致 | 最终一致 |

### 决策结论

**推荐架构：混合演进架构（方案A为基础，预留方案B扩展能力）**

| 阶段 | 架构 | 关键技术 | 目标规模 |
|------|------|----------|----------|
| Phase 1 | 分层存储 | GPU→SSD→Remote，跳层直达 | 1-10节点 |
| Phase 2 | 分层+分离混合 | 引入RDMA，远端DRAM池 | 10-50节点 |
| Phase 3 | 完全分离式（可选） | Conductor全局调度 | 50+节点 |

**核心设计原则**：
1. **分层优先**：本地SSD作为主要Offloading层，成本最优
2. **跳层优化**：热数据直接晋升到HBM，冷数据直接淘汰到SSD，跳过中间层
3. **异步预取**：Pipeline隐藏传输延迟，计算与传输重叠
4. **预留扩展**：Connector抽象屏蔽底层差异，支持RDMA后端无缝接入

### 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| SSD带宽成为瓶颈 | 中 | 高 | 监控SSD利用率，>70%触发扩容或引入远程层 |
| 分离式扩展改造成本高 | 低 | 中 | 初期设计预留RDMA接口，Connector抽象屏蔽差异 |
| 多方案并存增加复杂度 | 中 | 中 | 清晰的阶段划分，每阶段明确技术栈 |
| 团队技能不足 | 低 | 高 | 分阶段能力建设，先掌握分层再学习RDMA |

### 关键假设（需验证）

1. **假设1**：目标workload的本地命中率>70%（验证方法：原型测试1周生产数据）
2. **假设2**：SSD延迟在可接受范围（验证方法：P99 TTFT < 100ms压力测试）
3. **假设3**：团队能在6个月内掌握RDMA技术（验证方法：第4个月进行技术评估）

| 决策因素 | 倾向方案 |
|---------|---------|
| 预算优先、已有SSD资源 | A |
| 大规模生产、长期演进 | B |
| vLLM生态、快速落地 | C |
| 延迟敏感、实时交互 | 需A+C混合或定制 |
| 多租户隔离要求高 | B或C |
