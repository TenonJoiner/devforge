# ADR-002: 存储引擎设计决策

- **状态**: 草稿
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（70-80%）

---

## 背景

本系统为面向 AI 推理场景的 KV Cache 分布式存储系统，运行在 4~16 节点裸机 GPU 集群，全 NVMe SSD 存储介质，RDMA 网络。用户明确约束：

1. 最底层必须用 SPDK 访问 NVMe SSD
2. 不依赖本地文件系统（ZFS/Lustre/Btrfs 等）
3. 直接基于裸盘管理

### 问题定义

KV Cache 场景对存储引擎提出了与传统通用存储截然不同的需求：

| 特征 | KV Cache 场景 | 传统通用存储 | 对存储引擎的启示 |
|------|--------------|-------------|-----------------|
| **写模式** | 大块顺序写（Prefill 阶段一次性写入完整上下文） | 随机小写、原地更新 | 适合日志结构（Log-Structured） |
| **读模式** | 随机小块读（Decode 阶段逐层/逐 token 加载） | 顺序大读、全文件读取 | 需要高效的小块随机读索引 |
| **数据生命周期** | 短（分钟~小时），命中后高频复用，最终淘汰 | 长（月~年），持续累积 | 无需复杂 GC，但需要快速分配/释放 |
| **一致性要求** | 写入后立即可读，无并发修改 | 强一致、事务隔离 | 简化并发模型（WORM 语义） |
| **数据局部性** | 强（同请求的多层 KV Cache 一起访问） | 弱 | 适合按请求聚合存储 |
| **容错要求** | 可重算（丢失后重新 Prefill），但重计算成本高 | 不可丢失 | 副本/纠删码可用，但非极端可靠 |

核心问题：**如何在 SPDK 裸盘之上，构建一个针对 KV Cache 写后读模式优化的存储引擎，实现低延迟（P99 < 10ms）、高吞吐（饱和 NVMe 带宽）、低空间放大、低写放大的数据持久化层？**

## 候选方案

### 方案 A：SPDK Blobstore + 自研 KV Cache 对象层

**核心思路**：基于 SPDK 提供的 Blobstore（bs）层管理裸盘空间分配和持久化，上层自研 KV Cache 语义的对象层，将 KV Cache chunk 映射为 blob，利用 Blobstore 的事务和快照能力保证一致性。

**参考来源**：SPDK Blobstore（Intel 开源）+ Ceph BlueStore 裸盘管理思想

**架构草图**：

```
┌─────────────────────────────────────────┐
│         KV Cache Object Layer            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Chunk   │ │ Request │ │ Metadata│   │
│  │ Index   │ │ Router  │ │ Manager │   │
│  └────┬────┘ └─────────┘ └────┬────┘   │
│       │                       │         │
│       └───────────┬───────────┘         │
│                   ▼                     │
│  ┌─────────────────────────────────┐   │
│  │  Object → Blob Mapping          │   │
│  │  (1 KV Cache = 1+ blobs)        │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│         SPDK Blobstore Layer             │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Blob    │ │ Cluster │ │ Metadata│   │
│  │ Manager │ │ Alloc   │ │ Page    │   │
│  └─────────┘ └─────────┘ └─────────┘   │
│  事务支持、快照、写时复制、GC              │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│         SPDK Bdev / NVMe Driver          │
│  用户态 NVMe 驱动，轮询模式，零拷贝        │
└─────────────────────────────────────────┘
```

**优点**：
1. **裸盘管理由 Blobstore 负责**：无需自行实现块分配、磨损均衡、坏块管理，降低实现复杂度
2. **事务和快照原生支持**：Blobstore 提供原子写和快照能力，简化一致性保证
3. **SPDK 生态成熟**：Blobstore 经过验证，社区活跃，文档完善
4. **WORM 语义天然适配**：KV Cache 写入后只读，与 Blobstore 的写时复制（COW）语义一致
5. **GC 由 Blobstore 处理**：空间回收和碎片整理下沉到 Blobstore 层

**缺点**：
1. **Blobstore 通用性开销**：Blobstore 为通用对象存储设计，元数据页（metadata page）和集群分配粒度可能不适合 KV Cache 的小对象场景
2. **C 语言绑定**：SPDK 为 C 语言，但 Blobstore API 设计偏向底层，上层需要较多封装
3. **GC 不可控**：Blobstore 的 GC 策略不透明，可能在关键时刻触发，影响延迟稳定性
4. **单盘扩展性**：Blobstore 最初为单盘设计，多盘聚合需要自行实现 RAID 或条带化

**适用场景**：希望快速落地、利用成熟裸盘管理层、对 GC 可控性要求不极端的场景。

### 方案 B：SPDK 底层 I/O + 自研日志结构存储引擎

**核心思路**：直接使用 SPDK 的 NVMe 驱动（spdk_nvme）进行底层 I/O，自研日志结构（Log-Structured）存储引擎。数据以追加方式写入，通过索引（内存 + 持久化）定位数据，利用 KV Cache 的 WORM 特性简化 GC。

**参考来源**：WiscKey（FAST 2016）+ Seastar SeaStore 日志结构设计 + SPDK NVMe 驱动

**架构草图**：

```
┌─────────────────────────────────────────┐
│         KV Cache 存储引擎层              │
│                                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │ 索引    │  │ 写入    │  │ 读取    │ │
│  │ 管理器  │  │ 缓冲    │  │ 缓存    │ │
│  │(内存B树 │  │(DRAM   │  │(DRAM   │ │
│  │ + WAL)  │  │ 缓冲)   │  │ 缓存)   │ │
│  └────┬────┘  └────┬────┘  └─────────┘ │
│       │            │                    │
│       └────────────┼────────────────────┘
│                    ▼
│  ┌─────────────────────────────────┐   │
│  │  日志结构存储（Log-Structured）  │   │
│  │  - 追加写（Append-Only）         │   │
│  │  - 分段（Segment）管理           │   │
│  │  - 轻量 GC（仅清理已删除段）      │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│         SPDK NVMe 驱动层                 │
│  用户态提交 I/O，轮询完成队列（CQ）        │
│  多队列（Multi-Queue）并行 I/O            │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│         NVMe SSD（裸盘）                  │
│  分段布局：                               │
│  [SuperBlock][WAL][Segment 0][Segment 1]...│
│  [Segment N][Metadata Checkpoint]         │
└─────────────────────────────────────────┘
```

**优点**：
1. **极致性能**：绕过 Blobstore 的中间层，I/O 路径最短，延迟最低
2. **KV Cache 语义原生支持**：索引、分段、GC 策略完全针对 KV Cache 场景定制
3. **GC 完全可控**：由于 KV Cache 是 WORM + 显式删除（淘汰），GC 只需清理整个 segment，无需读-修改-写，实现简单
4. **写放大极低**：追加写无原地更新，写放大接近 1x（不考虑元数据）
5. **多盘聚合灵活**：可自行实现条带化或 RAID，不受 Blobstore 限制

**缺点**：
1. **实现复杂度高**：需自行实现块分配、坏块管理、崩溃恢复、元数据持久化等
2. **开发周期长**：核心存储引擎从零构建，预计 4-6 个月才能达到生产可用
3. **测试验证成本高**：需要大量故障注入测试验证数据完整性
4. **人才依赖**：需要深入理解 NVMe 协议、日志结构存储、SPDK 编程的工程师

**适用场景**：追求极致性能、团队有存储引擎开发经验、可接受较长开发周期的场景。

### 方案 C：SPDK 底层 I/O + 简化 LSM-Tree（键值分离）

**核心思路**：基于 SPDK NVMe 驱动，采用简化版 LSM-Tree 作为存储结构，但进行键值分离（Key-Value Separation）——小 key 存储在 LSM-Tree 中，大 value（KV Cache 数据）以追加方式写入独立的数据文件。参考 WiscKey 的设计思想。

**参考来源**：WiscKey（FAST 2016）+ RocksDB（LevelDB 架构）+ SPDK NVMe 驱动

**架构草图**：

```
┌─────────────────────────────────────────┐
│         KV Cache 存储引擎层              │
│                                          │
│  ┌─────────┐  ┌─────────────────────┐  │
│  │ LSM-Tree│  │  Value Log (vLog)   │  │
│  │ (Keys + │  │  (Append-Only       │  │
│  │  Value  │  │   Large Values)     │  │
│  │  Ptrs)  │  │                     │  │
│  │         │  │  [vLog 0][vLog 1]...│  │
│  │ MemTable│  │                     │  │
│  │ SSTable │  │  GC: 扫描 key 引用  │  │
│  │ Level 0 │  │  清理未引用 value   │  │
│  │ Level 1 │  │                     │  │
│  │ ...     │  │                     │  │
│  └────┬────┘  └─────────────────────┘  │
│       │                                 │
│       └────────────┬────────────────────┘
│                    ▼
│  ┌─────────────────────────────────┐   │
│  │  SPDK Blob / NVMe 驱动          │   │
│  │  （块设备抽象或直接 NVMe）        │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**优点**：
1. **键值分离降低写放大**：LSM-Tree 仅存储 key 和 value 指针，compaction 时无需移动大 value
2. **范围查询支持**：LSM-Tree 天然支持有序遍历，适合前缀扫描（Prefix Cache 场景）
3. **成熟理论基础**：LSM-Tree 经过大量生产验证（RocksDB、LevelDB、Cassandra）
4. **崩溃恢复简单**：WAL + checkpoint 机制成熟

**缺点**：
1. **Compaction 开销**：即使键值分离，LSM-Tree 的 compaction 仍引入不可预测的延迟抖动
2. **读放大**：读取需要先在 LSM-Tree 中查找 key，再读取 vLog，两次 I/O
3. **GC 复杂**：vLog 的 GC 需要扫描 LSM-Tree 确认 value 是否仍被引用，实现复杂
4. **不适合 WORM**：LSM-Tree 为频繁更新设计，KV Cache 的 WORM 语义无法充分利用其优势

**适用场景**：需要范围查询、key 更新频繁、可接受 compaction 开销的场景。KV Cache 场景下 WORM 语义使 LSM-Tree 的优势无法发挥。

### 方案 D：SPDK 底层 I/O + 扁平对象存储（Flat Object Store）

**核心思路**：放弃树形或日志结构，采用最简单的扁平对象存储——每个 KV Cache chunk 作为一个独立对象，对象直接追加写入磁盘，通过内存哈希表索引对象位置。对象不修改、不覆盖，删除时仅标记并异步回收空间。

**参考来源**：MinIO 对象存储底层 + InfiniStore 内存管理思想（bitmap 分配器适配到磁盘）+ Flat File System 设计

**架构草图**：

```
┌─────────────────────────────────────────┐
│         扁平对象存储引擎                 │
│                                          │
│  ┌─────────────────────────────────┐   │
│  │  内存索引（哈希表）               │   │
│  │  object_id → (offset, length,   │   │
│  │              checksum, status)  │   │
│  └─────────────────────────────────┘   │
│                   │                      │
│                   ▼                      │
│  ┌─────────────────────────────────┐   │
│  │  对象分配器（Extent-based）      │   │
│  │  - 大对象：连续 extent 分配      │   │
│  │  - 小对象：slab 合并存储         │   │
│  └─────────────────────────────────┘   │
│                   │                      │
│                   ▼                      │
│  ┌─────────────────────────────────┐   │
│  │  裸盘布局                        │   │
│  │  [SuperBlock][Index WAL]        │   │
│  │  [Object Data Region]           │   │
│  │  [Free Space Bitmap]            │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│         SPDK NVMe 驱动层                 │
└─────────────────────────────────────────┘
```

**优点**：
1. **实现最简单**：无树结构、无 compaction、无复杂 GC，代码量最小
2. **读性能最优**：一次哈希查找 + 一次直接 I/O，无中间层
3. **延迟最稳定**：无后台 compaction/GC，延迟可预测
4. **WORM 语义完美适配**：追加写入、不修改、标记删除，与 KV Cache 生命周期完全匹配

**缺点**：
1. **内存索引容量受限**：哈希表需常驻内存，大规模数据下内存消耗大
2. **空间回收效率低**：标记删除后空间回收需扫描或延迟整理，可能产生碎片
3. **无范围查询能力**：哈希表不支持有序遍历，前缀扫描需额外索引层
4. **扩展性受限**：单盘容量和内存索引规模成为瓶颈

**适用场景**：数据规模可控（单节点 TB 级）、追求极致简单和稳定延迟、无范围查询需求的场景。

## 方案量化对比矩阵

| 维度 | 方案 A：Blobstore + 对象层 | 方案 B：自研日志结构 | 方案 C：LSM-Tree 键值分离 | 方案 D：扁平对象存储 |
|------|---------------------------|---------------------|-------------------------|-------------------|
| **参考来源** | SPDK Blobstore + BlueStore | WiscKey + SeaStore | WiscKey + RocksDB | MinIO + InfiniStore |
| **P99 写延迟** | ~2ms（Blobstore 事务） | ~1ms（直接追加） | ~3ms（含 WAL + MemTable） | ~1ms（直接追加） |

| **顺序写吞吐** | 高（接近裸盘） | 极高（接近裸盘） | 中高（compaction 消耗） | 极高（接近裸盘） |
| **随机读吞吐** | 中高（blob 元数据开销） | 高（索引在内存） | 中（两次 I/O） | 高（哈希索引） |
| **写放大** | 低（COW） | 极低（~1x） | 低（键值分离后） | 极低（~1x） |
| **空间放大** | 中（COW 快照开销） | 低（仅存活数据） | 中（LSM 层级冗余） | 低~中（碎片） |
| **实现复杂度** | 中（利用 Blobstore） | 高（自研引擎） | 中高（LSM-Tree） | 低（简单哈希） |
| **与 SPDK 集成度** | 高（原生集成） | 高（直接驱动） | 高（直接驱动） | 高（直接驱动） |
| **崩溃恢复** | 中（依赖 Blobstore） | 中（WAL + checkpoint） | 中（WAL + SST） | 简单（WAL + 重放） |
| **GC 可控性** | 低（Blobstore 内部） | 高（完全自研） | 中（vLog GC） | 高（标记删除 + 异步回收） |
| **多盘聚合** | 需自行实现 | 灵活自研 | 灵活自研 | 需自行实现 |
| **内存消耗** | 中（Blobstore 元数据） | 中（索引 + 缓冲） | 高（MemTable + 块缓存） | 高（全内存哈希索引） |
| **开发周期** | 2-3 月 | 4-6 月 | 3-4 月 | 1-2 月 |

> **评分说明**：延迟基于 NVMe SSD（P99 读 ~100us，写 ~10us）+ SPDK 用户态驱动的假设估算。写放大 = 实际写入量 / 用户数据量。

## 关键 Trade-off

### 1. 开发速度 vs 性能极致

- **方案 A（Blobstore）**：利用成熟组件，2-3 月可出 MVP，但性能受 Blobstore 通用设计限制
- **方案 B（自研日志结构）**：性能最优，但开发周期 4-6 月，且需要存储引擎专家
- **判断**：4~16 节点规模下，Blobstore 的性能损耗（~1-2ms）是否可接受？需原型验证。

### 2. GC 可控性 vs 实现复杂度

- **KV Cache 的 WORM 语义**：写入后只读，删除是显式的（淘汰），这使得 GC 可以非常简单——只需回收整个 segment/object 的空间
- **Blobstore 的 GC**：内部实现，不透明，可能在关键时刻触发
- **自研 GC**：完全可控，可配合负载特征调度（低峰期回收）
- **判断**：KV Cache 场景下 GC 频率低（数据生命周期明确），Blobstore 的通用 GC 可能过度设计。

### 3. 索引结构：哈希 vs 树形

- **哈希（方案 D）**：O(1) 查找，但不支持范围查询。KV Cache 的访问模式是精确查找（通过 object_id），无需范围查询
- **树形（方案 C LSM-Tree）**：支持范围查询，但 KV Cache 场景不需要
- **判断**：KV Cache 的访问模式（精确 object_id 查找）使哈希索引成为最优选择，树形结构引入不必要的开销。

### 4. 元数据持久化：全内存 vs 分层

- **全内存索引（方案 D）**：重启后需从磁盘重建，恢复时间长（与数据量成正比）
- **分层索引（方案 A/B/C）**：checkpoint + WAL，恢复时间短
- **判断**：4~16 节点、TB 级数据规模下，全内存索引的恢复时间可能达分钟级，需评估可接受性。

### 5. 裸盘管理：自研 vs 复用

- **自研（方案 B/D）**：完全控制，但需处理坏块、磨损均衡、多盘聚合等复杂问题
- **复用 Blobstore（方案 A）**：这些能力已内置，但增加一层抽象
- **判断**：团队是否有裸盘管理经验？如无，Blobstore 可显著降低风险。

## 决策

**初步倾向：方案 B（SPDK 底层 I/O + 自研日志结构存储引擎）的变体**

选择理由：

1. **KV Cache 场景与日志结构天然匹配**：WORM 语义 + 追加写 + 显式删除，使日志结构的实现大幅简化（无需复杂 GC）
2. **性能天花板最高**：直接驱动 NVMe，I/O 路径最短，可充分饱和 NVMe 带宽
3. **GC 完全可控**：可配合 KV Cache 的生命周期特征设计 GC 策略（如低负载时批量回收）
4. **与整体架构方向一致**：decision-overall.md 倾向"混合 PD 原生"架构，存储层需要足够灵活以支持 PD 语义优化

**变体调整**（降低风险）：
- **Phase 1**：采用方案 A（Blobstore）快速验证端到端可行性（2-3 月）
- **Phase 2**：在 Blobstore 基础上验证性能瓶颈，若 Blobstore 成为瓶颈，迁移到方案 B
- **Phase 3**：方案 B 成熟后，Blobstore 作为备选/降级路径保留

## 替代方案排除理由

- **方案 A（Blobstore）**：未完全排除，作为 Phase 1 保底方案。长期看 Blobstore 的通用 GC 和元数据开销可能成为性能瓶颈。
- **方案 C（LSM-Tree 键值分离）**：KV Cache 场景不需要范围查询和频繁更新，LSM-Tree 的 compaction 引入不必要的延迟抖动和实现复杂度。WORM 语义下 LSM-Tree 的优势无法发挥。
- **方案 D（扁平对象存储）**：全内存索引在 10M+ 对象规模下内存消耗过大（假设每个索引项 64 字节，10M 对象需 640MB，可接受；但 100M 对象需 6.4GB，接近极限）。且重启恢复时间长，不适合生产环境。
