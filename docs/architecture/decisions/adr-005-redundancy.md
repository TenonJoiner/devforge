# ADR-005: 数据冗余策略

- **状态**: 草稿
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（76%）

## 背景

KV Cache 存储系统的数据冗余策略需要解决以下核心问题：

1. **数据耐久性**：单盘/单节点故障时，KV Cache 数据不丢失
2. **重建性能**：故障后重建数据的速度（MTTR 和重建风暴风险）
3. **存储效率**：冗余开销（多副本 2x vs 纠删码 1.2x~1.5x）
4. **读写性能影响**：冗余策略对正常 I/O 路径的影响

**场景约束**：
- 数据特征：KV Cache 数据块大（32MB~100MB+），写入后只读，生命周期从分钟到小时/天
- 容量目标：1TB~100TB 的 KV Cache 数据
- 成本敏感：用户以"低成本外部存储替代 HBM"为核心价值主张
- 硬件：全 NVMe SSD，4~16 节点
- 数据性质：**可重算但代价高**（与数据库的不可丢失数据不同）

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|----------|
| **VAST Data** | 宽条带 LDC（146+4，2.7% 开销） | 重建仅需读 1/4 幸存数据，但条带宽至 146，小集群（4~16 节点）完全不可用 |
| **Ceph** | 多副本（默认 3x）+ 纠删码（可选 k+m） | 默认 3 副本成熟稳定；纠删码 k+m 在小集群下不灵活（k+m=6+2 时最小需 8 节点） |
| **MoonCake** | `ReplicateConfig`（默认 1，尽力而为复制） | 作为缓存层，冗余为尽力而为，可丢失后重新计算；与本系统定位不同 |
| **MinIO** | Reed-Solomon 纠删码（4+2 或 8+4） | 对象存储纠删码参考，默认 50% 开销；小文件需要对象级条带化 |
| **Facebook HDFS RAID** | 热数据多副本 + 冷数据纠删码 | 混合策略的工业级实践，按数据年龄动态迁移 |

## 候选方案

| 方案 | 参考来源 | 优点 | 缺点 | 置信度 |
|------|----------|------|------|--------|
| **A: 3 副本** | Ceph/HDFS/MoonCake | 1. 实现最简单（无编码/解码）<br>2. 读性能最好（任意副本读取，负载均衡）<br>3. 重建最快（直接拷贝）<br>4. 小文件友好（无需条带对齐） | 1. 存储开销最高（200%，3x）<br>2. 写放大（写入需复制 3 份）<br>3. 跨节点写入延迟（同步复制） | 83% |
| **B: Reed-Solomon 纠删码（k+m）** | Ceph/MinIO | 1. 存储效率最高（冗余开销 = m/k，如 4+2 仅 50%）<br>2. 可配置容错（m=2 容忍 2 节点故障） | 1. 重建读取量大（需读 k 个幸存块）<br>2. 单节点故障时读性能下降（需解码）<br>3. 实现复杂（Galois Field 编解码）<br>4. 条带对齐要求（小对象空间浪费）<br>5. 最小集群规模要求（k+m=4+2 需 ≥6 节点） | 70% |
| **C: 混合策略（热 3 副本 + 冷纠删码）** | Facebook HDFS RAID / VAST | 1. 热数据享受多副本性能<br>2. 冷数据享受纠删码效率<br>3. 渐进式迁移，前台 I/O 无影响<br>4. 成本最优（介于 3x 和 1.5x 之间） | 1. 实现最复杂（两套存储路径 + 迁移逻辑）<br>2. 迁移开销（后台消耗网络/磁盘带宽）<br>3. 冷热判定阈值等参数需调优 | 72% |
| **D: 局部可修复码 LRC** | VAST LDC / Azure LRC | 1. 重建性能接近多副本（仅读局部组内数据）<br>2. 存储效率接近 RS 码（开销略高于 RS，远低于 3 副本） | 1. 实现复杂度高（LRC 编码/解码算法比 RS 更复杂）<br>2. 参数调优困难<br>3. **小集群不适用**：LRC 优势在 100+ 盘/大规模集群才体现，4~16 节点下优势有限 | 63% |

## 关键 Trade-off

### T1：存储成本 vs 重建性能（核心取舍）

| 维度 | 方案 A（3 副本） | 方案 B（RS 4+2） |
|------|------------------|-------------------|
| 存储开销 | 200% | 50% |
| 可容忍故障 | 2 节点 | 2 节点 |
| 重建读取量 | 1x（单副本拷贝） | 4x（读取 4 个幸存数据块） |
| 重建计算量 | 0 | 高（Galois Field 解码） |
| 重建时间（1TB，10Gbps） | ~15 分钟 | ~60 分钟（含解码） |
| 写性能 | 中（3x 复制） | 较好（并行条带写入） |
| 读性能（故障时） | 降级（直接读幸存副本） | 降级严重（需解码） |

**量化决策点**：
- 存储 10TB 有效数据时，3 副本需 30TB 物理容量，纠删码仅需 15TB
- 在 10Gbps 网络下，1TB 数据重建时间差距：3 副本 ~15 分钟 vs 纠删码 ~60 分钟
- **关键判断**：用户以"低成本"为核心价值 → 存储效率重要，但重建性能同样不可忽视

### T2：KV Cache 数据性质的特殊性

KV Cache 数据是"**可重算但代价高**"的数据：
- 与通用数据库（不可丢失）不同：KV Cache 丢失后可重新 Prefill 计算
- 与纯缓存（LMCache，无冗余）不同：重算代价高（GPU 算力、时间、TTFT 增加）
- **影响**：允许比通用数据库更激进的存储效率优化，但不允许像纯缓存那样完全放弃冗余

**量化**：
- 重算 1TB KV Cache 估算：LLaMA3-70B @ H800，假设 50% 缓存命中率，约需 30-60 分钟 GPU 算力
- 存储 1TB 数据 vs 重算 1TB 数据的成本比：SSD 存储 ~$0.03/GB/月 vs GPU 算力 ~$2-3/GB（一次性）

**结论**：存储成本远低于重算成本 → **应优先保证数据不丢失（冗余必须），但可在可靠性和效率间权衡**

### T3：小集群（4~16 节点）下的纠删码有效性

| 节点数 | 3 副本适用性 | RS 纠删码适用性 | LRC 适用性 |
|--------|-------------|----------------|-----------|
| 4 节点 | 最佳（每副本一个节点） | ❌ 不适用（需 ≥6 节点） | ❌ 不适用（优化效果差） |
| 6 节点 | 良好（可跨 NUMA 域放置） | 可用（4+2 配置） | ⚠️ 勉强可用（局部优势不明显） |
| 8~16 节点 | 良好 | 良好 | ⚠️ 可用（但 4+2 纠删码已足够好） |

**关键洞察**：VAST 的 146+4 LDC（2.7% 开销）在 4~16 节点下**完全不可用**，因为需要超大规模集群才能发挥宽条带优势。小集群应采用窄条带配置（4+2 或 2+1），但 2+1 配置仅容忍 1 个故障，与 3 副本的 2 个故障容忍度不对等。

### T4：MVP 阶段 vs 演进阶段的复杂度权衡

| 阶段 | 推荐方案 | 理由 |
|------|---------|------|
| MVP（v1.0） | 3 副本 | 简单可靠，正确性优先，快速交付 |
| 演进（v2.0） | 混合策略 | 热数据 3 副本 + 冷数据纠删码，按需升级 |

## 决策

选择 **方案 C（混合策略）的分阶段实现**，具体为：

### 阶段一（v1.0 MVP）：3 副本起步

**立即执行**：
- 所有数据默认 3 副本
- 异步复制：写入先确认到本地节点，后台异步复制到 2 个副本节点
- 副本放置：跨节点分布，故障域隔离（跨 NUMA 域 or 跨机架）
- 最小集群：3 节点（每副本在不同节点）

**量化指标**：
- 存储开销：200%
- 可容忍故障：任意 2 节点
- 写确认延迟：本地写入 + 2 个副本确认（异步模式下为本地写入即返回）
- 重建时间：~15 分钟/TB（10Gbps 网络）

### 阶段二（v2.0 演进）：混合策略

**触发条件**：节点数 ≥ 6 且存储利用率超过 60%（基于监控数据）

**执行方案**：
- **热数据**（最近 24h 写入 或 访问频率 > 10次/天）：保持 3 副本
- **温数据**（7d 内写入 且 访问频率 1-10次/天）：迁移到 2+1 纠删码
- **冷数据**（>7d 未写入）：保持 2+1 纠删码

**量化指标**：
- 有效存储开销：~120%（假设 30% 热 + 70% 冷）
- 可容忍故障：热数据 2 节点，温/冷数据 1 节点
- 迁移时机：后台低峰期（02:00-06:00），限速 30% 带宽

**演进触发条件**：
- 节点数达到 6 个且存储利用率超过 60%
- 或用户明确要求降低存储成本

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 3 副本存储成本超出用户预算 | 中 | 高 | 1. 提供成本计算器工具<br>2. 明确演进到混合策略的路径<br>3. 允许用户配置副本数（2 或 3） |
| 重建风暴影响正常 I/O | 中 | 高 | 1. 重建限速：总带宽 30%<br>2. 优先级调度：前台 I/O 优先<br>3. 监控告警：重建流量 >20% 时告警 |
| 纠删码实现 bug 导致数据丢失 | 低 | 极高 | 1. 使用成熟库（ISA-L）<br>2. 写时计算校验和，读时验证<br>3. 充分测试：故障注入验证 |
| 混合策略迁移逻辑 bug | 中 | 高 | 1. 迁移操作原子化（两阶段提交）<br>2. 迁移前后数据校验<br>3. 回滚机制（保留旧副本直到新副本确认） |
| 异步复制期间数据丢失 | 低 | 高 | 1. 写入确认前确保本地持久化（fsync）<br>2. 副本丢失时触发重同步<br>3. 定期对账（checksum 校验） |

## 待验证假设

1. **H1：存储成本 vs 重算成本的比值**：假设存储 $0.03/GB/月 vs 重算 $2/GB，需验证用户场景下此比值是否成立。若重算成本接近存储成本，则 3 副本的存储开销更不可接受。**验证方法**：与目标用户确认实际 GPU 算力成本和存储预算。
2. **H2：重建限速 30% 不影响前台 I/O**：假设 10Gbps 网络下，重建占用 3Gbps 不会影响正常读写。**验证方法**：混合负载压测（正常 I/O 70% + 重建 I/O 30%），测量延迟 P99。
3. **H3：冷热数据分布符合预期**：假设 30% 数据为热数据（7d 内写入），70% 为冷数据。**验证方法**：基于目标用户的实际 traces 分析数据年龄分布。
4. **H4：2+1 纠删码在小集群的容错性足够**：6 节点下，2+1 配置仅容忍 1 个节点故障，是否满足用户的 SLA？**验证方法**：与用户确认可用性要求。

## 替代方案排除理由

- **方案 B（纯 RS 纠删码）**：排除。4+2 配置需要 6 节点，在 4~5 节点下不适用；且单节点故障时读性能下降严重，与 KV Cache 的低延迟目标冲突。
- **方案 D（LRC）**：排除。小集群（4~16 节点）下 LRC 的局部修复优势无法体现；实现复杂度高；参数调优困难。在 100+ 节点规模才值得考虑。
- **方案 A（纯 3 副本，不演进）**：部分排除。M VP 阶段采用 3 副本是正确的，但不演进到混合策略会导致长期存储成本过高。决策采用分阶段策略，方案 A 作为起点而非终点。
