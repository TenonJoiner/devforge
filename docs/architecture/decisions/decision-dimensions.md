## 概述

基于 4 个标杆研究（LMCache / MoonCake / NVIDIA CMX / vLLM PagedAttention）和整体架构发散（见 [exploration-kvcache-offloading.md](exploration-kvcache-offloading.md)），识别出 **9 个关键技术维度** 需要独立探索。

**识别标准**：
- 标杆研究中出现分歧的技术点
- 对系统性能/可靠性/成本影响显著
- 存在多种成熟方案可选

---

## 维度 1：缓存索引与查找机制

**探索问题**：如何快速定位一个 token 序列对应的 KV 缓存块？在 prefix 匹配、部分命中、前缀树维护成本之间如何取舍？

**为何独立探索**：标杆分歧明显，且直接决定命中率和查找延迟（P99 路径上的关键开销）。

**标杆对比**：
- **LMCache**：Chunk-based + hash 索引，chunk 粒度 prefix 匹配
- **MoonCake**：全局 Conductor 维护元数据，集中式查找
- **vLLM PagedAttention**：Block Table 按请求维护，无全局 prefix 索引
- **NVIDIA CMX**：硬件侧分层索引，细粒度管理

**候选方向**：
1. Radix Tree / Trie 前缀树（SGLang RadixAttention 风格）
2. Chunk-hash + 扁平哈希表（LMCache 风格）
3. 全局元数据服务 + 本地缓存（MoonCake Conductor 风格）
4. 分布式一致性哈希索引（无中心）

---

## 维度 2：分层存储介质组织

**探索问题**：几层？每层容量配比？数据在层间如何流动（pull/push/prefetch）？冷热判定策略？

**为何独立探索**：不同标杆分层策略差异巨大，直接影响总拥有成本与命中延迟分布。

**标杆对比**：
- **LMCache**：HBM → DRAM → Disk 三层，显式 put/get
- **NVIDIA CMX**：硬件加速的 HBM ↔ DRAM 快速通路
- **MoonCake**：分离式，远端 DRAM 池为主
- **vLLM**：仅 HBM + CPU swap

**候选方向**：
1. 严格分层 LRU（逐层降级）
2. 跳层直达（热数据可跨层直接晋升/淘汰）
3. 分离式远端池（本地仅 HBM，其余远端聚合）
4. Tiering + 异步预取（基于访问预测）

---

## 维度 3：Cache Block 粒度与对齐

**探索问题**：以多大粒度组织 KV 块？固定 block size 还是可变 chunk？粒度如何影响碎片、传输效率、命中率？

**为何独立探索**：粒度是核心取舍点——粗粒度命中率低但元数据开销小，细粒度相反。

**标杆对比**：
- **vLLM**：固定 16 tokens/block
- **LMCache**：可配置 chunk（256 tokens 常见）
- **MoonCake**：page 粒度分配
- **CMX**：硬件亲和粒度

**候选方向**：
1. 固定小 block（16/32 tokens，对齐 PagedAttention）
2. 固定大 chunk（256/512 tokens，优化传输）
3. 分层异构粒度（HBM 小块，SSD 大块）
4. 可变长 chunk（按语义边界切分）

---

## 维度 4：数据传输与 IO 路径

**探索问题**：KV 在 GPU ↔ CPU ↔ SSD ↔ 网络间如何搬运？是否支持零拷贝、GPUDirect、RDMA？同步还是异步？

**为何独立探索**：传输带宽常是瓶颈，不同路径性能差 10x 以上。

**标杆对比**：
- **MoonCake**：RDMA + GPUDirect 为核心卖点
- **NVIDIA CMX**：NVLink/NVSwitch 硬件通路
- **LMCache**：CUDA memcpy + 普通文件 IO
- **vLLM**：cudaMemcpyAsync swap

**候选方向**：
1. 传统 cudaMemcpy + pread/pwrite
2. GPUDirect Storage（GPU 直访 NVMe）
3. RDMA + GPUDirect RDMA（远端池）
4. 批量流水线（overlap 计算与传输）

---

## 维度 5：淘汰与准入策略

**探索问题**：容量不足时淘汰谁？是否所有数据都应准入缓存？如何避免缓存污染？

**为何独立探索**：LLM 场景的访问模式与传统缓存差异大（prefix 重用、会话局部性），需专门设计。

**标杆对比**：
- **LMCache**：LRU + 引用计数
- **vLLM**：引用计数 + COW，不主动淘汰
- **MoonCake**：调度器全局决策
- **学术界**：LRU-K、LFU、ARC 等多种

**候选方向**：
1. 简单 LRU + pin
2. 基于热度评分（访问频率 + 最近性 + 前缀长度）
3. 成本感知淘汰（重算成本 vs 存储成本）
4. 全局调度器驱动（非本地决策）

---

## 维度 6：一致性与并发控制

**探索问题**：多请求并发读写同一 prefix 时如何保证一致性？COW 还是加锁？跨节点缓存一致性如何保证？

**为何独立探索**：分布式存储的核心问题，标杆做法差异显著。

**标杆对比**：
- **vLLM**：Block 级引用计数 + COW
- **MoonCake**：Conductor 仲裁
- **LMCache**：较弱的一致性保证
- **CMX**：硬件辅助一致性

**候选方向**：
1. 引用计数 + COW（vLLM 风格）
2. 不可变块 + 版本号
3. 分布式锁 + 租约
4. 最终一致（接受短暂不一致换性能）

---

## 维度 7：元数据管理架构

**探索问题**：元数据（block 位置、引用计数、LRU 信息）集中还是分布式？持久化还是纯内存？如何应对故障恢复？

**为何独立探索**：元数据架构决定可扩展性上限和故障域大小。

**标杆对比**：
- **MoonCake**：集中式 Conductor
- **LMCache**：本地元数据
- **vLLM**：进程内
- **分布式 KV 系统**：通常用 etcd / Raft

**候选方向**：
1. 本地内存元数据（单机）
2. 集中式元数据服务（Conductor）
3. Raft 复制的元数据组
4. 一致性哈希 + gossip（去中心）

---

## 维度 8：与推理引擎的集成接口

**探索问题**：以何种形态与 vLLM/SGLang/TRT-LLM 集成？plugin？sidecar？patch？接口抽象层次？

**为何独立探索**：决定产品的可采用性和生态兼容性，是产品成败关键。

**标杆对比**：
- **LMCache**：vLLM plugin + 独立进程
- **MoonCake**：深度 patch vLLM
- **NVIDIA CMX**：闭源内建
- **通用方案**：KV Connector 接口

**候选方向**：
1. 引擎内嵌 plugin（侵入式，性能最优）
2. 独立进程 + IPC / 共享内存
3. Sidecar + 标准 KV Connector 协议
4. Library 模式（被引擎链接）

---

## 维度 9：预取与预测策略（可选，优先级低）

**探索问题**：是否基于访问模式预取？如何识别会话延续？预取命中率能否抵消带宽开销？

**为何独立探索**：标杆中很少涉及，是潜在差异化点。

**候选方向**：
1. 不做预取（按需加载）
2. 会话级预取（同 session 历史块预热到 HBM）
3. 预测模型驱动
4. 用户 hint 驱动
