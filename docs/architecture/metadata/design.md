## 1. 设计目标

### 1.1 核心使命

Metadata 子系统的唯一使命是：**在分布式环境中，让推理引擎能够在微秒到毫秒量级内，定位到任意 token 序列对应的 KV Cache 数据的逻辑位置。**

这个使命的边界非常清晰：Metadata 只回答"数据在哪里"，绝不负责"把数据搬过来"。如果缺少 Metadata，推理引擎将不得不在茫茫的分布式存储池中盲目搜索，或者更糟——放弃所有缓存复用机会，每次都重新计算 KV Cache，这将导致 TTFT（首 token 时间）暴涨数倍乃至数十倍。

### 1.2 为什么需要一个独立的 Metadata 子系统

在小型单节点部署中，索引结构通常是隐式的——比如直接嵌入在 vLLM 的 `BlockTable` 或 SGLang 的 `RadixTree` 中。但当系统扩展为分布式后，这种内嵌式索引会遇到以下根本性问题：

1. **引擎无关性**：不同推理引擎（vLLM、TensorRT-LLM、SGLang）使用各自的数据结构，紧耦合索引会导致其他引擎无法共享缓存数据。
2. **调度全局视图**：多 prefill / decode 节点场景下，调度器需要知道"哪个节点缓存了这段前缀"才能做全局最优分配。
3. **存储分层抽象**：同一段 KV Cache 可能同时存在于 GPU HBM、本地 DRAM、远程 DRAM 或 SSD。引擎应只看到统一地址空间，不感知介质迁移细节。

因此，独立的 Metadata 子系统不仅是一个索引层，更是**引擎与存储之间的定位与调度中介**。它向上为 Connector 提供统一的"token 序列 → 位置列表"映射，向下为 Storage 提供"哪些数据需要被晋升/淘汰"的元数据支撑。

### 1.3 设计约束

基于 ADR-002 的明确决策，Metadata 子系统的设计受以下关键约束：

- **L1 本地索引**：采用 Radix Tree，覆盖单节点 GPU HBM + 本地 DRAM，负责前缀匹配。
- **L2 分布式索引**：采用 Chunk-level 哈希表 + 一致性哈希环，覆盖跨节点的远程 DRAM/SSD 池，支持任意位置复用（以 chunk 为粒度）。
- **一致性模型**：最终一致，允许 100ms 的不一致窗口。这是因为 KV Cache 本身是可重算的，短暂的索引滞后不会导致数据丢失，只会导致一次 miss。
- **并发控制**：版本号 + 乐观锁。读取时不阻塞写入，写入时通过版本冲突检测避免覆盖更新。

## 2. 总体架构与模块设计

### 2.1 模块关系图

### 2.2 模块职责与协作

#### API Gateway

- **职责一句话定义**：接收来自 Connector 和内部其他子系统的查询/更新请求，进行协议适配和权限校验后，转发给 Query Router。
- **输入**：token 序列（整型数组）、操作类型（`lookup` / `insert` / `invalidate`）、请求上下文（租户 ID、优先级标签）。
- **输出**：经过校验的内部请求对象，或直接返回的错误响应（如格式非法、权限不足）。
- **与其他模块的协作方式**：只与 Query Router 单向交互。它是 Metadata 子系统对外的唯一门面，屏蔽底层索引实现细节。

#### Query Router

- **职责一句话定义**：根据查询的语义和索引分片规则，决定请求应当路由到 L1 Index Manager 还是 L2 Index Manager，或者在多层索引间做级联查询。
- **输入**：API Gateway 转发的内部请求对象、L1/L2 索引的健康状态信号。
- **输出**：路由决策（`L1_ONLY`、`L1_THEN_L2`、`L2_ONLY`），以及带有时序戳的追踪日志。
- **与其他模块的协作方式**：
  - 向上接收 API Gateway 请求；
  - 向下调用 L1/L2 Index Manager；
  - 在级联查询场景中，异步读取 L2 结果并与 L1 结果合并后传递给 Result Assembler。

#### L1 Index Manager（Radix Tree）

- **职责一句话定义**：维护单节点内存中的前缀树索引，实现 token 序列到本地 GPU/DRAM 块的高速前缀匹配。
- **输入**：token 序列、块位置句柄（包含层级别标识：HBM/DRAM）、引用计数变更指令。
- **输出**：前缀匹配结果（最长匹配长度、匹配路径上的块位置列表、未匹配后缀）。
- **与其他模块的协作方式**：
  - 被 Query Router 直接查询；
  - 当本地块被淘汰时，异步通知 Storage 子系统，并标记树中对应节点为"待降级"；
  - 定期向 Heatmap Collector 上报节点访问频率。

#### L2 Index Manager（Chunk-Hash + Consistent Hash Ring）

- **职责**：维护跨节点分布式存储的 chunk 级内容寻址索引，通过一致性哈希环定位远程节点上的 KV Cache 副本。
- **输入**：按 chunk 切分的 token 子序列、chunk hash、远程位置坐标（节点 ID + 偏移量）。
- **输出**：chunk 级别命中/未命中列表，附带远程位置及故障转移路径。
- **协作**：被 Query Router 在 L1 未命中时调用；与其他 Metadata 节点周期性同步哈希环拓扑；接收 Storage 的失效通知并异步更新本地视图。

#### Heatmap Collector

- **职责**：收集 L1/L2 访问模式元数据，为 Tiering Manager 提供热度统计和趋势预测。
- **输入**：L1/L2 访问事件流、时间窗口配置。
- **输出**：热度分位数报告、冷数据候选列表、预取建议（可选）。
- **协作**：以非阻塞订阅者模式订阅 L1/L2 事件流；向 Tiering Manager 推送报告；不在查询关键路径上。

#### Result Assembler

- **职责**：将 L1/L2 原始命中结果整合为统一的按 token 分段 `SegmentMap`。
- **输入**：L1 前缀匹配结果、L2 chunk 命中结果、冲突消解策略。
- **输出**：`SegmentMap`（每段 token 范围对应本地/远程/缺失）。
- **协作**：接收 Query Router 级联结果；L1/L2 短暂冲突时以 L1 为准；结果回传 API Gateway。

#### Versioned Metadata Store

- **职责**：为索引更新提供 WAL 和内存快照，支撑版本号 + 乐观锁语义。
- **输入**：索引变更记录、单调递增版本号。
- **输出**：最新成功提交版本号、内存快照（用于故障恢复）。
- **协作**：被 L1/L2 Index Manager 写操作时调用；定期异步刷盘到本地 SSD；故障恢复时从快照重放 WAL 重建索引。

### 2.3 为什么拆成这六个模块

模块划分为六个核心是基于三条原则：
1. **查询路径与索引结构解耦**：API Gateway、Query Router、Result Assembler 构成控制平面，与 L1/L2 具体索引实现无关，保护上层接口稳定性。
2. **查询关键路径与后台任务隔离**：Heatmap Collector 和 Versioned Metadata Store 均不在查询关键路径上，确保高频查询不被统计或持久化阻塞。
3. **L1 与 L2 语义边界清晰**：前缀树与内容哈希是两种完全不同的索引语义，拆分为独立 Index Manager 后各模块只需专注自身的生命周期管理、并发控制和故障恢复。

## 3. 核心概念与抽象

### 3.1 Token 序列的地址空间模型

Metadata 子系统将整个世界建模为一个巨大的、以 token 序列为键的地址空间。任何一个 token 序列 `[t0, t1, ..., t_{n-1}]` 都可以被映射为零个、一个或多个物理位置。这不是一个一对一的映射，因为：

1. **前缀共享**：不同请求可能共享同一个前缀，这意味着同一段 token 子序列可能对应同一个物理块。
2. **分层副本**：同一段 KV Cache 可能同时存在于本地 HBM 和远程 DRAM 上。
3. **Chunk 切分**：L2 索引不关心单个 token，只关心固定大小的 chunk（例如 256 tokens），因此 token 序列必须对齐到 chunk 边界。

### 3.2 L1 索引：压缩 Radix Tree 的抽象

L1 索引的核心数据结构是一个压缩前缀树（Compressed Radix Tree）。其高层抽象可概括为：

- **节点**：保存 token 片段（edge label）、指向子节点的映射关系、以及对物理位置的引用集合。每个节点携带一个版本号以支持乐观并发控制。
- **压缩**：single-child 节点与父节点合并，减少指针跳转和空间开销。
- **分层引用**：任意节点均可挂载多个块引用，表示该前缀在本地 HBM/DRAM 中的位置。
- **引用计数保护**：活跃请求路径上的节点引用计数递增，防止后台淘汰线程误删正在使用的块。

### 3.3 L2 索引：Chunk-Hash + 一致性哈希环的抽象

L2 索引采用内容寻址（content-addressed）思想。我们将 token 序列按 chunk 大小（如 256 tokens）切分，对每个 chunk 计算一个内容指纹（Hash）。L2 索引的本质就是一个从 `Hash` 到 `物理位置` 的映射表。

L2 索引条目的高层抽象包含：
- **内容指纹**：128 位 chunk 级内容哈希，用于内容寻址去重；
- **远程位置列表**：记录该 chunk 在分布式存储池中的物理位置（节点 ID、池内偏移、副本序号）；
- **虚拟节点位置**：该 chunk 在一致性哈希环上的逻辑映射位置，支撑节点变更时的快速重定位；
- **存活元数据**：上次确认存在的时间戳和 TTL，用于过期清理和失效判定。

**一致性哈希环的抽象**：

集群所有存储节点映射到逻辑环（0 ~ 2^32-1），每个物理节点拥有多个虚拟节点以保证负载均衡。定位算法：
1. 计算 chunk_hash 在环上的投影 P；
2. 顺时针找到第一个负载低于阈值的虚拟节点；
3. 该虚拟节点对应物理节点为首选存储节点；
4. 继续顺时针找 R-1 个节点作为副本（R 为副本因子）。

当节点加入或离开时，仅影响相邻区间（约 1/N 数据重新映射），绝大多数 chunk 位置保持不变。

### 3.4 版本号 + 乐观锁的并发控制抽象

Metadata 子系统采用乐观并发控制（OCC）而非悲观锁，原因是在读多写少的索引场景中，悲观锁会严重阻塞查询关键路径。

**核心思想**：

- 每个索引节点（L1 的 Tree Node 和 L2 的 Hash Entry）带有一个单调递增的版本号。
- **读取**：不加锁，读取时记录当前版本号。读取完成后，如果版本号发生了变化，说明期间有其他写入，本次读取结果可能不一致，调用方决定是重试还是接受最终一致性。
- **写入**：提交修改时，检查目标节点的版本号是否与读取时一致。若不一致，则放弃本次写入并返回冲突；若一致，则原子性地更新数据和版本号。

**为什么适合 Metadata 场景**：

- 查询操作（lookup）占绝对主导，乐观锁让查询线程零阻塞。
- 写入操作主要是插入新请求产生的增量索引，冲突概率低（不同请求通常写入不同的前缀路径或 chunk）。
- 冲突时的回退成本可接受：kv cache 未命中只会导致重算，不会导致数据错误。

### 3.5 SegmentMap 的数据抽象

`SegmentMap` 是 Metadata 对外暴露的统一结果格式，每个 Segment 包含：
- `token_start` / `token_end`：token 范围（含/不含）
- `status`：`HIT` / `MISS` / `IN_TRANSIT`
- `location`：当 status=HIT 时有效（layer、node_id、block_refs）

**设计思想**：
- **优先合并**：相邻同层命中段合并为更大 Segment，减少 Connector 加载次数。
- **MISS 明确标出**：Connector 只需关注 `MISS` 段，由推理引擎重算。
- **IN_TRANSIT 过渡态**：层间迁移时可能短暂返回该状态，Connector 自行决定等待或降级读取。

在设计 Metadata 子系统之前，必须首先明确本场景下不可妥协的核心约束：

1. **延迟约束**：L1 索引查询必须在 10us 内完成（微秒级），因为它在推理请求的 TTFT 关键路径上。L2 索引查询可以放宽到 1ms 以内，因为它涉及网络 RTT。
2. **扩展性约束**：系统需要支持从单节点（8 GPU）扩展到百节点集群。单节点索引不能成为扩展天花板。
3. **前缀匹配约束**：LLM 推理 workload 中 80% 以上的缓存复用是前缀复用（如系统提示词、多轮对话历史）。索引必须原生、高效地支持前缀匹配。
4. **任意位置复用约束**：在 RAG 和长上下文场景中，非前缀的 chunk 级复用（如重复文档片段）也占有显著比例。索引需要在可接受的复杂度内支持这一点。
5. **一致性约束**：允许最终一致，100ms 窗口。强一致性需要分布式锁，会大幅增加延迟和实现复杂度。
6. **内存约束**：索引结构必须能常驻内存。若索引本身需要频繁换页或查询磁盘，则违背了"快速定位"的核心目标。

### 4.2 候选方案评估

基于上述约束，我们评估以下四个候选方案：

#### 候选 A：单一全局 Radix Tree（SGLang 风格扩展）

- **延迟**：单节点内前缀匹配 O(L)，平均 ~2us，满足约束 1。
- **扩展性**：单机内存天花板，百节点分片困难（前缀热点集中）。不满足约束 2。
- **前缀匹配**：原生支持，10/10。满足约束 3。
- **任意位置复用**：不支持。不满足约束 4。
- **结论**：作为全局唯一方案不可行，但作为 L1 局部方案极其优秀。

#### 候选 B：Chunk-Hash 分布式索引（LMCache / 一致性哈希环风格）

- **延迟**：chunk 查找 O(1)，但前缀匹配需逐 chunk 查表（2K tokens ≈ 8 次查找，总计 ~4us）。勉强满足约束 1。
- **扩展性**：按 hash 范围分片，支持水平扩展；百节点规模下采用一致性哈希环作为分布式定位实现变体。满足约束 2。
- **前缀匹配**：支持但非原生，效率次优。部分满足约束 3。
- **任意位置复用**：原生支持内容寻址。满足约束 4。
- **结论**：作为 L2 分布式方案优秀，作为 L1 本地方案是次优解。一致性哈希环是 Chunk-Hash 分布式索引在百节点规模下的推荐实现变体，二者不是并列的独立方案，而是同一语义层（内容寻址）的不同工程实现。

#### 候选 C：全局集中式元数据服务（MoonCake Conductor 风格）

- **延迟**：每次查询需网络 RTT（~50-200us），不满足 L1 的 10us 约束 1。
- **扩展性**：中心服务是瓶颈，百节点下单点风险高。部分满足约束 2。
- **前缀匹配 / 任意位置复用**：取决于下层实现。
- **结论**：适合全局调度决策，不适合作为高频查询的索引层。

#### 候选 D：纯一致性哈希环（无 Chunk-Hash 内容寻址）

- **延迟**：本地查环 O(1)，满足约束。
- **扩展性**：线性扩展，节点变更迁移最小。满足约束 2。
- **前缀匹配**：不支持。不满足约束 3。
- **任意位置复用**：仅支持按 key 定位，不支持基于内容指纹的去重复用（因为没有 Chunk-Hash）。部分满足约束 4。
- **结论**：若 L2 仅采用一致性哈希环而不做 Chunk-Hash，则丧失了内容寻址带来的去重收益。因此一致性哈希环不能单独作为 L2 方案，必须与 Chunk-Hash 结合。

### 4.3 约束冲突与决策推导

四个候选方案在核心约束上的表现如下：

| 约束 \ 方案 | 候选 A (Radix Tree) | 候选 B (Chunk-Hash + 一致哈希) | 候选 C (集中式服务) | 候选 D (纯一致哈希) |
|-------------|---------------------|-------------------------------|---------------------|---------------------|
| 延迟 < 10us (L1) | 满足 | 勉强满足 | 不满足 | 满足 |
| 扩展性 (百节点) | 不满足 | 满足 | 部分满足 | 满足 |
| 前缀匹配 | 优秀 | 中等 | 取决于下层 | 不支持 |
| 任意位置复用 | 不支持 | 优秀 | 取决于下层 | 中等 |
| 一致性 | 满足 | 满足 | 满足 | 满足 |

**核心冲突发现**：

- **冲突 1**："L1 延迟 < 10us" 与 "扩展性 (百节点)" 直接冲突。没有任何单一方案能同时极致满足这两者。
- **冲突 2**："原生前缀匹配" 与 "任意位置复用" 存在实现效率冲突。Radix Tree 擅长前者但完全无法做后者；Chunk-Hash 分布式索引能做两者但前缀匹配效率次优；纯一致性哈希环则放弃了内容寻址去重能力。

**决策推导**：

在约束冲突下，单一的候选方案均无法获胜。因此必须采用**分层混合架构**：

1. **用候选 A 解决 L1 本地前缀匹配**：在单节点内存中部署 Radix Tree，最大限度地压榨 TTFT 关键路径上的延迟性能，同时放弃跨节点扩展性（L1 本来就不应该跨节点）。
2. **用候选 B 解决 L2 分布式索引**：在 L2 层采用 Chunk-Hash 分布式索引实现内容寻址和任意位置复用，其分布式定位层通过一致性哈希环实现（百节点规模）或固定分片表实现（小集群规模）。L2 天然接受更高的延迟（因为它覆盖的是远程存储）。
3. **明确排除候选 C 作为索引层**：集中式元数据服务被剥离到调度层（如果有必要），Metadata 子系统本身保持去中心化的 L1 + 分布式 L2 结构。
4. **明确排除候选 D 作为独立 L2 方案**：纯一致性哈希环（无 Chunk-Hash）虽然扩展性优秀，但不支持内容寻址去重，会丧失大量 chunk 级复用收益，因此不能单独作为 L2 索引语义层。

**牺牲的代价（Y）**：

- **层间一致性和协同复杂度**：需要维护 L1 到 L2 的数据降级/升级协议，以及短暂不一致时的冲突消解策略。这个代价被"延迟和扩展性必须兼得"的硬约束所压倒，因此是可接受的。
- **实现复杂度**：相比单一方案，团队需要同时精通两种完全不同的索引结构和并发控制模型。

### 4.4 为什么 L2 的 Chunk-Hash 分布式索引需要一致性哈希环作为定位实现变体

在 ADR-002 的决策表格中，L2 被描述为 "Chunk-Hash"。本设计在 Chunk-Hash 的基础上增加一致性哈希环，是因为**纯 Chunk-Hash 分片表在百节点规模下会遇到三个具体瓶颈**，而一致性哈希环恰好能消解它们。本节给出增量推导。

#### 4.4.1 固定分片表实现变体的瓶颈分析

Chunk-Hash 分布式索引有多种工程实现方式。其中一种简单的实现是"固定分片表"：每个 chunk 的 hash 值按固定范围映射到某个分片（如 hash % N），每个分片由一个 Metadata 节点负责维护。这在节点数固定时工作良好，但在百节点、动态扩缩容场景下存在以下问题：

1. **分片路由表的膨胀与同步成本**
   - 百节点意味着每个节点需要维护一张包含 100 个分片位置的路由表。
   - 当节点加入或离开时，路由表需要全量刷新。假设每 5 秒 gossip 一次，100 个节点 × 100 个分片项 = 10K 状态项需要在集群间传播，单次 gossip 包大小可达数百 KB。
   - 更关键的是，Chunk-Hash 的每个条目都需要绑定到"当前哪个节点负责这个分片"，这意味着索引条目本身也隐含了对路由表的依赖。路由表变更时，大量条目的"负责节点"信息需要更新（即使数据本身没有迁移）。

2. **节点变更时的全量重分布成本**
   - 当节点数从 100 增加到 101 时，固定范围分片（hash % N）会导致**几乎所有 chunk 的映射结果改变**。约 99% 的 chunk 需要重新计算分片归属，触发索引条目的批量迁移。
   - 虽然 KV Cache 数据本身不需要全量搬移（因为索引和物理数据是分离的），但索引条目的重分布会导致 Metadata 节点在短时间内承受巨大的写入负载和一致性校验开销。

3. **热点分片的不可逃逸性**
   - 某些热门 prompt（如系统提示词）会产生大量命中同一 hash 范围的查询，这些查询会全部涌向同一个分片节点。
   - 纯分片表只能通过"再分片"来缓解热点，但这需要停服或复杂的在线重分片协议。相比之下，一致性哈希环可以通过虚拟节点和负载边界检查（CHWBL）将热点自然疏散到多个物理节点。

#### 4.4.2 一致性哈希环实现变体的适配度分析

一致性哈希环通过将 chunk_hash 映射到一个逻辑环（0 ~ 2^32-1），再顺时针寻找负责的物理节点，恰好消解了上述三个瓶颈：

| 瓶颈 | 固定分片表实现变体 | 一致性哈希环实现变体 |
|------|---------------------|---------------------------|
| 路由表大小 | O(N) 分片项，节点变更需全量刷新 | O(N) 虚拟节点，但每个虚拟节点仅覆盖相邻区间；gossip 增量更新，变更影响范围 ~1/N |
| 节点变更重分布 | 几乎所有 chunk 重新映射 | 仅相邻区间 (~1/N) 的 chunk 需要更新映射 |
| 热点逃逸能力 | 无，必须通过在线重分片 | 有，虚拟节点 + CHWBL 可将热点疏散到多个物理节点 |
| 查询本地额外开销 | 查表后直接向目标节点发请求 | 查环为 O(log V)（V=虚拟节点数，通常 1000-2000），额外开销 <1us，可忽略 |

**关键量化**
- 在 100 节点集群中，新增 1 个节点时：
  - 纯分片表导致的索引重分布比例：~99%
  - 一致性哈希环导致的索引重分布比例：~1%
- 这个 99:1 的差距是决定性的——它决定了系统能否在百节点规模下实现无感知的弹性扩缩容。

#### 4.4.3 选择一致性哈希环作为 L2 分布式定位层的代价与收益权衡

**代价（Y）**：
1. **本地环的副本一致性维护**：每个 Metadata 节点都持有一致性哈希环的本地副本。节点变更时，需要通过 gossip 协议在秒级内达成一致。在一致窗口内，不同节点对同一 chunk 的目标节点选择可能略有不同（但最终一致）。
2. **热点倾斜风险**：虽然虚拟节点能均衡负载，但极端热点 workload（如 90% 查询命中同一个 chunk）仍可能突破 CHWBL 的疏散能力。需要通过多副本机制进一步缓解（第 8 章风险 3 已覆盖）。

**收益（X）**：
1. **百节点扩展性**：节点变更时仅影响 ~1/N 的数据映射，支撑弹性扩缩容。
2. **去中心化**：无需集中式分片路由表或配置中心，每个节点独立计算 chunk 位置。
3. **故障隔离**：单个 Metadata 节点故障时，其环上区间自动由邻居节点接管，无需全局协调。

**决策标准**：若集群长期 <20 节点，纯 Chunk-Hash 分片表足够简单；若目标 ≥50 节点或需弹性扩缩容，一致性哈希环的收益压倒复杂度。

**修正声明**：ADR-002 的 "Chunk-Hash" 指 L2 的索引语义为内容寻址。一致性哈希环是 Chunk-Hash 分布式索引在百节点规模下的推荐分布式定位实现变体，而非与其并列的独立候选方案。若实际规模长期 <20 节点，允许回退为固定分片表实现变体。

### 5.1 查询生命周期时序图

以下时序图展示了一个典型 Lookup 请求在 Metadata 子系统内部的处理流程，包含 L1 命中、L1 未命中再查 L2 的两种路径：

```
                    Lookup Request
                         |
                         v
    +---------------------------------------------+
    |              API Gateway                    |
    |         (接收 token 序列 + 操作类型)         |
    +------------------+--------------------------+
                       | 内部请求对象
                       v
    +---------------------------------------------+
    |              Query Router                   |
    |         (决策：优先查 L1，级联查 L2)          |

             |                     |
     L1 路径 v                     v L2 路径（L1 未命中时触发）
    +-----------------+    +-----------------+
    | L1 Index Mgr    |    | L2 Index Mgr    |
    | (Radix Tree)    |    | (Chunk-Hash)    |
    |                 |    |                 |
    |  1. 最长前缀匹配 |    |  1. 按 chunk    |
    |  2. 返回匹配长度 |    |     切分 token  |
    |     和块引用    |    |  2. 逐个查哈希表 |
    |  3. 标记访问时间 |    |  3. 一致哈希环   |

                       | SegmentMap
                       v
    +---------------------------------------------+
    |              API Gateway                    |
    |         (返回给 Connector)                   |
    +---------------------------------------------+
```

**异常路径说明**：
- **L1 部分命中**：Query Router 仅将**未命中后缀**转发给 L2，最小化 L2 负载。
- **L1 与 L2 冲突**：Result Assembler 优先采纳 L1（"以 L1 为准"），因为 L1 数据更新鲜。
- **版本号冲突**：写入时若目标节点版本号已变，记录为软冲突并在 10ms 后重试；仍冲突则提交异步修复队列。

 ### 5.2 写入路径：索引注册与淘汰

 Storage 数据变更后，Metadata 异步更新索引：
 `Storage -> API Gateway -> Query Router(本地->L1/远程->L2) -> L1/L2 Mgr -> Versioned Metadata Store(乐观锁+WAL) -> 返回OK / 记录软冲突10ms后重试`

 **关键状态**：
 - **PENDING**：索引变更已提交但尚未完成持久化确认，查询不可见。
 - **ACTIVE**：索引变更已完成持久化确认，对查询生效。
 - **EVICTING**：L1 节点标记为"待降级"，新查询不再命中，但已被活跃请求引用的数据可继续使用。
 - **EXPIRED**：L1 已删除该节点；若存在远程副本，由 L2 继续维护。

 ### 5.3 L1→L2 降级/晋升状态机

 L1 与 L2 之间的数据迁移是 Metadata 子系统最复杂的动态行为。以下状态机精确定义了单个块（或 chunk）在层间迁移时的全生命周期，以及各状态转换的触发条件、超时阈值和失败回滚路径。

 ```
 +------------------+     +------------------+     +------------------+
 |      NULL        |---->|     PENDING      |---->|     ACTIVE       |
 |    (无索引)       |     |  (WAL 待持久化)   |     |  (L1 可查询)      |
 +------------------+     +------------------+     +--------+---------+
                                                            |
              +--------------------+--------+---------------+
              |                    |        |
              v                    v        v
      +------------------+  +------------------+  +------------------+
      |    EVICTING      |  |   L2 PENDING     |  |  L2 ACTIVE       |
      | (L1 只读,待降级)  |->|  (L2 WAL 待提交)  |->|  (L2 可查询)      |
      +------------------+  +------------------+  +--------+---------+
                                                           |
              +--------------------------------------------+
              |
              v
      +------------------+     +------------------+
      |    EXPIRED       |---->|      NULL        |
      |  (L1 删除,仅L2)   |     |   (索引完全注销)  |
      +------------------+     +------------------+
 ```

 **状态流转说明**：
 - `NULL ↔ PENDING ↔ ACTIVE`：Storage 注册/注销本地块，Metadata 持久化 WAL。
 - `ACTIVE → EVICTING`：L1 内存达 80% 水线或 Storage 发起迁移，原子标记为只读。
 - `ACTIVE → L2 PENDING`：跳层直达，Storage 直接将块写入远程后通知 Metadata。
 - `EVICTING → L2 PENDING → L2 ACTIVE`：`ref_count=0` 后异步降级到 L2，WAL 提交生效。
 - `L2 ACTIVE → ACTIVE`：Tiering Manager 晋升，远程块加载回本地后重新注册 L1。
 - `L2 ACTIVE → EXPIRED → NULL`：Storage 删除通知，Metadata 逐级注销索引。

 **状态转换条件与超时策略**：

 | 转换边 | 触发条件 | 权威方 | 超时/失败处理 |
 |--------|----------|--------|---------------|
 | NULL → PENDING | Storage `register_block` | Storage | 100ms 内未提交则重试 3 次；失败丢弃 update |
 | PENDING → ACTIVE | WAL 持久化确认 | Versioned Store | 同上 |
 | ACTIVE → EVICTING | L1 内存达 80% 水线或 Storage 迁移通知 | Tiering Manager / L1 Mgr | 无超时；需原子标记 EVICTING |
 | ACTIVE → L2 PENDING | Storage 远程块落盘后通知（跳层直达） | Storage | Storage 保证物理数据已落盘 |
 | EVICTING → L2 PENDING | `ref_count=0` 且异步降级完成 | L1 Mgr / Storage | 500ms 未完成重试 2 次；失败回滚 ACTIVE 或标记 EXPIRED |
 | L2 PENDING → L2 ACTIVE | L2 WAL 提交成功 | Versioned Store | 500ms 超时重试 3 次；失败则暂不可见 |
 | L2 ACTIVE → EXPIRED | L1 确认安全删除且 L2 生效 | L1 Mgr | 无超时；L1/L2 至少保留一处 ACTIVE |
 | L2 ACTIVE → ACTIVE | Tiering Manager 晋升完成 | Tiering Manager / Storage | 晋升 2s 超时放弃，保留 L2 |
 | L2 ACTIVE → NULL | Storage 删除通知 | Storage | 100ms 超时重试 3 次 |

 **核心设计 rationale**：
 1. **双 ACTIVE 安全窗口**：EVICTING→L2 ACTIVE 过程中允许 L1/L2 短暂共存，确保查询不 miss；Connector 优先采纳本地结果。
 2. **EVICTING 只读语义**：新查询不再命中，但已持有引用的活跃请求可继续使用，避免后台降级中断读取。
 3. **失败回滚**：L2 降级失败时，若 L1 物理数据仍在则回滚至 ACTIVE；仅当 L1 已释放且 L2 也失败时才进入 EXPIRED（只影响命中率，不影响正确性）。
 4. **权威方唯一**：Metadata 从不主动决定"把块搬到哪"，只响应 Storage 或 Tiering Manager 的完成通知。

 ### 5.4 故障恢复视图

 Metadata 节点可能因 OOM、硬件故障或软件崩溃而重启。以下视图描述了从崩溃到完全恢复服务的完整流程，以及与其他 Metadata 节点在 L2 拓扑同步上的交互。

 ```
 Metadata Node Failure
        |
        v
 +---------------------------+
 | 1. Detect failure         |
 | (health check 3 misses)   |
 +------------+--------------+
              |
      +-------+-------+
      |               |
      v               v
 +----------------+  +------------------+
 | 2a. Local Rec. |  | 2b. ClusterAware |
 | Load snapshot  |  | Gossip node DOWN |
 | Replay WAL     |  | Update hash ring |
 | Rebuild L1/L2  |  | Trigger repair   |
 +--------+-------+  +---------+--------+
          |                    |
          +--------+-----------+
                   v
          +-------------------+
          | 3. L2 Topology Sync |
          | (pull full or merge |
          |  delta gossip)      |
          +----------+----------+
                     v
          +-------------------+
          | 4. Service Resume |
          | (L2 CONSISTENT)   |
          +----------+----------+
                     v
          +-------------------+
          | 5. Background     |
          |    Repair         |
          +-------------------+
 ```

 **恢复阶段与 RTO 估算**：

 | 阶段 | 操作内容 | 耗时 | 一致性边界 |
 |------|----------|------|------------|
 | 1. 故障检测 | 健康检查 3 次未响应（间隔 2s） | ~6s | L2 查询路由到其他副本 |
 | 2a. 本地恢复 | 加载快照(2GB)+重放WAL(100MB) | ~5s | L1 恢复，L2 可能 stale |
 | 2b. 集群感知 | Gossip 传播节点 DOWN | ~5s | 邻居接管环上区间 |
 | 3. L2 拓扑同步 | 全量拉取(<10MB) 或增量 merge | ~100ms | L2 最终一致 |
 | 4. 服务恢复 | L2 标记 CONSISTENT | 瞬时 | 可对外服务 |
 | 5. 后台修复 | 异步扫描 stale 条目对账 | ~10s（后台） | 不影响查询 |

 **RTO**：关键路径 ≈ **11s**；完全恢复 ≈ 21s。

 **设计 rationale**：L1 必须靠本机快照+WAL 重建（建议 30s 异步刷盘）。L2 允许暂时 stale，由邻居接管区间。选择**先恢复服务、后后台修复**，控制 RTO，这是最终一致性的自然结果。

 #### 5.4.1 索引完全丢失恢复

 如果 Metadata 节点的本地 SSD 同时损坏，导致快照和 WAL 均不可读，该节点进入**索引完全丢失**状态。此时本地 L1 索引必须从零重建，L2 索引可从邻居节点恢复：

 1. **L1 重建路径（从 Storage 扫描）**：
    - Storage 子系统保存了物理块到 token 范围的映射（虽然粒度较粗）。
    - Metadata 向 Storage 发起 `scan_local_blocks()` 请求，获取当前节点上所有活跃的 block/token_range 列表。
    - 基于返回列表重新构建 Radix Tree 的叶节点（缺少精细的前缀压缩，但功能可用）。
    - **RTO 估算**：1TB 本地数据对应约 10K-50K 个块，扫描和重建约需 **60-120s**。

 2. **L2 重建路径（从邻居拉取）**：
    - 一致性哈希环上的邻居节点持有该节点负责区间的副本（虽然可能是 stale 的）。
    - 故障节点向邻居请求 `pull_hash_ring_partition(range)`，获取该区间内的全部 L2 索引条目。
    - 由于 L2 索引总大小通常 < 10MB/节点，全量拉取可在 **1-5s** 内完成。
    - 拉取完成后，L2 标记为 ACTIVE 但带有 `STALE_REPLICA` 标记；后台 Repair 任务与 Storage 对账，删除已不存在的条目。

 **最坏情况 RTO**：L1 从 Storage 扫描重建 ≈ 120s；L2 从邻居拉取 ≈ 5s；服务在 L2 恢复后即可对外提供查询（虽然前缀匹配精度暂时下降），完全恢复 ≈ 120s。

 #### 5.4.2 网络分区与脑裂处理

 在网络分区期间，Metadata 节点可能被割裂为两个或多个子集。由于我们采用最终一致性，必须明确分区期间的写入策略：**允许写入**（permit-all 策略），但在恢复后需要通过对账协议解决冲突。

 **分区期间的写入策略**：
 - 每个分区的 Metadata 节点继续接受本地 L1 写入（因为 L1 是单节点内存结构，不受网络影响）。
 - L2 写入：节点只在分区内处理本地 Chunk-Hash 更新；由于一致性哈希环的 gossip 中断，不同分区可能对同一 chunk 产生 divergent 的位置信息。

 **恢复后的对账协议**：
 1. **版本向量比对**：每个 L2 条目携带 `(node_id, version)` 列表。恢复后，冲突双方交换版本向量。
 2. **时间戳最新胜利（Last-Writer-Wins）**：若版本向量无法直接比较（并发写入），以最后一次更新时间戳较大者为准；若时间戳差值 < 1s，则认为存在 true conflict，标记为 `AMBIGUOUS` 并降级为 MISS（由引擎重算）。
 3. **Merkle Tree 批量对账**：对于差异较大的分区，使用 Merkle Tree 快速定位不一致的 chunk 范围，然后逐条应用 LWW 规则。
 4. **与乐观锁的自洽性**：3.4 节的"版本号 + 乐观锁"仅用于单节点内的并发控制。在网络分区场景下，跨节点的版本号不保证全局单调递增，因此需要 LWW 作为跨节点冲突消解的补充机制。文档明确：**单节点乐观锁 ≠ 全局分布式一致性协议**。

 **数据丢失风险评估**：
 - 由于 KV Cache 是可重算的，LWW 失败只会导致 miss，不会导致数据丢失或引擎结果错误。
 - 最坏情况是：分区期间同一 chunk 被两个分区各自更新，恢复后较晚写入者胜出，较早写入者的数据变为不可达，最终通过 Storage 的 RecycleManager 回收。

 ## 6. 与周边子系统的边界和交互

 Metadata 子系统处于整个 KV Cache Offloading 架构的核心枢纽位置，但它必须严守边界：只产出"位置信息"，不执行实际的数据搬运。

 ### 6.1 边界接口表格

| 方向 | 调用方 | 被调用方 | 事件类型 | 传递内容 | 权威方 | 一致性要求 |
|------|--------|----------|----------|----------|--------|------------|
| 被调用 | Connector | Metadata API Gateway | `lookup` / `invalidate` | token 序列 + 上下文标识 | Connector | 最终一致（100ms 窗口） |
| 被调用 | Tiering Manager | Metadata Heatmap Collector | `query_heatmap` | 热度查询请求（时间窗口、粒度） | Tiering Manager | 最终一致（秒级） |
| 被调用 | Storage | Metadata L1/L2 Index Manager | `block_created` | chunk/token_range + block_id + location | Storage | 最终一致（100ms 窗口） |
| 被调用 | Storage | Metadata L1/L2 Index Manager | `block_migrated` | 旧位置 + 新位置 + block_id | Storage | 最终一致（100ms 窗口） |
| 被调用 | Storage | Metadata L1/L2 Index Manager | `block_deleted` | block_id + location | Storage | 最终一致（100ms 窗口） |
| 主动调用 | Metadata L1 Index Manager | Storage | `local_evict_suggestion` | block_id + reason（memory_pressure / explicit_ttl） | Metadata | 异步，不阻塞查询 |
| 主动调用 | Metadata L1/L2 Index Manager | Storage | `index_inconsistency_detected` | block_id + 不一致描述 + 建议操作 | Metadata | 异步，不阻塞查询 |
| 主动调用 | Metadata Heatmap Collector | Tiering Manager | `heatmap_report` | chunk 级别冷热分布 + 预取建议 | Metadata | 批量推送，允许 1 分钟延迟 |
| 主动调用 | Metadata L2 Index Manager | 其他 Metadata 节点 | `ring_topo_delta` | 节点增删 + 虚拟节点变更列表 | 发起节点 | 最终一致（秒级） |

### 6.2 边界说明

- **与 Storage 的双向边界**：
  - **Storage → Metadata**（权威方：Storage）：Storage 在物理块创建、迁移、删除**完成后**通知 Metadata 更新索引。Metadata 绝不发起 `read_block` 或 `write_block` 调用。
  - **Metadata → Storage**（权威方：Metadata）：当 L1 内存达水线时，Metadata 可向 Storage 发送 `local_evict_suggestion`，建议淘汰某些本地块。这是**建议而非命令**——Storage 的 Tiering Manager 有权接受或拒绝该建议，并自主决定最终的淘汰时机和迁移目标。当 Metadata 检测到索引与 Storage 实际状态不一致时（如对账发现 block 存在但索引缺失），发送 `index_inconsistency_detected`，Storage 负责修复物理状态或重新通知 Metadata。
  - **调度决策边界明确声明**：任何涉及跨节点数据迁移的调度决策（"这块数据应该搬到哪个节点"）由 Storage 子系统内部的 Tiering Manager 或上层 Conductor 做出。Metadata **仅被动接收迁移完成后的索引更新通知**，不参与迁移目标的决策。这一原则防止了索引层与调度层之间的职责重叠和循环依赖。
- **与 Connector 的边界**：Connector 只看到一个统一的 API：`lookup(token_sequence) -> SegmentMap`。Connector 不需要知道 SegmentMap 中的位置是本地 HBM、本地 DRAM 还是远程节点。位置的实际解析由 Connector 根据部署配置决定。
- **与 Tiering Manager 的边界**：Tiering Manager 是 Storage 子系统内部负责冷热调度的模块。它需要查询 Heatmap Collector 获取热度统计，以决定哪些 chunk 应该从 SSD 晋升到 DRAM，或从 DRAM 淘汰到 SSD。Metadata 只提供"读多还是读少"的信息，**不参与晋升/淘汰的物理执行**。

### 6.3 接口抽象的层次设计

Metadata 对外暴露两类接口：
- **类型 A（面向 Connector）**：高抽象层次，只处理 `token 序列 -> SegmentMap`，屏蔽底层索引和介质差异。示例：`lookup(tokens) -> SegmentMap`。
- **类型 B（面向 Storage/Tiering Manager）**：中抽象层次，暴露 `block_id`、`location` 等细节以支持精确注册/注销，但 Metadata 不介入数据搬运。示例：`register_block(...) -> version`。

**为什么区分**：单一接口要么让 Connector 被迫理解块细节（增加耦合），要么让 Storage 陷入高抽象繁琐操作（降低效率）。分层接口在"简洁性"和"精确性"之间取得平衡。

### 6.4 关键接口契约草案

以下用精简的伪代码/IDL 风格定义 5 个最关键接口的契约。注意这不是 C 结构体或函数签名，而是架构层面的抽象接口约定。

**1. `lookup`（面向 Connector）**

```
SegmentMap metadata_lookup(
    const token_sequence_t *tokens,
    const query_context_t *ctx,
    lookup_status_t *status
);
/* 契约：
 * - tokens 长度上限由部署配置决定（建议 ≤ 128K tokens）。
 * - status 返回 OK / PARTIAL / TIMEOUT。
 * - 调用方在 TIMEOUT 时可选择重试或直接降级为 miss（命中率为 0 的 SegmentMap）。
 * - 返回值中相邻同层命中段已合并。
 */
```

**2. `register_block`（面向 Storage）**

```
int metadata_register_block(
    const block_info_t *info,
    version_t *committed_version
);
/* 契约：
 * - 调用方（Storage）保证物理块已落盘后发起调用。
 * - 返回 0 表示索引已提交；返回 CONFLICT 表示版本冲突，调用方应在 10ms 后重试。
 * - 若返回 OOM，表示 L1 内存达上限，Storage 应将新数据写入 L2/L3。
 */
```

**3. `notify_migration`（面向 Storage）**

```
int metadata_notify_migration(
    const location_t *old_loc,
    const location_t *new_loc,
    const block_id_t *block_id
);
/* 契约：
 * - 异步接口：Storage 迁移完成后调用，不阻塞查询关键路径。
 * - 在 100ms 不一致窗口内，Metadata 可能同时返回 old_loc 和 new_loc 给不同查询；Connector 优先采纳本地（层号较小）结果。
 * - 返回 0 表示已接受；返回 UNFOUND 表示 block_id 不在索引中（可能已被淘汰）。
 */
```

**4. `query_heatmap`（面向 Tiering Manager）**

```
heatmap_report_t metadata_query_heatmap(
    const time_window_t *window,
    heatmap_status_t *status
);
/* 契约：
 * - 查询时间窗口内的冷热分布。允许 1 分钟数据延迟。
 * - status 返回 OK / EMPTY（该窗口无采样数据）。
 * - 调用方不应在同步关键路径上调用此接口。
 */
```

**5. `index_inconsistency_detected`（Metadata → Storage）**

```
int storage_handle_index_inconsistency(
    const block_id_t *block_id,
    const inconsistency_desc_t *desc,
    repair_hint_t *hint
);
/* 契约：
 * - 异步通知。Storage 收到后应在 5s 内完成自检并修复，或再次调用 register_block 补偿。
 * - hint 提供建议操作：RE_SCAN / RE_REGISTER / IGNORE。
 */
```

## 7. 设计 Rationale

### 7.1 "为什么不是方案 A（纯 Radix Tree）？"

纯 Radix Tree 在单节点内提供了无与伦比的前缀匹配性能，但它有三个致命的扩展性限制：

1. **内存天花板**：当序列数量和长度增加时，树节点数量会膨胀。虽然压缩树能有效降低空间开销，但在最坏情况下（前缀几乎不共享）会退化为接近哈希表的空间复杂度。单节点的 DRAM 无法支撑百节点集群的全局索引。
2. **不支持非前缀复用**：现代 LLM workload（尤其是 RAG）中，大量复用发生在 prompt 的中间或后部，而非前缀。放弃这部分复用机会意味着显著降低缓存利用率。
3. **无法表达分布式位置**：Radix Tree 的边和节点存储的是本地块引用，如果要将其扩展为跨节点索引，需要在每个节点中引入远程指针和同步协议，这会彻底破坏树的简洁性和查询性能。

因此，纯 Radix Tree 只能作为 L1 局部方案，绝不能作为全局唯一方案。

### 7.2 "为什么不是方案 C（集中式元数据服务）？"

集中式元数据服务（如 MoonCake Conductor）在全球调度层面有其价值，但作为 Metadata 查询层存在结构性缺陷：

1. **网络 RTT 不可接受**：即使是最快的 RDMA 网络，局域网内往返也需要数十微秒到数百微秒。对比 L1 Radix Tree 的微秒级查询，集中式服务会让 TTFT 增加一个数量级。
2. **单点和热点风险**：所有查询汇聚到一个（或一组）中心节点，高并发下容易成为系统瓶颈。
3. **调度与索引职责混淆**：Conductor 更适合做"节点选择"和"负载均衡"这类调度决策，而不应该承担"每个请求都要查一次"的高频索引定位任务。

### 7.3 "为什么 L1 不采用纯 Chunk-Hash？"

文档 4.2 节已经指出 Chunk-Hash 作为 L1 方案是"次优解"，但"次优"不等于"不可接受"。本节通过量化对比说明：尽管纯 Chunk-Hash 的延迟仍在 10us 约束内，但 Radix Tree 在延迟确定性、内存效率和 TTFT 收益上的综合优势压倒性地 justify 了额外的实现复杂度。

#### 7.3.1 查询延迟对比（基于理论建模估算，待验证）

我们基于一个典型的多轮对话 workload 进行仿真：平均序列长度 2K tokens，前缀相似度 30%，chunk 大小 256 tokens。

| 指标 | 压缩 Radix Tree (L1) | 纯 Chunk-Hash (L1) | 备注 |
|------|---------------------|---------------------|------|
| 平均查询延迟 | ~1.5 us | ~4.2 us | Radix Tree 单次前缀匹配 O(L)，平均只需 2-3 次比较；Chunk-Hash 需要 8 次哈希查表 (2K/256) |
| P50 延迟 | ~1.2 us | ~3.8 us | — |
| P99 延迟 | ~2.8 us | ~8.5 us | Chunk-Hash 的尾延迟来自逐 chunk 比较时的缓存未命中和哈希冲突链；Radix Tree 的压缩路径使内存访问更局部化 |
| P99.9 延迟 | ~4.5 us | ~18 us | Chunk-Hash 在高并发下哈希表重新哈希或锁竞争会导致突发延迟；Radix Tree 采用乐观锁，查询零阻塞 |
| 10us 约束裕量 | +5.5 us | +0 us（P99.9 已逼近）| 若 workload 波动或并发突增，Chunk-Hash 的 P99.9 极易突破 10us |

**关键发现**：纯 Chunk-Hash 的 P99.9 延迟（~18us）已突破 10us 约束边界，存在系统性超标风险；Radix Tree P99.9 ~4.5us，裕量充足。

#### 7.3.2 内存占用对比（前缀相似度 30% 场景，基于理论建模估算）

| 结构 | 内存占用 | 计算依据 |
|------|----------|----------|
| 压缩 Radix Tree | ~2.1 GB | 内部节点 ~100MB（前缀共享压缩）+ 叶节点 ~2.0 GB |
| 纯 Chunk-Hash | ~3.8 GB | 10K 序列 × 8 chunks × ~480 字节/条目 |

**Rationale**：Radix Tree 的压缩特性使共享前缀只存一次；Chunk-Hash 因每个 chunk 需保留独立位置上下文，内存更高。若前缀相似度跌至 ≤30%，触发回退为纯 Chunk-Hash。

#### 7.3.3 量化数据来源与验证状态说明

7.3.1 和 7.3.2 中的延迟与内存数值来源于**基于公开工具与理论建模的估算**，而非大规模生产环境的实测数据。为确保评审透明度，本节明确列出计算假设：

- **仿真工具**：使用内部开发的 `metadata_index_sim.c` 原型（基于 ART/Radix Tree 开源实现 + 开放寻址哈希表），尚未经过完整 CI 回归验证。
- **workload 参数**：平均序列长度 2K tokens，前缀相似度 30%（基于公开多轮对话 benchmark 的解读），chunk 大小 256 tokens，总请求数 10K。
- **数据结构参数**：
  - Radix Tree：edge 压缩阈值 = 2 个 token，叶节点平均挂载 1.5 个 block_refs。
  - Chunk-Hash：负载因子 0.7，条目大小 ~480 字节（包含位置列表和元数据），开放寻址法处理冲突。
- **硬件假设**：AMD EPYC 9004 系列（L3 缓存 256MB），DDR5-4800 内存，单线程查询模型。

**验证状态**：上述数值为**理论估算，待原型验证**。关键回退条件：若 4 周内原型测试显示纯 Chunk-Hash 的 P99.9 < 10us 且内存占用与 Radix Tree 差异 < 20%，则重新评估 L1 方案选择的必要性。

#### 7.3.4 C 语言实现复杂度对比

| 维度 | 压缩 Radix Tree | 纯 Chunk-Hash（开放寻址） |
|------|----------------|--------------------------|
| 核心代码量（估算） | ~3K 行 | ~800 行 |
| 并发控制 | 版本号 + 乐观锁（OCC），查询零阻塞；插入时需 CAS 更新节点版本和子节点指针 | 版本号 + 乐观锁（OCC），查询零阻塞；插入时 CAS 更新槽位和版本号 |
| 锁粒度 | 无全局锁，冲突只发生在共享前缀路径的节点上，并发度与请求前缀发散度正相关 | 无全局锁，冲突只发生在同一 hash 槽位，并发度与 chunk hash 分布均匀性正相关 |
| 内存安全关键测试点 | 压缩/解压边界、节点分裂合并、edge_label 越界检查、ref_count 生命周期 | 哈希表扩容重哈希、槽位空标记歧义、开放寻址探测终止条件 |
| 测试覆盖要求 | 必须覆盖：单节点压缩路径、多线程并发插入同一前缀路径、LRU 淘汰与活跃引用冲突 | 必须覆盖：高负载扩容、hash 碰撞链路、并发删除与查询的时序 |

**复杂度差异的理性评估**：Radix Tree 代码量约为 Chunk-Hash 的 4 倍，但对有 C 语言分布式存储经验的团队并非不可逾越。更重要的是，Chunk-Hash 的"隐性复杂度"（并发扩容、rehash 竞态、删除标记与插入的时序边界）容易被低估，而这些正是 bug 高发区。Radix Tree 的复杂度是显性的，更易于系统性测试覆盖。

#### 7.3.4 决策标准与加权阈值

| 评估维度 | 权重 | Radix Tree 得分 | Chunk-Hash 得分 |
|----------|------|-----------------|-----------------|
| P99.9 延迟裕量 | 30% | 9/10 | 4/10 |
| 内存效率 | 25% | 8/10 | 5/10 |
| TTFT 收益 | 25% | 9/10 | 6/10 |
| 实现/维护可控性 | 20% | 6/10 | 7/10 |
| **加权总分** | 100% | **8.1** | **5.4** |

**决策结论**：L1 采用压缩 Radix Tree。失败回退条件：若真实 workload 前缀相似度 < 30%，或 4 周内原型未通过压力测试（P99.9 < 10us @ 10K QPS），则无条件回退为纯 Chunk-Hash。

### 7.4 最优场景与最差场景

**最优场景**：

- **Workload 具有强烈的前缀局部性**：例如多轮对话系统、Agent 工作流、Few-shot 学习。L1 Radix Tree 能命中 80% 以上的前缀请求，L2 几乎不被触发，TTFT 接近本地缓存的极限。
- **中等规模集群（10-50 节点）**：L2 的 Chunk-Hash 表和一致性哈希环在此规模下管理简单，节点变更频率低，层间同步开销可忽略。

**最差场景**：

- **前缀发散严重且请求完全随机**：例如独立单轮 QA，每个请求的 token 序列几乎不共享前缀。此时 L1 Radix Tree 退化为哈希表，内存占用高且命中率低，大部分查询落到 L2，查询延迟和层间同步开销凸显。
- **节点频繁加入/离开**：一致性哈希环需要频繁重新均衡，L2 索引处于持续不稳定状态，100ms 的最终一致窗口内可能出现大量定位失败或数据 miss。

### 7.4 未来脆弱点

当前设计的最大脆弱点在于**层间协同协议**。

L1 和 L2 是两个语义不同的索引结构（前缀树 vs chunk 哈希），当数据在本地和远程之间迁移时，必须同时更新两个索引层。如果在高并发场景下层间同步出现 bug（例如 L1 已标记某块为"本地可用"，但 L2 尚未完成降级确认），会导致短暂但难以复现的数据不一致。由于我们选择了最终一致性，这类问题不会立刻以错误结果的形式暴露，而是表现为"偶发性的缓存 miss"，调试和定位成本极高。

## 8. 风险点和缓解策略

### 高影响风险 1：Radix Tree 内存溢出导致 OOM

- **风险描述**：若 token 前缀共享度远低于预期，Radix Tree 内存可能迅速膨胀，触发 OOM。
- **可能性**：中。
- **影响**：高。Metadata 崩溃导致全部缓存查询失败。

**缓解策略**：
1. **硬性内存上限与水线**：L1 内存上限 5GB；80% 水线触发 LRU 淘汰，95% 强制暂停新插入。
2. **降级通道**：持续超限时新索引直入 L2，牺牲部分延迟保可用性。
3. **监控告警**：定期上报"节点数/叶节点数/内存占用"，内存增速 >100MB/分钟触发紧急告警。

### 风险 2：Chunk-Hash 碰撞导致错误命中

- **风险描述**：128 位 Hash 在极端规模下仍存在统计碰撞可能，collision 会导致加载错误 KV Cache。
- **可能性**：低。
- **影响**：高。

**缓解策略**：
1. **碰撞检测回退**：L2 返回命中时附带前 16 个 token 的校验和，Storage 加载前快速校验，不匹配即回退为 miss。
2. **定期扫描**：每周随机抽样比对 L2 条目与存储内容，发现碰撞标记为"脏"并异步重建。

### 风险 3：一致性哈希环的负载倾斜

- **风险描述**：热门 prompt 可能导致部分物理节点负载远高于其他节点，带宽饱和、尾延迟增加。
- **可能性**：中。
- **影响**：中。

**缓解策略**：
1. **CHWBL**：当候选节点负载超过"平均 × (1+ε)"时继续顺时针寻找，设 ε=0.1，控制最大偏差 ≤10%。
2. **热点多副本**：Heatmap 标记的热点 chunk 主动提升副本数（2→3-4），分散读取压力。
