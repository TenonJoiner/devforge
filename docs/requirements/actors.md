# KV Cache Offloading 存储系统 — Actor 识别的Actor列表

## 说明

本文档识别所有与系统发生交互的 Actor。Actor 划分为三类：人类（直接操作系统的用户）、外部系统（与系统进行数据或控制流交换的独立系统）、内部组件（从外部视角可感知的系统内部边界实体）。

文档纪律：不出现具体代码模块名（如 BlockAllocator、IndexService），只使用用户或外部系统可理解的角色名称。

---

## 一、人类 Actor

### 1. AI 应用开发者 (AI Application Developer)

| 属性 | 说明 |
|------|------|
| **类型** | 人类 |
| **职责** | 将 KV Cache Offloading 能力集成到 LLM 推理应用中；配置推理引擎以使用本系统；调优缓存策略以匹配应用特征 |
| **交互方式** | 通过推理引擎配置文件、SDK API、环境变量启用和配置缓存服务；阅读集成文档和 Quickstart |
| **关注点** | 易集成性、与 vLLM/SGLang 等引擎的兼容性、TTFT 降低效果、API 稳定性、文档清晰度 |

### 2. 平台工程师 / MLOps (Platform Engineer)

| 属性 | 说明 |
|------|------|
| **类型** | 人类 |
| **职责** | 在生产环境中部署、扩展、监控缓存服务；制定容量规划；处理日常运维和故障响应 |
| **交互方式** | 通过 Helm/K8s manifest 部署；通过 Prometheus/Grafana 监控；通过运维 CLI 执行扩容、缩容、升级 |
| **关注点** | 可观测性、K8s 原生支持、故障自动恢复能力、滚动升级无中断、容量规划公式、多租户隔离 |

### 3. 系统管理员 (System Administrator)

| 属性 | 说明 |
|------|------|
| **类型** | 人类 |
| **职责** | 管理底层硬件和操作系统环境；处理磁盘故障、网络配置、节点替换；执行安全策略 |
| **交互方式** | 通过 SSH/系统管理工具操作节点；通过日志和告警系统响应事件；通过 RBAC 控制访问权限 |
| **关注点** | 节点级故障诊断、数据完整性校验、安全合规（加密、访问控制）、硬件兼容性确认 |

## 二、外部系统 Actor

### 4. 推理引擎 (Inference Engine)

| 属性 | 说明 |
|------|------|
| **类型** | 外部系统 |
| **职责** | 执行 LLM 推理计算；在 prefill 阶段生成 KV Cache；在 decode 阶段加载复用 KV Cache；向缓存系统发起存储/检索请求 |
| **交互方式** | 通过标准化 Connector API 或 SDK 调用缓存服务：查询 prefix 匹配、启动异步加载、逐层同步、异步存储 |
| **关注点** | 最低侵入性集成、异步 API 零阻塞、层级别流水线支持、加载失败时的安全降级（fallback 到重算） |

### 5. 集群编排器 (Cluster Orchestrator)

| 属性 | 说明 |
|------|------|
| **类型** | 外部系统 |
| **职责** | 管理缓存服务实例的生命周期：调度 Pod/容器到节点、根据负载自动扩缩容、处理节点失效时的重新调度 |
| **交互方式** | 通过容器运行时接口（CRI）、健康检查探针（liveness/readiness）、资源指标（HPA/VPA）与本系统实例交互 |
| **关注点** | 健康检查响应及时性（<2s）、实例启动/关闭速度、状态持久化与编排器迁移的兼容性 |

### 6. 监控系统 (Monitoring System)

| 属性 | 说明 |
|------|------|
| **类型** | 外部系统 |
| **职责** | 采集缓存服务的运行时指标；生成告警和仪表盘；支撑容量规划和故障定位 |
| **交互方式** | 通过 Prometheus /metrics 端点拉取指标；通过 OpenTelemetry 或 StatsD 接收 trace/event |
| **关注点** | 指标覆盖完整性（延迟、吞吐、命中率、错误率、资源利用率）、指标采集开销低（<5% CPU）、标签维度清晰 |

### 7. 外部持久化存储 (External Persistent Storage)

| 属性 | 说明 |
|------|------|
| **类型** | 外部系统 |
| **职责** | 在缓存服务节点故障或数据淘汰后，作为更冷一层的持久化后端（如 S3、MinIO、DFS） |
| **交互方式** | 通过对象存储 API（S3 API）或文件系统接口与本系统进行大粒度数据交换 |
| **关注点** | 高可用性（至少 99.9%）、延迟上限可接受（P99 < 200ms）、带宽充足、与其他租户的 I/O 隔离 |

## 三、内部组件 Actor（外部可感知边界）

### 8. 缓存服务实例 (Cache Service Instance)

| 属性 | 说明 |
|------|------|
| **类型** | 内部组件 |
| **职责** | 运行在单个节点上的缓存服务进程，负责管理本节点的 HBM、DRAM、SSD 缓存资源；响应引擎的加载/存储请求；执行本地数据的晋升和淘汰 |
| **交互方式** | 对推理引擎暴露数据加载/存储接口；对集群管理平面暴露心跳和状态上报；对其他缓存实例暴露数据共享/迁移接口 |
| **关注点** | 本节点资源不超用、数据在本地各层级间高效迁移、故障时快速释放资源避免影响引擎 |

### 9. 全局元数据服务 (Global Metadata Service)

| 属性 | 说明 |
|------|------|
| **类型** | 内部组件 |
| **职责** | 管理集群级的 KV Cache 位置映射、prefix 索引的全局视图、实例加入/退出协调（在分布式部署模式下存在） |
| **交互方式** | 对缓存服务实例提供 segment/对象定位查询；接收实例的心跳和状态更新；在实例变更时重新分配元数据 |
| **关注点** | 查询低延迟（P99 < 5ms）、高可用性（避免单点故障）、最终一致性边界可控（不一致窗口 < 100ms） |

### 10. 数据传输服务 (Data Transfer Service)

| 属性 | 说明 |
|------|------|
| **类型** | 内部组件 |
| **职责** | 承担缓存数据在不同层级（GPU↔DRAM↔SSD）和不同节点间的实际数据搬运；封装零拷贝、RDMA、Pipeline 等传输优化 |
| **交互方式** | 被缓存服务实例调用执行异步数据迁移；向监控系统上报各路径的带宽和延迟指标 |
| **关注点** | 传输带宽高、CPU 开销低、错误可捕获、超时和降级路径明确、不同硬件能力（GDS/RDMA/TCP）的自动适配 |

## Actor → Feature Domain 映射

| Actor | 主要关注的 Feature Domain | 说明 |
|-------|---------------------------|------|
| AI 应用开发者 | `engine-integration`, `cache-offloading` | 关心集成难易度和缓存命中收益 |
| 平台工程师 / MLOps | `cluster-management`, `observability`, `data-lifecycle` | 关心部署、扩容、监控、容量规划和生命周期策略 |
| 系统管理员 | `cluster-management`, `observability` | 关心节点故障、数据完整性、安全和合规 |
| 推理引擎 | `cache-offloading`, `engine-integration`, `data-lifecycle` | 核心数据消费者和生产者 |
| 集群编排器 | `cluster-management` | 实例生命周期和弹性伸缩 |
| 监控系统 | `observability` | 指标采集和告警 |
| 外部持久化存储 | `data-lifecycle` | 冷数据归档和恢复 |
| 缓存服务实例 | `cache-offloading`, `data-lifecycle`, `cluster-management` | 本地缓存的存取、迁移和故障处理 |
| 全局元数据服务 | `cluster-management`, `cache-offloading` | 全局视图和一致性 |
| 数据传输服务 | `cache-offloading`, `data-lifecycle` | 实际数据搬运效率和可靠性 |
