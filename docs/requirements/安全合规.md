## Feature: 强制 TLS 传输加密

### Feature 价值论证

**用户痛点**：边缘侧传感数据在边云传输链路上以明文传输时存在被窃听或篡改的风险，部分行业监管要求（等保 2.0、电力二次安防规定）明确规定传输必须加密。

**现有方案不足**：可选式 TLS 配置依赖运维人员手动开启，实际部署中大量节点因配置遗漏或证书管理复杂而跳过加密；证书轮换需重启服务导致连接中断。

**本 Feature 价值**：在协议层强制 TLS，拒绝任何非加密连接，从架构上消除"配置遗漏导致明文传输"的可能性；支持证书在线热重载，轮换期间零中断。

**成功指标**：
- 所有节点间通信及客户端连接 100% 经过 TLS 加密
- 证书轮换期间服务中断时间 = 0
- 证书宽限期到期前 30min 内 100% 触发告警

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 平台运维工程师部署新节点，节点启动时自动加载 TLS 证书并仅接受加密连接 | 未配置有效证书的节点拒绝启动 | 自动化 |
| 2 | IoT 设备群通过 MQTT 写入，完成 TLS 握手后建立加密通道 | TLS 握手延迟 <= 50ms | 自动化 |
| 3 | 平台运维工程师执行证书在线轮换，宽限期配置为 12h | 轮换过程中活跃连接零中断，宽限期在 1h-24h 范围内可配置 | 自动化 |
| 4 | 证书宽限期到期前 30min，系统向平台运维工程师发送告警通知 | 告警在到期前 25-35min 内送达，告警渠道包含邮件和 webhook | 自动化 |
| 5 | 客户端尝试以明文方式连接 | 连接被重置，拒绝响应耗时 <= 10ms | 自动化 |
| 6 | 节点证书过期后新客户端连接被拒绝 | 过期证书连接失败率 100%，告警在过期后 <= 1min 触发 | 自动化 |
| 7 | 攻击者尝试 TLS 降级攻击 | 仅允许 TLS 1.3+，低版本协商 100% 被拒绝 | 自动化 |
| 8 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后 TLS 连接正常建立，证书配置不丢失 | 自动化 |
| 9 | 客户端不支持证书热重载（旧版本 SDK），在服务器证书轮换后 | 客户端在现有连接断开后使用新证书重新握手成功 | 自动化 |

## Feature: 身份认证与凭证管理

### Feature 价值论证

**用户痛点**：边云架构中接入方类型多样（IoT 设备、边缘应用、云端服务），缺乏统一认证机制时安全审计员无法统一审计"谁在什么时候访问了什么"。

**现有方案不足**：分散式认证导致凭证泄露后无法快速全局吊销；认证日志格式不统一；网络分区时新设备缺乏安全接入机制。

**本 Feature 价值**：提供统一认证入口，支持多种凭证类型，统一凭证生命周期管理；凭证吊销 30s 内全集群生效；网络分区下新接入设备 100% 拒绝。

**成功指标**：
- 凭证吊销后全集群生效时间 <= 30s
- 单节点认证吞吐 >= 10,000 次/秒
- 本地认证缓存 TTL <= 30s
- 网络分区场景下新接入设备拒绝率 100%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | IoT 设备群使用预分配 Token 向边缘节点发起写入请求 | 认证校验延迟 <= 1ms（本地缓存命中） | 自动化 |
| 2 | 应用服务集群使用 API Key 通过 gRPC 接入 | 首次认证延迟 <= 5ms，会话内免重复认证 | 自动化 |
| 3 | 运维工程师批量轮换一组边缘设备的 Token | 批量轮换 1000 台设备凭证 <= 10s | 自动化 |
| 4 | 安全审计员发现凭证泄露执行紧急吊销 | 吊销后目标凭证在所有可达节点 <= 30s 内失效 | 自动化 |
| 5 | 认证服务不可用时边缘节点使用本地缓存继续验证已知设备 | 本地缓存 TTL <= 30s；已缓存凭证的设备写入不受影响；未缓存的新设备被拒绝 | 自动化 |
| 6 | 网络分区期间新设备尝试接入边缘节点 | 分区下新接入设备 100% 被拒绝，已有会话设备在缓存有效期内继续服务 | 自动化 |
| 7 | 攻击者使用过期或伪造 Token 尝试接入 | 无效凭证 100% 被拒绝，审计日志记录来源 IP 和时间戳 | 自动化 |
| 8 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后已有凭证继续有效，新凭证签发正常 | 自动化 |
| 9 | 节点故障重启后认证状态恢复 | 重启后 <= 5s 内恢复认证服务，已签发凭证无需重新注册 | 自动化 |

## Feature: 基于角色的访问控制（RBAC）

### Feature 价值论证

**用户痛点**：企业客户的时序数据库服务多个业务团队，缺乏细粒度权限控制时要么全员最高权限（安全风险），要么依赖应用层自行实现（重复开发、标准不一）。

**现有方案不足**：开源版仅提供认证不提供授权，企业客户被迫在应用层叠加权限逻辑；权限变更存在传播延迟窗口，期间可能产生脏权限。

**本 Feature 价值**：在数据库引擎层提供角色-权限映射，支持数据库级和表级的读/写/管理权限控制；脏权限窗口期间越权写入不持久化；与 TLS+加密叠加后累计性能开销可控。

**成功指标**：
- 权限校验延迟 <= 0.1ms
- 角色权限变更后全集群生效时间 <= 5s
- 单独启用 RBAC 时写入吞吐下降 <= 2%
- 与 TLS+静态加密叠加启用时累计写入吞吐下降 <= 15%
- 与 TLS+静态加密+敏感标签（L3）四者叠加启用时累计写入吞吐下降 <= 20%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 安全审计员创建"只读分析师"角色并授予 SELECT 权限 | 绑定用户执行写入返回权限拒绝，响应时间 <= 1ms | 自动化 |
| 2 | 安全审计员为"设备写入"角色授予指定表的 INSERT 权限 | 写入目标表成功，未授权表返回拒绝，权限校验影响 <= 0.1ms | 自动化 |
| 3 | 安全审计员修改某角色权限（收回写权限） | 变更后 <= 5s 内所有节点对该角色写请求返回拒绝 | 自动化 |
| 4 | 用户尝试访问未授权的数据库或表 | 越权请求 100% 被拦截，审计日志记录 | 自动化 |
| 5 | 角色权限变更传播窗口期间，用户执行越权写入 | 脏权限窗口期间越权写入不持久化，变更传播完成后数据状态正确 | 自动化 |
| 6 | 角色继承：父角色权限收缩后子角色同步收缩 | 父角色变更后 <= 5s 子角色权限同步更新 | 自动化 |
| 7 | 高并发写入场景下同时启用 TLS+静态加密+RBAC | 累计写入吞吐下降 <= 15%，各功能独立启用时下降分别 <= 10% | 自动化 |
| 7a | 高并发写入场景下同时启用 TLS+静态加密+RBAC+敏感标签（L3） | 四者叠加累计写入吞吐下降 <= 20% | 自动化 |
| 8 | 网络分区期间节点继续执行权限校验 | 分区节点使用本地权限副本继续校验，分区恢复后 <= 5s 同步最新权限 | 自动化 |
| 9 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后角色权限映射不丢失，权限校验正常执行 | 自动化 |
| 10 | 用户同时受 RBAC 角色限制和敏感标签（L3）限制，RBAC 允许但标签拒绝 | 评估顺序：先 RBAC 后敏感标签；RBAC 拒绝时直接返回，标签不再评估；RBAC 通过但标签拒绝时返回"敏感数据访问受限" | 自动化 |

## Feature: 数据保留策略合规执行

### Feature 价值论证

**用户痛点**：传感数据同时受 TTL 清理策略和法规合规要求的双重约束，TTL 触发清理时若无合规拦截，受法定保护的数据（如事故调查期数据、司法保全数据）会被静默删除。

**现有方案不足**：常规 TTL 机制是"到期即删"，不具备按数据段标记合规保留的能力；合规策略服务不可用时缺乏安全降级机制。

**本 Feature 价值**：在 TTL 清理流程中引入合规策略检查点，命中合规保留规则时跳过删除并记录审计事件；明确合规 vs TTL vs GDPR 三方优先级；策略服务不可用时采取零删除保守策略。

**成功指标**：
- 受合规保护的数据在 TTL 到期后 0 条被误删
- 合规拦截事件从产生到可审计查询延迟 <= 5s
- 策略服务不可用时零数据删除

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | TTL 清理触发器检测到数据段命中合规规则，跳过删除并生成审计事件 | 命中规则的数据段删除被拦截，审计日志 5s 内可查 | 自动化 |
| 2 | 安全审计员查询因合规策略被保留的数据段清单 | 返回结果包含保留原因和时间信息，响应 <= 2s | 自动化 |
| 3 | 数据工程师新增合规保留规则，规则对已存在的过期数据立即生效 | 下一次清理周期内命中新规则的数据不被删除 | 自动化 |
| 4 | 数据段同时命中合规保留、TTL 到期、GDPR "被遗忘权"请求 | 三方优先级：合规保留 > GDPR 被遗忘权 > TTL 清理（需待澄清 #4 确认 GDPR 优先级） | 自动化 |
| 5 | 合规策略检查服务异常时 TTL 清理采取保守策略 | 策略检查异常期间零数据删除，产生告警 | 自动化 |
| 6 | 存储空间因合规保留数据累积接近耗尽 | 系统提前 24h 发出存储耗尽预警，提供手动释放合规保留数据的审批流程 | 自动化 |
| 7 | 合规保留规则到期后数据段自动进入正常 TTL 清理流程 | 规则到期后下一个清理周期内数据被正常删除 | 自动化 |
| 8 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后合规保留规则不丢失，TTL 清理继续受合规拦截 | 自动化 |
| 9 | 网络分区期间边缘节点执行本地 TTL 清理 | 分区节点本地缓存合规规则继续生效，无缓存时采取零删除策略 | 自动化 |

## Feature: 操作审计日志

### Feature 价值论证

**用户痛点**：等保三级、ISO 27001 等合规审计中需要提供完整的数据库操作记录，当前缺乏系统化审计能力导致审计取证耗时数小时甚至数天。

**现有方案不足**：OS syslog 只记录进程级事件，无法区分数据库内部操作；应用层埋点覆盖率无法保证；缺乏与 SIEM 系统的标准化对接。

**本 Feature 价值**：在数据库引擎层内置审计日志，自动捕获管理面和关键数据面操作，形成不可篡改的审计轨迹；支持 SIEM 标准化对接；定义吞吐上限。

**成功指标**：
- 审计日志覆盖率：管理面操作 100% 记录，关键数据面操作（DELETE/DROP/TRUNCATE/Schema 变更/权限变更）100% 记录，L3-机密级别数据的 SELECT 查询 100% 记录
- 审计取证效率：查询到完整操作轨迹 <= 30s
- 审计日志吞吐上限 >= 50,000 条/秒

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 安全审计员按时间+操作者+操作类型查询审计日志 | 10 万条日志范围内返回 <= 30s，包含全部必填字段 | 自动化 |
| 2 | 管理员执行权限变更后审计日志确认记录 | 操作到日志可查延迟 <= 5s，包含变更前后状态 | 自动化 |
| 3 | 管理员执行 DELETE/DROP/TRUNCATE 操作 | 关键数据面操作 100% 记录，包含操作对象、影响行数、执行计划 | 自动化 |
| 4 | 管理员执行 Schema 变更（CREATE/ALTER/DROP TABLE） | Schema 变更操作 100% 记录，包含变更前后 DDL | 自动化 |
| 5 | 用户执行 L3-机密级别数据的 SELECT 查询 | 查询主体、对象、时间戳 100% 记录，审计日志 5s 内可查 | 自动化 |
| 6 | 高写入负载期间管理员执行批量删除，审计日志吞吐达 50,000 条/秒 | 审计事件丢失率 = 0，审计日志写入不导致数据面吞吐下降超 5% | 自动化 |
| 7 | 审计日志通过 Syslog/CEF 协议实时推送至 SIEM 系统 | SIEM 侧接收延迟 <= 5s，事件格式符合 CEF 标准 | 自动化 |
| 8 | 审计日志存储空间达到配额上限 | 系统 <= 10s 内发出告警，已有日志零丢失，新日志继续写入环形缓冲 | 自动化 |
| 9 | 安全审计员对审计日志执行完整性校验 | 任意单条记录被篡改后 <= 1s 内检测到，误报率 = 0 | 自动化 |
| 10 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后审计日志不丢失，审计功能正常运作 | 自动化 |
| 11 | 审计日志子系统自身故障时数据面操作不受影响 | 审计子系统故障期间数据面读写成功率 >= 99.9%，故障恢复后审计日志无丢失 | 自动化 |

## Feature: 静态数据加密

### Feature 价值论证

**用户痛点**：边缘侧物理服务器缺乏数据中心级物理安全防护，硬盘被盗或设备遗失时明文数据可被直接读取，造成数据泄露。

**现有方案不足**：OS 级全盘加密在无人值守场景需预置密钥文件，粒度过粗无法按敏感等级差异化；历史数据加密迁移需停机。

**本 Feature 价值**：在存储引擎层实现文件级透明加密，加密对查询透明；支持在线迁移历史数据；迁移过程崩溃后无半加密文件残留。

**成功指标**：
- 加密启用后写入吞吐量下降 <= 10%
- 加密数据文件在无密钥情况下熵值 >= 7.99 bits/byte
- 在线迁移 1TB 数据 <= 12h
- 迁移崩溃恢复后无半加密文件
- 进程终止后内存中明文数据 <= 5s 内清零

## Feature: 密钥管理与 KMS 集成

### Feature 价值论证

**用户痛点**：企业安全策略要求密钥定期轮换、使用可审计、主密钥不落地存储在业务系统中；边缘节点网络不稳定时需有降级策略。

**现有方案不足**：数据库自行管理密钥存在密钥与数据同机存储、轮换需停机、缺乏审计链等风险；KMS 长期不可用时系统行为不明确。

**本 Feature 价值**：采用信封加密架构，数据密钥由数据库管理，主密钥由外部 KMS 托管；支持多 KMS 兼容（Vault/AWS KMS/阿里云 KMS）；KMS 长期不可用时按租户策略可选拒写模式或本地保护模式。

**成功指标**：
- 密钥轮换期间读写请求成功率 >= 99.99%
- KMS 不可用时系统依赖本地缓存可持续正常读写 >= 4 小时（边缘节点 24h）
- 多 KMS 兼容：Vault/AWS KMS/阿里云 KMS 均通过兼容性验证

## Feature: 多租户数据隔离

### Feature 价值论证

**用户痛点**：云端部署模式下多个客户共享同一集群，客户间数据必须严格隔离；共享资源争用可能通过时延侧信道泄露信息。

**现有方案不足**：物理隔离成本过高；逻辑隔离仅依赖应用层过滤存在 SQL 注入或 Bug 导致跨租户泄露的风险；元数据侧信道（错误码、时延、EXPLAIN 输出）未做防护。

**本 Feature 价值**：在存储引擎层实现租户命名空间隔离，即使绕过应用层也无法跨租户读取；拦截元数据侧信道；支持物理隔离与逻辑隔离可选；删除租户后设置隔离期防止命名空间复用攻击。

**成功指标**：
- 跨租户数据访问尝试 100% 被拦截
- 元数据侧信道（错误码/时延/EXPLAIN）泄露拦截率 100%
- 单租户资源超配额时其他租户写入延迟波动 <= 5%
- 租户删除后命名空间 1h 内不可复用

## Feature: 降采样策略的数据精度治理

### Feature 价值论证

**用户痛点**：数据分析师查询历史数据时无法区分返回结果是原始精度还是降采样聚合精度，可能将平滑效应误判为"无异常"，导致错误决策；合规审计中审计员需要复核分析报告的精度声明。

**现有方案不足**：降采样执行后聚合结果与原始数据存储在同一查询路径中，缺乏精度元数据标记；多级降采样链路无法追溯。

**本 Feature 价值**：本 Feature 为**跨域共享 Feature**——核心域归属为数据生命周期管理（降采样策略执行），安全合规域关注其合规属性：审计员需复核分析报告精度声明、精度元数据是合规审计的必要证据链。安全合规视角要求：精度元数据必须持久化存储不可篡改、精度违规必须触发审计事件、多级追溯链路必须完整。为每条降采样结果自动附加精度元数据；支持多级降采样链路追溯。

**成功指标**：
- 精度标记覆盖率：所有降采样产出数据 100% 携带精度元数据
- 精度违规检出率 100%，误报率 <= 1%
- 多级降采样链路可追溯：从最终聚合结果追溯到原始数据源 <= 3 跳

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师创建降采样策略后系统自动为产出数据附加精度元数据 | 降采样产出 100% 携带完整精度元数据（原始区间、聚合函数、采样间隔） | 自动化 |
| 2 | 数据工程师定义精度治理规则，不满足规则的查询收到警告 | 违规查询 100% 触发警告，查询仍返回结果 | 自动化 |
| 3 | 安全审计员审查某查询记录的数据精度 | 可通过查询 ID 追溯到精度元数据，响应 <= 5s | 自动化 |
| 4 | 安全审计员复核分析报告的精度声明 | 报告中的精度声明与系统元数据 100% 一致，不一致项自动标红 | 自动化 |
| 5 | 降采样任务异常导致部分数据未携带精度标记 | 无标记数据在查询结果中标记为"精度未知"，审计日志记录异常事件 | 自动化 |

## Feature: 敏感数据分类与标签

### Feature 价值论证

**用户痛点**：工业时序数据中混合了不同敏感等级的信息（如设备运行参数、工艺配方、客户隐私数据），缺乏分类机制时所有数据采用相同访问控制策略，无法实施差异化保护。

**现有方案不足**：应用层实现数据分类分散在各业务系统中，数据库层面无感知无法实施差异化控制；敏感等级降级缺乏审批管控。

**本 Feature 价值**：在数据库元数据层引入 L1/L2/L3 三级敏感等级标签体系，访问控制引擎基于标签实施差异化策略；敏感等级降级需数据工程师 + 安全审计员双人审批，24h 时效失效，单人降级 100% 拦截。

**成功指标**：
- 标签变更后访问控制策略生效延迟 <= 30s
- 安全审计员可在 <= 5s 内获取全库敏感数据分布报告
- 单人敏感等级降级 100% 被拦截
- 双人审批降级 24h 后自动失效

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师为某列设置敏感等级标签 L3-机密 | 无对应权限的用户查询该列返回拒绝 | 自动化 |
| 2 | 安全审计员查询全库敏感数据分布报告 | 报告 5s 内返回，包含各等级（L1/L2/L3）的表数、列数、数据量 | 自动化 |
| 3 | 数据工程师将某列敏感等级从 L2 提升到 L3 | 变更后 <= 30s 内新策略生效 | 自动化 |
| 4 | 用户执行跨表联合查询涉及不同敏感等级列 | 按最高敏感等级（L3）实施访问控制 | 自动化 |
| 5 | 数据工程师发起敏感等级从 L3 降级到 L2，安全审计员审批通过 | 双人审批后降级生效，审批记录审计日志留存 >= 180 天 | 自动化 |

| 8 | 攻击者尝试直接修改元数据表绕过标签系统 | 元数据表修改 100% 被拦截，触发安全告警 | 自动化 |
| 9 | 平台运维工程师执行版本升级后回滚到旧版本 | 回滚后敏感标签不丢失，访问控制策略继续生效 | 自动化 |
