#### HIGH（3 个，共 9 分）

**H1. Feature 2 成功指标中"磁盘 IO 带宽饱和时前台写入劣化 <= 20%"缺乏基线定义**
- **所属维度**: Feature 价值 / 验收标准量化
- **问题位置**: Feature 2（节点动态增删与数据再平衡）成功指标第 3 条
- **理由**: "劣化 <= 20%"是相对值，但未定义基线是什么（正常 P99 延迟是多少？）。如果正常 P99 是 10ms，劣化 20% 是 12ms；如果是 1s，劣化 20% 是 1.2s。基线缺失导致验收时无法判定是否达标。此外，该指标与 Feature 2 场景 8 的验收标准重复，但 Feature 6 场景 7 也引用了同一阈值，跨 Feature 指标一致性需确认。
- **修复建议**: 明确基线延迟值（如"基线 P99 < 100ms 时，再平衡期间 P99 < 120ms"），或在非功能需求中统一定义基线。

**H2. Feature 3 的"网络分区期间边缘节点继续本地写入"场景与边缘自治域边界模糊**
- **所属维度**: 场景覆盖 / 跨域依赖
- **问题位置**: Feature 3 场景 8；跨域依赖声明段落
- **理由**: 场景 8 的验收标准包含"分区期间本地写入不丢失"，但跨域依赖声明明确说"分区期间本地写入策略由边缘自治域定义"。这意味着 Feature 3 的验收标准实际上依赖另一个域的实现，但验收标准中没有标注这种跨域依赖关系。如果边缘自治域未实现或实现不满足要求，Feature 3 的验收将失败，责任边界不清。
- **修复建议**: 在场景 8 的验收标准中增加跨域依赖标注，如"（依赖边缘自治域本地写入策略实现）"，或将该场景的验收标准调整为仅验证"分区恢复后同步触发"和"同步完成后完整性"，将本地写入不丢失的验收留给边缘自治域。

**H3. Feature 5 灰度发布场景 2 的验收标准"仅接收 <= 配置比例的读流量"不完整**
- **所属维度**: 场景覆盖 / 验收标准可测试性
- **问题位置**: Feature 5 场景 2
- **理由**: 灰度节点的流量控制只提到"读流量"，未提及写流量。在滚动升级场景中，升级节点是否接收写入？如果不接收写入，那写入如何路由？如果接收写入，灰度比例是否也控制写入？这直接影响升级期间的数据一致性风险。此外，"配置比例"未给出默认值或典型值范围，验收时缺乏参照。
- **修复建议**: 明确灰度节点对写流量的处理方式（完全拒绝写入 / 按比例接收 / 全部接收但标记为实验流量），并在验收标准中补充写流量策略。

#### MEDIUM（5 个，共 5 分）

**M1. Feature 1 成功指标"故障检测误判率 <= 0.1%/月"难以在开发阶段验证**
- **所属维度**: Feature 价值 / 成功指标可验证性
- **问题位置**: Feature 1 成功指标第 2 条
- **理由**: 0.1%/月 的误判率需要至少运行数月才能统计出可信数据，无法在迭代验收或 CI 中验证。该指标更适合作为上线后的运营指标，而非开发阶段的验收标准。作为产品级需求，应区分"开发可验收指标"和"上线运营指标"。
- **修复建议**: 补充一个开发阶段可验证的替代指标，如"连续 1000 次模拟故障注入中误判次数 = 0"，保留 0.1%/月 作为上线运营目标。

**M2. 跨 Feature 复合场景与 Feature 内场景存在重复，但未明确关系**
- **所属维度**: 场景覆盖 / 文档一致性
- **问题位置**: 跨 Feature 复合场景 #3 与 Feature 3 场景 9；复合场景 #4 与 Feature 5 场景 9 和 Feature 8 场景 7
- **理由**: 复合场景 #3（注册后配置下发）与 Feature 3 场景 9 描述几乎相同；复合场景 #4 与 Feature 5 场景 9 和 Feature 8 场景 7 也高度重叠。重复场景如果不标注关系，会导致特性级开发时重复设计测试用例，或产生不一致的验收标准（如 Feature 3 场景 9 要求"<= 10s"，复合场景 #3 也是"<= 10s"，但 Feature 7 场景 1 要求"<= 30s"——三者涉及同一链路但时效要求不一致）。
- **修复建议**: 在复合场景表格中增加"关联 Feature 场景"列，标注每个复合场景由哪些 Feature 场景组合而成；或删除与 Feature 内场景完全重复的复合场景，仅保留真正需要跨 Feature 协同验证的复合场景。

**M3. Feature 6 资源配额粒度停留在"节点级"，未覆盖租户级/查询级**
- **所属维度**: Feature 价值 / 用户假设质疑
- **问题位置**: Feature 6（节点级资源配额与过载保护）整体
- **理由**: 文档标题和场景都限定在"节点级"配额，但 product-spec.md 中安全合规域有"多租户数据隔离"（P2 企业版）Feature。如果集群管理域只提供节点级配额，多租户场景下同一节点上的不同租户无法隔离资源，一个租户的流量风暴会影响其他租户。这是真实痛点——运维工程师需要的不只是节点级保护，还需要在共享节点上保护不同业务线/租户。当前设计可能低估了多租户场景的资源隔离需求。
- **修复建议**: 在 Feature 6 中增加一个场景或备注，说明节点级配额与多租户配额的关系：是"节点级为底层，租户级在上层实现"还是"本 Feature 仅覆盖节点级，租户级由其他域负责"。

**M4. Feature 7 配置下发"版本历史保留数量上限"待澄清问题标记为 LOW 优先级不合理**
- **所属维度**: 场景覆盖 / 故障模式
- **问题位置**: 已知待澄清问题 #6
- **理由**: 配置版本历史保留数量直接影响回滚能力。如果保留数量过少（如只保留 1 个版本），一旦连续两次配置错误就无法回滚到正确版本；如果保留在云端但边缘节点长期离线，回滚时可能无法获取历史版本。该问题与 Feature 7 场景 3（一键回滚）和场景 7（重启循环自动回滚）的可靠性直接相关，标记为 LOW 可能导致实现阶段被忽略，进而影响生产环境的配置安全。
- **修复建议**: 将优先级从 LOW 提升至 MEDIUM 或 HIGH，并在待澄清问题中增加"最小保留版本数"的候选建议（如至少保留 5 个版本或 30 天历史）。

**M5. 故障模式覆盖矩阵中 Feature 4（容器编排平台生命周期）的"节点故障"列引用场景 #4、#5，但 #5 是探针超时而非节点故障**
- **所属维度**: 核心表格针对性
- **问题位置**: 故障模式覆盖矩阵 Feature 4 行
- **理由**: 场景 #5 描述的是"节点内部 OOM 或死锁导致 liveness 探针超时"，这属于"资源耗尽"或"进程故障"，而非严格意义上的"节点故障"（节点硬件/网络完全不可用）。将 #5 归入"节点故障"列会误导特性级设计时对该故障模式的覆盖评估。Feature 4 真正覆盖的"节点故障"场景只有 #4（SIGKILL 场景），但 #4 更偏向"优雅关闭失败"而非"节点故障"。
- **修复建议**: 将 Feature 4 的"节点故障"列调整为仅引用 #4，并在备注中说明 Feature 4 主要覆盖容器生命周期信号响应，节点故障检测由 Feature 1 覆盖；或将 #5 移至"资源耗尽"列（已有一个 ✅）。

#### LOW（2 个，共 0.2 分）

**L1. Feature 8 场景 9 的"部署回滚"术语与 Feature 5 的"升级回滚"术语不一致**
- **所属维度**: 命名一致性
- **问题位置**: Feature 8 场景 9（"部署回滚"）vs Feature 5 场景 4/7/8（"升级回滚"）
- **理由**: Feature 8 场景 9 描述的是"取消部署后已完成的节点自动清理"，这实际上是"部署取消/清理"而非"回滚"（回滚通常指恢复到之前的状态）。术语混用可能导致特性级设计时对"回滚"语义理解不一致。
- **修复建议**: 将 Feature 8 场景 9 的"部署回滚"改为"部署取消清理"或"部署撤销"，与 Feature 5 的"升级回滚"区分。

**L2. 本域验收标准汇总表中缺少"配置下发成功率 >= 99.5%"**
- **所属维度**: 本域验收标准完整性
- **问题位置**: 本域验收标准表格
- **理由**: Feature 7 成功指标中定义了"配置下发成功率 >= 99.5%"，但本域验收标准汇总表（跨 Feature 核查表）中未包含该项。汇总表的目的是跨 Feature 核查，遗漏该指标会导致验收时忽略配置下发的可靠性。
- **修复建议**: 在本域验收标准表格中增加一行："配置下发成功率 | >= 99.5%（100 节点规模）| 自动化"。

#### HIGH（4 个，共 12 分）

**H1. Feature 1 与 Feature 5 的副本切换语义在滚动升级场景下存在时序冲突**
- **所属维度**: 跨 Feature 一致性 / 时序耦合
- **问题位置**: Feature 1 场景 9（计划内维护触发副本切换）与 Feature 5 场景 1（滚动升级）
- **理由**: Feature 1 场景 9 定义"升级暂停时间 <= 10s"，Feature 5 场景 1 定义"升级全程写入成功率 >= 99.9%"，但两者未定义交互时序。滚动升级时，旧实例先接收 SIGTERM（Feature 4 场景 1），然后 Feature 1 的副本切换被触发——如果切换耗时接近 30s RTO 上限，叠加升级暂停 10s，实际不可用窗口可能远超 99.9% 可用性所隐含的 43.2s/月（99.9% ≈ 43.2s 不可用/月）。更关键的是，Feature 5 场景 6 说"升级进行到一半时另一个未升级节点意外宕机，升级流程自动暂停"，但 Feature 1 场景 1 说"节点崩溃后自动检测并切换"——两者对"自动"的触发条件和优先级未协调：是优先完成副本切换再暂停升级，还是暂停升级再执行切换？这个时序决策直接影响数据一致性风险。
- **修复建议**: 在跨 Feature 复合场景 #1 中明确"副本切换优先于升级暂停"的优先级，并补充复合验收标准："副本切换完成前升级流程保持阻塞状态，切换完成后 <= 5s 内恢复升级"。同时验证 RTO 30s + 恢复 5s 的串联时序是否与 99.9% 可用性兼容。

**H2. Feature 2 数据再平衡与 Feature 6 资源配额的交互存在隐性绕过风险**
- **所属维度**: 跨 Feature 一致性 / 资源竞争
- **问题位置**: Feature 2 场景 8（磁盘 IO 饱和竞争）与 Feature 6 场景 7（再平衡与前台写入同时触发配额）

**H3. 边缘节点与云端节点的 NFR 规模目标存在数量级断层，架构推导困难**

**H4. 配置下发（Feature 7）与滚动升级（Feature 5）的协调缺少一致性边界定义**
- **所属维度**: 跨 Feature 一致性 / 状态机耦合
- **问题位置**: Feature 7 场景 8（升级期间修改配置）与 Feature 5 场景 1-9
- **理由**: Feature 7 场景 8 说"配置变更在升级暂停期间下发，升级恢复后新配置生效"，但未定义"升级暂停期间"的状态机边界。滚动升级涉及多节点逐一轮换，每个节点有自己的升级窗口——配置下发是全局操作还是按节点操作？如果是全局操作，在节点 A 升级暂停期间下发配置，节点 B（未升级）是否立即生效？如果是按节点操作，配置版本如何在不同节点间保持一致？此外，Feature 7 场景 3（一键回滚）与 Feature 5 场景 7（升级回滚）同时涉及"回滚"语义，但一个是配置回滚、一个是版本回滚，两者并发时的优先级未定义（如运维同时触发配置回滚和升级回滚）。

### 三、隐性假设识别

**假设 A1: 副本切换与升级流程的协调由单一协调器串行调度**
- **位置**: 跨 Feature 复合场景 #1、Feature 5 场景 6
- **内容**: 文档假设存在一个全局协调器，能在故障检测触发时自动暂停升级、在升级期间自动阻塞副本切换。但这个协调器本身是高风险单点——如果协调器故障，副本切换和升级可能并发执行，导致数据不一致。
- **验证方法**: 特性级设计时需明确协调器的实现（是嵌入现有一致性协议还是独立组件）、自身的故障转移机制。
- **验证失败的 fallback**: 如果协调器无法做到高可用，需降级为人工确认模式（运维工程师手动审批切换/升级的并发执行）。

**假设 A2: 边缘节点的本地写入缓冲区在分区期间不会溢出**
- **位置**: Feature 3 场景 8、跨域依赖声明
- **内容**: 文档假设网络分区期间边缘节点可以继续本地写入且"不丢失"，但未定义缓冲区上限。边缘自治域的 Scenario 4 提到"存储达阈值前写入成功率 >= 99.99%"，但集群管理域未引用此约束。如果分区持续时间超过缓冲区容量（如 7 天离线 + 高频写入），数据必然丢失。
- **验证方法**: 需明确集群管理域与边缘自治域在"本地写入不丢失"承诺上的责任边界——集群管理域负责"分区恢复后同步触发"，边缘自治域负责"分区期间本地写入持久化"。
- **验证失败的 fallback**: 在 Feature 3 场景 8 的验收标准中增加"在边缘自治域定义的缓冲区容量范围内"的前置条件。

**假设 A3: 配置下发的"全部目标节点生效"包含边缘节点和云端节点**
- **位置**: Feature 7 成功指标、场景 1
- **内容**: Feature 7 的"100 节点 <= 30s"隐含假设所有节点类型（云端+边缘）使用同一配置下发通道和时效标准。但边缘节点网络不稳定，30s 全量生效可能比云端困难得多。文档未区分边缘和云端的配置下发路径差异。
- **验证方法**: 特性级设计时需验证边缘节点的配置下发是否走边云协同通道（异步、可能延迟）还是独立通道（如 MQTT/CoAP）。
- **验证失败的 fallback**: 将配置下发时效按节点类型拆分（云端 <= 10s，边缘 <= 60s），或在成功指标中增加"网络正常时"的前提条件。

### 四、遗漏 Trade-off 补充

**Trade-off T1: 故障检测灵敏度 vs 误判率 vs 网络抖动容忍**
- **遗漏位置**: Feature 1 成功指标、Feature 3 场景 5
- **当前文档**: 仅定义了目标值（误判率 <= 0.1%/月，连续 3 个周期判定故障），未分析三者的 trade-off 关系。
- **补充分析**: 心跳周期缩短（2s→1s）可提升故障检测灵敏度、降低 RTO，但会增加网络抖动误判（尤其边缘节点 10s 周期下误判率可能更高）。连续判定阈值增加（3→5）可降低误判，但会延长 RTO。文档未说明这些参数的调整对整体可用性的影响——如果运维将心跳周期调至 1s 以追求更快 RTO，0.1%/月 的误判率目标是否仍然可达？建议增加备注："心跳周期和判定阈值的调整会影响 RTO 和误判率的 trade-off，具体参数组合需在特性级架构评审中验证"。

**Trade-off T2: 再平衡速度 vs 前台写入质量 vs 集群可用容量**
- **遗漏位置**: Feature 2 场景 1-9、Feature 6 场景 7
- **当前文档**: 分别定义了再平衡期间可用性 >= 99.9% 和前台写入劣化 <= 20%，但未分析三者的资源竞争关系。
- **补充分析**: 再平衡速度越快，前台写入劣化越大（IO 竞争）；再平衡速度越慢，集群处于"非均衡"状态的时间越长，可用容量降低（某些节点过载）。在 3 节点最小集群中，再平衡期间如果一个节点故障，剩余两节点可能无法维持最小副本数——文档未定义"再平衡期间集群的故障容忍度降级"。建议增加场景或备注："再平衡期间集群的故障容忍度临时降级（如从 1 节点降级为 0 节点），再平衡完成后恢复"。
