### R2 新发现问题清单

#### HIGH-1（维度：Actor 遗漏 — 数据分析师）
- **位置**：涉及 Actors 列表（第 25-33 行）、降采样 Feature 全部 Scenario
- **问题**：`数据分析师`作为 product-spec.md 中定义的高权力高利益 Actor，在本域完全缺失。降采样直接影响数据分析师的查询结果——原始数据 vs 聚合数据的可视化差异、聚合精度对趋势判断的影响、查询时自动切换数据源的透明度等，均未从数据分析师视角定义验收标准。当前文档仅关注"系统正确执行"，忽视了"用户正确感知"。
- **理由**：数据分析师是核心利益相关方。降采样后，分析师查询历史时间范围时，系统返回的是聚合数据而非原始数据，这一行为变化必须被分析师理解和接受。若缺少分析师视角的 Scenario，可能导致上线后用户投诉"历史数据精度变低了但系统没有提示"。
- **修正建议**：补充至少 1 个数据分析师视角的 Scenario，例如"数据分析师查询已降采样的时间范围时，查询结果自动返回聚合数据并标注精度级别与原始数据可用性状态"，验收标准量化：聚合数据查询延迟 P95 <= 原始数据查询延迟的 120%，精度标注准确率 = 100%。

#### HIGH-2（维度：Feature 价值 — 降采样 ROI 未量化）
- **位置**：降采样 Feature 价值论证（第 71-84 行）
- **问题**：降采样的核心价值主张是"降低存储成本"（痛点中"存储成本每月增长 30%+"），但成功指标中未包含任何与"成本节省"相关的可衡量指标。三个成功指标（及时性、吞吐影响、兼容性）均与成本无关。
- **理由**：Feature 上线后无法验证 ROI 是产品管理的大忌。同时，"30%+"的增长率缺乏行业基准支撑——VictoriaMetrics 的压缩比可达 10-20x，若本系统已采用列式压缩，实际增长率可能远低于 30%，该数字可能是主观臆断。
- **修正建议**：在成功指标中补充存储成本相关指标，例如"降采样后历史数据（>30 天）存储量相对于保留原始数据减少 >= 80%"或"单 TB 月度存储成本下降 >= 70%"，并明确测量口径（含压缩后的实际磁盘占用）。

#### MEDIUM-1（维度：场景覆盖 — TTL 清理暂停/恢复运维场景缺失）
- **位置**：TTL Scenario 清单（第 53-66 行）
- **问题**：缺少运维场景下暂停/恢复 TTL 清理的 Scenario。实际运维中，SRE 经常需要在合规审计、重大故障排查、数据恢复验证等时段暂停清理。
- **理由**：若文档不定义暂停/恢复行为，实现阶段可能遗漏该能力，导致审计期间数据被意外清理引发合规风险，或故障排查期间清理干扰诊断。暂停期间的过期数据积压如何在恢复后处理，也需要定义。
- **修正建议**：补充 Scenario"运维 SRE 在审计期间暂停 TTL 清理，审计结束后恢复执行"，验收标准：暂停期间零清理触发；恢复后 48h 内完成积压过期数据清理；暂停/恢复操作 100% 记入审计日志。

#### MEDIUM-2（维度：验收标准量化 — "空间回收率 >= 95%"测量口径模糊）
- **位置**：TTL Feature 成功指标（第 49 行）、本域验收标准（第 120 行）、Scenario #1（第 57 行）
- **问题**："空间回收率 >= 95%"的测量口径未定义，存在多种解读：(a) 逻辑删除记录数 / 应删除记录数，(b) 实际释放磁盘空间 / 过期数据占用空间，(c) 考虑压缩后的有效空间释放。
- **理由**：不同存储引擎（LSM-tree vs B-tree）的物理删除延迟不同。若采用 LSM-tree，compaction 是异步的，"24h 内清理完成"与"空间回收率 95%"在物理层面存在时间窗口冲突，验收时必然产生争议。
- **修正建议**：明确空间回收率定义口径，例如"逻辑删除的数据记录数 / 应删除的记录数 >= 95%，物理空间释放由存储层 compaction 异步完成，在 compaction 完成后 7 天内达到 90%"；或拆分为两个独立指标：逻辑清理率 + 物理回收率。

#### MEDIUM-3（维度：场景覆盖 — 降采样聚合数据 TTL 未定义）
- **位置**：降采样 Feature 全部 Scenario
- **问题**：降采样产生的聚合数据（1min 精度、1h 精度）本身的保留策略（TTL）完全未定义。
- **理由**：这是数据生命周期管理的核心问题。若聚合数据与原始数据同 TTL（如均 30 天），则降采样的长期价值大打折扣（用户无法查询 30 天前的趋势）。若聚合数据独立 TTL，则需要定义默认策略和配置方式。当前文档的缺失会导致实现阶段的产品决策真空。
- **修正建议**：补充聚合数据 TTL 策略定义，例如"聚合数据支持独立 TTL 配置，默认聚合数据保留周期 = 原始数据 TTL × 10（最低 1 年）"；并增加 Scenario"数据工程师为 1h 精度聚合数据配置 2 年独立保留策略"。

#### MEDIUM-4（维度：依赖关系 — 边缘自治域 TTL 依赖未识别）
- **位置**：自检清单（第 153 行）声称"依赖关系已确认"
- **问题**：product-spec.md 中"边缘自治"Feature（P0）明确包含"离线期间独立完成 TTL 管理"，但本文档未识别与边缘自治域的任何依赖关系。
- **理由**：边缘节点离线时的 TTL 清理行为（是否执行？积压如何处理？恢复后如何与云端策略同步？）与本文档定义的云端 TTL 策略存在直接依赖。若两边独立定义，可能导致边云 TTL 策略不一致（如云端 30 天、边缘 7 天），数据同步时产生冲突。
- **修正建议**：在已知待澄清问题中增加"边缘节点离线期间 TTL 清理策略与云端策略的同步机制"，或在文档中明确声明"本文档仅覆盖云端集群 TTL/降采样，边缘自治域 TTL 管理由边缘自治域文档定义，边云策略一致性由边云协同域保障"。

#### LOW-1（维度：格式 — R1 评审记录缺陷密度数值异常）
- **位置**：评审记录表（第 163 行）
- **问题**：R1 缺陷密度写为"15.15/Feature"，但按 1 CRITICAL(10) + 4 HIGH(3×4=12) = 22 分，密度应为 11.0/Feature，数字对不上。
- **理由**：数值不一致会导致评审历史可信度受损，后续轮次无法正确追踪改进趋势。
- **修正建议**：修正为准确值"11.0/Feature"，或注明计算方式。
