# 特性域：cache-offloading（KV Cache 核心卸载能力）

## Feature 价值论证

### F1: KV Cache 写入存储（P0）

#### 痛点

当前 LLM 推理面临 HBM 容量瓶颈：
- 单个 LLaMA-3 70B 模型的 KV Cache 占用：L=80 层 × 2（K/V）× 8KB/token × 128K tokens ≈ **128 GB HBM/请求**
- 实际部署中，HBM 满载后新请求被拒绝或触发 Eviction，导致重复 Prefill 计算
- GPU 显存成本高昂（HBM3 单价 ≈ $20/GB），扩展成本线性增长

标杆数据佐证：
- MoonCake 生产环境：有效请求容量提升 59%-498%（Kimi 月活 2000 万用户）
- LMCache 实测：LLaMA-3 8B 模型，32K tokens 上下文，HBM 卸载后显存占用从 28 GB 降至 4 GB

#### 当前方案不足

- **裸 HBM 扩容**：成本极高，HBM 单价是 DRAM 的 10x+，容量天花板明显
- **MoonCake/LMCache**：缺乏企业级能力（多租户隔离、QoS、故障自愈）
- **通用对象存储（MinIO/Ceph）**：无 KV Cache 语义，与推理引擎集成需自行开发适配层

#### 用户价值

- **直接价值**：将 KV Cache 异步写入外部存储（DRAM/SSD），释放 HBM 压力，使单 GPU 可服务更长上下文
- **量化收益**：同 HBM 配置下，上下文长度从 32K 提升至 128K；或同上下文长度下，GPU 并发请求数提升 3-5x
- **成本收益**：以 $1-2/GB 的 DRAM/SSD 替代 $20/GB 的 HBM，TCO 降低 60-80%

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| HBM 卸载率 | ≥ 80%（HBM 满载时，80% 的 KV Cache 块写入外部存储） | 基准测试：模拟 HBM 满载场景，统计卸载比例 |
| 写入吞吐 | 单节点 ≥ 500K ops/s（64KB chunk） | 压测工具：单连接写入 1M 次，统计吞吐 |
| 写入延迟影响 | 异步写入不增加推理延迟（TTFT/TBT 无显著变化） | 对比测试：写入前后 TTFT/TBT 分布 |
| 写入可靠性 | 写入成功的数据在读取时可被正确加载 | 端到端测试：写入→故障模拟→读取→数据校验 |
| 写入可中断 | 推理引擎崩溃时，已写入数据不丢失 | 故障演练：推理引擎 SIGKILL，已写入数据可恢复 |

### F2: KV Cache 读取加载（P0）

#### 痛点

长上下文和多轮对话场景下，历史 KV Cache 加载延迟直接影响 TTFT：
- LMCache 实测：无缓存时多轮对话 TTFT = 28s，有缓存时 TTFT = 3.7s（**7.7x 改善**）
- MoonCake 实测：Prefix 命中时 TTFT 降低 84%
- 当前方案中，加载延迟不可预测——跨节点传输时 RDMA 故障回退 TCP 导致 TTFT 超标

#### 当前方案不足

- **加载延迟分布宽**：通用缓存层缺乏分层感知的预取策略，导致 P99 延迟远高于 P50
- **加载失败无优雅降级**：缓存未命中时系统无感知，推理请求继续发送导致超时
- **跨实例不共享**：每个推理实例独立缓存，多实例部署时缓存重复度高达 60%+

#### 用户价值

- 从外部存储按需加载 KV Cache 到推理引擎，支撑**长上下文**（128K+ tokens）和**多轮对话**场景
- 提供**可预测的加载延迟**（P99 ≤ 500 μs，同一 RoCE fabric 内）
- 支持**全局缓存池**，跨推理实例共享已缓存的 KV Cache，消除重复计算

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 加载延迟 P99 | ≤ 500 μs（跨节点 RDMA，同一 RoCE fabric） | 基准测试：跨节点读取 10 万次，测量 P99 |
| 加载延迟 P99 | ≤ 150 μs（DRAM 层命中，节点内） | 基准测试：节点内读取 10 万次，测量 P99 |
| 缓存未命中处理 | 未命中时推理引擎收到明确错误码，可选择回退到重新计算 | 集成测试：故意清空缓存后请求，验证错误码和回退 |
| 预取命中率 | ≥ 80%（多轮对话场景，前 N 轮对话 KV Cache 在 Decode 开始前已预取） | 轨迹回放：多轮对话 trace，统计预取命中率 |
| 全局缓存共享 | 多推理实例间缓存共享率 ≥ 50%（相同 prefix 的请求） | 压测：多实例并发请求相同输入，统计缓存共享 |

### F3: Prefix Cache 匹配（P0）

#### 痛点

RAG 和多轮对话场景中存在大量**公共前缀**（System Prompt、RAG 检索结果）：
- 公共前缀重复计算浪费 HBM 和计算资源
- 每次新请求都需重新计算公共前缀的 KV Cache，即使之前已计算过
- MoonCake 数据显示：公共前缀占比 20%-40% 时，Prefix 命中可使 TTFT 降低 84%

#### 当前方案不足

- **Prefix 索引不精确**：现有方案（vLLM 内置 prefix caching）仅支持 token 级别精确匹配，无法处理语义相似但 token 序列略有不同的前缀
- **Prefix 共享粒度粗**：以请求为粒度，无法跨请求共享部分匹配的前缀块
- **Prefix 失效机制缺失**：KV Cache 过期或驱逐时，Prefix 索引未同步更新，导致误命中

#### 用户价值

基于 token 序列前缀匹配命中历史缓存，**避免重复 Prefill 计算**，TTFT 可降低 50-84%（benchmark 数据）

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| Prefix 匹配命中率 | 30%-50%（典型工作负载，RAG/多轮对话） | 轨迹回放：实际工作负载 trace，统计命中率分布 |
| TTFT 改善 | Prefix 命中时 TTFT 降低 ≥ 70% | 对比测试：同输入有/无 prefix 匹配 |
| 匹配精度 | 误匹配率 ≤ 0.1%（匹配结果与预期完全一致） | 单元测试：已知输入的预期匹配结果，统计精度 |
| Prefix 索引性能 | 索引查询延迟 ≤ 1 ms（100M token 序列规模） | 基准测试：大规模索引下的查询延迟 |
| Prefix 失效传播 | KV Cache 驱逐后，索引同步更新，延迟 ≤ 100 ms | 故障演练：驱逐块，验证索引一致性 |

### F4: PD 分离传输（P1）

#### 痛点

PD 分离架构（Prefill-Decode Disaggregation）已成为主流推理优化方向：
- Prefill 节点计算密集，Decode 节点内存密集，两者资源需求差异高达 5-10x
- 传统同节点部署导致资源利用率低（Decode 阶段 Prefill 资源闲置）
- MoonCake 的 Transfer Engine 实现单节点 190 GB/s 传输，验证了 PD 分离的技术可行性

#### 当前方案不足

- **传输协议不统一**：不同推理框架（vLLM/SGLang/Dynamo）使用不同的传输接口，互不兼容
- **传输与缓存耦合**：传输层直接依赖特定缓存后端，切换后端时传输链路需重建
- **拥塞控制缺失**：大块 KV Cache 传输时缺乏拥塞控制，影响同网络上的推理请求

#### 用户价值

支持 Prefill 节点到 Decode 节点的 KV Cache **高效传输**，使 PD 分离架构可行，
节点间传输吞吐 ≥ 10 GB/s（目标），使 PD 分离的资源利用率提升 2-3x

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 传输吞吐 | ≥ 10 GB/s/节点 | 基准测试：1GB KV Cache 跨节点传输，测量吞吐 |
| 传输延迟 | 1GB KV Cache 端到端传输 ≤ 200 ms | 基准测试：测量 1GB 数据传输时间 |
| 拥塞控制 | 多并发传输时，单传输延迟增加 ≤ 20% | 压测：10 并发传输，测量延迟分布 |
| 传输中断恢复 | 传输中断后自动续传，已传输数据不丢失 | 故障演练：传输中断后恢复，验证数据完整性 |

### F5: 热点数据多副本（P1）

#### 痛点

高频访问的 KV Cache 块（热点数据）存在**单节点网络拥塞**问题：
- 多个 Decode 实例同时请求同一 KV Cache 块时，单节点网络带宽成为瓶颈
- 单副本时该节点故障导致热点数据不可用

#### 当前方案不足

- **缺乏热点感知**：无法识别热点数据，只能被动处理拥塞
- **被动复制**：仅在读取失败时触发复制，延迟不可预测
- **副本一致性**：多副本写入时缺乏一致性协议，导致版本不一致

#### 用户价值

对频繁访问的 KV Cache 块**自动多副本复制**，避免单节点网络拥塞，提升热点数据可用性

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 热点识别延迟 | 热点数据识别延迟 ≤ 1 分钟（连续访问 ≥ 100 次/分钟） | 压测：指定块高频访问，测量热点识别时间 |
| 副本放置 | 副本均匀分布，副本间距离（网络跳数）最大化 | 压测：创建多副本，验证副本分布合理性 |
| 副本一致性 | 多副本读取返回相同数据（强一致） | 并发压测：多客户端并发读，验证一致性 |
| 副本重建时间 | 节点故障后，热点数据副本重建时间 ≤ 5 分钟 | 故障演练：热点节点故障，测量重建时间 |

### S1: 推理引擎正常写入 KV Cache

**类型**：正常场景
**Feature**：KV Cache 写入存储（F1）

#### 前置条件

- 存储集群运行正常（3+ 节点，所有存储层可用）
- 推理引擎通过 Connector 注册到存储集群
- KV Cache chunk 划分策略已配置（默认 64KB/chunk）
- 认证 Token 有效，租户权限正常

#### 用户旅程

> **时间标注说明**：所有时间均为**相对时间**（T=0 为当前旅程起点）。

```
T=0      推理引擎在 Prefill 阶段生成 KV Cache
    ↓
T=0      Connector 调用 kv_cache_write_async(chunk_id, kv_data)
    ↓
T=0      Connector 计算 chunk_hash，作为全局唯一标识
    ↓
T=1ms    数据写入 L1 层（DRAM，分层策略决定）
    ↓
T=2ms    I/O 调度层决策：L1 满载时异步下沉到 L2（SSD）
    ↓
T=5ms    写入确认返回，Connector 收到 ACK
    ↓
T=5ms    推理引擎继续 Decode 阶段，不阻塞主流程
```

#### 验收标准

- [ ] 写入请求在 10ms 内返回成功（异步模式）
- [ ] 写入后数据可通过 hash 检索到
- [ ] 数据分层策略生效（热数据在 DRAM 层）
- [ ] 写入吞吐达到 ≥ 500K ops/s（64KB chunk）
- [ ] 异步写入不增加推理 TTFT/TBT

#### 异常分支

| 条件 | 预期行为 |
|------|----------|
| L1 层满载 | 自动下沉到 L2 层，写入延迟增加到 SSD 层水平（P99 ≤ 2ms） |
| 租户配额超限 | 返回配额超限错误码，推理引擎可选择降级回 HBM |
| Token 过期 | 返回认证失败，Connector 自动刷新 Token 并重试 |

### S2: 多轮对话场景加载 KV Cache

**类型**：正常场景
**Feature**：KV Cache 读取加载（F2）

#### 前置条件

- 存在历史对话的 KV Cache 数据（已写入成功）
- Prefix 索引已构建（至少经过一次完整前缀计算）
- 加载策略配置正确（预取/按需加载）

#### 用户旅程

> **时间标注说明**：所有时间均为**相对时间**（T=0 为当前旅程起点）。

```
T=0      用户发起第 2 轮对话请求
    ↓
T=0      Connector 调用 prefix_cache_lookup(prefix_tokens)
    ↓
T=1ms    Prefix 索引返回匹配块列表 [chunk_id_1, chunk_id_3, chunk_id_7]
    ↓
T=1ms    预取调度器发起批量预取请求
    ↓
T=50ms   预取数据陆续到达（跨节点 RDMA，≤ 500 μs/块）
    ↓
T=200ms  预取完成，Decode 阶段使用已缓存的 KV Cache
    ↓
T=200ms  推理引擎继续生成，无需等待 Prefill
```

#### 验收标准

- [ ] Prefix 匹配命中时 TTFT 降低 ≥ 70%
- [ ] 预取在 Decode 开始前完成（预取完成 ≤ TTFT 目标）
- [ ] 未命中时明确返回错误码，推理引擎可回退重新计算
- [ ] 缓存未命中不影响推理正确性（只是延迟增加）
- [ ] 加载延迟 P99 ≤ 500 μs（RDMA 跨节点）

#### 异常分支

| 条件 | 预期行为 |
|------|----------|
| 部分 Prefix 命中（chunk_id_5 缺失） | 仅加载命中的 chunk，缺失部分回退到重新 Prefill |
| RDMA 传输失败（硬件故障） | 自动降级到 TCP 传输，延迟增加到 2-5x 但服务不中断 |
| 缓存数据损坏（CRC 校验失败） | 丢弃损坏数据，返回未命中，触发重新 Prefill |

### S3: Prefix Cache 匹配

**类型**：正常场景
**Feature**：Prefix Cache 匹配（F3）

#### 前置条件

- 存在包含公共前缀的历史请求（如 System Prompt、RAG 检索结果）
- Prefix 索引已构建并正常工作

#### 用户旅程

> **时间标注说明**：所有时间均为**相对时间**（T=0 为当前旅程起点）。

```
[第 1 轮对话]
T=0      用户请求含 System Prompt + 用户 Query
    ↓
T=100ms  Prefill 计算完成，KV Cache 写入存储
    ↓
T=100ms  Connector 更新 Prefix 索引
    ↓
T=100ms  第一轮对话结束

[第 2 轮对话]
T=0      用户新请求复用相同 System Prompt（仅 Query 变化）
    ↓
T=0      Connector 调用 prefix_cache_lookup(System_Prompt_Token_Array)
    ↓
T=1ms    Prefix 索引命中（命中块 chunk_id: 1, 2, 3, 4）
    ↓
T=1ms    加载命中块，Decode 阶段使用
    ↓
T=50ms   推理引擎仅计算用户 Query 部分的 Prefill
    ↓
T=200ms  完整回复生成（TTFT 比无缓存降低 84%）
```

#### 验收标准

- [ ] 公共前缀匹配时 TTFT 降低 ≥ 70%
- [ ] 匹配精度 ≥ 99.9%（无误匹配）
- [ ] Prefix 索引查询延迟 ≤ 20ms P99（100M token 规模，需并行批量查找优化）
- [ ] 匹配结果可用于推理（数据格式正确）

#### 异常分支

| 条件 | 预期行为 |
|------|----------|
| 前缀部分命中（仅前 80% 匹配） | 加载匹配部分，未匹配部分回退计算 |
| Prefix 索引损坏 | 自动重建索引，延迟 ≤ 5 分钟，期间回退到无缓存 |
| 新请求包含历史未见的 Token | 返回未命中，正常 Prefill |

### S4: PD 分离架构节点间传输

**类型**：正常场景
**Feature**：PD 分离传输（F4）

#### 前置条件

- 集群配置为 PD 分离模式（Prefill 节点池 + Decode 节点池分离）
- 两池节点间网络连通（同一 RoCE fabric）
- KV Cache chunk 大小 ≥ 1MB（PD 传输的典型场景）

#### 用户旅程

```
T+0ms    Prefill 节点完成 Prefill，生成 KV Cache chunk
    ↓
T+0ms    Connector 调用 kv_cache_transfer(chunk_id, decode_node_id)
    ↓
T+0ms    传输层与 Decode 节点建立 RDMA 连接
    ↓
T+100ms  1GB KV Cache 数据通过 RDMA 传输（吞吐 ≥ 10 GB/s）
    ↓
T+100ms  Decode 节点接收确认
    ↓
T+100ms  Decode 节点开始 Decode 阶段
```

#### 验收标准

- [ ] 1GB KV Cache 端到端传输 ≤ 200ms
- [ ] 传输吞吐 ≥ 10 GB/s
- [ ] 传输中断后自动续传
- [ ] 多并发传输时拥塞控制生效

#### 异常分支

| 条件 | 预期行为 |
|------|----------|
| Decode 节点故障 | 传输中断，目标节点重新上线后自动续传 |

### S8: 缓存雪崩（Cache Avalanche）

**类型**：故障场景
**Feature**：KV Cache 写入存储（F1）、数据冗余与自愈

#### 触发条件

```
触发条件：
1. 大量 KV Cache 块同时过期（如 TTL 统一设置导致批量过期）
2. 或大量节点同时故障（如机房断电）
```

#### 系统响应

```
T+0s     大量 KV Cache 块同时失效（TTL 过期）
    ↓
T+0s     大量读取请求触发缓存未命中
    ↓
T+0s     大量请求同时触发重新 Prefill（回退到 GPU 计算）
    ↓
T+0s     GPU HBM 压力急剧上升
    ↓
T+1s     速率限制器介入，限制并发 Prefill 数量
    ↓
T+5s     队列调度：公平排队，避免 HBM 过载
    ↓
T+10s    缓存逐步重建，HBM 压力下降
    ↓
T+2min   缓存重建完成，系统恢复正常
```

#### 验收标准

- [ ] 缓存雪崩发生时系统不崩溃（优雅降级）
- [ ] 速率限制器在 ≤ 1s 内介入
- [ ] 队列调度保证公平性（避免单一租户占用全部资源）
- [ ] 重建完成后 TTFT 恢复到正常水平
- [ ] 雪崩期间有明确告警通知运维

#### 预防措施

| 措施 | 说明 |
|------|------|
| TTL 随机化 | 不同 chunk 的 TTL 加入随机偏移（如 TTL±10%），避免批量过期 |
| 渐进式过期 | 缓存分批过期，而非一次性全部过期 |
| 熔断机制 | Prefill 请求超过阈值时触发熔断，返回服务降级提示 |
