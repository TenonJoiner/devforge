## Feature: Prefix Cache 加载

**做什么**：当推理引擎收到新请求时，系统根据请求的 token 序列前缀，查询缓存中是否存在可复用的 KV Cache。若存在，则将匹配的 KV Cache 异步加载到引擎的 GPU 内存中，减少重复计算。

**为什么**：Prefix Cache 命中是 KV Cache Offloading 的核心收益来源。在长上下文和 RAG/多轮对话场景中，prefix 复用率可达 30%~80%，TTFT 可降低 2×~22×。

**涉及 Actor**：推理引擎、缓存服务实例、全局元数据服务（分布式模式下）、AI 应用开发者、平台工程师。

### Scenario: 正常路径 — 前缀命中，TTFT 显著降低

**Actor**：推理引擎（vLLM）、缓存服务实例

**前置条件**：缓存服务已启动并与引擎完成握手；目标 prefix 已存在于 DRAM 层；引擎请求的 token 序列前 N 个 token 与缓存完全匹配；baseline workload 为平均序列长度 >= 2K tokens、prefix 命中长度 >= 512 tokens、模型大小 >= 7B。

**触发动作**：引擎调度器向缓存服务发起 prefix 匹配查询；匹配成功后，Worker 调用 `start_load_kv()` 启动异步加载。

**预期行为**：异步加载在 prefill 计算前完成或与之重叠；引擎仅对未命中部分执行计算；TTFT 相比无缓存时降低 ≥ 50%（P50）/ ≥ 33%（P99）。

**验证方法**：采集 1 万条请求样本，prefix hit 场景下 P50 TTFT < 基准值/2，P99 TTFT < 基准值/1.5；命中率统计误差 < 1%。

### Scenario: 短 prefix 命中 — 命中长度过短，不劣化 TTFT

**Actor**：推理引擎（vLLM）、缓存服务实例

**前置条件**：目标 prefix 存在于 DRAM 层，但命中长度 < 128 tokens。

**触发动作**：引擎发起 prefix 匹配查询，匹配成功但命中长度较短。

**预期行为**：系统仍执行异步加载，但由于加载 overhead 可能抵消收益，TTFT 不劣化于无缓存基线（增量 < 15%）。

**验证方法**：采集 1000 条命中长度 < 128 tokens 的请求：P99 TTFT < 无缓存基线 × 1.15；0 次因短 prefix 命中导致请求失败或崩溃。

### Scenario: 节点故障 — 存储节点宕机，请求降级处理

**Actor**：推理引擎、缓存服务实例、集群编排器、全局元数据服务

**前置条件**：目标 prefix 原本存储在节点 A 的 DRAM 中；节点 A 突然宕机或被编排器判定为不健康。

**触发动作**：引擎发起 prefix 查询；系统发现节点 A 不可达。

**预期行为**：全局元数据服务在 3 个心跳周期（默认 10s）内感知故障；查询自动降级为 "miss"；引擎 fallback 到全量 prefill 计算，请求不失败；错误率 = 0%。

**验证方法**：故障注入测试：随机 kill -9 一个缓存服务实例，连续发送 1000 个请求，0 个请求返回 5xx/崩溃；TTFT 恢复至无缓存基线的时间 < 15s。

### Scenario: 同节点并发冲突 — 多个引擎实例同时读取本地缓存块

**Actor**：多个推理引擎实例、缓存服务实例、数据传输服务

**前置条件**：同一 prefix 被 3 个及以上引擎实例同时请求；目标数据位于本节点 DRAM 或 SSD 层。

**触发动作**：多个引擎在 100ms 时间窗口内并发发起加载请求。

**预期行为**：缓存服务实例支持读共享，不阻塞并发读取；各引擎获取的数据内容一致；总读取带宽不超过介质带宽上限的 90%。

**验证方法**：压力测试：4 个引擎并发读取同一 2GB 缓存块，每个引擎数据校验（checksum）100% 通过；峰值带宽 ≥ 介质理论带宽的 80%；无死锁或超时。

### Scenario: 跨节点并发冲突 — RDMA 环境下多节点同时读取同一缓存块

**Actor**：多个推理引擎实例、多个缓存服务实例、数据传输服务

**前置条件**：同一 prefix 存储在远程节点；3 个及以上引擎实例通过 RDMA 从该节点并发读取。

**触发动作**：多个引擎在 100ms 时间窗口内并发发起跨节点加载请求。

**预期行为**：系统支持跨节点读共享，不阻塞并发读取；各引擎获取的数据内容一致；当 RDMA QP 数量或网络带宽成为瓶颈时，P99 延迟 < 正常值的 150%；不因资源竞争导致连接断开或请求失败。

**验证方法**：压力测试：4 个引擎跨节点并发读取同一 2GB 缓存块：各引擎数据校验（checksum）100% 通过；正常网络条件下峰值带宽 ≥ 理论带宽的 60%；人为限制 QP 数量至上限的 50% 时，P99 延迟 < 正常值的 150%；0 次 RDMA 连接断开或请求失败。

### Scenario: 网络分区 — 查询期间网络分区，恢复后数据一致

**Actor**：推理引擎、缓存服务实例、全局元数据服务

**前置条件**：引擎与缓存服务实例之间的网络出现临时分区（如 iptables DROP 10s）。

**触发动作**：分区期间引擎发起 prefix 查询；分区恢复后再次发起查询。

**预期行为**：分区期间查询在 2s 内超时并返回 miss；分区恢复后，系统通过心跳重新同步状态；再次查询能在 50ms 内正常返回命中结果；无数据不一致或脏读。

**验证方法**：使用 tc/iptables 注入 10s 网络分区，分区期间 100% 请求降级为 miss；恢复后 30s 内命中率恢复至分区前水平的 95% 以上；校验恢复后加载的数据 checksum 与分区前一致。

### Scenario: 数据损坏 — 加载时发现校验失败，触发失效与降级

**Actor**：推理引擎、缓存服务实例、平台工程师

**前置条件**：某缓存块在存储过程中因硬件异常导致部分数据损坏；损坏块仍被索引引用。

**触发动作**：引擎加载该缓存块；缓存服务在加载前/后进行 CRC64 校验。

**预期行为**：校验失败时，立即标记该缓存块为 "invalid"，并向监控系统上报 `cache_data_corruption` 告警；引擎收到 load failure 信号后 fallback 到全量 prefill；该 prefix 的后续请求不再尝试加载此块。

**验证方法**：故意篡改 1% 的缓存块数据，触发加载：100% 损坏块被检测并标记失效；0 个引擎实例因加载损坏数据而崩溃或产生错误结果；告警在 5s 内出现在监控面板。

## Feature: 异步 KV Cache 存储

**做什么**：在 prefill 或 decode 阶段产生新的 KV Cache 后，系统通过异步流水线将其卸载到 DRAM、SSD 或远程存储层，不阻塞引擎的连续 batching 和 token 生成。

**为什么**：GPU HBM 是昂贵且稀缺的资源。将已生成但暂不立即复用的 KV Cache 卸载到低层存储，可以释放 GPU 内存供更多请求使用，从而提升并发度和整体吞吐。

**涉及 Actor**：推理引擎、缓存服务实例、数据传输服务、平台工程师。

### Scenario: 正常路径 — 异步存储完成，不阻塞解码

**Actor**：推理引擎、缓存服务实例

**前置条件**：引擎完成某请求的 prefill 计算；系统配置为将新 KV Cache 异步卸载到 DRAM/SSD。

**触发动作**：Worker 调用 `start_store_kv()` 启动异步存储；引擎继续执行下一个 step 的调度 / decode。

**预期行为**：存储操作在后台完成；引擎的 decode TPOT 增加 < 5%（与无缓存场景对比）；存储完成后，该 KV Cache 可被后续请求查询命中。

**验证方法**：对比基准（无 offloading）和开启 offloading 的 TPOT：P99 增量 < 5%；存储完成到可查询的延迟 < 100ms；1 万条请求样本中 0 次因存储阻塞导致 decode 超时。

### Scenario: 资源耗尽 — SSD/DRAM 空间不足，优雅拒绝或降级

**Actor**：推理引擎、缓存服务实例、平台工程师

**前置条件**：本地 SSD 或 DRAM 利用率达到配置上限（如 95%）。

**触发动作**：引擎持续产生新的 KV Cache 并触发存储请求。

**预期行为**：当空间不足时，系统优先保留热度更高的旧数据；新数据若无法写入，则向引擎返回 "store_skipped" 信号；引擎继续正常推理，不因存储失败而中断；系统向监控上报 `storage_capacity_full` 告警。

**验证方法**：人为填满 SSD/DRAM 至 95%，继续注入请求：0 个请求失败或崩溃；存储拒绝率 100%（即新数据不再写入）；5s 内触发 `storage_capacity_full` 告警；引擎 TPOT 与正常场景差异 < 5%。

### Scenario: 节点故障 — 存储过程中节点宕机，数据不丢失或已降级

**Actor**：推理引擎、缓存服务实例、集群编排器

**前置条件**：某缓存服务实例正在执行异步存储；存储完成前该实例所在节点突然宕机。

**触发动作**：kill -9 缓存服务实例进程。

**预期行为**：引擎检测到存储实例不可达后，将该请求标记为 "store_failed"；引擎的 GPU 内存中的 KV Cache 仍可用于当前请求的 decode 完成；由于存储未完成，该数据自然不可被后续请求复用，系统不承诺未确认写入的数据持久化。

**验证方法**：在 1000 次存储过程中随机注入 3 次实例 kill：0 次引擎崩溃；当前请求 100% 正常完成 decode；未完成的存储不留下半写数据（通过 fsck/数据校验确认）。

### Scenario: 升级/回滚 — 缓存服务版本升级期间，存储不中断

**Actor**：推理引擎、缓存服务实例、平台工程师

**前置条件**：集群执行滚动升级，旧版本实例逐个被新版本替换。

**触发动作**：升级过程中引擎持续发送存储请求。

**预期行为**：旧实例在收到 SIGTERM 后，完成进行中的异步存储，拒绝新存储请求；引擎的 Connector 在 2s 内重连到新实例；升级期间请求的存储成功率 > 99%。

**验证方法**：滚动升级 3 节点集群，持续注入存储请求：存储成功率 ≥ 99%；0 次引擎因重连失败而 stuck > 5s；回滚操作在 60s 内完成且数据可继续读取。

## Feature: 跨层/跨节点 KV Cache 迁移

**做什么**：根据访问热度和容量策略，系统自动将 KV Cache 在不同存储层级（HBM→DRAM→SSD→Remote）之间晋升或降级，并在分布式部署下支持跨节点数据迁移以达到负载均衡。

**为什么**：静态放置无法适应 workload 变化。热数据需要靠近 GPU 以降低加载延迟，冷数据需要下沉到廉价介质以释放高速缓存空间。

**涉及 Actor**：缓存服务实例、数据传输服务、全局元数据服务、平台工程师。

### Scenario: 正常路径 — 热数据从 SSD 晋升到 DRAM

**Actor**：缓存服务实例、数据传输服务

**前置条件**：某 prefix 存储在 SSD 层（256 tokens 粒度）且最近 5 分钟内被访问 3 次以上；DRAM 层有可用空间。

**触发动作**：系统后台迁移任务检测到热度阈值触发，启动从 SSD 到 DRAM 的晋升。

**预期行为**：晋升在后台异步完成；晋升期间该 prefix 仍可从 SSD 正常读取；晋升完成后，后续查询从 DRAM（256 tokens 粒度）响应，加载延迟降低 ≥ 50%。

**验证方法**：监控某 prefix 的存储层标签变化：从 L3 变为 L2 的时间 < 30s；晋升期间 100 次读取全部成功；晋升后 P99 加载延迟 < 晋升前 P99 的 50%。

### Scenario: 晋升目标层空间不足 — 晋升任务可中断并降级读取

**Actor**：缓存服务实例、数据传输服务

**前置条件**：某 prefix 正在从 SSD 晋升到 DRAM；晋升过程中 DRAM 被新写入占满。

**触发动作**：后台晋升任务执行到 50% 时，DRAM 可用空间降至 0。

**预期行为**：晋升任务在 1s 内检测到空间不足并安全中断；已复制的部分数据不阻塞前台读取；系统保留原 SSD 数据作为 fallback；后续查询继续从 SSD 加载；晋升任务在 DRAM 空间释放后自动重试或放弃。

**验证方法**：人为耗尽 DRAM 空间：晋升任务 100% 安全中断，0 次阻塞前台读取；后续查询成功率 100%，P99 延迟与晋升前 SSD 基线差异 < 10%；0 次双份数据同时暴露或索引不一致。

### Scenario: 运维操作 — 管理员在线扩容，客户端无感知

**Actor**：平台工程师、缓存服务实例、全局元数据服务、集群编排器

**前置条件**：集群需要扩容以增加缓存容量；新节点已加入集群并完成初始化。

**触发动作**：平台工程师通过运维 CLI 触发数据重平衡；或系统自动根据 `free_ratio_first` 策略将新数据导向新节点。

**预期行为**：新节点在 60s 内开始接收写入和新晋升的数据；已有数据在后台逐步迁移（不阻塞前台读写）；迁移期间客户端 P99 TTFT 增加 < 10%。

**验证方法**：向 2 节点集群添加第 3 节点，持续注入读写请求：0 个请求失败；新节点在 60s 内接收 ≥ 10% 的新写入；扩容后 10 分钟内各节点存储量标准差 < 平均值的 30%。

### Scenario: 网络分区 — 迁移任务执行中发生网络分区

**Actor**：缓存服务实例、数据传输服务、全局元数据服务

**前置条件**：一个跨节点迁移任务正在进行中；源节点与目标节点之间出现 15s 网络分区。

**触发动作**：tc 注入 15s 网络延迟/丢包。

**预期行为**：迁移任务在 5s 内检测到超时并暂停；源节点保持原数据不变；分区恢复后，迁移任务自动重试；最终数据在目标节点完整可用，无半写块。

**验证方法**：注入分区后检查：迁移状态变为 "paused"；源数据 checksum 不变；恢复后 60s 内迁移完成；目标数据 checksum 与源一致。

### Scenario: 并发冲突 — 迁移期间读取请求并发访问

**Actor**：推理引擎、缓存服务实例、数据传输服务

**前置条件**：某 prefix 正在从节点 A 迁移到节点 B；此时引擎发起对该 prefix 的读取。

**触发动作**：高并发读取与后台迁移同时发生。

**预期行为**：读取优先：系统优先从源节点（A）或已完成复制的目标节点（B）响应读取；迁移不阻塞读取；读取响应延迟增加 < 20%。

**验证方法**：在持续 1GB 数据迁移的同时，以 100 QPS 读取同一 prefix：读取成功率 100%；P99 延迟 < 正常读取 P99 的 120%；迁移最终完成时间增加 < 50%。

## 本域非功能需求

| 子项 | 目标值 |
|------|--------|
| TTFT (cache hit) | P50 降低 ≥ 50% vs 无缓存基线；P99 降低 ≥ 33% |
| TTFT (cache miss) | 增量 < 15% vs 无缓存基线 |
| TPOT 影响 | 开启 offloading 后 P99 增量 < 5% |
| 加载延迟 | HBM→GPU < 1ms；DRAM→GPU P99 < 10ms；SSD→GPU P99 < 100ms；Remote→GPU P99 < 50ms (RDMA) / < 200ms (TCP) |
| 吞吐上限 | 单机 DRAM 传输 ≥ 40 GB/s；SSD 顺序读 ≥ 标称带宽 70% |
| 故障降级率 | 单节点/单盘故障时 100% 请求可降级为 miss 并完成推理 |
| 故障感知时间 | 节点无心跳到系统标记为故障 < 15s |
| RTO | 单节点故障后副本恢复或降级完成 < 5 分钟 |
| 不一致窗口 | 允许最终一致性；索引不一致窗口 < 100ms |
