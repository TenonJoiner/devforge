## 涉及 Actors

> 本节由 A 部分与 B 部分协作维护，列表为本特性域统一 Actor 清单。

- [数据分析师](../requirements/product-spec.md#actor-数据分析师)（主 Actor）——通过 SQL/PromQL/PG Wire 等接口发起交互式分析查询
- [可视化仪表盘系统](../requirements/product-spec.md#actor-可视化仪表盘系统)（主 Actor）——周期性拉取查询用于面板渲染
- [告警规则引擎](../requirements/product-spec.md#actor-告警规则引擎)（主 Actor）——高频低延迟评估查询
- [批量数据消费系统](../requirements/product-spec.md#actor-批量数据消费系统)（主 Actor）——大规模历史数据扫描与基于快照的批量导出
- [流式订阅消费者](../requirements/product-spec.md#actor-流式订阅消费者)（主 Actor）——长连接订阅实时数据
- [应用开发者](../requirements/product-spec.md#actor-应用开发者)（主 Actor）——通过 SDK/ORM/PG Wire Protocol 在业务应用中发起查询，关注客户端兼容性、连接管理、错误处理
- [数据工程师](../requirements/product-spec.md#actor-数据工程师)（协作 Actor）——配置查询资源组、缓存策略、敏感数据脱敏规则
- [平台运维工程师](../requirements/product-spec.md#actor-平台运维工程师)（协作 Actor）——监控查询服务健康度，处理慢查询、连接耗尽等运维事件
- [安全审计员](../requirements/product-spec.md#actor-安全审计员)（协作 Actor）——审计查询访问记录、敏感数据查询事件、SQL 注入告警

**跨域依赖说明**：
- **与边缘自治域的关系**：本域定义云端及全局查询能力。边缘节点离线期间的本地诊断查询由「边缘自治」域定义（参见 `边缘自治.md` Feature: 支持边缘节点离线自治运行 Scenario #2），本域 Scenario 默认运行在边云协同正常或云端独立场景下。
- **与边云协同域的关系**：跨边云的查询路由、边缘节点不可达时的降级策略由「边云协同」域定义（参见 `边云协同.md`），本域跨冷热边界的查询（SQL 兼容查询 Scenario #6）默认在云端单点完成，不涉及边缘下沉。
- **与安全合规域的关系**：身份认证、RBAC、TLS 由「安全合规」域统一定义，本域仅在 NFR 中声明对其依赖与查询特有的安全要求（查询权限粒度、SQL 注入防护、查询审计、敏感数据脱敏），不重复定义认证/授权机制本身。

**读路径语义分层**：
- **告警评估通道**：默认走实时读路径（不经过快照隔离层），保证写入后 2s 内数据可见；支持显式 `AS OF SNAPSHOT 'xxx'` 语法切换为快照视图
- **PG Wire / SQL 默认查询**：默认走实时读路径；支持显式 `AS OF SNAPSHOT 'xxx'` 语法请求快照一致性视图
- **PromQL 查询**：默认走实时读路径；范围查询的瞬时向量评估基于查询发起时刻的实时数据状态
- **批量导出**：必须基于显式创建的快照执行，保证导出结果时间点一致
- **流式订阅**：基于实时写路径推送，不经过快照层

## Feature: 提供 SQL 兼容的时序查询语言

### Feature 价值论证

**用户痛点**：数据分析师需要对传感器时序数据执行窗口聚合、降采样、趋势分析。当前时序数据库多采用私有查询语言，分析师需额外学习成本，且无法复用已有 SQL 工具链。

**现有方案不足**：纯 InfluxQL/Flux 等私有语言生态封闭；通用 SQL 数据库缺乏时序语义（时间桶对齐、插值、降采样）。

**本 Feature 价值**：基于标准 SQL 扩展时序语义（TIME_BUCKET、INTERPOLATE），分析师零学习成本上手，保留时序查询的性能优化路径。

**成功指标**：
- 标准 SQL 语法兼容率 >= 90%
- 典型时序查询（单时间线 1 天范围 1 分钟桶聚合）P95 响应时间 < 500ms

**协议语义一致性声明**：SQL 查询与 PromQL/PG Wire 查询访问同一份时序数据时，对等价逻辑查询的结果允许偏差：时间戳一致到秒级；聚合数值偏差 < 0.1%；NULL 与"无数据点"语义等价（均表示该时间点无有效数据）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据分析师对单条时间线执行 TIME_BUCKET + AVG 降采样查询，时间范围 24h、桶宽 1min | P95 响应时间 < 500ms，结果行数 = 1440 | 自动化 |
| 2 | 数据分析师使用标准 SQL 窗口函数（LAG/LEAD/RANK）对时序数据执行趋势变化检测 | 查询语法解析成功率 100%，窗口计算结果与参考实现一致 | 自动化 |
| 3 | 数据分析师对含 NULL 间断的时间线执行 INTERPOLATE 线性插值查询 | 插值结果与线性插值公式计算值误差 < 0.001% | 自动化 |
| 4 | 数据分析师提交语法错误的 SQL 查询 | 错误响应时间 < 50ms，错误信息包含行号+列号+建议修正 | 自动化 |
| 5 | 数据分析师对 30 天范围执行未加时间桶的全量 SELECT *，触发大结果集保护 | 扫描行数超过 100 万行时主动终止，终止延迟 < 2s | 自动化 |
| 6 | 数据分析师查询的时间范围跨越冷热存储边界 | 查询透明跨越冷热层，P95 响应时间 < 2s，结果完整 | 自动化 |
| 7 | 数据分析师执行查询期间目标数据节点宕机，系统自动重试到副本节点 | 查询最终成功返回，端到端延迟增幅 < 3s，结果与正常路径一致 | 自动化 |

## Feature: 提供 PromQL 兼容查询接口

### Feature 价值论证

**用户痛点**：Grafana 和告警引擎已深度绑定 PromQL 生态。若不支持 PromQL，运维团队需维护独立 Prometheus 实例做查询代理。

**现有方案不足**：部署独立 Prometheus 导致数据双写、存储冗余、一致性窗口（同步延迟 10-30s）。

**本 Feature 价值**：原生实现 PromQL 查询 API，Grafana/Alertmanager 直连时序数据库，消除中间层。

**成功指标**：
- PromQL 兼容率 >= 95%（Prometheus 官方合规测试套件）
- 告警规则引擎评估延迟（数据写入到查询可见）< 15s

**协议语义一致性声明**：PromQL 查询与 SQL/PG Wire 查询访问同一份时序数据时，对等价逻辑查询的结果允许偏差：时间戳一致到秒级；聚合数值偏差 < 0.1%；"无数据点"与 SQL NULL 语义等价（均表示该时间点无有效数据）。PromQL rate() 与 SQL 窗口差分在乱序数据下的结果偏差 < 0.1%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 可视化仪表盘系统通过 /api/v1/query_range 发起 rate() + sum by() 范围查询 | P95 响应时间 < 500ms，数值与 Prometheus 参考实现偏差 < 0.1% | 自动化 |
| 2 | 告警规则引擎以 15s 间隔持续拉取 avg_over_time() 聚合值，模拟 100 条并发告警规则 | 单次评估延迟 P99 < 500ms，100 条规则并发时无查询失败 | 自动化 |
| 3 | 可视化仪表盘系统查询包含 label_replace、histogram_quantile 等高级函数 | 函数计算结果与 Prometheus 参考实现一致 | 自动化 |
| 4 | 告警规则引擎查询引用不存在的 metric name | 返回空 vector 而非错误，响应时间 < 50ms | 自动化 |
| 5 | 可视化仪表盘系统发起跨 7 天的高基数 metric 范围查询（10000+ 时间线） | 系统限制返回时间线数量至配置上限，响应包含 warning 头部 | 自动化 |
| 6 | 告警规则引擎在数据写入暂停 30s 期间查询最近窗口数据 | 查询返回截至最后写入时刻的完整数据，不返回虚假零值 | 自动化 |
| 7 | 告警规则引擎查询目标节点不可达时，系统自动路由到副本节点 | 路由切换延迟 < 2s，查询结果与正常路径一致，无数据缺失 | 自动化 |

## Feature: 实现查询资源隔离与优先级调度

### Feature 价值论证

**用户痛点**：告警规则引擎的秒级评估查询与数据分析师的大范围扫描共享计算资源。大查询耗尽资源时告警评估延迟飙升导致漏报。

**现有方案不足**：简单连接数限制无法区分查询类型，缺乏按查询类型分配资源的能力。

**本 Feature 价值**：按查询来源和类型划分资源组，高优先级查询在资源争抢时获得优先调度。缓存命中查询的资源计费策略：缓存命中不消耗资源组 CPU/内存配额，但每个资源组对缓存命中查询设置每秒上限（防止缓存被滥用为无限容量通道），超限后回退到实时计算并正常计费。

**成功指标**：
- 批量扫描满载时，告警查询 P99 延迟劣化 < 20%
- 批量查询资源组超配额时，1s 内触发限流

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 告警规则引擎持续发起高优先级查询，同时批量消费系统发起 10 个并发全表扫描 | 告警查询 P99 延迟劣化 < 20% | 自动化 |
| 2 | 可视化仪表盘系统发起交互查询，系统按配置的资源组分配 CPU 和内存配额 | 实际内存使用不超过配额 110% | 自动化 |
| 3 | 批量消费系统的资源组配额耗尽，新批量查询进入排队 | 告警和交互查询延迟波动 < 5% | 自动化 |
| 4 | 单个交互查询执行时间超过超时阈值（30s），系统强制终止 | 查询在超时后 1s 内被终止，返回超时错误码 | 自动化 |
| 5 | 批量查询内存逼近资源组上限时，系统触发溢出到磁盘的降级策略 | 查询继续执行，系统内存不超过配额 105%，无 OOM | 自动化 |
| 6 | 资源组配置动态变更，变更在不中断现有查询的情况下生效 | 配置变更后 10s 内新查询按新配额调度 | 自动化 |
| 7 | 低优先级资源组的缓存命中查询超出每秒上限 | 超限查询回退到实时计算，响应时间退化 < 50%，不突破资源组配额 | 自动化 |

## Feature: 实现查询结果缓存与智能失效

### Feature 价值论证

**用户痛点**：可视化仪表盘系统每 15-30s 自动刷新，多用户同时查看相同仪表盘时相同查询被重复执行，浪费 90%+ 计算资源。

**现有方案不足**：应用层缓存无法感知时序数据的时间语义——不知道何时缓存因新数据写入而过期。

**本 Feature 价值**：查询引擎内置时间感知缓存，基于数据写入水位线判断缓存有效性，兼顾高命中率和数据新鲜度。

**成功指标**：
- 仪表盘典型查询缓存命中率 >= 80%（限定条件：仪表盘刷新间隔 15-30s，底层数据更新频率 < 1 次/分钟）
- 缓存数据新鲜度延迟（写入到缓存失效）< 5s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 可视化仪表盘系统对同一 metric 连续发起 10 次相同聚合查询 | 第 2-10 次命中缓存，响应时间 < 10ms | 自动化 |
| 2 | 可视化仪表盘系统查询滚动窗口（每次右移 15s），系统复用已缓存的历史桶结果 | 滚动查询响应时间相比全量重算降低 >= 60% | 自动化 |
| 3 | 新数据写入后，查询包含该数据点的时间桶，缓存自动失效 | 数据写入后 5s 内查询返回包含新数据的聚合结果 | 自动化 |
| 4 | 缓存内存达到配置上限，系统按 LRU 策略淘汰 | 淘汰后查询回退到实时计算，无错误返回 | 自动化 |
| 5 | 查询跨越多个已关闭桶和一个当前活跃桶 | 已关闭桶从缓存读取、活跃桶实时计算，合并结果正确 | 自动化 |
| 6 | 数据发生乱序写入（迟到数据落入已关闭的历史桶） | 乱序写入后 5s 内对应桶缓存失效 | 自动化 |
| 7 | 缓存节点宕机后冷启动，系统降级为直接查询存储层 | 降级期间查询成功率 100%，响应时间退化至无缓存水平但不超过 P95 < 2s；在每秒 ≥ 10 次查询持续负载下，缓存节点恢复后 60s 内命中率回升至 50%+ | 自动化 |

## Feature: 支持跨时间线关联查询与多维聚合

### Feature 价值论证

**用户痛点**：数据分析师需要关联分析多条时间线的相关性（如温度、振动、电流），当前需分别查询后在应用层手动对齐时间戳并计算，开发成本高且性能差。更复杂的场景需要将关联查询与时序 SQL 语义组合使用，如"按 1 分钟桶对齐两条时间线后 JOIN，再对 JOIN 结果执行窗口函数计算滑动相关性"。

**现有方案不足**：应用层关联需要将原始数据点全量拉取到客户端（网络传输 GB 级），时间对齐逻辑手写容易出错。当需要与 SQL 引擎的 TIME_BUCKET、窗口函数等时序语义组合时，应用层需自行实现桶对齐+JOIN+窗口计算的三段逻辑，复杂度指数级增长。

**本 Feature 价值**：在查询引擎内原生支持多时间线 JOIN 和多维 GROUP BY，在不传输全量原始数据的前提下完成关联计算。同时与 SQL 引擎的时序扩展（TIME_BUCKET、INTERPOLATE、窗口函数）无缝组合，使分析师无需切换工具即可完成"桶对齐+关联+窗口聚合"的一体化分析。

**成功指标**：
- 3 条时间线 7 天范围关联查询 P95 响应时间 < 2s
- 1000+ 时间线多维聚合 P95 响应时间 < 3s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据分析师对同一设备的 3 条时间线执行时间对齐 JOIN，计算 7 天范围内的皮尔逊相关系数 | P95 响应时间 < 2s，相关系数与离线参考值偏差 < 0.001 | 自动化 |
| 2 | 数据分析师对 1000+ 时间线按 3 个标签维度执行 GROUP BY 聚合 | P95 响应时间 < 3s，聚合分组数与标签基数笛卡尔积一致 | 自动化 |
| 3 | 数据分析师关联两条采样频率不同的时间线，系统自动执行时间对齐插值 | 对齐后数据点时间戳一致，插值方式可通过查询参数指定 | 自动化 |
| 4 | 关联查询中一条时间线在指定范围内无数据 | 查询不报错，无数据列标记为 NULL，响应包含缺失标识 | 自动化 |
| 5 | 多维聚合中部分时间线的标签值缺失 | 缺失标签归入默认分组，聚合结果完整无数据丢失 | 自动化 |
| 6 | 跨 10000 条时间线的关联查询超出复杂度限制 | 系统在查询规划阶段（< 500ms）拒绝并返回拆分建议 | 自动化 |
| 7 | 关联查询涉及的部分数据节点不可达（3 节点中 1 节点宕机） | 若缺失数据可从副本获取则返回完整结果；若无副本可用则返回明确错误码标识缺失分片，不返回静默的部分结果 | 自动化 |
| 8 | 数据分析师使用 TIME_BUCKET 对两条时间线分别降采样为 1 分钟桶，再执行 JOIN 并对 JOIN 结果施加 LAG 窗口函数计算滑动相关性 | 查询语法解析成功，结果与离线 Python 计算的滑动相关系数偏差 < 0.001，P95 响应时间 < 5s | 自动化 |
| 9 | 长时间执行的关联查询在中途遇到内部状态刷新（如配置变更、热数据迁移）时不中断 | 状态刷新后查询继续执行，总延迟增幅 < 200ms，最终结果正确 | 自动化 |

## Feature: 提供实时数据流式订阅推送

### Feature 价值论证

**用户痛点**：实时告警系统需要在数据写入后毫秒级触发规则引擎。轮询模式下消费者无法感知"是否有新数据"，高频轮询浪费资源，低频轮询牺牲实时性。

**现有方案不足**：轮询无法保证同一时间线内数据点的有序性，两次轮询之间可能漏掉乱序到达的迟到数据。

**本 Feature 价值**：通过长连接订阅机制，系统在数据写入落盘后主动推送，内置背压控制和断线补推。

**成功指标**：
- 写入到推送延迟 P99 < 1s
- 断线重连后数据补推完整率 100%（断线时长 <= 30 分钟）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 流式订阅消费者订阅 100 条时间线，系统按写入顺序逐条推送新数据点 | 同一时间线内推送顺序与写入顺序一致，延迟 P99 < 1s | 自动化 |
| 2 | 系统在 10 万点/秒持续写入下稳定推送 30 分钟无积压 | 推送队列深度不超过 10s 写入量 | 自动化 |
| 3 | 流式订阅消费者处理速度低于推送速度时，系统触发背压 | 背压触发后推送端内存不超过配置上限，恢复后自动补发 | 自动化 |
| 4 | 流式订阅消费者网络断开 15 分钟后重连，系统从断点补推 | 补推数据与断线期间写入数据逐条一致，无重复无遗漏 | 自动化 |
| 5 | 断线超过保留窗口（30 分钟），重连时系统告知数据已过期 | 返回明确错误码标识补推窗口过期 | 自动化 |

## Feature: 提供高频低延迟的告警评估查询通道

### Feature 价值论证

**用户痛点**：告警规则引擎每 5-15 秒执行一轮评估，在写入高峰期常规查询通道被大查询占满，告警评估超时导致漏报。漏报在工业场景下直接造成设备损坏或安全事故。

**现有方案不足**：共享查询通道无法区分查询优先级，告警引擎只能通过缩短超时来自保，但超时后无法区分"正常"与"未完成"。即使有资源隔离（Feature 3），通用资源组机制仍按公平调度分配 CPU 时间片，不会为告警类查询自动选择最优执行计划。

**本 Feature 价值**：为告警引擎开辟独立查询通道，具备以下三层差异化能力（区别于资源隔离 Feature 3 的通用机制）：
1. **独立 SLA 承诺**：承诺告警查询 P99 < 500ms（写入负载 ≤ 80% 峰值且分析通道满载时）；写入负载 > 80% 峰值时进入降级窗口，承诺退化为 P99 < 1s 并触发系统告警。资源隔离仅保证"比例分配"不保证绝对延迟
2. **独立连接入口和请求队列**：避免与通用查询共享连接池瓶颈，连接池耗尽不影响告警通道接入
3. **专用快路径**：针对告警查询的小范围、高频、低延迟特征，在通用查询规划阶段拥塞时仍保证规划阶段耗时 < 50ms，查询执行选择最短数据访问路径（具体实现机制由子系统架构决定）

**与资源隔离的关系**：资源隔离（Feature 3）是底层资源配额机制，告警评估通道是在其之上的专用快车道——资源隔离保证"带宽不被抢占"，告警通道保证"路径最短+入口独立"。

**与流式订阅的关系**：流式订阅（Feature 5）面向需要逐条处理数据点的实时消费者（如实时计算引擎、事件驱动架构），采用推模式、长连接、背压控制；告警通道面向周期性批量评估的规则引擎，采用拉模式、短请求、固定周期。两者适用场景边界清晰：流式订阅适合"数据到达即处理"的流式场景，告警通道适合"定时评估窗口聚合值"的规则匹配场景。

**成功指标**：
- 告警评估查询 P99 延迟 < 500ms（系统写入负载 ≤ 80% 峰值时）；写入负载 > 80% 峰值时退化为 P99 < 1s
- 端到端评估延迟（写入到告警触发）< 30s，达标率 >= 99.9%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 告警规则引擎以 5s 间隔通过专用通道查询最近 60s 聚合值 | 查询 P99 延迟 < 200ms，数据时间戳与当前时间差 < 5s | 自动化 |
| 2 | 告警规则引擎并发评估 200 条规则 | 所有查询在同一评估周期内完成，P99 延迟 < 500ms | 自动化 |

| 5 | 告警查询目标节点短暂不可达（< 3s） | 重试成功率 >= 99%，单次评估端到端延迟 < 10s | 自动化 |

## Feature: 提供 PostgreSQL Wire Protocol 兼容查询接口

### Feature 价值论证

**用户痛点**：数据分析师习惯使用 psql、DBeaver、Grafana 等通用 SQL 工具，当前只能通过私有 API 访问。应用开发者在业务应用中通过 ORM（如 SQLAlchemy、Hibernate）或数据库驱动（如 psycopg2、pgx）访问时序数据，需额外适配私有协议。

**现有方案不足**：私有协议要求每个工具都开发专用插件，生态覆盖不全。应用开发者无法复用已有的 ORM 和数据库驱动，被迫使用 REST/gRPC 客户端重写数据访问层。

**本 Feature 价值**：实现 PostgreSQL wire protocol 查询子集，任何支持 PostgreSQL 的客户端工具和 ORM 框架无需修改即可连接。对数据分析师提供零门槛接入，对应用开发者提供标准数据库驱动复用能力。

**成功指标**：
- psql、DBeaver、Grafana PostgreSQL 数据源三种主流客户端可直接连接
- 支持 SELECT、WHERE、GROUP BY、ORDER BY、LIMIT 及常用聚合函数
- psycopg2、pgx 两种主流驱动可正常建立连接和执行查询

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据分析师使用 psql 客户端连接并执行聚合查询 | 连接建立 < 1s，查询返回正确结果，格式与 PostgreSQL 标准输出一致 | 自动化 |
| 2 | 数据分析师通过 Grafana PostgreSQL 数据源配置连接 | 数据源连接测试通过，面板正常渲染 | 手动 |
| 3 | 数据分析师执行带 LIMIT/OFFSET 的分页查询 | 分页结果有序无重复，单页返回延迟 P95 < 2s | 自动化 |
| 4 | 数据分析师执行 PG Wire 当前不支持的 SQL 语法（子查询、DDL） | 返回标准 PostgreSQL 错误码和人类可读错误消息，连接保持可用 | 自动化 |
| 5 | 数据分析师通过连接池中间件（PgBouncer）建立连接 | 连接池模式下查询正常返回，连接复用无状态泄漏 | 自动化 |
| 6 | 数据分析师使用 ORM（SQLAlchemy）发起 JOIN 查询，PG Wire 将 JOIN 路由到 SQL 引擎执行 | JOIN 查询通过 PG Wire 正确返回结果，ORM 模型映射无异常 | 自动化 |
| 7 | 数据分析师查询执行中途网络连接断开 | 服务端在 TCP keepalive 超时（默认 30s）后释放连接资源，不泄漏内存或文件描述符 | 自动化 |

| 11 | 系统升级 PG Wire Protocol 实现版本（如新增消息类型支持），旧版客户端（psycopg2 2.8、pgx 4.x）在新版服务端上正常工作 | 协议版本协商后降级到客户端支持的子集，旧客户端查询功能不退化 | 自动化 |
| 12 | 系统回滚 PG Wire Protocol 到前一版本，回滚期间活跃连接自动断开后客户端可重连 | 回滚后 30s 内客户端重连成功率 100%，查询功能不退化 | 自动化 |
| 13 | 平台运维工程师动态调整 PG Wire 服务端最大连接数（200->500），变更不中断已有连接 | 配置变更后 5s 内新连接按新上限受理，已有连接不受影响 | 自动化 |

### NFR 交叉兼容性声明

以下指标组合在**标准硬件配置**（单节点 32 核 / 128GB 内存 / NVMe SSD）下**确认可同时达成**：

| 指标组合 | 同时达成条件 | 优先级 |
|---------|-------------|--------|
| 告警查询 P99 < 500ms + 流式推送 P99 < 1s + 并发查询 ≥ 100 | 写入负载 ≤ 80% 峰值，无批量导出 | 高 |
| 批量导出 500MB/s + 在线查询 P99 增幅 < 20% | 批量导出单独资源组，告警通道独立 | 高 |
| 缓存命中率 ≥ 80% + 缓存新鲜度 < 5s | 仪表盘刷新间隔 15-30s，数据更新频率 < 1 次/分钟 | 中 |
| 快照创建 < 100ms + 10 万点/秒写入 | 快照创建不阻塞写入路径 | 高 |

**互斥场景与降级策略**：
- 批量导出 500MB/s + 流式订阅 1000 连接 + 告警查询并发：批量导出吞吐优先保证 ≥ 300MB/s，告警 P99 退化为 < 1s，流式推送 P99 退化为 < 2s
- 写入峰值 100% + 分析通道满载 + 批量导出并发：告警查询进入降级窗口（P99 < 1s），批量导出吞吐退化为 ≥ 200MB/s
- **优先级排序**：告警 P99 > 批量导出吞吐 > 流式推送延迟 > 缓存命中率
