## Feature: 多协议统一写入接入

### Feature 价值论证

**用户痛点**：工业现场设备协议碎片化严重——PLC 网关只支持 MQTT，边缘网关走 CoAP，上层应用用 HTTP/gRPC。数据工程师被迫为每种协议部署独立的数据管道，运维成本随协议数量线性增长。

**现有方案不足**：多数时序数据库仅原生支持 HTTP 写入，其余协议依赖外部 broker 或适配层转发，引入额外跳数和延迟，且每个适配层的背压、重试、确认语义不统一，导致数据丢失难以定位。

**本 Feature 价值**：在数据库内核层统一多协议接入，所有协议共享同一写入路径和确认语义。数据源无需关心下游存储细节，只需选择自身最适合的传输协议即可完成写入。

**成功指标**：
- 支持协议数 >= 5（MQTT 3.1.1/5.0、CoAP、HTTP/1.1+2、gRPC、InfluxDB Line Protocol over TCP）
- 任意协议写入同一 measurement 的端到端延迟差异 < 20%（同等 payload 大小下）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | IoT 设备群通过 MQTT QoS 1 向边缘节点推送单条传感数据点，系统解析 topic 映射到 measurement 并返回 PUBACK | 写入确认延迟 P99 < 50ms（单点 payload < 256B） | 自动化 |
| 2 | 应用服务集群通过 gRPC 流式接口批量写入 1000 条数据点（Protobuf 编码），系统在单次 RPC 内完成解码、写入并返回批量确认 | 单批 1000 点写入延迟 P99 < 200ms，吞吐 >= 50k points/s | 自动化 |
| 3 | 数据采集代理通过 HTTP POST 以行协议格式提交混合 measurement 的批量数据（50 个不同 measurement，共 5000 点） | 单请求 5000 点写入成功率 100%，响应时间 P95 < 500ms | 自动化 |
| 4 | IoT 设备群发送格式不合法的 MQTT payload 时，系统拒绝该条数据并返回明确错误码，不影响同连接后续正常数据写入 | 错误数据拒绝延迟 < 5ms，同连接后续合法写入成功率 100% | 自动化 |
| 5 | 应用服务集群在 HTTP 写入过程中连接中断（半包），系统丢弃不完整请求且不产生脏数据 | 半包请求不落盘，连接资源释放时间 <= 30s，无内存泄漏 | 自动化 |

| 7 | 数据采集代理发送混合了合法与非法行的批量请求时，系统逐行解析，合法行正常写入，非法行单独报错 | 合法行写入成功率 100%，错误响应包含具体行号和原因 | 自动化 |

## Feature: 时序数据模型与 Schema 管理

### Feature 价值论证

**用户痛点**：数据工程师在设备上线初期无法预知所有 field，设备固件升级后会新增遥测字段。如果 schema 不支持演进，要么预留大量空列浪费存储，要么每次变更都需要停机迁移。

**现有方案不足**：InfluxDB 采用 schema-on-write 完全无约束，导致 tag/field 命名混乱、类型冲突频发。而传统 RDBMS 的 ALTER TABLE 在 TB 级表上耗时数小时，无法满足 7x24 连续写入要求。

**本 Feature 价值**：提供 measurement/tag/field 三层模型定义，支持在线 schema 演进（新增 field、tag 扩展），同时通过类型约束防止写入污染。

**成功指标**：
- Schema 变更（新增 field）生效时间 < 5s，变更期间写入零中断
- 类型冲突写入拒绝率 100%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师通过管理 API 为已有 measurement 新增一个 float64 类型 field，新增完成后 IoT 设备立即写入包含该新 field 的数据点 | 从 API 调用返回成功到新 field 可写入的生效延迟 < 5s | 自动化 |
| 2 | 应用服务集群首次写入一个尚未定义的 measurement，系统根据首条数据自动推断 schema 并持久化 | 自动创建的 schema 可通过查询 API 获取；类型推断与首条数据语法形式（整数→int64、浮点→float64、布尔→bool、引号字符串→string）一一对应，偏差数 = 0 | 自动化 |
| 3 | IoT 设备写入数据时某 field 值类型与 schema 定义冲突，系统拒绝该点，同批次中类型正确的其他数据点正常写入 | 冲突点被拒绝且错误响应包含具体 field 名和期望类型 | 自动化 |
| 4 | 数据工程师提交不兼容的 schema 变更（如将 float 字段改为 string）时，系统拒绝变更并返回冲突字段列表 | 不兼容变更 100% 被拒绝，响应包含冲突字段名和类型对比 | 自动化 |
| 5 | 两个数据工程师并发修改同一 measurement 的 schema 时，系统保证只有一个变更成功 | 并发冲突检测成功率 100%，无 schema 损坏或部分应用 | 自动化 |
| 6 | IoT 设备群写入数据触发 schema 自动推断时，若推断结果与已有 schema 冲突，该批数据被拒绝 | schema 完整性不被破坏，拒绝响应延迟 < 50ms | 自动化 |
| 7 | 系统在 schema 变更过程中发生进程崩溃，重启后 schema 状态为变更前或变更后的完整状态 | 崩溃恢复后 schema 一致性校验通过，无中间态残留 | 自动化 |
| 8 | IoT 设备群在某 tag 维度写入大量去重值（如随机 UUID），触发单 measurement tag 基数超过配置上限 | 基数超限时新增 series 拒绝率 100%，已有正常 series 写入不受影响，拒绝响应包含具体 tag 键和当前基数 | 自动化 |

## Feature: 写入流量治理

### Feature 价值论证

**用户痛点**：工业场景中设备上线具有潮汐特征——早班开机时数千设备同时开始上报，写入流量在 10 秒内从基线跃升 10-50 倍。如果系统无保护机制，内存和磁盘 IO 被瞬间打满，导致所有写入超时甚至 OOM 崩溃。

**现有方案不足**：简单的连接数限制无法区分正常突发和持续过载；客户端侧限流需要改造所有设备固件，在已部署的棕地环境中不可行。

**本 Feature 价值**：在服务端提供分层流量治理——背压反馈让客户端感知系统负载并主动降速，速率限制保护系统不被单一来源压垮，批量聚合提升吞吐效率，异步确认解耦写入延迟与持久化延迟。

**成功指标**：
- 写入洪峰（10x 基线）期间系统内存使用增幅 < 30%，无 OOM
- 背压触发后客户端可感知的降速信号延迟 < 1s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 应用服务集群以 10x 基线速率持续写入 60s，系统触发背压机制返回 HTTP 429 / gRPC RESOURCE_EXHAUSTED | 背压信号在负载达阈值后 1s 内发出，携带 Retry-After 头，已接受数据零丢失 | 自动化 |
| 2 | 数据采集代理配置异步写入模式，提交 10000 点后立即收到接受确认，系统后台完成持久化 | 异步接受确认延迟 P99 < 10ms，最终持久化确认延迟 P99 < 2s | 自动化 |
| 3 | 多个 IoT 设备群以不同速率写入，系统按租户/来源维度执行独立速率限制 | 超限来源收到限流响应，未超限来源写入延迟 P99 波动 < 10% | 自动化 |
| 4 | 写入队列因下游持久化变慢而积压至容量上限时，系统拒绝新写入，队列内已有数据不丢弃 | 队列满后新请求拒绝延迟 < 5ms，队列内数据最终全部落盘 | 自动化 |
| 5 | IoT 设备群异步写入后在确认超时窗口内未收到 ACK，设备重试写入，系统保证幂等去重 | 重试数据去重成功率 100%，超时窗口 <= 配置值 +10% 容差 | 自动化 |
| 6 | 系统在内存不足时主动降级批量聚合缓冲区大小，保持写入通路可用 | 内存压力下写入通路不中断，吞吐量降级幅度 <= 50% | 自动化 |
| 7 | 多个 Actor 同时写入且总速率超过系统容量时，系统按优先级策略公平限流 | 高优先级 Actor 吞吐量下降 <= 10%，低优先级 Actor 收到限流信号 | 自动化 |
| 8 | 数据工程师为不同设备组配置写入优先级，断网缓冲空间不足时按优先级淘汰 | 低优先级数据先被淘汰，高优先级数据零丢失，配置变更 < 10s 生效 | 自动化 |
| 9 | 网络恢复后续传阶段，边缘节点对上行带宽进行治理，确保续传不挤占实时数据通道 | 实时数据上行延迟 < 正常值的 1.5x，续传流量自适应占用剩余带宽 | 自动化 |
| 10 | 监控采集系统按 15s 周期拉取 /metrics，获取写入 QPS、延迟 P99、背压队列深度等指标 | 指标暴露完整性 100%，拉取自身延迟 < 100ms，对写入吞吐影响 < 1% | 自动化 |

## Feature: 写入持久化保障

### Feature 价值论证

**用户痛点**：7x24 高写入场景下，系统崩溃（断电、OOM Kill、硬件故障）是不可避免的。如果已确认的写入在崩溃后丢失，传感器数据不可重采，直接影响工艺分析和故障溯源。

**现有方案不足**：纯内存缓冲写入在崩溃时全部丢失；同步刷盘每条记录则写入延迟不可接受（毫秒级退化到十毫秒级）。需要在持久化保障和写入性能之间找到平衡。

**本 Feature 价值**：在写入确认与最终落盘之间建立持久化保障层——数据先顺序追加到持久化日志（顺序 IO，高吞吐），确认后再异步写入存储引擎。崩溃恢复时通过日志回放恢复所有已确认数据。

**成功指标**：
- 崩溃恢复后数据完整性 = 100%（已确认的写入零丢失）
- 持久化日志写入对写入延迟的额外开销 <= 10%（相对纯内存缓冲）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师确认写入成功后系统进程崩溃，重启后通过持久化日志回放恢复所有已确认数据 | 崩溃恢复后数据完整性 = 100% | 自动化 |
| 2 | 持久化日志写入过程中磁盘空间耗尽，系统立即拒绝新写入，已写入的日志记录完整可回放 | 磁盘满后新写入 100% 被拒绝，已有日志文件无截断损坏 | 自动化 |
| 3 | 持久化日志文件尾部因断电出现不完整记录，系统启动时跳过损坏尾部，恢复其前所有完整记录 | 损坏记录前的数据恢复率 = 100%，损坏记录数 <= 1 | 自动化 |
| 4 | IoT 设备群高并发写入时多个写者竞争持久化日志追加，系统保证记录顺序一致且无交错损坏 | 并发写入下日志记录 CRC 校验通过率 = 100% | 自动化 |
| 5 | 持久化日志段文件数量达到回收阈值时，系统确认数据已持久化后回收旧段，回收期间写入不阻塞 | 日志回收期间写入延迟增加 <= 10%，回收后磁盘空间释放 | 自动化 |
| 6 | 边缘节点在 eMMC/SD 卡等低 IOPS 存储介质上执行持久化日志写入 | 采用顺序追加写，写入吞吐 >= 5k points/s，存储介质擦写次数增幅 < 2x | 自动化 |
| 7 | 边缘节点持久化日志文件在存储空间不足时自动回收已确认续传成功的历史段 | 回收后释放空间可立即复用，回收操作对写入延迟影响 < 5ms（P99） | 自动化 |

## Feature: 边缘本地存储与断网缓冲

### Feature 价值论证

**用户痛点**：工业现场网络不可靠是常态——工厂车间电磁干扰、远程站点卫星链路中断、施工导致光缆切断，中断时长从秒级到天级不等。传感数据一旦丢失不可重采，直接影响工艺分析和故障溯源。

**现有方案不足**：纯云端时序库在断网时写入直接失败；边缘侧简单文件缓存缺乏持久化保障和有序续传能力，恢复后易出现乱序、重复或丢失；设备侧缓冲受限于传感器内存（通常 KB 级），无法承担小时级断网的数据暂存。

**本 Feature 价值**：在边缘节点提供具备持久化日志保障的本地存储层，断网期间零丢点持续接收设备数据，网络恢复后按时间序自动续传至云端。

**成功指标**：
- 断网 72 小时内数据零丢失（丢点率 = 0）
- 边缘节点崩溃重启后数据完整性 100%，恢复时间 < 30s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | IoT 设备群在网络正常时以 10k points/s 持续写入边缘节点，边缘节点本地持久化后实时转发至云端 | 本地写入 P99 延迟 < 50ms，云端确认后本地缓冲空间在 30s 内释放 | 自动化 |
| 2 | 网络中断发生后，IoT 设备群继续以相同速率写入，边缘节点自动切换为纯本地缓冲模式 | 切换延迟 < 1s，切换期间零丢点，设备侧无错误返回 | 自动化 |
| 3 | 网络恢复后，边缘节点自动按写入时间序将积压数据续传至云端，同时不阻塞新数据写入 | 续传期间新数据写入 P99 延迟劣化 < 20%，积压数据按时间序到达云端 | 自动化 |
| 4 | 边缘节点断网期间本地存储空间使用率达到 90% 阈值，触发降级策略 | 高优先级数据零丢失，降级触发后 P99 写入延迟 < 100ms | 自动化 |
| 5 | 边缘节点在写入过程中发生崩溃，重启后自动恢复本地缓冲状态 | 重启后数据完整性 100%，恢复时间 < 30s | 自动化 |
| 6 | 长时间断网（72 小时）期间 IoT 设备群以 5k points/s 持续写入，边缘节点在 64GB 存储上完成全量缓冲 | 72h 累计数据全部持久化，存储空间利用率可预测 | 自动化 |
| 7 | 网络在续传过程中再次中断，边缘节点中止续传并无缝回到纯本地缓冲模式 | 续传中断不导致数据重复或丢失，再次恢复后从断点续传 | 自动化 |
| 8 | 边缘节点存储介质出现坏块，部分本地缓冲数据不可读 | 受损数据段被标记跳过并生成告警，不影响其余正常数据的续传和新数据写入 | 手动 |

## Feature: 兼容 Prometheus remote_write 协议

### Feature 价值论证

**用户痛点**：数据采集代理（Prometheus、Grafana Agent、Vector）已广泛部署，其监控数据通过 remote_write 协议外发。如果不兼容该协议，客户需要额外部署协议转换层。

**现有方案不足**：部分时序数据库声称兼容 remote_write 但仅支持基础 TimeSeries 写入，不处理 metadata、exemplar 等扩展字段。

**本 Feature 价值**：实现完整的 remote_write v1 协议端点，数据采集代理零配置改动即可将本系统作为远端存储后端。

**成功指标**：
- Prometheus 官方兼容性测试套件通过率 >= 95%
- 单节点 remote_write 写入吞吐 >= 100k samples/s

| 2 | 数据采集代理批量推送包含 1000 个不同 metric 的 remote_write 请求 | 单请求处理延迟 P95 < 500ms，吞吐 >= 100k samples/s | 自动化 |

## Feature: 流式数据管道接入

### Feature 价值论证

**用户痛点**：大型工业客户已有 Kafka 作为数据总线，所有设备数据先入 Kafka 再分发给下游系统。如果时序数据库只支持推送写入，客户需要额外开发 consumer 应用做数据搬运。

**现有方案不足**：自建 consumer 应用需要处理 offset 管理、rebalance、反序列化、批量写入优化等复杂逻辑，且难以保证 exactly-once 语义。

**本 Feature 价值**：内置 Kafka consumer 能力，数据工程师只需配置 topic、反序列化格式和目标 measurement 映射，系统自动完成消费、解析、写入的全链路。

**成功指标**：
- Consumer lag 稳态 < 10s（单 partition 写入速率 10MB/s）
- 数据从 Kafka produce 到可查询的端到端延迟 P95 < 30s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师通过管理 API 创建 Kafka 消费任务，系统启动 consumer 并开始持续消费写入 | 任务创建后 30s 内开始消费，consumer lag 持续下降 | 自动化 |
| 2 | Kafka topic 有 8 个 partition，系统自动分配对应数量的消费线程并行消费 | 8 partition 并行消费吞吐 >= 单 partition 的 6 倍 | 自动化 |
| 3 | 数据工程师更新消费任务的 measurement 映射规则，系统热加载新配置不中断消费 | 配置变更生效时间 < 10s，变更窗口内无数据丢失 | 自动化 |
| 4 | Kafka 分区再平衡发生时，系统暂停消费，完成后从正确位点恢复 | 再平衡后数据重复率 = 0，丢失率 = 0，恢复时间 <= 30s | 自动化 |
| 5 | 系统进程崩溃后重启，从最后一次提交的消费位点恢复消费 | 崩溃恢复后数据丢失 = 0，重复数据点 = 0 | 自动化 |
| 6 | Kafka broker 不可达导致消费中断时，系统持续重试并在恢复后自动追赶积压 | 连接恢复后 lag 降至 < 10s 的时间 <= 5 分钟 | 自动化 |
| 7 | 消费速率持续低于生产速率导致 lag 超过告警阈值时，系统发出告警 | lag 超阈值后告警触发延迟 <= 60s | 自动化 |

## Feature: 兼容 InfluxDB Line Protocol 写入

### Feature 价值论证

**用户痛点**：InfluxDB Line Protocol 是时序数据领域的事实标准文本协议，Telegraf、Vector、Fluentd 等主流采集工具均原生支持。不兼容该协议意味着客户迁移时需要改造所有采集端配置。

**现有方案不足**：部分竞品仅支持 Line Protocol 的基础子集，不支持批量多行提交、纳秒精度时间戳、转义字符等完整语法。

**本 Feature 价值**：完整实现 InfluxDB Line Protocol 语法规范，数据采集代理和应用服务只需将写入 URL 切换到本系统即可完成迁移，零代码改动。

**成功指标**：
- InfluxDB Line Protocol 语法兼容率 100%
- 批量写入吞吐 >= 200k points/s（单节点，平均每点 200B）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据采集代理提交标准 Line Protocol 格式的批量数据（含 tag 值转义、field 值含空格和逗号） | 写入后查询返回的 tag/field 值与原始数据逐字节一致 | 自动化 |
| 2 | 应用服务集群以纳秒精度时间戳写入，同时另一来源以秒精度写入同一 measurement | 两种精度写入的数据点时间戳在查询时均以纳秒精度返回且数值正确 | 自动化 |
| 3 | 数据采集代理单次 HTTP POST 提交 10000 行 Line Protocol 数据 | 单请求解析+写入延迟 P95 < 300ms，吞吐 >= 200k points/s | 自动化 |
| 4 | 数据采集代理发送的行协议中包含解析错误行，系统逐行处理，错误行被拒绝，正确行正常写入 | 正确行写入成功率 100%，错误响应包含行号和解析错误描述 | 自动化 |
| 5 | 数据采集代理发送的时间戳精度与系统配置不匹配，系统按配置精度截断并写入 | 精度截断后数据正常写入，截断行为在响应 header 中标注 | 自动化 |

| 7 | 数据采集代理在网络恢复后批量补传 30 分钟前的历史数据，与当前实时数据混合写入 | 迟到数据按原时间戳落盘，可被时间范围查询到，且实时数据写入延迟 P99 劣化 < 20% | 自动化 |

| 9 | 数据工程师通过管理 API 切换默认时间戳精度（ns→ms），新写入按新精度处理 | 精度切换生效延迟 < 5s，切换后新写入时间戳精度正确，历史数据不受影响 | 自动化 |
| 10 | 单连接持续以最大速率提交数据导致连接池耗尽时，系统触发限流保护 | 限流触发后新连接建立成功率 100%（排队/拒绝策略），已有连接写入不受影响 | 自动化 |
