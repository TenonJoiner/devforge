## Feature: 暴露结构化运行指标端点

### Feature 价值论证

**用户痛点**：边缘侧节点分布广、数量多（百台至万台规模），运维团队 2-5 人无法逐台登录查看内部状态，故障发现依赖用户投诉而非主动告警，平均故障感知时间 > 15min。

**现有方案不足**：
- 自建脚本拉取日志再 grep 关键字，指标口径不统一、采集延迟 > 30s
- 同类时序数据库竞品（如 InfluxDB）虽暴露 /metrics，但未针对边缘场景对采集成本做优化（指标拉取本身可能消耗 5%+ 写入吞吐）
- TimescaleDB 虽支持 Prometheus 指标，但无背压保护，基数膨胀时直接拖垮主进程

**本 Feature 价值**：提供 OpenMetrics 合规的 /metrics 端点，监控采集系统可按 10-30s 间隔拉取全量运行指标，相比竞品具备：(a) 高负载下采集影响 < 1%；(b) 基数膨胀有显式降级而非崩溃；(c) 端点字段保持版本兼容，升级回滚不破坏监控面板。

**成功指标**：
- 运维团队平均故障感知时间从 > 15min 降至 < 5min（通过指标端点主动暴露异常替代用户投诉驱动）
- 监控告警面板可用率 ≥ 99.9%（升级/回滚/高负载下不破坏既有 dashboard）
- 指标采集对主路径写入吞吐影响 ≤ 1%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 监控采集系统每 15s 拉取 /metrics 端点，获取 QPS、写入延迟 P99、WAL 积压等指标 | 返回体包含 ≥ 20 个核心指标项，格式通过 OpenMetrics 合规校验 | 自动化 |
| 2 | 监控采集系统间隔 30s 再次拉取，指标值反映该时间窗口内的真实变化 | 计数器单调递增，gauge 反映瞬时值，histogram 桶累计值正确 | 自动化 |
| 3 | 数据库处于高写入负载（≥ 80% 设计吞吐）时拉取 /metrics | 端点响应延迟 P99 < 200ms，写入吞吐下降 ≤ 1% | 自动化 |
| 4 | 某指标维度出现基数膨胀（tag 组合 > 10000）时监控采集系统拉取指标 | 端点能感知并暴露 cardinality_overflow 标记位 = 1，不阻塞响应 | 自动化 |
| 5 | 在基数膨胀已被感知的情况下，端点对超限维度执行降级 | 超限维度被截断，整体响应体 < 5MB，响应延迟 < 500ms，截断行为可通过指标自描述 | 自动化 |
| 6 | 数据库节点启动但尚未完成初始化时监控采集系统拉取 /metrics | 返回 HTTP 503 + Retry-After header，不返回不完整指标 | 自动化 |
| 7 | 系统从 vN 升级至 vN+1 后监控采集系统继续使用旧版告警规则拉取 /metrics | 旧版本字段名/标签名保持兼容，新增字段不破坏既有 dashboard，弃用字段标注 deprecated 至少保留 1 个大版本 | 自动化 |
| 8 | 升级失败回滚至 vN 后监控采集系统继续拉取 /metrics | 端点字段集与升级前完全一致，监控告警链路不需任何调整 | 自动化 |
| 9 | 未授权客户端访问 /metrics 端点 | 端点强制访问控制（mTLS/Token/IP 白名单之一），未授权请求返回 401/403，事件被安全审计日志记录 | 自动化 |

## Feature: 提供结构化日志输出与级别控制

### Feature 价值论证

**用户痛点**：边云环境下故障定位需关联多节点日志，但非结构化文本日志无法被日志平台高效索引；调整日志级别需重启进程导致故障现场丢失；日志中可能携带认证凭证/查询参数等敏感字段，触发合规风险。

**现有方案不足**：
- 纯文本日志无法被 ELK/Loki 直接结构化索引，运维需写正则
- 多数开源时序库（InfluxDB/TDengine）日志级别调整需重启进程，故障现场丢失
- 业内常见 JSON 日志方案缺乏内置 PII 脱敏，认证 token / SQL 查询参数 / API Key 直接进入日志聚合平台，违反等保和 GDPR 要求
- 缺乏背压机制，DEBUG 级别下日志暴增直接拖累主业务

**本 Feature 价值**：JSON 结构化输出 + 运行时动态调级 + 内置脱敏白名单 + 写入背压。相比竞品，本 Feature 第一次将"合规可审计"作为日志输出的内建能力，而非依赖外部日志平台后处理。

**成功指标**：
- 动态调整日志级别从 API 调用到生效延迟 < 2s
- 结构化日志写入对主路径写入延迟的影响 < 3%（INFO 级别）
- 凭证类敏感字段在日志输出中脱敏覆盖率 100%（按已确认白名单）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 监控采集系统从节点拉取 / 推送日志至日志平台，平台索引最近 5 分钟 WARN 及以上日志 | 每条日志包含 timestamp/level/module/message/trace_id 字段，JSON 格式合法可被 ELK/Loki 直接索引 | 自动化 |
| 2 | 平台运维工程师通过管理 API 将日志级别从 INFO 动态调整为 DEBUG | 调用后 2s 内后续日志包含 DEBUG 条目，无需重启；调级行为被独立的安全审计日志记录（含操作者、时间、变更前后值） | 自动化 |
| 3 | 平台运维工程师将日志级别恢复为 INFO | 级别恢复后 2s 内不再产生 DEBUG 条目，恢复行为同样被审计日志记录 | 自动化 |
| 4 | 高写入负载且 DEBUG 级别下日志输出速率超过磁盘写入能力 | 触发背压丢弃低优先级条目，主业务写入延迟增幅 < 5%，丢弃数量通过 log_dropped_total 指标暴露 | 自动化 |
| 5 | 通过 API 传入非法日志级别值（空字符串、未知级别名、大小写错误、非字符串类型、超长字符串） | 返回 400 错误码并附带具体原因，当前日志级别保持不变，事件计入审计日志 | 自动化 |
| 6 | 节点磁盘空间不足（< 100MB）时持续产生日志 | 日志自动降级为 ERROR-only，/metrics 暴露 log_disk_pressure 指标，降级动作被审计日志记录 | 自动化 |
| 7 | 业务路径中出现含认证 token / API Key / SQL 查询参数 / 用户标识的请求，被日志记录 | 脱敏引擎对配置文件中声明的敏感字段模式 100% 匹配并触发脱敏（如 token 仅保留前 4 位 + 长度），脱敏后字段对安全审计员可识别但不可重建原值；具体字段白名单内容待 Q2 跟进确认后补充覆盖率验证 | 自动化 |
| 8 | 安全审计员定期审查日志保留期与合规字段完备性 | 日志保留期符合配置策略（满足等保/GDPR 配置），每条日志可追溯至 trace_id 和操作者，不存在缺失审计关键字段 | 手动 |
| 9 | 系统从 vN 升级至 vN+1，日志 schema 新增字段或重命名字段 | 既有日志平台索引规则不报错，下游解析兼容性测试通过；废弃字段标注 deprecated 至少保留 1 个大版本 | 自动化 |
| 10 | 升级失败回滚至 vN 后日志输出格式 | schema 与升级前完全一致，日志平台索引模板无需调整 | 自动化 |

## Feature: 支持分布式请求追踪集成（远期规划，本规格仅作前瞻设计）

> **优先级声明**：本 Feature 标记为 P2 远期规划，**当前迭代不进入开发**。本规格仅作前瞻性设计，验收门槛在前瞻阶段相应下调；正式纳入开发前需重新评审并对齐 SLA。
> **跨 Feature 依赖**：本 Feature 的 trace_id 字段强依赖 「提供结构化日志输出与级别控制」 Feature 中已定义的 trace_id 日志字段，必须在日志 Feature 上线后才能落地。

### Feature 价值论证

**用户痛点**：边云架构下一次写入请求经过边缘接入 → 边缘缓冲 → 边云同步 → 云端落盘多个环节，端到端延迟异常时无法判断瓶颈，平均故障定位时间 > 45min。

**现有方案不足**：
- 各节点独立记录日志，缺乏统一 trace_id 贯穿全链路
- 业内时序库（InfluxDB/TDengine）尚无内建 OpenTelemetry 集成；自建追踪需用户自行注入 SDK 并维护采样策略
- 边云之间在弱网环境下 context 传播缺乏标准协议适配（W3C Trace Context 在断网+批量补传场景下行为未定义）

**本 Feature 价值**（远期）：原生支持 OpenTelemetry W3C Trace Context，跨边云 RPC 自动注入 context，运维工程师可在 Jaeger/Tempo 一键查看完整调用链；针对弱网批量补传场景定义明确的 trace 行为。

**成功指标**（前瞻阶段，正式开发前需重审）：
- 跨边云请求 trace context 在线场景下传播成功率 ≥ 99%（前瞻阶段下调，正式开发前对齐至 99.9% 需要专项可用性论证）
- trace 注入/提取对单次请求延迟额外开销 < 0.5ms
- 默认采样率配置下 trace 后端存储增量 < 业务数据量 5%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 外部请求携带 W3C traceparent header 写入边缘节点，数据同步至云端 | 云端 span 与边缘 span 共享同一 trace_id，parent-child 关系正确 | 自动化 |
| 2 | 外部请求未携带 traceparent header | 边缘节点自动生成 trace_id 和 root span，trace_id 同步写入日志 Feature 定义的 trace_id 字段 | 自动化 |
| 3 | 平台运维工程师在追踪后端按 trace_id 查询完整调用链，并在日志平台用同一 trace_id 反查日志 | 调用链包含 ≥ 3 个 span，每个含 service_name/operation/duration/status；同一 trace_id 在日志平台可定位至对应日志条目 | 自动化 |
| 4 | 边云网络中断后恢复，积压数据批量同步 | 批量同步数据保留原始 trace_id，云端 span 标注同步延迟与原始时间戳 | 自动化 |
| 5 | 边缘节点收到格式非法的 traceparent header（结构错误、版本不支持、字段超长） | 丢弃非法 context 并生成新 trace_id，请求正常处理，事件被监控指标计数 | 自动化 |
| 6 | 安全审计员审查追踪数据中的字段 | span 属性中的请求参数遵循与日志相同的脱敏规则，不出现凭证类原值 | 手动 |

> **前瞻阶段待补充 Scenario**（正式开发前评审）：自适应采样率调整（QPS 超阈值时自动降采样）、trace 传播成功率持续监控。详见 Q3。
