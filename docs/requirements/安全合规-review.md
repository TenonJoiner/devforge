### R2 新发现问题清单

#### H-01【HIGH】TLS 最低版本与 product-spec 安全目标冲突

- **维度**：Feature 价值 / 跨文档一致性
- **位置**：Feature 1 Scenario #7（第 72 行）"仅允许 TLS 1.2+"；本域 NFR（第 447 行）"TLS 最低版本 | TLS 1.2+"
- **理由**：`product-spec.md` 安全合规目标表（第 462 行）明确要求"边云通信强制 TLS 1.3"，但本文档将最低版本降为 TLS 1.2+。这是下层文档对上层约束的向下不兼容放松。虽然 TLS 1.2 在 PCI DSS 4.0 中仍被接受，但对于 2026 年面向金融/工业场景的新产品，TLS 1.3 在握手性能和安全性上均有显著优势。两个文档的不一致会导致架构设计阶段产生"到底按哪个版本实现"的歧义。
- **修复建议**：统一为 TLS 1.3+（推荐），或在 product-spec 中同步更新为 TLS 1.2+ 并说明兼容性考量。

#### H-02【HIGH】审计日志数据面覆盖范围缺少 SELECT 查询审计

- **维度**：场景覆盖 / 合规完整性
- **位置**：Feature 5 成功指标（第 184 行）"关键数据面操作（DELETE/DROP/TRUNCATE/Schema 变更/权限变更）100% 记录"；Scenario #3、#4
- **理由**：等保 2.0 三级和 ISO 27001 均要求对敏感数据的访问进行审计。当前审计范围仅覆盖破坏性写操作（DELETE/DROP/TRUNCATE）和结构变更，未覆盖 SELECT 查询——尤其是 Feature 10 定义的 L3-机密级别数据的读取。在工业场景中，工艺配方、客户隐私等敏感数据的"谁查看了什么"与"谁修改了什么"同等重要。缺少 SELECT 审计会导致合规声称存在缺口。
- **修复建议**：在"关键数据面操作"定义中补充"L3 敏感数据的 SELECT 查询"，或增设独立 Scenario 验证"用户查询 L3 列后审计日志记录查询主体、对象、时间戳"。

#### M-01【MEDIUM】故障矩阵 3 个"—"单元格缺少备注说明

- **维度**：模板合规性
- **位置**：Feature 2（第 370 行）"资源耗尽"列 = "—"；Feature 9（第 377 行）"网络分区"和"资源耗尽"列 = "—"
- **理由**：模板强制要求"'—'表示该故障模式与本 Feature 无关或由其他域覆盖，需在备注中说明"。当前 7 条备注仅覆盖了 7 个"—"单元格，但矩阵中共有 13 个"—"（Feature 2 有 2 个、Feature 9 有 2 个、Feature 10 有 3 个等）。Feature 2 的"资源耗尽"、Feature 9 的"网络分区"和"资源耗尽"共 3 个单元格无任何备注解释。这会导致读者无法判断是遗漏还是有意排除。
- **修复建议**：补充 3 条备注，例如：
  - "身份认证'资源耗尽'列标记为'—'：认证服务资源耗尽由集群管理域的过载保护机制覆盖"
  - "降采样精度治理'网络分区'列标记为'—'：精度元数据为本地生成，分区不影响元数据正确性"
  - "降采样精度治理'资源耗尽'列标记为'—'：精度元数据存储开销由数据存储域的容量管理覆盖"

#### M-02【MEDIUM】Feature 8（多租户数据隔离）缺少网络分区场景

- **维度**：场景覆盖 / 边云架构一致性
- **位置**：Feature 8 故障矩阵（第 376 行）"网络分区"列 = "—"，备注称"租户隔离为本地命名空间机制，分区不影响隔离性"
- **理由**：本文档其他 7 个 Feature 中，有 5 个明确包含网络分区场景（Feature 1/2/3/4/7），体现了边云架构对网络分区的重视。Feature 8 作为 P2/H 复杂度的企业版特性，在边缘节点多租户场景下，分区期间新租户创建、租户配额同步、物理隔离租户的状态一致性等行为均未验证。备注"分区不影响隔离性"过于笼统，未覆盖边缘侧租户管理的具体行为。
- **修复建议**：添加 Scenario"网络分区期间边缘节点继续执行租户隔离校验"，验收标准为"分区期间租户 A 用户无法访问租户 B 数据，分区恢复后 <= 5s 同步最新租户元数据"。

#### M-03【MEDIUM】Actors 列表缺少"数据分析师"

- **维度**：Actor 完整性
- **位置**：涉及 Actors 章节（第 34-44 行）
- **理由**：Feature 9 Scenario #6（第 326 行）直接使用"数据分析师"作为场景 Actor（"数据分析师执行跨精度联合查询"），但"数据分析师"未在本文档的 Actors 列表中出现。虽然 product-spec.md 中定义了"数据分析师" Actor，但本文档作为独立特性域文档，其 Actors 列表应自包含——读者不应需要跨文档查找才能理解场景中的角色。
- **修复建议**：在涉及 Actors 章节补充"[数据分析师](../requirements/product-spec.md#actor-数据分析师)（协作 Actor：执行跨精度查询、依赖精度元数据进行分析决策）"。

#### L-01【LOW】3 个 Actor 链接在 product-spec.md 中无对应锚点

- **维度**：文档一致性
- **位置**：涉及 Actors 章节中的"外部身份提供者（IDP）""安全信息与事件管理系统（SIEM）""被作用主体"
- **理由**：这 3 个 Actor 的 Markdown 链接指向 `product-spec.md#actor-xxx`，但 product-spec.md 中并未定义这 3 个 Actor（仅定义了"外部密钥管理系统（KMS）"等 19 个 Actor）。链接会失效。虽然概念上这 3 个 Actor 合理（IDP 用于认证、SIEM 用于审计对接、被作用主体是被授权对象），但应在 product-spec 中补充定义，或本文档改为不链接的文本描述。
- **修复建议**：在 product-spec.md 中补充这 3 个 Actor 的定义，或本文档中移除超链接改为纯文本引用。

#### L-02【LOW】Feature 10 矩阵备注过于笼统

- **维度**：可追溯性
- **位置**：矩阵备注（第 387 行）"敏感标签'节点故障/网络分区/资源耗尽'列标记为'—'：标签为轻量元数据，故障场景由底层存储和 RBAC 域覆盖"
- **理由**：备注称"由底层存储和 RBAC 域覆盖"但未指明具体是哪些 Feature 的哪些 Scenario。读者无法验证该声明是否属实。例如，RBAC 域的节点故障场景是 #8（网络分区期间权限校验），存储域的节点故障场景是 #5（加密迁移崩溃恢复）——这些是否真的能覆盖敏感标签的故障场景？
- **修复建议**：备注中引用具体 Scenario 编号，例如"由底层存储域 Feature 6 #5（加密迁移崩溃恢复）和 RBAC 域 Feature 3 #8（分区期间权限校验）覆盖"。
