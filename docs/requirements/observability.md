## Feature: 全链路性能指标采集

**做什么**：持续采集并暴露 KV Cache 从查询、加载、传输、存储到淘汰的全链路性能指标，包括延迟分布、吞吐、命中率、以及各层资源利用率。

**为什么**：可量化的指标是优化和运维的基础。没有指标，无法回答 "命中率是多少"、"瓶颈在 SSD 还是网络"、"扩容是否有效" 等问题。

**涉及 Actor**：缓存服务实例、数据传输服务、全局元数据服务、监控系统、平台工程师。

### Scenario: 正常路径 — 指标持续采集和暴露

**Actor**：缓存服务实例、数据传输服务、监控系统

**前置条件**：系统正常运行；Prometheus 已配置抓取本系统的 /metrics 端点。

**触发动作**：监控系统每 15s 抓取一次指标。

**预期行为**：/metrics 端点响应时间 < 50ms；指标覆盖 TTFT/TPOT 分解、各层加载延迟(P50/P90/P99)、吞吐量(GB/s)、命中率、错误率、CPU/内存/磁盘/网络利用率；采集任务对主流程 CPU 占用 < 3%。

**验证方法**：连续运行 24 小时：指标缺失率 = 0%；/metrics P99 响应时间 < 50ms；采集期间主请求延迟增加 < 2%；Prometheus 中可见 ≥ 30 个核心指标。

### Scenario: 节点故障 — 故障节点指标缺失触发告警

**Actor**：缓存服务实例、监控系统、平台工程师

**前置条件**：某缓存服务实例突然故障；Prometheus 连续 2 个采集周期未收到该实例指标。

**触发动作**：实例进程崩溃。

**预期行为**：监控系统在 1 分钟内触发 `instance_down` 告警；告警标签包含实例 ID、节点名、最后已知状态；其他节点指标采集不受影响。

**验证方法**：kill 一个实例后：Prometheus 在 30s 内标记该 target 为 DOWN；1 分钟内 Grafana 出现 `instance_down` 告警；其他实例指标正常更新。

### Scenario: 运维操作 — 管理员动态调整指标采样率

**Actor**：平台工程师、缓存服务实例

**前置条件**：当前 histogram 采样率为 100%；CPU 负载较高，管理员决定降低延迟直方图采样率至 10%。

**触发动作**：通过运维 API 或热更新配置修改采样率。

**预期行为**：配置变更在 30s 内对所有实例生效；histogram 数据量减少约 90%；核心计数器（命中率、吞吐量）仍保持 100% 精确采集；延迟趋势仍可通过低采样率推断。

**验证方法**：变更后 60s 内：新采集的 histogram sample_count 下降 ≥ 85%；counter 指标无遗漏；/metrics 响应大小下降 ≥ 50%；主请求延迟无显著变化。

### Scenario: 升级/回滚 — 升级期间指标不中断

**Actor**：缓存服务实例、监控系统、平台工程师

**前置条件**：集群执行滚动升级；被升级实例需要重启。

**触发动作**：实例逐个重启。

**预期行为**：升级实例的 /metrics 在重启后 10s 内恢复可访问；Prometheus 最多丢失 1 个采集周期的数据；升级期间全局仪表盘不中断；历史数据连续性不受影响。

**验证方法**：滚动升级 3 节点集群：单个实例指标缺失周期 ≤ 1（即 ≤ 15s）；全局聚合指标（如集群总命中率）曲线无断崖式下跌；升级期间 Dashboard 可正常刷新。

### Scenario: 引擎侧指标集成

**Actor**：推理引擎、监控系统、缓存服务实例

**前置条件**：系统需要计算 "TTFT 降低 >= 50%" 等收益指标；引擎已暴露 baseline 指标端点。

**触发动作**：监控系统同时抓取引擎 baseline 指标和缓存服务侧指标。

**预期行为**：引擎暴露 `vllm_ttft_without_cache`、`vllm_cache_hit_rate`、`vllm_tpot_with_cache` 等 baseline 指标；缓存服务侧采集 `cache_ttft_with_load`、`cache_hit_rate` 等指标；收益计算公式为 `(baseline_ttft - cache_ttft) / baseline_ttft`，误差 < 5%；Dashboard 同时展示引擎侧和缓存服务侧的关联视图。

**验证方法**：在相同 workload 下分别采集引擎 baseline 和开启缓存后的指标：收益计算误差 < 5%；引擎指标与缓存服务指标在 Grafana 中可关联展示；0 次因指标缺失导致收益面板为空。

## Feature: 故障诊断与追踪

**做什么**：为每个 KV Cache 请求和后台任务生成可关联的 trace、日志和事件记录，支持跨节点、跨层级的端到端故障定位。

**为什么**：分布式系统中的故障往往跨越多个节点和层级。没有 trace，很难回答 "为什么这个请求的 TTFT 突然 spike"、"数据在哪个环节丢失" 等问题。

**涉及 Actor**：缓存服务实例、数据传输服务、全局元数据服务、平台工程师、监控系统。

### Scenario: 正常路径 — Trace 完整记录请求生命周期

**Actor**：推理引擎、缓存服务实例、数据传输服务

**前置条件**：OpenTelemetry Collector 已配置；系统开启 1% trace 采样。

**触发动作**：引擎发起一个 prefix 匹配 + 加载请求。

**预期行为**：trace 包含以下 span：prefix 查询、索引查找、数据传输启动、逐层 wait、加载完成、引擎消费；各 span 包含延迟、状态码、错误信息、数据来源节点/层级标签；端到端 trace 在 95% 情况下完整（无 span 丢失）。

**验证方法**：采集 1000 条 trace：完整 trace 比例 ≥ 95%；各 span 时间戳单调递增；trace ID 在引擎和缓存服务间正确传递；Jaeger/Tempo 中可检索到全部 trace。

### Scenario: 网络分区 — 分区期间 Trace 标记异常

**Actor**：缓存服务实例、数据传输服务、全局元数据服务

**前置条件**：某加载请求执行过程中发生 10s 网络分区。

**触发动作**：分区导致某 span（如数据传输）超时。

**预期行为**：超时 span 标记 `error=true` 并附带 `timeout` 类型；上游 span 能清晰展示超时发生的层级和节点；trace 不中断，请求 fallback 路径的 span 也被完整记录。

**验证方法**：注入分区后检索受影响 trace：100% 显示 timeout 错误标签；timeout span 的 duration 与配置超时偏差 < 500ms；fallback 路径（如重算）的 span 可被检索。

### Scenario: 数据损坏 — 检测到异常命中，Trace 定位根因

**Actor**：缓存服务实例、平台工程师

**前置条件**：某请求的加载数据 checksum 校验失败。

**触发动作**：引擎加载数据并发现校验失败。

**预期行为**：trace 中加载 span 标记 `checksum_mismatch`；关联日志输出数据来源节点、数据块 ID、存储层级、最后修改时间；平台工程师可在 1 分钟内从 trace 定位到具体源节点和文件路径。

**验证方法**：模拟 10 次校验失败：100% 的 trace 中包含 `checksum_mismatch` 标签；日志中包含准确的 node_id + chunk_id + layer 信息；人工排查平均定位时间 < 1 分钟。

### Scenario: 并发冲突 — 高并发下 Trace 采样不丢关键事件

**Actor**：缓存服务实例、数据传输服务

**前置条件**：系统处于高并发状态；trace 采用 1% 采样；某个请求恰好是已采样的 trace 且涉及并发竞争。

**触发动作**：高并发读取与后台迁移/淘汰同时发生。

**预期行为**：即使并发竞争激烈，已采样 trace 的关键 span（如竞争开始、锁获取、操作完成）不丢失；trace 能正确展示请求在并发控制下的等待时间和实际执行时间。

**验证方法**：在 10 万 QPS 下运行 1 小时：抽样检查 100 条高并发期间的 trace：关键 span 缺失率 < 0.1%；并发等待时间被准确记录；无因 trace buffer 满导致的 span 丢弃告警。

### Scenario: 基于结构化日志的非采样故障定位

**Actor**：平台工程师、缓存服务实例、监控系统

**前置条件**：某请求未命中 1% trace 采样，但在执行过程中发生了 checksum mismatch 或网络超时。

**触发动作**：平台工程师接收到 `cache_data_corruption` 或 `load_timeout` 告警，需要定位根因。

**预期行为**：所有关键操作（查询发起、加载开始、加载完成、错误标记）均输出结构化日志，包含 request_id、trace_id（可选）、node_id、chunk_id、layer、存储层级、错误码；平台工程师可通过日志聚合查询（如 Loki / Elasticsearch）在 1 分钟内定位到具体源节点和文件路径；关键错误事件（checksum mismatch、timeout、node_down）实施 100% trace 尾部采样，不受全局 1% 采样率限制。

**验证方法**：模拟 10 次未命中 trace 采样的 checksum mismatch 事件：通过日志查询在 1 分钟内定位到 node_id + chunk_id 的成功率 100%；日志中 request_id 与引擎侧日志可关联率 >= 95%。

## Feature: 容量与成本分析

**做什么**：基于缓存命中率、数据生命周期、存储介质成本模型，为平台工程师提供容量预测、成本归因和扩容建议。

**为什么**：缓存服务的运营成本与存储介质选择直接相关。平台工程师需要知道 "当前 DRAM/SSD 配比是否最优"、"下个月是否需要扩容"、"不同业务线的缓存成本分别是多少"。

**涉及 Actor**：平台工程师、监控系统、缓存服务实例。

**MVP 范围声明**：成本模型采用可配置参数（DRAM 单价 $/GB、SSD 单价 $/GB、附加费用率 %）。MVP 阶段仅支持单租户、单一云厂商的近似成本估算；多租户成本拆分、跨云厂商定价差异、IOPS / 网络出口等精细化计费列为 P2 后续迭代。

### Scenario: 正常路径 — 容量预测准确

**Actor**：平台工程师、监控系统、缓存服务实例

**前置条件**：系统已运行 7 天以上；有历史存储增长曲线和命中率数据。

**触发动作**：监控系统每日生成容量预测报告。

**预期行为**：报告预测未来 7 天的 DRAM/SSD 使用趋势；预测误差 < ±15%（基于历史同期）；当预测利用率将超过 85% 时发出预警。

**验证方法**：对比 7 天前的预测和实际使用量：平均绝对百分比误差（MAPE）< 15%；预警触发时间比实际达到 85% 的时间提前 ≥ 24 小时。

### Scenario: 资源耗尽 — 容量预警提前发出

**Actor**：监控系统、平台工程师、缓存服务实例

**前置条件**：某层存储按当前增长速率将在 3 天内达到 95%。

**触发动作**：监控系统每日容量评估任务检测到风险。

**预期行为**：在达到 95% 前至少 72 小时发出 `capacity_warning` 告警；告警包含建议措施（如清理冷数据、扩容节点、调整淘汰策略）；告警按严重程度分级（ INFO / WARNING / CRITICAL ）。

**验证方法**：人为陡增写入速率使某层 3 天内将达 95%：72 小时前收到 WARNING 告警；48 小时前收到 CRITICAL 告警；告警建议措施中至少包含 "扩容" 和 "加速淘汰" 两项。

### Scenario: 节点故障 — 故障后容量重新评估

**Actor**：监控系统、全局元数据服务、平台工程师

**前置条件**：集群中 1 个节点故障下线；剩余节点需要承担该节点的数据副本或流量。

**触发动作**：节点故障自动恢复流程完成后，监控系统更新容量视图。

**预期行为**：监控系统在 5 分钟内更新可用容量视图；重新计算剩余节点的安全边际；若剩余容量不足，立即触发 CRITICAL 告警。

**验证方法**：kill 1 个节点后：5 分钟内监控面板可用容量减少量 = 该节点原容量；若剩余节点平均利用率 > 80%，CRITICAL 告警在 5 分钟内触发。

### Scenario: 运维操作 — 管理员手动触发容量报告

**Actor**：平台工程师、缓存服务实例、监控系统

**前置条件**：平台工程师需要评审当前缓存成本结构。

**触发动作**：通过运维 CLI 触发即时容量与成本报告。

**预期行为**：报告在 30s 内生成；包含：各层当前使用量/总量/增长率、命中率按业务标签拆分、预估月度存储成本（按 DRAM $/GB 和 SSD $/GB 计算）、Top 10 占用 prefix 列表。

**验证方法**：触发报告后：生成时间 < 30s；报告中数据与实时监控差异 < 5%；Top 10 prefix 总占用量与实际扫描结果差异 < 3%。

## 本域非功能需求

| 子项 | 目标值 |
|------|--------|
| 指标覆盖 | ≥ 50 个 Prometheus 指标，覆盖延迟、吞吐、命中率、错误率、资源利用率 5 大维度 |
| 指标采集开销 | /metrics 响应 P99 < 50ms；主流程 CPU 占用 < 3% |
| 指标缺失率 | 连续运行 24 小时，指标缺失率 = 0% |
| 告警时效 | 关键故障（节点 down、数据损坏、容量满）1 分钟内触发告警 |
| Trace 完整性 | 1% 采样下端到端 trace 的 span 缺失率 < 5%；关键错误事件（checksum mismatch、timeout、node_down）实施 100% 尾部采样 |
| 关键 span 缺失率 | 高并发期间已采样 trace 的关键 span 缺失率 < 0.1%；关键 span 包括查询发起、加载完成、错误标记 |
| 故障定位时效 | 数据损坏场景下，人工从 trace 定位到源节点和文件路径的平均时间 < 1 分钟 |
| 容量预测准确度 | MAPE < 15%；预警比实际达到 85% 提前 ≥ 24 小时 |
| 容量预警时效 | 达到 95% 前至少 72 小时发出 WARNING；48 小时前发出 CRITICAL |
| 报告生成时效 | 手动触发容量与成本报告，生成时间 < 30s；数据与实时监控差异 < 5% |
