### 架构问题清单

#### H-A-01【HIGH】Scenario 中"专用查询计划优化路径"泄漏内部子系统设计

- **维度**：架构耦合识别
- **位置**：Feature「告警评估通道」价值论证 第 242 行 第 3 项："专用查询计划优化路径：跳过通用查询优化器的成本估算阶段，直接选择索引扫描+最近分区裁剪的最短路径"
- **理由**：产品需求文档不应规定"跳过成本估算阶段""选择索引扫描+最近分区裁剪"等执行计划生成机制——这是查询引擎子系统的内部架构决策，应在 ADR/子系统 design 中讨论。需求层只需声明"告警通道在通用查询优化器满载时仍能保证 P99 < 500ms"。该描述会反向锁死实现方案，剥夺架构师在子系统设计阶段评估其他方案（如 plan cache、专用 short-path executor、混合策略）的空间。
- **修复建议**：将第 3 项重写为效果型描述："专用快路径：在通用查询规划阶段拥塞时仍保证规划阶段耗时 < 50ms（具体实现机制由子系统架构决定）"。

#### H-A-02【HIGH】"独立 SLA 承诺"与"承诺 P99 < 500ms（不受通用查询负载影响）"存在不可证伪表述

- **维度**：NFR 合理性 / 可验收性
- **位置**：Feature「告警评估通道」第 240-244 行 + 第 259 行 Scenario #4："系统写入负载达峰值 80% 且分析查询通道满载时，告警查询 P99 < 500ms"
- **理由**：(1)"不受通用查询负载影响"是绝对化表述，无法落地——任何共享存储/网络 IO 的系统都会有耦合，即使资源完全独立的进程也会因磁盘带宽、PageCache 受影响。(2) 80% 写入 + 分析通道满载只是单一负载点，未定义"超出 80%"或"分析通道极端拥塞"时的 SLA 退化曲线。架构上需要清晰的退化策略（拒绝/降级/排队），但当前承诺暗示"无论何种负载都 < 500ms"。
- **修复建议**：补充负载-延迟分段承诺表，例如"写入 ≤ 80% 峰值 + 分析满载：P99 < 500ms；写入 > 80% 时进入降级窗口，承诺退化为 P99 < 1s 并触发告警"。

#### H-A-03【HIGH】SQL/PromQL/PG Wire 三协议语义对齐缺失

- **维度**：跨 Feature 一致性
- **位置**：Feature 1（SQL）、Feature 2（PromQL）、Feature 10（PG Wire）
- **理由**：三种协议都允许查询同一份时序数据，但文档未约束以下语义对齐：(1) 时间范围语义：SQL 默认本地时区 vs PromQL UTC vs PG Wire timestamp with time zone，是否要求三者对同一查询返回一致结果？(2) 空值处理：SQL 的 NULL vs PromQL 的"无数据点"vs PG Wire NULL，是否要求语义等价？(3) 聚合差异：PromQL 的 `rate()` 与 SQL 的窗口差分在乱序数据下结果可能不同。架构上必须在需求层先声明"三协议结果在等价查询下的允许偏差"，否则子系统设计阶段会陷入"对齐到何种程度"的反复返工。
- **修复建议**：在 Feature 10 后增设跨协议语义对齐 Feature 或在「本域非功能需求」补一行"协议语义一致性"——明确三协议对同一逻辑查询的结果偏差容忍度（如时间戳一致到秒、聚合数值偏差 < 0.1%、NULL 语义等价）。

#### H-A-04【HIGH】资源隔离与缓存的交互未定义

- **维度**：跨 Feature 一致性
- **位置**：Feature 3「资源隔离」 + Feature 4「查询缓存」
- **理由**：缓存命中是否绕过资源配额？两种合理设计相互冲突：(1) 若绕过：低优先级查询可通过缓存命中"白嫖"配额，破坏隔离；(2) 若不绕过：缓存意义削弱，命中查询仍占用配额计算的"虚拟时间"。当前需求未表态。Feature 3 #1 "告警查询 P99 延迟劣化 < 20%"是否假设缓存被禁用以保证隔离纯净？这是架构上必须先明确的需求层决策。
- **修复建议**：在 Feature 3 或 Feature 4 价值论证补一段"缓存命中的资源计费策略"——明确选择哪种语义，并在 Scenario 中加一例验证（例如"低优先级缓存命中查询不消耗资源组配额，但每秒最多 N 次"）。

#### H-A-05【HIGH】快照与告警通道的实时性语义冲突

- **维度**：跨 Feature 一致性
- **位置**：Feature 9「一致性快照」 + Feature 8「告警评估通道」 第 258 行 Scenario #3："查询刚写入 2s 内的数据点，返回包含最近 2s 内写入的数据点"
- **理由**：告警通道要求"刚写入 2s 内可见"，意味着查询路径上不能有快照隔离层；而快照 Feature 提供"严格时间点一致"的快照视图。如果二者共用同一查询执行栈，则需要在路由层区分"snapshot read"与"realtime read"，但当前需求未声明这是同一路径还是分离路径。架构上影响巨大：(a) 同一路径需要双语义执行计划，复杂度高；(b) 分离路径需要两套读路径子系统。
- **修复建议**：在「跨域依赖说明」后增"读路径语义分层"小节，明确告警通道、PG Wire、SQL 默认是否走快照视图，是否支持显式 `AS OF SNAPSHOT 'xxx'` 语法。

#### H-A-06【HIGH】流式订阅与批量导出的资源竞争场景缺失

- **维度**：故障模式完整性
- **位置**：Feature 5「流式订阅」 + Feature 6「批量导出」
- **理由**：两者都是长时间持续占用资源的查询模式（流式订阅长连接持续推送，批量导出长时间扫描），故障模式矩阵中"资源耗尽"列各自只引用了内部 Scenario，未覆盖"流式订阅 1000 个长连接 + 批量导出 4 路并行"的叠加场景。架构上需要明确流式推送的内存配额是否受批量导出影响、批量导出是否会拖慢流式推送的 P99 延迟。
- **修复建议**：在两者的 Scenario 中增补一例叠加场景，或在「本域验收标准」加"持续写入 10 万点/秒 + 流式订阅 1000 连接 + 批量导出并发 4 路时，告警 P99 < 500ms 且推送 P99 < 1s"。

#### H-A-07【HIGH】NFR 互相兼容性未做交叉校验

- **维度**：NFR 合理性
- **位置**：第 388-401 行 NFR 表 + 各 Feature 成功指标
- **理由**：以下指标在同一集群是否能同时达成未论证：
  - 告警评估 P99 < 500ms（满载场景） + 批量导出 500MB/s 单节点 + 流式订阅写入到推送 P99 < 1s + 缓存新鲜度 < 5s + 快照创建 < 100ms + 并发查询 ≥ 100。
  - 例：批量导出 500MB/s 在 SSD 顺序读已逼近极限，此时 PageCache 被冲刷，告警通道还能保证 < 500ms 吗？流式推送的写入感知延迟还能保证 < 1s 吗？
  - 缺少"在所有 NFR 同时达成时的最小硬件配置"或"哪些 NFR 不能同时达成（互斥）"的声明。
- **修复建议**：在「本域非功能需求」表后追加"NFR 交叉兼容性"段，列出确认可以同时达成的指标组合，以及互斥场景下的优先级（如告警 P99 优先于批量吞吐）。

#### H-A-08【HIGH】跨时间线 JOIN 的下推语义未声明

- **维度**：架构耦合 / 演进路径
- **位置**：Feature 5（实为 Feature「跨时间线关联查询」）价值论证 第 155 行："计算下推到存储层"
- **理由**：(1) "下推到存储层"已经是子系统职责划分，需求层不应规定下推策略；(2) 下推语义影响存储层接口契约（必须支持 join hint、列裁剪、谓词下推），需要在子系统设计阶段评估，需求层只需说"3 条时间线 7 天关联 P95 < 2s"。当前表述会导致存储域设计被反向锁死。
- **修复建议**：删除"计算下推到存储层"，替换为"在不传输全量原始数据的前提下完成关联计算"——保留性能效果，去除架构机制。

#### M-A-09【MEDIUM】告警评估通道的"两个评估周期内必须完成"未定义周期长度

- **维度**：可验收性
- **位置**：第 261 行 Scenario #6："单条规则最大评估延迟不超过 2 个评估周期（30s）"
- **理由**："2 个评估周期 = 30s"暗示评估周期是 15s，但 Feature 价值论证（第 235 行）说"每 5-15 秒执行一轮"。周期长度是配置项还是固定值？若可配置，"2 个周期"的承诺在 5s 周期下是 10s，在 15s 周期下是 30s，验收标准随之浮动，需要明确。
- **修复建议**：声明评估周期为 Actor 提供的请求参数，最大延迟统一表达为"≤ 2 × 评估周期且 ≤ 30s"。

#### M-A-10【MEDIUM】PG Wire 不支持的 SQL 语法清单不闭合

- **维度**：演进路径
- **位置**：Feature 10 第 319 行 Scenario #4："执行不支持的 SQL 语法（JOIN、子查询、DDL）"
- **理由**：当前列举"JOIN、子查询、DDL"为不支持，但 Feature 5 明确支持跨时间线 JOIN。两者矛盾：是 PG Wire 子集刻意不暴露 JOIN 还是 JOIN 仅限私有 API？演进路径上若未来需要在 PG Wire 暴露 JOIN，旧客户端语义如何处理？
- **修复建议**：在 Feature 10 增补"PG Wire 当前 SQL 子集与 Feature 1 SQL 引擎全集的差异说明"，并在待澄清问题中加入"PG Wire 是否计划达到 SQL 引擎全集"。

#### M-A-11【MEDIUM】缓存节点宕机的 60s 命中率回升 SLA 缺乏依据

- **维度**：可验收性
- **位置**：第 145 行 Scenario #7："缓存节点恢复后 60s 内命中率回升至 50%+"
- **理由**：60s 与 50% 的具体阈值缺乏建模——回升速度取决于查询模式（仪表盘刷新周期 15-30s）和缓存预热策略。在低 QPS 场景下 60s 内可能根本没有 50% 的查询发生过。
- **修复建议**：将"60s 内"改为"在每秒 ≥ N 次查询持续负载下，缓存节点恢复后 60s 内命中率回升至 50%+"。

#### M-A-12【MEDIUM】"协议升级"故障模式被矩阵备注 [3] 默认免除

- **维度**：演进路径
- **位置**：第 351 行备注 [3]："查询协议版本兼容性由各自协议 Feature 的兼容性保证（PromQL/SQL 语义版本化），不单独设计升级 Scenario"
- **理由**：备注称"PromQL/SQL 语义版本化"在文档中没有任何 Feature/Scenario 体现，没有 SQL/PromQL 的版本协商或语义版本号声明，备注 [3] 实际上是空头支票。只有 PG Wire 有升级 Scenario（#10、#11），其他两协议升级时如何保证兼容性是真空。
- **修复建议**：补充"SQL/PromQL 函数集语义版本"NFR，或承认这是已知缺口（加入待澄清问题列表）。

#### L-A-13【LOW】跨时间线 JOIN 的"查询计划缓存被淘汰"是子系统术语

- **维度**：架构耦合
- **位置**：第 173 行 Scenario #9："关联查询执行期间查询计划缓存被淘汰，系统重新生成计划后继续执行"
- **理由**:"查询计划缓存"是查询引擎子系统的内部机制，需求层不应直接引用。该 Scenario 的实际意图是验证"重规划不影响正在执行的查询"，应该用效果性描述。
- **修复建议**：改写为"长时间执行的关联查询在中途遇到内部状态刷新（如配置变更、热数据迁移）时不中断"。

### 隐性假设识别（≥2）

1. **隐含假设：缓存路径与资源隔离正交**——文档默认两者独立设计，未讨论缓存命中是否计入资源配额。该假设被证伪（任一选择都需要在 Feature 间协调）。
2. **隐含假设：所有 NFR 在单一硬件配置下可同时达成**——批量导出 500MB/s + 告警 P99 500ms + 流式 P99 1s + 缓存新鲜度 5s 之间没有冲突分析。
3. **隐含假设：三协议（SQL/PromQL/PG Wire）共用同一查询执行栈**——价值论证暗示通过统一引擎实现，但快照语义、告警通道差异化执行计划暗示需要分层路径，假设未被显式验证。

### 遗漏 Trade-off 补充

1. **缓存新鲜度 vs 命中率 vs 资源隔离**：缩短缓存失效窗口（< 5s）会降低命中率（80%→？）；当前文档单方面承诺两者都达标，未做 trade-off 量化。
2. **告警通道独立性 vs 集群资源利用率**：专用通道+独立 SLA 承诺需要预留资源容量，集群整体利用率必然下降。需求层未声明能接受多大的预留浪费。
