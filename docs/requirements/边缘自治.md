## Feature: 支持边缘节点离线自治运行

### Feature 价值论证

**用户痛点**：工厂/站点边缘节点部署在网络条件恶劣的环境（偏远矿区、海上平台、地下管廊），与云端连接可能中断数小时甚至数天。中断期间 IoT 设备群仍在持续产生传感数据，运维工程师仍需查询近期数据判断设备健康状态。

**现有方案不足**：典型边云架构中边缘节点依赖云端下发配置或确认写入状态。一旦断网，节点要么拒绝写入（强一致模式），要么进入只读状态等待重连。

**本 Feature 价值**：边缘节点具备完整的离线自治能力——写入路径零依赖云端确认，本地查询引擎独立工作，TTL 清理按本地时钟自主执行。

**成功指标**：
- 基础目标（P0 必须）：连续 24 小时完全断网期间写入成功率 >= 99.99%
- 增强目标（P0 努力方向）：连续 7 天完全断网期间写入成功率 >= 99.99%
- 断网期间本地查询可用性 >= 99.9%，P99 响应时间 <= 200ms

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | IoT 设备群在边缘节点与云端完全断网后持续推送写入请求，边缘节点独立完成数据持久化 | 断网期间写入确认延迟 P99 < 100ms，写入成功率 >= 99.99% | 自动化 |
| 2 | 平台运维工程师在断网期间对边缘节点执行本地诊断查询（通过本地终端或局域网内管理接口） | 查询可用性 >= 99.9%，P99 响应时间 <= 200ms | 自动化 |
| 3 | TTL 清理触发器在断网期间按本地时钟到达清理周期，边缘节点自主执行过期数据删除 | 过期数据在 TTL 到期后 1 小时内被清理，清理期间写入延迟增幅 <= 10% | 自动化 |
| 4 | 边缘节点持续离线 7 天，本地存储使用率从 30% 增长至接近容量上限 | 存储达阈值（可配置，默认 90%）前写入成功率 >= 99.99% 且写入延迟 P99 < 100ms，达阈值后触发告警并加速 TTL 清理 | 自动化 |
| 5 | 边缘节点在离线期间发生进程崩溃或硬件重启，重启后无需云端参与即自主恢复 | 冷启动至可接受写入 <= 30 秒，WAL 恢复后数据零丢失 | 自动化 |
| 6 | 边缘节点离线期间本地存储物理耗尽，系统执行降级策略 | 存储耗尽时拒绝新写入并返回明确错误码，已有数据查询不受影响 | 自动化 |
| 7 | 边缘节点离线期间需要变更配置，但无法从云端拉取新配置 | 返回"离线不可变更"提示，系统按断网前最后生效配置稳定运行 | 自动化 |
| 8 | 边缘节点从离线状态恢复网络连接，系统自动识别重连并从自治模式平滑切换回协同模式 | 重连后 60 秒内启动增量同步，切换期间本地写入和查询服务不中断，性能下降 <= 15% | 自动化 |
| 9 | 边缘节点离线期间检测到本地存储数据损坏（WAL CRC 校验失败） | 损坏段被标记跳过，新数据写入不受影响，系统生成告警且告警内容包含损坏数据的时间范围 | 自动化 |
| 10 | 边缘节点离线期间本地时钟发生显著漂移（>1 分钟，无 NTP 同步），重连后系统检测到时钟偏差并修正数据时间戳 | 重连后时间戳修正机制生效，不因时钟漂移导致数据丢失或 TTL 误删，修正后时间戳偏差 <= 1 秒 | 自动化 |
