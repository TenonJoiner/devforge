## Feature: 实现节点故障自动检测与副本切换

### Feature 价值论证

**用户痛点**：7x24 高写入场景下节点宕机时，人工发现并手动切换需 5-15 分钟，期间该节点负责的分片写入全部失败。

**现有方案不足**：简单 ping 检测无法区分"进程假死但端口仍在"与"真正宕机"；切换依赖人工介入；缺乏对数据一致性边界的明确定义。

**本 Feature 价值**：通过多维度健康探测实现秒级故障判定，结合自动副本提升，将故障恢复从"人工分钟级"降至"自动秒级"；显式定义 RPO 边界，让运维工程师清楚知道故障期间可能丢失的数据范围。

**成功指标**：
- 单节点宕机后受影响分片写入恢复时间（RTO）<= 30 秒
- 故障检测误判率 <= 0.1%/月
- 异步复制模式下 RPO <= 1 秒；quorum 写入模式下 RPO = 0
- 心跳检测周期默认 2 秒，可配置范围 1-10 秒

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 某数据节点进程崩溃后系统自动检测并将分片切换到健康副本 | 从节点不可达到写入恢复 <= 30s | 自动化 |
| 2 | 容器编排平台对实例执行健康检查，实例返回正确健康状态 | 健康检查响应延迟 <= 200ms（P99） | 自动化 |
| 3 | 副本切换完成后运维工程师查看切换事件记录 | 切换事件 <= 5s 内可查询，包含故障节点 ID、耗时、新主副本 ID | 自动化 |
| 4 | 某节点因网络抖动导致心跳短暂丢失后恢复，系统不误判 | 单次心跳丢失不触发切换，连续 >= 3 个周期（默认 6s）未响应才判定故障 | 自动化 |
| 5 | 集群出现网络分区，少数派侧不提升新主，多数派侧正常切换 | 分区期间不出现双主写入；分区恢复后少数派自动降级 | 自动化 |
| 6 | 副本切换过程中已确认写入在新主上可读，未确认写入返回可重试错误 | 已确认写入完整性 100%；未确认写入返回可重试错误码；异步模式 RPO <= 1s，quorum 模式 RPO = 0 | 自动化 |
| 7 | 故障节点恢复上线后自动重新加入集群作为副本并追赶数据 | 恢复后 <= 60s 内重新加入，追赶期间主副本吞吐下降 <= 5% | 自动化 |
| 8 | 副本切换完成后客户端读取数据，系统保证单调读一致性并标记可能过期的数据 | 切换后读取同一 key 的值不会回退；stale 标记通过查询响应元数据暴露，由客户端 SDK 自动处理或供应用开发者显式判断 | 自动化 |
| 9 | 运维工程师执行计划内节点维护，系统主动触发副本切换后暂停原节点写入 | 升级暂停时间 <= 10s，期间受影响分片通过副本继续服务 | 自动化 |
| 10 | 应用开发者客户端在副本切换期间写入数据，SDK 自动重定向到新的主副本 | 切换期间未确认写入返回可重试错误码（含指数退避建议：初始 100ms，最大 5s，因子 2）；切换完成后 SDK 自动重连新主副本，重连耗时 <= 5s | 自动化 |

## Feature: 支持集群节点动态增删与数据再平衡

### Feature 价值论证

**用户痛点**：时序数据量随传感器数量线性增长，运维工程师需要定期扩容。传统方案扩容需停机迁移数据。

**现有方案不足**：停机扩缩容不可接受；在线扩容时再平衡占用过多 IO 会挤压写入吞吐；缺乏磁盘带宽饱和时的前台写入保护机制。

**本 Feature 价值**：提供不停机的节点增删能力，配合带宽可控的数据再平衡机制；在磁盘 IO 带宽饱和时保证前台写入劣化可控。

**成功指标**：
- 节点增删期间集群写入可用性 >= 99.9%
- 再平衡期间写入延迟 P99 劣化 <= 20%（基线：无再平衡时 P99 < 100ms，劣化后 P99 < 120ms）
- 磁盘 IO 带宽饱和时前台写入劣化 <= 20%（基线同上）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 运维工程师向集群添加新节点，系统自动将部分分片迁移到新节点 | 再平衡期间写入成功率 >= 99.9% | 自动化 |
| 2 | 运维工程师标记某节点为待下线，系统自动迁移数据后安全移除 | 迁移完成后该节点数据副本数为 0，不导致任何分片副本数低于最小值 | 自动化 |
| 3 | 容器编排平台触发滚动升级，逐个替换节点实例 | 升级全程写入可用性 >= 99.9% | 自动化 |
| 4 | 再平衡进行中运维工程师调整带宽限制 | 带宽调整后 <= 10s 生效 | 自动化 |
| 5 | 再平衡进行中目标节点突然故障 | 30s 内迁移任务切换到新目标，已迁移数据不丢失 | 自动化 |
| 6 | 运维工程师同时添加多个节点（批量扩容） | 单个源节点同时迁出分片数 <= 配置上限，写入延迟 P99 劣化 <= 20% | 自动化 |
| 7 | 集群仅剩最小副本数时运维工程师尝试下线节点 | 操作被拒绝并返回受影响分片列表 | 自动化 |
| 8 | 磁盘 IO 带宽饱和时前台写入与再平衡流量竞争 | 前台写入延迟增幅 <= 20%；再平衡流量受 Feature 6 配额体系约束，自动降速，最终完成时间延长但不失败 | 自动化 |
| 9 | 再平衡过程中运维工程师触发升级回滚 | 回滚指令 <= 10s 内生效，已迁移数据不丢失，未迁移分片保持原位置 | 自动化 |

## Feature: 管理边缘节点注册与拓扑发现

### Feature 价值论证

**用户痛点**：边缘侧节点数量多、网络环境复杂，运维工程师无法逐一手动配置连接关系，也无法实时掌握哪些节点在线。

**现有方案不足**：静态配置无法适应边缘节点的动态性，当规模达数百时人工管理成本不可接受；网络分区期间边缘节点完全不可用，丢失本地采集数据。

**本 Feature 价值**：提供自动注册、心跳保活和拓扑自动发现机制，新节点即插即用，失联节点自动标记；网络分区期间边缘节点可继续本地写入，分区恢复后自动同步。

**成功指标**：
- 新边缘节点从首次启动到完成注册 <= 60 秒
- 失联后云端拓扑状态更新延迟 <= 心跳超时周期 + 10 秒
- 网络分区恢复后数据同步触发延迟 <= 2 个心跳周期

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 新边缘节点首次启动，携带预置凭证向云端注册 | 注册完成时间 <= 60s，注册后立即出现在拓扑视图 | 自动化 |
| 2 | 已注册节点按配置周期发送心跳，运维工程师查看各节点在线状态 | 心跳周期可配置（默认 10s），拓扑视图展示在线/离线状态 | 自动化 |
| 3 | 运维工程师查询当前集群拓扑 | 响应时间 <= 1s，包含节点 ID、角色、所属区域、连通状态 | 自动化 |
| 4 | 大量边缘节点同时启动（注册风暴） | 100 个节点 10s 内同时注册，成功率 >= 99% | 自动化 |
| 5 | 边缘节点因网络波动连续心跳丢失 | 连续 3 次标记"疑似离线"，超过 5 个周期标记"离线"，恢复后自动回到"在线" | 自动化 |
| 6 | 云端与某区域边缘节点群发生网络分区 | 分区恢复后 <= 2 个心跳周期内重新建立连接 | 自动化 |
| 7 | 某边缘节点凭证过期或被吊销 | 请求被拒绝并返回错误码，云端生成安全告警 | 自动化 |
| 8 | 网络分区期间边缘节点继续本地写入，分区恢复后触发数据同步 | 分区期间本地写入不丢失（依赖边缘自治域本地持久化策略实现）；恢复后 <= 2 个心跳周期内开始同步；同步完成后云端数据完整性 100% | 自动化 |
| 9 | 边缘节点注册完成后系统 <= 10s 内自动触发配置下发 | 注册到配置生效 <= 10s；首条数据写入在配置生效后 <= 5s | 自动化 |

> **跨域依赖声明**：
> - 分区期间本地写入策略（写入缓冲区大小、持久化方式、淘汰策略）由[边缘自治域](../requirements/边缘自治.md)定义
> - 分区恢复后的数据同步触发条件、同步优先级、冲突解决策略由[边云协同域](../requirements/边云协同.md)定义
> - 本 Feature 负责：注册通道建立、心跳保活、拓扑状态维护、分区/恢复事件通知

## Feature: 响应容器编排平台的生命周期管理信号

### Feature 价值论证

**用户痛点**：K8s 环境中节点被调度重启或驱逐时，未正确响应 SIGTERM 导致写入缓冲区数据丢失、WAL 未刷盘。

**现有方案不足**：依赖默认信号处理或简单 preStop hook，无法感知当前写入负载状态；缺乏 startupProbe 与 livenessProbe 的区分处理；WAL 回放超时场景无明确策略；K8s 重启循环缺乏防护。

**本 Feature 价值**：数据库进程内建生命周期感知能力，根据当前负载动态调整关闭流程；区分启动探测与存活探测；定义 WAL 回放超时策略；防止重启循环耗尽集群资源。

**成功指标**：
- 优雅关闭期间零数据丢失
- 健康探针响应延迟 P99 < 200ms
- 重启循环检测：连续 3 次启动失败后在 5 分钟内不再自动重启

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 容器编排平台发送 SIGTERM，节点在写入负载下执行优雅关闭 | 关闭耗时 <= terminationGracePeriodSeconds（默认 60s）且 WAL 完整性校验通过 | 自动化 |
| 2 | 容器编排平台周期性调用 liveness 探针 | 探针响应延迟 P99 < 200ms | 自动化 |
| 3 | 容器编排平台调用 startupProbe，节点完成数据恢复后才标记 ready | 未完成 WAL 回放期间返回 not-ready；startupProbe 超时阈值默认 60s，可配置；回放完成后 <= 5s 转为 ready | 自动化 |
| 4 | SIGTERM 后节点未在 grace period 内完成关闭被 SIGKILL | 重启后 WAL 回放恢复数据完整性 | 自动化 |
| 5 | 节点内部 OOM 或死锁导致 liveness 探针超时 | 探针连续失败达 failureThreshold 后节点被重启，重启后 <= 60s 恢复 | 自动化 |
| 6 | 容器编排平台调用 readiness 探针，节点在再平衡或追赶数据期间不接收新流量 | 再平衡/追赶期间 readiness 返回 not-ready，完成后 <= 5s 转为 ready | 自动化 |
| 7 | WAL 回放超时（数据量过大或磁盘故障） | 回放超过 startupProbe 超时阈值后节点被标记为 failed，触发副本切换（RTO 30s 适用于节点完全不可用场景，回放超时导致的启动延迟由 startupProbe 超时阈值覆盖）；重启后从上次成功位置继续回放 | 自动化 |
| 8 | 节点因配置错误连续启动失败，K8s 进入重启循环 | 连续 3 次 startupProbe 失败后，系统在 5 分钟内阻止自动重启并上报告警 | 自动化 |
| 9 | 容器编排平台触发节点滚动升级，旧实例优雅关闭新实例启动 | terminationGracePeriodSeconds 默认 60s，升级全程写入可用性 >= 99.9% | 自动化 |
| 10 | 应用开发者客户端在副本切换期间写入数据，SDK 自动重定向到新的主副本 | 切换期间未确认写入返回可重试错误码（含指数退避建议：初始 100ms，最大 5s，因子 2）；切换完成后 SDK 自动重连新主副本，重连耗时 <= 5s | 自动化 |

> **隐性假设与约束**（特性级设计时作为架构约束输入）：
> - 协调器串行调度副本切换与升级流程，协调器自身需具备故障转移能力（特性级设计时明确），否则降级为人工确认模式
> - 边缘节点心跳周期（10s）独立于云端副本心跳（2s），原因：边缘网络不稳定、ARM 资源受限，需降低心跳开销
> - 再平衡流量纳入 Feature 6 配额体系，受统一 IO 配额约束；再平衡 IO 控制由配额系统实现，Feature 2 不独立实现
> - 配置下发时效目标（<= 30s）隐含前提：网络正常时；边缘节点间歇网络场景下时效可能延长
> - 边缘节点本地写入缓冲区容量由边缘自治域定义，集群管理域承诺"分区恢复后同步触发"，不承诺无限期离线数据不丢失

## Feature: 提供滚动升级与版本灰度发布

### Feature 价值论证

**用户痛点**：2-5 人运维团队管理数十到数百个节点，版本升级如果需要停机窗口，在 7x24 场景下几乎找不到安全维护时间。

**现有方案不足**：手动逐节点升级耗时长；全量升级一旦新版本有缺陷则影响面 100%；升级失败回滚策略不明确。

**本 Feature 价值**：支持节点逐一升级并自动迁移读写流量，结合灰度比例控制将缺陷爆炸半径限制在可控范围；定义明确的升级回滚策略。

**成功指标**：
- 滚动升级期间写入成功率 >= 99.9%
- 单节点升级耗时 <= 120s
- 升级回滚耗时 <= 180s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 运维工程师触发 3 节点集群滚动升级 | 升级全程写入成功率 >= 99.9%；单节点升级耗时 <= 120s | 自动化 |
| 2 | 运维工程师配置灰度比例 20%，仅 1 个节点升级到新版本 | 灰度节点读流量 <= 配置比例（默认 20%，范围 5%-50%）；写流量完全路由到未升级节点，灰度节点不接收写入 | 自动化 |
| 3 | 灰度验证通过后扩大至 100% 完成全量升级 | 所有节点版本一致 | 自动化 |
| 4 | 升级过程中正在升级的节点启动失败，系统自动回滚 | 故障节点 <= 180s 内回滚到旧版本 | 自动化 |
| 5 | 新旧版本存在数据格式不兼容，升级预检阻止继续 | 预检在停止旧版本前完成，输出不兼容项清单 | 自动化 |
| 6 | 升级进行到一半时另一个未升级节点意外宕机 | 升级流程自动暂停，待恢复或运维确认后继续 | 自动化 |
| 7 | 运维工程师在灰度阶段发现异常，手动触发全量回滚 | 回滚指令 <= 30s 内生效，所有已升级节点恢复旧版本，无数据丢失 | 自动化 |
| 8 | 升级回滚过程中节点再次启动失败 | 系统标记节点为 failed 并停止自动重试，等待人工介入 | 自动化 |

## Feature: 控制节点级资源配额与过载保护

### Feature 价值论证

**用户痛点**：边缘节点资源有限，单个异常数据源的流量风暴可耗尽节点资源导致所有写入失败。

**现有方案不足**：OS 级 cgroup 只能做粗粒度隔离，无法感知数据库内部业务级指标，触发 OOM Kill 时已来不及降级。

**本 Feature 价值**：在数据库层面实现业务感知的资源配额管理，接近阈值时主动触发背压。

**成功指标**：
- 过载保护触发后核心写入链路可用性 >= 99%
- 从检测到过载到背压生效 <= 5s

|---|------|---------|---------|

| 2 | 运维工程师配置磁盘配额，达到 90% 时拒绝新建时间线 | 新建时间线请求返回错误码，已有时间线写入不受影响 | 自动化 |
| 3 | 运维工程师配置并发查询上限 | 超出配额的查询 <= 1s 内收到 429 | 自动化 |

| 5 | 配额值设置为不合理的极低值 | 节点启动时校验并拒绝，使用上一次有效配置 | 自动化 |
| 6 | 过载保护触发期间运维工程师动态调高配额 | 配额变更 <= 10s 内生效，背压解除 | 自动化 |
| 7 | 再平衡流量与前台写入同时触发内存配额上限 | 前台写入优先，再平衡流量自动降速，前台写入延迟增幅 <= 20% | 自动化 |
| 8 | 资源配额触发背压期间节点收到 SIGTERM | 优雅关闭优先于背压，关闭过程中数据完整性 100% | 自动化 |
