## Feature: TTL 数据保留与过期清理

### Feature 价值论证

**用户痛点**：边缘侧设备每秒产生数万条高频采样数据，数据量以 TB/月 速度增长。数据工程师需要手动编写清理脚本，且无法按 measurement/tag 维度差异化保留；同时部分行业（电力、轨交、医疗）法规要求最短保留期（如 6 个月/2 年），违规清理将面临监管处罚。

**现有方案不足**：手动清理脚本无法感知法定保留期限，存在违规风险；清理操作与在线查询争抢 IO；策略变更需修改脚本并重新部署；多节点环境下节点宕机会造成过期数据积压无人清理。

**本 Feature 价值**：提供声明式保留策略配置（measurement/tag 粒度），系统自动在低负载窗口执行过期清理；与安全合规域对接强制执行法定最短保留期限，杜绝违规清理。

**合规边界（反向冲突说明）**：当业务方配置的 TTL 短于安全合规域下发的法定最短保留期限时，**合规约束优先级高于成本/性能优化**——系统必须拒绝该策略配置，即使会造成额外存储成本。本 Feature 的"成本最优"目标在合规边界内才有效。

**成功指标**：
- 过期数据在 TTL 到期后 24 小时内完成清理（起算点：分区中最新一条数据的事件时间到达 TTL 阈值的时刻），逻辑清理率 >= 95%（应删除记录数 / 到期记录数），物理空间释放由存储层 compaction 异步完成，compaction 完成后 7 天内物理回收率 >= 90%
- 清理执行期间 P99 查询延迟增幅 <= 15%
- **合规违规清理事件 = 0**（任何短于法定保留期的策略必须在配置阶段被拦截，不得进入执行阶段）

| # | 场景 | 验收标准 | 可测试性 |

| 2 | 数据工程师为同一 measurement 下不同 tag 配置差异化保留周期（单集群策略数 <= 1000 条） | 各 tag 分别在各自 TTL 到期后 24h 内独立清理，无交叉影响 | 自动化 |
| 3 | 数据工程师在线修改保留策略，TTL 触发器在下一个清理周期按新策略执行 | 策略变更后首次触发即按新策略执行，无需重启，操作记入审计日志 | 自动化 |
| 4 | TTL 触发器执行清理期间，在线查询并发访问同一时间范围数据 | 清理期间 P99 查询延迟增幅 <= 15%，无查询错误 | 自动化 |

| 9 | 因元数据损坏导致某分区的 TTL 元信息丢失 | 系统检测到元数据缺失后阻止清理并告警，不得按"无 TTL"或默认值清理任何数据 | 自动化 |
| 10 | 运维 SRE 在合规审计期间暂停 TTL 清理，审计结束后恢复执行 | 暂停期间零清理触发；恢复后 48h 内完成积压过期数据清理；暂停/恢复操作 100% 记入审计日志 | 自动化 |

## Feature: 定时降采样与连续聚合

### Feature 价值论证

**用户痛点**：数据工程师需要同时满足两个矛盾需求——近期数据保留秒级精度用于异常诊断，历史数据只需分钟/小时级趋势。若全量保留原始数据，存储成本每月增长 30%+。

**现有方案不足**：外挂 ETL 方案无法利用存储层分区信息做增量聚合，迟到数据无法修正已完成的聚合结果，聚合任务失败后缺乏幂等重试，规则升级后历史聚合数据格式不兼容导致历史数据查询失败。

**本 Feature 价值**：内置连续聚合引擎，声明降采样策略后系统自动增量聚合，支持迟到数据修正与规则平滑升级，全程不阻塞在线写入。

**与 TTL 的依赖优先级**：当某时间窗口同时满足"降采样未完成"与"原始数据 TTL 到期"时，**降采样优先于 TTL 清理**——TTL 触发器在清理前必须查询该时间范围是否存在未完成的降采样任务，若存在则推迟该范围清理直至降采样完成或显式告警。该编排避免出现 Scenario #8 描述的源数据缺失情况。

**聚合数据 TTL 策略**：降采样产生的聚合数据（如 1min 精度、1h 精度）支持独立 TTL 配置，默认聚合数据保留周期 = 原始数据 TTL × 10（最低 1 年）。聚合数据 TTL 到期后的清理遵循与原始数据 TTL 相同的合规约束（不得短于法定保留期限）。

**成功指标**：
- 降采样任务在触发周期结束后 10 分钟内完成聚合写入
- 降采样执行期间写入吞吐下降 <= 10%
- 规则升级后历史聚合数据可读率 = 100%（含按旧格式读取的兼容路径）
- **降采样后历史数据（>30 天）存储量相对于保留原始数据减少 >= 80%**（测量口径：含压缩后的实际磁盘占用，按相同时间范围、相同 measurement 对比）
- 聚合数据查询 P95 延迟 <= 同时间范围原始数据查询 P95 延迟的 50%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据工程师配置将 1 秒精度数据按 5 分钟窗口聚合为 avg/max/min 摘要 | 每个聚合周期结束后 10min 内完成写入；max/min 与全量重算结果**完全一致**（误差 = 0）；avg 相对误差 <= 1e-9（IEEE 754 双精度累加上限） | 自动化 |
| 2 | 数据工程师配置多级降采样链（1s → 1min → 1h），系统按依赖顺序逐级执行 | 各级按依赖顺序完成，端到端延迟 <= 30min，链路任一级失败时下游级别不启动 | 自动化 |
| 3 | 降采样触发器执行聚合期间，在线写入持续进行 | 聚合期间写入吞吐下降 <= 10%，P99 写入延迟增幅 <= 20% | 自动化 |
| 4 | 降采样任务执行期间用户查询正在被聚合的时间窗口原始数据 | 查询返回完整原始数据并标注"聚合进行中"状态，无并发冲突错误 | 自动化 |
| 5 | 降采样任务执行中途因节点故障中断，故障恢复后幂等重试 | 重试后聚合结果与正常执行**逐字节一致**（采用幂等 key 去重），无重复写入 | 自动化 |
| 6 | 迟到数据（默认 24h 迟到窗口内）到达时其所属时间窗口的降采样已完成，系统增量修正 | 修正后聚合值与全量重算结果一致（按指标类型遵守 #1 的误差阈值），修正延迟 <= 下一个聚合周期；超过 24h 窗口的迟到数据被拒绝并告警 | 自动化 |
| 7 | 数据工程师修改降采样策略（含运维操作审计），系统从策略生效时间点起按新策略执行 | 新策略在下一触发周期生效，历史聚合结果不变，操作记入审计日志 | 自动化 |
| 8 | 降采样源数据存在但其上游 TTL 即将到期（编排冲突场景） | TTL 触发器查询到该时间范围存在未完成降采样任务时推迟清理；若降采样持续失败超过 SLO，告警运维 SRE 而非清理源数据 | 自动化 |
| 9 | 降采样规则/聚合格式版本升级后，查询访问按旧版本格式落盘的历史聚合数据 | 历史聚合数据可读率 = 100%；新旧格式查询结果在指标定义未变情况下数值一致；版本回滚后旧版本仍可读取新格式数据（前向兼容窗口 = 1 个 minor 版本） | 自动化 |
| 10 | 集群发生网络分区，少数派节点上的降采样触发器与多数派失联 | 少数派暂停聚合写入避免脑裂，分区恢复后通过幂等 key 安全续跑，无重复聚合 | 自动化 |
