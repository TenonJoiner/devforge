### 评审摘要

R1 修正循环显著提升了 Scenario 的量化程度与故障覆盖完整性，但在两个**模板硬性约束**项上仍未达标——`兼容 InfluxDB Line Protocol` 与 `兼容 Prometheus remote_write` 的故障覆盖矩阵单行 ✅ 数量不足 5 个；其中 Prometheus 行的"升级回滚 ✅ #4"分类错置（#4 实为协议版本兼容性，不属于系统升级回滚）。此外，时序库的两个经典痛点（**tag 高基数**、**乱序/迟到样本写入**）在本域未被任何 Scenario 覆盖，与本域作为时序数据库写入核心路径的定位不符。安全审计 Actor、监控采集系统 Actor 在写入路径上的 Scenario 仍存在显性缺位。

### 问题清单

#### CRITICAL

**C1. 故障覆盖矩阵硬性约束违反 [维度：场景覆盖质疑]**
- **位置**：`故障模式覆盖矩阵` 表，`兼容 InfluxDB Line Protocol` 行
- **问题**：该行只有"正常路径 #1,#2,#3"+"数据损坏 #4,#5,#6"两个维度有 ✅，节点故障/网络分区/并发冲突/资源耗尽/运维操作/升级回滚 6 个维度均为 "—"。模板硬性要求"每个 Feature 行至少 5 个 ✅（正常路径 + 4 种故障模式）"，本行仅 2 类维度被覆盖，未达模板基线。
- **理由**：Line Protocol 同样会面对资源耗尽（HTTP 连接池打满、解析时 CPU 饱和）、运维操作（精度配置变更）、并发提交场景，这些都是真实工程问题，不能以"由其他 Feature 覆盖"为由跳过——本 Feature 是兼容协议的独立验收单元。
- **修正建议**：补充至少 3 个新场景使覆盖维度达到 5 类，例如：
  - 资源耗尽：单连接持续提交导致 HTTP keep-alive 池耗尽时的限流/排队行为
  - 并发冲突：多个采集端同时写入同 measurement 同 timestamp 的处理策略
  - 运维操作：通过管理 API 切换默认时间戳精度（ns/us/ms/s）后，新写入按新精度处理且历史数据不受影响

#### HIGH

**H1. Prometheus remote_write 矩阵分类错置且数量不达标 [维度：场景覆盖质疑]**
- **位置**：`故障模式覆盖矩阵` 表，`兼容 Prometheus remote_write` 行的"升级回滚 ✅ #4"
- **问题**：#4 描述为"使用不支持的 remote_write 协议版本发送数据"，属于协议版本兼容/契约校验，不是"系统升级回滚"语义。剔除分类错置后该行实际只有 5 个 ✅（其中 3 个集中于"正常路径"），本质上仅覆盖正常路径 + 数据损坏 2 个维度。
- **理由**：Prometheus 兼容场景中真正的工程风险是 OOO（out-of-order）samples、staleness markers、native histograms 这些 Prometheus 2.x 引入的扩展，而非系统升级回滚。
- **修正建议**：① 将 #4 从"升级回滚"列移除，归入"数据损坏"或新增"契约校验"列归类；② 补充 OOO samples 与 staleness marker 处理场景（详见 H3）。

**H2. 时序库核心痛点 tag 高基数未覆盖 [维度：用户假设质疑]**
- **位置**：Feature `时序数据模型与 Schema 管理` Scenario 清单
- **问题**：本 Feature 价值论证强调"防止写入污染"和"schema 演进"，但完全没有 Scenario 覆盖 tag 基数爆炸（cardinality explosion）这一时序数据库经典痛点——例如 IoT 设备将随机 UUID 或时间戳写入 tag 字段，导致索引规模线性失控。InfluxDB/Prometheus 社区将此列为生产事故 TOP3。
- **理由**：用户假设"schema 类型约束"已足够，但类型正确不等于基数受控。无该场景，验收阶段无法判断系统是否会被恶意/疏忽客户端拖垮。
- **修正建议**：补充 1 个 Scenario，例如：
  > IoT 设备群在某 tag 维度的去重值数量超过配置上限（如单 measurement tag 基数 > 100k）时，系统拒绝新基数注入或触发降级（限流/告警）｜验收：基数超限时新增 series 拒绝率 100%，已有正常 series 写入不受影响。

**H3. Prometheus 兼容关键扩展场景遗漏 [维度：场景覆盖质疑]**
- **位置**：Feature `兼容 Prometheus remote_write 协议` Scenario 清单
- **问题**：兼容性成功指标声明"官方兼容性测试套件通过率 >= 95%"，但 Scenario 完全未涉及 OOO samples（Prometheus 2.39+）、staleness markers、exemplars 等会显著影响兼容率得分的扩展项。当前 Scenario 仅覆盖标准 TimeSeries + metadata + 错误路径。
- **理由**：兼容率指标无法仅靠当前 6 个 Scenario 验证。验收阶段会出现"代码通过 6 条 Scenario 但兼容套件得分仅 70%"的脱节。
- **修正建议**：补充 1-2 个 Scenario：
  - 数据采集代理推送包含乱序 timestamp 的 sample（旧于已写入的最大 timestamp），系统按配置的 OOO 容忍窗口接受或拒绝｜验收：窗口内乱序写入成功率 100%，超窗写入按策略明确拒绝。
  - 数据采集代理推送 staleness marker（特殊 NaN value），系统正确标记 series 终态｜验收：后续查询返回 staleness 状态而非空值。

**H4. Line Protocol 乱序/迟到时间戳场景遗漏 [维度：场景覆盖质疑]**
- **位置**：Feature `兼容 InfluxDB Line Protocol 写入` Scenario 清单
- **问题**：Line Protocol 实际部署中常见 Telegraf 重连后批量补传历史数据（写入时间戳早于当前最大 timestamp）。该场景在所有协议接入 Feature 中均缺失，但 Line Protocol 是主要触发协议（兼容 InfluxDB 的客户依赖此能力）。
- **理由**：现有 Feature 1 的 #4（半包/中断）和当前 Feature 的 #4（错误行）只覆盖"语法错误"，未覆盖"语法正确但时间戳乱序"。
- **修正建议**：补充 1 个 Scenario：
  > 数据采集代理在网络恢复后批量补传 30 分钟前的历史数据，与当前实时数据混合写入｜验收：迟到数据按原 timestamp 落盘，可被时间范围查询到，且不影响实时数据写入延迟（P99 劣化 < 20%）。

#### MEDIUM

**M1. 隐性 Actor 遗漏：安全审计/写入鉴权 [维度：Actor 完整性]**
- **位置**：`涉及 Actors` 段及全部 Feature Scenario
- **问题**：本域涉及多协议外网入站写入路径，Scenario 中无任何"鉴权失败/未授权写入/审计日志"场景。即便鉴权机制由"安全合规"特性域定义，本域 Scenario 中至少应有钩子说明"未通过鉴权的写入请求被拒绝且不消耗写入配额"。
- **理由**：写入路径是攻击面最大的接入点，没有鉴权失败的验收标准会导致安全域与写入域职责真空。
- **修正建议**：在 `多协议统一写入接入` 或在文档头部备注"鉴权场景由安全合规域定义，本域仅保证未授权请求在协议解析前被拒绝"，并补充 1 个 Scenario：
  > 客户端使用无效凭证发起写入，系统在协议解析前完成鉴权拒绝｜验收：拒绝响应延迟 < 10ms，不进入 WAL 与背压队列，鉴权失败计数可观测。

**M2. 监控采集系统 Actor 写入指标场景缺位 [维度：Actor 完整性]**
- **位置**：`涉及 Actors` 段；非功能需求"可观测性"项
- **问题**：product-spec.md 第 188 行将"监控采集系统"列为 Actor，关注 /metrics 拉取写入相关指标（QPS/延迟/积压），但本域 Actor 列表与 Scenario 中均未出现该 Actor。非功能需求的"/metrics 端点验证"是单点检查，缺少端到端 Scenario。
- **修正建议**：在 Actors 段补充"监控采集系统（协作 Actor）"，并在 `写入流量治理` 或独立处补 1 个 Scenario：
  > 监控采集系统按 15s 周期拉取 /metrics，获取 ingest_qps/ingest_latency_p99/wal_backlog 等写入指标｜验收：指标暴露完整性 100%，拉取自身延迟 < 100ms，对写入吞吐影响 < 1%。

**M3. Schema 自动推断验收标准模糊 [维度：场景覆盖质疑]**
- **位置**：Feature `时序数据模型与 Schema 管理` Scenario #2
- **问题**：验收标准"类型推断准确率 100%"未定义何为"准确"。无量化基准下不可独立判定通过/失败。
- **修正建议**：明确为：自动推断的类型与首条数据中各 field 的语法形式（整数→int64、浮点→float64、布尔→bool、引号字符串→string）一一对应；与对照规则表的偏差数 = 0。

#### LOW

**L1. "协议解析层内存占用"边界模糊 [维度：可测试性]**
- **位置**：Feature `多协议统一写入接入` Scenario #9
- **问题**："协议解析层内存占用 < 64MB" 中"协议解析层"非用户可观察边界。
- **修正建议**：改为以进程 RSS 增量度量，例如"在 512MB 总内存约束下，启用 MQTT+CoAP 双协议时进程 RSS < 256MB"。
