## 业务上下文

### 行业背景

- **所属行业领域**：企业级 IT 基础设施 / 分布式存储
- **当前行业核心趋势与挑战**：
  - 传统行业（金融、制造、医疗、政府）数字化转型加速，对存储性能要求持续提升
  - 现有开源方案（Ceph）在随机 IOPS、长尾延迟、扩容线性度方面存在明显瓶颈
  - 商业存储（OceanStor 等）成本高昂且存在 vendor lock-in 风险
  - NVMe-oF 协议逐步成熟，成为高性能块存储的主流接入方式
- **技术演进方向对本品的影响**：NVMe-oF 生态成熟为新品类入场提供窗口期；客户对"开源替代 + 性能提升"的需求明确

### 竞争格局

| 竞品/方案 | 定位 | 核心优势 | 主要短板 | 与本品关系 |
|-----------|------|----------|----------|------------|
| Ceph | 开源统一存储 | 生态成熟、免费、功能全面 | 随机 IOPS 低、长尾延迟不可控、扩容性能不线性 | **直接替代目标** |
| OceanStor | 商业企业存储 | 性能稳定、服务完善、品牌信任 | 成本高、vendor lock-in、灵活性差 | **竞品对标** |
| 国内中小存储厂商 | 本土化商业存储 | 本地化服务、价格适中 | 技术深度参差、生态不成熟 | **竞品对标** |

### 商业模式假设

- **目标客户群体**：传统行业政企客户（金融、制造、医疗、政府），私有部署场景
- **价值主张**：在保持 Ceph 级成本的同时，提供比 Ceph 提升一个数量级的块存储性能
- **收入模式假设**：待补充（软件许可 / 订阅 / 服务支持）
- **关键成功因素**：
  1. 性能目标达成（IOPS、吞吐、长尾延迟三维度）
  2. 与 Ceph 持平的容量成本 + IOPS 成本降低一个数量级
  3. 协议兼容（NVMe-oF + iSCSI + vhost）降低迁移成本

## 约束与假设

### 技术约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 集群规模 | 单集群 ~10 节点，几十 TB 量级 | 架构设计、一致性协议选择、元数据管理 |
| 协议支持 | NVMe-oF 为主，兼容 iSCSI、标准块设备接口、vhost | 数据面设计、客户端 SDK、虚拟化集成 |
| 性能基线 | 比 Ceph 提升一个数量级（IOPS、吞吐、P99 延迟） | 存储引擎设计、网络模型、缓存策略 |
| 成本基线 | 容量成本持平 Ceph，IOPS 成本降低一个数量级 | 硬件选型、数据冗余策略、资源调度 |

### 法规与合规约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 行业合规 | 不做特定行业合规认证（等保、HIPAA 等） | 产品边界明确，减少初期认证成本 |

### 用户能力约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 运维水平 | 目标用户有存储管理经验，但期望比 Ceph 更简化的运维 | 管理界面设计、自动化运维、故障诊断 |

## 使用场景与上下文

**产品定位**：面向传统行业政企客户的私有部署分布式块存储，以 NVMe-oF 为主协议，目标在 IOPS、吞吐、长尾延迟三维度比 Ceph 提升一个数量级。

**部署环境**：
- 部署模式：私有部署（on-premise）
- 典型规模：单集群 10 节点左右，总容量几十 TB
- 硬件假设：标准 x86 服务器 + NVMe SSD（具体配置待架构阶段确认）

**典型使用流程**：待第 2/3 阶段补充 Scenario 后完善

## 全局非功能需求

| 类型 | 目标值 | 验收方法 |
|------|--------|----------|
| 性能 | 比 Ceph 提升一个数量级（IOPS、顺序吞吐、P99 延迟） | 同硬件条件下对比基准测试 |
| 可用性 | 待架构阶段明确（如 99.9% / 99.99%） | 故障注入测试 |
| 一致性 | 待架构阶段明确（强一致 / 最终一致及适用场景） | 一致性验证测试 |
| 安全性 | 传输加密、访问控制、操作审计 | 安全扫描 + 审计日志验证 |
| 可观测性 | 核心指标全覆盖、P0 故障 1 分钟内告警 | 指标覆盖检查 + 故障演练 |
| 扩展性 | 单集群线性扩容至目标规模（10 节点，几十 TB） | 扩容性能回归测试 |
| 兼容性 | NVMe-oF 标准协议 + iSCSI + vhost | 协议兼容性测试 |

### 可观测性目标

| 指标类型 | 目标 | 具体说明 |
|----------|------|----------|
| 指标覆盖 | 核心子系统全部暴露关键指标 | 存储引擎、网络、元数据、协议接入层 |
| 日志规范 | 结构化日志、统一 Trace ID | 支持端到端请求追踪 |
| 告警时效 | P0 故障 1 分钟内触达 | 磁盘故障、节点宕机、网络分区 |
| 诊断深度 | 支持从请求到存储的端到端追踪 | 延迟分解、热点定位 |

### 安全合规目标

| 目标项 | 要求 | 验收方式 |
|--------|------|----------|
| 身份认证 | 支持多租户身份认证 | 功能测试 |
| 传输加密 | NVMe-oF/TCP 支持 TLS | 抓包验证 |
| 访问审计 | 关键操作可审计 | 审计日志检查 |
| 数据隔离 | 多租户间数据隔离 | 隔离性测试 |
| 合规认证 | 不做特定行业认证（等保、HIPAA） | 产品边界声明 |
