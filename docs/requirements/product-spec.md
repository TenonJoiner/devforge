# 产品需求总纲

## 业务上下文

### 行业背景

- 所属行业领域：时序数据库（Time-Series Database），聚焦工业物联网与基础设施监控
- 当前行业核心趋势与挑战：边缘计算兴起推动数据处理下沉至设备侧；传感器密度指数增长导致写入规模持续膨胀；云边协同成为刚需但网络不可靠性是核心挑战
- 技术演进方向对本品的影响：列式存储+压缩技术成熟降低存储成本；Prometheus 生态成为事实标准要求兼容；边缘侧资源受限要求极致轻量化

### 竞争格局

| 竞品/方案 | 定位 | 核心优势 | 主要短板 | 与本品关系 |
|-----------|------|----------|----------|------------|
| InfluxDB 3.0 | 云原生通用时序数据库 | 对象存储分离架构、SQL 支持、成熟生态 | 边缘能力弱、开源版功能受限 | 直接竞品，需在边云协同上差异化 |
| TDengine | 工业 IoT 时序数据库 | 超级表模型、内置流计算、边云一体 | 生态封闭、Prometheus 兼容弱 | 直接竞品，需在生态兼容性上差异化 |
| QuestDB | 高性能分析型时序数据库 | 极致查询性能、PostgreSQL 兼容 | 无原生集群、边缘能力缺失 | 间接竞品，可借鉴查询性能优化 |
| VictoriaMetrics | 大规模监控时序存储 | 极致压缩、成本效率、Prometheus 全兼容 | 非 IoT 定位、无边缘方案 | 间接竞品，需在成本效率上对标 |

### 商业模式假设

- **目标客户群体**：拥有大规模传感器网络的工业企业（制造、能源、交通）、需要边云协同监控的 SRE 团队
- **价值主张**：从边缘到云端的一体化时序数据管理，7×24 高写入不丢数据，低成本长期存储
- **收入模式假设**：开源核心 + 企业版增值（多租户、高级安全、商业支持）
- **关键成功因素**：写入性能领先、边云同步可靠、Prometheus 生态兼容、运维复杂度低

## 约束与假设

### 技术约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 最终一致性 | 不做跨地域强一致复制，仅支持最终一致性 | 边云同步、多副本读取 |
| 无完整 ACID | 不提供完整事务支持 | 写入模型、并发控制 |
| 边缘资源受限 | 边缘节点 CPU/内存/存储有限（ARM/x86 低配） | 边缘组件设计、内存管理 |

### 法规与合规约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 数据保留合规 | 工业数据可能有法定保留期限要求 | TTL 策略、数据删除 |
| 传输加密 | 边云通信必须支持 TLS | 网络协议、性能开销 |

### 用户能力约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 运维团队规模小 | 目标用户运维团队通常 2-5 人 | 自动化运维、自愈能力 |
| SQL 为通用技能 | 分析师熟悉 SQL 但不熟悉专有查询语言 | 查询语言设计 |

## 使用场景与上下文

**产品定位**：边云结合的时序数据库，面向海量传感数据 7×24 高写入场景，提供从边缘到云端的一体化数据管理。

**部署环境**：
- 边缘侧：工厂/站点本地服务器或工控机，资源受限，网络可能间歇性中断
- 云端侧：中等规模集群（3-10 节点），TB 级数据存储，承载聚合查询和长期归档

**典型使用流程**：
1. **数据写入**：传感设备通过 MQTT/行协议将测量数据写入边缘节点，边缘节点本地存储并异步同步至云端
2. **数据存储与管理**：云端集群接收同步数据，执行降采样和 TTL 清理，分层存储（热/温/冷）
3. **数据消费**：分析师通过 SQL 查询云端数据，Grafana 拉取实时仪表盘，告警引擎评估阈值触发通知

## Actors

### Actor 严格定义与判定决策树

**Actor = 产品黑盒视角下，与运行中的系统发生持续、直接交互的外部实体。** 类型限定为 3 类：

| 类型 | 判定条件 | 本产品示例 |
|------|---------|-----------|
| **人类** | 物理存在的人 | 数据工程师、数据分析师、平台运维工程师 |
| **外部系统** | 系统边界外、独立运行的另一个系统/设备 | IoT 设备群、Grafana、Kubernetes |
| **时间触发源** | 时间事件 + 驱动了一个有用户价值的独立用例 | 定时降采样、TTL 过期清理 |

---

### Actor: 数据工程师

**类型**：人类
**职责**：设计数据模型（measurement/tag/field schema）、编写数据写入管道、管理数据生命周期策略
**交互方式**：通过管理面配置 schema、设置写入策略（批量大小/重试/背压）、定义 retention policy 和降采样规则
**关注点**：写入吞吐量、写入延迟 P99、数据完整性（丢点率）、schema 演进兼容性

### Actor: 数据分析师

**类型**：人类
**职责**：对已入库的时序数据执行查询分析，发现趋势、异常和业务洞察
**交互方式**：通过 SQL/时序查询语言执行聚合/窗口函数/降采样查询，配置告警规则阈值，创建仪表盘定义
**关注点**：查询响应时间（P95 < 2s）、查询表达力（聚合/窗口操作）、跨时间范围查询性能稳定性、数据新鲜度

### Actor: 平台运维工程师

**类型**：人类
**职责**：负责时序数据库集群（含边缘节点和云端节点）的部署、扩缩容、故障恢复、日常巡检
**交互方式**：执行集群部署/升级、配置节点拓扑、设置告警阈值、执行故障转移、调整资源配额
**关注点**：系统可用性（SLA）、故障恢复时间（RTO）、存储容量规划、边缘节点离线容忍时长、升级零停机

### Actor: 应用开发者

**类型**：人类
**职责**：基于时序数据库的 SDK/API 开发上层应用（IoT 平台、监控系统、能源管理等）
**交互方式**：调用写入/查询 API、配置连接池参数、使用 SDK 集成到应用代码、订阅数据变更通知
**关注点**：API 易用性和稳定性、SDK 语言覆盖、错误码语义清晰度、连接可靠性（断线重连）、API 版本兼容性

### Actor: 安全审计员

**类型**：人类
**职责**：审计数据访问权限、检查合规性（数据保留/删除策略是否符合法规）、审查操作日志
**交互方式**：查询审计日志、检查权限配置、执行合规扫描、设置数据分类标签
**关注点**：操作可追溯性（审计日志完整性）、最小权限原则执行情况、数据保留策略合规性、敏感数据访问控制

---

### Actor: IoT 设备群

**类型**：外部系统
**职责**：在边缘侧持续产生时序测量数据并直接写入系统
**交互方式**：通过 MQTT/CoAP/私有二进制协议，以推送模式向边缘节点批量写入时间戳+测量值；单设备 1Hz-1kHz，长连接保持
**关注点**：写入确认延迟（P99 < 100ms）、断网续传能力、协议开销最小化（受限带宽/算力）

### Actor: 应用服务集群

**类型**：外部系统
**职责**：云端业务系统产生的运行指标（metrics）、审计日志、业务事件写入
**交互方式**：通过 HTTP/gRPC 短请求，以行协议或 Protobuf 格式批量写入；写入频率 10s-60s 一批
**关注点**：写入吞吐（万级 TPS）、批量写入原子性、背压信号（过载时 429 退避）

### Actor: 消息队列/流平台

**类型**：外部系统
**职责**：作为数据管道上游，将已汇聚/预处理的时序数据流式推送到本系统
**交互方式**：本系统作为 consumer 从 Kafka topic 拉取数据，或由流平台通过 Sink Connector 推送；持续消费 MB/s-GB/s
**关注点**：消费吞吐与 lag 控制（consumer lag < 10s）、exactly-once 语义、分区并行度匹配

### Actor: 数据采集代理

**类型**：外部系统
**职责**：部署在目标主机/容器旁，采集基础设施指标并定期批量写入本系统
**交互方式**：通过 HTTP remote_write 协议或 InfluxDB 行协议，定时（10s-30s）推送采集批次
**关注点**：写入兼容性（Prometheus remote_write / InfluxDB line protocol）、采集端资源开销最小化、写入失败时本地缓冲与重试

### Actor: 可视化仪表盘系统

**类型**：外部系统
**职责**：周期性拉取时序数据并渲染为图表/仪表盘供人类查看
**交互方式**：通过 HTTP/REST 或 PromQL 兼容 API 发起范围查询，按时间窗口+标签过滤拉取聚合数据；同步请求-响应
**关注点**：查询延迟 P95（< 2s）、数据点完整性、并发查询吞吐（多面板同时刷新）

### Actor: 告警规则引擎

**类型**：外部系统
**职责**：持续评估时序数据是否触发告警阈值，产生告警事件
**交互方式**：以固定间隔（秒级~分钟级）通过查询 API 拉取最近窗口的聚合值；交互频率高、单次数据量小
**关注点**：评估延迟（数据写入到告警触发端到端 < 30s）、数据新鲜度（不读过期缓存）、查询稳定性（不因超时导致漏报）

### Actor: 流式订阅消费者

**类型**：外部系统
**职责**：实时订阅特定时序数据流，接收推送的新数据点用于下游实时处理
**交互方式**：通过长连接（WebSocket/gRPC streaming）建立订阅通道，系统主动推送符合条件的新写入数据点
**关注点**：推送延迟（写入到推送 < 1s）、消息有序性（同一时间线内有序）、背压处理、断线重连后数据补推

### Actor: 批量数据消费系统

**类型**：外部系统
**职责**：定期从时序数据库中批量抽取历史数据，加载到数据仓库/数据湖/ML 平台
**交互方式**：通过批量扫描 API 或数据导出接口（CSV/Parquet），一次性读取大时间范围（天~月级）数据；GB~TB 级
**关注点**：扫描吞吐量（GB/s）、对在线查询的影响隔离、数据一致性快照、断点续传能力

### Actor: 容器编排平台

**类型**：外部系统
**职责**：管理数据库实例的生命周期调度——创建、销毁、重启、弹性伸缩
**交互方式**：向实例发送健康探针（liveness/readiness）、下发启停信号、调度 Pod 到节点
**关注点**：实例启动时间、健康检查响应延迟、资源请求与实际使用匹配度、优雅关闭时间

### Actor: 监控采集系统

**类型**：外部系统
**职责**：持续采集数据库自身暴露的运行指标，供系统健康监控和告警使用
**交互方式**：按固定间隔（10-30s）拉取 /metrics 端点，获取结构化指标数据（写入 QPS、查询延迟分布、WAL 积压量等）
**关注点**：指标暴露完整性和时效性、采集端点可用性、指标命名规范一致性

### Actor: 备份与灾备系统

**类型**：外部系统
**职责**：执行数据库的定期备份、跨区域灾备复制、灾难恢复时的数据还原
**交互方式**：调用快照/备份 API、请求增量数据导出、触发恢复流程
**关注点**：备份 RPO 达标率、备份窗口对写入性能的影响、恢复 RTO、备份数据完整性校验

### Actor: 外部密钥管理系统（KMS）

**类型**：外部系统
**职责**：为系统提供密钥的创建、轮换、撤销和审计能力，是静态数据加密和传输加密的密钥信任锚
**交互方式**：系统通过标准 KMS API（如 AWS KMS / HashiCorp Vault / Azure Key Vault）请求加解密操作；系统不存储主密钥，仅缓存数据密钥（DEK）并在 TTL 过期后重新请求
**关注点**：密钥可用性（KMS 不可用时加密写入是否阻断）、密钥轮换对在线业务的影响、API 调用延迟和速率限制、密钥访问审计日志完整性

---

### Actor: 定时降采样触发器

**类型**：时间触发源
**职责**：按预设时间策略触发历史数据的降采样，将高精度原始数据聚合为低精度摘要数据
**交互方式**：时间事件（每小时/每天）触发降采样任务，读取指定时间范围原始数据，写回聚合结果
**关注点**：降采样及时性、聚合正确性、对在线写入的影响（不阻塞热路径）、策略可配置性

### Actor: TTL 数据过期清理触发器

**类型**：时间触发源
**职责**：按数据保留策略（TTL）定期清理过期的时序数据，释放存储空间
**交互方式**：时间事件（每天）触发过期数据扫描和删除；对外暴露 TTL 策略配置和清理执行报告
**关注点**：清理及时性、清理对在线服务的影响（不导致查询延迟抖动）、空间回收效率、策略粒度

## Feature 总览

> 以下 Feature 按特性域组织。每个 Feature 包含一句话价值摘要（做什么 + 为什么）。详细的价值论证（痛点 → 不足 → 价值 → 成功指标）留给各特性域文档展开。
>
> P0 = 关键成功因素直接映射 + 核心数据面闭环 + 法规硬约束，缺一则产品不可发布。P1 = 重要增强，MVP 后首个迭代应覆盖。P2 = 远期规划。社区贡献 = 不计入核心排期。

> 标注"企业版"的 Feature 属于企业版增值能力，开源版不提供。

| 术语 | 定义 | 英文全称/别名 | 备注 |
|------|------|---------------|------|
| WAL | 预写日志，在数据写入磁盘前先记录到日志，确保崩溃后不丢失已确认数据 | Write-Ahead Log | 时序数据库持久化的基础机制 |
| TTL | 数据存活时间，超过该时间的数据将被自动清理 | Time-To-Live | 用于控制存储成本和满足合规要求 |
| 降采样 | 将高精度原始数据按时间窗口聚合为低精度摘要数据的过程 | Downsampling | 降低长期存储成本，保留趋势分析能力 |
| 时间线 | 由唯一标签组合标识的一组按时序排列的数据点 | Time Series | 时序数据库的核心数据单元 |
| 边云协同 | 边缘节点与云端集群之间的数据同步和协同计算架构 | Edge-Cloud Collaboration | 本产品的核心架构特征 |
| 最终一致性 | 数据写入后，在所有副本上最终达到一致状态，不保证实时一致 | Eventual Consistency | 产品明确的技术约束 |
| 热/温/冷存储 | 按访问频率将数据存放在不同性能和成本的存储介质上 | Hot/Warm/Cold Tiering | 用于平衡查询性能和存储成本 |
| 背压 | 当系统处理能力不足时，向上游发送反压信号使其降低发送速率 | Backpressure | 防止系统因过载而崩溃 |
| 连续聚合 | 对实时流入的数据按时间窗口持续执行聚合计算 | Continuous Aggregation | 与批量降采样相对，用于实时场景 |
| 物化视图 | 预计算并持久化存储的查询结果，用于加速重复查询 | Materialized View | 连续聚合的输出形式 |
