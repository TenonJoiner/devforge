## Feature: 提供版本化 REST API 并规范错误码体系

### Feature 价值论证

**用户痛点**：
应用开发者在集成时序数据库时，常面临 API 变更导致代码中断的问题。REST API 缺乏明确的版本标识，服务端升级后客户端行为不可预期；错误响应格式不统一，开发者无法程序化解析错误原因，只能依赖模糊的状态码猜测问题，调试成本极高。

**现有方案不足**：
- InfluxDB 2.x API 版本通过请求头传递，易被中间代理剥离，导致版本协商失败
- VictoriaMetrics 错误响应为纯文本，无结构化错误码，自动化重试逻辑难以编写
- 多数竞品错误码与 HTTP 状态码简单映射，无法表达业务层语义（如"时间线不存在" vs "测量值类型不匹配"）

**本 Feature 价值**：
- 通过 URL 路径版本化（如 `/v1/write`）确保版本标识不可被代理剥离，客户端可明确锁定 API 版本
- 定义结构化错误响应体（含机器可读错误码、人类可读消息、请求追踪 ID），支持程序化错误处理
- 错误码分层设计：HTTP 状态码表达传输层语义，业务错误码表达数据层语义，使重试策略、降级逻辑可自动化决策

**跨域错误码继承关系**：
- API 层错误码面向客户端（SDK/CLI/直接调用），安全合规域错误码面向服务端内部审计和策略执行。两者存在映射关系：
  - `AUTHENTICATION_FAILED`（API 层，HTTP 401）← 映射自安全合规域 Feature 2（身份认证）的无效凭证拒绝事件。API 层仅暴露"认证失败"语义，不暴露安全合规域内部细节（如凭证过期 vs 伪造 vs 吊销的具体原因），具体拒绝原因记录在安全审计日志中
  - `AUTHORIZATION_DENIED`（API 层，HTTP 403）← 映射自安全合规域 Feature 3（RBAC）的权限拒绝事件。API 层不暴露"哪个角色缺少哪个权限"的内部信息
  - `TLS_HANDSHAKE_FAILED`（API 层，HTTP 403）← 映射自安全合规域 Feature 1（强制 TLS）的证书校验失败事件
- 映射规则：安全合规域错误码为"源"，API 层错误码为"面"；同一安全事件在不同暴露层级使用不同粒度描述，确保安全性与可调试性的平衡

**成功指标**：
- 首次集成调试时间 ≤ 30 分钟（从拿到 API 文档到成功写入第一条数据）
- API 版本升级导致客户端代码变更率 ≤ 5%（N-1 版本兼容期内）
- 错误响应可程序化解析率 = 100%（所有 4xx/5xx 响应均包含结构化错误码）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 应用开发者通过 `/v1/write` 路径向指定版本 API 写入时序数据 | 请求成功返回 HTTP 200，响应体包含写入确认点数 = 请求点数；URL 路径版本号格式为 `v{major}`，与语义版本的主版本号对应 | 自动化 |
| 2 | 应用开发者通过 `/v2/query` 路径执行跨版本查询 | 请求成功返回 HTTP 200，响应格式与 v2 文档一致 | 自动化 |
| 3 | 应用开发者使用不存在的 API 版本发起请求 | 返回 HTTP 404，错误码 `API_VERSION_NOT_FOUND`，包含支持的版本列表 | 自动化 |
| 4 | 应用开发者发送格式非法的请求体（如缺失必填字段） | 返回 HTTP 400，错误码 `INVALID_REQUEST_BODY`，指明具体缺失字段 | 自动化 |
| 5 | 应用开发者向不存在的 measurement 写入数据 | 返回 HTTP 404，错误码 `MEASUREMENT_NOT_FOUND`，提示 measurement 名称 | 自动化 |
| 6 | 应用开发者使用无效认证凭据调用 API | 返回 HTTP 401，错误码 `AUTHENTICATION_FAILED`，不暴露内部实现细节 | 自动化 |
| 7 | 服务端内部异常时 API 返回统一错误响应 | 返回 HTTP 500，错误码 `INTERNAL_ERROR`，包含唯一追踪 ID（request_id） | 自动化 |
| 8 | 应用开发者收到错误响应后通过追踪 ID 查询日志 | 日志系统可通过 request_id 检索到完整请求链路，定位时间 < 30s | 自动化 |

## Feature: 提供多语言原生 SDK

### Feature 价值论证

**用户痛点**：
应用开发者团队通常使用多种编程语言构建系统（Go 写微服务、Python 写数据分析、Java 写企业应用、Rust/C 写高性能采集端）。缺乏官方原生 SDK 时，开发者需自行封装 HTTP 客户端、处理序列化/反序列化、实现重试和连接管理，每个语言重复造轮子，且质量参差不齐。

**现有方案不足**：
- InfluxDB 官方 SDK 仅覆盖主流语言，C/Rust 社区 SDK 维护状态不稳定
- VictoriaMetrics 无官方 SDK，社区实现功能不完整，缺乏类型安全保证
- 多数 SDK 仅做薄封装，未针对时序数据特征优化（如批量聚合、行协议序列化）

**本 Feature 价值**：
- 为 Go/Java/Python/Rust/C 提供官方维护的原生 SDK，保证功能完整性和长期维护
- SDK 内置时序数据类型系统（measurement/tag/field/value/timestamp），编译期捕获类型错误
- 针对各语言生态 idiomatic 设计（Go 的 context 支持、Python 的 async/await、Java 的 CompletableFuture）

**成功指标**：
- 官方 SDK 覆盖语言数 ≥ 5（Go/Java/Python/Rust/C）
- 各语言 SDK 发布包下载/安装成功率 ≥ 99.5%（通过包管理器）
- SDK API 与 REST API 功能覆盖率 ≥ 95%（写入/查询/管理操作）
- 版本兼容矩阵准确率 = 100%（每个 SDK 版本标注的兼容 API 版本范围经 CI 自动验证）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 应用开发者使用 Go SDK 批量写入时序数据 | 单次调用写入 1000 点，SDK 返回写入确认，耗时 P99 < 200ms | 自动化 |
| 2 | 应用开发者使用 Python SDK 执行异步查询 | 查询结果以迭代器形式返回，内存占用不随结果集大小线性增长 | 自动化 |
| 3 | 应用开发者使用 Java SDK 构建带标签的时序数据点 | SDK 类型系统拒绝非法标签值（如空字符串标签键），编译期报错 | 自动化 |
| 4 | 应用开发者 SDK 版本与服务器 API 版本不匹配 | SDK 采用"请求前本地兼容矩阵预检 + 请求后服务端响应确认"双层检测：本地矩阵预检不通过时直接抛出 `VersionMismatchError`（零网络开销）；预检通过但服务端响应 HTTP 410/Deprecation 头时抛出带服务端详情的 `VersionMismatchError` | 自动化 |
| 5 | 应用开发者 SDK 序列化大批量数据时内存不足 | SDK 流式序列化，单批次内存占用上限 = 10MB，超出时自动分片 | 自动化 |
| 6 | 应用开发者使用 Rust SDK 在边缘设备上编译 | SDK 无 std 依赖可选（no_std 模式），编译产物体积 < 500KB | 自动化 |
| 7 | 应用开发者 SDK 请求超时后收到结构化异常 | 抛出 `RequestTimeoutError`，含超时时间、重试次数、最终错误原因 | 自动化 |
| 8 | 应用开发者 SDK 遭遇服务端限流（429）时自动退避 | SDK 按指数退避重试，最大重试次数 = 3，总等待时间 ≤ 30s | 自动化 |

## Feature: 提供 SDK 连接管理能力

### Feature 价值论证

**用户痛点**：
应用开发者在使用 SDK 与分布式时序数据库交互时，需自行管理 TCP 连接生命周期（创建、复用、健康检查、断线重连）。在高并发写入场景下，连接管理不当导致连接泄漏、端口耗尽、请求排队超时；网络闪断时无自动恢复机制，应用需人工介入重启。

**现有方案不足**：
- InfluxDB Go 客户端连接池配置复杂，缺乏智能负载均衡，单节点故障时全部请求失败
- VictoriaMetrics 无官方客户端，社区实现连接管理质量参差，无断线重连标准行为
- 多数 SDK 未针对时序写入的高并发、长连接特征优化（如连接预热、批量管道复用）

**本 Feature 价值**：
- SDK 内置连接池自动管理（创建上限、空闲超时、健康探活），开发者无需手动配置
- 断线自动重连：网络恢复后自动重建连接并恢复写入，应用层无感知
- 客户端负载均衡：多节点部署时 SDK 自动分发请求，单节点故障自动剔除并标记恢复探测

**成功指标**：
- 网络闪断 30s 内恢复后，写入成功率恢复至 100%，无需应用重启
- 单 SDK 实例物理 TCP 连接数上限 ≥ 100（用于多节点部署场景），在 1000 并发请求下连接复用率 ≥ 80%
- 单节点故障时请求失败率 < 1%（负载均衡自动切换）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 应用开发者初始化 SDK 时配置多节点地址 | SDK 自动建立到所有健康节点的连接池，连接数 = min(节点数×4, 上限) | 自动化 |
| 2 | 应用开发者高并发写入时连接自动复用 | 1000 并发写入请求使用 ≤ 50 个物理连接，连接复用率 ≥ 80% | 自动化 |
| 3 | 网络闪断 10s 后自动恢复，SDK 无感知续传（直连云端模式） | 断网期间写入请求进入本地缓冲队列，恢复后自动重发，数据不丢失；边缘部署模式下 SDK 不启用本地缓冲，依赖边缘节点持久化能力 | 自动化 |
| 4 | 服务端单节点宕机，SDK 自动切换至健康节点 | 服务端通过响应头或独立推送通道通知拓扑变更，SDK 在收到通知后 5s 内完成重连；SDK 主动探测作为兜底机制，探测周期与集群管理域心跳周期对齐（默认 2s）；故障节点请求在 3 个探测周期内被路由至其他节点，失败率 < 1% | 自动化 |
| 5 | 连接池达到上限时新请求行为可控 | 超出上限的请求排队等待，排队超时 = 10s，超时后返回 `POOL_EXHAUSTED` 错误 | 自动化 |
| 6 | 空闲连接超过超时时间后自动关闭 | 空闲连接在 60s 无活动后关闭，不触发服务端异常日志 | 自动化 |
| 7 | SDK 进程崩溃后服务端连接正确释放 | SDK 进程终止后，服务端在 30s 内检测到连接关闭，无连接泄漏 | 自动化 |
| 8 | 边缘网络高延迟（RTT > 500ms）下连接保持可用 | 连接心跳间隔自适应延长至 RTT×3，持续误判断线率 = 0（允许 3 个心跳周期内偶发丢包不触发断线） | 自动化 |

## Feature: API 版本兼容性承诺与废弃策略

### Feature 价值论证

**用户痛点**：
应用开发者在生产环境中依赖时序数据库 API 后，最担忧的是服务端升级导致现有代码失效。缺乏明确的版本生命周期承诺时，开发团队被迫在每次服务端升级前进行全量回归测试，升级决策被阻塞。

**现有方案不足**：
- InfluxDB 1.x → 2.x 升级引入破坏性变更，社区迁移工具不完善，大量用户滞留旧版本
- VictoriaMetrics 承诺向后兼容但无明确的废弃预告机制，新功能仅在最新版本可用
- 多数竞品未区分主版本（breaking）和次版本（non-breaking）的废弃策略，开发者无法评估升级风险

**本 Feature 价值**：
- 明确版本生命周期：主版本维护期 ≥ 2 年，次版本维护期 ≥ 1 年，废弃前 6 个月通过多渠道预告
- 废弃 API 保留只读兼容层：标记废弃的端点继续响应至少 6 个月，响应头携带废弃警告
- 提供版本迁移指南：每次 breaking change 附带自动化迁移脚本和兼容性检查工具

**SDK 与 API 版本兼容矩阵维护机制**：
- **责任方**：发布工程团队负责矩阵的编制、验证和发布
- **更新渠道**：兼容矩阵随 SDK 发布包分发（嵌入 SDK 二进制或作为同级 YAML 文件），同时在文档站点提供在线版本
- **矩阵格式**：YAML（机器可读），包含字段 `sdk_version`、`compatible_api_versions`、`deprecated_api_versions`、`minimum_api_version`、`eol_date`
- **CI 自动验证**：每次 API 发布触发兼容性矩阵校验流水线，验证所有活跃 SDK 版本与新 API 版本的兼容性；矩阵变更需通过代码评审，与 SDK 发布流程绑定

**成功指标**：
- N-1 版本 API 废弃预告期 ≥ 6 个月
- 版本升级导致客户端代码变更率 ≤ 5%（在兼容承诺期内）
- 废弃 API 的继续使用率在预告期结束后 30 天内降至 < 1%

## Feature: 提供命令行管理工具（CLI）

### Feature 价值论证

**用户痛点**：
平台运维工程师和应用开发者在日常操作（集群状态查看、数据导入导出、schema 管理、配置检查）时，缺乏统一的管理工具。被迫直接操作 HTTP API 或编写一次性脚本，操作繁琐且易出错，无法形成标准化的运维流程。

**现有方案不足**：
- InfluxDB CLI（influx）功能集中在数据查询，集群管理和配置操作仍需直接调用 API
- VictoriaMetrics 无官方 CLI，社区工具功能碎片化，缺乏统一的配置管理和认证集成
- 多数竞品 CLI 未针对脚本化运维设计（如结构化输出、非交互模式、批量操作）

**本 Feature 价值**：
- 统一 CLI 覆盖数据操作（写入/查询/导出）、集群管理（节点状态/拓扑/配置）、诊断工具（连接测试/性能探测）
- 支持交互式和非交互式两种模式：交互式用于探索，非交互式（`--output json`）用于脚本集成
- 内置配置文件管理：支持多环境配置切换（dev/staging/prod），避免命令行重复输入连接参数

**CLI 操作范围边界（与集群管理域的暴露策略）**：
| 集群管理域 Feature | CLI 暴露策略 | 说明 |
|-------------------|-------------|------|
| 节点故障自动检测与副本切换 | 只读 | CLI 可查询切换事件记录（见[集群管理域 Feature 1 Scenario #3](../requirements/集群管理.md#feature-实现节点故障自动检测与副本切换)），不可触发切换 |
| 集群节点动态增删与数据再平衡 | 不暴露 | 扩缩容操作通过集群管理域专用工具执行，CLI 不覆盖 |
| 边缘节点注册与拓扑发现 | 只读 | CLI 可查询拓扑状态（见[集群管理域 Feature 3 Scenario #3](../requirements/集群管理.md#feature-管理边缘节点注册与拓扑发现)），不可修改注册信息 |
| 容器编排平台生命周期管理 | 不暴露 | K8s 探针和信号处理为内部机制，CLI 不介入 |
| 滚动升级与版本灰度发布 | 只读 | CLI 可查询升级状态和版本信息，不可触发升级/回滚 |
| 节点级资源配额与过载保护 | 只读 | CLI 可查看配额使用情况，配额修改通过集群管理域管理接口 |
| 集群配置集中管理 | 不暴露 | 配置下发由集群管理域配置中心负责，CLI 不覆盖 |
| 边缘节点批量部署与初始化 | 不暴露 | 批量部署由独立部署工具负责，CLI 不覆盖 |

**成功指标**：
- CLI 覆盖操作类型 ≥ 10（写入、查询、导出、导入、节点列表、配置查看、连接测试、版本检查、用户管理、备份触发）
- 非交互模式脚本化成功率 ≥ 99%（CI/CD 流水线中可稳定调用）
- 常见运维操作（查看集群状态、导出数据）通过 CLI 完成时间 ≤ 30s

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 平台运维工程师通过 CLI 查看集群节点状态 | CLI 调用集群管理域公开 API（见[集群管理域 Feature 3 Scenario #3](../requirements/集群管理.md#feature-管理边缘节点注册与拓扑发现)）获取拓扑信息，命令返回所有节点健康状态、角色、版本，输出格式与文档一致，耗时 < 1s | 自动化 |
