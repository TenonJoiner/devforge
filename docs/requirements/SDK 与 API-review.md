#### LOW-1：Feature 5 Scenario 边界覆盖缺口

- **级别**：LOW
- **维度**：Scenario 完整性
- **位置**：Feature 5 / Scenario 清单
- **理由**：当前 7 个 Scenario 覆盖了"正常调用 N-1 版本"、"已标记废弃的 API"、"废弃期结束后"三种状态，但缺少"预告期内但尚未正式标记废弃"的过渡态场景，以及"开发者忽略废弃警告继续使用"的边界场景。版本生命周期有 4 个状态（正常→预告→废弃→移除），当前只覆盖了 3 个。
- **修复建议**：新增 Scenario："应用开发者在预告期内调用即将废弃的 API"——响应正常但头部增加 `Sunset` 警告；以及"应用开发者忽略废弃警告继续在生产环境使用"——系统统计并暴露废弃 API 调用率指标。

#### LOW-2：Feature 4 Scenario #1 可验收性不足

- **级别**：LOW
- **维度**：验收标准可验收性
- **位置**：Feature 4 / Scenario #1（文档导航）
- **理由**：验收标准"文档搜索'写入'返回 API 参考页作为首条结果"依赖搜索引擎的具体实现和索引状态，在不同环境（本地构建 vs 线上部署）可能产生不同排序，导致"通过/失败"判定存在主观性。
- **修复建议**：将验收标准改为更稳定的判定方式，例如"文档搜索'写入'返回的结果中 API 参考页位于前 3 条"，或增加"搜索响应时间 < 2s"作为辅助量化指标。

#### LOW-3：Feature 4 Scenario #2 可复现性不足

- **级别**：LOW
- **维度**：验收标准可验收性
- **位置**：Feature 4 / Scenario #2（快速入门）
- **理由**：验收标准"从安装到成功写入 ≤ 10 分钟"未定义前置条件（网络环境、依赖包是否已缓存、操作系统类型）。首次安装（需下载所有依赖）与重复安装（依赖已缓存）的时间差异可能达数倍，导致同一测试在不同环境下结果不一致。
- **修复建议**：明确环境假设，例如"在已预装语言运行时和包管理器的环境中，从安装 SDK 到成功写入 ≤ 10 分钟"，或拆分为两个标准：冷启动（无缓存）≤ 15 分钟，热启动（有缓存）≤ 5 分钟。

#### LOW-4：Feature 3 成功指标定义边界模糊

- **级别**：LOW
- **维度**：Feature 价值论证
- **位置**：Feature 3 / 成功指标
- **理由**："单 SDK 实例并发连接数上限 ≥ 100"与"连接复用率 ≥ 80%"两个指标同时存在时存在语义张力——如果连接复用率达到 80%，意味着大部分请求复用了已有连接，实际需要的并发连接数应远低于请求并发数。两个指标未澄清是"或"关系还是"且"关系，也未定义"并发连接数"的计量口径（物理 TCP 连接数 vs 逻辑请求通道数）。
- **修复建议**：澄清指标定义，例如"单 SDK 实例物理 TCP 连接数上限 ≥ 100（用于支撑多节点部署场景），在 1000 并发请求下连接复用率 ≥ 80%"，明确两个指标分别对应不同场景。

#### LOW-5：Feature 4 缺少数据工程师视角 Scenario

- **级别**：LOW
- **维度**：Actor 角色视角覆盖
- **位置**：Feature 4 / Scenario 清单
- **理由**："涉及 Actors"中明确列出"数据工程师（协作 Actor，集成示例库使用者）"，但 8 个 Scenario 全部为应用开发者视角。Scenario #3 虽提到数据工程师，但场景动作仍是"参考批量写入示例优化生产代码"，本质仍是应用开发行为。数据工程师作为独立 Actor 有其专属关注点（schema 管理、数据生命周期、降采样策略），文档域应至少覆盖一个数据工程师视角场景。
- **修复建议**：新增数据工程师视角 Scenario，例如"数据工程师查阅 schema 演进文档规划字段变更"——文档包含 schema 版本迁移指南、向后兼容性说明、破坏性变更检查清单。

### 新发现问题列表

#### CRITICAL

**C-01: Feature 3（SDK 连接管理）与「数据写入」域 Feature 5（边缘本地存储与断网缓冲）存在能力重叠和职责边界模糊**

- **问题描述**：Feature 3 Scenario #3 定义 SDK "断网期间写入请求进入本地缓冲队列，恢复后自动重发"；「数据写入」域 Feature 5 Scenario #2/#3/#7 定义边缘节点 "自动切换为纯本地缓冲模式"、"按时间序续传"。两者都涉及"断网缓冲+续传"，但以下关键问题未回答：
  1. SDK 层的"本地缓冲队列"与边缘节点层的"本地持久化日志"是同一物理存储还是两层？
  2. 如果是两层，SDK 缓冲的数据在边缘节点本地持久化后，网络恢复时是否会重复续传（SDK 续传一次 + 边缘节点续传一次）？
  3. 数据一致性如何保证——SDK 续传成功后边缘节点是否感知并去重？
- **影响**：架构设计阶段若不清除此重叠，可能导致实现时两套缓冲机制冲突或数据重复。
- **建议**：明确划分边界——SDK 连接管理负责"连接生命周期+请求重试策略"，边缘本地存储负责"数据持久化+有序续传"。SDK 在边缘部署模式下应禁用本地缓冲，完全依赖边缘节点的持久化能力；仅在直连云端模式下启用 SDK 本地缓冲。

---

#### HIGH

**H-01: Feature 2（多语言 SDK）版本检测机制与 Feature 5（API 版本兼容性）的服务端响应机制未对齐**

- **问题描述**：Feature 2 Scenario #4 定义 "SDK 检测到版本不兼容时抛出 `VersionMismatchError`"，但未说明检测机制——是基于服务端响应（HTTP 410/Deprecation 头）还是 SDK 本地维护的兼容矩阵？Feature 5 Scenario #2/#3 定义服务端返回 `Deprecation: true` 头和 HTTP 410，但 SDK 如何提前获知"不兼容"（在请求发送前还是收到响应后）未定义。
- **影响**：若 SDK 采用本地矩阵，矩阵的更新/分发机制缺失；若采用服务端响应，则 "VersionMismatchError" 应在收到响应后抛出，但此时请求已发出，与"检测"语义矛盾。
- **建议**：明确检测机制为"请求前本地矩阵预检 + 请求后服务端响应确认"双层模型，并定义兼容矩阵的维护责任方和更新渠道。

**H-02: Feature 6（CLI）集群状态查询与「集群管理」域状态来源未定义接口契约**

- **问题描述**：Feature 6 Scenario #1 定义 CLI "返回所有节点健康状态、角色、版本"，但未说明状态来源——是调用集群管理域 API 还是 CLI 自行探测？「集群管理」域 Feature 1 Scenario #3 定义了切换事件记录，Feature 3 Scenario #3 定义了拓扑查询响应。若 CLI 直接调用集群管理 API，接口契约（端点、字段、延迟 SLA）未在本域引用；若 CLI 自行探测，则两套健康判定逻辑可能导致状态不一致。
- **影响**：实现阶段可能出现 CLI 显示状态与集群管理面板不一致，或 CLI 对集群管理 API 的隐式依赖未被识别。
- **建议**：明确 CLI 集群状态查询走集群管理域公开 API，在跨域依赖中引用集群管理域的具体 Scenario 和接口定义。

**H-03: SDK 连接管理的"客户端负载均衡"与「集群管理」域"副本切换后自动重定向"的实现机制不一致**

- **问题描述**：「集群管理」域 Feature 1 Scenario #10 定义 "切换完成后 SDK 自动重连新主副本，重连耗时 <= 5s"，Feature 4 Scenario #10 也有相同描述。但 Feature 3（SDK 连接管理）Scenario #4 定义的是 "故障节点请求在 3 个探测周期（每个 5s）内被路由至其他节点"。两个域对"自动切换"的实现机制描述不一致：
  - 集群管理域暗示 SDK 被动接收拓扑变更通知后重连（5s 内完成）
  - SDK 连接管理域描述的是 SDK 主动探测+路由（15s 窗口）
- **影响**：架构设计时两套机制可能重复实现或冲突——服务端推送拓扑变更 vs SDK 主动健康探测，两者并存时行为不可预期。
- **建议**：统一机制——服务端通过响应头或独立推送通道通知拓扑变更，SDK 在收到通知后 5s 内完成重连；SDK 的主动探测作为兜底机制，探测周期应与集群管理域的心跳周期对齐（默认 2s）。

#### MEDIUM

**M-01: CLI 与「集群管理」域的能力边界未清晰划分**

- **问题描述**：待澄清问题 #2 已记录 "CLI 是否覆盖边缘节点管理操作"，但仅标记为 P1，未在文档中给出明确的边界划分原则。Feature 6 的价值论证提到 CLI 覆盖 "集群管理（节点状态/拓扑/配置）"，但集群管理域的 Feature 2（节点增删与再平衡）、Feature 5（滚动升级）、Feature 7（配置集中管理）等是否也通过 CLI 暴露？若 CLI 只读而集群管理 API 可写，CLI 的价值主张"统一的管理工具"被削弱。
- **建议**：在 Feature 6 价值论证中增加"CLI 操作范围边界"段落，明确 CLI 对集群管理域各 Feature 的暴露策略（只读/读写/不暴露），并引用集群管理域的接口定义。

**M-02: Feature 1 错误码与「安全合规」域错误码未明确对齐关系**

- **问题描述**：Feature 1 Scenario #6 定义错误码 `AUTHENTICATION_FAILED`，「安全合规」域 Feature 2（身份认证）Scenario #7 也定义了无效凭证拒绝。两者是否为同一错误码？如果是，应在跨域依赖中明确引用安全合规域的定义；如果不是，需说明差异（如安全合规域的错误码包含审计日志记录要求，而 API 层的错误码面向客户端）。
- **建议**：在 Feature 1 错误码体系说明中增加"跨域错误码继承关系"段落，明确 API 层错误码与安全合规域错误码的映射规则。

**M-03: 故障模式覆盖矩阵 Feature 4（开发者文档）的"节点故障"列标记为"—"，但 Scenario #8 涉及资源受限场景**

- **问题描述**：矩阵中 Feature 4 "节点故障"列标记为"—"（由其他域覆盖），但 Scenario #8 "边缘示例展示本地缓冲+重试策略，可在内存 <= 256MB 设备运行"实际上涉及资源耗尽场景。矩阵中 Feature 4 的"资源耗尽"列也标记为"—"，该 Scenario 未在矩阵中体现。
- **建议**：将 Scenario #8 归入"资源耗尽"列（边缘设备内存受限属于资源耗尽范畴），或补充说明为何文档 Feature 的故障矩阵可不覆盖此场景。

**M-04: NFR "SDK 与 API 版本矩阵"的维护机制缺失**

- **问题描述**：NFR 兼容性维度要求 "每个 SDK 版本明确标注兼容的 API 版本范围"，但 Feature 2 的成功指标仅要求 "SDK API 与 REST API 功能覆盖率 >= 95%"，未要求版本矩阵的维护机制（谁维护、如何分发、更新频率、与 CI 的集成方式）。
- **建议**：在 Feature 5（API 版本兼容性）或 Feature 2 中补充版本矩阵的维护责任方（建议由发布工程团队维护，随 SDK 发布包分发）、矩阵格式（YAML/JSON）、以及 CI 自动验证矩阵准确性的机制。

#### LOW

**L-01: API 版本命名规则未统一定义**

- **问题描述**：Feature 1 Scenario #2 提到 `/v2/query`，但 Feature 5 的废弃策略中只提到 `/v1/write`。v1/v2 对应 major 版本的规则未在文档中定义（如 v1 = major 1，v2 = major 2）。
- **建议**：在 Feature 1 或 Feature 5 中增加一行说明 "URL 路径版本号格式为 `v{major}`，与语义版本的主版本号对应"。

**L-02: Feature 3 Scenario #8 "误判断线率 = 0"过于绝对**

- **问题描述**：边缘网络高延迟场景下偶发丢包可能导致心跳误判，"误判断线率 = 0"在实际网络环境中不可达成。
- **建议**：改为"持续误判断线率 = 0（单次心跳丢失不触发断线）"，或增加容错窗口说明（如"允许在 3 个心跳周期内偶发丢包"）。

**L-03: 自检清单中"依赖关系已确认"无法独立验证**

- **问题描述**：自检清单勾选 "本域 Feature 在 product-spec.md 中的依赖关系已确认"，但文档中未提供 product-spec.md 中依赖关系的具体引用（如章节号、Feature 编号对应关系）。
- **建议**：在文档元数据或跨域依赖说明中增加 product-spec.md 的具体引用锚点。

### 新发现问题列表

#### MEDIUM

**M-05: Feature 5 Scenario #10（离线降级）与 Feature 2 Scenario #4（双层检测）的本地矩阵缓存策略未对齐**

- **问题描述**：Feature 5 Scenario #10 定义"离线状态下使用本地缓存的兼容矩阵继续工作，缓存有效期 7 天"；Feature 2 Scenario #4 定义"请求前本地兼容矩阵预检"。两者都依赖"本地兼容矩阵"，但以下问题未回答：
  1. 矩阵缓存的初始来源——是 SDK 首次启动时从服务端拉取，还是随 SDK 发布包内置？
  2. 矩阵更新机制——服务端发布新 API 版本后，矩阵如何在已运行的 SDK 实例中更新？
  3. 7 天缓存有效期与 CI 自动验证（Feature 5 维护机制中定义）的时效关系——CI 验证新矩阵后，已部署 SDK 的缓存何时失效？
- **影响**：若矩阵更新依赖 SDK 重启或重新安装，生产环境中 SDK 实例可能长期运行旧矩阵，导致预检结果与服务端实际状态不一致。
- **建议**：在 Feature 2 或 Feature 5 中补充矩阵缓存的更新策略，例如"SDK 每次启动时拉取最新矩阵，运行期间按可配置周期（默认 24h）后台刷新缓存；缓存刷新失败时继续使用旧缓存直至 7 天有效期届满"。

**M-06: Feature 3 Scenario #3 的"边缘部署模式"判定条件未定义**

- **问题描述**：Feature 3 Scenario #3 定义"边缘部署模式下 SDK 不启用本地缓冲，依赖边缘节点持久化能力"，但未说明 SDK 如何判定自身处于"边缘部署模式"——是通过配置显式声明（如 `deployment_mode: edge`），还是通过自动探测（如检测到边缘节点代理存在）？
- **影响**：实现阶段可能出现模式误判——直连云端的 SDK 被误判为边缘模式（导致断网时数据丢失），或边缘部署的 SDK 被误判为直连模式（导致与边缘节点持久化日志重复缓冲）。
- **建议**：明确模式判定机制，例如"边缘部署模式通过 SDK 初始化配置显式声明（`deployment_mode: edge/cloud`），默认值为 `cloud`；SDK 不自动探测部署模式，避免环境误判"。
