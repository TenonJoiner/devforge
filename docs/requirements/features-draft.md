# KV Cache Offloading 存储系统 — Feature 与 Scenario 草案

## 文档说明

本文档按 **Feature Domain** 组织需求，不从内部架构子系统出发。所有描述均从用户或外部系统视角书写，不出现内部模块名（如 BlockAllocator、IndexService）。

文档结构：
- 每个 Feature Domain 下包含 2~4 个 Feature
- 每个 Feature 说明：做什么、为什么（价值论证）、涉及 Actor
- 每个 Feature 下包含 ≥5 个 Scenario，强制覆盖正常路径 + 至少 4 种故障/异常/运维模式
- 每个 Scenario 包含：Actor、前置条件、触发动作、预期行为、**可量化的验证方法**

---

## Feature Domain 1: cache-offloading

**领域价值**：将 LLM 推理产生的 KV Cache 从昂贵的 GPU HBM 卸载到更廉价的存储介质（DRAM/SSD/远程节点），并在后续请求的匹配前缀上快速加载复用，从而降低 TTFT、提升 GPU 利用率和整体吞吐。

---

### Feature 1.1: Prefix Cache 加载

**做什么**：当推理引擎收到新请求时，系统根据请求的 token 序列前缀，查询缓存中是否存在可复用的 KV Cache。若存在，则将匹配的 KV Cache 异步加载到引擎的 GPU 内存中，减少重复计算。

**为什么**：Prefix Cache 命中是 KV Cache Offloading 的核心收益来源。根据标杆研究，在长上下文和 RAG/多轮对话场景中，prefix 复用率可达 30%~80%，TTFT 可降低 2×~22×。

**涉及 Actor**：推理引擎、缓存服务实例、全局元数据服务（分布式模式下）、AI 应用开发者、平台工程师。

#### Scenario 1.1.1: 正常路径 — 前缀命中，TTFT 显著降低

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 推理引擎（vLLM）、缓存服务实例 |
| **前置条件** | 缓存服务已启动并与引擎完成握手；目标 prefix 已存在于 DRAM 层；引擎请求的 token 序列前 N 个 token 与缓存完全匹配 |
| **触发动作** | 引擎调度器向缓存服务发起 prefix 匹配查询；匹配成功后，Worker 调用 `start_load_kv()` 启动异步加载 |
| **预期行为** | 异步加载在 prefill 计算前完成或与之重叠；引擎仅对未命中部分执行计算；TTFT 相比无缓存时降低 ≥50% |
| **验证方法** | 采集 1 万条请求样本，prefix hit 场景下 P50 TTFT < 基准值/2，P99 TTFT < 基准值/1.5；命中率统计误差 < 1% |

#### Scenario 1.1.2: 节点故障 — 存储节点宕机，请求降级处理

| 字段 | 内容 |
|------|------|
| **类型** | 节点故障 |
| **Actor** | 推理引擎、缓存服务实例、集群编排器、全局元数据服务 |
| **前置条件** | 目标 prefix 原本存储在节点 A 的 DRAM 中；节点 A 突然宕机或被编排器判定为不健康 |
| **触发动作** | 引擎发起 prefix 查询；系统发现节点 A 不可达 |
| **预期行为** | 全局元数据服务在 3 个心跳周期（默认 10s）内感知故障；查询自动降级为 "miss"；引擎 fallback 到全量 prefill 计算，请求不失败；错误率 = 0% |
| **验证方法** | 故障注入测试：随机 kill -9 一个缓存服务实例，连续发送 1000 个请求，0 个请求返回 5xx/崩溃；TTFT 恢复至无缓存基线的时间 < 15s |

#### Scenario 1.1.3: 并发冲突 — 多个引擎实例同时读取同一缓存块

| 字段 | 内容 |
|------|------|
| **类型** | 并发冲突 |
| **Actor** | 多个推理引擎实例、缓存服务实例、数据传输服务 |

| **触发动作** | 多个引擎在 100ms 时间窗口内并发发起加载请求 |
| **预期行为** | 缓存服务实例支持读共享，不阻塞并发读取；各引擎获取的数据内容一致；总读取带宽不超过介质带宽上限的 90% |
| **验证方法** | 压力测试：4 个引擎并发读取同一 2GB 缓存块，每个引擎数据校验（checksum）100% 通过；峰值带宽 ≥ 介质理论带宽的 80%；无死锁或超时 |

#### Scenario 1.1.4: 网络分区 — 查询期间网络分区，恢复后数据一致

| 字段 | 内容 |
|------|------|
| **类型** | 网络分区 |
| **Actor** | 推理引擎、缓存服务实例、全局元数据服务 |
| **前置条件** | 引擎与缓存服务实例之间的网络出现临时分区（如 iptables DROP 10s） |
| **触发动作** | 分区期间引擎发起 prefix 查询；分区恢复后再次发起查询 |
| **预期行为** | 分区期间查询在 2s 内超时并返回 miss；分区恢复后，系统通过心跳重新同步状态；再次查询能在 50ms 内正常返回命中结果；无数据不一致或脏读 |
| **验证方法** | 使用 tc/iptables 注入 10s 网络分区，分区期间 100% 请求降级为 miss；恢复后 30s 内命中率恢复至分区前水平的 95% 以上；校验恢复后加载的数据 checksum 与分区前一致 |

#### Scenario 1.1.5: 数据损坏 — 加载时发现校验失败，触发失效与降级

| 字段 | 内容 |
|------|------|
| **类型** | 数据损坏 |
| **Actor** | 推理引擎、缓存服务实例、平台工程师 |
| **前置条件** | 某缓存块在存储过程中因硬件异常导致部分数据损坏；损坏块仍被索引引用 |
| **触发动作** | 引擎加载该缓存块；缓存服务在加载前/后进行 CRC64 校验 |
| **预期行为** | 校验失败时，立即标记该缓存块为 "invalid"，并向监控系统上报 `cache_data_corruption` 告警；引擎收到 load failure 信号后 fallback 到全量 prefill；该 prefix 的后续请求不再尝试加载此块 |
| **验证方法** | 故意篡改 1% 的缓存块数据，触发加载：100% 损坏块被检测并标记失效；0 个引擎实例因加载损坏数据而崩溃或产生错误结果；告警在 5s 内出现在监控面板 |

### Feature 1.2: 异步 KV Cache 存储

**做什么**：在 prefill 或 decode 阶段产生新的 KV Cache 后，系统通过异步流水线将其卸载到 DRAM、SSD 或远程存储层，不阻塞引擎的连续 batching 和 token 生成。

**为什么**：GPU HBM 是昂贵且稀缺的资源。将已生成但暂不立即复用的 KV Cache 卸载到低层存储，可以释放 GPU 内存供更多请求使用，从而提升并发度和整体吞吐。

**涉及 Actor**：推理引擎、缓存服务实例、数据传输服务、平台工程师。

#### Scenario 1.2.1: 正常路径 — 异步存储完成，不阻塞解码

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 推理引擎、缓存服务实例 |
| **前置条件** | 引擎完成某请求的 prefill 计算；系统配置为将新 KV Cache 异步卸载到 DRAM/SSD |
| **触发动作** | Worker 调用 `start_store_kv()` 启动异步存储；引擎继续执行下一个 step 的调度 / decode |
| **预期行为** | 存储操作在后台完成；引擎的 decode TPOT 增加 < 5%（与无缓存场景对比）；存储完成后，该 KV Cache 可被后续请求查询命中 |
| **验证方法** | 对比基准（无 offloading）和开启 offloading 的 TPOT：P99 增量 < 5%；存储完成到可查询的延迟 < 100ms；1 万条请求样本中 0 次因存储阻塞导致 decode 超时 |

#### Scenario 1.2.2: 资源耗尽 — SSD/DRAM 空间不足，优雅拒绝或降级

| 字段 | 内容 |
|------|------|
| **类型** | 资源耗尽 |
| **Actor** | 推理引擎、缓存服务实例、平台工程师 |
| **前置条件** | 本地 SSD 或 DRAM 利用率达到配置上限（如 95%） |
| **触发动作** | 引擎持续产生新的 KV Cache 并触发存储请求 |
| **预期行为** | 当空间不足时，系统优先保留热度更高的旧数据；新数据若无法写入，则向引擎返回 "store_skipped" 信号；引擎继续正常推理，不因存储失败而中断；系统向监控上报 `storage_capacity_full` 告警 |
| **验证方法** | 人为填满 SSD/DRAM 至 95%，继续注入请求：0 个请求失败或崩溃；存储拒绝率 100%（即新数据不再写入）；5s 内触发 `storage_capacity_full` 告警；引擎 TPOT 与正常场景差异 < 5% |

#### Scenario 1.2.3: 节点故障 — 存储过程中节点宕机，数据不丢失或已降级

| 字段 | 内容 |
|------|------|
| **类型** | 节点故障 |
| **Actor** | 推理引擎、缓存服务实例、集群编排器 |
| **前置条件** | 某缓存服务实例正在执行异步存储；存储完成前该实例所在节点突然宕机 |
| **触发动作** | kill -9 缓存服务实例进程 |
| **预期行为** | 引擎检测到存储实例不可达后，将该请求标记为 "store_failed"；引擎的 GPU 内存中的 KV Cache 仍可用于当前请求的 decode 完成；由于存储未完成，该数据自然不可被后续请求复用，系统不承诺未确认写入的数据持久化 |
| **验证方法** | 在 1000 次存储过程中随机注入 3 次实例 kill：0 次引擎崩溃；当前请求 100% 正常完成 decode；未完成的存储不留下半写数据（通过 fsck/数据校验确认） |

#### Scenario 1.2.4: 升级/回滚 — 缓存服务版本升级期间，存储不中断

| 字段 | 内容 |
|------|------|
| **类型** | 升级/回滚 |
| **Actor** | 推理引擎、缓存服务实例、平台工程师 |
| **前置条件** | 集群执行滚动升级，旧版本实例逐个被新版本替换 |
| **触发动作** | 升级过程中引擎持续发送存储请求 |
| **预期行为** | 旧实例在收到 SIGTERM 后，完成进行中的异步存储，拒绝新存储请求；引擎的 Connector 在 2s 内重连到新实例；升级期间请求的存储成功率 > 99% |
| **验证方法** | 滚动升级 3 节点集群，持续注入存储请求：存储成功率 ≥ 99%；0 次引擎因重连失败而 stuck > 5s；回滚操作在 60s 内完成且数据可继续读取 |

### Feature 1.3: 跨层/跨节点 KV Cache 迁移

**做什么**：根据访问热度和容量策略，系统自动将 KV Cache 在不同存储层级（HBM→DRAM→SSD→Remote）之间晋升或降级，并在分布式部署下支持跨节点数据迁移以达到负载均衡。

**为什么**：静态放置无法适应 workload 变化。热数据需要靠近 GPU 以降低加载延迟，冷数据需要下沉到廉价介质以释放高速缓存空间。

**涉及 Actor**：缓存服务实例、数据传输服务、全局元数据服务、平台工程师。

#### Scenario 1.3.1: 正常路径 — 热数据从 SSD 晋升到 DRAM

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 缓存服务实例、数据传输服务 |
| **前置条件** | 某 prefix 存储在 SSD 层且最近 5 分钟内被访问 3 次以上；DRAM 层有可用空间 |
| **触发动作** | 系统后台迁移任务检测到热度阈值触发，启动从 SSD 到 DRAM 的晋升 |
| **预期行为** | 晋升在后台异步完成；晋升期间该 prefix 仍可从 SSD 正常读取；晋升完成后，后续查询从 DRAM 响应，加载延迟降低 ≥50% |
| **验证方法** | 监控某 prefix 的存储层标签变化：从 L3 变为 L2 的时间 < 30s；晋升期间 100 次读取全部成功；晋升后 P99 加载延迟 < 晋升前 P99 的 50% |

#### Scenario 1.3.2: 运维操作 — 管理员在线扩容，客户端无感知

| 字段 | 内容 |
|------|------|
| **类型** | 运维操作 |
| **Actor** | 平台工程师、缓存服务实例、全局元数据服务、集群编排器 |
| **前置条件** | 集群需要扩容以增加缓存容量；新节点已加入集群并完成初始化 |
| **触发动作** | 平台工程师通过运维 CLI 触发数据重平衡；或系统自动根据 `free_ratio_first` 策略将新数据导向新节点 |
| **预期行为** | 新节点在 60s 内开始接收写入和新晋升的数据；已有数据在后台逐步迁移（不阻塞前台读写）；迁移期间客户端 P99 TTFT 增加 < 10% |
| **验证方法** | 向 2 节点集群添加第 3 节点，持续注入读写请求：0 个请求失败；新节点在 60s 内接收 ≥10% 的新写入；扩容后 10 分钟内各节点存储量标准差 < 平均值的 30% |

#### Scenario 1.3.3: 网络分区 — 迁移任务执行中发生网络分区

| 字段 | 内容 |
|------|------|
| **类型** | 网络分区 |
| **Actor** | 缓存服务实例、数据传输服务、全局元数据服务 |
| **前置条件** | 一个跨节点迁移任务正在进行中；源节点与目标节点之间出现 15s 网络分区 |
| **触发动作** | tc 注入 15s 网络延迟/丢包 |
| **预期行为** | 迁移任务在 5s 内检测到超时并暂停；源节点保持原数据不变；分区恢复后，迁移任务自动重试；最终数据在目标节点完整可用，无半写块 |
| **验证方法** | 注入分区后检查：迁移状态变为 "paused"；源数据 checksum 不变；恢复后 60s 内迁移完成；目标数据 checksum 与源一致 |

#### Scenario 1.3.4: 并发冲突 — 迁移期间读取请求并发访问

| 字段 | 内容 |
|------|------|
| **类型** | 并发冲突 |
| **Actor** | 推理引擎、缓存服务实例、数据传输服务 |
| **前置条件** | 某 prefix 正在从节点 A 迁移到节点 B；此时引擎发起对该 prefix 的读取 |
| **触发动作** | 高并发读取与后台迁移同时发生 |
| **预期行为** | 读取优先：系统优先从源节点（A）或已完成复制的目标节点（B）响应读取；迁移不阻塞读取；读取响应延迟增加 < 20% |
| **验证方法** | 在持续 1GB 数据迁移的同时，以 100 QPS 读取同一 prefix：读取成功率 100%；P99 延迟 < 正常读取 P99 的 120%；迁移最终完成时间增加 < 50% |

## Feature Domain 2: data-lifecycle

**领域价值**：管理 KV Cache 数据从产生、持久化、分层存储、淘汰到最终清理的完整生命周期，确保缓存空间高效利用、数据在需要时可用、过期/损坏数据不影响系统正确性。

---

### Feature 2.1: 缓存写入与持久化

**做什么**：将推理引擎生成的 KV Cache 可靠地写入各存储层级，确保在进程正常退出或异步刷盘策略下，数据满足持久化承诺。

**为什么**：写入路径是缓存服务与推理引擎的边界。不可靠的写入会导致 "伪 miss"（数据本已生成但未能复用），直接降低缓存命中率并增加 GPU 计算浪费。

**涉及 Actor**：推理引擎、缓存服务实例、外部持久化存储。

#### Scenario 2.1.1: 正常路径 — 写入成功并立即可查询

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 推理引擎、缓存服务实例 |
| **前置条件** | 引擎完成 prefill；系统配置写入到 DRAM（立即持久化）或 SSD（异步刷盘） |
| **触发动作** | Worker 调用存储接口提交 KV Cache 数据 |
| **预期行为** | 数据写入完成后，索引在 100ms 内更新；后续请求对该 prefix 的查询返回命中；写入确认到可查询的延迟 < 100ms |
| **验证方法** | 写入完成后立即（0ms/50ms/100ms/200ms 间隔）发起查询：100ms 时命中率 ≥ 99%；200ms 时命中率 100%；0ms 时允许 miss |

#### Scenario 2.1.2: 数据损坏 — 写入后校验发现不一致，触发重写

| 字段 | 内容 |
|------|------|
| **类型** | 数据损坏 |
| **Actor** | 缓存服务实例、平台工程师 |
| **前置条件** | 写入过程中发生位翻转或存储介质异常；系统配置了写入后校验策略 |
| **触发动作** | 缓存服务在写入完成后执行 CRC64 校验 |
| **预期行为** | 校验失败时，该写入标记为失败；系统尝试最多 1 次重写；若重写仍失败，丢弃该数据并向监控上报 `write_checksum_mismatch` 事件；不将损坏数据加入索引 |
| **验证方法** | 使用 LD_PRELOAD 或 fault injection 使 5% 的写入产生位翻转：100% 损坏数据被检测；0% 损坏数据进入索引；告警事件在 5s 内触发 |

#### Scenario 2.1.3: 资源耗尽 — 写入时缓冲区满，执行流控

| 字段 | 内容 |
|------|------|
| **类型** | 资源耗尽 |
| **Actor** | 推理引擎、缓存服务实例 |
| **前置条件** | 突发高并发写入导致异步刷盘缓冲区达到容量上限（如 1GB） |
| **触发动作** | 引擎以峰值 2 倍于常态的速率持续提交存储请求 |
| **预期行为** | 缓冲区满时，新写入在 100ms 内返回 `store_backpressure`；引擎收到信号后可选择跳过存储或等待；系统不崩溃、不丢已确认的数据；缓冲区水位在 30s 内恢复正常 |
| **验证方法** | 注入 2x 峰值写入持续 60s：`store_backpressure` 返回率 > 80%；0 次 OOM 或 panic；已确认写入的数据 100% 可查询；60s 后缓冲区利用率 < 70% |

#### Scenario 2.1.4: 节点故障 — 写入过程中节点宕机，数据完整性可恢复

| 字段 | 内容 |
|------|------|
| **类型** | 节点故障 |
| **Actor** | 缓存服务实例、集群编排器 |
| **前置条件** | 写入到 SSD 层的过程正在进行；此时节点突然断电或进程被 kill -9 |
| **触发动作** | 在写入中途注入 kill -9 |
| **预期行为** | 节点重启后，系统执行写日志（write-ahead log）或文件系统一致性检查；已完整写入的 chunk 保留；半写 chunk 被识别并删除；索引自动修复，不引用损坏或半写数据 |

### Feature 2.2: 缓存淘汰与分层迁移

**做什么**：当高速缓存层（HBM/DRAM）空间不足时，依据热度、访问时间和容量策略，将冷数据降级到更低速层级或永久淘汰，为新数据腾出空间。

### Feature 3.3: Worker 侧异步加载/存储流水线

**做什么**：在推理引擎的 Worker 执行模型 forward 时，通过层级别（layer-wise）的异步 API，将 KV Cache 的加载和存储与计算重叠，最小化传输对推理延迟的影响。

**为什么**：层级别流水线是隐藏传输延迟的关键技术。根据标杆研究，该优化可将 offloading 的吞吐提升 ~10×，并将加载延迟对 TTFT 的影响降到最低。

**涉及 Actor**：推理引擎（Worker）、缓存服务实例、数据传输服务。

#### Scenario 3.3.1: 正常路径 — 层级别流水线隐藏加载延迟

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 推理引擎（Worker）、缓存服务实例、数据传输服务 |
| **前置条件** | 请求需要加载 32 层 KV Cache；引擎支持 layer-wise pipeline（如 vLLM V1） |
| **触发动作** | `start_load_kv()` 在 forward 前启动；每层 attention 计算前调用 `wait_for_layer_load(layer_id)` |
| **预期行为** | 层 N 的计算与层 N+1 的加载重叠；实际增加的 TTFT < 原始加载时间的 20%；decode 阶段 TPOT 不受影响 |
| **验证方法** | 测量无 pipeline（同步加载）vs 有 pipeline 的 TTFT：pipeline 将额外 TTFT 降低 ≥70%；各层 `wait_for_layer_load` 平均等待时间 < 0.5ms；GPU 利用率保持 > 90% |

| **验证方法** | 在加载第 10 层时注入 5s 网络分区：请求 100% 最终成功；超时检测时间 < 配置超时 + 200ms；回退重算产生的额外延迟 < 原始全量 prefill 的 50% |

#### Scenario 3.3.3: 数据损坏 — 层加载数据校验失败

| 字段 | 内容 |
|------|------|
| **类型** | 数据损坏 |

| **验证方法** | 随机损坏 5% 的 layer 数据：100% 损坏 layer 被检测；0 次引擎 forward 产生错误结果；回退重算成功率 100%；损坏数据 60s 内从缓存索引中移除 |

### Feature 4.1: 节点加入与退出

### Feature 4.2: 实例故障检测与自动恢复

**做什么**：通过心跳、健康探针和异常指标监控，及时发现缓存服务实例的故障，并触发自动的数据重平衡、副本修复和路由切换。

**为什么**：分布式系统中节点故障是常态。若故障不能被快速检测和恢复，会导致缓存命中率下降、请求堆积，甚至影响推理引擎的稳定性。

**涉及 Actor**：缓存服务实例、全局元数据服务、集群编排器、监控系统、平台工程师。

### Feature 4.3: 负载均衡与容量调度

**做什么**：在分布式部署下，将新的 KV Cache 写入和加载请求合理分布到各个缓存服务实例上，避免单节点过载；同时根据各节点的容量利用率动态调整流量分配。

**为什么**：不均衡的负载会导致热点节点成为瓶颈，出现高延迟、高拒绝率，甚至触发 OOM 或磁盘满错误。负载均衡是扩展性的前提。

**涉及 Actor**：全局元数据服务、缓存服务实例、推理引擎、平台工程师。

#### Scenario 4.3.1: 正常路径 — 请求均匀分布到各节点

| **类型** | 正常路径 |
| **Actor** | 推理引擎、全局元数据服务、缓存服务实例 |
| **前置条件** | 3 节点集群；各节点容量和性能一致；写入请求无特定偏斜 |
| **触发动作** | 引擎以 1000 QPS 持续发送存储请求 |
| **预期行为** | 各节点写入量标准差 < 平均值的 20%；查询延迟 P99 在各节点间差异 < 30%；无单节点利用率超过 85% |
| **验证方法** | 运行 30 分钟：各节点存储量差异 < 20%；CPU/DRAM/SSD 利用率最高值 < 85%；P99 延迟差异 < 30% |

## Feature Domain 5: observability

| **触发动作** | 实例逐个重启 |
| **预期行为** | 升级实例的 /metrics 在重启后 10s 内恢复可访问；Prometheus 最多丢失 1 个采集周期的数据；升级期间全局仪表盘不中断；历史数据连续性不受影响 |
| **验证方法** | 滚动升级 3 节点集群：单个实例指标缺失周期 ≤ 1（即 ≤ 15s）；全局聚合指标（如集群总命中率）曲线无断崖式下跌；升级期间 Dashboard 可正常刷新 |

---

### Feature 5.2: 故障诊断与追踪

**做什么**：为每个 KV Cache 请求和后台任务生成可关联的 trace、日志和事件记录，支持跨节点、跨层级的端到端故障定位。

**为什么**：分布式系统中的故障往往跨越多个节点和层级。没有 trace，很难回答 "为什么这个请求的 TTFT 突然 spike"、"数据在那个环节丢失" 等问题。

**涉及 Actor**：缓存服务实例、数据传输服务、全局元数据服务、平台工程师、监控系统。

#### Scenario 5.2.1: 正常路径 — Trace 完整记录请求生命周期

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 推理引擎、缓存服务实例、数据传输服务 |
| **前置条件** | OpenTelemetry Collector 已配置；系统开启 1% trace 采样 |
| **触发动作** | 引擎发起一个 prefix 匹配 + 加载请求 |
| **预期行为** | trace 包含以下 span：prefix 查询、索引查找、数据传输启动、逐层 wait、加载完成、引擎消费；各 span 包含延迟、状态码、错误信息、数据来源节点/层级标签；端到端 trace 在 95% 情况下完整（无 span 丢失） |
| **验证方法** | 采集 1000 条 trace：完整 trace 比例 ≥ 95%；各 span 时间戳单调递增；trace ID 在引擎和缓存服务间正确传递；Jaeger/Tempo 中可检索到全部 trace |

#### Scenario 5.2.2: 网络分区 — 分区期间 Trace 标记异常

| 字段 | 内容 |
|------|------|
| **类型** | 网络分区 |
| **Actor** | 缓存服务实例、数据传输服务、全局元数据服务 |
| **前置条件** | 某加载请求执行过程中发生 10s 网络分区 |
| **触发动作** | 分区导致某 span（如数据传输）超时 |
| **预期行为** | 超时 span 标记 `error=true` 并附带 `timeout` 类型；上游 span 能清晰展示超时发生的层级和节点；trace 不中断，请求 fallback 路径的 span 也被完整记录 |
| **验证方法** | 注入分区后检索受影响 trace：100% 显示 timeout 错误标签；timeout span 的 duration 与配置超时偏差 < 500ms；fallback 路径（如重算）的 span 可被检索 |

#### Scenario 5.2.3: 数据损坏 — 检测到异常命中，Trace 定位根因

| 字段 | 内容 |
|------|------|
| **类型** | 数据损坏 |
| **Actor** | 缓存服务实例、平台工程师 |
| **前置条件** | 某请求的加载数据 checksum 校验失败 |
| **触发动作** | 引擎加载数据并发现校验失败 |
| **预期行为** | trace 中加载 span 标记 `checksum_mismatch`；关联日志输出数据来源节点、数据块 ID、存储层级、最后修改时间；平台工程师可在 1 分钟内从 trace 定位到具体源节点和文件路径 |
| **验证方法** | 模拟 10 次校验失败：100% 的 trace 中包含 `checksum_mismatch` 标签；日志中包含准确的 node_id + chunk_id + layer 信息；人工排查平均定位时间 < 1 分钟 |

#### Scenario 5.2.4: 并发冲突 — 高并发下 Trace 采样不丢关键事件

| 字段 | 内容 |
|------|------|
| **类型** | 并发冲突 |
| **Actor** | 缓存服务实例、数据传输服务 |
| **前置条件** | 系统处于高并发状态；trace 采用 1% 采样；某个请求恰好是已采样的 trace 且涉及并发竞争 |
| **触发动作** | 高并发读取与后台迁移/淘汰同时发生 |
| **预期行为** | 即使并发竞争激烈，已采样 trace 的关键 span（如竞争开始、锁获取、操作完成）不丢失；trace 能正确展示请求在并发控制下的等待时间和实际执行时间 |
| **验证方法** | 在 10 万 QPS 下运行 1 小时：抽样检查 100 条高并发期间的 trace：关键 span 缺失率 = 0%；并发等待时间被准确记录；无因 trace buffer 满导致的 span 丢弃告警 |

### Feature 5.3: 容量与成本分析

**做什么**：基于缓存命中率、数据生命周期、存储介质成本模型，为平台工程师提供容量预测、成本归因和扩容建议。

**为什么**：缓存服务的运营成本与存储介质选择直接相关。平台工程师需要知道 "当前 DRAM/SSD 配比是否最优"、"下个月是否需要扩容"、"不同业务线的缓存成本分别是多少"。

**涉及 Actor**：平台工程师、监控系统、缓存服务实例。

#### Scenario 5.3.1: 正常路径 — 容量预测准确

| 字段 | 内容 |
|------|------|
| **类型** | 正常路径 |
| **Actor** | 平台工程师、监控系统、缓存服务实例 |
| **前置条件** | 系统已运行 7 天以上；有历史存储增长曲线和命中率数据 |
| **触发动作** | 监控系统每日生成容量预测报告 |
| **预期行为** | 报告预测未来 7 天的 DRAM/SSD 使用趋势；预测误差 < ±15%（基于历史同期）；当预测利用率将超过 85% 时发出预警 |
| **验证方法** | 对比 7 天前的预测和实际使用量：平均绝对百分比误差（MAPE）< 15%；预警触发时间比实际达到 85% 的时间提前 ≥ 24 小时 |

#### Scenario 5.3.2: 资源耗尽 — 容量预警提前发出

| 字段 | 内容 |
|------|------|
| **类型** | 资源耗尽 |
| **Actor** | 监控系统、平台工程师、缓存服务实例 |
| **前置条件** | 某层存储按当前增长速率将在 3 天内达到 95% |
| **触发动作** | 监控系统每日容量评估任务检测到风险 |
| **预期行为** | 在达到 95% 前至少 72 小时发出 `capacity_warning` 告警；告警包含建议措施（如清理冷数据、扩容节点、调整淘汰策略）；告警按严重程度分级（ INFO / WARNING / CRITICAL ） |
| **验证方法** | 人为陡增写入速率使某层 3 天内将达 95%：72 小时前收到 WARNING 告警；48 小时前收到 CRITICAL 告警；告警建议措施中至少包含 "扩容" 和 "加速淘汰" 两项 |

#### Scenario 5.3.3: 节点故障 — 故障后容量重新评估

| 字段 | 内容 |
|------|------|
| **类型** | 节点故障 |
| **Actor** | 监控系统、全局元数据服务、平台工程师 |
| **前置条件** | 集群中 1 个节点故障下线；剩余节点需要承担该节点的数据副本或流量 |
| **触发动作** | 节点故障自动恢复流程完成后，监控系统更新容量视图 |
| **预期行为** | 监控系统在 5 分钟内更新可用容量视图；重新计算剩余节点的安全边际；若剩余容量不足，立即触发 CRITICAL 告警 |
| **验证方法** | kill 1 个节点后：5 分钟内监控面板可用容量减少量 = 该节点原容量；若剩余节点平均利用率 > 80%，CRITICAL 告警在 5 分钟内触发 |

#### Scenario 5.3.4: 运维操作 — 管理员手动触发容量报告

| 字段 | 内容 |
|------|------|
| **类型** | 运维操作 |
| **Actor** | 平台工程师、缓存服务实例、监控系统 |
| **前置条件** | 平台工程师需要评审当前缓存成本结构 |
| **触发动作** | 通过运维 CLI 触发即时容量与成本报告 |
| **预期行为** | 报告在 30s 内生成；包含：各层当前使用量/总量/增长率、命中率按业务标签拆分、预估月度存储成本（按 DRAM $/GB 和 SSD $/GB 计算）、Top 10 占用 prefix 列表 |
| **验证方法** | 触发报告后：生成时间 < 30s；报告中数据与实时监控差异 < 5%；Top 10 prefix 总占用量与实际扫描结果差异 < 3% |

## 非功能需求量化

以下维度定义了系统必须满足的量化目标及相应的验收方法。所有目标值均基于标杆研究中的生产级数据和我方 ADR 决策推导而来。

---

### 1. 性能 (Performance)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **TTFT (cache hit)** | P50 降低 ≥ 50% vs 无缓存基线；P99 降低 ≥ 33% | 使用标准 workload（ prefix 重叠率 50% ），采集 1 万条请求，对比有/无缓存的 TTFT 分布 |
| **TTFT (cache miss)** | 增量 < 15% vs 无缓存基线 | 强制所有请求 miss，测量 TTFT 增量；P99 增量 < 15% |
| **TPOT 影响** | 开启 offloading 后 TPOT 增量 < 5% | 对比基准和开启异步存储/load 后的 TPOT；P99 增量 < 5% |
| **加载延迟** | HBM→GPU < 1ms; DRAM→GPU P99 < 10ms; SSD→GPU P99 < 100ms; Remote→GPU P99 < 50ms (RDMA) / < 200ms (TCP) | 使用微基准测试，裸测各层传输延迟分布 |
| **吞吐上限** | 单机 DRAM 传输吞吐 ≥ 40 GB/s; SSD 顺序读吞吐 ≥ SSD 标称带宽的 70% | 使用 fio / 自定义 bench 测试最大可持续带宽 |

---

### 2. 可用性 (Availability)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **服务可用性** | 缓存服务本身 99.9%（单节点模式）/ 99.99%（分布式 HA 模式） | 通过 30 天连续运行或故障注入测试统计 down time；分布式模式下采用 3 副本 + 元数据 HA |
| **故障降级率** | 单节点/单盘故障时，100% 请求可降级为 miss 并正常完成推理 | 故障注入测试：kill 节点 / 拔盘 / 网络分区，连续发送 1 万请求，0 失败 |
| **故障感知时间** | 节点无心跳到系统标记为故障 < 15s | 注入 kill -9，测量从故障发生到路由表更新的时间 |
| **恢复时间目标 (RTO)** | 单节点故障后，副本恢复或降级完成 < 5 分钟 | 测量从故障检测到请求恢复基线成功率 100% 的时间 |
| **数据恢复点目标 (RPO)** | 已确认写入的数据 0 丢失；未确认写入允许丢失 | 故障注入后校验已确认数据 100% 可查询 |

---

### 3. 容量 (Capacity)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **单机缓存上限** | 支持管理 ≥ 20TB 的本地 SSD 缓存；≥ 512GB 的本地 DRAM 缓存 | 在测试节点上配置最大容量，运行 7 天无性能衰减 |
| **分布式缓存上限** | 10 节点集群支持 ≥ 100TB 聚合缓存 | 写入 100TB 等效数据，验证命中率、查询延迟稳定在目标范围内 |
| **元数据内存上限** | 单机索引内存占用 < 5GB（支持 10K 活跃序列） | 使用内存分析工具（如 massif）测量 Radix Tree + Chunk-Hash 内存；极端 workload 下不 OOM |
| **容量预警** | 利用率达到 85% 前至少提前 72 小时发出告警 | 模拟写入陡增，验证告警触发时间 |

---

### 4. 扩展性 (Scalability)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **节点扩展** | 新增节点在 60s 内开始承担 ≥ 5% 写入流量；10 分钟内达到负载均衡 | 向运行中的集群加入新节点，监控流量分配曲线 |
| **查询吞吐量** | 单机缓存服务实例支持 ≥ 10,000 QPS 的 prefix 查询 | 使用压测工具（如 wrk / custom loadgen）测量查询吞吐 |
| **线性扩展比** | 分布式模式下，3 节点相对单节点的有效查询吞吐 ≥ 2.5x | 对比 1 节点和 3 节点在相同 workload 下的总吞吐 |
| **元数据扩展** | 全局元数据服务支持 ≥ 100 个缓存实例的心跳和查询 | 模拟 100 个实例持续心跳 + 查询，元数据服务 CPU < 70% |

---

### 5. 一致性 (Consistency)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **索引一致性** | 允许最终一致性；不一致窗口 < 100ms | 在高并发写入场景下，测量从写入完成到查询 100% 命中所需的稳定时间 |
| **数据完整性** | 已确认写入的数据 100% 通过 checksum 校验 | 对全部存储数据执行后台扫描，校验失败率 = 0% |
| **并发安全** | 多引擎并发读写同一 prefix 时，返回数据 100% 一致 | 并发读取测试 + checksum 比对；0 不一致 |
| **跨节点一致性** | 网络分区恢复后，各节点元数据视图在 30s 内达成一致 | 注入分区后恢复，比较各节点状态与全局视图一致性 |

---

### 6. 安全性 (Security)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **访问控制** | 支持基于 token/RBAC 的 API 访问控制；未授权请求 100% 拒绝 | 安全测试：使用无效/缺失凭证访问管理 API，返回 401/403 |
| **传输加密** | 跨节点 TCP 传输支持 TLS 1.3；RDMA 路径支持 IPSec（可选） | 抓包验证：TCP 流量无法被明文解码 |
| **数据隔离** | 多租户场景下，不同租户的缓存数据不可互相访问 | 跨租户访问测试：租户 A 的引擎尝试加载租户 B 的 prefix，100% 返回 miss 或权限错误 |
| **审计日志** | 所有管理操作（配置变更、数据清理、扩缩容）记录审计日志，保留 ≥ 30 天 | 检查审计日志覆盖率和保留策略；模拟操作后 100% 可查 |

---

### 7. 可观测性 (Observability)

| 子项 | 目标值 | 验收方法 |
|------|--------|----------|
| **指标覆盖** | ≥ 50 个 Prometheus 指标，覆盖延迟、吞吐、命中率、错误率、资源利用率 5 大维度 | 检查 /metrics 端点输出的指标数量和分类 |
| **指标采集开销** | /metrics 响应对主流程 CPU 占用 < 3%；响应时间 P99 < 50ms | 压测期间采集 CPU 和 /metrics 延迟 |
| **Trace 完整性** | 1% 采样下，端到端 trace 的 span 缺失率 < 5% | 采集 1000 条 trace，统计缺失 span 的比例 |
| **告警时效** | 关键故障（节点 down、数据损坏、容量满）在 1 分钟内触发告警 | 故障注入后测量告警触发时间 |
| **日志可检索** | 所有 ERROR 级别日志结构化输出并支持按 trace_id/request_id 关联检索 | 注入已知错误，验证日志中包含 trace_id 且可被检索 |
