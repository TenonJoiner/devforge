## Feature 价值论证

### F1: 数据分层存储（P0）

#### 痛点

KV Cache 数据热度分布不均——最新生成的块热度高，历史块迅速冷却：
- 推理引擎 HBM 满载后，新请求的 KV Cache 只能存入 DRAM 或 SSD
- 没有分层策略时，冷数据占用 DRAM，热数据被驱逐到 SSD
- 分层不合理时，热点数据在 SSD 层，P99 延迟远超预期

标杆数据：
- MoonCake 分层策略：GPU→CPU→磁盘三层，验证了分层有效性
- LMCache 后端支持：GPU→CPU→Redis/本地/分布式存储，粒度可配置

#### 当前方案不足

- **分层策略静态**：无法根据实际访问模式动态调整分层
- **跨节点分层缺失**：无法感知其他节点的缓存状态，导致重复缓存
- **降级路径不完善**：故障时无法优雅降级到更慢层

#### 用户价值

按热度自动在 HBM/DRAM/SSD 间迁移数据，**平衡性能与成本**：
- 热数据在 DRAM/Pmem 层，P99 读取延迟 ≤ 200 μs
- 冷数据下沉到 SSD，节省 DRAM 成本
- 访问模式变化时自动重新分层

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 热数据 DRAM 层占比 | ≥ 90%（识别为热数据的块在 DRAM 层） | 压测：构造热度分布，验证分层结果 |
| 分层决策延迟 | ≤ 10 ms/块（分层决策计算） | 基准测试：模拟访问模式，测量决策延迟 |
| 迁移吞吐 | ≥ 100 MB/s/节点（层间迁移） | 基准测试：触发迁移，测量吞吐 |
| 迁移对业务影响 | 迁移期间读取延迟增加 ≤ 10% | 对比测试：迁移前/中/后读取延迟分布 |
| 降级路径 | L1 故障时自动降级到 L2，延迟增加到 ≤ 2ms P99 | 故障演练：DRAM 层故障，测量降级延迟 |

### F2: 数据冗余与自愈（P0）

#### 痛点

分布式存储环境中，磁盘故障、节点故障、网络分区不可避免：
- 单副本时，任何节点故障都可能导致数据丢失
- 多副本时，故障恢复速度直接影响系统可用性
- Ceph 的 11 个 9 耐久性是行业标准（多站点纠删码配置），但实现复杂

标杆数据：
- Ceph 生产案例：3000+ 节点集群，1 EB+ 总容量，故障自愈机制成熟
- MinIO：多站点配置 11 个 9（跨数据中心副本）；本地 3 副本配置约 6 个 9

> **耐久性澄清**：6 个 9（99.9999%）对应年数据丢失率 ≤ 0.0001%，是本地 3 副本配置的合理目标。11 个 9 是跨站点多副本（纠删码）的行业顶级目标，不适用于本地 3 副本。

#### 当前方案不足

- **MoonCake/LMCache 缺失持久化**：仅提供缓存语义，故障后数据丢失
- **自愈速度慢**：故障后重建时间长，期间可用性降低
- **无一致性验证**：重建过程中可能出现数据损坏未被发现

#### 用户价值

多副本保护，磁盘/节点故障自动重建，**保证数据不丢**：
- 3 副本配置，容忍同时 1 个节点故障
- 自动故障检测和重建，RTO ≤ 5 分钟（单节点）
- 数据完整性端到端校验（CRC/SHA）

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 数据耐久性 | ≥ 99.9999%（6 个 9，3 副本配置） | 计算推导：年故障率假设 + 数学验证，区分本地 3 副本（6 个 9）与跨站点多副本（11 个 9） |
| 故障容忍 | 容忍同时 1 个节点故障（≥ 5 节点集群） | 故障演练：kill -9 节点进程 |
| RTO（热数据优先级恢复） | ≤ 1 分钟（最热 1GB 块） | 故障演练：节点故障，测量最热块从剩余副本恢复时间 |
| 重建吞吐量 | ≥ 3.3 GB/s（聚合，多节点并行） | 故障演练：单节点故障，测量集群聚合重建吞吐 |
| 重建期间可用性 | 读取服务不中断（从剩余副本读取） | 故障演练：重建期间读取请求正常 |
| 数据完整性 | 全链路 CRC-32C 校验，数据损坏检测率 100% | 故障演练：注入数据损坏，验证检测 |

### F3: 数据压缩（P1）

#### 痛点

KV Cache 数据具有高度重复性——LLM 输出的 token 分布具有局部相关性：
- 未压缩时，KV Cache 存储成本高（KV Cache 压缩潜力 30-50%）
- 长上下文（128K+ tokens）场景，存储容量需求成倍增长
- SSD 层存储成本是 DRAM 的 1/10，压缩可显著降低 SSD 成本

#### 当前方案不足

- **压缩率与延迟矛盾**：高压缩率算法（如 zstd）计算开销大，增加延迟
- **压缩粒度不匹配**：通用压缩算法无法感知 KV Cache 的结构
- **在线/离线压缩选择**：在线压缩影响写入延迟，离线压缩浪费存储

#### 用户价值

对 KV Cache 数据进行压缩存储，**降低存储成本 30-50%**：
- 在线压缩（写入时）：轻微增加写入延迟（P99 ≤ 10%），显著节省存储
- 离线压缩（后台）：不影响写入延迟，周期性地压缩冷数据

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 压缩率 | ≥ 30%（KV Cache 典型数据） | 数据集测试：使用真实推理 trace，测量压缩率 |
| 压缩延迟 | ≤ 5 ms/GB（在线压缩） | 基准测试：压缩 1GB 数据，测量延迟 |
| 解压延迟 | ≤ 2 ms/GB（不影响读取延迟） | 基准测试：解压 1GB 数据，测量延迟 |
| 解压对读取影响 | 解压后读取延迟 P99 增加 ≤ 10% | 对比测试：压缩/非压缩数据读取延迟 |
| 压缩算法可配置 | 支持 zstd（默认）、lz4、gzip | 配置测试：切换压缩算法，验证生效 |

### F4: 快照与克隆（P1）

#### 痛点

ML 推理场景中，数据版本管理需求普遍：
- 模型版本切换时，需要快速恢复历史 KV Cache
- 实验场景需要克隆当前状态进行对比实验
- 租户需要备份/恢复能力防止误操作

#### 当前方案不足

- **MoonCake/LMCache 无快照**：无法保存时间点状态
- **克隆成本高**：全量复制 KV Cache 数据，耗时长、占用空间大
- **快照一致性**：并发写入时的快照一致性难以保证

#### 用户价值

支持 KV Cache 数据的时间点快照和快速克隆，**支撑模型版本回滚**：
- 秒级快照创建（Copy-on-Write 机制）
- 快照占用额外存储 ≤ 10%（变更量小时）
- 克隆操作秒级完成（写时复制，不复制数据）

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 快照创建时间 | ≤ 1 秒（元数据操作，无数据复制） | 基准测试：创建快照，测量时间 |
| 快照存储开销 | 快照后 1 小时内额外存储 ≤ 10%（增量快照） | 压测：创建快照后写入新数据，测量存储增长 |
| 克隆操作时间 | ≤ 1 秒（写时复制元数据） | 基准测试：克隆快照，测量时间 |
| 克隆读取一致性 | 克隆数据与快照创建时刻完全一致 | 数据校验：比对克隆与快照数据 hash |
| 快照数量上限 | 单命名空间 ≥ 100 个快照 | 压测：创建 100 个快照，验证功能 |
| 快照并发一致性 | 并发写入期间创建的快照为时间点一致状态（Copy-on-Write 语义） | 压测：快照创建期间有并发读写，验证快照数据为创建时刻的一致视图 |

### F5: 跨区域复制（P2）

#### 痛点

多数据中心部署场景中，数据需要跨地域同步：
- 主备数据中心切换时，需要备用站点已有数据
- 数据合规要求数据不能仅存储在单一地区
- 灾难恢复需要跨区域的数据副本

#### 当前方案不足

- **无原生跨区域复制**：MoonCake/LMCache 无此能力
- **复制延迟不可控**：WAN 延迟差异大，无法保证一致性
- **复制带宽成本**：跨区域带宽昂贵，需要压缩和增量复制

#### 用户价值

跨数据中心复制 KV Cache 数据，**支撑异地多活和灾备**：
- 异步复制，延迟容忍度大
- 增量复制，最小化带宽消耗
- RPO 可配置（从 1 分钟到 1 小时）

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 复制延迟 | ≤ 1 分钟（WAN RTT ≤ 100ms 时） | 压测：写入后测量跨区域复制完成时间 |
| 复制带宽 | ≤ 原始数据量 30%（增量复制压缩） | 压测：写入 1GB 新数据，测量实际复制流量 |
| 复制冲突处理 | 支持 Last-Write-Wins（默认）和人工仲裁 | 配置测试：并发写入，验证冲突处理 |
| 复制链路故障恢复 | 故障恢复后自动续传，无数据丢失 | 故障演练：WAN 链路中断后恢复 |

## 场景（Scenario）

### S1: 正常写入触发数据分层迁移

**类型**：正常场景
**Feature**：数据分层存储（F1）

#### 前置条件

- 存储集群 3 层配置（L0=HBM，L1=DRAM，L2=SSD）
- 租户配额：L1 层 256GB，L2 层 4TB
- 当前 L1 层使用率 80%

#### 用户旅程

```
T+0ms    推理引擎写入 KV Cache chunk
    ↓
T+0ms    I/O 调度层计算 chunk 热度评分（访问频率 + 最近访问时间）
    ↓
T+5ms    热度评分 = 0.8（高热），决定写入 L1 层
    ↓
T+10ms   L1 层写入完成
    ↓
T+1min   后台调度器扫描 L1 层，发现使用率已达 80%
    ↓
T+1min   调度器识别 L1 层最冷的 10GB 数据
    ↓
T+1min   调度器发起层间迁移：将冷数据从 L1 迁移到 L2
    ↓
T+30s    迁移完成，L1 层使用率降至 75%
```

#### 验收标准

- [ ] 写入时自动分层（热数据 → L1，冷数据 → L2）
- [ ] 分层决策延迟 ≤ 10 ms
- [ ] 后台迁移不阻塞前台写入
- [ ] 迁移后读取路径正确（自动从新层读取）

### S2: 单节点磁盘故障触发自动重建

**类型**：故障场景
**Feature**：数据冗余与自愈（F2）

#### 前置条件

- 存储集群 5 节点，3 副本配置
- 节点 B 上存储了 chunk_id=777 的一个副本（另两个副本在节点 A 和 C）

#### 故障注入

```
故障注入：节点 B 磁盘故障（smartctl -t long 模拟坏道，读取 chunk_id=777 触发 IO Error）
```

#### 系统响应

```
T+0s     存储客户端读取 chunk_id=777 时收到 IO Error
    ↓
T+1s     客户端从节点 A 或 C 读取（故障转移）
    ↓
T+1s     节点 B 的存储进程检测到磁盘故障
    ↓
T+5s     故障确认，元数据服务更新 chunk_id=777 的副本状态
    ↓
T+5s     调度器发起重建：从节点 A 的副本重建到节点 D
    ↓
T+2min   1TB 数据重建完成（限速 10MB/s，避免影响业务）
    ↓
T+2min   chunk_id=777 恢复到 3 副本状态
```

#### 验收标准

- [ ] 故障检测到副本重建启动 ≤ 30 秒
- [ ] 故障期间读取服务不中断（从健康副本读取）
- [ ] 重建完成后数据完整性校验通过
- [ ] 重建期间写入服务正常
- [ ] 重建进度可通过指标监控

### S3: L1 层（DRAM）故障降级到 L2

**类型**：故障场景
**Feature**：数据分层存储（F1）

#### 故障注入

```
故障注入：节点 DRAM 故障（通过 memfd 耗尽内存模拟 OOM）
```

#### 系统响应

```
T+0s     L1 层检测到 DRAM 不可用
    ↓
T+0s     所有 L1 层数据标记为不可读
    ↓
T+0s     读取请求自动降级到 L2 层
    ↓
T+1ms    降级读取完成（从 SSD 层）
    ↓
T+5s     告警：L1 层故障，降级运行
    ↓
T+10s    运维工程师收到告警，开始处理
    ↓
T+1min   DRAM 恢复（或节点替换），L1 层重建
    ↓
T+2min   L1 层恢复正常，读取延迟恢复到 ≤ 200 μs P99
```

#### 验收标准

- [ ] L1 故障时读取自动降级到 L2，延迟增加到 ≤ 2ms P99
- [ ] 降级过程不丢失数据
- [ ] 降级期间写入正常（继续写入 L2 层）
- [ ] L1 恢复后自动切回，数据重新分层

### S4: 数据压缩写入（在线压缩）

**类型**：正常场景
**Feature**：数据压缩（F3）

#### 前置条件

- 压缩功能已启用（算法 zstd-3，默认压缩级别）
- 推理引擎通过 Connector 写入 KV Cache

#### 用户旅程

```
T+0ms    Connector 发送 KV Cache 数据（未压缩，1MB）
    ↓
T+0ms    压缩模块对数据进行 zstd 压缩
    ↓
T+5ms    压缩完成，压缩率 35%（原始 1MB → 压缩后 650KB）
    ↓
T+5ms    压缩数据写入存储层
    ↓
T+10ms   写入确认返回

[读取时]
T+0ms    Connector 请求读取 chunk
    ↓
T+0ms    存储层返回压缩数据
    ↓
T+2ms    解压缩完成（650KB → 1MB）
    ↓
T+2ms    返回解压后数据给推理引擎
```

#### 验收标准

- [ ] 写入时压缩生效，压缩率 ≥ 30%
- [ ] 压缩后存储空间节省 ≥ 30%
- [ ] 压缩延迟 ≤ 5 ms/GB，不显著影响写入延迟
- [ ] 解压延迟 ≤ 2 ms/GB，不显著影响读取延迟
- [ ] 压缩算法可切换（zstd/lz4/gzip）

### S5: 快照创建与回滚

**类型**：运维场景
**Feature**：快照与克隆（F4）

#### 前置条件

- 存在租户的当前 KV Cache 数据
- 租户配额：快照数量上限 100 个

#### 用户旅程

```
T+0s     租户管理员执行创建快照命令
    ↓
T+0s     元数据服务记录快照点（Copy-on-Write）
    ↓
T+1s     快照创建完成（无数据复制）
    ↓
T+0s     ... [后续写入新数据] ...
    ↓
T+0s     租户发现新数据有问题，决定回滚到快照
    ↓
T+0s     管理员执行回滚命令（快照 ID = snap_001）
    ↓
T+1s     元数据服务将 KV Cache 数据回滚到 snap_001 时刻
    ↓
T+1s     回滚完成，数据恢复
```

#### 验收标准

- [ ] 快照创建时间 ≤ 1 秒
- [ ] 快照创建后写入新数据，快照数据不变
- [ ] 回滚后数据与快照时刻完全一致
- [ ] 回滚操作不影响其他快照

#### 验收标准

- [ ] 链路中断期间本地写入正常
- [ ] 链路恢复后自动续传，无数据丢失
- [ ] 续传期间主站点服务不中断

### S7: 存储容量耗尽（紧急驱逐）

**类型**：异常场景
**Feature**：数据分层存储（F1）

#### 触发条件

```
触发条件：所有层（L1+L2）存储容量均达到 95%
```

#### 系统响应

```
T+0ms    写入请求到达
    ↓
T+1ms    存储系统检测到容量即将耗尽（≥ 95%）
    ↓
T+1ms    LRU 驱逐器立即启动，紧急驱逐最冷的 20% 数据
    ↓
T+10ms   驱逐完成，释放 ≥ 10% 存储空间
    ↓
T+10ms   写入请求继续执行
    ↓
T+20ms   告警：存储容量告急，触发紧急驱逐
    ↓
T+1min   运维工程师收到告警，安排扩容
```

#### 验收标准

- [ ] 容量告急时自动触发紧急驱逐
- [ ] 紧急驱逐优先驱逐最冷数据（LRU）
- [ ] 驱逐过程 ≤ 1 秒，不阻塞写入
- [ ] 驱逐有明确的日志和指标记录
- [ ] 告警及时触发（容量 ≥ 85% 触发常规告警，≥ 95% 触发紧急告警）

### S8: 热数据从 DRAM 层读取（正常场景）

**类型**：正常场景
**Feature**：数据分层存储（F1）
**Actor**：推理引擎（外部系统）、推理服务开发者

#### 前置条件

- 存储集群正常运行（3+ 节点）
- KV Cache chunk 已存在于 L1 层（DRAM）
- L1 层使用率 50%

#### 用户旅程

```
T=0      推理引擎请求读取 KV Cache chunk_id=555
    ↓
T=0      元数据服务返回 chunk 位置：L1 层，节点 B
    ↓
T=0      从 L1 层（DRAM）读取数据
    ↓
T=150μs  数据返回（P99 目标 ≤ 200 μs）
    ↓
T=150μs  推理引擎使用数据继续 Decode 阶段
```

#### 验收标准

- [ ] L1 层读取延迟 P99 ≤ 200 μs
- [ ] 读取后热度评分更新（访问频率 +1）
- [ ] 读取不影响数据分层状态（不触发迁移）

### S9: 压缩数据从 SSD 层读取（正常场景）

**类型**：正常场景
**Feature**：数据分层存储（F1）、数据压缩（F3）
**Actor**：推理引擎、推理服务开发者

#### 前置条件

- L1 层（DRAM）使用率达到 95%，部分数据已下沉到 L2
- 压缩功能已启用（算法 zstd-3）
- 冷数据在 L2 层（SSD）以压缩格式存储

#### 用户旅程

```
T=0      推理引擎请求读取 chunk_id=777（位于 L2 层）
    ↓
T=0      存储系统从 L2 层读取压缩数据（650KB）
    ↓
T=2ms    解压缩完成（650KB → 1MB）
    ↓
T=2ms    返回解压后数据
    ↓
T=2ms    推理引擎使用数据继续 Decode 阶段
```

#### 验收标准

- [ ] L2 层读取 + 解压缩总延迟 P99 ≤ 5 ms（含 2ms 解压）
- [ ] 解压后数据与原始数据完全一致（CRC 校验通过）
- [ ] 读取后热度评分更新，可能触发数据升层（L2 → L1）
- [ ] L2 层命中率监控指标正确更新

### S10: 快照创建与读取（正常场景）

**类型**：正常场景
**Feature**：快照与克隆（F4）
**Actor**：推理服务开发者、集群运维工程师

#### 前置条件

- 租户 namespace 下有已存储的 KV Cache 数据
- 快照配额未耗尽（< 100 个）

#### 用户旅程

```
T=0      租户管理员创建快照
    ↓
T=1s     快照创建完成（Copy-on-Write 元数据操作，无数据复制）
    ↓
T=0      管理员列出快照：snap_001（2026-04-21T10:00:00Z）
    ↓
T=0      管理员读取快照数据（克隆）
    ↓
T=1s     克隆创建完成（写时复制元数据）
    ↓
T=1s     克隆数据可读，与快照创建时刻一致
```

#### 验收标准

- [ ] 快照创建时间 ≤ 1 秒
- [ ] 快照列出操作返回正确的创建时间和状态
- [ ] 克隆操作 ≤ 1 秒
- [ ] 克隆数据与快照时刻一致（hash 校验通过）
- [ ] 快照创建后，新写入不影响已有快照
