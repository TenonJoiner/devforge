## Feature: 缓存写入与持久化

**做什么**：将推理引擎生成的 KV Cache 可靠地写入各存储层级，确保在进程正常退出或异步刷盘策略下，数据满足持久化承诺。

**为什么**：写入路径是缓存服务与推理引擎的边界。不可靠的写入会导致 "伪 miss"（数据本已生成但未能复用），直接降低缓存命中率并增加 GPU 计算浪费。

**涉及 Actor**：推理引擎、缓存服务实例、外部持久化存储。

### Scenario: 正常路径 — 写入成功并立即可查询

**Actor**：推理引擎、缓存服务实例

**前置条件**：引擎完成 prefill；系统配置写入到 DRAM（立即持久化）或 SSD（异步刷盘）。

**触发动作**：Worker 调用存储接口提交 KV Cache 数据。

**预期行为**：数据写入完成后，索引在 100ms 内更新；后续请求对该 prefix 的查询返回命中；写入确认到可查询的延迟 < 100ms。

**验证方法**：写入完成后立即（0ms/50ms/100ms/200ms 间隔）发起查询：100ms 时命中率 ≥ 99%；200ms 时命中率 100%；0ms 时允许 miss。

### Scenario: 数据损坏 — 写入后校验发现不一致，触发重写

**Actor**：缓存服务实例、平台工程师

**前置条件**：写入过程中发生位翻转或存储介质异常；系统配置了写入后校验策略。

**触发动作**：缓存服务在写入完成后执行 CRC64 校验。

**预期行为**：校验失败时，该写入标记为失败；系统尝试最多 1 次重写；若重写仍失败，丢弃该数据并向监控上报 `write_checksum_mismatch` 事件；不将损坏数据加入索引。

**验证方法**：使用 LD_PRELOAD 或 fault injection 使 5% 的写入产生位翻转：100% 损坏数据被检测；0% 损坏数据进入索引；告警事件在 5s 内触发。

### Scenario: 资源耗尽 — 写入时缓冲区满，执行流控

**Actor**：推理引擎、缓存服务实例

**前置条件**：突发高并发写入导致异步刷盘缓冲区达到容量上限（如 1GB）。

**触发动作**：引擎以峰值 2 倍于常态的速率持续提交存储请求。

**预期行为**：缓冲区满时，新写入在 100ms 内返回 `store_backpressure`；引擎收到信号后可选择跳过存储或等待；系统不崩溃、不丢已确认的数据；缓冲区水位在 30s 内恢复正常。

**验证方法**：注入 2x 峰值写入持续 60s：`store_backpressure` 返回率 > 80%；0 次 OOM 或 panic；已确认写入的数据 100% 可查询；60s 后缓冲区利用率 < 70%。

### Scenario: 节点故障 — 写入过程中节点宕机，数据完整性可恢复

**Actor**：缓存服务实例、集群编排器

**前置条件**：写入到 SSD 层的过程正在进行；系统采用 write-ahead log（WAL）作为写入路径的持久化技术；此时节点突然断电或进程被 kill -9。

**触发动作**：在写入中途注入 kill -9。

**预期行为**：节点重启后，系统执行 WAL 回放或文件系统一致性检查（fsck）；已完整写入的 chunk 保留；半写 chunk 被识别并从索引中清除；fsck 扫描时间 < 30s；扫描完成后无效索引条目 < 0.1%。

**验证方法**：注入 10 次随机 kill：重启后 fsck 扫描；半写文件数量 = 被删除数量；fsck 运行时间 < 30s；无效索引条目率 < 0.1%；系统进行中的请求可正常调度。

## Feature: 缓存淘汰与分层迁移

**做什么**：当高速缓存层（HBM/DRAM）空间不足时，依据热度、访问时间和容量策略，将冷数据降级到更低速层级或永久淘汰，为新数据腾出空间。

**为什么**：缓存容量永远小于数据总量。没有有效的淘汰策略，缓存会变成 "只写不读" 的存储黑洞，命中率迅速 decay 到接近 0。

**涉及 Actor**：缓存服务实例、平台工程师、AI 应用开发者。

### Scenario: 正常路径 — 冷数据按 LRU 从 DRAM 淘汰到 SSD

**Actor**：缓存服务实例

**前置条件**：DRAM 层利用率达到 85% 阈值；存在超过 10 分钟未被访问的数据。

**触发动作**：后台淘汰线程扫描并选择最冷数据块。

**预期行为**：冷数据按 chunk 粒度异步写入 SSD；DRAM 空间在 60s 内释放；淘汰期间前台查询延迟增加 < 10%；淘汰到 SSD 的数据仍可通过查询加载。

**验证方法**：保持 DRAM 利用率在 80%-90%：后台淘汰速率 ≥ 写入速率的 90%；P99 查询延迟 < 正常值的 110%；被淘汰数据 100% 仍可从 SSD 命中。

### Scenario: 并发冲突 — 淘汰和读取同时发生

**Actor**：推理引擎、缓存服务实例

**前置条件**：某 chunk 正在被后台淘汰到 SSD；此时引擎恰好发起对该 chunk 的读取。

**触发动作**：高并发读取与后台淘汰重叠。

**预期行为**：读取优先：若 DRAM 中数据尚未删除，直接返回；若已删除但 SSD 写入已完成，从 SSD 加载；读取与淘汰不互相阻塞；无数据丢失或重复删除。

**验证方法**：在持续淘汰任务中以 100 QPS 读取冷热混合数据：读取成功率 100%；0 次数据找不到（not found）；0 次重复释放导致的 double-free / 崩溃。

### Scenario: 资源耗尽 — 所有层级空间均达上限

**Actor**：缓存服务实例、平台工程师

**前置条件**：HBM、DRAM、SSD 全部达到配置上限（如 95%）。

**触发动作**：引擎继续产生新 KV Cache 并请求存储。

**预期行为**：系统启动强制淘汰：按 TTL 和 LRU 组合策略删除最旧数据，直到空间释放 5% 以上；若仍不足，拒绝新存储请求；优先保证已有数据的读取可用性；强制淘汰期间读取成功率 >= 99%，miss 增量 < 10%。

**验证方法**：人为填满所有层至 95%，继续注入请求：系统强制淘汰速率 > 0；拒绝率稳定后 < 20%（即大部分新存储仍可通过释放旧空间完成）；读取成功率 >= 99%；miss 增量 < 10%（对比强制淘汰前基线）；0 次 OOM 或崩溃。

### Scenario: 运维操作 — 管理员调整淘汰策略

**Actor**：平台工程师、缓存服务实例

**前置条件**：当前策略为 LRU；管理员决定切换为 TTL-优先策略。

**触发动作**：通过运维 API 或配置文件更新策略参数。

**预期行为**：策略变更在 30s 内对所有实例生效；新写入按新策略执行淘汰；已有数据在下一次淘汰扫描时按新策略重新评估；策略切换期间服务不中断。

**验证方法**：切换策略后 30s 内，采集淘汰日志：新淘汰决策 100% 符合新策略；前台请求成功率 100%；P99 延迟波动 < 15%。

## Feature: 缓存数据清理与失效

**做什么**：主动识别并清理过期、损坏、或业务上不再需要的 KV Cache 数据，防止存储空间被死数据永久占用。

**为什么**：KV Cache 与请求上下文强相关。当模型版本更新、业务 session 超时、或数据被显式删除时，对应的缓存应立即失效，否则会造成命中率虚高、存储浪费，甚至加载已不兼容的数据。

**涉及 Actor**：缓存服务实例、全局元数据服务、平台工程师、AI 应用开发者。

### Scenario: 正常路径 — 过期数据自动清理

**Actor**：缓存服务实例

**前置条件**：某 prefix 设置了 TTL = 1 小时；超过 TTL 后未被访问。

**触发动作**：后台清理扫描任务检测到 TTL 过期。

**预期行为**：过期数据在 TTL 后最多延迟 10 分钟被删除；索引同步移除对应条目；存储空间释放；后续查询正常返回 miss。

**验证方法**：写入 1000 个带 1h TTL 的数据块，等待 1h10m：过期数据删除率 100%；索引中 0 个过期条目；存储空间恢复至写入前水平的 ±5%。

### Scenario: 数据损坏 — 发现损坏数据块，批量标记失效

**Actor**：缓存服务实例、平台工程师

**前置条件**：后台完整性扫描发现某批次数据存在系统性校验失败（如写入批次使用了错误的 dtype）。

**触发动作**：平台工程师通过运维 CLI 触发按前缀模式或时间范围的批量失效操作。

**预期行为**：指定范围内的数据在 60s 内被标记为 invalid；索引更新以排除这些条目；已加载到引擎内存中的数据不受影响（引擎自行管理生命周期）；后续查询返回 miss。

**验证方法**：触发批量失效后 60s 内：目标范围数据 100% 不可查询；索引扫描确认 0 个残留条目；非目标范围数据查询成功率 100%。

### Scenario: 网络分区 — 清理命令在分区期间下发

**Actor**：平台工程师、缓存服务实例、全局元数据服务

**前置条件**：管理员向全局下发清理命令；部分节点与元数据服务处于网络分区状态。

**触发动作**：分区期间命令无法到达部分节点。

**预期行为**：可达节点立即执行清理；分区节点在恢复后 30s 内（与 overview.md 定义的最终一致性窗口一致）通过状态同步补执行遗漏命令；最终所有节点一致完成清理；无重复删除或遗漏；在分区恢复后到清理完成前的窗口期内，若引擎查询到尚未清理的数据，系统返回 "stale" 标记或直接按 miss 处理。

**验证方法**：注入 20s 分区后恢复：分区恢复后 30s 内，全部节点目标数据删除率 100%；无 "already deleted" 错误日志（重复删除）；元数据服务与各节点状态一致；窗口期内返回 stale 标记的比例 100%。

### Scenario: 升级/回滚 — 升级期间清理任务暂停与恢复

**Actor**：平台工程师、缓存服务实例

**前置条件**：集群正在执行滚动升级；升级前存在定期清理任务。

**触发动作**：升级脚本自动暂停清理任务。

**预期行为**：升级期间清理任务暂停执行；升级完成后 60s 内自动恢复；暂停期间不产生因并发清理导致的升级失败；若回滚，清理任务在回滚完成后恢复正常调度。

**验证方法**：升级期间监控清理任务运行状态：0 次清理任务执行；升级完成后 60s 内清理任务恢复；回滚场景下同样验证；0 次因清理导致的数据访问失败。

### Scenario: 服务端失效与引擎内存状态不一致

**Actor**：推理引擎、缓存服务实例

**前置条件**：某 prefix 已被引擎加载到 GPU 内存中；随后服务端通过批量失效或 TTL 过期将该 prefix 标记为无效。

**触发动作**：引擎在后续多轮对话中尝试扩展该 prefix（追加 token）。

**预期行为**：引擎在扩展请求中携带 `generation_version` 或 `cache_timestamp`；服务端校验版本，若发现数据已失效，返回 `invalid_version` 错误；引擎收到错误后，将该 prefix 及后续 token 回退为未计算状态，执行本地重算；请求最终成功完成，不返回错误结果。

**验证方法**：模拟 100 次 "加载后服务端失效再扩展" 场景：100% 的扩展请求被服务端正确识别为版本无效；0 次引擎使用过期缓存数据；回退重算成功率 100%；请求最终结果正确性 100%。

## 本域非功能需求

| 子项 | 目标值 |
|------|--------|
| 写入确认到可查询 | 延迟 < 100ms |
| 数据完整性 | 已确认写入的数据 100% 通过 checksum 校验 |
| 淘汰延迟影响 | 淘汰期间前台查询延迟增加 < 10% |
| 强制淘汰可用性 | 所有层 95% 时 0 次 OOM 或崩溃；拒绝率稳定后 < 20%；读取成功率 >= 99%，miss 增量 < 10% |
| TTL 清理延迟 | 过期数据在 TTL 后最多延迟 10 分钟被删除 |
| 批量失效时效 | 指定范围数据在 60s 内标记为 invalid |
| 积压恢复 | 刷盘缓冲区满后 30s 内水位恢复正常 |
| 节点故障恢复 | kill -9 后 fsck 时间 \u003c 30s；无效索引条目 \u003c 0.1% |
