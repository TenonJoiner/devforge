## Feature: 节点加入与退出

**做什么**：支持新的缓存服务实例动态加入集群以扩展容量，或将旧实例优雅移出集群以进行维护或缩容，期间不影响引擎的缓存读写。

**为什么**：缓存系统需要随着业务负载增长而扩展。节点的动态加入和退出是弹性部署和滚动升级的基础能力。

**涉及 Actor**：缓存服务实例、全局元数据服务、集群编排器、平台工程师。

### Scenario: 正常路径 — 新节点成功加入集群

**Actor**：新缓存服务实例、全局元数据服务、集群编排器

**前置条件**：新节点已完成容器/进程启动；网络可达；配置与集群一致。

**触发动作**：新实例向全局元数据服务注册；开始发送心跳。

**预期行为**：注册在 10s 内完成；新节点开始接收新的 KV Cache 写入和部分数据副本；全局元数据服务更新路由表；已有请求不中断。

**验证方法**：向 2 节点集群添加第 3 节点：注册时间 < 10s；新节点在 60s 内承担 ≥ 5% 的写入流量；0 个正在运行的请求失败。

### Scenario: 运维操作 — 管理员手动缩容

**Actor**：平台工程师、缓存服务实例、全局元数据服务、集群编排器

**前置条件**：集群有 3 个节点；需要缩容到 2 节点。

**触发动作**：管理员向目标节点发送缩容命令；全局元数据服务首先检查目标节点上所有数据块的可用副本数是否 >= 1。

**预期行为**：若存在可用副本数 < 1 的数据块，缩容命令被阻塞，并上报 `scale_in_blocked_due_to_replica_risk` 告警；若所有数据块均满足副本安全条件，目标节点进入 DRAINING 状态，停止接收新写入；已有数据在后台迁移到剩余节点；迁移完成后节点优雅退出；缩容期间 P99 TTFT 增加 < 15%。

**验证方法**：缩容过程中持续注入请求：0 个请求失败；缩容命令在存在唯一副本时被 100% 阻塞；目标节点在 5 分钟内完成数据迁移并退出；缩容期间 P99 TTFT < 基准值的 115%。

### Scenario: 节点故障 — 节点异常退出，数据通过副本恢复

**Actor**：缓存服务实例、全局元数据服务、集群编排器

**前置条件**：3 节点集群；每个数据块有 2 副本；其中一个节点突然宕机。

**触发动作**：kill -9 目标节点进程。

**预期行为**：全局元数据服务在 3 个心跳周期（默认 10s）内感知故障；丢失副本的数据触发后台复制，从存活副本补全到新的目标节点；读取请求自动路由到存活副本；0 个请求因节点故障而失败。

**验证方法**：注入节点故障后连续请求 1000 次：0 失败；故障感知时间 < 15s；副本恢复在 5 分钟内完成；恢复期间 P99 延迟 < 基准值的 120%。

### Scenario: 并发冲突 — 节点加入期间大量请求并发

**Actor**：推理引擎、新缓存服务实例、全局元数据服务

**前置条件**：业务高峰期；新节点正在加入；大量读写请求同时发生。

**触发动作**：新节点注册和路由表更新与大量请求并发。

**预期行为**：路由表更新采用原子切换，不存在请求被路由到不存在的节点；新节点逐步接收流量（不瞬间承接大量请求）；无请求丢失或路由错误。

**验证方法**：以峰值 2x QPS 注入请求的同时加入新节点：0 个 5xx/路由错误；新节点流量 ramp-up 曲线平滑（60s 内达到稳定负载）；P99 延迟波动 < 20%。

## Feature: 实例故障检测与自动恢复

**做什么**：通过心跳、健康探针和异常指标监控，及时发现缓存服务实例的故障，并触发自动的数据重平衡、副本修复和路由切换。

**为什么**：分布式系统中节点故障是常态。若故障不能被快速检测和恢复，会导致缓存命中率下降、请求堆积，甚至影响推理引擎的稳定性。

**涉及 Actor**：缓存服务实例、全局元数据服务、集群编排器、监控系统、平台工程师。

### Scenario: 正常路径 — 心跳检测正常

**Actor**：缓存服务实例、全局元数据服务

**前置条件**：所有实例正常运行；网络稳定。

**触发动作**：实例每 3s 向全局元数据服务发送心跳。

**预期行为**：全局元数据服务 100% 收到心跳；无 false positive 故障判定；集群状态显示为 HEALTHY。

**验证方法**：运行 24 小时：心跳丢失率 = 0%；false positive 告警数 = 0；元数据服务状态与实例实际状态一致性 100%。

### Scenario: 节点故障 — 存储实例心跳丢失，触发数据重平衡

**Actor**：缓存服务实例、全局元数据服务、集群编排器

**前置条件**：某缓存服务实例因硬件故障或 OOM 崩溃；心跳连续 3 次未到达。

**触发动作**：全局元数据服务在 10s 内标记该实例为 UNHEALTHY。

**预期行为**：该实例从路由表中移除；需要修复的副本在后台启动复制；受影响 prefix 的查询自动降级或路由到存活副本；5 分钟内副本健康度恢复至故障前水平。

**验证方法**：随机 kill 实例：故障检测时间 < 12s（3 次心跳 + 余量）；路由表更新 < 15s；副本修复启动 < 30s；1000 次连续请求 0 失败。

### Scenario: 网络分区 — 脑裂场景下的仲裁机制

**Actor**：缓存服务实例、全局元数据服务、集群编排器

**前置条件**：集群发生网络分区，部分实例与元数据服务主节点失联，但与从节点保持连接。

**触发动作**：iptables 模拟分区，将集群分为 A、B 两个分区。

**预期行为**：拥有多数节点的分区继续提供服务；少数分区实例进入只读或降级模式；不存在双主同时写入同一数据； minority 分区在脑裂期间接受的未确认写入，在恢复后 100% 被回滚或标记为无效；优先采用 majority 分区版本作为协调基准；分区恢复后，数据通过版本号或时间戳自动协调，无冲突。

**验证方法**：注入 30s 脑裂：0 次双主写入同一 prefix 的情况（通过全局唯一 write ID 或元数据锁验证）；少数分区请求降级率 > 80%； minority 分区写入在恢复后 100% 被回滚或标记无效；恢复后 60s 内所有节点状态一致。

### Scenario: 升级/回滚 — 滚动升级中的优雅退出

**Actor**：缓存服务实例、全局元数据服务、集群编排器、平台工程师

**前置条件**：集群执行滚动升级；当前轮次需要升级某实例。

**触发动作**：编排器向实例发送 SIGTERM，实例进入优雅关闭流程。

**预期行为**：实例完成进行中的数据传输；将未持久化的数据刷盘；向元数据服务注销；正在服务的请求在 5s 内完成或安全转移；实例正常退出，无强制 kill。

**验证方法**：升级期间持续注入请求：优雅退出时间 < 30s；0 次强制 kill（除异常情况外）；升级期间请求失败率 < 0.1%；回滚时同样验证。

## Feature: 负载均衡与容量调度

**做什么**：在分布式部署下，将新的 KV Cache 写入和加载请求合理分布到各个缓存服务实例上，避免单节点过载；同时根据各节点的容量利用率动态调整流量分配。

**为什么**：不均衡的负载会导致热点节点成为瓶颈，出现高延迟、高拒绝率，甚至触发 OOM 或磁盘满错误。负载均衡是扩展性的前提。

**涉及 Actor**：全局元数据服务、缓存服务实例、推理引擎、平台工程师。

### Scenario: 正常路径 — 请求均匀分布到各节点

**Actor**：推理引擎、全局元数据服务、缓存服务实例

**前置条件**：3 节点集群；各节点容量和性能一致；写入请求无特定偏斜。

**触发动作**：引擎以 1000 QPS 持续发送存储请求。

**预期行为**：各节点写入量标准差 < 平均值的 20%；查询延迟 P99 在各节点间差异 < 30%；无单节点利用率超过 85%。

**验证方法**：运行 30 分钟：各节点存储量差异 < 20%；CPU/DRAM/SSD 利用率最高值 < 85%；P99 延迟差异 < 30%。

### Scenario: 资源耗尽 — 某节点容量达 90%，流量切换

**Actor**：全局元数据服务、缓存服务实例、推理引擎

**前置条件**：节点 A 的 DRAM 或 SSD 利用率达到 90%；其他节点仍有空闲容量。

**触发动作**：持续写入导致节点 A 触达容量警戒线。

**预期行为**：全局元数据服务在 10s 内将节点 A 的新写入权重降为 0；查询流量仍可从 A 读取（若数据存在）；新数据导向其他节点；节点 A 水位在 5 分钟内下降或稳定。

**验证方法**：人为填满节点 A 至 90%：30s 内新写入到 A 的流量降为 0；读取流量不变；其他节点分担写入后最大利用率 < 90%；系统 0 次因容量不足拒绝存储。

### Scenario: 数据损坏 — 检测到某节点数据异常，停止调度

**Actor**：全局元数据服务、缓存服务实例、平台工程师

**前置条件**：后台完整性扫描发现节点 B 上某批次数据存在系统性校验失败。

**触发动作**：扫描任务上报 `node_data_corruption` 事件。

**预期行为**：全局元数据服务在 5s 内将节点 B 的写入和查询权重降为 0；若数据有副本，读取自动路由到副本节点；节点 B 进入维护模式，等待管理员介入；扫描任务上报后 5s 内，100% 的新请求不再路由到异常节点；扫描前已发出的少量请求由 data-lifecycle 的逐层校验兜底。

**验证方法**：模拟扫描发现 10% 数据损坏：5s 内节点 B 停止接收新请求；已有副本的查询 100% 路由到存活节点；扫描上报后 5s 内新请求 100% 不再路由到节点 B；维护模式状态在监控中可见。

### Scenario: 并发冲突 — 调度决策与高并发写入冲突

**Actor**：推理引擎、全局元数据服务、缓存服务实例

**前置条件**：全局元数据服务正在更新某节点的路由权重；引擎同时大量写入该节点。

**触发动作**：路由表更新与突发写入并发。

**预期行为**：路由表更新采用原子操作，旧决策和新决策不混合使用；引擎收到的路由决策要么完全基于旧表，要么完全基于新表；不存在部分 block 路由到旧节点、部分路由到新节点导致的写分裂。

**验证方法**：在路由表更新瞬间注入 1000 次写入：0 次写分裂（即同一请求的所有 block 都路由到同一节点集合）；0 次因路由不一致导致的数据丢失；P99 调度延迟 < 10ms。

## 本域非功能需求

| 子项 | 目标值 |
|------|--------|
| 节点注册时效 | 新节点注册 < 10s；60s 内承担 ≥ 5% 写入流量 |
| 缩容影响 | 缩容期间 P99 TTFT 增加 < 15%；目标节点 5 分钟内完成迁移并退出 |
| 故障感知时间 | 心跳连续 3 次未到达后 < 15s 标记为故障 |
| 故障恢复 RTO | 单节点故障后请求恢复 100% 基线成功率 < 5 分钟 |
| 脑裂仲裁 | 0 次双主写入同一 prefix；少数分区降级率 > 80%；恢复后 60s 内状态一致 |
| 负载均衡 | 3 节点写入量标准差 < 平均值的 20%；P99 延迟差异 < 30% |
| 容量告警响应 | 节点达 90% 后 30s 内新写入流量降为 0；其他节点最大利用率 < 90% |
| 路由原子性 | 路由切换瞬间 0 次写分裂；P99 调度延迟 < 10ms |
| 集群可用性 | 单节点模式 99.9%；分布式 HA 模式 99.99% |
| 元数据服务扩展 | 支持 ≥ 100 个缓存实例的心跳和查询；元数据服务 CPU < 70% |
| 线性扩展比 | 3 节点相对单节点有效查询吞吐 ≥ 2.5x |
