#### HIGH（2 个，共 6 分）

**H1. Feature 1 成功指标与 Scenario 验收标准口径不一致，存在自我矛盾**
- **所属维度**：Feature 价值 / 验收标准一致性
- **位置**：Feature 1「成功指标」第 1 条 vs Scenario #3 验收标准
- **问题描述**：成功指标声明"指标端点响应延迟 P99 < 200ms（高负载场景）"，但 Scenario #3 的验收标准也是"端点响应延迟 P99 < 200ms，写入吞吐下降 ≤ 1%"。同一句话同时出现在"成功指标"和"Scenario 验收标准"中，导致成功指标没有独立衡量价值，只是重复了验收标准。成功指标应回答"上线后业务结果如何"，而非重复技术验收门槛。
- **修复建议**：将成功指标第 1 条改为业务结果导向的指标，例如"运维团队平均故障感知时间从 >15min 降至 <5min"或"监控告警面板可用率 ≥ 99.9%"，与 Scenario 的技术验收标准形成互补而非重复。

**H2. Feature 2 脱敏 Scenario 的验收标准依赖未确认白名单，存在验收不可执行风险**
- **所属维度**：验收标准可测试性
- **位置**：Feature 2 Scenario #7 验收标准
- **问题描述**：验收标准声明"凭证类字段脱敏覆盖率 100%（按已确认白名单）"，但 Q2 明确标注"敏感字段脱敏白名单/黑名单尚未与安全团队对齐"且计划澄清时间为 2026-05-22。这意味着在 Q2 澄清前，Scenario #7 的验收标准无法执行——"已确认白名单"不存在。虽然括号备注诚实，但验收标准本身不应依赖尚未存在的前置条件。
- **修复建议**：将验收标准拆分为两个阶段：(a) 框架验收——"脱敏引擎对配置白名单中的字段 100% 触发脱敏"；(b) 内容验收——"白名单确认后补充具体字段覆盖率验证"。或在验收标准中改为"脱敏引擎对配置文件中声明的敏感字段模式 100% 匹配并脱敏，白名单内容由 Q2 跟进确认"。

#### MEDIUM（4 个，共 4 分）

**M1. Feature 3（P2 远期规划）Scenario 数量 = 8 个，超出前瞻阶段合理范围**
- **所属维度**：需求蔓延 / Feature 范围控制
- **位置**：Feature 3 Scenario 清单
- **问题描述**：Feature 3 明确标注为"P2 远期规划，当前迭代不进入开发"，且声明"验收门槛在前瞻阶段相应下调"。但 Scenario 数量 = 8 个，与 Feature 1（9 个）和 Feature 2（10 个）几乎持平。对于不进入开发的远期 Feature，8 个 Scenario 的前瞻设计投入产出比偏低，且部分 Scenario（如 #6 采样率自动调整）涉及较复杂的自适应逻辑，可能诱导特性级阶段过度设计。
- **修复建议**：将 Feature 3 Scenario 精简至 5-6 个（保留核心链路：trace 传播、trace 生成、查询关联、弱网补传、非法 context 处理），将 #6（自适应采样）和 #7（成功率监控）合并或降级为"已知待澄清问题"，明确标注"正式开发前需重新评审"。

**M2. 故障模式覆盖矩阵中"—"备注存在推卸责任式表述**
- **所属维度**：场景覆盖完整性
- **位置**：故障模式覆盖矩阵 Feature 1 和 Feature 2 的"—"备注
- **问题描述**：Feature 1 对"网络分区"和"并发冲突"标记为"—"，备注"由监控采集系统域承担"和"端点为只读拉取，无并发写入冲突"。Feature 2 对"节点故障"和"网络分区"标记为"—"，备注"由监控采集系统域处理"和"跨网络传输不在本 Feature 范围"。这些备注虽然逻辑成立，但属于"这不是我的问题"式表述。从用户视角看，可观测性数据（指标/日志）在网络分区或节点故障时的可用性是一个真实问题，即使具体实现由其他域承担，本域也应定义跨域协作的验收标准（如"网络分区期间本地指标缓存不丢失"）。
- **修复建议**：在"—"备注中补充跨域协作验收标准引用，例如"网络分区期间本地 /metrics 缓存持续可用，分区恢复后数据完整性由监控采集系统域覆盖，本域要求缓存丢失率 = 0"；或在已知待澄清问题中增加一项跨域验收对齐问题。

**M3. 非功能需求中"可观测性自身的可观测"条目定义模糊**
- **所属维度**：非功能需求量化
- **位置**：非功能需求表格「可观测性」维度
- **问题描述**：目标值描述为"log_dropped_total / cardinality_overflow / sampling_rate / disk_pressure 等自描述指标全部暴露"，但"全部暴露"缺乏量化标准——是要求这些指标本身也出现在 /metrics 端点中？还是要求它们有独立的监控链路？"等"字暗示列表不完整，验收时可能产生争议。
- **修复建议**：明确列出必须暴露的自描述指标清单（而非用"等"），并给出量化验收标准，例如"自描述指标 ≥ 8 项，全部出现在 /metrics 端点中，命名符合 OpenMetrics 规范"。

**M4. Feature 2 缺少"日志轮转/归档"相关 Scenario**
- **所属维度**：场景覆盖完整性 / 运维场景
- **位置**：Feature 2 Scenario 清单
- **问题描述**：Feature 2 覆盖了日志输出、动态调级、背压、脱敏、磁盘压力降级、升级回滚，但缺少日志文件轮转（log rotation）和归档清理的 Scenario。在边缘节点磁盘受限（product-spec.md 约束）场景下，日志无限增长会导致磁盘耗尽，即使 Scenario #6 有磁盘压力降级，也缺乏主动轮转/归档机制的定义。Q1 提到"日志保留期"待澄清，但保留期策略的执行（自动删除过期日志）没有对应 Scenario。
- **修复建议**：增加一个 Scenario，例如"节点日志文件达到配置大小上限或保留期到期时，系统自动执行日志轮转/清理"，验收标准"轮转不丢失已写入日志条目，清理后磁盘空间释放，操作被审计日志记录"。
