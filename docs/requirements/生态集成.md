## Feature: 提供 Grafana 原生数据源插件

### Feature 价值论证

**用户痛点**：数据分析师在 Grafana 中只能通过通用 HTTP 数据源手写 JSON 查询，无法利用本系统特有的标签索引、降采样和流式分页能力，大时间范围查询频繁超时；同时缺乏多租户上下文传递机制，跨部门看板存在越权风险。

**现有方案不足**：通用插件缺乏查询构建器、无 metric/tag 自动补全、不支持服务端降采样委托与流式分页；用户自行封装中间层会重复实现鉴权透传与限流，且无法跟随 Grafana 大版本节奏。

**本 Feature 价值**：原生插件提供可视化查询构建器、metric/tag 自动补全、服务端降采样委托、租户上下文透传、查询级限流保护，并通过 Grafana marketplace 分发以保证签名与版本兼容。

**成功指标**：
- Grafana 面板从添加数据源到首次出图的端到端时间 ≤ 3 分钟（覆盖 80% 用户首次体验）
- 单面板 10 条时序线、7 天范围查询 P95 响应时间 < 2s（基线数据集 5 亿点）
- 插件在 Grafana 最近 2 个 LTS 大版本上的兼容通过率 100%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据分析师在 Grafana 中添加数据源插件，连接验证成功 | 连接验证响应时间 < 1s，失败时返回包含错误码和原因的明确提示 | 自动化 |
| 2 | 数据分析师通过查询构建器选择 metric、筛选 tag、设置降采样粒度 | 查询 P95 响应时间 < 2s，数据点数与服务端直查一致 | 自动化 |
| 3 | 数据分析师在变量面板输入 tag key 前缀，插件返回匹配列表 | 自动补全响应时间 < 500ms，匹配结果与服务端一致 | 自动化 |
| 4 | 可视化仪表盘系统同时刷新 20 个 panel 的 dashboard | 全部 panel 渲染完成时间 < 10s，无查询被拒绝 | 自动化 |
| 5 | 数据分析师携带过期 Token 或错误凭证发起查询（鉴权失败/Token 过期） | 插件返回 401/403 并提示重新登录，不泄露内部错误堆栈 | 自动化 |
| 6 | 单租户单查询扫描点数超过限流阈值（大查询限流） | 插件返回明确限流错误码与建议降采样窗口，不击穿后端 | 自动化 |
| 7 | 多租户环境下，租户 A 用户尝试查询租户 B 的 metric（多租户隔离） | 跨租户查询被拒绝并写入审计日志，租户 A 仅能看到自身命名空间 | 自动化 |
| 8 | 查询时间范围内存在数据空洞 | 空洞区间显示为断线而非零值填充，UI 标注空洞起止时间 | 手动 |

| 10 | 插件版本升级后回滚至前一稳定版（升级回滚） | 回滚后已保存的 dashboard 和数据源配置全部可用，无需手工修复 | 自动化 |

## Feature: 提供 Kafka Connect Sink Connector

### Feature 价值论证

**用户痛点**：企业已有 Kafka 数据管道，需要将数据落入本系统；自行开发 consumer 承担 offset 管理、重试、schema 映射与 task 在线扩缩容等工程负担，且发布到 Confluent Hub 需通过严格认证。

**现有方案不足**：自研 consumer 需独立处理 exactly-once 语义、背压控制、故障恢复、schema 演进与扩缩容；通用 JDBC sink 不支持时序数据的 metric/tag/field 三元结构。

**本 Feature 价值**：标准 Kafka Connect Sink Connector 利用 Connect 框架能力，用户通过 JSON 配置完成数据管道；支持 schema 演进、task 在线扩缩容，并通过 Confluent Hub 分发以保证签名与版本兼容。

**成功指标**：
- 稳态写入吞吐 ≥ 200k points/s（3 task 并行，单条 5 字段）
- Consumer lag 稳态 < 10s（生产速率 ≤ 吞吐上限的 80% 时）
- task 在线扩缩容时数据丢失率 = 0、重复率 ≤ 0.01%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 消息队列/流平台通过 Connect REST API 提交 connector 配置，connector 启动消费 | connector 状态变为 RUNNING，首批数据 30s 内写入可查 | 自动化 |
| 2 | 以 200k points/s 持续推送数据，connector 以 3 task 并行消费 | 稳态 consumer lag < 10s，无 task 重启 | 自动化 |
| 3 | 应用开发者通过 SMT 配置将嵌套 JSON 映射为 metric + tags + fields | 映射后写入数据与配置完全一致，字段缺失时返回明确错误 | 自动化 |
| 4 | Connector 运行中某个 task 崩溃，Connect 框架自动重启 | 重启后从上次 offset 继续，数据丢失率 = 0、重复率 ≤ 0.01% | 自动化 |
| 5 | 本系统集群短暂不可用 60s，connector 写入失败触发重试（网络分区） | 集群恢复后 120s 内追平积压，无数据丢失 | 自动化 |
| 6 | Kafka 消息包含不支持的数据类型或字段（数据损坏/脏数据） | 不支持的消息路由至死信队列并记录原因，后续消息不阻塞 | 自动化 |
| 7 | 上游 schema 演进新增字段（向后兼容变更） | 新字段自动作为 field 写入，无需重启 connector，旧消息仍按原映射处理 | 自动化 |
| 8 | 运维值班通过 REST API 在线将 task 数从 3 调整为 6（扩缩容） | 扩容期间无消息丢失，调整后 60s 内吞吐线性提升 | 自动化 |
| 9 | 集群 broker 节点故障切换（节点故障） | connector 自动连接到新 leader，写入恢复时间 < 30s | 自动化 |
| 10 | Connector 版本升级后回滚到前一版本（升级回滚） | 已有配置兼容旧版本，offset 不丢失，回滚后业务无感知 | 自动化 |

## Feature: 支持 MQTT Broker 直接写入集成

### Feature 价值论证

**用户痛点**：IoT 场景中数万设备通过 MQTT Broker 上报数据，需要额外部署桥接服务转写入时序数据库，桥接层成为单点故障并增加端到端延迟；同时 TLS 协商失败、订阅鉴权失败等安全异常缺乏统一处置。

**现有方案不足**：自建桥接服务需处理重连、QoS、解析、批量写入、背压、TLS 与凭证轮换，工程成本高且难以保证 QoS 0/1 语义在重启场景下的明确边界。

**本 Feature 价值**：系统内置 MQTT Client 直接订阅外部 Broker，省去桥接层；统一处理 TLS 协商失败、订阅鉴权失败、QoS 重启语义，凭证加密存储并写入审计日志。

**成功指标**：
- MQTT 消息发布到数据可查延迟 P99 < 100ms（本地 Broker，单节点 50k msg/s）
- Broker 断连后自动重连成功率 100%（短连接抖动 ≤ 5 分钟内）
- TLS 协商成功率 ≥ 99.9%（凭证有效期内）

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 应用开发者配置 Broker 地址、Topic 模式和 payload 解析规则，系统自动订阅 | 订阅建立后首条消息 100ms 内写入可查 | 自动化 |
| 2 | IoT 设备群以 50k msg/s 持续发布到订阅 Topic | 写入吞吐匹配发布速率，端到端延迟 P99 < 100ms | 自动化 |
| 3 | 应用开发者配置 Topic 通配符订阅，不同设备 ID 自动映射为不同 tag | 各设备数据按 tag 隔离可查，命名冲突时返回明确错误 | 自动化 |
| 4 | MQTT Broker 网络中断 30s 后恢复，系统自动重连（网络分区） | 重连在 5s 内完成，QoS 1 离线消息全部接收并写入 | 自动化 |
| 5 | MQTT 消息 payload 解析失败（脏数据/规则不匹配） | 解析失败消息计入错误计数器并采样进入死信主题，不影响后续正常消息 | 自动化 |
| 6 | 系统写入能力饱和，MQTT 消息到达速率超过处理能力（资源耗尽） | 对 MQTT Client 施加背压，QoS 1 消息无丢失，QoS 0 丢弃量被计数告警 | 自动化 |
| 7 | TLS 握手失败（证书过期/不被信任/版本不匹配） | 连接立即终止并写入审计日志，运维值班 1 分钟内收到告警 | 自动化 |
| 8 | 凭证有效但订阅鉴权失败（ACL 拒绝特定 Topic） | 系统不重试该 Topic 订阅，记录审计事件并向运维值班告警 | 自动化 |
| 9 | 系统重启后对 QoS 0 消息的语义边界（运维操作） | 重启窗口内 QoS 0 消息允许丢失但被计数，QoS 1 消息按 session 持久化恢复 | 自动化 |

## Feature: 兼容 OpenTSDB HTTP 写入协议（社区贡献级）

### Feature 价值论证

**用户痛点**：部分存量监控系统已对接 OpenTSDB `/api/put` 接口，迁移需逐一修改采集器配置；这部分用户群规模尚未量化，但社区调研显示存在迁移诉求。

**现有方案不足**：部署协议转换代理增加额外组件、运维负担和延迟；且代理无法享受本系统的写入批处理与 schema 校验。

**本 Feature 价值**：原生兼容 `/api/put` 端点，存量采集器只需修改目标地址即可完成迁移。**本 Feature 定位为社区贡献级**：目标用户群待 2026-06 调研后量化（见已知待澄清问题 #1），验收标准采用宽松基线，不承诺与原生接口同等性能与功能完备性；若调研结论为低需求，则将 Feature 从核心 Feature 降级为 example 维护模式。

**成功指标（宽松基线）**：
- 存量 OpenTSDB 采集器仅修改目标地址即可成功写入（覆盖 OpenTSDB 2.4 文档中 `/api/put` 必选字段）
- 兼容端点写入吞吐 ≥ 原生接口的 60%（社区基线，非性能 SLA）
- 通过 OpenTSDB 官方测试套件中 `/api/put` 用例的 80%

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 数据采集代理向 `/api/put` 发送单条 JSON 数据点 | 返回 HTTP 204，数据 1s 内可查 | 自动化 |
| 2 | 数据采集代理批量发送 1000 条数据点 | 返回 HTTP 204，吞吐 ≥ 原生接口的 60% | 自动化 |
| 3 | 请求带 `?summary` 或 `?details` 查询参数 | 响应体格式与 OpenTSDB 2.4 文档一致 | 自动化 |
| 4 | 数据点中 metric 名包含本系统不支持的字符（脏数据） | 不合规数据点返回错误码与位置，合规数据点正常写入 | 自动化 |
| 5 | 时间戳为秒级精度（OpenTSDB 默认） | 秒级时间戳自动转换为系统内部精度，不丢失语义 | 自动化 |
| 6 | 请求使用 OpenTSDB 特有但未实现的字段 | 未实现字段被忽略，响应通过 warning 字段提示 | 自动化 |
| 7 | 大批量请求触发后端限流（资源耗尽） | 返回 HTTP 429 与 Retry-After，采集器重试后成功 | 自动化 |
