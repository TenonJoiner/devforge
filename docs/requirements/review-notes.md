## 总体置信度调整

原评估: 高 (85%)
调整后: 中 (70%)
理由:
1. 多个关键非功能指标（如 TTFT P99 降低 >=33%、DRAM->GPU P99<10ms）缺乏 workload 特征和 baseline 验证假设，直接在需求层面锁定数值，设计阶段可能发现不可行。
2. 诸多 Scenario 的验收标准依赖未声明的隐性假设（如 vLLM V1 的 layer-wise pipeline 可用性、workload 的时间局部性、RDMA 网络环境成熟度）。
3. Actor 边界存在重叠，尤其是"平台工程师 / MLOps"与"系统管理员"在故障诊断、升级操作、策略调整等场景的职责未清晰切分，易导致实施歧义。
4. 多特性域之间存在未明确处理的冲突（如最终一致性窗口 <100ms 与跨节点数据迁移/复制的时间要求、1% trace 采样与故障定位时效的矛盾）。
5. 对标杆（vLLM 官方 prefix caching、各大厂商的 KV cache offloading 方案）的借鉴有选择性偏差：大量引用收益数据（TTFT 降低 2x~22x），但未分析其前置条件、失败场景和不可迁移到本系统的约束。

## 按特性域质疑

### cache-offloading

**质疑1**: Scenario "正常路径 — 前缀命中，TTFT 显著降低"的验收标准过于理想化，缺少 worst-case 约束。
- 理由: 文档 stating "P50 TTFT < 基准值/2，P99 TTFT < 基准值/1.5"，但未定义 baseline workload（序列长度分布、batch size、模型大小）。若 baseline 已包含高命中率场景，33% 的 P99 降低可能不现实。同时未限定 prefix 长度下限（若 prefix 仅 10 tokens，加载 overhead 可能抵消收益）。
- 建议: 补充 baseline workload 定义（如平均序列长度 >= 2K、prefix 命中长度 >= 512 tokens），并增加一个 "short-prefix hit" 的验收场景，明确短 prefix 命中时不劣化 TTFT。

**质疑2**: Scenario "并发冲突 — 多个引擎实例同时读取同一缓存块"假设了介质带宽是瓶颈，但未考虑跨节点 RDMA 场景下的带宽抖动和 QP 数量限制。
- 理由: 文档要求 "峰值带宽 >= 介质理论带宽的 80%"，但在分布式 RDMA 环境中，单个 prefix 可能被多个节点并发读取，瓶颈在网络 QP 或 PCIe 带宽，而非介质本身。该 Scenario 将 Actor 限定在 "缓存服务实例"，忽略了 RDMA NIC / 交换机的资源竞争。
- 建议: 拆分并发冲突场景为 "同节点并发读取" 和 "跨节点并发读取"，后者增加 RDMA QP 耗尽和网络带宽竞争的验收标准（如 P99 延迟 < 正常值的 150%）。

**质疑3**: Feature "跨层/跨节点 KV Cache 迁移" 的 "热数据从 SSD 晋升到 DRAM" Scenario 缺少对晋升失败 fallback 的验收。
- 理由: 文档仅描述了 "DRAM 层有可用空间" 时的正常晋升，但若 DRAM 在晋升过程中被新写入占满，晋升任务可能中断或阻塞。未定义此时系统的行为（暂停晋升？降级读取到 SSD？取消晋升？）。
- 建议: 增加 "晋升目标层空间不足" 的 Scenario，明确晋升任务的可中断性和 fallback 行为，并给出验收标准（如晋升任务失败不阻塞前台读取）。

**质疑4**: 缺少 "缓存块粒度" 的定义，导致多个 Scenario 的验收无法量化。
- 理由: 文档多次提到 "chunk 粒度"、"2GB 缓存块"、"按 chunk 粒度异步写入"，但未在需求层面定义 chunk 大小是否固定、是否可配置、chunk 大小对 prefix 命中精度的影响。若 chunk 是 1MB 还是 1GB，将直接影响 TTFT 和命中率。
- 建议: 在 overview.md 或 cache-offloading.md 中增加 "KV Cache 数据分块策略" Feature，明确 chunk 大小、对齐方式、动态调整策略及验收标准。

### data-lifecycle

**质疑1**: Scenario "节点故障 — 写入过程中节点宕机，数据完整性可恢复" 的验收标准 "半写 chunk 被识别并删除" 过于乐观，未涉及缓存服务重启时间窗口。
- 理由: 文档要求 kill -9 后 "节点重启后，系统执行写日志或文件系统一致性检查"。对于 C 语言实现的存储系统，若未使用 WAL 或 copy-on-write 文件系统，kill -9 后半写数据的识别在工程上并不 trivial。验收标准 "索引中 0 个条目指向无效文件" 在首次实现时几乎不可能达到 100%。
- 建议: 将该验收标准降级为 "索引中指向无效文件的条目 < 0.1%"，或增加对 fsck 运行时间的约束（如 < 30s）。同时明确写入路径使用的是 WAL、日志结构存储还是直接文件写入。

**质疑2**: Feature "缓存淘汰与分层迁移" 的 "所有层级空间均达上限" Scenario 中，"拒绝率稳定后 < 20%" 的指标与 "优先保证已有数据的读取可用性" 存在潜在冲突。
- 理由: 若系统启动强制淘汰，按 TTL+LRU 删除最旧数据，但最旧数据可能正在被读取（尤其在长上下文场景中，旧 prefix 的长尾读取很常见）。强制淘汰会造成这部分请求的 miss，降低命中率。20% 的拒绝率是 "新存储请求" 的，而非 "读取 miss 率"，需求未对读取影响做约束。
- 建议: 补充强制淘汰场景下的 "读取命中率变化" 验收标准（如强制淘汰期间读取成功率 >= 99%，miss 增量 < 10%）。

**质疑3**: Scenario "网络分区 — 清理命令在分区期间下发" 的验收要求 "最终所有节点一致完成清理；无重复删除或遗漏" 依赖于最终一致性，但未给出不一致窗口的上限。
- 理由: overview.md 中定义 "跨节点一致性: 网络分区恢复后，各节点元数据视图 30s 内达成一致"，但 data-lifecycle.md 的清理场景未引用该指标。若清理命令在分区期间下发，某节点在恢复后 30s 才执行清理，则在这 30s 内该节点仍可能响应已过期的 prefix 查询，导致引擎加载到不应存在的数据。
- 建议: 明确清理命令下发后的不一致窗口要求（如 <= 30s），并增加一个验收点：分区恢复后到清理完成前，被清理数据仍可被查询但返回结果必须被标记为 "stale" 或直接 miss。

**质疑4**: Feature "缓存数据清理与失效" 缺少对 "引擎已加载到 GPU 内存中的 KV Cache 在服务端被失效后" 的行为定义。
- 理由: 文档在 "批量标记失效" Scenario 中提到 "已加载到引擎内存中的数据不受影响（引擎自行管理生命周期）"，但未说明如果引擎在后续 decode 中需要扩展该 prefix（如多轮对话）时，如何处理服务端已失效的缓存。这会导致引擎和服务端状态不一致。
- 建议: 增加 "服务端失效与引擎内存状态不一致" Scenario，明确引擎需要支持的版本校验或 invalidate-on-read 机制。

### engine-integration

**质疑1**: Feature "Worker 侧异步加载/存储流水线" 高度依赖 "引擎支持 layer-wise pipeline（如 vLLM V1）"，但该假设未在需求中验证或声明为前提条件。
- 理由: 文档中 "层级别流水线隐藏加载延迟" Scenario 的前置条件直接假设 "引擎支持 layer-wise pipeline（如 vLLM V1）"，但 vLLM V0 与 V1 的 API 差异巨大，若 MVP 深度适配 vLLM 时 V1 尚未稳定或 API 变更，整个 Feature 的收益将无法兑现。这是整个 engine-integration 域的核心风险。
- 建议: 将该假设提升为显性假设，增加一个 "引擎不支持 layer-wise pipeline" 的 fallback Scenario（如退化为 block-wise 或同步加载），并定义此时 TTFT 的验收上限。

**质疑2**: Scenario "调度器侧 Prefix 匹配查询" 的 "元数据服务不可用，本地降级查询" 存在职责边界模糊问题。
- 理由: 文档定义 "缓存服务实例 fallback 到本地索引查询"，但在分布式模式下，本地索引可能并不完整（全局 prefix 可能存储在其他节点）。若本地索引 miss，系统返回 (0, false)，调度器会分配全量本地 block，这可能导致后续 Worker 侧发现其他节点实际有该 prefix 时产生资源浪费或调度不一致。
- 建议: 明确 "本地降级查询" 的适用边界：是仅用于读写分离的本地缓存实例，还是任意分布式节点都持有完整的本地索引副本？若不是完整副本，应增加一个 "本地降级导致后续全局 hit 不一致" 的验收约束。

**质疑3**: Feature "推理引擎连接与握手" 的兼容性 hash 未包含 "model parallel 策略" 或 "pipeline parallel 策略"。
- 理由: 只交换了 vllm_version, model_name, cache_dtype, block_size，但不同 TP/PP 配置下，KV Cache 的 layout 和 shard 方式可能不同。若引擎以 TP=4 运行，而缓存服务中的数据来自 TP=2 的引擎，即使 block_size 相同，数据也可能不兼容。
- 建议: 将 tp_size, pp_size 纳入 compatibility hash，或在 handshake capability 中显式交换 parallel config。

**质疑4**: Scenario "资源耗尽 — Pipeline 缓冲内存不足" 的 "系统减少 pipeline 并发深度" 缺少明确的控制 Actor 和触发条件。
- 理由: 文档说 "系统减少 pipeline 并发深度"，但未定义是谁检测、谁决策、谁通知。是缓存服务实例告诉引擎降低深度，还是引擎自行根据 GPU 显存预算调整？若是后者，这与 "缓存服务实例" 作为 Actor 的边界冲突。
- 建议: 明确该 Scenario 的控制闭环：引擎根据 GPU 显存自行调整，无需缓存服务参与。调整验收标准为 "引擎在显存预算不足时自主降低预取深度，0 次 OOM"。

### cluster-management

**质疑1**: Feature "节点加入与退出" 的缩容 Scenario 假设 "目标节点上的数据在其他节点有副本或可降级"，但未说明缩容前的副本检查机制。
- 理由: 文档要求 "迁移完成后节点优雅退出"，但如果目标节点上存在某些数据块的唯一副本（因副本修复尚未完成或其他原因），直接迁移并退出会导致这部分数据永久丢失。
- 建议: 增加缩容前置条件的验证步骤：全局元数据服务必须确认目标节点上所有数据的可用副本数 >= 副本数 - 1（或至少 >=1），否则缩容命令应被阻塞并告警。

**质疑2**: Feature "实例故障检测与自动恢复" 的脑裂仲裁 Scenario 过于简化，缺少对 minority 分区中 "已确认写入" 数据的处理。
- 理由: 文档说 "少数分区实例进入只读或降级模式；分区恢复后，数据通过版本号或时间戳自动协调，无冲突"，但未说明若 minority 分区在脑裂期间接受了写入（尽管需求说降级率 >80%，不是 100%），恢复后这些写入如何处理。是丢弃？合并？这对一致性模型有重大影响。
- 建议: 明确脑裂恢复后的数据协调策略：优先采用 majority 分区版本、minority 分区的未确认写入丢弃。并增加一个验收点：minority 分区在脑裂期间的写入在恢复后 100% 被回滚或标记为无效。

**质疑3**: Feature "负载均衡与容量调度" 的 "请求均匀分布" Scenario 的验收标准 "各节点写入量标准差 < 平均值的 20%" 在 workload 有天然热点 prefix 时几乎不可达。
- 理由: LLM 推理场景中，某些 prefix（如系统 prompt、热门 RAG query）天然是热点。若按 prefix hash 分布，热点 prefix 必然集中到某个节点，20% 的标准差要求过于严格。这会导致设计阶段被迫选择不合适的分布策略（如按请求随机分发，牺牲命中率）。
- 建议: 将验收标准调整为 "排除 Top 5% 热点 prefix 后，各节点写入量标准差 < 20%"，或增加 "热点 prefix 允许副本扩散到多个节点" 的机制约束。

**质疑4**: Scenario "数据损坏 — 检测到某节点数据异常，停止调度" 的 "0 次引擎加载到损坏数据" 与 data-lifecycle 域的逐层校验存在重复，但未明确触发时机。
- 理由: cluster-management 从调度侧停止写入/查询权重，而 data-lifecycle 从存储侧做 checksum。但如果损坏扫描任务的上报存在延迟（如 5s 内），在这 5s 内引擎仍可能从该节点读取。两个域的验收都号称 "0 次加载损坏数据"，这是不现实的。
- 建议: 将 cluster-management 的验收标准修改为 "扫描上报后 5s 内新请求 100% 不再路由到异常节点"，接受扫描前已发出的少量请求可能加载到损坏数据（由 data-lifecycle 的校验兜底）。

### observability

**质疑1**: Feature "故障诊断与追踪" 的 "1% trace 采样" 与 "故障定位时效 < 1 分钟" 存在根本矛盾。
- 理由: 1% 采样意味着 99% 的故障请求没有 trace。若某数据损坏事件恰好落在未采样的 99% 中，平台工程师将无法依赖 trace 在 1 分钟内定位。文档未提供基于日志或 metrics 的备用定位路径。
- 建议: 增加 "基于结构化日志的非采样故障定位" Scenario，要求日志中包含足够的上下文（request_id, node_id, chunk_id, layer），使得即使未命中 trace，也能在 1 分钟内定位。或者将关键错误（checksum mismatch）提升为 100% trace 采样事件。

**质疑2**: Feature "容量与成本分析" 的成本模型过于简化，缺少多租户和云厂商定价差异的考虑。
- 理由: 文档提到 "按 DRAM $/GB 和 SSD $/GB 计算"，但实际云环境中，DRAM 成本可能与实例类型强相关（如 HBM 节点 vs 普通 CPU 节点），SSD 成本还涉及 IOPS 计费、网络出口费等。单一定价模型会导致成本分析失真。
- 建议: 将成本模型设计为可配置参数（单价、附加费用率），并明确该 Feature 的 MVP 范围：先支持单租户、单一云厂商的近似成本估算，后续迭代支持多租户和精细化计费。

**质疑3**: Scenario "并发冲突 — 高并发下 Trace 采样不丢关键事件" 的验收标准 "关键 span 缺失率 = 0%" 在工程上不现实。
- 理由: 在高并发场景下，trace buffer 或 collector 端的背压必然导致少量 span 丢失或截断。要求 "0%" 会给实现团队带来不合理的压力，或者导致为保 trace 而牺牲主流程性能。
- 建议: 将验收标准调整为 "关键 span 缺失率 < 0.1%"，并明确 "关键 span" 的列表（如查询发起、加载完成、错误标记）。

**质疑4**: Feature "全链路性能指标采集" 未包含对推理引擎侧指标的采集要求。
- 理由: 文档只要求缓存服务实例暴露 /metrics，但 TTFT/TPOT 是端到端指标，需要引擎侧配合采集和关联。若引擎未上报 "无缓存 baseline" 的 TTFT，则缓存服务无法独立计算 "降低 >=50%" 的收益。
- 建议: 增加引擎侧指标集成的需求，明确引擎需要暴露的 baseline 指标（如 `vllm_ttft_without_cache`）以及缓存服务侧的对比计算方法。

## 隐性假设识别

| 假设 | 影响 | 验证建议 |
|------|------|----------|
| **假设1**: 目标推理引擎（vLLM）在 MVP 期间提供稳定可用的 layer-wise pipeline API，且 API 不会发生重大变更。 | 若该假设不成立，engine-integration 中 "Pipeline TTFT 收益降低 >=70%" 的核心承诺将无法兑现，导致整个产品收益模型崩塌。 | 在 architectural spike 中验证 vLLM V1 的 `start_load_kv`/`wait_for_layer_load` 等 API 的可用性和稳定性；若不可用，需在 MVP 前明确 fallback 方案。 |
| **假设2**: Workload 具有显著的时间局部性和 prefix 复用特征，命中率可达 30%~80%。 | 若 workload 以短 prompt、一次性请求为主，缓存命中率将远低于预期，TTFT 降低的收益无法覆盖系统复杂度和运维成本。 | 在目标业务场景中采集 7 天真实请求日志，计算 prefix 重合度和长度分布；若命中率 <20%，需重新评估产品定位。 |
| **假设3**: 生产环境中 RDMA (RoCE) 网络稳定可用，且延迟抖动 < 1ms。 | 若部署环境实际只能使用 TCP，或 RDMA 存在频繁的 PFC 风暴/丢包，则 "Remote->GPU P99 < 50ms" 的目标将不现实，分布式模式的可行性存疑。 | 在目标集群上运行 rdma_perftest 和 pertest3，测量 RDMA 读写延迟 P99 和抖动；若不达标，需调整延迟目标或限制分布式模式的适用范围。 |
| **假设4**: 缓存服务实例所在节点的 SSD 具备足够的顺序写带宽和耐久性，能够承受 KV Cache 的高频写入（工作负载写入放大可控）。 | SSD 寿命和写带宽是 data-lifecycle 的经济性关键。若每日写入量 > 数倍 SSD 容量，会导致 SSD 快速磨损或写放大严重，运维成本激增。 | 基于预期 workload 模型计算每日写入总量和 SSD DWPD（Drive Writes Per Day）；选择满足耐久性要求的 SSD 型号，并在验收测试中纳入写放大监控。 |
| **假设5**: 全局元数据服务的查询模式以 prefix 精确匹配或前缀匹配为主，不涉及复杂的模糊搜索或范围查询。 | 若未来需求中出现模糊 prefix 匹配，当前基于 hash 或 trie 的索引设计可能无法扩展，导致元数据服务成为查询瓶颈。 | 在产品级需求中明确禁止模糊/范围查询（除非作为 P2 需求），并在架构设计中为索引扩展预留接口。 |

## 遗漏 Trade-off

| 维度A | 维度B | 当前选择 | 建议补充分析 |
|-------|-------|----------|--------------|
| **最终一致性窗口 (<100ms)** | **跨节点数据迁移时效** | 选择了最终一致性，但未分析迁移任务对这 100ms 窗口的冲击。 | 补充分析：在节点扩容/缩容期间，大量后台迁移会产生大量元数据更新，是否会突破 100ms 一致性窗口？若突破，是放宽一致性窗口还是限制迁移并发度？ |
| **高命中率（数据尽量保留）** | **低延迟淘汰（快速释放空间）** | 各域都倾向于保留数据以获得高命中率，但 data-lifecycle 中的强制淘汰和容量告警说明空间有限。 | 补充一个 Feature：明确 "命中收益评分" 与 "存储成本" 的权衡策略，例如对长序列但低命中频率的 prefix 设置更高的淘汰优先级。 |
| **Trace 完整性（高采样）** | **系统性能（低开销）** | 选择 1% trace 采样，但 observability 域的故障定位时效又要求 1 分钟定位。 | 补充分析：1% 采样 vs 5% 采样 vs 100% 错误采样的性能开销对比；明确关键错误路径（如 checksum mismatch）是否应采用尾部采样或 100% 采样。 |
| **分布式 HA 的可用性 (99.99%)** | **资源成本（副本数）** | 设定了 99.99% 可用性，但未定义副本策略（2 副本？3 副本？）。 | 补充可用性-副本数-成本模型：在当前故障率假设下，需要多少副本才能达到 99.99%？该副本策略对存储成本的增量是多少？ |
| **MVP 深度适配 vLLM** | **扩展性（SGLang / TensorRT-LLM 接口预留）** | 深度绑定 vLLM 的 API 和内部实现，但 overview 提到要预留扩展接口。 | 补充分析：深度适配 vLLM 所需的引擎侵入性修改，与预留通用接口之间的冲突。建议将 "通用接口抽象" 提升到 architecture 设计目标中，避免 MVP 代码过度绑定 vLLM 内部结构。 |

## 风险补充

| 风险 | 可能性 | 影响 | 缓解建议 |
|------|--------|------|----------|
| **风险1: vLLM API 在开发周期内发生 Breaking Change** | 中 (40%) | 高 | engine-integration 域大量验收标准依赖 vLLM layer-wise pipeline。缓解：在需求中增加 "API 兼容性追踪" 机制，要求每两周验证一次目标 vLLM 版本的 API 稳定性；同时设计 block-wise fallback 方案。 |
| **风险2: RDMA 网络在大规模集群（>10 节点）中出现 PFC 风暴或拥塞** | 中 (35%) | 高 | 全局非功能需求中的 "Remote->GPU P99 < 50ms" 在 RoCE 环境下 >10 节点时很难保证。缓解：在架构验证阶段增加 10 节点 RDMA 压力测试；需求中增加 "当 RDMA 延迟持续 >100ms 时自动降级到 TCP" 的显式要求。 |
| **风险3: 强制淘汰策略与长上下文 workload 冲突，导致有效命中率急剧下降** | 中高 (50%) | 高 | data-lifecycle 中的强制淘汰可能淘汰掉长序列但低频访问的 prefix，而这类 prefix 的重新计算成本极高（TTFT 可能长达数十秒）。缓解：在淘汰策略中增加 "序列长度权重"，对长序列设置更低的淘汰优先级；需求中补充 "长上下文命中率" 的独立监控指标。 |

**回应**：接受。已在 observability.md 中新增 "基于结构化日志的非采样故障定位" Scenario；同时明确关键错误（checksum mismatch、timeout、node_down）实施 100% trace 采样（尾部采样），不受全局 1% 采样率限制。

**处理结果**：接受

**状态**：已关闭

### observability-02: 成本模型过于简化，缺少多租户和云厂商差异

**质疑点**：Feature "容量与成本分析" 的成本模型过于简化，缺少多租户和云厂商定价差异的考虑。

**回应**：接受。已在 "容量与成本分析" Feature 中说明成本模型采用可配置参数（DRAM 单价、SSD 单价、附加费用率）；明确 MVP 范围支持单租户、单一云厂商的近似成本估算，多租户和精细化计费列为后续迭代（P2）。

**处理结果**：接受

**状态**：已关闭

### observability-03: "关键 span 缺失率 = 0%" 不现实

**质疑点**：Scenario "并发冲突 — 高并发下 Trace 采样不丢关键事件" 的验收标准 "关键 span 缺失率 = 0%" 在工程上不现实。trace buffer 背压必然导致少量 span 丢失。

**回应**：接受。已将 observability.md 的该验收标准调整为 "关键 span 缺失率 < 0.1%"，并明确 "关键 span" 列表：查询发起、加载完成、错误标记。

**处理结果**：接受

**状态**：已关闭

### observability-04: 未包含引擎侧指标采集要求

**处理结果**：接受

## 修正说明

### 已补充的隐性假设（>= 5 个）

1. **假设 A (engine-integration)**: 引擎支持 layer-wise pipeline（如 vLLM V1）是可选优化路径，而非 MVP 必备前提。若引擎不支持，系统必须能退化为 block-wise 或同步加载，且此时额外 TTFT 不得超过原始全量 prefill 的 50%。
2. **假设 B (data-lifecycle)**: 写入路径采用 write-ahead log（WAL）作为 kill -9 后数据恢复的技术基础。若实际实现改用 copy-on-write 文件系统或日志结构存储，需重新评估 fsck 时间和无效条目率的上限。
3. **假设 C (overview/cache-offloading)**: 缓存服务的 chunk/block 粒度策略遵循 ADR-004 的分层异构粒度决策。HBM 层 16 tokens、DRAM 层 256 tokens、SSD 层 256 tokens（2MB）、Remote 层 65536 tokens。同一 prefix 在不同层之间命中时，可能存在粒度不对齐导致的部分加载 overhead。
4. **假设 D (cluster-management)**: 缩容操作前，全局元数据服务能够完整掌握各数据块的副本分布。若元数据服务本身处于网络分区或不一致状态，缩容命令应被阻塞，而非基于不完整视图执行。
5. **假设 E (observability)**: 关键错误事件（checksum mismatch、timeout、node_down）的 trace 采样不受全局 1% 采样率限制，实施 100% 尾部采样。若 trace collector 在高并发下仍出现背压，关键 span 缺失率允许的上限为 < 0.1%。
6. **假设 F (observability)**: 引擎侧必须暴露 `vllm_ttft_without_cache` 等 baseline 指标，否则缓存服务无法独立计算 "TTFT 降低 >= 50%" 的收益。若引擎未暴露该指标，收益评估将依赖离线对比测试。
7. **假设 G (data-lifecycle)**: 强制淘汰策略以 "读取可用性优先" 为原则。即使所有层级空间均达上限，强制淘汰期间读取成功率仍须 >= 99%，miss 增量 < 10%。

### 已补充的 Trade-off（>= 3 组）

1. **TTFT 收益 vs 引擎兼容性**: 深度绑定 vLLM V1 的 layer-wise pipeline 可获得最佳 TTFT 收益（额外 TTFT 降低 >= 70%），但会牺牲对 vLLM V0 及其他引擎的兼容性。需求选择保留同步降级路径，以兼容性换取收益的部分折损。
2. **数据完整性 vs 恢复现实性**: kill -9 后要求 100% 识别并清除半写数据在工程上过于乐观。需求选择明确使用 WAL 技术，并将验收标准调整为 "无效索引条目 < 0.1%" + "fsck 时间 < 30s"，以现实可达成性换取理论完美性。
3. **负载均衡严格性 vs 热点 prefix 现实**: 严格的 "写入量标准差 < 20%" 在存在天然热点 prefix 的 workload 下几乎不可达。 reviewer 建议引入热点例外或放宽标准。需求保留待讨论状态，倾向于在 MVP 中放宽标准或仅对非热点数据执行严格均衡。
4. **Trace 完整性 vs 系统开销**: 高 trace 采样率（如 5% 或更高）可提升故障定位能力，但会增加主流程 CPU / 网络开销。需求选择 1% 全局采样 + 100% 关键错误尾部采样的混合策略，以定位时效性换取常规开销可控。
5. **分布式 HA 可用性 vs 副本/运维成本**: 99.99% 的 HA 可用性目标需要多副本和自动故障恢复机制支撑，但未在需求中锁定具体副本数。需求将副本策略的细化留给架构设计阶段，以可用性目标换取设计的灵活性。

### 已降低的风险（>= 3 个）

1. **vLLM API Breaking Change 风险**: 通过新增 "引擎不支持 layer-wise pipeline" 的 fallback Scenario，将该风险对产品收益模型的影响从 "高" 降为 "中"。即使 V1 API 不稳定或变更，MVP 仍可基于同步降级路径交付基本功能。
2. **缩容导致数据丢失风险**: 通过增加缩容前副本可用性检查机制（可用副本数 >= 1 方可继续），将缩容操作的数据丢失风险从 "中高" 降为 "低"。
3. **trace 采样导致故障无法定位风险**: 通过引入关键错误 100% 尾部采样和结构化日志定位方案，将 1% 采样与 1 分钟定位时效之间的矛盾风险从 "高" 降为 "中低"。
4. **引擎加载损坏数据风险**: 通过区分 cluster-management 的调度侧拦截（扫描上报后 5s 内 100% 停止路由）和 data-lifecycle 的存储侧校验，将 "0 次加载损坏数据" 的不现实承诺调整为可工程实现的分层防御，降低了验收不可达风险。
5. **节点故障后半写数据不可恢复风险**: 通过明确 WAL 技术选型和放宽验收标准（< 0.1% 无效条目），将 kill -9 后数据恢复的实现风险从 "高" 降为 "中"。
