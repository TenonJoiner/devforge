# Systematic Debugging — 系统化调试

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖了根本问题。

**核心原则**：在尝试修复之前，必须找到根因。只修复症状就是失败。

**违反本流程的字面要求，就是违反调试的精神。**

## 铁律

```
没有根因调查，不得提出修复方案
```

如果未完成第一阶段，你不能提出任何修复。

## 何时使用

用于任何技术问题：
- **测试失败** — 单元测试、集成测试或 CI 流水线中出现的任何测试不通过，包括断言失败、超时、flaky test。
- **生产环境 bug** — 线上报错、监控告警、用户反馈的异常，无论是否可复现都必须先定位再修复。
- **意外行为** — 代码运行结果与预期不符，逻辑看似正确但输出错误，或状态偏离了设计意图。
- **性能问题** — 响应延迟、吞吐量下降、内存泄漏、CPU 飙升等，需先量化再定位瓶颈而非盲目优化。
- **构建失败** — 编译错误、链接失败、依赖缺失、构建脚本异常，排除问题前不要猜测原因。
- **集成问题** — 多组件/服务交互时的接口不匹配、数据格式错误、环境配置差异、版本兼容性问题。

**以下情况尤其要使用：**
- 时间紧迫（紧急情况下更容易产生猜测冲动）
- "只改一下就好了"看起来很明显
- 你已经尝试了多种修复
- 之前的修复没起作用
- 你不完全理解问题

**以下情况也不要跳过：**
- 问题看起来简单（简单 bug 也有根因）
- 你在赶时间（仓促保证返工）
- 要求立即修复（系统化比盲目尝试更快）

## 四个阶段

你必须完成每个阶段才能进入下一阶段。

### 第一阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们经常包含确切的解决方案
   - 完整阅读 stack trace
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**

   **当系统包含多个组件时（客户端 → 协调层 → 存储节点，写入 → WAL → 刷盘 → 复制）：**

   **在提出修复方案之前，先添加诊断日志：**
   ```
   对每个组件边界：
     - 记录进入组件的数据（请求参数、payload 大小、时间戳）
     - 记录离开组件的数据（返回码、写入字节数、耗时）
     - 验证配置传递（超时值、副本数、一致性级别）
     - 检查每层状态（队列深度、锁持有、资源水位）

   运行一次收集证据，显示问题在哪里中断
   然后分析证据，定位失败的组件
   然后调查该特定组件
   ```

   **示例（分布式存储写入路径）：**
   ```c
   /* Layer 1: 客户端请求到达协调层 */
   log_info("client_write: key=%s size=%zu consistency=%d",
            key, size, consistency_level);

   /* Layer 2: 协调层写入本地 WAL */
   log_info("wal_append: lsn=%lu payload_crc=0x%08x", lsn, crc32(data, size));

   /* Layer 3: WAL 刷盘 */
   log_info("wal_flush: lsn=%lu fsync_rc=%d elapsed_ms=%ld",
            lsn, rc, elapsed_ms);

   /* Layer 4: 复制到多数派 */
   log_info("replicate: lsn=%lu acked=%d required=%d timeout_ms=%d",
            lsn, acked_count, quorum_size, timeout_ms);
   ```

   **这会揭示**：哪一层失败了（client → coord ✓, coord → WAL ✓, WAL → disk ✗）

5. **追溯数据流**

   **当错误发生在调用栈深处时：**

   参见本目录下的 `root-cause-tracing.md` 获取完整的反向追溯技术。

   **快速版本：**
   - 坏值从哪里产生？
   - 谁用坏值调用了这个？
   - 持续向上追溯直到找到源头
   - 在源头修复，不在症状处修复

### 第二阶段：模式分析

**修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中找到类似的工作代码
   - 什么能工作，什么坏了？

2. **对照参考实现**
   - 如果实现某种模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作代码和坏代码之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那个不可能有影响"

4. **理解依赖**
   - 这个组件需要什么其他组件？
   - 需要什么设置、配置、环境？
   - 它做了什么假设？

### 第三阶段：假设与验证

**科学方法：**

1. **形成单一假设**
   - 清楚表述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体，不模糊

2. **最小化测试**
   - 做最小的可能变更来验证假设
   - 一次只改变一个变量
   - 不要同时修复多个问题

3. **验证后再继续**
   - 有效？→ 进入第四阶段
   - 无效？→ 形成新假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 做更多研究

### 第四阶段：实施修复

**修复根因，不是症状：**

1. **创建失败测试用例**
   - 最简单的复现
   - 尽可能自动化测试
   - 如果没有框架，用一次性测试脚本
   - 修复前必须有
   - 使用 `devforge-tdd-workflow` skill 编写正确的失败测试

2. **实施单一修复**
   - 针对已识别的根因
   - 一次只改一处
   - 不要"顺便"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 没有其他测试被破坏？
   - 问题确实解决了？

4. **如果修复无效**
   - 停止
   - 计数：你已经尝试了多少修复？
   - 如果 < 3：返回第一阶段，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下文第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 个修复

5. **如果 3+ 修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每个修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每个修复在别处制造新症状

   **停止并质疑根本问题：**
   - 这个模式从根本上是否合理？
   - 我们是否"仅凭惯性坚持"？
   - 应该重构架构还是继续修复症状？

   **在尝试更多修复之前与主人讨论**

   这不是失败的假设——这是错误的架构。

## 危险信号 —— 停止并遵循流程

如果你发现自己在想：
- "先快速修复，以后再调查"
- "试试改 X 看有没有用"
- "同时改多个地方，跑测试"
- "跳过测试，我手动验证"
- "可能是 X，我来修"
- "我不完全理解但也许能工作"
- "模式说要 X 但我要改一下"
- "主要问题是：[列出修复方案但没有调查]"
- 在追溯数据流之前提出解决方案
- **"再试一次修复"（已经试了 2+ 次）**
- **每次修复在不同地方揭示新问题**

**这些都意味着：停止。返回第一阶段。**

**如果 3+ 修复失败：** 质疑架构（见第四阶段第 5 步）

## 主人的信号 —— 说明你做错了

**注意这些引导：**
- "是不是没发生？" —— 你未经验证就假设了
- "能不能给我们看……？" —— 你应该添加证据收集
- "停止猜测" —— 你在不理解的情况下提出修复
- "深入思考" —— 质疑根本问题，不只是症状
- "我们卡住了？"（沮丧）—— 你的方法不起作用

**当你看到这些：** 停止。返回第一阶段。

## 常见辩解

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 bug 也很快。 |
| "紧急情况，没时间走流程" | 系统化调试比猜测尝试更快。 |
| "先试试这个，然后再调查" | 第一次修复就定下了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复不持久。先写测试才能证明。 |
| "同时改多个地方省时间" | 无法隔离什么起作用。会制造新 bug。 |
| "参考太长，我改一下模式" | 部分理解保证出 bug。完整阅读。 |
| "我看到问题了，我来修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 失败 = 架构问题。质疑模式，不要再修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 阅读错误、复现、检查变更、收集证据 | 理解 WHAT 和 WHY |
| **2. 模式** | 找到工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当流程揭示"无根因"

如果系统调查揭示问题确实是环境相关的、时序相关的或外部的：

1. 你已完成流程
2. 记录你调查了什么
3. 实施适当处理（重试、超时、错误信息）
4. 添加监控/日志以便未来调查

**但是：** 95% 的"无根因"案例都是调查不完整。

## 支持技术

这些技术是系统化调试的一部分，在本目录中可用：

- **`root-cause-tracing.md`** —— 沿着调用栈反向追溯找到原始触发点
- **`defense-in-depth.md`** —— 找到根因后在多层添加验证
- **`condition-based-waiting.md`** —— 用条件轮询替代任意超时

**相关 skill：**
- **devforge-tdd-workflow** —— 用于创建失败测试用例（第四阶段第 1 步）

## 真实影响

来自调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时盲目尝试
- 首次修复成功率：95% vs 40%
- 引入新 bug：接近零 vs 常见
