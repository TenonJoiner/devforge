# devforge-arch-extract-topic — 技术主题逆向提取

## 概述

从已有源码逆向提取一个技术主题的深度设计文档。技术主题是按技术关注点组织的横向概念（如"协程调度模型""本地存储引擎""Raft 共识"），跨目录存在，不按代码目录组织。

**核心原则**：
1. **源码 = 行为真相**：代码是唯一权威的事实来源
2. **社区文档 = 参考线索**：可选的社区文档提供概念框架和设计背景，但与代码矛盾时以代码为准
3. **分析密度优先**：深度分析阶段用多 agent 按维度切分并行，确保每个维度读够源码
4. **单次生成单篇文档**：每次执行只产出当前目标 topic 的一篇文档

**与 subsystem/system skill 的关系**：
- topic 是最深层的设计文档，聚焦单个技术主题的"怎么做"和"为什么这样做"
- subsystem 文档为 topic 提供架构上下文（组件拓扑、模块间关系）
- topic 文档不重复 subsystem 已有的上下文描述

## 工作目录约定

- **输入**：当前分支的源码（事实来源） + 可选的社区文档目录 `--doc=<dir>`（参考来源）
- **输出**：`docs/architecture/<subsystem>/<topic>.md`
- **模板**：`skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md`

**输出路径确定**：
- `<subsystem>` 由阶段 3 范围确认时与用户商定
- `<topic>` 文件名由主会话根据主题名生成（小写、连字符分隔，如 `coroutine-scheduler`）

## 启动检测

### 步骤 1：解析参数

- `--target=<主题名>`（必选）：要分析的技术主题，如"协程调度模型""本地存储引擎"
- `--doc=<dir>`（可选）：社区文档目录路径，不传则不启用社区文档

若无 `--target` 参数，报错提示用法。

### 步骤 2：检测输出文件

检查 `docs/architecture/<subsystem>/<topic>.md` 是否已存在（<subsystem> 和 <topic> 待阶段 3 确认后确定路径）：
- **不存在** → 进入初次生成模式
- **已存在** → 反问主人「修订 / 补全」

---

## 初次生成模式

### 第 1 阶段：主题定位

**职责**：根据主题名在代码库中跨目录定位相关文件、识别入口点

派遣 1 个 researcher agent：

```
**任务模式**：代码库主题定位
**任务**：根据主题名 "<target>" 在代码库中定位所有相关源文件

**输入**：
- 主题名：<target>
- 代码库根目录：当前工作目录

**分析内容**：
1. 扫描目录结构，匹配主题关键词（含同义词、英文对应词）
2. 识别核心入口文件（头文件、主模块文件）
3. 识别相关实现文件（跨目录追踪依赖）
4. 识别相关测试文件（了解使用方式）

**输出**（写入 `/tmp/arch-extract-topic-locate-<ts>.md`）：
- 文件清单：按目录分组的文件列表
- 入口点：3-5 个核心入口文件/函数
- 依赖关系：文件间的 include/import 关系概览
- 规模估算：总行数、文件数
- 边界建议：主题的合理范围建议（哪些文件属于本主题、哪些属于边界外）
```

**质量门禁**：
- 找到 ≥3 个相关文件（否则主题名可能不对，向用户确认）
- 入口点已识别

### 第 2 阶段：文档发现与分析

**前置条件**：`--doc=<dir>` 已提供。未提供则跳过本阶段。

派遣 2 个 researcher agent 并行：

**Agent A — 文档发现与筛选**：

```
**任务模式**：社区文档发现
**任务**：在 <doc_dir> 下搜索与主题 "<target>" 相关的文档

**输入**：
- 文档目录：<doc_dir>
- 主题名：<target>

**匹配逻辑**：
1. 扫描目录下所有 .md/.rst/.txt 文件
2. 按 target 关键词（及其同义词/英文对应词）匹配文件名和标题
3. 筛选最相关的 1-5 篇

**输出**（写入 `/tmp/arch-extract-topic-doclist-<ts>.md`）：
- 匹配到的文档列表（路径 + 标题 + 相关度评分）
```

**Agent B — 设计声明提取**：

```
**任务模式**：设计声明提取
**任务**：从匹配到的社区文档中提取设计层面的声明

**输入**：
- 文档列表：由 Agent A 产出
- 主题名：<target>

**提取维度**：
1. 架构声明：组件划分、模块关系
2. 接口声明：API 定义、契约约定
3. 数据模型声明：核心数据结构描述
4. 并发/一致性声明：锁策略、一致性保证
5. 设计约束声明：明确的设计约定和限制

**输出**（写入 `/tmp/arch-extract-topic-claims-<ts>.md`）：
- 声明清单：按维度组织的设计声明列表
- 每条声明标注出处（文档路径 + 章节）
```

**质量门禁**（若启用了文档）：
- Agent A 找到 1-5 篇相关文档
- Agent B 提取 ≥3 条设计声明

### 第 3 阶段：范围确认

**职责**：向用户展示定位结果 + 匹配到的文档列表 + 文档声明清单，确认主题边界

主会话读取阶段 1 和阶段 2 的产出摘要（≤5 行），展示给用户：

1. **主题定位结果**：找到 N 个相关文件，跨 M 个目录
2. **入口点**：3-5 个核心入口
3. **文档匹配**（若启用）：匹配到 K 篇文档，提取 L 条设计声明
4. **边界建议**：建议的主题范围

使用 AskUserQuestion 确认：
- 主题边界是否合适（可调整范围）
- 输出路径：`docs/architecture/<subsystem>/<topic>.md`（确认 subsystem 和 topic 文件名）

### 第 4 阶段：代码分析

**职责**：深度分析源码，从四个维度并行切入

派遣 4 个 researcher agent 并行：

**Agent A — 架构与组件**：

```
**任务模式**：架构与组件分析
**任务**：分析主题的组件拓扑、模块划分、控制流、数据流

**输入**：
- 文件清单：阶段 1 产出
- 主题名：<target>

**分析维度**：
1. 组件拓扑：识别所有组件及其层次关系
2. 模块划分：模块边界、职责划分
3. 控制流：主要控制路径、分支逻辑
4. 数据流：数据在各组件间的流动路径

**输出**（写入 `/tmp/arch-extract-topic-arch-<ts>.md`）：
- ASCII Art 组件拓扑图
- 每个组件的职责描述
- 1-2 条核心控制/数据流的端到端描述
```

**Agent B — 数据结构与算法**：

```
**任务模式**：数据结构与算法分析
**任务**：分析核心 struct/class、存储布局、关键算法逻辑

**输入**：
- 文件清单：阶段 1 产出
- 主题名：<target>

**分析维度**：
1. 核心数据结构：struct/class 定义、字段语义、内存布局
2. 存储布局：数据在内存/磁盘中的组织方式
3. 关键算法：核心业务逻辑的算法描述（伪代码）

**输出**（写入 `/tmp/arch-extract-topic-data-<ts>.md`）：
- ≥2 个核心数据结构的详细分析
- 每个数据结构的存储布局和选择理由
- 1-2 个关键算法的逻辑描述
```

**Agent C — 并发与资源**：

```
**任务模式**：并发与资源分析
**任务**：分析锁策略、线程模型、内存管理、生命周期

**输入**：
- 文件清单：阶段 1 产出
- 主题名：<target>

**分析维度**：
1. 锁策略：锁类型、粒度、获取顺序
2. 线程/协程模型：角色、数量、调度方式
3. 内存管理：分配策略、所有权、释放时机
4. 生命周期：关键对象的创建/使用/销毁

**输出**（写入 `/tmp/arch-extract-topic-concur-<ts>.md`）：
- 锁清单（名称/类型/保护资源/持有范围）
- 线程模型描述
- 关键对象的生命周期分析
```

**Agent D — 接口与异常**：

```
**任务模式**：接口与异常分析
**任务**：分析公开 API、错误码、异常传播、边界条件

**输入**：
- 文件清单：阶段 1 产出
- 主题名：<target>

**分析维度**：
1. 公开接口：函数签名、参数、返回值
2. 前置/后置条件：调用约束和保证
3. 异常语义：错误码定义、传播路径
4. 边界条件：极限输入、资源耗尽、并发冲突

**输出**（写入 `/tmp/arch-extract-topic-iface-<ts>.md`）：
- ≥2 个关键接口的详细分析
- 错误码体系描述
- 边界条件清单
```

**质量门禁**：4 个 agent 全部返回，每个产出包含各自维度的分析内容。

### 第 5 阶段：文档-代码核对

**前置条件**：阶段 2 产出了设计声明清单。未启用 `--doc` 则跳过。

派遣 2 个 researcher agent 并行：

**Agent A — 接口契约核对**：

```
**任务模式**：接口契约核对
**任务**：将社区文档中的接口声明与阶段 4-D 的接口分析逐条对比

**输入**：
- 设计声明清单：阶段 2 产出
- 接口分析：阶段 4-D 产出

**对比维度**：函数签名、参数语义、前置/后置条件、异常语义

**输出**（写入 `/tmp/arch-extract-topic-check-iface-<ts>.md`）：
每条声明的对比结果：一致 / 文档有代码无 / 代码有文档无 / 矛盾
```

**Agent B — 设计约束核对**：

```
**任务模式**：设计约束核对
**任务**：将社区文档中的架构、数据模型、并发约束声明与阶段 4-A/B/C 的分析逐条对比

**输入**：
- 设计声明清单：阶段 2 产出
- 架构/数据/并发分析：阶段 4-A/B/C 产出

**对比维度**：组件划分、模块职责、数据组织、并发策略

**输出**（写入 `/tmp/arch-extract-topic-check-design-<ts>.md`）：
每条声明的对比结果：一致 / 文档有代码无 / 代码有文档无 / 矛盾
```

**文档-代码核对规则**：
- **一致**：产出文档可引用社区文档的描述，标注"与上游文档一致"
- **文档有，代码无**：以代码为准；在"已知问题"中记录为可能的版本差
- **代码有，文档无**：正常提取到产出文档，不特别标注
- **矛盾**：以代码为准；区分版本差（上游已演进但当前分支未跟进）和分叉差（本分支主动做出的不同选择）

### 第 6 阶段：分析确认

**职责**：向用户展示架构推断 + 核对结果，确认设计意图

主会话汇总阶段 4（四个维度分析摘要）和阶段 5（核对结果摘要，若有），展示给用户：

1. **架构推断**：组件拓扑、核心设计思路
2. **文档-代码差异**（若有）：列出一致/矛盾/未验证条目
3. **待确认的问题**：需要用户判断的设计意图（如矛盾差异的性质是版本差还是分叉差）

使用 AskUserQuestion 确认关键判断。矛盾处以用户判断为准。

### 第 7 阶段：文档生成

**职责**：按 topic 模板生成完整文档

派遣 3 个 architect agent 并行（按章切分），然后 1 个 architect agent 合并：

**Agent A — 设计背景与问题 + 核心设计**：

```
**任务模式**：topic 文档写作
**任务**：撰写"设计背景与问题"和"核心设计"两个章节

**输入**：
- 阶段 4 全部分析（A/B/C/D）
- 阶段 5 核对结果（若有）
- 阶段 6 用户确认的设计意图
- 模板：skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md（写作前 MUST 先读取）

**输出**（写入 `/tmp/arch-extract-topic-sec1-<ts>.md`）：
- 设计背景与问题：明确 1 个核心技术矛盾
- 核心设计：ASCII Art 组件图 + 1 条核心流程的端到端描述
```

**Agent B — 关键机制 + 数据结构**：

```
**任务模式**：topic 文档写作
**任务**：撰写"关键机制"和"数据结构"两个章节

**输入**：
- 阶段 4 全部分析（A/B/C/D）
- 模板：skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md（写作前 MUST 先读取）

**输出**（写入 `/tmp/arch-extract-topic-sec2-<ts>.md`）：
- 关键机制：2-4 个关键机制，每个含触发条件 + 执行步骤 + 边界条件
- 数据结构：≥2 个核心数据结构，每个附存储布局和选择理由
```

**Agent C — 接口 + 依赖 + 已知问题**：

```
**任务模式**：topic 文档写作
**任务**：撰写"接口"、"依赖与交互"和"已知问题"三个章节

**输入**：
- 阶段 4 全部分析（A/B/C/D）
- 阶段 5 核对结果（若有）
- 模板：skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md（写作前 MUST 先读取）

**输出**（写入 `/tmp/arch-extract-topic-sec3-<ts>.md`）：
- 接口：≥2 个接口，含前置/后置/异常语义
- 依赖与交互：上下游 + 同步/异步 + 耦合强度
- 已知问题：文档-代码差异按版本差/分叉差分类记录；无则写"未发现"
```

**Agent D — 合并**：

```
**任务模式**：文档合并
**任务**：将三个章节文件合并为完整文档

**输入**：
- `/tmp/arch-extract-topic-sec1-<ts>.md`
- `/tmp/arch-extract-topic-sec2-<ts>.md`
- `/tmp/arch-extract-topic-sec3-<ts>.md`
- 模板：skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md（参照自检清单）

**输出**（写入 `/tmp/arch-extract-topic-draft-<ts>.md`）：
- 完整的 topic 文档（≤500 行）
- 去掉章节间的重复内容
- 术语统一
```

**质量门禁**：文档 ≤500 行，所有必填章节均已覆盖。

### 第 8 阶段：评审

**职责**：独立评审 + 修正循环（最多 3 轮）

派遣 3 个 architect-reviewer agent 并行（按视角切分）：

**Agent A — 技术准确性**：

```
**任务模式**：独立技术评审
**评审视角**：
1. 技术陈述是否准确反映代码实现
2. 组件拓扑和数据流是否正确
3. 数据结构描述是否与代码一致
4. 并发/资源分析是否有遗漏或错误

**被评审对象**：`/tmp/arch-extract-topic-draft-<ts>.md`
**review_output_path**：`/tmp/arch-extract-topic-review-tech-<ts>.md`
```

**Agent B — 内容完整性**：

```
**任务模式**：独立评审
**评审视角**：
1. 是否覆盖了模板 `skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md` 的所有必填章节
2. 各章节是否达到最低深度要求
3. 关键机制是否遗漏重要子机制
4. 已知问题是否遗漏了明显的设计局限或文档-代码差异

**被评审对象**：`/tmp/arch-extract-topic-draft-<ts>.md`
**review_output_path**：`/tmp/arch-extract-topic-review-comp-<ts>.md`
```

**Agent C — 表达质量**：

```
**任务模式**：独立评审
**评审视角**：
1. 文档篇幅是否 ≤500 行
2. 图表（ASCII Art/Mermaid）是否清晰、正确
3. 术语是否统一
4. 是否出现"因为模板要求所以要有"的占位填充
5. 安全/性能/错误语义是否融入关键机制描述而非独立成章

**被评审对象**：`/tmp/arch-extract-topic-draft-<ts>.md`
**review_output_path**：`/tmp/arch-extract-topic-review-expr-<ts>.md`
```

**评审修正循环**：
1. 汇总 3 个 agent 的问题清单，计算缺陷密度
2. 无 CRITICAL + 缺陷密度 ≤ 2.0 → 通过，进入第 9 阶段
3. 否则 → 派遣 1 个 architect agent 修正，回到本阶段重新评审
4. 最多 3 轮。3 轮后仍未通过 → 标注残留问题，进入第 9 阶段

**问题分值**：CRITICAL=10, HIGH=3, MEDIUM=1, LOW=0.1
**缺陷密度** = 问题分数之和 / 章节数

### 第 9 阶段：写入

主会话将最终文档写入 `docs/architecture/<subsystem>/<topic>.md`（路径由阶段 3 确认）。

**终端汇报**：
- 文档路径
- 章节数、图示数
- 文档-代码核对结果（若有社区文档输入）
- 置信度（评审通过 / 带债通过）

---

## 修订模式

反问主人「想修订哪一块」：
1. 修订核心设计与关键机制
2. 修订数据结构与接口
3. 修订已知问题

只跑对应范围的阶段 4-9，merge 结果回原文档。评审只检查变更范围。

## 补全模式

用模板自检清单扫描文档，识别缺失项，跑阶段 4-9 补全。

---

## 禁忌项

- 禁止跳过主题定位直接进入代码分析
- 禁止跳过范围确认（阶段 3 必须与用户交互）
- 禁止跳过评审循环直接输出
- 禁止安全/性能/错误语义独立成章（必须融入关键机制描述）
- 禁止将社区文档描述当作本产品的设计声明（必须标注"来自上游社区文档"）
- 禁止写"因为模板要求所以要有"的占位章节
- 禁止超过 500 行
- 禁止不读取模板就直接写作
