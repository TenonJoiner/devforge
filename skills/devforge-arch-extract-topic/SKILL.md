# devforge-arch-extract-topic — 技术主题逆向提取

## 概述

从已有源码逆向提取一个技术主题的深度设计文档。技术主题是按技术关注点组织的横向概念（如"协程调度模型""本地存储引擎""Raft 共识"），跨目录存在，不按代码目录组织。

**核心原则**：
1. **源码 = 行为真相**：代码是唯一权威的事实来源
2. **社区文档 = 参考线索**：可选的社区文档提供概念框架和设计背景，但与代码矛盾时以代码为准
3. **分析密度优先**：深度分析阶段用多 agent 按维度切分并行，确保每个维度读够源码
4. **单次生成单篇文档**：每次执行只产出当前目标 topic 的一篇文档

**与 subsystem/system skill 的关系**：
- topic 是最深层的设计文档，聚焦单个技术主题的"怎么做"和"为什么这样做"
- subsystem 文档为 topic 提供架构上下文（组件拓扑、模块间关系）
- topic 文档不重复 subsystem 已有的上下文描述

## 工作目录约定

- **输入**：
  - 源码目录 `--src=<dir>`（默认：当前工作目录）—— 事实来源
  - 社区文档目录 `--ref-doc=<dir>`（默认：空，不启用社区文档）—— 参考来源
- **输出**：`--output-docs=<dir>`（默认：`docs`），实际路径为 `<output-docs>/architecture/<subsystem>/<topic>.md`
- **模板**：`skills/devforge-arch-extract-topic/templates/arch-reverse-topic.md`
- **临时文件**：所有中间产出写入 `/tmp/arch-extract-topic-*-<ts>.md`，其中 `<ts>` = `$(date +%s)$$`（秒级时间戳 + PID，保证并发安全）

**输出路径确定**：
- `<subsystem>` 由阶段 3 范围确认时与用户商定
- `<topic>` 文件名由主会话根据主题名生成（小写、连字符分隔，如 `coroutine-scheduler`）

## 启动检测

### 步骤 1：解析参数

- `--target=<主题名>`（必选）：要分析的技术主题，如"协程调度模型""本地存储引擎"
- `--src=<dir>`（可选，默认：当前工作目录）：源码目录路径
- `--ref-doc=<dir>`（可选，默认：空）：社区文档目录路径，不传则不启用社区文档
- `--output-docs=<dir>`（可选，默认：`docs`）：输出文档根目录，实际输出路径为 `<output-docs>/architecture/<subsystem>/<topic>.md`

若无 `--target` 参数，报错提示用法。

## 初次生成模式

### 第 1 阶段：主题定位

**职责**：根据主题名在代码库中跨目录定位相关文件、识别入口点

派遣 1 个 Explore agent（`subagent_type: "Explore"`）：

```
**任务模式**：代码库主题定位
**任务**：根据主题名 "<target>" 在 <src> 中定位所有相关源文件

**输入**：
- 主题名：<target>
- 代码库根目录：<src>

**分析内容**：
1. 扫描目录结构，匹配主题关键词（含同义词、英文对应词）
2. 识别核心入口文件（头文件、主模块文件）
3. 识别相关实现文件（跨目录追踪依赖）
4. 识别相关测试文件（了解使用方式）

**输出**（写入 `/tmp/arch-extract-topic-locate-<ts>.md`）：
- 文件清单：按目录分组的文件列表
- 入口点：3-5 个核心入口文件/函数
- 依赖关系：文件间的 include/import 关系概览
- 规模估算：总行数、文件数
- 边界建议：主题的合理范围建议（哪些文件属于本主题、哪些属于边界外）
```

**质量门禁**：
- 找到 ≥3 个相关文件（否则主题名可能不对，向用户确认）
- 入口点已识别

### 第 2 阶段：文档发现与分析

**前置条件**：`--ref-doc=<dir>` 已提供。未提供则跳过本阶段。

派遣 1 个 general-purpose agent：

```
**任务**：在 <ref-doc> 下搜索与主题 "<target>" 相关的文档，从中提取设计层面的声明

**输入**：
- 文档目录：<ref-doc>
- 主题名：<target>

**步骤 1 — 文档发现与筛选**：
- 搜索目录索引文件（SUMMARY.md、INDEX.md、README.md、toc.md 等），从中按主题逐层导航定位相关文档
- 按 target 关键词（及其同义词/英文对应词）全文搜索所有文档内容，作为补充
- 合并去重，筛选最相关的 1-5 篇

**步骤 2 — 设计声明提取**（从步骤 1 匹配到的文档中提取）：
1. 设计动机/背景：这个主题为何存在、解决了什么技术矛盾、设计目标是什么
2. 架构声明：组件划分、模块关系
3. 接口声明：API 定义、契约约定
4. 数据模型声明：核心数据结构描述
5. 关键机制/流程声明：核心流程描述、触发条件、执行步骤
6. 并发/一致性声明：锁策略、一致性保证
7. 设计约束声明：明确的设计约定和限制

**输出**（写入 `/tmp/arch-extract-topic-claims-<ts>.md`）：
- 匹配到的文档列表（路径 + 标题 + 相关度评分）
- 声明清单：按维度组织的设计声明列表，每条标注出处（文档路径 + 章节）
```

**质量门禁**（若启用了文档）：
- 找到 1-5 篇相关文档
- 提取 ≥3 条设计声明

### 第 3 阶段：范围确认

**职责**：向用户展示定位结果 + 匹配到的文档列表 + 文档声明清单，确认主题边界

主会话读取阶段 1 和阶段 2 的产出摘要（≤5 行），展示给用户：

1. **主题定位结果**：找到 N 个相关文件，跨 M 个目录
2. **入口点**：3-5 个核心入口
3. **文档匹配**（若启用）：匹配到 K 篇文档，提取 L 条设计声明
4. **边界建议**：建议的主题范围

使用 AskUserQuestion 确认：
- 主题边界是否合适 → 不合适时提供调整选项：
  - 缩小：从建议清单中剔除部分目录/文件
  - 扩大：追加用户指定的目录/文件
  - 重新定位：修改 target 关键词，回到阶段 1 重新搜索
- 输出路径：`<output-docs>/architecture/<subsystem>/<topic>.md`（确认 subsystem 和 topic 文件名）

确认后检查输出文件是否已存在：
- **不存在** → 进入初次生成模式（第 4-8 阶段）
- **已存在** → 进入补全模式（用模板自检清单扫描缺失项，跑阶段 4-9 补全）
  - 若非补全而是修订已有内容，不适用本 skill，主人直接用 `/df:design` 或手动编辑

### 第 4 阶段：代码分析

**职责**：深度分析源码，从五个维度并行切入

**上下文注入**（主会话准备，注入各 agent prompt）：
- 阶段 1 定位结果中的文件清单和入口点
- 若启用了 `--ref-doc`：阶段 2 的设计声明清单（`/tmp/arch-extract-topic-claims-<ts>.md`），agent 分析时对照声明逐条验证代码是否支持（"设计动机/背景"类声明无需验证，留作阶段 7 写作参考）
- 若 `<output-docs>/architecture/<subsystem>/` 下已有 subsystem/topic 文档：留作阶段 7 写作参考（避免重复已有上下文），不注入阶段 4

派遣 5 个 architect agent 并行：

**Agent A1 — 代码组织与调用链**：

```
**任务**：阅读源码，理清代码组织方式和从入口点出发的完整调用链

**输入**：
- 文件清单与入口点：阶段 1 产出（入口点是控制流追踪的起点）
- 主题名：<target>
- <若启用了 ref-doc> 设计声明清单（架构部分）：/tmp/arch-extract-topic-claims-<ts>.md，逐条与代码对照

**分析要求**：
1. 代码组织：从 #include/import 关系和目录结构出发，画出文件之间的依赖关系图。列出每个目录/命名空间下有哪些关键文件、各文件之间的 include 关系
2. 调用链：从阶段 1 给出的每个入口函数出发，向下追踪内部调用（不超出阶段 1 的文件清单范围），列出调用路径上经过的函数和文件行号。分支路径只追到主题边界内的叶子函数

**输出**（写入 `/tmp/arch-extract-topic-arch-<ts>.md`）：
- 文件依赖关系图（ASCII Art，标注目录和关键文件路径）
- 每个入口函数的调用链（函数名 → 函数名 → ...，每步标注文件和行号）
- <若启用了 ref-doc> 对照声明清单中架构部分的逐条验证：代码支持 / 代码不支持 / 无法验证（附具体行号）
```

**Agent A2 — 数据流追踪**：

```
**任务**：追踪核心 struct 在调用链上的完整传递路径——从创建到消费，覆盖传递方式、所有权转移、生命周期

**输入**：
- 文件清单与入口点：阶段 1 产出
- 主题名：<target>
- <若启用了 ref-doc> 设计声明清单（关键机制部分）：/tmp/arch-extract-topic-claims-<ts>.md，逐条与代码对照

**分析要求**：
1. 核心 struct 选择：阶段 1 文件清单中出现频率最高的 2-3 个核心 struct/class
2. 传递路径：追踪每个 struct 从创建到消费的完整路径，标注每步传递方式（函数参数/返回值/全局变量/队列）及函数和行号
3. 所有权转移：标注指针传递、move 语义等所有权变更点

**输出**（写入 `/tmp/arch-extract-topic-dataflow-<ts>.md`）：
- 2-3 个核心 struct 的传递路径（从创建到消费，标注每步的函数、行号、传递方式）
- <若启用了 ref-doc> 对照声明清单中关键机制部分的逐条验证：代码支持 / 代码不支持 / 无法验证（附具体行号）
```

**Agent B — 数据结构与算法**：

```
**任务**：提取阶段 1 入口点涉及的核心 struct/class 的完整定义、内存/磁盘布局、以及入口函数内的关键算法实现

**输入**：
- 文件清单与入口点：阶段 1 产出（优先分析入口函数直接操作的数据结构）
- 主题名：<target>
- <若启用了 ref-doc> 设计声明清单（数据模型部分）：/tmp/arch-extract-topic-claims-<ts>.md>

**分析要求**：
1. 核心 struct/class 选择：阶段 1 入口函数的参数类型和返回值类型优先；若入口函数涉及的 struct 不足 2 个，再扩展到入口函数调用的内部函数所操作的 struct
2. 字段提取：完整列出每个 struct 的所有字段（字段名、类型、源码行号），附源码中的注释原文。计算 sizeof 并列出对齐填充
3. 内存布局：列出嵌套 struct 的包含关系、union 的变体、柔性数组等影响内存布局的结构。若代码中有 static_assert(sizeof(...) == N)，一并列出
4. 磁盘/网络布局：找到序列化/反序列化函数，逐字段描述写入和读出的顺序、格式（如 varint、固定宽度、length-prefixed）、字节序处理
5. 关键算法：选择入口函数内部最复杂的 1-2 段业务逻辑（非参数校验、非错误处理），从源码中提取算法步骤，用伪代码描述，标注源函数和行号

**输出**（写入 `/tmp/arch-extract-topic-data-<ts>.md`）：
- 每个核心 struct 的字段表（字段名 | 类型 | 行号 | 注释原文）
- 每个 struct 的内存布局描述（sizeof、对齐、嵌套关系、static_assert）
- 序列化/反序列化的逐字段格式描述（附序列化函数和行号）
- 1-2 个关键算法的伪代码（标注源函数和行号）
- <若启用了 ref-doc> 对照声明清单中数据模型部分的逐条验证
```

**Agent C — 并发与资源**：

```
**任务**：识别项目并发框架，基于框架提取同步原语和执行单元的创建点，追踪关键对象的分配和释放路径

**输入**：
- 文件清单与入口点：阶段 1 产出（入口函数直接创建/操作的对象即为关键对象）
- 主题名：<target>
- <若启用了 ref-doc> 设计声明清单（并发/一致性部分）：/tmp/arch-extract-topic-claims-<ts>.md

**分析要求**：
1. 识别并发框架：从阶段 1 文件清单的 #include 中识别并发/同步库（pthread.h、abt.h、folly/executors、seastar、std::thread 等），确定项目的并发模型和对应的同步原语。若项目使用单线程无锁模型（如 Seastar reactor），后续步骤 2 改为描述实际的同步机制（future/promise、continuation、消息队列等）而非搜索传统锁
2. 同步原语：从步骤 1 识别的框架同步原语出发，列出每个同步原语的类型（框架对应的 mutex/rwlock/spinlock/atomic 或等价 API）、保护对象（它守卫哪个成员/哪段逻辑）、获取/释放调用点（含 RAII 守卫）、是否存在跨函数持有，全部标注行号。若步骤 1 判定为无锁模型，描述代码中实际使用的同步机制（如等待 future、提交回调、写入消息队列），标注行号
3. 执行单元：从步骤 1 识别的创建 API 出发，搜索执行单元的创建点（ULT/tasklet/goroutine/fiber/thread 的 spawn/create），列出每个创建点的入口函数和传入参数。描述入口函数内部的主循环或核心逻辑，标注行号。不推断执行单元的"角色"，只陈述入口函数做了什么
4. 内存管理：从阶段 1 文件清单的 #include 中识别内存管理方案——先搜索自定义 allocator/内存池/pmem 等封装的分配器头文件，再针对性搜索其分配/释放 API。若未找到封装，回退到 malloc/free/new/delete/智能指针。对阶段 1 入口函数涉及的关键对象，追踪其分配点、所有权转移点（指针传递、move）、释放点，标注行号
5. 生命周期：以阶段 1 入口点为起点，追踪入口函数创建的关键对象从 init/create → 运行中使用（被哪些函数访问/修改）→ deinit/destroy 的完整阶段，每阶段标注函数和行号

**输出**（写入 `/tmp/arch-extract-topic-concur-<ts>.md`）：
- 同步原语清单（变量名 | 框架/API | 保护对象 | 获取/释放行号 | 是否跨函数持有；无锁模型则描述同步机制 + 行号）
- 执行单元清单（创建点行号 | 框架/API | 入口函数 | 传入参数 | 主逻辑位置）
- 关键对象的内存分配/释放路径（对象 | 分配点行号 | 所有权转移点 | 释放点行号）
- 关键对象的生命周期阶段表（对象 | 创建 | 使用中 | 销毁，每阶段函数+行号）
- <若启用了 ref-doc> 对照声明清单中并发/一致性部分的逐条验证
```

**Agent D — 接口与异常**：

```
**任务**：以阶段 1 入口点为关键接口，提取每个接口的签名、前置条件、错误返回、边界检查，并从测试文件中提取调用模式

**输入**：
- 文件清单与入口点：阶段 1 产出（入口点列表即关键接口列表）
- 主题名：<target>
- <若启用了 ref-doc> 设计声明清单（接口/契约部分）：/tmp/arch-extract-topic-claims-<ts>.md

**分析要求**：
1. 函数签名：对阶段 1 的每个入口函数，从头文件中提取完整的函数签名（参数名+类型+是否 const/指针、返回值类型），标注声明行号
2. 前置条件：从函数体内的 assert、参数校验（NULL 检查、范围检查、状态检查）中提取调用方必须满足的条件，标注每个检查的行号
3. 错误返回：从函数体内的错误码定义（enum/#define）和 return 语句中提取所有可能的错误返回值及其触发条件；沿调用链向下追一层，看被调用函数的错误如何向上传播（直接返回/包装后返回/日志后吞掉），标注行号
4. 边界检查：从函数体内的边界判断（buffer 满/空、计数器溢出、超时、资源耗尽）中提取边界条件及其处理方式（返回错误/阻塞等待/abort），标注行号
5. 测试中的调用模式：从相关测试文件中提取每个入口函数的典型调用序列——测试先构造什么参数、调用后检查什么返回值、如何清理，用伪代码描述

**输出**（写入 `/tmp/arch-extract-topic-iface-<ts>.md`）：
- 每个入口函数的接口表（签名行号 | 前置条件 | 成功返回值 | 错误返回值及触发条件）
- 错误码表（错误码 | 含义 | 触发条件 | 传播方式，每项附行号）
- 边界条件表（边界 | 检测位置行号 | 处理方式）
- 每个入口函数的测试调用序列（伪代码，标注来源测试文件和行号）
- <若启用了 ref-doc> 对照声明清单中接口/契约部分的逐条验证
```

**质量门禁**：5 个 agent 全部返回，每个产出包含各自维度的分析内容；若启用了 `--ref-doc`，每个产出须包含对照声明清单的验证结果。

### 第 5 阶段：架构推断

**职责**：综合阶段 4 五个维度的代码事实和阶段 4 验证结果，归纳架构设计思路

派遣 1 个 architect agent：

```
**任务**：读取阶段 4 的代码分析产出和验证结果，归纳主题的架构设计思路

**输入**：
- 阶段 4-A1：代码组织与控制流（文件依赖关系图、调用链）
- 阶段 4-A2：数据流追踪（struct 传递路径）
- 阶段 4-B：数据结构与算法（字段表、内存/磁盘布局、算法伪代码）
- 阶段 4-C：并发与资源（同步原语清单、执行单元清单、分配路径、生命周期）
- 阶段 4-D：接口与异常（入口函数接口表、错误码表、边界条件表）
- 阶段 4 验证结果（若有）：文档-代码差异表，辅助判断哪些代码行为是刻意设计 vs 版本差异

**推断要求**：
1. 组件拓扑：从 4-A1 的文件依赖关系图和调用链中，归纳出逻辑组件边界（文件 → 组件的聚类），画出 ASCII Art 组件拓扑图
2. 核心设计思路：从调用链和 struct 传递路径中，归纳出 1 条核心设计模式（如 pipeline、event-driven、layered 等），用 3-5 句话描述
3. 核心技术矛盾：从数据结构选择（4-B）、并发策略（4-C）、接口设计（4-D）中，归纳出 1 个核心技术矛盾——面向上游的约束是什么，为此做了什么关键取舍
4. 设计动机推断：若 4-B 的字段注释和 4-C 的生命周期中有设计意图的线索，归纳出"为什么这样设计"的推断；无法从代码中推断的部分标注"推断依据不足"

**输出**（写入 `/tmp/arch-extract-topic-synth-<ts>.md`）：
- ASCII Art 组件拓扑图（从文件依赖聚类为逻辑组件）
- 核心设计思路（3-5 句）
- 1 个核心技术矛盾
- 设计动机推断（附推断依据和置信度）
```

**质量门禁**：组件拓扑图和核心设计思路已产出，核心技术矛盾已明确。

### 第 6 阶段：分析确认

**职责**：向用户展示架构推断 + 核对结果，确认设计意图

主会话汇总阶段 5（架构推断摘要）和阶段 4（验证结果摘要，若有），展示给用户：

1. **架构推断**：阶段 5 产出的组件拓扑、核心设计思路、核心技术矛盾
2. **文档-代码差异**（若有）：列出一致/矛盾/未验证条目
3. **待确认的问题**：需要用户判断的设计意图（如有）

使用 AskUserQuestion 确认关键判断。矛盾处以用户判断为准。

**质量门禁**：
- 用户明确确认了主题边界和输出路径 → 推进到第 7 阶段
- 用户要求调整范围 → 若缩小（剔除文件），直接重跑阶段 4-5；若扩大或重新定位，回退到阶段 1 重跑定位，再重跑阶段 4-5。阶段 2（文档发现）不重跑
- 用户对分析结果有疑问但能给出明确指示 → 记录指示，推进到第 7 阶段（writer 以用户指示为准覆盖分析结论）
- 用户无法判断 → 标记为"待确认"，推进到第 7 阶段（writer 在已知问题中记录差异，标注"需人工核实"）
- 若无文档-代码差异且无待确认问题 → 跳过 AskUserQuestion，自动推进到第 7 阶段（避免无意义的用户打断）

### 第 7 阶段：文档生成

**共享写作约束**（三个 agent 均须遵守）：

- 模板中的引用块（> 开头）和自检清单是写作指引，禁止写入最终文档。合并 agent 必须在输出前删除所有模板脚手架残留
- **叙事性**：主要面向 coding agent，兼顾人的阅读体验。每个设计决策回答"解决了什么、为什么这样解决、舍弃了什么"；章节间有逻辑递进，给出必要背景铺垫，让读者无需先读源码也能理解设计脉络

**Agent B — 关键设计**：

```
**任务**：撰写"关键设计"章节

**输入**：
```

```
- 去掉章节间的重复内容（同一概念在多个章节出现时，保留首次出现的完整描述，后续章节仅引用）
```

```
3. 术语是否统一
4. 是否出现"因为模板要求所以要有"的占位填充

6. 叙事性：章节间是否有逻辑递进（背景→设计→关键设计→接口与交互→问题），核心设计章节是否为关键设计章节做了铺垫，而非孤立罗列
7. 可读性：是否过度依赖条目罗列或表格对比——设计方案和思路应以文字阐述为主，表格仅用于结构化信息（字段表、接口表、锁清单等）；避免连续三个以上纯列表段落
```

**评审修正循环**：
1. 汇总 3 个 agent 的问题清单
2. 存在任何 CRITICAL 问题 → **直接判定不通过**（CRITICAL 一票否决，不受缺陷密度阈值豁免）
3. 无 CRITICAL 且缺陷密度 ≤ 2.0 → 通过，进入第 9 阶段
4. 否则 → 主会话汇总三份评审报告，合并为修正清单（每条问题附所属评审报告路径），注入 1 个 architect agent 的派遣 prompt。agent 按清单逐项修正后，回到本阶段重新评审
5. 最多 3 轮。3 轮后仍未通过 → 标注残留问题，进入第 9 阶段

**问题分值**：CRITICAL=10, HIGH=3, MEDIUM=1, LOW=0.1
**缺陷密度** = 问题分数之和 / 章节数（5）

**注**：CRITICAL 问题在密度公式中占 10 分已远超 2.0 阈值，一票否决规则确保了 CRITICAL 不会因章节数多而侥幸通过。

## 禁忌项

- 禁止跳过主题定位直接进入代码分析
- 禁止跳过范围确认（阶段 3 必须与用户交互）
- 禁止跳过评审循环直接输出
- 禁止安全/性能/错误语义独立成章（必须融入关键设计描述）
- 禁止将社区文档描述直接当作本产品的设计声明（代码与文档一致时直接写，不一致时以代码为准并注明差异）
- 禁止写"因为模板要求所以要有"的占位章节
- 禁止为凑深度而堆砌内容（宁缺毋滥）
- 最终文档控制在 500 行以内
- 禁止不读取模板就直接写作
- 禁止将内部过程制品（声明编号、临时文件路径、核对清单 ID 等）泄漏到最终文档
- 禁止将模板脚手架（引用块 >、自检清单节、写作指引）写入最终文档——合并 agent 必须删除
