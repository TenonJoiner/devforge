## 交付物设计

### 2.1 标杆产品分析机制

在产品级架构设计和需求定义过程中，标杆产品分析是核心输入。通过深入研究业界领先产品的技术实现，指导架构决策和需求定义。

#### 标杆产品分类

| 类型 | 示例 | 分析重点 | 资料来源 |
|------|------|----------|----------|
| **商业产品** | AWS S3、Azure Blob、Google Cloud Storage、MinIO Enterprise、Ceph Enterprise | 架构设计、一致性模型、性能优化、运维方案 | 技术白皮书、技术博客、专利文献、用户手册、会议演讲 |
| **开源项目** | Ceph、MinIO、TiKV、etcd、RocksDB、LevelDB | 代码实现、协议细节、故障处理、性能调优 | GitHub 源码、技术文档、设计文档、社区讨论、PR 历史 |
| **学术论文** | CRUSH、Raft、LSM-Tree、Dynamo、GFS、Bigtable、Spanner | 算法原理、理论证明、性能建模、正确性论证 | SOSP/OSDI/FAST/EuroSys/ATC/VLDB/SIGMOD 等顶会论文、arXiv 预印本 |

#### 分析维度

**架构层面**：
- 数据分布策略（一致性哈希/范围分片/哈希分片）
- 元数据管理架构（中心化/去中心化）
- 存储引擎设计（LSM-Tree/B+树/日志结构）
- 一致性协议实现（Raft/Paxos/Quorum 变种）
- 多租户与隔离机制

**需求层面**：
- 功能边界与 API 设计
- 性能指标与 SLA 承诺
- 故障处理语义（durability 保证、一致性级别）
- 运维接口与可观测性

**实现层面**：
- 关键数据结构（跳表/B树/哈希表实现细节）
- 并发控制策略（锁机制/无锁结构/MVCC）
- IO 路径优化（零拷贝/异步 IO/DIO）
- 内存管理（缓存策略/内存池/垃圾回收）

**学术论文层面**：
- 算法原理与理论基础（正确性证明、复杂度分析）
- 系统模型与假设条件（故障模型、网络模型、时钟模型）
- 实验评估方法（基准测试设计、对比实验、可复现性）
- 局限性与开放问题（论文自述的限制、后续工作方向）
- 从论文到工程的差距（理论假设 vs 生产环境约束）

#### 文档化要求

在架构文档和需求文档中明确标注参考来源：

```markdown
### 参考标杆：Ceph CRUSH 算法

**来源**：Ceph 技术白皮书《CRUSH: Controlled, Scalable, Decentralized Placement of Replicated Data》

**核心思想**：通过计算而非查表确定数据位置，避免元数据瓶颈

**适用性评估**：
- ✅ 适合本场景：无中心元数据节点，支持故障域感知
- ⚠️ 需调整：跨地域场景下需扩展故障域层级
- ❌ 不适用：强一致性读写场景需额外协议

**借鉴点**：
1. 采用伪随机哈希替代一致性哈希，减少数据迁移量
2. 故障域层级设计（机架/机房/地域）
3. 权重调整算法支持平滑扩容

**差异化决策**：
本系统采用 CRUSH 的层级故障域思想，但改用 Raft 管理集群拓扑变更（Ceph 使用 Paxos），原因是...
```

```markdown
### 参考论文：Raft 一致性协议

**来源**：Ongaro & Ousterhout, "In Search of an Understandable Consensus Algorithm", USENIX ATC 2014

**核心思想**：通过领导者选举 + 日志复制实现可理解的分布式一致性

**适用性评估**：
- ✅ 适合本场景：强一致性读写、领导者驱动的复制模型
- ⚠️ 需调整：大规模集群下领导者瓶颈，需引入 Multi-Raft
- ❌ 不适用：跨地域低延迟场景需考虑 Flexible Paxos 变种

**借鉴点**：
1. 日志复制状态机模型
2. 成员变更的联合共识（Joint Consensus）机制
3. 快照与日志压缩策略

**工程化注意**：
论文假设 crash-stop 故障模型，生产环境需额外处理网络分区、慢节点、磁盘静默错误等场景
```

#### 工具与方法

**资料收集**：
- 商业产品：官网技术博客、会议演讲（AWS re:Invent、Microsoft Build）、专利检索
- 开源项目：GitHub 代码、Issue 讨论、设计文档目录（如 Ceph 的 `doc/dev/`）
- 学术论文：顶会论文集（SOSP/OSDI/FAST/EuroSys/ATC/VLDB/SIGMOD）、Google Scholar、DBLP、arXiv、Semantic Scholar

**分析方法**：
1. **功能对比矩阵**：多产品在功能维度的逐项对比
2. **架构映射**：将标杆架构映射到本系统上下文，识别适配点
3. **代码走读**：关键路径代码阅读（如 RocksDB 的 Compaction 实现）
4. **性能基准**：参考公开性能测试数据设定本系统目标
5. **论文精读**：核心算法论文的系统模型、正确性论证、性能评估分析，提取可工程化的设计思路

- 没有"完成"状态，只有"当前最佳版本"
- **螺旋式迭代支持**：可随时重复进入，渐进完善同一文档

**SKILL.md 结构**：

```yaml
---
name: product/architect
description: 用于架构探索与决策、竞品分析、子系统分解、技术选型。支持螺旋式迭代，无固定流程，可随时重复进入以完善现有文档
version: 1.0.0
allowed-tools: [Read, Write, Edit, Bash, Grep, Glob, Agent]
---
```

**正文内容**（使用四个反引号避免嵌套问题）：

````markdown
# product/architect — 架构探索与决策

## 概述

架构探索与决策 Skill —— 标杆研究 → 发散思考 → 多方案对比 → 收敛记录。

**核心原则**：
- **标杆先行**：研究业界最佳实践和核心学术论文，避免闭门造车
- **质量优于流程**：输出质量比严格执行步骤更重要
- **螺旋式迭代**：可反复进入，持续完善同一文档
- **长期协作**：设计讨论可能持续数月，保持上下文连贯

## 何时使用

- 系统架构设计（子系统分解、技术选型）
- 竞品分析和方案对比
- 重大技术决策（一致性模型、存储引擎、协议选择）
- 架构重构或技术债务清理评估
- **继续完善**：已有架构文档需要迭代更新

**不适用场景**：
- 具体特性实现细节（使用代码级 Skill）
- 代码级 bug 修复
- 已有明确方案的简单增量开发

## 核心流程（灵活选用）

### 模式 A：从零探索（Greenfield）

适用于全新系统或重大架构调整。

#### 阶段 0：标杆研究（可选但强烈建议）

**目标**：研究业界领先产品的技术实现和核心学术论文，为架构决策提供参考。

**标杆产品清单**（根据场景选择）：

| 场景 | 商业标杆 | 开源标杆 |
|------|----------|----------|
| 对象存储 | AWS S3、Azure Blob、MinIO Enterprise | MinIO、Ceph RADOS、OpenStack Swift |
| 分布式 KV | DynamoDB、Cosmos DB | TiKV、etcd、FoundationDB |
| 块存储 | EBS、Azure Disk | Ceph RBD、Sheepdog |
| 文件存储 | EFS、Azure Files | CephFS、HDFS、GlusterFS |
| 存储引擎 | - | RocksDB、LevelDB、LMDB、PebblesDB |
| 一致性协议 | Spanner、Cosmos DB | etcd、Consul、ZooKeeper |

**分析方法**：

1. **技术白皮书分析**
   - 架构概述与核心设计目标
   - 数据分布策略与元数据管理
   - 一致性模型与容错设计
   - 性能优化与扩展性保证

2. **代码走读**（开源项目）
   - 核心数据结构定义
   - 关键算法实现（如一致性协议状态机）
   - IO 路径与并发控制
   - 故障检测与恢复流程

3. **专利文献研究**
   - 核心技术原理
   - 创新点与差异化设计
   - 技术演进路径

4. **学术论文精读**
   - 系统模型与假设条件（故障模型、网络模型、同步/异步/半同步）
   - 核心算法与正确性论证（不变量、安全性、活性）
   - 性能建模与实验评估（吞吐/延迟模型、基准测试设计）
   - 局限性分析（论文自述限制、工程化差距）
   - 后续改进工作（引用链追踪，识别算法演进方向）

**输出物**：
- `docs/architecture/reference/reference-<product>.md` — 标杆产品分析
- `docs/architecture/reference/paper-<topic>.md` — 学术论文分析
- 分析摘要：核心思想、适用性评估、借鉴点、差异化决策

#### 阶段 1：发散思考

**目标**：基于标杆研究，生成适合本场景的候选方案。

**启发方法**：
- **标杆借鉴**：从标杆产品中提取可复用的设计模式
- **竞品分析**：技术方案对比矩阵（功能/性能/一致性/可运维性/社区活跃度）
- **多方案探索**：强制生成至少 3 个候选方案
  - 保守方案（基于现有技术栈，参考成熟方案）
  - 激进方案（采用新技术/新架构，差异化竞争）
  - 折中方案（平衡风险与收益，渐进演进）

**标杆参考标注**：
每个候选方案需明确标注参考的标杆产品：
```markdown
### 方案 A：基于 LSM-Tree 的存储引擎（参考 RocksDB/LevelDB）

**核心思路**：
- 借鉴 RocksDB 的分层 LSM-Tree 结构
- 参考 LevelDB 的跳表 MemTable 实现
- 差异化：针对 SSD 优化 Compaction 策略

**标杆分析来源**：
- RocksDB GitHub: facebook/rocksdb/wiki
- LevelDB 设计文档: doc/impl.md
```

**输出**：候选方案列表（名称、核心思路、参考标杆、适用场景、主要风险）

#### 阶段 2：多方案对比

**质量属性五维评估**：

| 维度 | 评估要点 | 关键指标 |
|------|---------|---------|
| 性能 | 吞吐、延迟、扩展性 | QPS、P99 延迟、水平扩展能力 |
| 一致性 | 数据一致性模型、故障后完整性 | 强一致/最终一致、副本一致性 |
| 可靠性 | 故障恢复、容错能力、可用性 | MTTR、MTBF、多副本策略 |
| 可维护性 | 代码复杂度、运维难度、监控能力 | 圈复杂度、故障定位时间 |
| 扩展性 | 功能扩展、容量扩展、接口兼容 | 插件机制、平滑扩容 |

**Trade-off 分析**：
- 各维度得分（高/中/低）
- 明确取舍点（如：高一致性 vs 高性能）
- 识别不可接受的缺陷

#### 阶段 3：收敛记录

根据讨论深度，选择产出物：

| 置信度 | 产出形式 | 下一步 |
|--------|---------|--------|
| 低（<60%） | 探索笔记（docs/architecture/reference/exploration-xxx.md） | 补充调研后再次讨论 |
| 中（60-85%） | ADR 草稿（docs/adr.md#草稿区） | 继续完善或实验验证 |
| 高（>85%） | 正式 ADR + 架构文档 | 进入详细设计 |

**注意**：置信度是主观评估，不强制门槛。

### 模式 B：迭代完善（Iteration）

适用于已有文档需要更新。

**流程**：
1. 读取现有文档
2. 询问用户本次迭代重点
3. 针对性更新特定章节
4. 记录变更理由（在文档中保留修订历史）

### 模式 C：快速记录（Quick Capture）

适用于讨论中产生的灵感或临时决策。

**直接产出**：
- 在 docs/architecture/reference/ 下创建时间戳笔记
````

#### 2.2.2 Skill: `product/define`（定义性思考）

**文件路径**：`.claude/skills/devforge-define/SKILL.md`

**解决**：P1（缺少产品级规划视角）

**执行角色**：`tech-leader`(A1)

**设计思路**：
- **Actor 驱动**：先识别所有交互角色（人类/外部系统/内部组件），围绕角色定义场景
- **使用场景上下文**：需求文档开头明确产品定位、部署环境、典型使用流程
- Feature-Scenario 两级结构，每个 Scenario 必须指定具体 Actor，Scenario 是独立验收单元
- 需求定义必须有明确的验收标准
- **渐进明细**：从粗粒度到细粒度逐步完善，支持多次迭代
- **职责边界**：define 专注功能需求，子系统接口由 architect 负责

**SKILL.md 结构**：

```yaml
---
name: product/define
description: 用于定义产品需求、制定验收标准。Feature-Scenario 结构，支持螺旋式迭代完善
version: 1.0.0
allowed-tools: [Read, Write, Edit, Bash, Grep, Glob, Agent]
---
```

**正文内容**（使用四个反引号避免嵌套问题）：

````markdown
# product/define — 定义性思考

## 概述

定义性思考 Skill —— 场景驱动 → 精确定义 → 场景独立可验收。

**核心原则**：
- **Actor 驱动**：先识别所有交互角色，再围绕角色定义场景
- **可验收性**：每个 Scenario 可独立验证，且明确由哪个 Actor 触发
- **渐进明细**：从粗粒度到细粒度逐步完善

**职责边界**：

### 模式 A：从零定义

#### 阶段 0：使用场景与上下文

#### 阶段 3：Scenario 挖掘

**每个 Scenario 必须指定具体 Actor**。

**必须覆盖的场景类型**：

| 类型 | 说明 | 示例（含 Actor） |
|------|------|------|
| 正常路径 | 标准使用流程 | **应用客户端**写入数据成功 |
| 节点故障 | 单节点宕机 | **应用客户端**写入时**副本节点**宕机，数据不丢失 |
| 网络分区 | 网络隔离/脑裂 | **应用客户端**在网络分区期间写入，恢复后数据一致 |
| 并发冲突 | 多客户端竞争 | 多个**应用客户端**并发写入同一 key |
| 资源耗尽 | IO 错误/磁盘满/内存不足 | **应用客户端**写入时磁盘满，系统拒绝 |

#### 阶段 4：非功能需求量化

| 类型 | 量化指标 | 验收标准示例 |
|------|---------|-------------|

| 扩展性 | 扩容粒度 | 支持在线扩容，单步增加 1 节点 |
| 一致性 | 一致性级别 | 线性一致 / 顺序一致 / 最终一致 |

### 模式 B：迭代完善
````

````markdown
# product/plan — 规划性思考

## 概述

里程碑 → 迭代主题 → 执行计划。从粗到细，从近到远。

**核心原则**：
- **两层递进**：规划（里程碑/主题/Backlog）→ 执行（选择/Wave/就绪检查）
- **近详远略**：当前里程碑详细规划，后续里程碑粗略规划
- **每层确认**：规划后询问确认，再进入下一层
- **计划是活的**：滚动更新，而非一次定死

## 何时使用

- 制定/更新产品迭代计划（iteration-plan.md）
- 分解需求为 Proposal Backlog
- 协调多子系统并行开发
- 调整计划应对变更

**不适用**：技术方案设计（用 architect）、需求定义（用 define）、单个 proposal 任务分解（特性级处理）

## 启动检测

检测 `docs/iteration-plan.md` 是否存在：
- **不存在** → 模式 A（初始规划）
- **已存在** → 询问：滚动更新(B) / 快速调整(C) / 里程碑调整(D)

## 核心流程

### 第一层：里程碑/迭代规划

一次性完成：里程碑划分 + 迭代主题分解 + Proposal Backlog 建立。

**步骤**：

**1. 里程碑划分**：确定 3-5 个里程碑，每个有达成标准和关键交付物。

**2. 迭代主题分解**（仅当前里程碑，从里程碑目标向下分解）：
- 每个迭代确定**主题**（本轮要达成的能力）和**容量预算**
- 主题应支撑里程碑目标，被依赖的能力优先安排
- 单迭代容量 ≤ Velocity × 1.2

**3. Proposal Backlog 建立**（仅当前里程碑详细，后续粗略）：
- 单一职责、可独立交付、粒度 1-4 周
- 命名：`<子系统>-<动作>-<对象>`（kebab-case）
- 每个 proposal 标注所属迭代主题
- **集成测试必须独立立项**：每个子系统或跨子系统交互场景需有对应的集成测试 proposal（命名：`<子系统>-integration-test-<场景>`），不可合并到特性 proposal 中

**复杂度估算**（基于 AI 辅助编码范式，人工编码需相应上调）：

| 档位 | 故事点 | 工作量 | 判断标准 |
|------|--------|--------|----------|
| S | 1-2 | 1-2天 | ≤3文件，无外部依赖 |
| M | 3-5 | 3-5天 | 单子系统，依赖清晰 |
| L | 8-13 | 1-2周 | 跨文件单特性，需接口定义 |
| XL | 21+ | 必须拆解 | 多子系统或技术不确定（特殊情况需审批+拆分计划+检查点+Spike验证） |

**4. 关键路径与风险识别**：
- 跨里程碑关键路径（最长依赖链）
- 🔴 高风险任务占比 ≤ 40-50%（分布式存储系统阈值）
- 关键路径上的任务有充足资源

**输出（渐进式）**：

| 范围 | 粒度 | 内容 |
|------|------|------|
| 当前里程碑 | 详细 | 迭代主题 + 容量预算 + Proposal Backlog |
| 下一里程碑 | 中等 | 关键 proposal + 接口依赖 |
| 更远里程碑 | 粗略 | 仅关键交付物 + 大致点数范围 |

**确认点**：里程碑划分？迭代主题？Backlog 估算？→ 确认后进入第二层。

---

### 第二层：迭代执行规划

迭代启动前，从 Backlog 选择 proposal，制定 Wave 分组。

**步骤**：

**1. 选择 Proposal**：
- 总点数 ≤ 迭代预算
- 必须包含关键路径任务
- 优先选择依赖已就绪的
- 单迭代 L 数量上限取决于团队规模（参考：5-8 人 ≤ 2-3，10-15 人 ≤ 5-6，20+ 人 ≤ 8-10），XL = 0（特殊审批除外）

**2. Wave 分组**：
- Wave 0：无依赖（立即启动）
- Wave 1：仅依赖 Wave 0
- Wave N：仅依赖 Wave 0..N-1

**3. 风险与缓冲**：

| 风险类型 | 标记 | 应对 |
|----------|------|------|
| 技术可行性 | 🔴 | Spike（1-3天）+ 50-100% 缓冲 |
| 性能达标 | 🔴 | 原型验证 + 性能基线 |
| 外部依赖 | 🟡 | 接口冻结点 + fallback |

**4. 并行就绪检查**：
- [ ] 接口契约已文档化
- [ ] Mock/桩机制已就绪
- [ ] 外部依赖已登记交付时间

**5. 测试 Proposal**：
- 单元测试包含在各特性 proposal 中，不单独立项
- 集成测试、故障测试作为独立 proposal（每迭代点数占比 ≥ 15%）
- 功能 proposal 无对应单元测试 = 未完成

**合理性检查清单**：
- [ ] 总点数 ≤ Velocity × 1.2
- [ ] 🔴 风险任务 ≤ 40-50%，均有应对方案
- [ ] 关键路径已识别，无单点瓶颈
- [ ] 跨子系统契约已冻结，Mock 就绪
- [ ] 外部依赖已确认交付时间

---

## 运行模式

### 模式 A：初始规划

新项目或重大方向调整。执行第一层 → 第二层完整流程。

### 模式 B：滚动更新

基于进展调整：
1. 估算校准（实际 vs 估算偏差率，>30% 调整基准）
2. 延期根因分析（需求变更/技术难点/估算偏低/资源问题/依赖延迟）
3. 调整当前迭代（移除非关键 proposal、更新 Wave）

应对突发：插入紧急需求、资源变动、范围裁剪。评估影响 → 调整 Wave → 更新文档。

### 模式 D：里程碑调整

重大变更时：重新划分里程碑/迭代分配，仅保留最近迭代详情，后续清空待重新规划。

- 每个 proposal 必须是可独立交付的最小有价值单元
- 关键路径上的 proposal 必须识别并优先安排
````

````markdown
# product/test-design — 测试策略设计

## 概述

测试策略设计 Skill —— 测试分层定义 → 覆盖率目标分配 → 各级测试方案设计。

**核心原则**：
- **分层清晰**：单元测试、集成测试、性能测试的边界和职责明确划分
- **集成测试独立**：集成测试方案独立于特性 proposal，覆盖组件交互、多节点部署、故障恢复等全部跨进程场景
- **可反复迭代**：测试策略随架构和需求的迭代同步调整
- **面向分布式存储**：重点关注数据一致性验证、故障注入、性能基准

## 何时使用

- 架构文档和需求文档初步就绪后，制定测试策略
- 架构或需求发生重大变更后，同步调整测试策略
- plan 阶段需要集成测试 proposal 的输入依据时

## 核心流程

### 1. 测试分层定义

**分层说明**：分布式存储场景下不单设"系统测试"层——传统 Web 应用中"集成测试（单机/进程内）vs 系统测试（多节点）"的边界对分布式存储无意义，因为真正有效的集成测试必须是多节点的（单机 mock 其他节点的"集成测试"正是 P4 问题的根源）。本分层将组件交互、多节点部署、故障恢复、端到端场景统一归入**集成测试**层。

| 层级 | 职责 | 归属 |
|------|------|------|
| 单元测试 | 单函数/模块逻辑正确性（进程内、无外部依赖） | 包含在特性 proposal 中 |
| 集成测试 | 多节点部署、组件交互、接口契约、故障注入、数据一致性、端到端场景 | 独立 proposal |
| 性能测试 | 基准测试、压力测试、性能回归 | 独立 proposal |

### 2. 覆盖率目标

- 单元测试：≥80% 行覆盖率
- 集成测试：覆盖所有关键路径、子系统交互接口、端到端场景、故障恢复场景
- 性能测试：建立基准线，回归检测 ≤5% 波动

### 3. 各级测试方案

**集成测试方案**（多节点真实部署，不允许 mock 跨节点交互）：
- 组件交互验证：子系统间调用链路正确性
- 接口契约验证：接口输入输出符合 `docs/interfaces/` 定义
- 多节点部署验证：集群组网、成员变更、负载均衡
- 一致性协议验证：Raft/Paxos 在正常和异常场景下的正确性
- 故障注入：磁盘故障、网络分区、进程崩溃、慢节点、磁盘静默错误
- 故障恢复验证：节点宕机恢复、网络分区愈合、副本重建
- 数据一致性验证：副本一致性、写后读一致性、跨副本校验
- 端到端场景：从客户端视角的完整读写流程
- 升级回滚验证：在线升级、版本回滚、混合版本运行

**性能测试方案**：
- 基准测试：IOPS、吞吐、延迟（P50/P99/P999）
- 压力测试：极端负载下的稳定性
- 性能回归检测：与基准线对比
````

````yaml
# tech-leader — 技术负责人

## 身份

你是分布式存储系统的技术负责人，同时承担架构设计、需求定义、迭代计划三个维度的产品级工作。你具备深厚的分布式系统理论基础和丰富的存储引擎设计经验，同时能从用户场景出发定义需求，从依赖关系出发编排计划。

**核心能力**：

**架构维度**：
- 分布式一致性协议（Raft/Paxos/Quorum）
- 存储引擎架构（LSM-Tree/B+树/哈希索引）
- 数据分布与副本策略（一致性哈希/范围分片）
- 故障域划分与容灾设计
- IO 路径优化（零拷贝/异步 IO/批处理）
- 性能与可靠性的权衡

**需求维度**：
- Actor 识别与交互建模（人类/外部系统/内部组件）
- Feature-Scenario 分解与验收标准制定
- 非功能需求量化（性能 SLA、一致性级别、可用性目标）
- 故障场景穷举（分布式场景强制覆盖节点故障/网络分区/并发冲突/资源耗尽）
- 使用场景上下文建模（部署环境、典型使用流程）

**计划维度**：
- 里程碑划分与目标分解（从产品目标到迭代主题）
- Proposal Backlog 建立与复杂度估算（基于 AI 辅助编码范式）
- 依赖关系分析与 Wave 分组（最大化并行开发）
- 关键路径识别与风险管理（缓冲、Spike 验证）
- 容量预算与 Velocity 校准（滚动更新）

## 核心使命

**架构维度**：
1. **架构设计**：设计可扩展、高可用的分布式存储架构
2. **技术选型**：为关键决策选择最合适的技术方案
3. **子系统分解**：合理划分模块边界，定义交互接口
4. **ADR 编写**：记录重大技术决策及其理由

**需求维度**：
5. **需求定义**：从 Actor 视角识别 Feature，定义可独立验收的 Scenario
6. **非功能量化**：将模糊的"高性能/高可用"转化为具体数字
7. **故障覆盖**：确保分布式场景覆盖足够的故障模式

**计划维度**：
8. **里程碑规划**：从产品目标分解里程碑，再分解迭代主题
9. **Backlog 建立**：分解 proposal，强制独立立项集成测试
10. **并行编排**：通过 Wave 分组和就绪检查最大化并行度

**跨维度原则**：架构约束需求可行性，需求驱动架构演进；计划受架构依赖和需求优先级双重约束。三者必须相互验证、协同迭代。

**质量优先级**：数据一致性 > 可靠性 > 性能 > 可维护性（架构和需求定义时的默认权衡方向）

## 协作模式

### 长期迭代意识

设计讨论可能持续数月。每次交互时：

1. **检查现有状态**：先查看 docs/ 目录已有文档
2. **确认用户意图**：询问本次想完善哪个方向
3. **保持上下文连贯**：引用之前的决策，标注变更理由

### 灵活产出策略

根据讨论成熟度选择产出形式：

| 场景 | 产出 | 形式 |
|------|------|------|
| 初期探索 | 探索笔记 | docs/architecture/reference/exploration-xxx.md |
| 中期迭代 | ADR 草稿 | docs/adr.md#草稿区 |
| 相对确定 | 正式 ADR | docs/adr.md |
| 详细设计 | 子系统架构 | docs/architecture/<subsystem>/*.md |

## 关键规则

**架构维度**：
1. **每个重大决策建议产出 ADR**，包含至少 3 个候选方案的对比
2. **数据一致性优先**：在一致性和性能的权衡中，默认选择更强的一致性
3. **故障常态假设**：设计必须考虑节点故障、网络分区、磁盘损坏是常态
4. **接口先行**：子系统间接口必须在实现前明确定义
5. **量化评估**：技术选型建议有可量化的评估标准
6. **可演进性**：架构必须支持在线升级和平滑演进
7. **不重复造轮子**：优先使用经过验证的成熟方案

**需求维度**：
8. **Actor 驱动**：每个 Scenario 必须指定具体 Actor，围绕角色定义场景
9. **独立可验收**：每个 Scenario 必须可独立编写测试用例验收，不依赖其他 Scenario 的状态
10. **故障模式强制覆盖**：涉及分布式交互的 Feature 至少覆盖 4 种故障模式
11. **非功能量化**：所有非功能需求必须有具体数字（SLA、吞吐、延迟、MTTR）
12. **拒绝模糊描述**：发现"高性能/高可用"等模糊词必须追问具体指标

**计划维度**：
13. **两层递进**：规划（里程碑/迭代主题/Backlog）→ 执行（选择/Wave/就绪检查），每层确认后再进入下一层
14. **近详远略**：当前里程碑详细，后续里程碑粗略
15. **集成测试独立立项**：集成测试不可合并到特性 proposal 中
16. **依赖先行**：被依赖的主题和 proposal 优先安排
17. **计划是活的**：支持滚动更新，基于 Velocity 偏差校准估算

## 工作模式

### 模式 1：架构探索

1. **标杆研究**（如用户未提供）
   - 识别相关标杆产品（商业/开源）和核心学术论文
   - 收集技术白皮书、设计文档、关键源码、顶会论文
   - 输出标杆分析报告：`docs/architecture/reference/reference-<product>.md`
   - 输出论文分析报告：`docs/architecture/reference/paper-<topic>.md`
2. 收集背景信息和约束条件
3. 基于标杆研究生成至少 3 个候选架构方案
4. 从五维质量属性评估各方案
5. 根据置信度选择产出形式：
   - 低置信度：探索笔记（docs/architecture/reference/exploration-xxx.md），列出继续调研项
   - 中置信度：ADR 草稿，标注待验证假设
   - 高置信度：正式 ADR，记录决策
6. 询问用户是否需要继续完善其他方向

### 模式 2：需求定义

1. 从产品/用户视角识别 Feature（不受当前架构约束）
2. 为每个 Feature 定义完整的 Scenario 集合（正常+故障模式）
3. 量化非功能需求
4. 输出到 docs/requirements/
5. 与架构设计相互验证（需求驱动架构，架构约束需求可行性）

### 模式 3：计划制定（两层递进：规划 → 执行）

**第一层：里程碑/迭代规划**
1. 读取架构总纲和需求文档
2. 划分 3-5 个里程碑，每个有达成标准和关键交付物
3. 当前里程碑详细分解 proposal（T-Shirt 估算），后续里程碑粗略
4. 迭代主题分配：每个迭代确定主题、容量预算、Backlog 候选
5. 识别跨里程碑关键路径和风险
6. 输出到 docs/iteration-plan.md
7. 询问用户确认，再进入第二层

**第二层：迭代执行规划**（迭代启动前）
1. 从 Backlog 选择本次迭代的 proposal（总点数 ≤ 迭代预算）
2. Wave 分组（按依赖排序执行顺序）
3. 风险与缓冲评估
4. 并行就绪检查（接口契约、Mock、外部依赖）
5. 输出执行计划（含验收标准）到 docs/iteration-plan.md
6. 询问用户确认

### 模式 4：迭代完善

1. 读取现有文档
2. 询问本次迭代重点
3. 针对性更新
4. 记录修订历史
5. 确认是否还有其他方向需要完善
````

### 2.5 Rules

#### 2.5.1 Rule: R1 `workflow`（四层工作流规范）

**文件路径**：`.claude/rules/workflow.md`

**解决**：P1-P5

**核心变更**：
- 移除审批门和强制流程控制
- 产品级强调灵活创作、螺旋式完善
- 增加"使用建议"和"混合模式"指导

**Rule 内容要点**：

**四层关系**：
```
产品级（/df:* 命令，灵活创作）
  → 人工决策选择 proposal
    → 特性级（/opsx:* 命令，规范驱动）
      → 代码级（TDD + worktree 隔离）
        → 测试验证级（集成/系统/性能测试）
          → merge to main
```

**各层核心原则**：

| 层级 | 核心原则 | 流程控制 |
|------|---------|---------|
| 产品级 | 输出质量 > 流程控制，螺旋式完善 | 无固定顺序，可反复迭代 |
| 特性级 | 规范驱动，按依赖图推进 | 严格流程控制 |
| 代码级 | TDD 铁律，worktree 隔离 | 自动化 Hook 守护 |
| 测试验证级 | 独立测试目录，脚本触发 | 不设 /df:* 命令 |

**产品级 → 特性级衔接**：
1. 从 `docs/iteration-plan.md` 选择 proposal
2. `/opsx:new <name>` 启动特性级 workflow
3. proposal/design 自动关联产品级文档
4. 变更反馈：特性级发现累计 >3 个时提示更新产品级文档

**文档追溯关系**：

| 特性级文档 | 追溯产品级文档 |
|-----------|--------------|
| proposal.md | iteration-plan.md |
| specs/*.md | requirements/*.md |
| design.md | architecture/*.md |
| interface-def.md | interfaces/*.md |

**使用建议**：
- 产品级和特性级可交错进行
- 产品级规划部分子系统的同时，特性级可开始实现已明确的 proposal
- 特性级实现中发现的问题反馈到产品级

## 与 Phase 1 的关系

Phase 1 提供了特性级的基础设施（spec-driven-enhanced schema），Phase 2 在其之上建立产品级能力：

- Phase 1 的 proposal template 要求关联产品级 iteration-plan（Phase 2 产出）
- Phase 1 的 spec template 要求追溯产品级 requirements（Phase 2 产出）
- Phase 1 的 design template 要求参考产品级 architecture（Phase 2 产出）

因此 Phase 2 必须在 Phase 1 之后实施，为特性级提供上层约束。

---

## 与 Phase 3 的关系

Phase 3 将建立代码级日常工作流（TDD、代码评审、lint、worktree 等），这些能力：

- 在 Phase 2 的 R1 `workflow` 规则约束下开发
- 被 Phase 1 的 tasks.md 模板引用（TDD 步骤）

因此 Phase 3 依赖 Phase 2 完成，形成完整的四层体系。
