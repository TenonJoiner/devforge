# coding-style-generator — 仓库编码规范生成器

## 概述

本 skill 把一份代码仓库的真实编码风格，转化为可直接供 coding agent 使用的 `.claude/rules/coding-style-<lang>.md` 规范文件。

流程分为 5 个阶段：
1. 生成仓库地图
2. 提取实际风格事实
3. 冲突分析
4. 生成规范文件
5. 自动验证与修复

所有中间产物和最终规范都写入当前仓库的 `.claude/rules/` 目录。

主会话负责任务调度、用户确认和推进判定；深度分析、内容创作由子 agent 完成。agent 产出的报告主会话可直接阅读，用于确认关键决策点。

## 触发场景

- “帮我给这个仓库生成一份编码规范”
- “分析这个 repo 的代码风格”
- “创建 coding-style-c.md”
- “给 coding agent 写规则”
- “统一团队代码风格”

## 启动检测

### 1. 确认输出目录

检查当前仓库根目录是否存在 `.claude/rules/`：
- 存在 → 继续
- 不存在 → 询问用户是否创建：`mkdir -p .claude/rules`

### 2. 确认外部规范参考（可选）

如果团队已经有一份编码规范，可以作为生成参考；如果没有，后续流程基于仓库实际风格事实直接生成。

通过 `AskUserQuestion` 询问用户：
> 你是否已有团队编码规范文件希望作为参考？如果有，请提供文件路径（可多选，支持具体文件或目录）；如果没有，直接回复“无”。这些文件不会被修改。

记录用户提供的 `<external-specs>`：
- 用户回复“无” → 空列表 `[]`
- 用户提供路径 → 路径列表（若是目录，agent 会按需遍历其中相关文件）

后续阶段中，这些外部规范作为**可选输入**：提供了则读取对比，未提供则跳过。

### 3. 检查前置产物

用 `Glob` 扫描 `.claude/rules/`：
- 若已有 `codebase-analysis-report.md`，询问用户是否跳过第 1 阶段
- 若已有 `code-style-facts-<lang>.md`，询问用户是否跳过第 2 阶段
- 若已有 `gap-analysis-<lang>.md`，询问用户是否跳过第 3 阶段

## 第 1 阶段：生成仓库地图

**目标**：为当前仓库生成 `codebase-analysis-report.md`，作为后续阶段的事实基础。

### 主会话动作

1. 确认 `.claude/rules/` 目录存在
2. 派遣 1 个通用 agent 完成仓库地图分析
3. agent 返回后，阅读报告末尾的检查摘要和关键章节，确认覆盖度和语言列表
4. 若检查摘要不满足通过标准，让同一 agent 补充缺失项

### Agent 任务

```
分析当前仓库，生成 .claude/rules/codebase-analysis-report.md。

请完成：
1. 列出仓库主语言、构建系统、主要子系统/模块目录
2. 按“子系统分层 → 分层抽样”方式选取代表性源文件（按语言分别列出）
   - 分层要求：覆盖 3–5 个不同子系统/模块，具体数量根据仓库复杂度、规模和子系统边界动态决定
   - 每层优先选取以下类型文件：
     - 错误处理/返回值路径复杂的文件
     - 锁/并发原语密集的文件
     - 内存分配/资源释放关键路径
     - 对外公共头文件或接口定义
     - 近期活跃修改的文件（反映当前团队共识）
     - 存在明显历史债务的文件（用于识别例外）
   - 数量建议：每种主语言 8–15 个文件
   - 每个文件附一句话选取理由
3. 列出该仓库使用的第三方框架/库
4. 列出仓库内已有的编码规范文件、.clang-format、lint 配置等（仅作背景参考，不作为规范内容来源）
5. 评估各语言代码占比，判断需要为哪些语言生成 coding-style-<lang>.md

输出要求：
- 将完整报告写入 .claude/rules/codebase-analysis-report.md
- 在报告末尾附加一个“检查摘要”小节，仅包含客观计数，格式如下：
  - 子系统覆盖数：N（目标 3–5）
  - 每种主语言抽样文件数：
  - 建议生成规范的语言列表：
  - 主要风险/不确定项数量：N（应 ≤2）
```

### 通过标准

检查摘要中的客观计数满足：
- 3 ≤ 子系统覆盖数 ≤ 5
- 每种主语言 8 ≤ 抽样文件数 ≤ 15
- “建议生成规范的语言列表”非空
- 主要风险/不确定项数量 ≤ 2

## 第 2 阶段：提取实际风格事实

**目标**：为每种主要语言生成 `code-style-facts-<lang>.md`。

### 主会话动作

1. 读取第 1 阶段报告末尾的“建议生成规范的语言列表”
2. 通过 `AskUserQuestion` 向用户确认需要生成规范的语言列表
3. 针对每种确认的语言，并行派遣 1 个通用 agent
4. agent 返回后，阅读各语言文件末尾的检查摘要，确认是否满足通过标准
5. 未通过的语言 → 让对应 agent 修复

### Agent 任务

```
阅读 .claude/rules/codebase-analysis-report.md，针对语言 <lang> 分析其实际编码风格。

输出文件：.claude/rules/code-style-facts-<lang>.md

内容格式：
| 维度 | 实际做法 | 是否有子系统差异 | 是否有历史例外 | 示例（文件路径 + 短片段）|
|------|---------|----------------|--------------|------------------------|
| 错误处理与错误传播方式 | | | | |
| 日志宏/级别与选用场景 | | | | |
| 资源管理（内存/对象/锁生命周期） | | | | |
| 并发原语与选用判据 | | | | |
| 并发编程模型 | | | | |
| 领域框架 API / 惯用法 | | | | |
| 公共接口设计约定（所有权转移/释放责任/API 稳定性） | | | | |
| 命名模式与领域缩写惯例 | | | | |
| 目录↔前缀↔职责映射 | | | | |
| 性能敏感约定（零拷贝/缓冲区复用/大对象分配/热路径禁忌） | | | | |
| 字符串/字节处理约定 | | | | |
| 危险函数 / 安全红线 | | | | |
| 测试组织约定 | | | | |
| 代码演进约定（弃用/TODO/向后兼容） | | | | |
| 工程约定（工具无法强制的部分） | | | | |

要求：
- 只描述事实，不做评判
- 每个维度给出 1–3 个真实代码片段
- 聚焦工具无法自动修复的隐式约定
- 不收录 formatter/linter 可强制的缩进、花括号、import 顺序、命名大小写等

输出要求：
- 将完整表格写入 .claude/rules/code-style-facts-<lang>.md
- 在文件末尾附加一个“检查摘要”小节，仅包含客观计数，格式如下：
  - 15 个维度中无内容的数量：N（应为 0）
  - 示例缺失或不是真实代码的维度数量：N（应为 0）
  - 混入 formatter/linter 可修复格式细节的数量：N（应为 0）
```

### 通过标准

检查摘要中的客观计数满足：
- 无内容维度数量 = 0
- 示例缺失/非真实代码维度数量 = 0
- 混入 formatter/linter 可修复格式细节数量 = 0

## 第 3 阶段：冲突分析

**目标**：生成 `gap-analysis-<lang>.md`。
- 若提供了外部规范：找出外部规范与仓库实际的差距。
- 若未提供外部规范：分析仓库内部一致性，提炼红线规则、推荐风格与禁止反模式。

### 主会话动作

1. 确认当前 `<external-specs>` 是否非空
2. 针对每种语言，并行派遣 1 个通用 agent 完成冲突分析
3. agent 返回后，阅读 `gap-analysis-<lang>.md` 和检查摘要，确认关键决策点
4. 若检查摘要不满足通过标准，让对应 agent 补充
5. 对于有争议、需要用户拍板的点，通过 `AskUserQuestion` 询问用户
6. 派遣 agent 根据用户答复刷新 `gap-analysis-<lang>.md`

### Agent 任务（冲突分析）

```
读取以下输入：
- 外部规范参考（可选）：<external-specs> 中用户提供的文件/目录（若为空则跳过）
- 仓库风格事实：.claude/rules/code-style-facts-<lang>.md

完成以下 5 项分析并输出到 .claude/rules/gap-analysis-<lang>.md：

1. 外部规范对比（若未提供外部规范，本项输出“未提供外部规范，跳过”）
   - 哪些条款该仓库完全遵循？列出条款和证据（跳过 formatter/linter 可自动强制的条款）。
   - 哪些条款该仓库部分遵循？列出冲突点、代码示例，并评估：影响范围、迁移成本、能否自动化约束。
   - 哪些条款该仓库完全不适用？说明原因。
2. 该仓库有哪些重要实践是现有规范没有覆盖的？
3. 哪些规则应作为“必须遵守的红线”？（不遵守会导致 bug/安全/稳定性问题）
4. 哪些规则应作为“推荐风格”？（影响可读性/一致性，但不阻塞正确性）
5. 该仓库必须禁止哪些标准库或通用替代写法？列出反模式、错因、推荐替代写法及真实代码示例。

每条冲突/规则补充：
- 影响范围：全局 / 某子系统 / 历史遗留
- 迁移成本：高 / 中 / 低
- 是否可自动化约束：是 / 否 / 部分

输出要求：
- 将完整分析写入 .claude/rules/gap-analysis-<lang>.md
- 在文件末尾附加“检查摘要”小节，仅包含客观计数，格式如下：
  - 5 个分析项中未覆盖的数量：N（应为 0）
  - 无证据结论数量：N（应为 0）
  - 冲突点/规则标注缺失数量：N（应为 0，即每条冲突点/规则都标注了影响范围/迁移成本/可自动化性）
  - 需要用户拍板的争议点数量：N
```

### 通过标准

检查摘要中的客观计数满足：
- 未覆盖分析项数量 = 0
- 无证据结论数量 = 0
- 冲突点/规则标注缺失数量 = 0

## 第 4 阶段：生成规范文件

**目标**：为每种语言生成最终的 `.claude/rules/coding-style-<lang>.md`。

### 主会话动作

1. 确认所有输入存在
2. 针对每种语言，并行派遣 1 个通用 agent 生成规范
3. agent 返回后，阅读 `coding-style-<lang>.md` 和检查摘要，确认是否满足通过标准
4. 未通过的语言 → 让对应 agent 修复

### Agent 任务

```
为语言 <lang> 生成编码规范，写入 .claude/rules/coding-style-<lang>.md。

输入（需全部读取；.clang-format / lint 配置仅作背景，不作为规范内容来源）：
- 外部规范参考（可选）：<external-specs> 中用户提供的文件/目录（若为空则跳过）
- 仓库风格事实：.claude/rules/code-style-facts-<lang>.md
- 差距分析与决策：.claude/rules/gap-analysis-<lang>.md

要求：
1. 篇幅简洁完整，通常 200–500 行，宁短勿滥。每个规则必须基于输入文件中的事实或决策，禁止编造；规范正文不写来源引用，通过真实代码示例和验证阶段保证可追溯性
2. 完整自洽的编码规范（非差异增量），agent 只读一份即可编码。若提供了外部规范，最终规范必须同时包含：外部规范中适用/遵循的条款（按本仓库实际调整后写入）、与外部规范冲突且已按本仓库决策调整后的条款、仓库自身特有的规则；不能只写差异，也不能完全忽略外部规范
3. 结构如下：
   - 适用范围（1 段）
   - 必须遵守的规则
   - 推荐风格
   - 禁止反模式
   - 历史例外说明
4. 每条规则统一格式：
   - 规则陈述（使用“必须”/“推荐”/“禁止”措辞）
   - 适用场景
   - 原因（错因/收益）
   - 正确示例（文件路径 + 短片段）
   - 错误示例（如为禁止规则，文件路径 + 错因）
5. 内容焦点：只收录工具无法自动修复的隐式约定
   - 关注编码范式与工程领域：错误处理、并发、日志、资源管理、领域框架 API、目录↔前缀↔职责映射、并发编程模型、公共接口设计、性能敏感约定
   - 不收录可由 formatter/linter 自动修复的风格（缩进 / 花括号 / import 顺序 / 命名大小写等）
   - 不含构建 / 测试执行命令、覆盖率标准、lint 配置
6. 质量要求：规则必须可执行
   - 判据而非枚举：每个选项 / 级别 / API 须给出“什么场景选哪个”的判据
   - 领域 API 要有深度：列出该仓库的核心 API / 惯用法，给正确用法 + 选用判据
   - 反模式必须给出推荐替代写法，不能只列错误
7. 历史例外说明必须包含：
   - 例外场景
   - 存在原因
   - 是否允许新代码继续使用
   - 建议迁移方式
8. 真实性约束
   - 忠于输入：技术名词、错误码、框架 API 必须与输入文件一致
   - 禁止编造输入未支持的量化标准（覆盖率 / 阈值 / 行数等）
   - 版权与许可证须核实仓库实际主体与年份
   - 代码示例须来自该仓库真实代码，标注文件路径；反模式须标注“错误写法”并说明错因
   - 安全红线不可削弱
9. 规则溯源：生成时确保每条规则都能对应到 gap-analysis-<lang>.md 或 code-style-facts-<lang>.md 中的事实/决策；规范正文不出现“来源：...”这类引用

输出要求：
- 将完整规范写入 .claude/rules/coding-style-<lang>.md
- 在文件末尾附加“检查摘要”小节，仅包含客观计数，格式如下：
  - 缺失章节数量：N（应为 0，即包含全部 5 个章节）
  - “必须”规则缺失真实代码示例数量：N（应为 0）
  - 禁止规则缺失错误示例或推荐替代写法的数量：N（应为 0）
  - 未经验证量化标准数量：N（应为 0）
  - 安全红线被削弱数量：N（应为 0）
```

### 通过标准

检查摘要中的客观计数满足：
- 缺失章节数量 = 0
- “必须”规则缺失真实代码示例数量 = 0
- 禁止规则缺失错误示例/替代写法数量 = 0
- 未经验证量化标准数量 = 0
- 安全红线被削弱数量 = 0

## 第 5 阶段：自动验证与修复

**目标**：检查生成的规范并自动修复问题。

### 主会话动作

1. 针对每种语言，派遣 1 个通用 agent 执行验证并直接修复问题
2. agent 返回后，阅读验证报告中的“残留问题汇总”，确认是否通过
3. 若存在未解决的 CRITICAL 或 HIGH 问题，再次派遣 agent 修复，最多 3 轮

### Agent 任务

```
读取 .claude/rules/coding-style-<lang>.md 及其输入文件：
- .claude/rules/code-style-facts-<lang>.md
- .claude/rules/gap-analysis-<lang>.md
- 外部规范参考（可选）：<external-specs> 中用户提供的文件/目录（若为空则跳过）

按以下清单验证，发现问题时直接修复并覆写 .claude/rules/coding-style-<lang>.md：

1. 输入忠实性
   - 技术名词、错误码、框架 API 是否与输入一致
   - 是否存在输入文件中未提及的量化标准、阈值、行数限制
2. 结构合规性
   - 是否包含：适用范围、必须遵守的规则、推荐风格、禁止反模式、历史例外说明
   - 若提供了外部规范，最终规范是否完整（而非差异增量），是否同时覆盖了外部规范适用条款、调整后的冲突条款、仓库特有规则
   - 是否出现未被确认的语言规范
3. 规则格式一致性
   - 每条规则是否包含：规则陈述、适用场景、原因、正确示例
   - 禁止规则是否额外包含错误示例 + 推荐替代写法
   - 每条规则是否能在 gap-analysis-<lang>.md 或 code-style-facts-<lang>.md 中找到对应的事实/决策依据（不要求规范正文标注来源）
4. 可执行性
   - 每条“必须”规则是否配有来自本仓库的真实代码示例
   - 每个选项 / API / 级别是否给出“什么场景选哪个”的判据
   - 反模式是否标注“错误写法”、错因、推荐替代写法及真实代码示例
5. 范围边界
   - 是否混入 formatter/linter 可自动修复的格式细节
   - 是否混入构建 / 测试命令、覆盖率标准、lint 配置
6. 自洽性
   - 同一语言规范内部是否存在矛盾条款
   - 若提供了外部规范，是否只输出差异增量或完全忽略外部规范
   - 如引用了外部规范，相关内容是否与外部规范一致（本文件为替换用途，冲突时不需声明“以本文件为准”，但不应自相矛盾）
7. 历史例外完整性
   - 每个例外是否包含：场景、原因、是否允许新代码使用、迁移方式

修复规则：
- 发现不真实、不可执行、超范围、自相矛盾的条款时，直接删除或改写
- 缺失真实代码示例时，从 code-style-facts-<lang>.md 中补充并标注文件路径
- 缺失推荐替代写法时，从 gap-analysis-<lang>.md 中补充
- 不得引入输入文件未支持的新规则
- 不得削弱安全红线

输出要求：
- 更新后的 .claude/rules/coding-style-<lang>.md
- 验证报告 .claude/rules/coding-style-validation-<lang>.md，列出：检查项、结果、修复动作、残留问题汇总（CRITICAL / HIGH / MEDIUM / LOW 各几条）
```

### 通过标准

验证报告的“残留问题汇总”显示：
- 无 CRITICAL 问题
- 所有 HIGH 问题已修复或记录为已知例外
- 规范文件结构完整

## 完成汇报

所有阶段结束后，向用户汇报：
- 为哪些语言生成了规范
- 输出文件清单（`.claude/rules/coding-style-<lang>.md`）
- 每个规范的规则条数（必须 / 推荐 / 禁止 / 历史例外）——取自最终检查摘要
- 残留问题（如有）——取自最终验证报告

## 问题分级

- **CRITICAL**：会导致 bug/安全/稳定性问题的规则冲突或遗漏
- **HIGH**：显著影响规范可执行性或一致性的问题
- **MEDIUM**：影响可读性/完整性的问题
- **LOW**：格式或表述建议

## 推进模式

- **自动推进**：单个 agent 可独立完成的阶段内步骤
- **确认后推进**：跨越阶段、需要用户决策（如跳过已存在产物、确认冲突结论）
- **等待人工决策**：涉及规范基准选择、红线规则拍板

## 禁忌项

- 禁止直接套用通用模板而不结合仓库真实代码
- 禁止在规范中编造未经验证的量化标准
- 禁止削弱安全红线
- 禁止将 formatter/linter 可自动修复的格式细节写入规范
- 禁止将构建/测试命令、覆盖率标准、lint 配置写入规范
- 禁止引用不存在的前置产物而不提示用户
