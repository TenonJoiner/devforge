# devforge-simplify — 三维度深度清理

## 概述

基于 git diff 对批量变更做三 agent 并行评审（复用 / 质量 / 效率），由 developer 汇总修复。每次简化必须通过一个简单测试："新团队成员理解简化后的代码是否比原来更快？"

## 何时使用

- 批量变更后的深度重构（QA.2 FULL-REVIEW 完成后，对整个 feature 的完整 diff 执行）
- archive 前最终整理
- 独立使用：一批变更累积较大（>200 行）且代码评审后仍有明显异味

## 何时不使用

- 代码已经干净可读——不要为了简化而简化
- 还没理解代码做什么——先理解再简化
- 性能关键代码且"更简单"的版本会可测量地变慢
- 即将整体重写的模块——简化即将丢弃的代码是浪费
- 变更量 <50 行——直接在 code review 中处理即可

## 核心原则

### 1. 行为不变（Preserve Behavior Exactly）

不改变代码做什么——只改变它如何表达。所有输入、输出、副作用、错误行为、边界情况必须保持一致。

```
每次修复前自问：
→ 对所有输入是否产生相同输出？
→ 错误行为是否保持一致？
→ 副作用和执行顺序是否不变？
→ 所有现有测试是否无需修改即可通过？
```

### 2. Chesterton's Fence（先理解再动手）

在修改或移除任何代码前，先理解它为什么存在。如果看到一段"不合理"的代码：

1. 检查 git blame 了解原始上下文
2. 检查是否有 `// perf-critical`、`// simplify-ignore`、`// intentional` 等标记
3. 检查是否有相关注释解释 WHY
4. 如果仍不理解原因，标记为 "需确认" 而非直接修复

### 3. 遵循项目规范

简化意味着让代码更符合项目已有风格，而非引入外部偏好。参考 `coding-style-*.md` 中的语言特定规范。

### 4. 清晰优于精巧（Prefer Clarity Over Cleverness）

显式代码优于紧凑代码——当紧凑版本需要"心理暂停"才能解析时，它不是简化。

- 嵌套三元 → if/else 或 switch
- 链式 reduce + 内联逻辑 → 命名中间变量 + for 循环
- 单行 lambda 嵌套三层 → 提取为命名函数

**检验**：新团队成员能否不暂停地读懂这段代码？如果不能，它不够清晰。

### 5. 保持平衡（Maintain Balance）

简化有失败模式——过度简化。警惕以下陷阱：

- **过度内联**：移除了给概念命名的 helper，调用点反而更难读
- **合并无关逻辑**：两个简单函数合成一个复杂函数不是简化
- **移除"不必要"的抽象**：有些抽象是为了可测试性或可扩展性存在的，不是复杂性
- **优化行数**：行数少不是目标，理解速度才是

### 6. 增量修复 + 即时验证

每次只做一个简化，修复后立即跑测试。如果测试失败，**立即 revert 该修复并标记为 skip**，不要尝试修改测试来适配简化。

## 执行流程

### Step 1：识别变更范围

**目标**：获取本次简化的全部目标代码变更。

**场景识别**（按优先级）：

1. **显式指定**：用户指定了文件/目录 → 直接用指定范围
2. **Feature branch**：当前分支非默认分支（main/master），且存在分岔点 →
   ```bash
   DEFAULT_BRANCH=$(git remote show origin | grep 'HEAD branch' | awk '{print $NF}' 2>/dev/null || echo main)
   BASE=$(git merge-base HEAD $DEFAULT_BRANCH)
   git diff $BASE..HEAD
   ```
3. **默认分支 / 无分岔**：→ 询问用户意图：
   - 最近 N 个 commit 的变更
   - 未提交的 staged/unstaged 变更
   - 指定文件/目录

**合并未提交变更**：无论哪种场景，最后都检查是否有未提交的 staged/unstaged 变更，一并纳入

**排除规则**：
- 遇到 `simplify-ignore-start` / `simplify-ignore-end` 标记的代码块，跳过不评审
- 遇到 `// perf-critical`、`// intentional` 等标记，跳过不报；若标记本身可能过期，标记为"需确认"而非"建议修复"

> 避免直接使用无参数的 `git diff`——那只会看到未提交的变更，前面的 tasks 会被漏掉。

### Step 2：启动三评审 Agent 并行

同时启动 3 个 `code-reviewer` subagent，各获得完整 diff 和以下指令约束。通过 prompt 限定各自的评审维度，重载 code-reviewer 的默认评审视角。

**code-reviewer 派遣字段（必填，由 skill 注入）**：

| 字段 | 说明 |
|------|------|
| `任务模式` | `simplify 三维度评审`（复用/质量/效率 之一） |
| `主语言` | skill 从项目文件系统探测后注入（如 `C`/`Go`） |
| `评审维度` | 本 subagent 负责的单一维度（`code-reuse` / `code-quality` / `code-efficiency`） |
| `diff_range` | skill 计算后注入的完整 diff 范围 |
| `report_output_path` | 本 subagent 的报告路径（如 `/tmp/simplify-reuse-<ts>.md`） |

**所有 agent 的通用约束**：
- 对于看起来"不合理"的代码，先检查 git blame 或注释确认是否有意为之，再标记为问题
- 遇到 `simplify-ignore` 标记的代码块，直接跳过
- 只报告有具体修复建议的问题，不报告模糊的"感觉不好"
- 对 false positive 宁可漏报也不误报

#### Agent 1：复用检查（Code Reuse Review）

**目标**：检查新来的代码是否重复造轮子。

1. 搜索现有 utilities / helpers / 公共模块（重点关注 `src/utils/`、`src/common/`、相邻模块）
2. 指出与已有函数仅参数/命名略有差异的新增函数
3. 指出手造的字符串/路径/环境处理逻辑，检查是否可用现有工具替代
4. 指出重复出现的 >3 行逻辑块（出现 >2 处）

**输出格式**：
```markdown
### 复用问题
- `[file:line]` 问题描述
  - **现有替代**：`src/xxx/yyy.c:func_name()`
  - **建议**：直接调用现有函数 / 提取公共辅助函数
```

#### Agent 2：质量检查（Code Quality Review）

**目标**：发现 hacky 模式、过度设计、坏味道。

1. **冗余状态**：新加的 state 是否可由已有 state 派生
2. **参数膨胀**：是否通过新增参数解决，而非重构数据结构
3. **复制粘贴**：仅命名/条件略有差异的重复代码块
4. **泄露抽象**：暴露了内部细节，或破坏了已有抽象边界
5. **过度包装**：无意义的中间层、冗余宏、空函数转发
6. **无意义注释**：解释了 WHAT 而非 WHY 的注释

**领域特有加项**：从代码结构推断检查重点。Agent 应扫描以下信号识别领域：
- 源码目录结构（如 `src/engine/`、`src/protocol/`、`src/optimizer/` 等子系统名）
- 核心数据结构名和模块间调用关系
- 变更涉及的已有子系统（对照 `docs/architecture/` 或源码中的模块边界）

结合推断出的领域，识别常见质量反模式。例如：
- 数据库：查询优化器执行计划缓存、索引批量更新、锁粒度与并发控制
- 分布式系统：序列化/反序列化延迟与批量、故障恢复全量 vs 增量
- 编译器：AST 遍历合并、优化 pass 延迟执行
- 存储系统：元数据路径与数据路径混合、存储引擎层间泄漏、WAL 与刷盘语义不一致
此列表仅为启发示例，Agent 不应局限于这些条目。

**输出格式**：
```markdown
### 质量问题
- `[file:line]` 问题描述
  - **类型**：冗余状态 / 参数膨胀 / 复制粘贴 / 泄露抽象 / 过度包装 / 无意义注释
  - **建议**：...
```

#### Agent 3：效率检查（Efficiency Review）

**目标**：发现性能陷阱和不必要的开销。

1. **不必要的工作**：循环内重复计算、重复文件读取、重复内存分配
2. **热路径膨胀**：新的阻塞调用/锁/分配是否加在了每次请求/IO 都走的路径上
3. **锁粒度**：全局锁保护细粒度操作、锁内嵌套过深
4. **批处理机会**：逐条 IO/请求是否可改为批量提交
5. **内存**：循环内的大缓冲区分配、无界数据结构增长、泄漏风险
6. **IO 开销**：预检查文件存在性再操作（TOCTOU）、读全文件只需部分

**领域特有加项**：从代码结构推断检查重点。Agent 应扫描以下信号识别领域：
- 源码目录结构（如 `src/engine/`、`src/protocol/`、`src/optimizer/` 等子系统名）
- 核心数据结构名和模块间调用关系
- 变更涉及的已有子系统（对照 `docs/architecture/` 或源码中的模块边界）

结合推断出的领域，识别常见效率反模式。例如：
- 数据库：查询优化器执行计划缓存、索引批量更新、零拷贝替代 memcpy
- 分布式系统：序列化/反序列化延迟与批量、故障恢复全量 vs 增量
- 编译器：AST 遍历合并、优化 pass 延迟执行
- 存储系统：元数据路径与数据路径混合、存储引擎层间泄漏、WAL 与刷盘语义不一致
此列表仅为启发示例，Agent 不应局限于这些条目。

**输出格式**：
```markdown
### 效率问题
- `[file:line]` 问题描述
  - **场景**：热路径 / 初始化路径 / 故障恢复路径
  - **影响**：延迟增加 / 吞吐下降 / 内存膨胀
  - **建议**：...
```

### Step 3：汇总与修复

主 Agent 等待三个 subagent 返回后：

1. **去重合并**：相同位置的问题合并为一条
2. **评估价值**：每个问题评估修复价值（高/中/低）和风险（行为变更风险）
3. **逐个修复**：
   - 每次只做一个简化
   - 修复后运行**相关模块的单元测试**（而非全量测试，避免编译/测试周期过长）
   - 测试通过 → 继续下一个
   - **测试失败 → 立即 revert，标记为 skip，不修改测试来适配简化**
4. **大规模变更策略**：如果修复量超过 500 行，优先使用自动化工具（codemod/sed/AST transform）而非手动逐行修改
5. **false positive 处理**：简单标注并跳过，不展开争论

### Step 4：验证

全部修复完成后，按以下顺序执行验证。任何一步失败则停止，不继续后续步骤。

**第一阶段：构建**
- [ ] 构建成功，无新增 warning

**第二阶段：测试**
- [ ] 单元测试通过（无需修改任何测试）
- [ ] L1 集成测试 / 冒烟测试通过（验证基本交互和关键路径，无需跑完整集成测试）

**第三阶段：静态分析 + 内存/并发安全**
- [ ] linter/formatter 通过（无格式回归）
- [ ] 内存/并发安全检查通过（根据语言选择工具，参考 `coding-style-*.md` 中的"内存检查工具"表）

**第四阶段：代码质量**
- [ ] 每个简化是可独立 review 的增量变更
- [ ] diff 干净——无无关变更混入
- [ ] 简化后的代码遵循项目规范（`coding-style-*.md`）
- [ ] 未移除或削弱任何错误处理
- [ ] 未遗留死代码（未使用的 import、不可达分支）

## 常见自我合理化（Agent 判断校准）

| 合理化借口 | 现实 |
|---|---|
| "代码能跑就不要动" | 难读的代码在下次出 bug 时会更难修。现在简化为未来每次变更节省时间 |
| "行数越少越简单" | 一行嵌套三元不比五行 if/else 简单。简单性是理解速度，不是行数 |
| "顺便把旁边的代码也清理了" | 无范围的简化制造噪音 diff，增加回归风险。保持聚焦 |
| "原作者一定有原因" | 也许有。检查 git blame——应用 Chesterton's Fence。但累积的复杂性往往没有原因，只是压力下迭代的残留 |
| "这个抽象以后可能有用" | 不要保留投机性抽象。现在没用就是无价值的复杂性。移除，需要时再加 |
| "重构的同时加个功能" | 重构和功能开发分开。混合变更更难 review、revert 和理解历史 |

## 红旗信号

### 评审侧红旗

- 三个 agent 同时指出同一文件有问题（高置信度，优先修复）
- 热路径上出现了 `malloc` 或锁
- 新文件/新函数的 50% 以上代码在别处已存在

### 修复侧红旗（立即停止）

- 简化后需要修改测试才能通过——说明改变了行为，必须 revert
- "简化"后代码反而更长更难懂——revert，原版更好
- 移除了错误处理因为"让代码更干净"——错误处理不是复杂性
- 简化了自己不完全理解的代码——先理解再动手
- 批量修改难以逐个 review——拆分为更小的独立修复
