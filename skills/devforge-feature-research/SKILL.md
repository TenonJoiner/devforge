# devforge-feature-research — 特性级研究

## 概述

特性级研究是 OpenSpec workflow 中 proposal 与 design 之间的桥梁。本 skill 产出**标杆方案分析报告**，包含：
1. **背景与目标**：从 proposal 提炼问题与标杆选择理由
2. **标杆方案分析**：按产品组织，每个标杆的完整方案（含模块架构图）
3. **关键决策维度对比**：横向对比 + 约束影响分析

**职责边界**（重要）：
- ✅ Research 提供**素材**：标杆方案、关键维度对比、约束识别
- ❌ Research **不做决策**：不推荐方案、不写实现细节（并发/内存/错误处理留给 design）

**核心原则**：
1. **可读性优先**：报告给人看，要有叙事性，保持标杆整体性，不要把方案拆散到多个章节
2. **篇幅可控**：按特性复杂度弹性调整，不做长篇大论
3. **约束嵌入**：约束直接作用于维度对比，不单独列清单
4. **即时确认**：未知约束立即用 AskUserQuestion 确认，不带问题往下跑
5. **图示辅助**：每个标杆 1 个模块架构图，最核心机制可加 1 个时序图

## 工作目录约定

skill 在**当前工作目录**查找输入文件、输出产出文件：
- **输入**：`proposal.md`（必需）
- **输出**：`research.md`
- **产品级文档**：通过项目根目录的 CLAUDE.md#产品级文档索引定位

**调用方式**：
- **OpenSpec workflow 调用**：workflow 先 `cd openspec/changes/<name>/`，然后调用 skill
- **手动调用**：用户先 `cd` 到包含 `proposal.md` 的目录，然后调用 `/df:research`

## 启动检测

检查当前工作目录的 `research.md`：
- **不存在** → 进入「初次生成」模式
- **已存在** → 反问主人「修订 / 补全」，按指定模式运行

如果当前工作目录无 `proposal.md`，立即报错并提示主人 `cd` 到正确目录。

## 篇幅控制

按特性复杂度弹性调整总篇幅：

| 复杂度 | 标杆数 | 每标杆篇幅 | 对比章节 | 总篇幅 |
|--------|--------|----------|---------|--------|
| 简单（单一算法/优化）   | 2     | 100-150 行 | 80-120 行  | 200-350 行 |
| 中等（缓存机制/调度）   | 2-3   | 150-200 行 | 100-200 行 | 350-600 行 |
| 复杂（分布式一致性）    | 3     | 200-250 行 | 150-300 行 | 600-900 行 |

**复杂度判断**：researcher agent 读 proposal 后自动判断，无需主人介入。

## 初次生成模式

### [1] 上下文准备

准备以下输入（无需 Read 工具，只是明确路径）：
1. **proposal.md**：本特性的动机、范围、Capabilities
2. **产品级文档索引**：CLAUDE.md#产品级文档索引（已在上下文）
3. **research.md template**：`openspec/schemas/spec-driven-enhanced/templates/research.md`

### [2] 标杆产品识别

派遣 1 个 researcher agent，任务：
- 读 proposal.md，理解特性的核心问题和技术挑战
- 识别 **2-3 个**相关标杆产品（开源项目 / 论文 / 商业产品）
- 每个标杆产品标注**互补性说明**（不是穷举，而是覆盖方案空间的不同范式）
- **数量控制**：
  - **2 个**：必须互补，覆盖方案空间两端（如弱一致 + 强一致、有锁 + 无锁）
  - **3 个**：覆盖核心范式（如 timeout + callback + DLM 三大一致性范式）
- **选择标准**：
  - 代表性强：业界主流方案，有广泛应用
  - 设计思路清晰：有详细文档或源码可追溯
  - **互补而非穷举**：同类标杆只选 1 个代表（如 CephFS 与 AFS 同属 callback 范式，选一个）
- **优先级规则**（高到低）：
  1. 业界主流 + 源码可得（如 Linux 内核子系统、知名开源项目）
  2. 学术论文 + 商业验证（如 OSDI/SOSP 论文 + 生产实践）
  3. 小众开源但设计清晰（如专项优化项目）

写入临时文件 `research-benchmarks-draft.md`，返回标杆产品数与互补性说明。

### [2.1] 标杆产品确认（必须确认）

主会话读取 `research-benchmarks-draft.md`，提取标杆清单（产品名 + 互补性说明），使用 AskUserQuestion 工具询问：

**问题**：「以下 2-3 个标杆是否合适用于研究本特性？」

**展示内容**：
- 标杆 1: [产品名]（代表 [范式/路线]，[互补性说明]）
- 标杆 2: [产品名]（代表 [范式/路线]，[互补性说明]）
- ...

**选项**：
- ✅ 确认，继续深度研究
- 🔄 调整标杆清单（说明需要补充/替换/删除哪些标杆）

**处理**：
- 选择「确认」→ 进入步骤 [3]
- 选择「调整」→ 根据主人指令修改 `research-benchmarks-draft.md`，回到本步骤重新确认

### [3] 标杆方案研究（按标杆并发）

对每个标杆产品，派遣 1 个 researcher agent（并发上限 3），任务：

**任务核心**：保持方案的整体性，让评审者能完整理解一个标杆的设计哲学。**不要**把方案拆散到多个孤立的"技术关注点"。

**输出结构**（与 template §2 对齐）：

1. **方案全景**（1-2 段）：用一段话说清楚该标杆的整体设计思路（如"NFSv4 = timeout 弱一致兜底 + delegation 可选升级 + COMPOUND 摊销"）

2. **整体架构图**（必须，1 个）：
   - **类型**：模块层级图（纵向分层），不是横向交互时序图
   - **内容**：客户端内部模块组织、协议边界、与服务端接口
   - **格式**：ASCII Art 方框图，10-20 行
   - **示例**：
     ```
     ┌─────────────┐
     │  VFS Layer  │
     ├─────────────┤
     │  Cache      │
     ├─────────────┤
     │  RPC        │
     └─────────────┘
          ↓
     [Server]
     ```

3. **核心机制**（2-3 个）：每个机制三层拆解
   - **原理**：核心思想，解决什么问题
   - **设计**：关键数据结构 / 控制流（不贴源码，最多引用类名/字段名）
   - **取舍**：量化 trade-off（性能数字 + 适用 + 不适用）

4. **交互流程图**（可选，0-1 个）：
   - 仅在**最能体现该标杆特色**的机制下加 1 个时序图
   - 类型：时序图（横向交互），10-20 行
   - 如果核心机制是纯本地算法（如 LRU），可不画
   - **不要**每个机制都画时序图

5. **适用场景与坑点**（1-2 段）：浓缩，1-2 个最关键坑点

**篇幅约束**：根据特性复杂度（简单 100-150 行 / 中等 150-200 行 / 复杂 200-250 行），由 agent 自行判断。

**产出格式约束**：
- 用设计语言描述，不贴源码（最多引用关键函数名/数据结构名）
- 量化 trade-off：必须有具体数字（如"延迟降低 60%"），避免模糊描述
- 精选坑点：1-2 个最关键的，不罗列所有问题
- **可读性**：报告给人看，叙事流畅，避免干枯罗列

产出：每个标杆一个 `research-benchmark-<product>-draft.md`

### [4] 对比分析生成

派遣 1 个 researcher agent，任务：

**子任务 1：提炼核心决策维度**
- 读所有标杆 draft
- 提炼 **3-5 个核心决策维度**（如"一致性机制"、"缓存粒度"、"淘汰策略"）
- 维度选择标准：
  - 跨标杆对比有意义（≥2 个标杆有不同选择）
  - 直接影响方案路线（如选 timeout 还是 callback 是路线分叉点）
- **不要**拆得过细（不要 8 个维度），聚焦本质决策点

**子任务 2：约束收集**
- 从标杆研究中提取关键约束维度（如"客户端规模"、"网络拓扑"、"一致性窗口"）
- 检索 proposal.md 与产品级文档：
  - 命中 → 提炼约束（附来源：文件路径 + 章节锚点）
  - 未命中 → 标记为"未知约束"，记录"待主人确认的具体问题"
- **约束筛选**：只保留**直接影响维度选择**的约束（控制在 5-10 条，不追求穷举）

**子任务 3：未知约束即时确认（条件触发）**

如果识别出未知约束（≥1 条），主会话使用 AskUserQuestion 工具**逐条确认**：

**确认策略**（按重要性筛选，不按数量截断）：
- 只确认"决定方案路线分叉的约束"——即不同取值会导致选不同标杆/不同方案的约束
- 不确认"仅影响调参不影响路线选择的约束"（如缓冲区大小、超时数值），这些留给 design 阶段
- 筛选后的关键约束通常 ≤ 3 条；若 > 5 条，提示主人"本特性约束过多，建议先做约束梳理再继续研究"

**对每条未知约束**：
- 问题：[约束维度]的具体取值？（如"客户端规模上限大概多少？"）
- 选项：
  - 提供 2-3 个典型范围（如"< 百级"、"千级"、"万级+"）
  - "暂时不确定，留作条件分支"

**处理**：
- 主人给出具体值 → 标记为"已确认约束"，更新约束清单
- 主人选"暂时不确定" → 保留为"条件分支"，在对比分析中用"如果 X=A 则..."表达

**子任务 4：对比分析撰写**

按维度组织，每个维度包含：

1. **标杆对比表格**：标杆 / 方案 / Trade-off / 适用场景 4 列
2. **约束影响分析**：把约束直接嵌入对比，**按对该维度方案选择的影响程度从高到低排序**（不按来源分类）：
   - 每条约束统一格式：`**[约束内容]**（来源：proposal.md#章节 或 生成期确认） → 影响：[如何影响该维度的方案选择]`
   - 排在最前的是"决定方案路线分叉"的约束（如本维度选 timeout 还是 callback）
   - 排在中间的是"收窄候选范围"的约束（如排除某些方案）
   - 排在最后的是"影响调参不影响路线"的约束（如性能目标、容量预算）
   - 条件分支（如有未确认约束）：在受影响的约束行后用"如果 X=A 则...，如果 X=B 则..."表达

**禁止**：
- ❌ 单独列"约束清单"章节（约束必须嵌入维度对比）
- ❌ 推荐方案（不写"建议选 X"，让 design 决策）
- ❌ 写实现细节（并发/内存/错误处理留给 design）

**写入**：当前工作目录 `research.md` 的 §1 §2 §3 §引用 章节。

### [5] 评审循环（最多 3 轮）

派遣 1 个 architect-reviewer agent，任务：
- 读 research.md 全文
- 检查评审维度（见下）
- 产出：问题清单（CRITICAL / HIGH / MEDIUM / LOW）

**评审维度**（视角清单）：
1. **可读性与叙事**：
   - 标杆整体性是否保持（不被拆散到多个章节）？
   - 报告结构是否清晰（背景 → 标杆 → 对比）？
   - 篇幅是否符合复杂度档位？
2. **图示完整性**：
   - 每个标杆是否有 1 个模块架构图（不是时序图）？
   - 时序图（如有）是否仅出现在最核心机制下，不滥用？
3. **维度对比质量**：
   - 维度数是否合理（3-5 个，不过度拆解）？
   - 标杆对比表格是否清晰（4 列：标杆 / 方案 / Trade-off / 适用场景）？
   - Trade-off 是否量化（具体数字而非模糊描述）？
4. **约束嵌入**：
   - 约束是否嵌入维度对比，而非单独列清单？
   - 约束是否按"对该维度方案选择的影响程度"从高到低排序（不按来源分类）？
   - 每条约束是否有来源标注（proposal.md#章节 或 生成期确认）？
   - 未知约束是否已通过 AskUserQuestion 确认（已确认）或显式标注为条件分支？
5. **职责边界**：
   - 是否未做方案推荐（不越界 design 职责）？
   - 是否未写实现细节（并发/内存/错误处理留给 design）？

**问题分值**：CRITICAL=10 分, HIGH=3 分, MEDIUM=1 分, LOW=0.1 分
**结构性元素总数**：标杆数 + 维度数（如 3 标杆 + 4 维度 = 7）
**缺陷密度** = 问题分数之和 / 结构性元素总数

**通过门禁**：无 CRITICAL + 缺陷密度 ≤ 0.3

- 通过 → 进入 [6]
- 不通过 → 派遣 1 个 researcher agent 修正，回到本步骤重新评审
- 3 轮后仍未通过 → 进入 [5.1] 带债推进确认

### [5.1] 带债推进确认（仅在 3 轮评审后未通过时触发）

主会话读取最终 review，提取残留问题清单与缺陷密度，使用 AskUserQuestion 询问：

**问题**：「评审 3 轮后仍未通过，是否接受带债推进？」

**展示内容**：缺陷密度 + CRITICAL/HIGH/MEDIUM/LOW 计数 + 残留问题标题

**选项**：
- ✅ 接受带债推进（在 research.md 开头标注残留问题）
- 🔄 继续修正（说明修正方向，进入第 4 轮）
- ❌ 暂停（终止 skill）

**带债推进格式**：在 research.md 开头插入

```markdown
## 0. 评审残留问题（带债推进）

评审轮次：3/3（未通过）
缺陷密度：X.XX

### CRITICAL
- [C1] ...

### HIGH
- [H1] ...
```

### [6] 落地输出与清理

**清理中间产物**（评审通过或带债推进后）：

```bash
rm -f research-benchmarks-draft.md
rm -f research-benchmark-*-draft.md
rm -f research-review.md
```

**最终目录只保留**：`proposal.md` + `research.md`

**终端汇报**：
- 文件位置：research.md
- 总行数 + 复杂度档位（简单/中等/复杂）
- 标杆产品数（2-3 个）
- 核心决策维度数（3-5 个）
- 约束统计：proposal 来源 N 条 / 生成期确认 N 条 / 条件分支 N 条
- 评审结果：通过（密度 X.XX）/ 带债推进

## 增量更新模式

反问主人「想更新哪一块」，提供选项：
1. 更新背景与目标（重写 §1）
2. 更新标杆方案（补充/替换/删除标杆）

根据主人选择，只跑对应范围的生成阶段，merge 结果回 research.md，不动其他章节。评审循环只检查变更范围。

## Agent 派遣 Prompt 模板

### researcher agent（标杆产品识别）

```
当前是特性级 research 阶段，识别相关标杆产品。

**任务**：识别 2-3 个相关标杆产品，强调互补性而非穷举。

**输入**：
- proposal.md：当前工作目录
- 产品级文档索引：CLAUDE.md#产品级文档索引

**output_path**：`research-benchmarks-draft.md`（当前工作目录）

**输出要求**：
标杆产品清单（2-3 个），每个标杆包含：
- 产品名称（开源项目 / 论文 / 商业产品）
- 代表的范式/路线（如"timeout 弱一致代表"、"callback 强一致代表"）
- 互补性说明（与其他标杆形成什么对比）
- 可追溯性（源码位置 / 论文链接 / 文档链接）

**数量控制**：
- 2 个：必须互补，覆盖方案空间两端
- 3 个：覆盖核心范式
- **不要 4+ 个**，特性级研究深度优先于广度

**选择标准**：
- 代表性强：业界主流方案
- 互补而非穷举：同类标杆只选 1 个代表
- 设计清晰：有详细文档或源码可追溯
```

### researcher agent（标杆方案研究）

```
当前是特性级 research 阶段，研究标杆产品「<product>」。

**任务**：产出该标杆在本特性领域的完整方案分析报告，**保持方案整体性**。

**输入**：
- proposal.md：当前工作目录
- 标杆产品源码 / 论文 / 官方文档

**输出结构**（与 research.md template §2 对齐）：

1. **方案全景**（1-2 段）：一段话概括完整方案

2. **整体架构图**（必须，1 个）：
   - 类型：**模块层级图**（纵向分层），不是横向交互时序图
   - 内容：客户端内部模块组织、协议边界
   - 格式：ASCII Art 方框图，10-20 行

3. **核心机制**（2-3 个）：每个三层拆解
   - 原理：核心思想，解决什么问题
   - 设计：关键数据结构 / 控制流（不贴源码，最多引用类名/字段名）
   - 取舍：量化 trade-off（具体数字 + 适用 + 不适用）

4. **交互流程图**（可选，0-1 个）：
   - 仅在最能体现该标杆特色的机制下加 1 个时序图
   - 10-20 行
   - 如果核心机制是纯本地算法可不画

5. **适用场景与坑点**（1-2 段）：1-2 个最关键坑点

**篇幅**：
- 简单特性：100-150 行
- 中等特性：150-200 行
- 复杂特性：200-250 行

**质量约束**：
- 量化优先：trade-off 必须有具体数字
- 整体性：方案的各部分相互呼应，不要写成孤立技术点列表
- 可读性：叙事流畅，避免干枯罗列

**禁忌**：
- ❌ 不贴源码（最多引用关键函数名/数据结构名）
- ❌ 不写过多边缘细节
- ❌ 不要每个机制都画时序图

```
当前是特性级 research 阶段，评审 research.md。

**被评审对象**：research.md（当前工作目录）
**被评审 template_path**：openspec/schemas/spec-driven-enhanced/templates/research.md
**review_output_path**：`research-review.md`（当前工作目录，多轮追加同一文件）

**评审维度**（视角清单）：

1. **可读性与叙事**：
   - 标杆整体性是否保持（不被拆散到多个章节）？
   - 报告结构是否清晰（背景 → 标杆 → 对比）？
   - 篇幅是否符合复杂度档位？

2. **图示完整性**：
   - 每个标杆是否有 1 个模块架构图（纵向分层，不是时序图）？
   - 时序图（如有）是否仅出现在最核心机制下，不滥用？

3. **维度对比质量**：
   - 维度数是否合理（3-5 个，不过度拆解）？
   - 标杆对比表格是否清晰（4 列：标杆 / 方案 / Trade-off / 适用场景）？
   - Trade-off 是否量化（具体数字而非模糊描述）？

4. **约束嵌入**：
   - 约束是否嵌入维度对比，而非单独列清单？
   - 约束是否按"对该维度方案选择的影响程度"从高到低排序（不按来源分类）？
   - 每条约束是否有来源标注（proposal.md#章节 或 生成期确认）？
   - 未知约束是否已确认（已确认）或标注为条件分支？

5. **职责边界**：
   - 是否未做方案推荐（不越界 design 职责）？
   - 是否未写实现细节（并发/内存/错误处理留给 design）？

**问题分值**：CRITICAL=10 分, HIGH=3 分, MEDIUM=1 分, LOW=0.1 分
**缺陷密度** = 问题分数之和 / (标杆数 + 维度数)

**输出**：问题清单（CRITICAL / HIGH / MEDIUM / LOW），计算缺陷密度。

## 禁忌项

- ❌ 标杆产品 ≥ 4 个（特性级研究 2-3 个足够）
- ❌ 单独列"约束清单"章节（约束必须嵌入维度对比）
- ❌ 维度 > 5 个（过度拆解，丧失对比价值）
- ❌ 推荐方案（research 提供素材，不做决策）
- ❌ 写实现细节（并发/内存/错误处理留给 design）
- ❌ 每个机制都画时序图（只在最核心机制下加 1 个，可选）
- ❌ 整体架构图画成横向交互时序图（应是纵向分层模块图）
- ❌ 跳过未知约束的即时确认，带问题往下跑
- ❌ 评审完成后不清理中间产物
