# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在本仓库中工作时提供指导。

## 项目概述

**teamskills** 是面向大型复杂基础软件开发团队的统一技能仓库，解决团队级 AI 辅助开发工作流的标准化问题。

**适用范围**：
- 分布式系统（存储、计算、消息队列、数据库）
- 操作系统/内核
- 编译器/运行时
- 网络协议栈
- 虚拟化/容器
- 高性能计算
- 复杂嵌入式系统

**不适用**：简单的 Web CRUD 应用、小型移动 App、原型项目

产品级使用启发式 Skills 辅助思考（与 OpenSpec 脱离），特性级基于 OpenSpec 的规范驱动开发，代码级和测试验证级使用 Claude Code 原生扩展。构建覆盖产品规划到代码交付全流程的 skill 体系。

## 核心定位

- **团队协作优先**：目标是团队整体提效和高效协作，减少并行开发冲突，而非个人开发者效率工具
- **分层驱动**：产品级用启发式 skill 辅助创意迭代（重输出质量），特性级用 OpenSpec 引擎驱动工作流（重流程控制）
- **统一技能仓库**：采用 Monorepo 模式，teamskills 作为产品级统一仓库，所有子系统代码位于同一仓库中，通过 `.claude/` 目录直接使用技能

## 设计原则

1. **四层分层驱动**：产品级（启发式 skill，重输出质量）→ 特性级（OpenSpec workflow，重流程控制）→ 代码级（TDD/评审/重构）→ 测试验证级（集成/性能测试）。上层约束下层，下层变更反馈上层
2. **团队协作 + Monorepo**：所有 skill 围绕多人并行开发场景设计，通过 `.claude/` 目录统一管理，无需额外安装
3. **用现成工具，不造轮子**：使用 Claude Code 内置能力、Linux 系统工具和外部成熟工具，不自研需要长期维护的工具链
4. **领域适配**：通过 `/df:define` 和 `/df:design` 的阶段 0 确定具体系统类型（分布式系统/OS/编译器/网络栈等）和编程语言（C/C++/Rust/Go/Python/Java），后续流程自动适配。以数据一致性、高性能、内存安全、并发正确性、错误处理完备性为第一优先级
5. **中文写作**：所有 skill、command、agent、hook 及相关文档均使用中文编写
6. **MCP 可选引入**：优先选择免费可用、无需翻墙的 MCP 服务，引入时注明名称、用途及访问方式

## Skill 多 Agent 深度协作设计规范

本规范约束所有 SKILL.md 的 Agent 调度设计，目标是通过多 agent 并行与分工，增加思考深度和广度，提升交付件质量上限。

### 核心信条

**交付件质量的上限，由产出阶段的 agent 数量与独立性决定；评审只能发现已有框架内的缺陷，无法弥补框架本身的盲区。**

### 一、复杂度驱动原则（首要原则）

Agent 数量不由"阶段数量"决定，而由**输出件复杂度**决定。

**复杂度是语义属性，不是语法属性。** 判定依据是文档承载的决策权重和技术交叉度，而非篇幅行数。一份 80 行的 ADR（如一致性协议选型）复杂度远高于一份 500 行的接口清单罗列。

#### 内容类型矩阵（核心判定依据）

| 内容类型 | 基准复杂度 | 说明 |
|---------|-----------|------|
| **架构决策**（ADR 级别） | **极高** | 影响多个子系统、一旦确定难以回退、涉及核心技术选型 |
| **系统级架构设计** | **极高** | 定义子系统边界、交互契约、全局数据流 |
| **需求核心 Feature / Actor 定义** | **极高** | 定义产品边界、Actor 权责、验收标准 |
| **子系统级设计** | **高** | 单一子系统内多技术域交叉（如存储引擎需同时考虑 WAL + 索引 + 缓存） |
| **接口 / 契约设计** | **高** | 跨模块边界、变更成本高、需兼容演进 |
| **Scenario / 用例设计** | **中等** | 基于已确定 Feature 的展开、有明确输入输出 |
| **调研报告** | **中等 / 低** | 信息整理、对比分析、不产生新的设计决策 |
| **模板化文档**（如迭代计划格式填充） | **低** | 结构固定、内容基于已有信息组合 |

#### 决策属性上调法（辅助判定）

文档若包含以下属性，复杂度在基准上上调：

| 属性 | 上调规则 |
|-----|---------|
| 跨子系统边界 | 复杂度 +1 级 |
| 不可逆决策 | 复杂度 +1 级 |
| 性能 / 一致性关键路径 | 复杂度 +1 级 |
| 无现成方案（需原创设计） | 复杂度 +1 级 |
| 安全 / 合规敏感 | 复杂度 +1 级 |

> 示例：一份 80 行的 ADR，若涉及跨子系统 + 不可逆 + 一致性关键路径，基准复杂度「极高」不变；一份 300 行的调研报告，若命中上述属性，复杂度从「中等」上调至「高」。

#### 复杂度与 agent 配置对照

| 复杂度 | 最小并行产出 agent 数 | 最小独立评审 agent 数 |
|--------|---------------------|---------------------|
| **极高** | **≥3** | **≥2** |
| **高** | **≥2** | **≥2** |
| **中等** | 1-2 | 1-2 |
| **低** | 1 | 1 |

#### 篇幅的辅助作用：异常检测

篇幅**不作为复杂度判定依据**，但用于发现异常：

- 高/极高复杂度文档篇幅过短（<100 行）：红旗——可能分析深度不足，需检查是否遗漏关键决策依据
- 低复杂度文档篇幅过长（>400 行）：红旗——可能信息密度过低，存在冗余或模板化填充
- 篇幅与同类文档历史均值偏差 >50%：提示复核，确认是合理差异还是质量问题

### 二、Plan 阶段原则

在极高/高复杂度场景下，**必须先解构再发散**。产出阶段之前的轻量级 Plan 用于确保视角覆盖的完整性，避免产出 agent 基于错误前提进行同构思维。

#### 何时需要 Plan 阶段

| 复杂度 | 是否需要 Plan | Plan 形式 |
|--------|--------------|-----------|
| **极高** | **必须** | 派遣 1 个 architect agent 做任务解构（切分维度、视角分配、关键假设清单、评审视角缺口分析） |
| **高** | **建议** | 主会话基于上下文自评估，或派遣 1 个 agent 快速确认 |
| **中等 / 低** | 不需要 | — |

#### Plan 阶段的边界

Plan 阶段**只决定"谁分析什么"**，不产出分析结论：

- ✅ 确定切分维度（按子系统 / 分析类型 / 职能侧重 / 范围深度）
- ✅ 分配每个并行 agent 的视角约束（不同角色、不同分析范式、不同约束侧重）
- ✅ 列出需要被显式质疑的关键假设
- ✅ 识别产出 agent 未覆盖的视角，指导评审 agent 配置
- ❌ 不产出深度分析内容
- ❌ 不预设定最终结论或倾向性方案

#### Plan 阶段与主会话职责的关系

轻量级的任务解构和视角分配属于**调度职责**，不越界。主会话可以基于已有上下文直接完成高复杂度的 Plan；极高复杂度建议派遣 1 个 architect agent 执行。

### 三、视角独立性原则

多 agent 并行必须是**真正独立的视角**，禁止以下替代：

- **禁止**一个 agent "内部发散多个方案"替代多个 agent 独立产出
- **禁止**多个 agent 使用同一 prompt 模板、同一输入源、同一分析框架
- **禁止**一个 agent 产出后，其他 agent 仅做"润色"或"格式调整"

**正确做法**：每个并行 agent 应从不同维度出发：
- 不同专业角色（architect vs product-manager vs researcher）
- 不同分析范式（自顶向下 vs 自底向上）
- 不同约束侧重（性能优先 vs 可维护性优先 vs 交付速度优先）
- 不同范围切分（按子系统、按特性域、按场景类型）

#### 预定义角色铁律

所有子 agent**必须是预定义角色**，禁止在主会话中临时构造 agent：

- **禁止**在调度 prompt 中临时描述 agent 角色（如"你是一个擅长性能分析的助手..."）
- **禁止**为单次任务创造新的 agent 类型名
- SKILL.md 中调度 agent 时只能使用 `.claude/agents/` 中已定义的 agent 名称

> 预定义角色确保 agent 行为的一致性和可复现性。临时构造的角色无法积累质量反馈，也无法在多轮迭代中保持一致的行为边界。

#### 同角色多实例区分规则

当同一预定义角色需要多个实例并行时，**必须通过 prompt 明确区分关注点**：

```
agent: researcher（关注一致性协议选型）
agent: researcher（关注存储引擎实现方案）
agent: researcher（关注网络传输层优化）
```

区分方式（选择其一或组合）：
- **按范围**：明确指定分析的子系统/模块/文件范围
- **按约束侧重**：如"性能优先视角"vs"可维护性优先视角"
- **按分析范式**：如"自顶向下架构推导"vs"自底向上实现约束推导"

**禁止**：同一角色的多个实例使用完全相同的 prompt，这将退化为"一个 agent 内部发散"的等价形式。

### 四、产出-评审对称原则

评审 agent 的视角丰富度必须**覆盖或超过**产出 agent 的视角丰富度。

| 产出阶段 agent 配置 | 评审阶段最低 agent 配置 | 说明 |
|-------------------|----------------------|------|
| 1 个 agent（单一视角） | **至少 2 个不同角色的 reviewer** | 发现单一视角的结构性盲区 |
| 2 个并行 agent（双视角） | **至少 2 个 reviewer，其中 1 个视角与产出不同** | 确保有外部视角审视整合结果 |
| 3 个并行 agent（多视角） | **至少 2 个 reviewer** | 整合后的复杂度可能引入新问题 |

**禁止模式**："1 个 agent 产出 + 3 个 reviewer 评审"——产出阶段的结构性缺陷无法通过评审阶段的丰富度来弥补。

### 五、并行最大化原则

无依赖的任务**必须并行**，不得以"流程描述简洁"为由串行化。

**必须并行的场景**：
- 多个独立文档的产出（如多个标杆研究、多个特性域需求、多个子系统架构）
- 多个独立维度的调研（如一致性协议、存储引擎、网络模型）
- 多个独立评审（如 architect-reviewer、pm-reviewer 同时评审不同文档）

**判断依赖的方法**：如果两个任务的输出互不引用对方的结论，即可并行。

### 六、拆分维度原则

复杂任务拆分时应选择以下维度之一，确保拆分后的子任务对单个 agent 而言仍是**完整的、有上下文的**：

| 拆分维度 | 适用场景 | 示例 |
|---------|---------|------|
| **按领域/子系统** | 各子系统相对独立 | 元数据子系统、存储子系统、网络子系统分别设计 |
| **按分析类型** | 同一对象需要多角度分析 | 正常路径 agent + 故障/异常路径 agent |
| **按职能侧重** | 同一文档的不同章节需要不同专业能力 | 结构规划 agent + 量化估算 agent + 风险分析 agent |
| **按范围深度** | 核心 vs 补充 | 核心 Feature agent + 边缘/扩展 Feature agent |

**禁止的拆分方式**：
- **禁止**按"前半部分/后半部分"机械拆分（agent 失去上下文）
- **禁止**按"写文档/改格式"拆分（后者不是 agent 工作）
- **禁止**过度拆分导致整合成本超过并行收益（>3 个并行 agent 产出同一份文档时需谨慎）

### 七、修正阶段多 Agent 原则

评审修正循环中，修正工作量的大小决定修正 agent 的配置：

| 修正工作量 | 判断标准 | agent 配置 |
|-----------|---------|-----------|
| 少量修正（<5 处，单点调整） | 1 个 agent 可直接完成 | 1 个对应角色 agent |
| 中量修正（5-15 处，结构性调整） | 涉及多章节或多维度 | **2 个 agent 分块并行修正 + 主会话整合** |
| 大量修正（>15 处或涉及重发散） | 接近重写 | **回退到产出阶段重新执行多 agent 并行发散** |

### 八、主会话职责边界

主会话在多 agent 协作中只承担以下职责，**超出即越界**：

1. **调度**：决定哪个阶段派哪些 agent、什么时机并行
2. **搬运**：将 agent 产出写入文件（仅搬运，不编辑内容）
3. **整合**：当多个 agent 产出需要合并时，选择/拼接/去重（不改写核心结论）
4. **门禁**：检查出口标准、置信度阈值、agent 执行记录完整性

**主会话禁止行为**：
- 直接产出需要深度分析的文档内容
- 对 agent 产出进行"润色"后写入（只能原样搬运或明确标注修改）
- 在没有 agent 执行记录的情况下声称某阶段已完成

### 多 Agent 配置自检清单

每个 SKILL.md 定稿前，对每一阶段的 agent 调度设计进行以下检查：

- [ ] 该阶段输出件复杂度已评估（基于内容类型矩阵 + 决策属性上调法，非篇幅）
- [ ] 极高/高复杂度任务已完成 Plan 阶段（切分维度、视角分配、关键假设清单）
- [ ] 产出阶段的 agent 数量与复杂度匹配（极高≥3、高≥2）
- [ ] 多个并行 agent 的视角是否真正独立（不同角色/范式/侧重/范围）
- [ ] 评审 agent 的视角丰富度 ≥ 产出 agent 的视角丰富度
- [ ] 无依赖的任务已明确标注并行执行
- [ ] 复杂任务的拆分维度已明确（领域/分析类型/职能/范围）
- [ ] 修正阶段的 agent 配置与修正工作量匹配
- [ ] 主会话职责已限定为"调度、搬运、整合、门禁"四件事
- [ ] 不存在"1 个 agent 产出 + 丰富评审"的不对称配置
- [ ] 不存在"一个 agent 内部发散多个方案"替代多 agent 并行
- [ ] Plan 阶段未产出分析结论，仅做任务解构与视角分配
- [ ] 所有子 agent 均为预定义角色，无临时构造角色
- [ ] 同角色多实例已通过 prompt 明确区分关注点

### 九、出口标准与评审修正循环全覆盖原则

**每个产生可交付文档的阶段/Step，必须同时定义明确的出口标准和评审修正循环机制，缺一不可。**

这是 skill 质量的最后一道防线。产出 agent 的数量和独立性决定了交付件质量的上限，但**如果没有出口标准做门禁、没有评审修正循环做收敛，多 agent 并行产出的只是多个待整合的草稿，而非经过验证的正式交付件。**

#### 出口标准最低要求

每个有产出的阶段/Step，必须用以下三项定义通过条件：

1. **结构性 checklist**：用 checkbox 列明通过条件
   - CRITICAL 问题处理规则（必须修正/阻塞推进）
   - 评审 agent 判定标准（"通过"或"有条件通过"的明确条件）
   - 文档版本/状态要求

2. **量化置信度评分**：维度权重表 + 进入门槛
   - 至少 3 个评分维度，每个维度配权重和评分标准
   - 一般门槛 ≥ 0.80；涉及不可逆决策或一致性关键路径的门槛 ≥ 0.85
   - 低于门槛的处理规则：自动再修正 / 等待人工决策 / 降级标注

3. **自动推进规则**：明确什么条件下自动进入下一阶段，什么条件下等待用户确认

#### 评审修正循环最低要求

产出写入文件后，必须立即进入"独立评审 → 修正 → 复核"循环：

1. **独立评审**：至少 2 个不同角色的 reviewer agent 并行执行，视角差异化
2. **修正工作量分级**：
   | 修正工作量 | 判断标准 | agent 配置 |
   |-----------|---------|-----------|
   | 少量修正（<5 处） | 单点调整 | 1 个对应角色 agent |
   | 中量修正（5-15 处） | 涉及多章节或多维度 | **2 个 agent 分块并行修正 + 主会话整合** |
   | 大量修正（>15 处） | 接近重写 | **回退到产出阶段重新执行多 agent 并行发散** |
3. **循环上限与降级**：最多 3 次评审修正循环；3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或**等待人工决策**
4. **循环终止条件**：所有 CRITICAL 问题已修正 + 所有评审 agent 给出"通过"或"有条件通过"

#### 豁免场景（仅限以下两种）

| 场景 | 处理方式 | 原因 |
|------|---------|------|
| 纯交互式信息收集阶段（如第 0 阶段） | 无需评审修正循环，但需有轻量级"准入检查清单"确认信息充分性 | 无正式文档产出，只有问答结论 |
| 维护模式单点修正 | 走快速通道：读取 → 定位 → 修正 → 落盘；**修正后仍需至少 1 个 reviewer 快速复核** | 变更范围可控，完整循环成本过高 |

**任何有正式文档产出的阶段/Step，若缺少出口标准或评审修正循环定义，视为 SKILL.md 定稿不合格，禁止投入使用。**

## Skill 四文件分工原则

每个 skill 由四类文件组成，各有明确职责边界。判断标准：**这段话是给谁看的？**

skill 按需使用四类文件：至少包含 SKILL.md + Command；有 Agent 调度时需要 Agent.md；有结构化文档产出时需要 Template。

### SKILL.md — 编排法典（主会话上下文）

主会话的唯一权威指令，定义"怎么编排"。包含：
- 阶段结构与状态检测逻辑（进入条件、产出物、推进规则）
- Agent 调度规则（哪个阶段派哪个 agent、prompt 骨架、模型选择）
- 质量门禁（置信度阈值、评审触发条件、自动修正 vs 人工确认）
- 文件 I/O 规则（写什么文件、写到哪里、执行记录声明）
- 禁止事项和红旗清单

不应包含：agent 人设描述、文档具体章节结构、模板内容复述。

### Command — 启动引导（主会话上下文）

用户看到的第一屏，目标是 3 秒内理解命令用途，然后交给 SKILL.md。包含：
- 一句话说明 skill 做什么
- 用法示例
- 产出物清单（文件路径列表）
- 调用 Skill 工具的指令

不应包含：流程细节、agent 调度逻辑、质量标准。上限 30 行。

### Agent — 独立人格（子 agent 上下文）

子 agent 被派遣时的完整上下文，主会话看不到这些内容。包含：
- 身份定义（专���领域、角色定位）
- 工作模式（研究模式 / 设计模式 / 评审模式）
- 输出标准（格式、深度、长度要求）
- 协作边界（能做什么、不能做什么、与其他 agent 的关系）
- 该角色通用的质量准则

不应包含：具体阶段流程（SKILL.md 的事）、文档模板（Template 的事）。

### Template — 内容契约（子 agent 按需读取）

文档结构的唯一权威，agent 写文档前必须先读取对应 template。包含：
- 必填章节与可选章节
- 每个章节的内容要求（写什么、不写什么、最低深度）
- 自检清单
- 示例片段（可选）

不应包含：流程逻辑、agent 调度、质量门禁。

### 上下文池隔离

```
用户输入 /df:<skill>
    │
    ▼
Command（注入主会话）→ 调用 Skill 工具
    │
    ▼
SKILL.md（注入主会话）→ 主会话成为编排器
    │
    ├─ 派遣 agent-A ──▶ Agent.md（注入子 agent）
    │                          └─ Read template → 产出文档
    │
    ├─ 派遣 agent-B ──▶ Agent.md（注入子 agent）
    │                          └─ Read template → 产出文档
    │
    └─ 主会话：检查质量门禁 → 写文件 → 推进下一阶段
```

主会话上下文 = Command + SKILL.md；子 agent 上下文 = Agent.md + 按需读取的 Template。两个上下文池互不干扰，各自精简。

### 去重原则

如果一段话同时需要两个地方看到，说明抽象层次没分清——要么提升到更通用的表述放一处，要么拆成两个不同粒度的版本各放各处。具体：
- Command 不复述 SKILL.md 的流程摘要
- SKILL.md 不描述文档该有哪些章节（指向 template）
- SKILL.md 不嵌入 agent 人设（调度 prompt 只说"派遣 researcher 研究 X"）
- Agent.md 不包含阶段流程（它只知道自己的角色，不知道整体编排）

### Skill 元数据约束

Skill 文件（SKILL.md / Command / Agent.md / Template）**不保留 version 和 changelog 信息**。版本历史由 git 管理，变更内容体现在提交信息中。禁止在 YAML frontmatter 或正文中添加 version、changelog、变更记录等字段。

## Skill 术语规范

本规范约束所有 skill（`product/*` 和 `code/*` 目录下）的 SKILL.md 流程术语。目标：用户在切换 `/df:*` 命令时，不因术语差异产生认知切换成本；新 skill 开发时有明确的术语参照系。

### 统一策略：三层分离

| 层次 | 说明 | 策略 |
|------|------|------|
| **基础设施层** | 所有 skill 都有的共通机制（入口检测、评审、文档写入等） | **强制统一**，术语必须一致 |
| **流程骨架层** | 各 skill 的主流程阶段划分 | **规范命名规则**，允许内涵差异 |
| **领域内容层** | 各 skill 特有的专业内容 | **不干预**，由各 skill 自行定义 |

### 基础设施层（强制统一）

以下术语在所有 skill 中必须一致使用，禁止自创同义替换。

#### 1. 入口检测

统一叫 **「启动检测」**。

所有 `/df:*` 命令启动时，通过扫描文件系统判断当前所处阶段的动作，一律叫"启动检测"。禁止叫"状态检测""相位检测"等其他名称。

#### 2. 文档写入规范

统一叫 **「文档写入铁律」**，作为 SKILL.md 中的独立章节。

内容最低要求：
- Agent 产出后，主会话必须立即写入文件
- 写入后必须验证文件存在（`ls`/`wc` 确认）
- 每轮迭代产出禁止仅在对话中输出

#### 3. 问题分级

统一四级制：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **CRITICAL** | 阻塞性问题，会导致方向错误 | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响质量 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | 可延期 |

#### 4. 评审机制

| 要素 | 统一命名 | 说明 |
|------|----------|------|
| 评审动作 | **独立评审** | 一个或多个 Agent 独立产出评审意见 |
| 评审后修正 | **评审修正循环** | 发现问题 → 修正 → 复核的迭代过程 |
| 通过门槛 | **出口标准** | 用 checkbox 列表明确定义通过条件 |
| 质量评分 | **置信度** | 0.00-1.00 小数制，配合维度权重表 |

#### 5. 用户协作节奏

| 模式 | 统一命名 | 说明 |
|------|----------|------|
| 全自动 | **自动推进** | 阶段内产出→评审→修正，无需用户确认 |
| 半自动 | **确认后推进** | 阶段/轮次之间，汇报后等待用户确认 |
| 阻塞等待 | **等待人工决策** | 置信度低于阈值或信息缺失时停止 |

#### 6. 维护/扩展机制

| 统一术语 | 含义 |
|----------|------|
| **维护模式** | 主体流程完成后进入的按需调用状态 |
| **快速通道** | 小修小补，不走完整轮次 |
| **回退机制** | 大变更回退到对应子流程重新执行 |

### 流程骨架层（规范命名规则）

各 skill 的主流程结构不同，不强制统一名称，但必须遵守以下命名规则：

#### 规则 1：主流程阶段统一用「第 X 阶段」

所有 skill 的主流程大阶段一律叫 **「第 X 阶段」**。plan 不再用"层"，design/define 不再混用"步"与"轮"。

| skill | 当前 | 规范后 |
|-------|------|--------|
| design | 第 0 步、第 1 轮、第 2 轮... | 第 0 阶段、第 1 阶段...第 4 阶段 |
| define | 前置步骤、第一轮...第四轮 | 第 0 阶段、第 1 阶段...第 4 阶段 |
| plan | 第一层、第二层 | 第 1 阶段、第 2 阶段 |

> plan 无第 0 阶段，直接从第 1 阶段开始。

#### 规则 2：阶段内子单元统一用「Step X」

阶段内如有明确子步骤，一律用 **Step 1、Step 2...**。禁止用小数编号（如"Step 2.1"），禁止混用中文数字。

> 层级只有两层：**阶段 → Step**。不设第三层。如果 Step 内部需要描述固定的多 agent 协作模式（如 architect→reviewer→修正），使用基础设施层已定义的「评审修正循环」术语内联描述，不再拆分子层级。

#### 规则 3：禁止混用同义词

以下词汇在 SKILL.md 中属于**独占词**，一个 skill 中只能选一个，且必须遵守上述规则：

| 独占词 | 允许的语义 | 禁止替代为 |
|--------|-----------|-----------|
| 阶段 | 主流程大单元（第 X 阶段） | 轮、层、步、Phase |
| Step | 阶段内的子单元 | 子步骤、环节、步、Phase |

### 领域内容层（不干预）

以下术语属于各 skill 的专业领域，保留差异：

- 产品级文档类型：design 的 ADR/子系统架构，define 的 Actor-Feature/Scenario，plan 的 milestone-plan/iteration-plan
- 产品级专业概念：design 的数据流/一致性协议，define 的验收标准/非功能需求，plan 的故事点/Velocity/Wave
- 代码级专业概念：tdd 的 RED/GREEN/REFACTOR，code-review 的评审级别，lint 的检查层级
- Agent 角色名：各 skill 按需选择 researcher/architect/product-manager/code-reviewer/developer 等

### 新 Skill 术语自检清单

开发新 skill 时，SKILL.md 定稿前检查：

- [ ] 启动检测是否叫"启动检测"
- [ ] 文档写入章节是否叫"文档写入铁律"
- [ ] 问题分级是否使用 CRITICAL/HIGH/MEDIUM/LOW
- [ ] 评审动作是否叫"独立评审"，修正流程是否叫"评审修正循环"
- [ ] 通过门槛是否叫"出口标准"，质量评分是否用"置信度"（0.00-1.00）
- [ ] 主流程大单元是否叫"第 X 阶段"
- [ ] 阶段内子单元是否叫"Step X"
- [ ] 是否存在"轮/层/步/Phase"与"阶段"混用
- [ ] 是否存在小数编号（如 Step 2.1）
- [ ] 多 agent 协作模式是否用"评审修正循环"内联描述，而非拆分子层级
