# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在本仓库中工作时提供指导。

## 项目概述

**teamskills** 是面向大型复杂基础软件开发团队的统一技能仓库，解决团队级 AI 辅助开发工作流的标准化问题。

- **团队协作优先**：目标是团队整体提效和高效协作，减少并行开发冲突，而非个人开发者效率工具
- **四层分层驱动**：产品级（启发式 skill，重输出质量）→ 特性级（OpenSpec workflow，重流程控制）→ 代码级（TDD/评审/重构）→ 测试验证级（集成/性能测试）。上层约束下层，下层变更反馈上层
- **用现成工具，不造轮子**：使用 Claude Code 内置能力、Linux 系统工具和外部成熟工具，不自研需要长期维护的工具链

## 领域适配设计规范

**适用领域**：大型复杂基础软件项目，包括分布式系统（存储、计算、消息队列、数据库）、操作系统/内核、编译器/运行时、网络协议栈、虚拟化/容器、高性能计算、复杂嵌入式系统。

**不适用领域**：简单的 Web CRUD 应用、小型移动 App、原型项目。

**普适性要求**：
- 所有 skill 设计必须具备跨领域普适性，不绑定特定系统类型或编程语言
- 使用前需通过交互式方式确定具体系统类型（分布式系统/OS/编译器/网络栈等）和编程语言（C/C++/Rust/Go/Python/Java）
- 后续流程根据确定的领域信息自动适配：选择对应的编码规范、测试框架、质量标准
- **禁止为单一领域或单一语言设计 skill**（必须通过适配机制支持多领域）

## Skill 多 Agent 深度协作设计规范

本规范约束所有 SKILL.md 的 Agent 调度设计，目标是通过多 agent 并行与分工，增加思考深度和广度，提升交付件质量上限。

### 核心信条

**交付件质量的上限，由产出阶段的 agent 数量与独立性决定；评审只能发现已有框架内的缺陷，无法弥补框架本身的盲区。**

### 一、复杂度驱动原则（首要原则）

Agent 数量不由"阶段数量"决定，而由**输出件复杂度**决定。

**复杂度是语义属性，不是语法属性。** 判定依据是文档承载的决策权重和技术交叉度，而非篇幅行数。一份 80 行的 ADR（如一致性协议选型）复杂度远高于一份 500 行的接口清单罗列。

#### 复杂度评估的前提

主会话在评估文档复杂度前，**必须先读取对应的 template**，了解文档的章节结构和内容要求。基于 template 的具体要求，结合"内容类型矩阵"和"决策属性上调法"做出判定。

不同项目的同类文档，template 可能差异很大。例如，同样是"子系统架构设计"，如果 template 要求 10 个章节（含性能分析、故障处理、安全设计等），复杂度可能是"高"；如果只要求 3 个章节（概览、接口、部署），可能只是"中等"。

#### 内容类型矩阵（核心判定依据）

| 内容类型 | 基准复杂度 | 说明 |
|---------|-----------|------|
| **架构决策**（ADR 级别） | **极高** | 影响多个子系统、一旦确定难以回退、涉及核心技术选型 |
| **系统级架构设计** | **极高** | 定义子系统边界、交互契约、全局数据流 |
| **需求核心 Feature / Actor 定义** | **极高** | 定义产品边界、Actor 权责、验收标准 |
| **子系统级设计** | **高** | 单一子系统内多技术域交叉（如存储引擎需同时考虑 WAL + 索引 + 缓存）。对于特别复杂的子系统（如分布式存储的存储引擎、OS 的内存管理），若同时涉及多个上调属性，可上调至极高 |
| **接口 / 契约设计** | **高** | 跨模块边界、变更成本高、需兼容演进 |
| **Scenario / 用例设计** | **中等** | 基于已确定 Feature 的展开、有明确输入输出 |
| **调研报告** | **中等** | 信息整理、对比分析、不产生新的设计决策 |
| **模板化文档**（如迭代计划格式填充） | **低** | 结构固定、内容基于已有信息组合 |

#### 决策属性上调法（辅助判定）

文档若包含以下属性，复杂度在基准上上调：

| 属性 | 上调规则 |
|-----|---------|
| 跨子系统边界 | 复杂度 +1 级 |
| 不可逆决策 | 复杂度 +1 级 |
| 性能 / 一致性关键路径 | 复杂度 +1 级 |
| 无现成方案（需原创设计） | 复杂度 +1 级 |
| 安全 / 合规敏感 | 复杂度 +1 级 |

> 示例：一份 80 行的 ADR，若涉及跨子系统 + 不可逆 + 一致性关键路径，基准复杂度「极高」不变；一份 300 行的调研报告，若命中上述属性，复杂度从「中等」上调至「高」。

#### 复杂度与 agent 配置对照

| 复杂度 | 最小并行产出 agent 数 | 最小独立评审 agent 数 |
|--------|---------------------|---------------------|
| **极高** | **≥5** | **≥3** |
| **高** | **≥3** | **≥2** |
| **中等** | 1-2 | 1-2 |
| **低** | 1 | 1 |

> 对于大型复杂基础软件，架构决策的错误代价极高，token 成本相对可以忽略。极高复杂度文档需要5个以上独立视角的产出 agent，确保每个 agent 可以专注于单一核心维度进行深度分析。

**说明**：
- 表格给出的是**最小 agent 数下限**，不是精确数量
- 中等/低复杂度：主会话根据 template 和系统上下文，在下限范围内决定具体数量和分工
- 极高/高复杂度：必须派遣 plan agent，由 plan agent 决定具体数量（≥下限）和分工方案

#### 篇幅的辅助作用：异常检测

篇幅**不作为复杂度判定依据**，但用于发现异常：

- 高/极高复杂度文档篇幅过短（<100 行）：红旗——可能分析深度不足，需检查是否遗漏关键决策依据
- 低复杂度文档篇幅过长（>400 行）：红旗——可能信息密度过低，存在冗余或模板化填充
- 篇幅与同类文档历史均值偏差 >50%：提示复核，确认是合理差异还是质量问题

### 二、Plan 阶段原则

极高/高复杂度场景下，**必须先解构再发散**。Plan 阶段生成"agent 调度方案"，确保视角覆盖完整性和独立性。

#### 何时需要 Plan

| 复杂度 | 是否需要 Plan | Plan 执行方式 |
|--------|--------------|-------------|
| **极高** | **必须** | 派遣对应的 Plan agent 做任务解构 |
| **高** | **必须** | 派遣对应的 Plan agent 做任务解构 |
| **中等 / 低** | 不需要 | — |

> Plan 必须由 agent 执行（不能由主会话执行），因为涉及专业判断：如何切分维度、如何分配视角、如何识别盲区。

#### Plan agent 选择规则（产品级）

| Skill 任务 | Plan agent | 理由 |
|-----------|-----------|------|
| `/df:design` | architect | 系统思维和技术权衡能力 |
| `/df:define` | product-manager | 用户视角和场景分解能力 |
| `/df:plan` | project-manager | 资源估算和依赖编排能力 |

#### Plan agent 工作流程

1. **读取 template**：了解目标文档的章节结构和内容要求
2. **分析系统上下文**：了解文档在系统中的位置、重要性、依赖关系
3. **决定 agent 数量**：基于复杂度下限（极高≥5、高≥3）和 template 具体要求，决定实际派遣的 agent 数量
4. **设计切分维度**：基于 template 结构和系统特点，选择合适的任务切分维度（按子系统/按分析类型/按章节/按职能）
5. **产出调度方案**：每个 agent 的配置（角色 + 视角约束 + 负责章节/维度 + 输入材料）

#### Plan 产出与使用

**Plan agent 产出**（结构化调度方案）：
- 并行产出 agent 配置（角色 + 视角约束 + 负责章节/维度 + 输入材料）
- 评审 agent 配置（基于盲区分析）
- 整合策略（合并规则 + 冲突处理）

**主会话使用**：
1. **原样执行** plan agent 的调度方案（不做修改）
2. 按方案并行派遣 agent（prompt 含视角约束）
3. 配置评审（补充盲区）
4. 整合产出（按策略合并）

**如果对方案有疑问**：与用户确认或重新派遣 plan agent，而不是自行修改方案。

#### Plan 边界

- ✅ 切分维度、分配视角、识别盲区、列出关键假设
- ❌ 不产出分析内容、不预设结论


### 三、视角独立性原则

多 agent 并行必须是**真正独立的视角**，禁止以下替代：

- **禁止**一个 agent "内部发散多个方案"替代多个 agent 独立产出
- **禁止**多个 agent 使用同一 prompt 模板、同一输入源、同一分析框架
- **禁止**一个 agent 产出后，其他 agent 仅做"润色"或"格式调整"

**正确做法**：每个并行 agent 应从不同维度出发：
- 不同专业角色（architect vs product-manager vs researcher）
- 不同分析范式（自顶向下 vs 自底向上）
- 不同约束侧重（性能优先 vs 可维护性优先 vs 交付速度优先）
- 不同范围切分（按子系统、按特性域、按场景类型）

#### 预定义角色铁律

所有子 agent**必须是预定义角色**，禁止在主会话中临时构造 agent：

- **禁止**在调度 prompt 中临时描述 agent 角色（如"你是一个擅长性能分析的助手..."）
- **禁止**为单次任务创造新的 agent 类型名
- SKILL.md 中调度 agent 时只能使用 `.claude/agents/` 中已定义的 agent 名称

> 预定义角色确保 agent 行为的一致性和可复现性。临时构造的角色无法积累质量反馈，也无法在多轮迭代中保持一致的行为边界。

#### 同角色多实例区分规则

当同一预定义角色需要多个实例并行时，**必须通过 prompt 明确区分关注点**：

```
agent: researcher（关注一致性协议选型）
agent: researcher（关注存储引擎实现方案）
agent: researcher（关注网络传输层优化）
```

区分方式（选择其一或组合）：
- **按范围**：明确指定分析的子系统/模块/文件范围
- **按约束侧重**：如"性能优先视角"vs"可维护性优先视角"
- **按分析范式**：如"自顶向下架构推导"vs"自底向上实现约束推导"

**禁止**：同一角色的多个实例使用完全相同的 prompt，这将退化为"一个 agent 内部发散"的等价形式。

### 四、产出-评审对称原则

评审 agent 的视角丰富度必须**覆盖或超过**产出 agent 的视角丰富度。

| 产出阶段 agent 配置 | 评审阶段最低 agent 配置 | 说明 |
|-------------------|----------------------|------|
| ≥5 个并行 agent（极高复杂度） | **≥3 个不同角色的 reviewer** | 多视角产出需要多视角评审，确保整合一致性 |
| ≥3 个并行 agent（高复杂度） | **≥2 个不同角色的 reviewer** | 多维度产出需要外部视角审视整合结果 |
| 1-2 个 agent（中等/低复杂度） | **≥1 个 reviewer** | 单一或双视角产出需要独立评审发现盲区 |

**禁止模式**："1 个 agent 产出 + 3 个 reviewer 评审"——产出阶段的结构性缺陷无法通过评审阶段的丰富度来弥补。

### 五、并行最大化原则

无依赖的任务**必须并行**，不得以"流程描述简洁"为由串行化。

**必须并行的场景**：
- 多个独立文档的产出（如多个标杆研究、多个特性域需求、多个子系统架构）
- 多个独立维度的调研（如一致性协议、存储引擎、网络模型）
- 多个独立评审（如 architect-reviewer、pm-reviewer 同时评审不同文档）

**判断依赖的方法**：如果两个任务的输出互不引用对方的结论，即可并行。

### 六、拆分维度原则

复杂任务拆分时应选择以下维度之一，确保拆分后的子任务对单个 agent 而言仍是**完整的、有上下文的**：

| 拆分维度 | 适用场景 | 示例 |
|---------|---------|------|
| **按领域/子系统** | 各子系统相对独立 | 元数据子系统、存储子系统、网络子系统分别设计 |
| **按分析类型** | 同一对象需要多角度分析 | 正常路径 agent + 故障/异常路径 agent |
| **按职能侧重** | 同一文档的不同章节需要不同专业能力 | 结构规划 agent + 量化估算 agent + 风险分析 agent |
| **按范围深度** | 核心 vs 补充 | 核心 Feature agent + 边缘/扩展 Feature agent |

**禁止的拆分方式**：
- **禁止**按"前半部分/后半部分"机械拆分（agent 失去上下文）
- **禁止**按"写文档/改格式"拆分（后者不是 agent 工作）
- **禁止**过度拆分导致整合成本超过并行收益（>3 个并行 agent 产出同一份文档时需谨慎）

### 七、修正阶段 Agent 配置原则

如果评审后决定进入修正（未回退），根据**总问题数量**（CRITICAL + HIGH + MEDIUM）决定 agent 配置：

| 总问题数量 | agent 配置 |
|-----------|-----------|
| <10 | 1 个对应角色 agent |
| 10-30 | **2 个 agent 分块并行修正 + 主会话整合** |
| >30 | **评估：分批修正 or 回退重做** |

> 总问题数量只计入需要修正的问题（CRITICAL/HIGH/MEDIUM），LOW 级别问题可延期处理。


### 八、主会话职责边界

#### 主会话应该做的事

1. **读取 template 进行复杂度评估**：在决定派遣多少个 agent 前，先读取对应 template 了解文档的章节结构和内容要求
2. **交互式调研（前置）**：通过启发式提问收集产品背景、系统定位、业务目标、技术约束等信息，为后续专业分析工作提供上下文
3. **调度编排**：
   - 中等/低复杂度：根据 template 和系统上下文，决定派遣哪些 agent、并行还是串行
   - 极高/高复杂度：派遣 plan agent，原样执行 plan agent 的调度方案
   - 决定何时推进下一阶段
4. **信息搬运**：将 agent 产出写入文件（原样搬运，不改写核心内容）
5. **结果整合**：合并多个 agent 产出（拼接/去重/选择，不改写核心结论）
6. **质量门禁**：检查出口标准、置信度阈值、agent 执行记录完整性
7. **轻量级判断**：文件存在性检查、阶段状态判断、简单的信息查找和确认

#### 主会话不应该做的事

- **深度专业分析**：架构设计、需求分解、技术选型等需要专业思维的工作（应派遣对应 agent）
- **内容创作**：直接编写需要深度分析的文档章节（应由 agent 产出）
- **润色改写**：对 agent 产出进行实质性修改（只能原样搬运或明确标注修改）
- **虚假声明**：在没有 agent 执行记录的情况下声称某阶段已完成
- **修改 plan agent 的方案**：如对方案有疑问，应与用户确认或重新派遣 plan agent，而不是自行修改


## Skill 出口标准与评审修正循环规范

**每个产生可交付文档的阶段/Step，必须同时定义明确的出口标准和评审修正循环机制，缺一不可。**

这是 skill 质量的最后一道防线。产出 agent 的数量和独立性决定了交付件质量的上限，但**如果没有出口标准做门禁、没有评审修正循环做收敛，多 agent 并行产出的只是多个待整合的草稿，而非经过验证的正式交付件。**

### 出口标准最低要求

每个有产出的阶段/Step，必须用以下三项定义通过条件：

1. **结构性 checklist**：用 checkbox 列明通过条件
   - CRITICAL 问题处理规则（必须修正/阻塞推进）
   - 评审 agent 判定标准（"通过"或"有条件通过"的明确条件）
   - 文档版本/状态要求

2. **量化置信度评分**：维度权重表 + 进入门槛
   - 至少 3 个评分维度，每个维度配权重和评分标准
   - 一般门槛 ≥ 0.80；涉及不可逆决策或一致性关键路径的门槛 ≥ 0.85
   - 低于门槛的处理规则：自动再修正 / 等待人工决策 / 降级标注

3. **自动推进规则**：明确什么条件下自动进入下一阶段，什么条件下等待用户确认

### 评审修正循环最低要求

产出写入文件后，必须立即进入"独立评审 → 回退判断 → 修正 → 复核"循环：

1. **独立评审**：reviewer agent 数量符合复杂度要求（极高≥3、高≥2、中等/低≥1，见"产出-评审对称原则"），不同角色并行执行，视角差异化

   **CRITICAL 控制要求**：
   - 评审 agent 必须严格按照"问题分级"的三条判定标准识别 CRITICAL
   - **CRITICAL 必须基于系统级影响判定**：某个被评审文档内的结构性错误 ≠ 系统级 CRITICAL
   - 主会话在调度评审 agent 时，应在 prompt 中说明**文档的系统上下文**（在系统中的位置、重要性、可替换性）

2. **回退判断**：根据 CRITICAL 数量和框架性错误判断是否回退重做

   | 条件 | 决策 |
   |------|------|
   | CRITICAL ≥3 **或** (CRITICAL 1-2 且为框架性错误) | **回退重做** |
   | CRITICAL 1-2 且非框架性错误 | 进入修正 |
   | CRITICAL = 0 | 进入修正 |

   > 框架性错误：整体方向错误、核心假设错误、架构选型根本不适用等。

3. **修正工作量分级**：如果不回退，根据总问题数量配置修正 agent（见"修正阶段 Agent 配置原则"）

4. **循环上限与降级**：最多 3 次评审修正循环；3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或**等待人工决策**

5. **循环终止条件**：所有 CRITICAL 问题已修正 + 所有评审 agent 给出"通过"或"有条件通过"

### 评审记录写入约束

评审与修正记录写入正式交付文档时，必须保持简略，避免喧宾夺主：

- **评审记录 ≤200 字**：仅保留评审 agent 名称、通过结论、遗留问题数及简要分类，不展开详细评审意见
- **修正记录 ≤200 字**：聚焦"改了什么"，不展开论证过程
- **详细评审意见留在对话上下文**，不写入正式交付文档

### 豁免场景（仅限以下两种）

| 场景 | 处理方式 | 原因 |
|------|---------|------|
| 交互式信息收集（主会话主导） | 无需评审修正循环，但需有轻量级"准入检查清单"确认信息充分性 | 主会话通过交互式提问收集并记录事实性信息，无 agent 深度分析 |
| 维护模式单点修正 | 走快速通道：读取 → 定位 → 修正 → 落盘；**修正后仍需至少 1 个 reviewer 快速复核** | 变更范围可控，完整循环成本过高 |

**任何由 agent 进行深度分析产出的阶段/Step，若缺少出口标准或评审修正循环定义，视为 SKILL.md 定稿不合格，禁止投入使用。**

## Skill 四文件分工原则

每个 skill 由四类文件组成，各有明确职责边界。判断标准：**这段话是给谁看的？**

本章节定义 skill 内部的文件组织规范。关于 skill 与外部环境的依赖边界，见"Skill 自解释规范"。

skill 按需使用四类文件：至少包含 SKILL.md + Command；有 Agent 调度时需要 Agent.md；有结构化文档产出时需要 Template。

### 文件位置规范

#### Skill 核心文件

| 文件类型 | 位置 | 示例 |
|---------|------|------|
| SKILL.md | `.claude/skills/<skill-name>/SKILL.md` | `.claude/skills/devforge-design/SKILL.md` |
| Command | `.claude/commands/df/<skill-name>.md` | `.claude/commands/df/design.md` |
| Agent.md | `.claude/agents/<agent-name>.md` | `.claude/agents/architect.md` |
| Template | `.claude/templates/<template-name>.md` | `.claude/templates/arch-design.md` |

**说明**：
- `<skill-name>` 为 skill 名称（如 `devforge-design`、`devforge-define`、`devforge-plan`）
- Command 按命令前缀分组（`df` 对应 `/df:*` 命令）
- Agent 和 Template 为跨 skill 共享，不按 skill 分类存放

#### 辅助目录

| 目录 | 用途 | 示例 |
|------|------|------|
| `.claude/rules/` | 编码规范、工作流规范、测试规范等规则文件 | `coding-style-c.md`、`git-workflow.md`、`testing.md` |
| `.claude/hooks/` | Claude Code hooks 配置，自动化触发的脚本和规则 | `pre-tool-use.json`、`post-edit.json` |

**说明**：
- Rules 按语言或领域分类（如 `coding-style-<lang>.md`）
- Hooks 按触发时机命名（pre-/post-）

### SKILL.md — 编排法典（主会话上下文）

主会话的唯一权威指令，定义"怎么编排"。包含：
- 阶段结构与状态检测逻辑（进入条件、产出物、推进规则）
- Agent 调度规则（哪个阶段派哪个 agent、prompt 骨架、模型选择）
- 质量门禁（置信度阈值、评审触发条件、自动修正 vs 人工确认）
- 文件 I/O 规则（写什么文件、写到哪里、执行记录声明）
- 禁止事项和红旗清单

不应包含：agent 人设描述、文档具体章节结构、模板内容复述。

### Command — 启动引导（主会话上下文）

用户看到的第一屏，目标是 3 秒内理解命令用途，然后交给 SKILL.md。包含：
- 一句话说明 skill 做什么
- 用法示例
- 产出物清单（文件路径列表）
- 调用 Skill 工具的指令

不应包含：流程细节、agent 调度逻辑、质量标准。上限 60 行。

### Agent — 独立人格（子 agent 上下文）

子 agent 被派遣时的完整上下文，主会话看不到这些内容。包含：
- 身份定义（专业领域、角色定位）
- 工作模式（研究模式 / 设计模式 / 评审模式）
- 输出标准（格式、深度、长度要求）
- 协作边界（能做什么、不能做什么、与其他 agent 的关系）
- 该角色通用的质量准则

不应包含：具体阶段流程（SKILL.md 的事）、文档模板（Template 的事）。

### Template — 内容契约（子 agent 按需读取）

文档结构的唯一权威，agent 写文档前必须先读取对应 template。包含：
- 必填章节与可选章节
- 每个章节的内容要求（写什么、不写什么、最低深度）
- 自检清单
- 示例片段（可选）

不应包含：流程逻辑、agent 调度、质量门禁。

### 上下文池隔离

```
用户输入 /df:<skill>
    │
    ▼
Command（注入主会话）→ 调用 Skill 工具
    │
    ▼
SKILL.md（注入主会话）→ 主会话成为编排器
    │
    ├─ Read template → 评估复杂度
    │
    ├─ 中等/低复杂度：
    │   ├─ 派遣 agent-A ──▶ Agent.md（注入子 agent）
    │   │                          └─ Read template → 产出文档
    │   └─ 派遣 agent-B ──▶ Agent.md（注入子 agent）
    │                              └─ Read template → 产出文档
    │
    ├─ 极高/高复杂度：
    │   ├─ 派遣 plan agent ──▶ Agent.md（注入子 agent）
    │   │                          ├─ Read template → 了解文档结构
    │   │                          └─ 产出调度方案（agent 数量 + 分工）
    │   │
    │   ├─ 主会话：原样执行方案
    │   │
    │   ├─ 派遣 agent-A ──▶ Agent.md（注入子 agent）
    │   │                          └─ Read template → 产出文档
    │   ├─ 派遣 agent-B ──▶ Agent.md（注入子 agent）
    │   │                          └─ Read template → 产出文档
    │   └─ ...（按方案派遣更多 agent）
    │
    └─ 主会话：检查质量门禁 → 写文件 → 推进下一阶段
```

**上下文池说明**：
- 主会话上下文 = Command + SKILL.md + Template（用于复杂度评估）
- plan agent 上下文 = Agent.md + Template（用于任务解构）
- 产出 agent 上下文 = Agent.md + Template（用于文档写作）
- 各上下文池互不干扰，各自精简

### 去重原则

如果一段话同时需要两个地方看到，说明抽象层次没分清——要么提升到更通用的表述放一处，要么拆成两个不同粒度的版本各放各处。

#### Command 与 SKILL.md

- Command 不描述阶段划分、执行逻辑、质量门禁
- Command 只列出产出物清单（文件路径），不描述如何产出
- SKILL.md 不重复 Command 中的"一句话说明"和用法示例

#### SKILL.md 与 Template

- SKILL.md 不列举文档的章节结构
- SKILL.md 不描述每个章节应该写什么内容
- SKILL.md 只引用 template（说"读取 template X"或"按 template 要求"）
- Template 不描述何时产出、由谁产出、如何评审

#### SKILL.md 与 Agent.md

- SKILL.md 不描述 agent 的角色定位、专业领域、工作模式
- SKILL.md 不描述 agent 的输出格式、深度、长度要求
- SKILL.md 调度 prompt 只说"派遣 architect 分析 X"，不描述 architect 是什么角色
- Agent.md 不描述何时被派遣、在哪个阶段、派遣条件
- Agent.md 不描述整体流程编排逻辑

#### Agent.md 与 Template

- Agent.md 不列举文档的章节划分
- Agent.md 只说"按 template 要求写作"，不复述 template 的章节结构
- Template 不描述由哪个 agent 写作、agent 的能力要求

#### 跨文件通用原则

- **权威定义唯一**：信息在 A 文件是权威定义，B 文件需要时应该引用而不是复制
- **禁止"方便"复制**：禁止"为了方便"而在多个文件中重复同一信息
- **重复即设计问题**：发现需要在多个文件中重复，说明抽象层次有问题，应该重新设计

## Skill 自解释规范

本章节定义 skill 的可移植性约束。关于 skill 内部的文件组织，见"Skill 四文件分工原则"。

**核心原则**：Skill 是可移植的技能包，必须自包含，不依赖项目特定的 CLAUDE.md。

### 自解释要求

- Skill 文件（SKILL.md / Command / Agent.md / Template）**不能引用 CLAUDE.md** 的内容
- Skill 文件之间可以相互引用（如 SKILL.md 引用 template、Agent.md）
- Skill 需要的所有信息必须在 skill 文件内部定义或通过参数/交互式提问获取

### 允许的引用

- SKILL.md 可以引用 template（`.claude/templates/xxx.md`）
- SKILL.md 可以引用 agent（`.claude/agents/xxx.md`）
- Command 可以引用 SKILL.md
- Agent.md 可以引用 template

### 禁止的引用

- Skill 文件不能引用 CLAUDE.md
- Skill 文件不能引用项目特定的规范文件（如 `.claude/rules/`）
- Skill 文件不能假设项目有特定的目录结构（除了 skill 自己创建的）

### 信息重复的处理

如果某个信息在 CLAUDE.md 和 skill 中都需要，应该在两处**分别定义**：
- **CLAUDE.md 中的定义**：项目级规范（如"复杂度驱动原则"的设计理念）
- **Skill 中的定义**：skill 的执行逻辑（如"极高复杂度派遣 ≥5 个 agent"）
- 两者可以内容相同，但不能通过引用关联

### 可移植性验证

- Skill 应该能够被打包、分发、安装到其他项目
- 安装后，skill 应该能够正常工作，不依赖原项目的 CLAUDE.md
- 如果 skill 需要项目提供信息（如编程语言、系统类型），应该通过交互式提问获取，而不是读取 CLAUDE.md

### Skill 元数据约束

Skill 文件（SKILL.md / Command / Agent.md / Template）**不保留 version 和 changelog 信息**。版本历史由 git 管理，变更内容体现在提交信息中。禁止在 YAML frontmatter 或正文中添加 version、changelog、变更记录等字段。

## Skill 术语规范

本规范约束所有 skill（`.claude/skills/` 目录下）的 SKILL.md 流程术语。目标：用户在切换 `/df:*` 命令时，不因术语差异产生认知切换成本；新 skill 开发时有明确的术语参照系。

### 统一策略：三层分离

| 层次 | 说明 | 策略 |
|------|------|------|
| **基础设施层** | 所有 skill 都有的共通机制（入口检测、评审、文档写入等） | **强制统一**，术语必须一致 |
| **流程骨架层** | 各 skill 的主流程阶段划分 | **规范命名规则**，允许内涵差异 |
| **领域内容层** | 各 skill 特有的专业内容 | **不干预**，由各 skill 自行定义 |

### 基础设施层（强制统一）

以下术语在所有 skill 中必须一致使用，禁止自创同义替换。

#### 1. 入口检测

统一叫 **「启动检测」**。

所有 `/df:*` 命令启动时，通过扫描文件系统判断当前所处阶段的动作，一律叫"启动检测"。禁止叫"状态检测""相位检测"等其他名称。

**行为规则**：
- **主流程未完成时**：根据文档状态直接判定并提示**唯一的下一步动作**，不罗列多个选项让用户选择
- **主流程已完成后**（维护模式）：允许给出多个改进方向供用户选择（如新增/完善/重构）
- **禁止跳过启动检测**直接假设从零开始

#### 2. 文档写入规范

统一叫 **「文档写入铁律」**，作为 SKILL.md 中的独立章节。

内容最低要求：
- Agent 产出后，主会话必须立即写入文件
- 写入后必须验证文件存在（`ls`/`wc` 确认）
- 每轮迭代产出禁止仅在对话中输出

#### 3. 问题分级

**核心原则：问题级别基于对整个产品系统级的影响判定，而非对单个文档所描述的局部影响判定。**

统一四级制：

| 级别 | 含义 | 判定基准 | 判定标准（必须同时满足） | 处理方式 |
|------|------|---------|----------------------|----------|
| **CRITICAL** | 阻塞性问题，会导致系统方向错误 | **系统级影响** | (1) 影响整个产品系统的核心决策或整体方向<br>(2) 不修正会导致后续工作推倒重来<br>(3) 涉及不可逆或高成本变更 | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响局部质量 | **文档/模块级影响** | 影响局部设计或实现质量，但不改变系统整体方向 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | **优化建议** | 优化建议，不影响功能正确性 | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | **非功能性** | 格式、命名、注释等非功能性问题 | 可延期 |

**判定时必须考虑的系统上下文**：
- 该文档在整个产品系统中的位置和重要性
- 该文档所描述的组件/模块的可替换性
- 错误的影响范围（仅限当前模块 vs 影响多个子系统 vs 影响整体架构）
- 修正成本（局部调整 vs 推倒重来）

**常见误判与纠正**：
- ❌ **错误**：某个子系统设计文档内部结构混乱 → 判定为 CRITICAL
- ✅ **正确**：该子系统可独立重构，不影响其他子系统 → 应为 HIGH

- ❌ **错误**：某个 Feature 的 Scenario 定义不完整 → 判定为 CRITICAL  
- ✅ **正确**：该 Feature 可后续补充，不影响产品整体方向 → 应为 HIGH

- ✅ **正确**：核心一致性协议选型错误 → CRITICAL（系统级方向性决策）
- ✅ **正确**：整体架构分层错误 → CRITICAL（影响所有子系统）

#### 4. 评审机制

| 要素 | 统一命名 | 说明 |
|------|----------|------|
| 评审动作 | **独立评审** | 一个或多个 Agent 独立产出评审意见 |
| 评审后修正 | **评审修正循环** | 发现问题 → 修正 → 复核的迭代过程 |
| 通过门槛 | **出口标准** | 用 checkbox 列表明确定义通过条件 |
| 质量评分 | **置信度** | 0.00-1.00 小数制，配合维度权重表 |

#### 5. 用户协作节奏

**核心边界**：自动推进与人工确认的切换点按**交付件是否完成闭环**划分，不按"阶段内/阶段间"简单切分。

| 模式 | 统一命名 | 说明 |
|------|----------|------|
| 全自动 | **自动推进** | 交付件内部：产出 → 评审 → 修正全自动闭环，不中途询问用户 |
| 半自动 | **确认后推进** | 交付件之间：产出物闭环后向用户汇报概况（产出文件、置信度、关键决策点），等待用户确认后再进入下一交付件 |
| 阻塞等待 | **等待人工决策** | 置信度低于阈值、CRITICAL 或 HIGH 未修正、或需用户做不可逆决策时，在交付件内部中断 |

#### 6. 维护/扩展机制

| 统一术语 | 含义 |
|----------|------|
| **维护模式** | 主体流程完成后进入的按需调用状态 |
| **快速通道** | 小修小补，不走完整轮次 |
| **回退机制** | 大变更回退到对应子流程重新执行 |

### 流程骨架层（规范命名规则）

各 skill 的主流程结构不同，不强制统一名称，但必须遵守以下命名规则：

#### 规则 1：主流程阶段统一用「第 X 阶段」

所有 skill 的主流程大阶段一律叫 **「第 X 阶段」**。

**命名规则**：
- 禁止使用"层""轮""步"等同义词
- 禁止混用中文数字和阿拉伯数字
- 统一使用"第 X 阶段"（X 为阿拉伯数字）

**示例**（仅展示命名规则，不约束具体阶段划分）：
- ❌ 错误：第一层、第 1 轮、第 0 步
- ✅ 正确：第 0 阶段、第 1 阶段、第 2 阶段

#### 规则 2：阶段内子单元统一用「步骤 X」

阶段内如有明确子步骤，一律用 **步骤 1、步骤 2...**。

**命名规则**：
- 禁止用小数编号（如"步骤 2.1"）
- 禁止混用中文数字（如"步骤一"）
- 统一使用"步骤 X"（X 为阿拉伯数字）

**与废弃术语的区分**：
- ❌ 废弃：主流程用"第 X 步"（已统一为"第 X 阶段"）
- ✅ 当前：阶段内用"步骤 X"

**层级结构**：
- 只有两层：**第 X 阶段 → 步骤 X**
- 不设第三层
- 如果步骤内部需要描述固定的多 agent 协作模式（如 architect→reviewer→修正），使用基础设施层已定义的「评审修正循环」术语内联描述，不再拆分子层级

#### 规则 3：禁止混用同义词

以下词汇在 SKILL.md 中属于**独占词**，一个 skill 中只能选一个，且必须遵守上述规则：

| 独占词 | 允许的语义 | 禁止替代为 |
|--------|-----------|-----------|
| 阶段 | 主流程大单元（第 X 阶段） | 轮、层、步、Phase |
| 步骤 | 阶段内的子单元（步骤 X） | 子步骤、环节、步、Step、Phase |

### 领域内容层（不干预）

以下术语属于各 skill 的专业领域，保留差异：

- 产品级文档类型：design 的 ADR/子系统架构，define 的 Actor-Feature/Scenario，plan 的 milestone-plan/iteration-plan
- 产品级专业概念：design 的数据流/一致性协议，define 的验收标准/非功能需求，plan 的故事点/Velocity/Wave
- 代码级专业概念：tdd 的 RED/GREEN/REFACTOR，code-review 的评审级别，lint 的检查层级
- Agent 角色名：各 skill 按需选择 researcher/architect/product-manager/code-reviewer/developer 等

## Skill 文件合规评审

Skill 四类文件（SKILL.md / Agent.md / Command.md / Template.md）定稿前，必须通过 `SKILL-COMPLIANCE.md` 的逐项合规检查。

**使用方式**：
- **设计阶段**：了解检查大类分布，确保设计不偏离规范方向
- **评审时**：严格执行 `SKILL-COMPLIANCE.md`，逐项打勾，禁止以"通读觉得没问题"替代

检查覆盖：术语规范 / 层级命名 / 多 Agent 协作设计 / 出口标准与评审修正循环 / 自包含性 / 四文件分工 / 核心信条体现 / 执行行为规范 / 评审结论记录
