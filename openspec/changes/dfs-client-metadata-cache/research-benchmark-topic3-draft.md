# 主题 3: 缓存淘汰策略 — 标杆方案研究

## 维度 3.1: 淘汰算法

### 方案 1: LRU (Least Recently Used)

**标杆来源**：Linux page cache、Memcached、Redis (allkeys-lru)

#### 原理层

**解决的问题**：
- 在缓存容量有限时，决定淘汰哪些条目以腾出空间
- 基于时间局部性假设：最近访问的数据未来更可能被访问

**核心算法**：
- 维护访问时间戳或双向链表，每次访问将条目移至链表头
- 淘汰时从链表尾移除最久未访问的条目
- 时间复杂度：O(1) 访问更新，O(1) 淘汰

**理论基础**：
- 时间局部性原理（Temporal Locality）
- 适用于访问模式具有短期重复性的工作负载

#### 设计层

**系统架构**：
- **Linux page cache**：双向链表 + Hash Table
  - `struct list_head lru` 链表节点嵌入 `struct page`
  - `radix_tree` 提供 O(1) 查找，链表提供 O(1) LRU 更新
- **Memcached**：分段 LRU（Segmented LRU）
  - HOT/WARM/COLD 三段链表，新条目进入 WARM
  - 访问时提升到 HOT，淘汰从 COLD 开始
  - 减少链表锁竞争（每段独立锁）

**关键设计机制**：
1. **访问更新策略**：
   - 严格 LRU：每次访问都移动到链表头（Memcached 早期）
   - 近似 LRU：批量更新或采样更新（Redis）
2. **淘汰触发时机**：
   - 被动淘汰：插入新条目时���查容量
   - 主动淘汰：后台线程定期扫描（Linux kswapd）
3. **数据流**：
   - 访问路径：Hash 查找 → 命中则移动到链表头 → 返回数据
   - 淘汰路径：从链表尾取 N 个候选 → 检查引用计数 → 释放内存

#### 优化层

**性能瓶颈**：
- 链表操作需要全局锁，高并发下锁竞争严重
- 每次访问都更新链表，写放大严重（缓存行失效）

**针对性优化**：
- **分段锁**（Memcached）：将链表分为多段，每段独立锁，并发度提升 N 倍
- **近似 LRU**（Redis）：
  - 采样 N 个条目（默认 5），选择其中最久未访问的淘汰
  - 避免维护全局链表，内存开销降低 50%
  - 命中率损失 <5%（N=5 时）
- **批量更新**（Linux）：访问时不立即移动，标记为 active，后台批量处理

**Trade-off 分析**：

| 维度 | 严格 LRU | 近似 LRU（Redis） | 分段 LRU（Memcached） |
