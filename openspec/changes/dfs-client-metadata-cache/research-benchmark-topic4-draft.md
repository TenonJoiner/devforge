# 主题 4: 预取与批量优化 — 标杆方案研究

## 维度 4.1: 预取触发策略

### 方案 A: 顺序预取（Sequential Prefetch）

**标杆来源**: Linux readahead, NFS client readahead

#### 原理层

**解决的问题**: 顺序访问场景下减少元数据查询延迟，将多次串行 RPC 转化为一次批量 RPC + 本地缓存命中。

**核心算法**:
- 检测连续访问模式：当前访问的 inode 编号与上次访问的 inode 编号差值 ≤ 阈值（如 16）
- 触发预取：一旦检测到顺序模式，预取后续 N 个 inode 的元数据（N 为预取窗口大小）
- 窗口调整：连续命中时窗口翻倍（如 4 → 8 → 16），随机访问时窗口重置为最小值

**理论复杂度**:
- 检测开销：O(1)（只需比较当前与上次访问的 inode 编号）
- 预取开销：O(N)（N 为预取窗口大小）
- 空间开销：O(N × M)（N 为窗口大小，M 为单个元数据条目大小）

#### 设计层

**系统架构**:
```
Client
  ├── Access Pattern Detector
  │     ├── 记录最近 K 次访问的 inode 序列
  │     └── 计算连续性得分（consecutive hits / total accesses）
  ├── Prefetch Engine
  │     ├── 窗口大小控制器（初始 4，最大 128）
  │     └── 批量 RPC 调度器
  └── Metadata Cache
        └── 预取条目标记（prefetched flag）
```

**关键设计机制**:
1. **窗口自适应**: 
   - 初始窗口 = 4 个条目
   - 连续命中 ≥ 75% 时窗口翻倍
   - 随机访问（连续性 < 25%）时窗口减半
2. **预取时机**: 
   - 同步预取：当前查询返回后立即发起预取 RPC
   - 异步预取：后台线程定期检查访问模式并预取
3. **预取取消**: 
   - 检测到访问模式切换（如从顺序变为随机）时取消未完成的预取请求
