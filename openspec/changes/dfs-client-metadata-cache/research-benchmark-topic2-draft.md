# 主题 2: 缓存一致性协议 — 标杆方案研究

## 维度 2.1: 一致性模型

### 方案 A: 强一致性（租约模型）

**标杆来源**: AFS callback、Ceph capability、GFS2 glock

#### 原理层

**解决的问题**: 保证客户端缓存与服务端数据严格一致，避免读到过期数据。

**核心机制**: 
- 客户端持有租约（lease/capability）期间，服务端保证不会修改数据
- 服务端需要修改时，先回收所有客户端租约，等待确认后才执行写操作
- 租约有明确的有效期，过期后客户端必须重新验证

**理论复杂度**:
- 读操作延迟: O(1) 本地缓存命中，O(RTT) 租约过期需重新获取
- 写操作延迟: O(N×RTT) N 为持有租约的客户端数，需逐个回收
- 一致性保证: 线性一致性（Linearizability）

#### 设计层

**AFS callback 机制**:
- 组件划分: 客户端 Cache Manager、服务端 File Server、callback 状态表
- 数据流: 客户端首次访问时获取数据 + callback promise，服务端记录 (file, client) 映射
- 控制流: 服务端修改文件前，遍历 callback 表向所有客户端发送 break callback RPC
- 关键设计: callback 是"承诺"而非"租约"，无明确过期时间，依赖服务端主动通知

**Ceph capability 机制**:
- 组件划分: 客户端 MDS client、MDS 服务端、capability 状态机
- 数据流: 客户端请求 capability（读/写/缓存权限），MDS 授予并记录 (inode, client, caps) 三元组
- 控制流: MDS 需要回收时发送 revoke 消息，客户端确认后释放，MDS 等待所有确认后继续
- 关键设计: capability 分为多种类型（FILE_RD/FILE_WR/FILE_CACHE/FILE_LAZYIO），支持细粒度权限控制

**GFS2 glock 机制**:
- 组件划分: 客户端 glock holder、DLM（分布式锁管理器）、锁状态机
- 数据流: 客户端通过 DLM 获取 glock（共享/排他模式），持有期间可缓存元数据
- 控制流: 其他客户端请求冲突锁时，DLM 触发 demote 回调，当前持有者降级或释放锁
- 关键设计: 基于 DLM 的分布式锁，支持锁升级/降级，避免频繁释放

#### 优化层

**性能瓶颈**:
- 写操作需要回收所有租约，延迟随客户端数线性增长
- 客户端故障或网络分区导致租约回收超时，阻塞写操作
- 租约过期后重新获取产生额外 RPC 开销

**针对��优化**:
- **租约分级**: Ceph 将 capability 分为多种类型，只回收冲突的权限（如写操作只需回收 FILE_WR，不影响 FILE_RD）
- **异步回收**: AFS 使用异步 callback break，不等待客户端确认即可继续（牺牲强一致性换取性能）
- **租约续期**: GFS2 支持锁降级而非完全释放，减少重新获取开销
- **超时机制**: 设置租约回收超时（如 30s），超时后强制失效客户端缓存

**Trade-off 取舍**:

| 维度 | 强一致性优势 | 强一致性代价 |
|------|-------------|-------------|
| 正确性 | 保证线性一致性，无脏读 | 实现复杂，需处理客户端故障、网络分区 |
| 性能 | 缓存命中时延迟低（本地访问） | 写操作延迟高（O(N×RTT)），扩展性差 |
| 可用性 | 客户端故障影响写操作可用性 | 需要超时和强制失效机制 |
| 适用场景 | 强一致性要求（如数据库元数据） | 不适合高并发写、大规模客户端场景 |

**可扩展性限制**:
- 客户端数 >1000 时，租约回收延迟可达数秒
- 需要引入租约分组、分层回收等优化
- 极端情况下可能退化为无缓存（每次都重新验证）

### 方案 B: 最终一致性（失效通知模型）

**标杆来源**: NFS v4 delegation、HDFS client cache、Lustre intent lock

#### 原理层

**解决的问题**: 在保证最终一致性的前提下，最大化缓存命中率，降低写操作延迟。

**核心机制**:
- 客户端缓存数据，服务端异步发送失效通知（invalidation）
- 客户端收到通知后清理缓存，下次访问时重新获取
- 允许短暂的不一致窗口（从服务端修改到客户端收到通知）

**理论复杂度**:
- 读操作延迟: O(1) 本地缓存命中，O(RTT) 缓存失效需重新获取
- 写操作延迟: O(RTT) 服务端直接执行，异步通知客户端
- 一致性保证: 最终一致性（Eventual Consistency），不一致窗口 = 通知延迟

#### 设计层

**NFS v4 delegation 机制**:
- 组件划分: 客户端 NFS client、服务端 NFS server、delegation 状态表
- 数据流: 客户端请求 delegation（读/写委托），服务端授予并记录 (file, client, type)
- 控制流: 服务端需要修改时发送 CB_RECALL，客户端异步返回 delegation，服务端不等待确认即可继续
- 关键设计: delegation 是"建议"而非"强制"，客户端可以延迟返回，服务端可以强制回收

**HDFS client cache 机制**:
- 组件划分: 客户端 DFSClient、NameNode、block location cache
- 数据流: 客户端缓存 block location 信息，NameNode 修改后不主动通知
- 控制流: 客户端访问时检测到 block location 变化（如 RPC 返回错误），清理缓存重新获取
- 关键设计: 被动失效模型，依赖客户端自行检测过期，无主动通知机制

**Lustre intent lock 机制**:
- 组件划分: 客户端 LDLM client、MDT（元数据服务器）、intent lock 状态机
- 数据流: 客户端请求 intent lock（如 IT_LOOKUP/IT_OPEN），MDT 授予锁并返回元数据
- 控制流: MDT 修改元数据时发送 blocking AST（异步回调），客户端收到后释放锁
- 关键设计: intent lock 将锁请求与元数据操作合并，减少 RPC 次数

#### 优化层

**性能瓶颈**:
- 失效通知延迟导致不一致窗口，可能读到过期数据
- 客户端故障或网络分区导致失效通知丢失，缓存永久过期
- 大量客户端时，失效通知风暴（N 个客户端 × M 次修改）

**针对性优化**:
- **版本号机制**: HDFS 使用 block version，客户端访问时检测版本变化，自动失效缓存
- **批量失效**: Lustre 将多个失效通知合并为一个 RPC，减少网络开销
- **失效通知重试**: NFS v4 支持失效通知重试，客户端必须确认收到
- **强制失效**: 设置失效通知超时（如 10s），超时后服务端强制标记客户端缓存失效

**Trade-off 取舍**:

| 维度 | 最终一致性优势 | 最终一致性代价 |
|------|---------------|---------------|
| 正确性 | 允许短暂不一致，简化实现 | 可能读到过期数据，需应用层容忍 |
| 性能 | 写操作延迟低（O(RTT)），扩展性好 | 失效通知延迟影响一致性窗口 |
| 可用性 | 客户端故障不影响写操作 | 失效通知丢失导致缓存永久过期 |
| 适用场景 | 高并发写、大规模客户端、容忍短暂不一致 | 不适合强一致性要求场景 |

**可扩展性限制**:
- 客户端数 >10000 时，失效通知风暴可能导致服务端过载
- 需要引入失效通知批量化、分组通知等优化
- 极端情况下可能退化为无缓存（频繁失效）

### 方案 C: 弱一致性（TTL 模型）

**标杆来源**: DNS cache、HTTP cache、CDN cache

#### 原理层

**解决的问题**: 在无需服务端参与的情况下，客户端自主管理缓存生命周期，最大化缓存命中率。

**核心机制**:
- 客户端缓存数据时设置 TTL（Time-To-Live），过期后自动失效
- 服务端无需维护客户端状态，无需发送失效通知
- 允许较长的不一致窗口（TTL 时长）

**理论复杂度**:
- 读操作延迟: O(1) 本地缓存命中，O(RTT) TTL 过期需重新获取
- 写操作延迟: O(RTT) 服务端直接执行，无需通知客户端
- 一致性保证: 弱一致性（Weak Consistency），不一致窗口 = TTL 时长

#### 设计层

**DNS cache 机制**:
- 组件划分: 客户端 resolver、DNS 服务器、本地缓存
- 数据流: 客户端查询 DNS 记录，服务器返回记录 + TTL 值
- 控制流: 客户端缓存记录，TTL 过期后自动失效，下次查询时重新获取
- 关键设计: TTL 由服务端指定，客户端严格遵守，无主动失效机制

**HTTP cache 机制**:
- 组件划分: 客户端浏览器、HTTP 服务器、缓存控制头（Cache-Control/Expires）
- 数据流: 客户端请求资源，服务器返回资源 + 缓存控制头
- 控制流: 客户端根据 max-age/Expires 计算过期时间，过期后重新验证（If-Modified-Since）
- 关键设计: 支持条件请求（304 Not Modified），减少数据传输

**CDN cache 机制**:
- 组件划分: 边缘节点、源站、缓存策略配置
- 数据流: 边缘节点缓存源站内容，设置 TTL（如 3600s）
- 控制流: TTL 过期后，边缘节点向源站发送条件请求，验证内容是否变化
- 关键设计: 支持主动刷新（purge）和预热（prefetch），但依赖外部触发

#### 优化层

**性能瓶颈**:
- TTL 过短导致缓存命中率低，频繁重新获取
- TTL 过长导致不一致窗口大，可能长时间读到过期数据
- 无法主动失效，服务端修改后客户端仍可能使用旧缓存

**针对性优化**:
- **自适应 TTL**: 根据数据修改频率动态调整 TTL（如热点数据 TTL 短，冷数据 TTL 长）
- **条件请求**: HTTP 使用 ETag/Last-Modified，客户端过期后先验证是否变化，未变化则继续使用缓存
- **主动刷新**: CDN 支持 API 触发缓存刷新，缩短不一致窗口
- **分层 TTL**: 不同类型数据设置不同 TTL（如目录元数据 TTL 长，文件属性 TTL 短）

**Trade-off 取舍**:

| 维度 | 弱一致性优势 | 弱一致性代价 |
|------|-------------|-------------|
| 正确性 | 实现���单，无需服务端状态 | 不一致窗口长，可能长时间读到过期数据 |
| 性能 | 缓存命中率高，无失效通知开销 | TTL 过期后需重新获取，无法主动失效 |
| 可用性 | 服务端故障不影响缓存使用 | 无法保证一致性，依赖应用层容忍 |
| 适用场景 | 读多写少、容忍长时间不一致 | 不适合强一致性或最终一致性要求场景 |

**可扩展性限制**:
- TTL 设置需要权衡一致性与性能，难以找到最优值
- 无法应对突发写操作（如批量修改），不一致窗口无法缩短
- 极端情况下可能退化为无缓存（TTL 设置为 0）

### 方案 D: 混合模型

**标杆来源**: Ceph MDS cache、GlusterFS metadata cache

#### 原理层

**解决的问题**: 结合多种一致性模型的优势，根据数据类型和访问模式选择合适的一致性策略。

**核心机制**:
- 对不同类型的元数据使用不同的一致性模型
- 如目录结构使用强一致性（租约），文件属性使用最终一致性（失效通知），访问时间使用弱一致性（TTL）
- 根据工作负载动态调整一致性策略

**理论复杂度**:
- 读操作延迟: O(1) ~ O(RTT)，取决于数据类型和一致性模型
- 写操作延��: O(RTT) ~ O(N×RTT)，取决于数据类型和一致性模型
- 一致性保证: 混合一致性，不同数据类型保证不同级别

#### 设计层

**Ceph MDS cache 混合机制**:
- 组件划分: 客户端 MDS client、MDS 服务端、capability 状态机、TTL 管理器
- 数据流: 
  - 目录结构: 使用 DIR capability（强一致性），MDS 修改前回收所有客户端 capability
  - 文件属性: 使用 FILE_CACHE capability（最终一致性），MDS 修改后异步发送 revoke
  - 访问时间: 使用 TTL（弱一致性），客户端本地更新，定期同步到 MDS
- 控制流: 根据 capability 类型选择不同的失效策略
- 关键设计: capability 分为多种类型，每种类型对应不同的一致性保证

**GlusterFS metadata cache 混合机制**:
- 组件划分: 客户端 FUSE client、brick 服务器、inode table、dentry cache
- 数据流:
  - inode 属性: 使用 TTL（弱一致性），默认 1s 过期
  - dentry 缓存: 使用失效通知（最终一致性），服务端修改后通知客户端
  - 文件锁: 使用租约（强一致性），持有锁期间保证独占访问
- 控制流: 根据数据类型和操作类型选择不同的一致性策略
- 关键设计: 分层缓存，不同层级使用不同的一致性模型

#### 优化层

**性能瓶颈**:
- 多种一致性模型增加实现复杂度，需要维护多套状态机
- 不同一致性模型之间可能存在冲突（如 TTL 过期但 capability 仍有效）
- 一致性策略选择不当可能导致性能下降或一致性问题

**针对性优化**:
- **策略自适应**: Ceph 根据工作负载动态调整 capability 类型（如只读工作负载使用 FILE_CACHE，读写工作负载使用 FILE_WR）
- **一致性降级**: GlusterFS 在高负载时自动降级一致性（如从强一致性降级为最终一致性）
- **分层失效**: 对不同层级的缓存使用不同的失效策略（如 inode 使用 TTL，dentry 使用失效通知）
- **冲突检测**: 在多种一致性模型之间建立优先级（如 capability 优先于 TTL）

**Trade-off 取舍**:

| 维度 | 混合模型优势 | 混合模型代价 |
|------|-------------|-------------|
| 正确性 | 根据数据类型选择合适的一致性保证 | 实现复杂，需处理多种一致性模型之间的冲突 |
| 性能 | 平衡一致性与性能，避免过度保守或过度激进 | 策略选择不当可能导致性能下降 |
| 可用性 | 部分数据类型故障不影响其他数据类型 | 需要维护多套状态机，增加维护成本 |
| 适用场景 | 复杂工作负载，不同数据类型有不同一致性要求 | 不适合简单工作负载或一致性要求统一的场景 |

**可扩展性限制**:
- 一致性策略数量增加时，状态机复杂度指数增长
- 需要引入策略配置和动态调整机制
- 极端情况下可能退化为单一一致性模型（如全部使用强一致性）

## 维度 2.2: 失效传播机制

### 方案 A: 主动推送（callback 模型）

**标杆来源**: AFS callback、Ceph capability revoke、SMB oplock break

#### 原理层

**解决的问题**: 服务端修改数据后，立即通知所有持有缓存的客户端失效，最小化不一致窗口。

**核心机制**:
- 服务端维护 (data, client) 映射表，记录哪些客户端持有哪些数据的缓存
- 服务端修改数据前，遍历映射表向所有客户端发送失效通知（callback/revoke）
- 客户端收到通知后，清理缓存并确认，服务端等待所有确认后继续

**理论复杂度**:
- 失效延迟: O(N×RTT) N 为持有缓存的客户端数
- 服务端状态空间: O(M×N) M 为数据项数，N 为客户端数
- 网络开销: O(N) 每次修改需发送 N 个失效通知

#### 设计层

**AFS callback 机制**:
- 组件划分: 客户端 Cache Manager、服务端 File Server、callback 状态表
- 数据流: 客户端首次访问时，服务端记录 (file, client) 到 callback 表
- 控制流: 服务端修改文件前，遍历 callback 表发送 break callback RPC，等待客户端确认
- 关键设计: callback 是双向 RPC，客户端必须响应，服务端超时后强制失效

**Ceph capability revoke 机制**:
- 组件划分: 客户端 MDS client、MDS 服务端、capability 状态机
- 数据流: MDS 维护 (inode, client, caps) 三元组，记录每个客户端持有的 capability
- 控制流: MDS 需要回收时发送 revoke 消息，客户端释放 capability 并确认，MDS 等待所有确认
- 关键设计: 支持部分回收（如只回收 FILE_WR，保留 FILE_RD），减少失效范围

**SMB oplock break 机制**:
- 组件划分: 客户端 SMB client、服务端 SMB server、oplock 状态表
- 数据流: 客户端持有 oplock（独占/批量/级别 II），服务端记录 (file, client, oplock_type)
- 控制流: 其他客户端请求冲突操作时，服务端发送 oplock break，当前持有者降级或释放
- 关键设计: 支持 oplock 降级（如从独占降级为级别 II），避免完全失效

#### 优化层

**性能瓶颈**:
- 客户端数量大时，失效通知延迟线性增长（O(N×RTT)）
- 服务端需要维护大量状态（O(M×N)），内存开销高
- 客户端故障或网络分区导致失效通知超时，阻塞写操作

**针对性优化**:
- **异步通知**: AFS 支持异步 callback break，不等待客户端确认即可继续（牺牲强一致性）
- **批量失效**: Ceph 将多个 revoke 消息合并为一个 RPC，减少网络开销
- **失效分组**: 将客户端分组，先通知活跃客户端，再通知非活跃客户端
- **超时强制失效**: 设置失效通知超时（如 30s），超时后强制标记客户端缓存失效

**Trade-off 取舍**:

| 维度 | 主动推送优势 | 主动推送代价 |
|------|-------------|-------------|
| 一致性 | 不一致窗口最小（RTT 级别） | 客户端故障或网络分区影响一致性 |
| 性能 | 缓存命中率高，无需客户端轮询 | 失效延迟随客户端数线性增长 |
| 可扩展性 | 客户端数 <1000 时性能可接受 | 客户端数 >1000 时失效延迟过高 |
| 服务端负载 | 需要维护大量状态（O(M×N)） | 内存和 CPU 开销高 |

**可扩展性限制**:
- 客户端数 >10000 时，失效通知风暴可能导致服务端过载
- 需要引入失效通知批量化、分组通知、异步通知等优化
- 极端情况下可能退化为被动拉取（客户端轮询）

---

### 方案 B: 被动拉取（版本号模型）

**标杆来源**: HDFS block version、Cassandra timestamp、DynamoDB version

#### 原理层

**解决的问题**: 避免服务端维护客户端状态，客户端自行检测数据是否过期，降低服务端负载。

**核心机制**:
- 服务端为每个数据项维护版本号（version/timestamp），每次修改时递增
- 客户端缓存数据时记录版本号，访问时将版本号发送给服务端
- 服务端比较版本号，如果客户端版本过期则返回新数据，否则返回 304 Not Modified

**理论复杂度**:
- 失效延迟: O(RTT) 客户端下次访问时检测
- 服务端状态空间: O(M) 只需维护数据项版本号，无需记录客户端
- 网络开销: O(1) 每次访问需发送版本号，但无需额外失效通知

#### 设计层

**HDFS block version 机制**:
- 组件划分: 客户端 DFSClient、NameNode、block location cache
- 数据流: 客户端缓存 block location + generation stamp（版本号）
- 控制流: 客户端访问 DataNode 时携带 generation stamp，DataNode 检测版本不匹配则返回错误
- 关键设计: 被动检测，客户端访问失败后清理缓存重新获取，无主动失效机制

**Cassandra timestamp 机制**:
- 组件划分: 客户端、Cassandra 节点、timestamp 列
- 数据流: 每次写入时附加 timestamp（微秒级时间戳），读取时返回最新 timestamp 的数据
- 控制流: 客户端缓存数据 + timestamp，下次读取时比较 timestamp，过期则使用新数据
- 关键设计: Last-Write-Wins 冲突解决，依赖时钟同步

**DynamoDB version 机制**:
- 组件划分: 客户端 SDK、DynamoDB 服务端、version 属性
- 数据流: 每个 item 包含 version 属性，每次更新时递增
- 控制流: 客户端读取时获取 version，更新时使用条件写入（version 匹配才更新）
- 关键设计: 乐观并发控制，冲突时客户端重试

#### 优化层

**性能瓶颈**:
- 客户端每次访问都需要发送版本号，增加请求大小
- 版本号不匹配时需要重新获取数据，增加延迟
- 无法主动失效，客户端可能长时间使用过期缓存（直到下次访问）

**针对性优化**:
- **条件请求**: HTTP 使用 If-None-Match（ETag）或 If-Modified-Since，服务端返回 304 时客户端继续使用缓存
- **版本号压缩**: 使用哈希值或短版本号代替完整时间戳，减少请求大小
- **批量验证**: 客户端批量发送多个版本号，服务端批量返回过期列表
- **混合模式**: 结合 TTL，TTL 未过期时不验证版本号，TTL 过期后再验证

**Trade-off 取舍**:

| 维度 | 被动拉取优势 | 被动拉取代价 |
|------|-------------|-------------|
| 一致性 | 客户端下次访问时保证一致性 | 不一致窗口 = 两次访问间隔，可能较长 |
| 性能 | 无失效通知开销，服务端无需维护客户端状态 | 每次访问需验证版本号，增加延迟 |
| 可扩展性 | 服务端状态空间 O(M)，与客户端数无关 | 客户端数增加时，版本验证请求增加 |
| 服务端负载 | 无需维护客户端状态，内存开销低 | 每次访问需比较版本号，CPU 开销增加 |

**可扩展性限制**:
- 客户端访问频率高时，版本验证开销可能超过主动推送
- 需要引入 TTL 或其他机制减少验证频率
- 极端情况下可能退化为无缓存（每次都验证版本号）

### 方案 C: 租约过期模型

**标杆来源**: Chubby lock service、ZooKeeper ephemeral node、etcd lease

#### 原理层

**解决的问题**: 通过租约自动过期机制，避免客户端故障导致的资源泄漏，同时保证一致性。

**核心机制**:
- 客户端获取租约时指定有效期（如 30s），租约过期后自动失效
- 客户端需要定期续约（如每 10s 发送心跳），否则租约过期
- 服务端检测到租约过期后，自动清理客户端状态

**理论复杂度**:
- 失效延迟: O(lease_timeout) 租约过期时间
- 服务端状态空间: O(N) 只需维护活跃租约
- 网络开销: O(N/renewal_interval) 客户端定期续约

#### 设计层

**Chubby lock service 机制**:
- 组件划分: 客户端 Chubby client、Chubby 服务端、lease 管理器
- 数据流: 客户端获取锁时获得 lease（默认 12s），定期发送 KeepAlive 续约
- 控制流: 服务端检测到 lease 过期后，自动释放锁并通知其他等待客户端
- 关键设计: lease 过期时间由服务端控制，客户端只能请求续约

**ZooKeeper ephemeral node 机制**:
- 组件划分: 客户端 ZooKeeper client、ZooKeeper 服务端、session 管理器
- 数据流: 客户端创建 ephemeral node 时绑定 session，session 超时后 node 自动删除
- 控制流: 客户端定期发送心跳（默认每 1/3 session timeout），服务端检测超时后删除 node
- 关键设计: session timeout 由客户端和服务端协商，服务端可以拒绝过短或过长的 timeout

**etcd lease 机制**:
- 组件划分: 客户端 etcd client、etcd 服务端、lease 管理器
- 数据流: 客户端创建 lease 时指定 TTL（如 60s），将 key 绑定到 lease
- 控制流: 客户端定期发送 LeaseKeepAlive 续约，lease 过期后所有绑定的 key 自动删除
- 关键设计: 一个 lease 可以绑定多个 key，批量续约减少网络开销

#### 优化层

**性能瓶颈**:
- 客户端需要定期续约，增加网络开销和服务端负载
- 租约过期时间过短导致频繁续约，过长导致故障检测延迟
- 网络分区时客户端无法续约，租约过期导致缓存失效

**针对性优化**:
- **自适应租约**: 根据客户端行为动态调整租约时长（如活跃客户端租约长，非活跃客户端租约短）
- **批量续约**: etcd 支持一个 lease 绑定多个 key，一次续约更新所有 key
- **租约续期优化**: Chubby 使用 piggybacking，将续约请求附加到其他 RPC 中，减少独立续约请求
- **租约降级**: 网络分区时客户端自动降级为只读模式，避免租约过期导致完全失效

**Trade-off 取舍**:

| 维度 | 租约过期优势 | 租约过期代价 |
|------|-------------|-------------|
| 一致性 | 租约过期后自动失效，避免永久不一致 | 租约过期时间内可能存在不一致 |
| 性能 | 无需主动失效通知，服务端自动清理 | 客户端需要定期续约，增加网络开销 |
| 可用性 | 客户端故障自动清理，无资源泄漏 | 网络分区时租约过期导致缓存失效 |
| 故障检测 | 租约过期时间 = 故障检测延迟 | 过短导致频繁续约，过长导致检测延迟 |

**可扩展性限制**:
- 客户端数 >10000 时，续约请求可能导致服务端过载
- 需要引入批量续约、续约合并等优化
- 极端情况下可能退化为无租约（租约时间设置为无限长）

### 方案 D: 混合失效机制

**标杆来源**: Ceph MDS cache、GFS2 glock、Lustre LDLM

#### 原理层

**解决的问题**: 结合主动推送、被动拉取、租约过期的优势，根据场景选择合适的失效机制。

**核心机制**:
- 对不同类型的数据使用不同的失效机制
- 如热点数据使用主动推送（低延迟），冷数据使用被动拉取（低开销），长期缓存使用租约过期（自动清理）
- 根据工作负载动态调整失效策略

**理论复杂度**:
- 失效延迟: O(RTT) ~ O(lease_timeout)，取决于失效机制
- 服务端状态空间: O(M) ~ O(M×N)，取决于失效机制
- 网络开销: O(1) ~ O(N)，取决于失效机制

#### 设计层

**Ceph MDS cache 混合失效**:
- 组件划分: 客户端 MDS client、MDS 服务端、capability 状态机、TTL 管理器
- 数据流:
  - 目录结构: 主动推送（revoke capability），MDS 修改前通知所有客户端
  - 文件属性: 租约过期（capability TTL），客户端定期续约或自动过期
  - 访问时间: 被动拉取（version 号），客户端访问时检测版本变化
- 控制流: 根据 capability 类型和数据类型选择失效机制
- 关键设计: 三种失效机制共存，通过 capability 类型区分

**GFS2 glock 混合失效**:
- 组件划分: 客户端 glock holder、DLM、锁状态机
- 数据流:
  - 元数据锁: 主动推送（DLM demote callback），其他客户端请求时触发
  - 数据锁: 租约过期（锁超时），客户端定期续约或自动释放
  - 只读锁: 被动拉取（锁版本号），客户端访问时检测版本变化
- 控制流: 根据锁类型和访问模式选择失效机制
- 关键设计: DLM 支持多种失效策略，客户端可以选择

**Lustre LDLM 混合失效**:
- 组件划分: 客户端 LDLM client、MDT、锁状态机
- 数据流:
  - intent lock: 主动推送（blocking AST），MDT 修改时通知客户端
  - extent lock: 租约过期（锁 TTL），客户端定期续约或自动释放
  - lookup lock: 被动拉取（版本号），客户端访问时检测版本变化
- 控制流: 根据锁类型和 intent 选择失效机制
- 关键设计: 锁类型与失效机制绑定，简化实现

#### 优化层

**性能瓶颈**:
- 多种失效机制增加实现复杂度，需要维护多套状态机
- 失效机制选择不当可能导致性能下降或一致性问题
- 不同失效机制之间可能存在冲突（如主动推送与租约过期同时触发）

**针对性优化**:
- **策略自适应**: Ceph 根据工作负载动态调整失效机制（如高负载时使用租约过期，低负载时使用主动推送）
- **失效优先级**: GFS2 建立失效机制优先级（如主动推送优先于租约过期）
- **失效合并**: Lustre 将多个失效通知合并为一个 RPC，减少网络开销
- **冲突检测**: 在多种失效机制之间建立协调机制，避免冲突

**Trade-off 取舍**:

| 维度 | 混合失效优势 | 混合失效代价 |
|------|-------------|-------------|
| 一致性 | 根据数据类型选择合适的失效机制 | 实现复杂，需处理多种失效机制之间的冲突 |
| 性能 | 平衡延迟与开销，避免过度保守或过度激进 | 策略选择不当可能导致性能下降 |
| 可扩展性 | 部分失效机制故障不影响其他机制 | 需要维护多套状态机，增加维护成本 |
| 适用场景 | 复杂工作负载，不同数据类型有不同失效要求 | 不适合简单工作负载或失效要求统一的场景 |

**可扩展性限制**:
- 失效机制数量增加时，状态机复杂度指数增长
- 需要引入策略配置和动态调整机制
- 极端情况下可能退化为单一失效机制（如全部使用主动推送）

## 维度 2.3: 并发写冲突处理

### 方案 A: 写穿透（write-through）

**标杆来源**: NFS v3、AFS、早期 Ceph

#### 原理层

**解决的问题**: 保证写操作立即持久化到服务端，避免数据丢失和一致性问题。

**核心机制**:
- 客户端写操作直接发送到服务端，等待服务端确认后才返回
- 客户端缓存只用于读操作，写操作不经过缓存
- 服务端串行化所有写操作，保证一致性

**理论复杂度**:
- 写操作延迟: O(RTT) 每次写都需要等待服务端确认
- 写吞吐量: 受限于服务端处理能力和网络带宽
- 一致性保证: 强一致性，所有客户端立即看到最新数据

#### 设计层

**NFS v3 write-through 机制**:
- 组件划分: 客户端 NFS client、服务端 NFS server、STABLE 标志
- 数据流: 客户端写操作发送到服务端，服务端写入磁盘后返回确认
- 控制流: 客户端等待服务端确认，确认后更新本地缓存
- 关键设计: STABLE 标志控制是否等待磁盘持久化，FILE_SYNC 模式保证强一致性

**AFS write-through 机制**:
- 组件划分: 客户端 Cache Manager、服务端 File Server、store 操作
- 数据流: 客户端关闭��件时，将整个文件发送到服务端（store 操作）
- 控制流: 服务端接收文件后，向所有持有 callback 的客户端发送 break callback
- 关键设计: 文件级写穿透，关闭时才同步，期间本地修改对其他客户端不可见

**早期 Ceph write-through 机制**:
- 组件划分: 客户端 MDS client、MDS 服务端、OSD
- 数据流: 客户端写元数据时，直接发送到 MDS，MDS 更新后返回确认
- 控制流: 客户端等待 MDS 确认，确认后更新本地缓存
- 关键设计: 元数据写穿透，数据写可以使用 write-back

#### 优化层

**性能瓶颈**:
- 每次写操作都需要等待服务端确认，延迟高（O(RTT)）
- 写吞吐量受限于服务端处理能力，无法利用客户端缓存
- 小文件写操作频繁时，网络开销大

**针对性优化**:
- **批量写**: 将多个小写操作合并为一个大写操作，减少 RPC 次数
- **异步确认**: 客户端发送写操作后立即返回，后台等待确认（牺牲强一致性）
- **写合并**: 服务端将多个客户端的写操作合并，减少磁盘 IO
- **写缓冲**: 服务端使用写缓冲（如 NVRAM），快速返回确认，后台持久化

**Trade-off 取舍**:

| 维度 | 写穿透优势 | 写穿透代价 |
|------|-----------|-----------|
| 一致性 | 强一致性，所有客户端立即看到最新数据 | 实现简单，无需处理缓存一致性 |
| 性能 | 写延迟高（O(RTT)），吞吐量低 | 无法利用客户端缓存，网络开销大 |
| 可靠性 | 数据立即持久化，无数据丢失风险 | 服务端故障导致写操作失败 |
| 适用场景 | 强一致性要求、写操作少、小文件 | 不适合高并发写、大文件、低延迟要求 |

**可扩展性限制**:
- 写吞吐量受限于服务端处理能力，无法水平扩展
- 客户端数增加时，服务端成为瓶颈
- 极端情况下可能退化为串行写（服务端单线程处理）

---

### 方案 B: 写回（write-back）

**标杆来源**: Lustre client cache、GlusterFS write-behind、GPFS

#### 原理层

**解决的问题**: 通过客户端缓存写操作，降低写延迟，提高写吞吐量。

**核心机制**:
- 客户端写操作先写入本地缓存，立即返回
- 后台异步将脏数据刷新到服务端
- 服务端需要处理多个客户端的并发写冲突

**理论复杂度**:
- 写操作延迟: O(1) 本地缓存写入
- 写吞吐量: 受限于客户端缓存大小和刷新速度
- 一致性保证: 最终一致性，存在不一致窗口

#### 设计层

**Lustre client cache write-back 机制**:
- 组件划分: 客户端 LLITE、OSC（Object Storage Client）、OST（Object Storage Target）
- 数据流: 客户端写操作先写入 page cache，后台 pdflush 线程异步刷新到 OST
- 控制流: 客户端持有 extent lock 期间可以缓存脏数据，锁被回收时强制刷新
- 关键设计: extent lock 保证独占写权限，避免多客户端并发写冲突

**GlusterFS write-behind 机制**:
- 组件划分: 客户端 FUSE client、write-behind translator、brick 服务器
- 数据流: 客户端写操作先写入 write-behind 缓冲区，立即返回，后台异步刷新到 brick
- 控制流: 缓冲区满或超时时触发刷新，fsync/close 时强制刷新
- 关键设计: write-behind 是可选的 translator，可以动态启用/禁用

**GPFS write-back 机制**:
- 组件划分: 客户端 GPFS client、token manager、NSD（Network Shared Disk）
- 数据流: 客户端持有 write token 期间可以缓存脏数据，后台异步刷新到 NSD
- 控制流: token 被回收时强制刷新，保证一致性
- 关键设计: token 机制保证独占写权限，支持 token 降级（如从写降级为读）

#### 优化层

**性能瓶颈**:
- 脏数据刷新延迟导致不一致窗口，可能读到过期数据
- 客户端故障导致脏数据丢失，需要恢复机制
- 多客户端并发写同一文件时，需要协调机制避免冲突

**针对性优化**:
- **锁机制**: Lustre 使用 extent lock，GPFS 使用 token，保证独占写权限
- **强制刷新**: fsync/close 时强制刷新脏数据，保证持久化
- **刷新优先级**: 根据数据重要性调整刷新优先级（如元数据优先于数据）
- **故障恢复**: 客户端故障后，服务端检测到锁超时，标记数据为可疑，需要恢复

**Trade-off 取舍**:

| 维度 | 写回优势 | 写回代价 |
|------|---------|---------|
| 一致性 | 最终一致性，存在不一致窗口 | 需要锁机制保证独占写权限 |
| 性能 | 写延迟低（O(1)），吞吐量高 | 脏数据刷新延迟影响一致性 |
| 可靠性 | 客户端故障导致脏数据丢失 | 需要故障恢复机制 |
| 适用场景 | 高并发写、大文件、低延迟要求 | 不适合强一致性要求、小文件 |

**可扩展性限制**:
- 客户端缓存大小限制脏数据量，缓存满时需要刷新
- 锁机制限制并发写能力，多客户端写同一文件时串行化
- 极端情况下可能退化为写穿透（缓存禁用或锁频繁回收）

---

### 方案 C: 写失效（write-invalidate）

**标杆来源**: Ceph MDS cache、GFS2 glock、分布式缓存系统

#### 原理层

**解决的问题**: 通过失效其他客户端缓存，保证写操作的一致性，同时允许客户端缓存写操作。

**核心机制**:
- 客户端写操作前，先向服务端请求独占权限
- 服务端失效所有其他客户端的缓存，授予独占权限
- 客户端持有独占权限期间可以缓存写操作，其他客户端无法访问

**理论复杂度**:
- 写操作延迟: O(N×RTT) 需要失效所有其他客户端缓存
- 写吞吐量: 受限于失效延迟和独占权限持有时间
- 一致性保证: 强一致性，独占权限保证无并发写

#### 设计层

**Ceph MDS cache write-invalidate 机制**:
- 组件划分: 客户端 MDS client、MDS 服务端、capability 状态机
- 数据流: 客户端请求 FILE_WR capability，MDS 回收所有其他客户端的 FILE_CACHE capability
- 控制流: MDS 等待所有客户端确认后，授予 FILE_WR capability，客户端可以缓存写操作
- 关键设计: capability 分为多种类型，FILE_WR 与 FILE_CACHE 互斥

**GFS2 glock write-invalidate 机制**:
- 组件划分: 客户端 glock holder、DLM、锁状态机
- 数据流: 客户端请求排他锁（EX mode），DLM 回收所有其他客户端的共享锁（SH mode）
- 控制流: DLM 等待所有客户端释放共享锁后，授予排他锁，客户端可以缓存写操作
- 关键设计: 锁模式转换（SH → EX），支持锁降级（EX → SH）

**分布式缓存系统 write-invalidate 机制**:
- 组件划分: 客户端缓存节点、中心协调器、缓存一致性协议
- 数据流: 客户端写操作前，向协调器请求独占权限，协调器失效所有其他节点缓存
- 控制流: 协调器等待所有节点确认后，授予独占权限，客户端可以写入
- 关键设计: 类似 MESI 协议的 Invalidate 状态转换

#### 优化层

**性能瓶颈**:
- 写操作需要失效所有其他客户端缓存，延迟随客户端数线性增长
- 独占权限持有时间过长导致其他客户端阻塞
- 频繁的独占权限请求导致缓存频繁失效，命中率下降

**针对性优化**:
- **细粒度锁**: Ceph 使用 inode 级 capability，GFS2 使用 extent 级锁，减少冲突范围
- **锁降级**: 写操作完成后立即降级为共享锁，允许其他客户端读取
- **批量失效**: 将多个失效通知合并为一个 RPC，减少网络开销
- **异步失效**: 不等待所有客户端确认，超时后强制失效

**Trade-off 取舍**:

| 维度 | 写失效优势 | 写失效代价 |
|------|-----------|-----------|
| 一致性 | 强一致性，独占权限保证无并发写 | 实现复杂，需要失效协议 |
| 性能 | 写延迟高（O(N×RTT)），但持有权限期间可以缓存 | 频繁失效导致缓存命中率下降 |
| 可扩展性 | 客户端数增加时，失效延迟线性增长 | 需要细粒度锁和批量失效优化 |
| 适用场景 | 读多写少、写操作集中在少数客户端 | 不适合高并发写、多客户端写同一文件 |

**可扩展性限制**:
- 客户端数 >1000 时，失效延迟可达数秒
- 需要引入细粒度锁、异步失效等优化
- 极端情况下可能退化为写穿透（无缓存）

---

### 方案 D: 混合写策略

**标杆来源**: Ceph 混合模式、GPFS 自适应模式

#### 原理层

**解决的问题**: 结合写穿透、写回、写失效的优势，根据工作负载选择合适的写策略。

**核心机制**:
- 对不同类型的数据使用不同的写策略
- 如元数据使用写穿透（强一致性），大文件使用写回（高吞吐量），小文件使用写失效（平衡一致性与性能）
- 根据工作负载动态调整写策略

**理论复杂度**:
- 写操作延迟: O(1) ~ O(N×RTT)，取决于写策略
- 写吞吐量: 受限于策略选择和工作负载
- 一致性保证: 混合一致性，不同数据类型保证不同级别

#### 设计层

**Ceph 混合写策略**:
- 组件划分: 客户端 MDS client、MDS 服务端、OSD、capability 状态机
- 数据流:
  - 元数据: 写穿透，客户端直接发送到 MDS，等待确认
  - 大文件数据: 写回，客户端缓存脏数据，后台刷新到 OSD
  - 小文件数据: 写失效，客户端请求 FILE_WR capability，失效其他客户端缓存
- 控制流: 根据数据类型和文件大小选择写策略
- 关键设计: capability 类型与写策略绑定，简化实现

**GPFS 自适应写策略**:
- 组件划分: 客户端 GPFS client、token manager、NSD
- 数据流:
  - 独占访问: 写回，客户端持有 write token 期间缓存脏数据
  - 共享访问: 写穿透，客户端直接发送到 NSD，等待确认
  - 冲突访问: 写失效，客户端请求 write token，失效其他客户端缓存
- 控制流: 根据 token 状态和访问模式选择写策略
- 关键设计: token 状态机自动切换写策略，无需手动配置

#### 优化层

**性能瓶颈**:
- 多种写策略增加实现复杂度，需要维护多套状态机
- 写策略选择不当可能导致性能下降或一致性问题
- 不同写策略之间可能存在冲突（如写回与写失效同时触发）

**针对性优化**:
- **策略自适应**: GPFS 根据访问模式动态调整写策略（如独占访问时使用写回，共享访问时使用写穿透）
- **策略优先级**: Ceph 建立写策略优先级（如元数据写穿透优先于数据写回）
- **策略切换**: 支持运行时切换写策略（如从写回切换到写穿透），无需重启
- **冲突检测**: 在多种写策略之间建立协调机制，避免冲突

**Trade-off 取舍**:

| 维度 | 混合写策略优势 | 混合写策略代价 |
|------|---------------|---------------|
| 一致性 | 根据数据类型选择合适的一致性保证 | 实现复杂，需处理多种写策略之间的冲突 |
| 性能 | 平衡延迟与吞吐量，避免过度保守或过度激进 | 策略选择不当可能导致性能下降 |
| 可扩展性 | 部分写策略故障不影响其他策略 | 需要维护多套状态机，增加维护成本 |
| 适用场景 | 复杂工作负载，不同数据类型有不同写要求 | 不适合简单工作负载或写要求统一的场景 |

**可扩展性限制**:
- 写策略数量增加时，状态机复杂度指数增长
- 需要引入策略配置和动态调整机制
- 极端情况下可能退化为单一写策略（如全部使用写穿透）
