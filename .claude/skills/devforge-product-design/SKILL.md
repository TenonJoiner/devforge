# devforge-product-design — 产品级架构探索与决策

## 概述

架构探索是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续多个阶段的思考-研究-发散-评审-收敛循环，产出高质量的架构设计文档。

> **内容规范强牵引**：架构交付件的所有内容结构、章节要求、自检清单**以 `.claude/templates/*.md` 为唯一准绳**。本 skill 只负责定义**努力程度、写作风格、质量门槛和流程控制**，不再重复模板中已有的内容要求。

**核心原则**：
1. **模板牵引**：所有产出必须严格遵循对应模板，写前先读模板是铁律
2. **标杆先行**：不研究业界同类产品的架构，不允许提出自研方案
3. **深度对比**：每个维度至少 ≥2 个候选方案，每个方案必须有明确的 Trade-off 分析
4. **方案 > 代码**：聚焦设计决策和 rationale，不讨论具体代码实现
5. **产品级颗粒度**：只定义核心框架（子系统边界、接口契约、关键路径、不可逆决策），特性级实现细节归 `/opsx:*`
6. **长时间迭代**：架构文档需要 5-20 个阶段才能定稿，禁止一次对话定架构

## 通用规范

### 通用术语

#### 问题分级

问题级别基于对**整个产品系统级**的影响判定，不按文档局部影响放大。

| 级别 | 含义 | 判定基准 | 处理方式 |
|------|------|---------|----------|
| **CRITICAL** | 阻塞性问题，会导致系统方向错误 | **系统级影响**（同时满足：影响核心决策/方向 + 不修正会推倒重来 + 不可逆或高成本） | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响局部质量 | **文档/模块级影响**，不改变系统整体方向 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | **优化建议**，不影响功能正确性 | 评估后决定 |
| **LOW** | 轻微问题 | 格式、命名、注释等非功能性 | 可延期 |

> 常见误判：子系统内部结构问题、单 ADR 论证不足 → 归 HIGH（局部问题，可独立修复）；核心一致性协议选型、整体架构分层错误 → 归 CRITICAL（系统级方向性决策）。reviewer agent 在评审 prompt 中需明确这个判定边界。

### 标准评审修正循环

各阶段遵循统一的评审修正循环。具体门槛值（缺陷密度阈值、评估对象数计算方式）见各阶段规则。

1. **独立评审**：并行启动 ≥2 个 reviewer，按模板自检清单提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **计算缺陷密度**：所有问题分数之和 / 评估对象数

   问题分值：**CRITICAL=10分, HIGH=3分, MEDIUM=1分, LOW=0.1分**

3. **判断退出条件**（两步判定）：
   - **先判断 CRITICAL 问题**（一票否决）：有 CRITICAL → 进入修正循环；无 CRITICAL → 继续
   - **再判断缺陷密度**：达标 → 成功退出；超标 → 进入修正循环
4. **修正策略**（此为唯一定义，各阶段文件不得重复）

   基于问题数量决定 agent 配置：
   - < 10 处：1 个 agent 读取文件 + review 文件后 Edit 修正
   - 10-30 处：2 个 agent 按产出对象/维度分块并行修正，各自 Edit 对应部分
   - \> 30 处：按各阶段规则判断是否回退到对应步骤重新执行多 agent 并行发散（回退规则见各阶段，各阶段不重复定义此处 3 档策略本身）
   - 分工原则：问题集中在某个产出对象 → 针对该对象；问题分散 → 按产出对象并行；问题集中在某个维度 → 按维度分工
   - 修正 agent 直接 Edit 文件，主会话只确认修正完成状态，不中转修正内容

5. **复核**：修正完成后回到步骤 1（独立评审）
6. **循环上限**：最多 3 次
7. **降级规则**：3 次循环后仍未达标 → 进入人工决策（见步骤 8）
8. **人工决策返回路径**：3 次循环未达标时，用户可选：接受现状（标记 `⚠️ ACCEPTED-WITH-DEBT` 并记录残留问题）、重启循环（按提示重新评审）、回退重做（按阶段规则回退到对应步骤）。
9. **评审状态标记契约**（机器可解析，启动检测依此判定阶段过渡）：

   ```markdown
   **评审状态**: ✅ PASS（缺陷密度 X/评估对象，无 CRITICAL，循环 N/3）
   ```

   - 仅当无 CRITICAL + 缺陷密度达标时写入 `✅ PASS`；评审中或未通过写入 `❌ FAIL` 或不写；接受带债放行写入 `⚠️ ACCEPTED-WITH-DEBT`（不视为 PASS，但允许阶段过渡）

### 缺陷密度门槛

| 阶段 | 门槛 | 评估单位 |
|------|------|---------|
| 第 1 阶段（标杆研究） | 2.0 分/标杆，CRITICAL 单个否决 | 每个标杆产品研究 |
| 第 2 阶段步骤 1（整体架构发散） | 2.0 分/方案，CRITICAL 单个否决 | 每个候选方案 |
| 第 2 阶段步骤 3（维度方案发散） | 1.5 分/方案，CRITICAL 单个否决 | 每个维度的候选方案 |
| 第 3 阶段步骤 1（ADR 收敛） | **0.5 分/ADR**，CRITICAL 单个否决 | 每条 ADR |
| 第 3 阶段步骤 2（系统总纲） | 1.5 分/评估对象，CRITICAL 单个否决 | 子系统数 + 全局章节数（数据流/一致性/部署拓扑等）|
| 第 3 阶段步骤 3（子系统设计） | 1.5 分/章节，CRITICAL 单个否决 | 章节（功能流/故障/接口/运维等）|
| 第 4 阶段（维护快速通道） | 3.0 分/修正点，CRITICAL 单个否决 | 修正点（单文件单章节）|

### 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

- **原始 agent**（researcher / architect）：将完整产出写入 `-draft-` 标记的中间文件，向主会话返回结构化摘要，包含以下字段（缺失项跳过，不超过 10 行）：
  - `产出`：写入的文件路径 + 核心产出数（方案数/章节数/ADR 数等）
  - `数字`：覆盖率、完整度等可量化指标
  - `异常`：覆盖缺口、数据矛盾、视角间分歧（附 1-2 句定位，供后续合并/评审参考）
  - `编排提示`：如"合并时需关注 X 与 Y 视角在 Z 上的分歧"、"建议 reviewer 侧重验证接口契约一致性"
- **合并 agent**（同类型新实例）：读取 draft 文件，执行合并/去重，写入最终文件，返回结构化摘要
- **Reviewer agent**：将评审意见追加到 `-review.md` 文件，返回数字摘要（issues 数、缺陷密度、CRITICAL 数）
- 所有文件写入后用 `ls` 验证存在，完成后才能汇报"本阶段完成"

**文件命名规范**：
- 所有非最终产出统一用 `-draft-` 标记，如 `decision-overall-draft-{view}.md`、`reference/<product>-draft-{view}.md`
- 评审意见统一追加到单 `-review.md` 文件，如 `decision-overall-review.md`、`reference/<product>-review.md`
- 最终文件不含 `-draft-` 和 `-review` 标记
- **子系统目录命名**：使用 kebab-case 英文标识，如 `docs/architecture/storage-engine/`，禁止使用中文字符
- 中文子系统名称仅用于文档内部标题和阅读显示

**draft 清理约束**（每个合并步骤的产出完整性检查通过后必须执行）：

- 时机：合并 agent 写入最终文件 → 主会话执行产出完整性检查 → **通过后立即清理 draft**，再进入评审步骤
- 命令：主会话用 Bash 执行 `rm <对应目录>/*-draft-*.md`（按各阶段文件的合并目录定位）
- 验证：清理后用 `ls` 确认 `*-draft-*.md` 已全部移除，且最终文件仍存在
- 禁止：draft 文件残留进入评审阶段或下一阶段；评审中需回退合并时，回退路径是重新派遣 agent 产出新 draft，不复用旧 draft
- 例外：若产出完整性检查未通过需回退到合并前的产出 agent 阶段，draft 暂时保留至重新合并并通过检查后再清理

**例外**：步骤内部的工作指令（如视角切分理由说明）属于执行中间件，可按对应阶段规则豁免落盘。

### 主会话职责边界

**主会话只负责**（必须由主会话亲自执行）：
- 调度 agent（决定派遣哪些 agent、视角切分、prompt 内容、并发节奏）
- 状态跟踪（记录各 agent 完成状态、文件路径、轻量摘要数字）
- 检查出口标准（文件存在性、✅ PASS 标记、数量门槛）
- 缺陷密度判定（从 reviewer 返回的数字摘要中读取，做通过/修正/回退决策）
- 与用户交互（确认、汇报、等待人工决策）

**已下放给 agent 的职责**（主会话不再执行）：
- 文件写入：agent 自行 Write/Edit 到 draft/review/最终文件
- 内容整合：由**同类型新 agent 实例**执行合并去重（researcher 合并 researcher 的 draft，architect 合并 architect 的 draft）
- 评审记录：reviewer 追加到 `-review.md`，合并 agent 将 PASS 标记写入最终文件

**边界警戒**（这些行为会破坏 agent 协作的质量保障机制）：
- 主会话不得读取 agent 完整产出或实质性修改其核心内容
- 禁止无记录声称完成或用单 agent 内部发散替代多 agent 独立产出
- 合并 agent 须与原始 agent 类型一致
- 禁止委托 plan agent 生成调度方案或设计决策树式"伪动态"

### Agent 并发控制

**核心参数**：`agent.max_concurrent` = **5**（单一 `Agent` 工具调用计为 1 个并发单元）

**滑动窗口策略**：初始启动 min(N, 5) 个 agent，每完成一个立即补位下一个，始终保持运行中 agent 数 = min(剩余, 5)。有依赖关系的 agent 必须等上游完成后才能入队。各阶段"并行启动"指令均受此限，禁止以并发限制为由压缩多 agent 步骤为单 agent。

### 评审纪要写作约束

> 写作风格约束，**不计入评审判定项**（reviewer 不应据此降级评审结论）。

**写作要求**：
- 写入评审记录前先读对应模板（`ref-architecture.md` / `arch-system.md` / `arch-adr.md` / `arch-subsystem.md`）的「评审记录」示例，模板定义了表格列、日期格式等具体要求
- 日期格式精确到分钟，如 `2026-05-09 14:30`
- 评审记录表格的「缺陷密度」列每一轮（含修正后的复核轮次）都必须有数值，不可留空
- 评审记录章节 ≤ 200 字
- 详细评审意见（逐条分析）留存对话上下文，不写入正式文档

### 视角切分原则

#### 发散视角（researcher / architect 等产出 agent）

多 agent 并行发散时，每个 agent 从独立、互补的视角产出。视角由主会话基于**本次产品的具体上下文**动态生成，在派遣 prompt 中说明切分理由。

**视角质量标准**：互补（不重叠）、独立（任一视角缺失会导致分析盲区）、有明确命名和切分理由。

**反面约束**：不得套用固定视角清单（如对所有产品都用"数据流/状态/性能/故障/演进"五视角）。视角服务于产品决策——不同产品的决策关注点不同，视角应反映这种差异。

#### 产出者派遣 prompt 必备字段（architect / researcher agent）

**核心原则**：产出者 agent 已解耦 `工作模式` / `template 路径` / `output 路径` 等流程性细节。skill 派遣时必须在 prompt 中显式注入：

1. **任务模式**（如 `架构探索主角` / `子系统设计主角` / `ADR 决策` / `计划评审顾问` / `需求评审顾问`）
2. **template_path**（如 `.claude/templates/arch-system.md` / `.claude/templates/adr.md` / `.claude/templates/subsystem-design.md`）
3. **mandatory_sections**（从 template 提取的必填章节清单）
4. **output_path**（如 `docs/architecture/design.md` / `docs/architecture/adr.md` / `docs/architecture/<subsystem>/design.md`）
5. **length_hint**（如 `ADR 80-200 行` / `系统总纲 300-600 行` / `子系统设计 200-500 行`）
6. **上游输入路径**（如 `docs/architecture/reference/<product>.md` 标杆研究、`docs/requirements/product-spec.md` 需求约束）

**researcher agent 额外字段**：
- **研究视角**（`架构研究`：系统/架构/实现视角，按三层剖析模型）
- **output_path**（`docs/architecture/reference/<product>.md`）

#### 评审视角（reviewer agent）

**核心原则**：评审视角不由 reviewer agent 自带，必须从被评审对象的相关来源获取。reviewer agent 只负责承载"评审思维风格"。三层视角分工如下：

| 视角层级 | 承担者 | 内容 |
|---------|------|------|
| **对象特异视角**（评审项） | 被评审 template | `mandatory-sections` + `checklist-at-end` 段（每个 template 已自带评审锚点） |
| **场景特异视角**（评审深度） | 主会话派遣 prompt | 基于 `domain_specific` / `quality_attributes.priorities` 前两项 / Non-Goals 等动态注入 |
| **评审思维风格** | reviewer agent 人设 | architect-reviewer = 技术/架构视角；product-reviewer = 业务/用户视角；project-reviewer = 进度/资源视角 |
| **评审报告格式** | `.claude/templates/review-report.md` | 统一输出格式，**不含**任何评审项；reviewer 按格式产出 |

**派遣 prompt 必备字段**（主会话每次派遣 reviewer 时必须传入）：

1. **被评审对象路径**（如 `docs/architecture/reference/<product>.md`）
2. **被评审 template 路径**（如 `.claude/templates/ref-architecture.md`，指示视角来源 1）
3. **特异性子维度清单**（基于本次产品上下文动态产出，指示视角来源 2）
4. **本次评审的复杂度指引**（简单/中等/复杂，对应最低质疑点数 3/5/7）
5. **评审报告产出路径**（如 `reference/<product>-review.md`，统一追加格式）
6. **被评审对象在系统中的位置说明**（作为问题分级判定的系统上下文）

**反面约束**：
- 禁止 skill 中硬编码 reviewer 的检查项清单（如"product-reviewer 看用户场景 / Non-Goals 覆盖度 / 量化指标合理性"——这类视角应在被评审 template 中或派遣 prompt 中表达）
- 禁止 skill 中硬编码 reviewer 数量（统一用 ≥N 描述下限）
- 禁止默认"两个不同类型 reviewer = 一种分工组合"——同一类型也可多实例做交叉验证（如 2 个 architect-reviewer 对比意见）
- 禁止 reviewer agent 在 prompt 之外携带任何评审项

各阶段文件引用本原则，不再重复定义。

## 复杂度判定

| 阶段 | 内容类型 | 复杂度 | agent 配置 |
|------|---------|-------|-----------|
| 第 0 阶段 | 交互式信息收集 | 低 | 1 个 architect（交互式，豁免评审循环） |
| 第 1 阶段 | 标杆研究（架构调研报告） | 高 | 每个标杆 ≥2 个 researcher + 整体 ≥2 个 reviewer |
| 第 2 阶段 步骤 1 | 整体架构发散（系统级方案） | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 2 阶段 步骤 3 | 维度方案发散 | 高 | 每维度 ≥3 个 architect + ≥2 个 reviewer |
| 第 3 阶段 步骤 1 | ADR 收敛（不可逆决策） | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 3 阶段 步骤 2 | 系统架构总纲定稿 | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 3 阶段 步骤 3 | 子系统架构定稿 | 高 | 每子系统 ≥3 个 architect + ≥2 个 reviewer |
| 第 4 阶段 | 维护模式快速通道 | 低 | 1 个 architect-reviewer 快速复核 |

**说明**：
- 极高/高复杂度的 agent 数下限是**阶段或步骤的总 agent 数**，不要求每个子单元独立达标
- 视角切分由主会话基于产品上下文动态决定，禁止套用固定视角清单
- **决策属性上调**：涉及「跨子系统边界 / 不可逆决策 / 性能或一致性关键路径 / 无现成方案 / 安全合规敏感」的文档，复杂度在基准上 +1 级（不超过"极高"）

## 启动检测

**核心原则**：状态判定以文件系统实时扫描（`ls`/`wc`/`Read`）为准。文件存在性是必要条件，**`✅ PASS` / `⚠️ ACCEPTED-WITH-DEBT` 评审状态标记是阶段过渡的充分条件**。**扫描与汇报范围仅限 `docs/architecture/` 与 `.claude/domain-config.yaml`**——其他 skill 的产出目录（`docs/requirements/` 等）不读、不列、不纳入判定。

**状态机**（按优先级从上到下匹配，命中即停）：

| # | 当前文件状态 | 下一步动作 |
|---|------------|-----------|
| 1 | `domain-config.yaml` 不存在 | → 第 0 阶段 |
| 2 | `reference/` 下标杆文件 < 2 个 | → 第 1 阶段步骤 1-2 |
| 3 | 任一标杆文件无 PASS 标记 | → 第 1 阶段步骤 3（评审修正） |

## 与用户协作模式

**核心边界**：自动推进与人工确认的切换点按**交付件是否完成闭环**划分，不按"阶段内/阶段间"简单切分。

**三种协作节奏**：
1. **自动推进**：交付件内部，产出→评审→修正循环全自动闭环，不中途询问用户
2. **确认后推进**：交付件闭环后，向用户汇报产出概况（产出文件、缺陷密度、关键决策点），等待确认后再进入下一交付件
3. **等待人工决策**：缺陷密度超过门槛、CRITICAL 未修正、信息缺失、或需用户做不可逆决策时停止

**阶段衔接规则**（确认后推进的具体约束）：

阶段 N 结束汇报时：
- **必须**：汇报阶段 N 的产出摘要（仅当前阶段的交付件成果）
- **必须**：告知下一阶段名称（如"下一步：进入第 1 阶段（标杆研究）"），等待用户确认推进
- **禁止**：预演下一阶段的具体步骤、工具、候选对象或中间产物
- **禁止**：提前执行下一阶段的任何步骤（包括搜索、筛选、候选列举）
- **禁止**：将下一阶段才应让用户确认的内容提前到当前阶段汇报中展示

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要

### 对话语言约束

Skill 内部的组织语汇（T1-T6 主题编号、字段名、枚举字面值、"锚点/闭合/评估对象/缺陷密度"等术语）是 agent 执行流程的施工图，**不出现在与用户的任何输出中**（提问、确认、推断展示、阶段汇报）。

- **提问**：把"信息槽要填什么"翻译成"为什么、在什么场景、为谁、放弃什么换什么"这类业务话术
- **展示推断**：用业务语言（如"几十节点的云端集群、单客户百 TB"），不展示"字段名 + 枚举值"两栏表
- **内部翻译**：用户回答 → 枚举值的映射由 agent 自己完成，不亮枚举字面值给用户
- **阶段汇报**：可提阶段名，但不展开内部主题清单或把 T 编号当 todo 进度条
