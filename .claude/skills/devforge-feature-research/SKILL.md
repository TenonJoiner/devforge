# devforge-feature-research — 特性级研究

## 概述

特性级研究是 OpenSpec workflow 中 proposal 与 spec/design 之间的桥梁。本 skill 产出三部分：
1. **自身约束清单**：本特性必须遵守的架构、协议、性能、范围约束
2. **标杆方案空间**：按取舍维度组织的标杆方案研究
3. **设计空间地图**：标杆 × 约束过滤后的可行候选空间

**与产品级 research 的区别**：
- 产品级（`/df:product-design` 第 1 阶段）：全系统标杆调研，产出 `docs/architecture/reference/*.md`
- 特性级（本 skill）：单特性约束发现 + 设计空间过滤，产出 `openspec/changes/<name>/research.md`

**核心原则**：
1. **约束驱动**：从产品级文档和 proposal 中提取约束，作为设计空间过滤依据
2. **标杆维度反查**：标杆方案的取舍维度反向驱动约束发现（"我们在这个维度的情况是什么"）
3. **设计空间地图**：让主人一眼看出"在我们的约束下，哪些候选可选、哪些被排除"
4. **skill 内化评审**：最多 3 轮 researcher → architect-reviewer 循环，保证基础质量

## 初次生成模式

### [1] 上下文准备

准备以下输入（无需 Read 工具，只是明确路径）：
1. **proposal.md**：本特性的动机、范围、Capabilities
2. **产品级文档索引**：CLAUDE.md#产品级文档索引（已在上下文）
3. **产品级架构文档**（按需）：`docs/architecture/` 下相关子系统设计、ADR
4. **产品级需求文档**（按需）：`docs/requirements/` 下相关 Feature 规格
5. **research.md template**：`openspec/schemas/spec-driven-enhanced/templates/research.md`

### [2] 标杆产品识别

派遣 1 个 researcher agent，任务：
- 读 proposal.md，理解特性的核心问题和技术挑战
- 识别 **3-5 个**相关标杆产品（开源项目 / 论文 / 商业产品）
- 每个标杆产品标注相关性（如"Redis 的元数据缓存机制"、"Linux page cache 的淘汰策略"）
- **选择标准**：
  - 代表性强：业界主流方案，有广泛应用
  - 设计思路清晰：有详细文档或源码可追溯
  - 与本特性相关：解决类似问题或有可借鉴的设计
  - **避免重复**：同类标杆选 1-2 个代表即可（如分布式文件系统选 Ceph 或 Lustre，不全选）

写入临时文件 `research-benchmarks-draft.md`，返回标杆产品数。

**数量控制**：严格控制在 3-5 个，聚焦核心标杆，不追求穷举。

### [3] 标杆方案研究（按标杆组织）

对每个标杆产品，派遣 1 个 researcher agent（并发上限 5），任务：
- **深度调研**：该标杆产品在本特性相关领域的完整设计
- 识别该标杆的核心决策主题（如"缓存淘汰策略"、"并发控制"、"失效传播"）
- **主题数量控制**：每个标杆识别 **2-4 个**核心主题（聚焦关键决策，不追求穷举）
- 每个主题描述**该标杆的方案**（不是列举多个候选方案）：
  - **原理与设计**：该标杆在这个主题下的具体方案（核心机制 + 关键数据结构）
  - **取舍分析**：该方案的 trade-off + 适用场景 + 不适用场景
  - **关键实践**：该标杆的 1-2 个已知坑点或最佳实践
- 标注标杆来源（产品名 / 论文 / 源码位置）
- **保持整体性**：不要割裂单个标杆的设计思路，各决策之间的关联要保留

产出：每个标杆一个 `research-benchmark-<product>-draft.md`

**示例**：
```
标杆：Redis
  主题 1: 缓存淘汰策略
    - Redis 的方案：近似 LRU（采样 5 个 key）
    - 原理与设计：...
    - 取舍分析：命中率损失 < 5%，性能提升 30%
    - 关键实践：采样数可配置（maxmemory-samples）
  
  主题 2: 并发控制
    - Redis 的方案：单线程 + IO 多路复用
    - 原理与设计：...
    - 取舍分析：避免锁竞争，但无法利用多核
    - 关键实践：6.0 后引入 IO 线程
```

**内容精简原则**：
- **聚焦核心**：只写核心机制，不写边缘细节或实现变种
- **量化优先**：trade-off 必须量化（如"性能提升 30%"），避免模糊描述（如"性能更好"）
- **精选坑点**：只写 1-2 个最关键的坑点，不罗列所有可能的问题
- **避免冗余**：不重复已知信息，不解释显而易见的内容

**��出格式约束**：
- 用设计语言描述，不贴源码（最多引用关键函数名或数据结构名）
- 关键数据结构用结构图或字段说明
- 算法用流程描述或决策树

**与产品级 research 的区别**：
- 产品级：**广度优先**，多个标杆产品的系统级架构（子系统分解、数据流、一致性协议）
- 特性级：**深度优先**，单特性的实现级理解（算法、数据结构、性能优化、已知坑点），但用设计语言表达

### [4] 设计空间地图生成

派遣 1 个 researcher agent，任务：

**子任务 1：主题提取与重组**
- 读所有标杆的 draft
- 提取跨标杆的共同主题（如"缓存淘汰"在 Redis / Memcached / Linux 都有）
- **主题数量控制**：识别 **3-8 个**共同主题（聚焦核心决策，不追求穷举）
- 将不同标杆在同一主题下的方案合并到一起
- **方案来源标注**：每个方案标注来自哪个标杆（如"近似 LRU（Redis）"、"严格 LRU（Memcached）"）
- 重组为按主题组织的标杆方案空间

**示例**：
```
主题 1: 缓存淘汰策略
  - 方案 A: 近似 LRU（Redis）
    - 原理与设计：采样 5 个 key，淘汰 idle time 最大的
    - 取舍分析：命中率损失 < 5%，性能提升 30%
  - 方案 B: 严格 LRU（Memcached）
    - 原理与设计：双向链表 + 哈希表，O(1) 访问和淘汰
    - 取舍分析：命中率最优，但锁竞争严重
  - 方案 C: 双 LRU 列表（Linux）
    - 原理与设计：active/inactive 分离，避免扫描污染
    - 取舍分析：适合混合访问模式，内存开销增加
```

**子任务 2：约束收集**
- 用 Grep + 局部 Read 检索产品级文档（按 CLAUDE.md 索引入口）
- 对每个标杆维度，生成"我们在这个维度的情况是什么"问题
- 命中 → 提炼约束（附来源）；未命中 → 标记「未知约束」
- **约束筛选原则**：只保留**直接影响设计决策**的约束
  - ✅ 保留：缓存大小上限 1GB（影响淘汰策略选择）
  - ❌ 删除：日志格式要求（不影响缓存设计）
- **未知约束合并**：相似的未知约束合并为一条
- 产出约束清单（控制在 10-15 条）

**子任务 3：空间过滤**
- 标杆方案 × 自身约束 → 过滤后的可行候选空间
- 标注每个候选方案的可行性（可行 / 需改造 / 不可行）
- 给出改造建议或排除理由（**简洁表达**，说明核心矛盾即可）

产出：research.md 的三个章节
- ## 1. 自身约束清单
- ## 2. 标杆方案空间（按主题重组）
- ## 3. 设计空间地图

**内容精简原则**：
- **约束清单**：每条约束包含"约束内容 + 来源 + 对设计的影响"，不写无关背景
- **标杆方案空间**：复用步骤 [3] 的内容，只做主题重组，不重新展开
- **设计空间地图**：每个方案只写"可行性判断 + 核心理由"，不重复标杆分析

### [5] 评审循环（最多 3 轮）

派遣 1 个 architect-reviewer agent，任务：
- 读 research.md 全文
- 检查：标杆覆盖度、约束追溯、未知约束是否合理标注
- **篇幅合理性检查**：根据特性复杂度判断总行数是否合理
  - 参考标准（非硬性要求）：
    - 简单特性（如单一算法优化）：300-600 行
    - 中等特性（如缓存机制）：500-1000 行
    - 复杂特性（如分布式一致性）：800-1500 行
  - 如果明显过长（如简单特性 > 1000 行）：提示"过于详细，建议精简冗余内容"
  - 如果明显过短（如复杂特性 < 400 行）：提示"可能遗漏关键内容或分析不够深入"
  - 篇幅问题不计入缺陷密度，只作为优化建议
- 产出：问题清单（CRITICAL / HIGH / MEDIUM / LOW）

计算缺陷密度（问题分数之和 / 标杆产品数）：
- 无 CRITICAL + 缺陷密度 ≤ 2.0 → 通过，进入 [6]
- 否则 → 派遣 1 个 researcher agent 修正，回到本步骤重新评审
- 3 轮后仍未通过 → 标注残留问题，进入 [6]

### [6] 落地输出

research.md 写入完成。在终端汇报：
- 文件规模（总行数）
- 标杆产品数
- 主题数（从标杆中提取）
- 约束清单条数（已知 / 未知）
- 设计空间地图候选数
- 置信度（评审通过 / 带债通过）

## 修订模式

反问主人「想修订哪一块」，提供选项：
1. 修订约束清单（重新检索产品级文档）
2. 修订标杆方案空间（补充新标杆或调整维度）
3. 修订设计空间地图（重新过滤）

只跑对应范围的生成阶段，merge 结果回 research.md，不动其他章节。评审循环只检查变更范围。

---

## 补全模式

用结构性元素清单扫描 research.md，识别缺失项：
- 约束清单缺失某类约束（如性能约束）
- 标杆方案空间缺失某维度
- 设计空间地图未覆盖某候选

直接在缺失位置生成补全内容，评审循环。

```
### architect-reviewer agent

```
当前是特性级 research 阶段，评审 research.md。

**被评审对象**：<路径>

- 标杆覆盖度：是否遗漏主流标杆产品或关键设计维度

```
**被评审 template**：openspec/schemas/spec-driven-enhanced/templates/research.md

**评审维度**：
- 标杆覆盖度：每个主题是否有 ≥2 个候选方案
- 约束追溯：每条约束是否有来源（文件路径 + 锚点）
- 未知约束标注：检索失败的约束是否显式标记

```

## 禁忌项

- 禁止在 research.md 中写实现细节（如具体代码、API 签名）
- 禁止在约束清单中编造约束（检索不到时必须标记「未知约束」）
- 禁止标杆方案空间按产品维度并列（如"X 产品 / Y 产品"），必须按取舍维度组织
- 禁止跳过评审循环直接输出

## 与其他 skill 的协作

- **上游**：proposal.md（由 OpenSpec 引擎或主人创建）
- **下游**：spec.md / design.md（由 `devforge-feature-define` / `devforge-feature-design` 读取 research.md）
- **并行**：无（research 是 spec/design 的前置依赖）
