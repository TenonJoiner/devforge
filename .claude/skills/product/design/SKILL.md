# product/design — 架构探索与决策

## 概述

架构探索是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 通过持续多个阶段的思考-研究-对比-完善循环，产出高质量的架构设计。

**核心原则**：
1. **标杆先行**：不研究业界方案，不允许提出自研方案
   > **与 `/df:define` 的关系**：本 skill 的标杆研究聚焦**技术架构视角**（原理层→设计层→优化层、子系统分解、数据流、一致性协议），与 define 的**需求视角**（用户场景、功能边界、验收标准）互补。两者选择的标杆不必一致。
2. **深度对比**：至少 3 个候选方案，每个方案必须有明确的取舍分析
3. **长时间迭代**：架构文档需要 5-20 个阶段才能定稿，禁止一次对话定架构
4. **方案 > 代码**：聚焦设计决策和 rationale，不讨论具体代码实现
5. **模板即法源**：所有正式交付文档的内容结构、必填章节、自检清单以 `.claude/templates/*.md` 为唯一标准

## Agent 调度总览

本 skill 使用三个 agent，详细人设见各自 Agent.md：

| Agent | 调度场景 | Prompt 骨架 |
|-------|---------|------------|
| researcher | 第 1 阶段标杆研究、第 2 阶段维度调研 | "研究 [标杆/维度]，按 ref-architecture.md 模板产出" |
| architect | 第 2 阶段方案发散、第 3 阶段 ADR/总纲/子系统初稿与修正 | "基于 [输入材料] 发散 ≥N 个候选方案，按 [模板] 产出" |
| architect-reviewer | 第 2 阶段出口评审、第 3 阶段独立评审 | "独立评审 [文档]，按 [模板] 提出 ≥N 个质疑点" |

## 复杂度判定

本 skill 各阶段的内容类型与基准复杂度判定如下（详见 CLAUDE.md「复杂度驱动原则」）：

| 阶段 | 内容类型 | 基准复杂度 | 判定依据 |
|------|---------|-----------|---------|
| 第 0 阶段：启动检测 + 前置调研 | 交互式信息收集 | 低 | 引导式问答，确定设计目标和约束 |
| 第 1 阶段：标杆研究 | 调研报告 | 中等 | 技术视角的信息整理与对比分析 |
| 第 2 阶段 Step 1：整体架构发散 | 系统级架构设计 | **极高** | 定义子系统边界、交互契约、全局数据流 |
| 第 2 阶段 Step 3-4：维度调研与发散 | 子系统级设计 | **高** | 单一技术域内多方案对比（如一致性协议选型） |
| 第 3 阶段 Step 1：ADR 收敛 | 架构决策（ADR） | **极高** | 影响多个子系统、一旦确定难以回退 |
| 第 3 阶段 Step 2：系统总纲定稿 | 系统级架构设计 | **极高** | 定义全局边界和数据流 |
| 第 3 阶段 Step 3：子系统架构定稿 | 子系统级设计 | **高** | 单一子系统内多技术域交叉 |
| 第 4 阶段：维护模式 | 单点修正 / 补充 | 低 | 结构固定、内容基于已有信息组合 |

**决策属性上调**（辅助判定）：
- 若涉及「跨子系统边界」或「一致性关键路径」，复杂度在基准上 +1 级
- 若涉及「不可逆决策」（一旦确定难以回退），复杂度在基准上 +1 级
- 若「无现成方案」（需原创设计），复杂度在基准上 +1 级
- 若涉及「安全 / 合规敏感」，复杂度在基准上 +1 级

**复杂度与 agent 配置对照**：

| 阶段 | 复杂度 | 并行产出 agent | 独立评审 agent |
|------|--------|---------------|---------------|
| 第 1 阶段 | 中等 | 每个标杆 1 个 researcher | architect 验收 |
| 第 2 阶段 Step 1 | 极高 | 3 个 architect | ≥2 个 architect-reviewer |
| 第 2 阶段 Step 3-4 | 高 | 每个维度 2 个 researcher + 1 个 architect | 每个维度 2 个 architect-reviewer |
| 第 3 阶段 Step 1 | 极高 | 3 个 architect | 2 个 architect-reviewer |
| 第 3 阶段 Step 2 | 极高 | 3 个 architect | 2 个 architect-reviewer |
| 第 3 阶段 Step 3 | 高 | 2 个 architect | 2 个 architect-reviewer |
| 第 4 阶段 | 低 | 1 个对应角色 agent | 按需 |

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **正式文档写入前，必须确认对应模板已被读取**

## Template-first 内容规范

所有正式交付文档的内容结构、章节深度、自检清单，**不在本 skill 中重复定义**，统一由以下模板规定：

| 文档类型 | 模板路径 | 说明 |
|----------|----------|------|
| 标杆分析 | `.claude/templates/ref-architecture.md` | reference/*.md |
| ADR | `.claude/templates/arch-adr.md` | adr.md |
| 系统架构总纲 | `.claude/templates/arch-system.md` | design.md |
| 子系统架构 | `.claude/templates/arch-subsystem.md` | `<subsystem>/design.md` |

**Skill 保留的流程与深度要求**（不重复模板中的章节清单）：
- 标杆分析：每个技术点必须完成"原理层→设计层→优化层"三层剖析
- 方案发散：至少 3 个候选方案（ADR 维度）、至少 2 个候选+1 个被排除（子系统维度）
- 决策 rationale：必须有量化推导，禁止"矩阵后直接给结论"
- 文档产出：必须先读模板，再按模板结构填充，缺失任何 `mandatory-sections` 视为不达标

## 执行流程

### 第 0 阶段：启动检测 + 前置调研与定位确认（交互式五步引导，不可跳过）

#### Step 1：启动检测（强制）

**目录归属铁律**：`docs/architecture/reference/`（本 skill）与 `docs/requirements/reference/`（`/df:define`）是两个**完全独立**的目录，禁止仅凭文件名判断文件归属。

**检测清单**：
1. 执行 `ls -la docs/architecture/reference/` 和 `ls -la docs/requirements/reference/` **两个命令**
2. 汇报每个文件时必须使用**完整绝对路径**
3. 若两个目录下存在**同名或类似名文件**，必须在汇报中明确标注归属
4. **禁止**：只检测一个目录就声称"标杆研究已完成"；将 `docs/requirements/reference/` 下的文件计入本 skill 的产出

**状态汇报格式**（强制）：
```
=== 状态汇报 ===
architecture/reference/（本 skill 产出）：
  - product-A.md（X KB）
requirements/reference/（/df:define 产出，仅参考）：
  - product-C.md（X KB）
```

#### Step 2：前置调研（交互式五步引导，不可跳过）

**核心原则**：不调研清楚背景和约束就开始标杆研究，等同于盲人摸象。必须与用户通过**多次交互式提问**逐层深入，禁止一次性抛出所有问题。

**准入条件**：无
**产出文件**：`docs/architecture/design.md`（初稿/WIP 状态，按 `arch-system.md` 模板写入「设计目标」和「全局设计约束」章节）

**交互纪律**：
- **一次只问一个问题**，在用户回答后再决定下一个什么
- **追问比提问更重要**——用户第一次回答往往只是表面意图，需要 1-2 次追问才能触及真实约束
- **提供选项但不限制选择**——给出常见模式帮助用户表达，但保留"其他"选项
- **每步确认理解**——用自己的话复述用户意图，确认无误后再推进

**五步引导**：

- **问 1: 用户动机与系统定位**（最关键，2-3 次对话）— 核心问题："这个系统解决的核心问题是什么？如果没有它，用户/系统现在怎么 workaround？" 收敛标准：能用一句话描述"系统在什么场景下，因为什么限制，无法达成什么目标"
- **问 2: 目标用户/Actor**（1-2 次对话）— 核心问题："谁会直接使用这个系统，或者它的输出？" 收敛标准：明确列出所有 Actor 类型，区分人类用户和外部系统
- **问 3: 规模与负载特征**（1-2 次对话）— 核心问题："系统需要处理的规模和负载特征是什么？" 收敛标准：至少明确数据量级、访问模式、流量特征中的 2 个维度
- **问 4: 环境与约束**（1-2 次对话）— 核心问题："系统最终跑在什么环境里？有什么必须满足的关键约束？" 收敛标准：明确部署拓扑和至少 2-3 个关键约束
- **问 5: 明确边界**（1 次对话）— 核心问题："什么是明确不在本系统范围内的？" 收敛标准：明确列出 2-3 个"明确不做"的事项

**汇总确认**：将五步收集到的信息整理为定位摘要，向用户确认后，**读取 `arch-system.md` 模板**，将调研信息写入 `docs/architecture/design.md` 初稿。

**阶段-章节映射表（执行边界，禁止越界填充）**：

| 章节 | 首次填充 | 定稿阶段 | 第 0 阶段 | 第 1 阶段 | 第 2 阶段 | 第 3 阶段 | 说明 |
|------|----------|----------|---------|---------|---------|---------|------|
| 设计目标 | 第 0 阶段 | Step 2 | ✅ | ✅ 修正 | ✅ 修正 | ❌ | 前置调研问答信息 |
| 性能基线 | 第 0 阶段 | Step 2 | ✅ | ❌ | ✅ 补充 | ❌ | 质量属性 |
| 故障假设 | 第 0 阶段 | Step 2 | ✅ | ❌ | ✅ 补充 | ❌ | 环境约束 |
| 安全边界 | 第 0 阶段 | Step 2 | ✅ | ❌ | ✅ 补充 | ❌ | 合规约束 |
| 可运维性约束 | 第 0 阶段 | Step 2 | ✅ | ❌ | ✅ 补充 | ❌ | 团队约束 |
| 子系统划分 | 第 2 阶段 | Step 2 | ❌ | ❌ | ✅ 预演 | ❌ | 需方案发散后才能划分 |
| 数据流总览 | 第 2 阶段 | Step 2 | ❌ | ❌ | ✅ 初稿 | ❌ | 需架构方向确定后 |
| 关键接口索引 | 第 2 阶段 | Step 2 | ❌ | ❌ | ✅ 初稿 | ❌ | 需子系统预演后 |
| 设计原则 | — | Step 2 | ❌ | ❌ | ❌ | ⛔ 禁止 | ADR 收敛后才能提炼 |
| 技术演进路线图 | — | Step 2 | ❌ | ❌ | ❌ | ⛔ 禁止 | 方案收敛后才能规划 |
| 风险与缓解策略 | — | Step 2 | ❌ | ❌ | ❌ | ⛔ 禁止 | 方案收敛后才能识别 |

**阶段越界检查清单（每次写入 design.md 前必须执行）**：
- [ ] 本次填充的章节是否在当前阶段的允许范围内？
- [ ] 禁止填充的章节是否仍保持占位状态？

> **豁免说明**：第 0 阶段为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。以准入检查清单确认信息充分性。

**第 0 阶段准入检查清单**（写入 `design.md` 初稿前必须确认）：
- [ ] 设计目标已收敛到一句话描述
- [ ] 所有 Actor 类型已明确列出
- [ ] 数据量级 / 访问模式 / 流量特征中至少 2 个维度已明确
- [ ] 部署拓扑和至少 2-3 个关键约束已确认
- [ ] 至少 2-3 个"明确不做"的事项已列出
- [ ] 用户已确认定位摘要无误

**第 0 阶段置信度评分**（信息充分性评估）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 系统定位清晰度 | 30% | 能用一句话描述系统场景、限制、目标 |
| Actor 完整性 | 20% | 所有关键 Actor 已识别，人类/系统区分明确 |
| 规模约束明确性 | 25% | 数据量级/访问模式/流量特征至少 2 个维度已量化 |
| 边界清晰度 | 15% | 至少 2-3 个"明确不做"事项已列出 |
| 用户确认度 | 10% | 用户已确认定位摘要无误 |

**进入第 1 阶段门槛：≥ 0.75**（信息收集阶段，门槛略低于正式阶段）
**低于门槛处理**：标记信息缺口清单，继续交互式提问补充，禁止进入第 1 阶段

**自动推进规则**：准入检查清单全部通过 + 置信度 ≥ 0.75 → 自动进入第 1 阶段；否则继续交互式提问

#### 终止条件

- 用户明确表示"只是为了练习架构设计方法"→ 使用默认假设，进入简化流程
- 用户已有详细的需求文档（如 `docs/requirements/*.md` 已存在）→ 直接读取现有文档，只补充缺失信息
- 用户表示"之前已经讨论过"→ 请求用户提供之前的调研结论摘要

### 第 1 阶段：标杆研究（强制，不可跳过）

**准入条件**：第 0 阶段前置调研已完成，标杆列表已与用户确认
**产出文件**：`docs/architecture/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
**模板**：`.claude/templates/ref-architecture.md`

**执行方式**：
1. 主会话读取 `ref-architecture.md` 模板，确认深度要求
2. 并行派遣 researcher，每个研究一个标杆
3. researcher 完成后，**主会话将内容写入** `docs/architecture/reference/<product>.md`
4. **并行启动 2 个 architect-reviewer 执行独立评审**
   - reviewer-A：评审研究深度与视角正确性（原理层→设计层→优化层三层剖析完整性、技术实现细节混入检查）
   - reviewer-B：评审参考来源可靠性与洞察质量（对我方架构的启示、trade-off 分析、可复用性）
   - 依据模板末尾自检清单逐项确认，提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
5. 主会话汇总两个 reviewer 的评审结论
6. 不达标时按修正工作量选择 agent 配置，进入**评审修正循环**

**评审修正循环**：
1. **独立评审**：2 个 architect-reviewer 并行执行，按模板自检清单验收，提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **修正工作量分级**：
   - 少量修正（<5 处）：1 个 researcher agent 补充分析
   - 中量修正（5-15 处）：**2 个 researcher agent 分块并行补充** + 主会话整合
     - researcher-A：补充原理层与协议分析相关内容
     - researcher-B：补充工程实现与部署运维视角相关内容
   - 大量修正（>15 处）：**回退到重新派遣 researcher 执行标杆研究**
3. **修正完成后，重新启动独立评审**（复核）
4. **循环上限**：最多 3 次评审修正循环
5. **降级规则**：3 次循环后仍有未达标项 → 降级置信度并标注"待第 2 阶段补充"，或等待人工决策
6. **循环终止条件**：所有 CRITICAL 问题已修正 + **2 个 architect-reviewer 均给出"通过"或"有条件通过"**

7. **确认所有文件已落盘**，全部验收通过后，进入下一阶段

**出口标准**：
- [ ] 每个标杆完成"原理层→设计层→优化层"三层剖析
- [ ] 每个标杆的调研深度通过 **2 个 architect-reviewer** 按模板自检清单验收
- [ ] 文档中无技术实现细节混入（纯架构技术视角）
- [ ] **2 个 architect-reviewer 均给出"通过"或"有条件通过"**
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] 所有文件已落盘并验证存在（`ls -la` 确认）
- [ ] 文档状态标记为"第 1 阶段完成"
- [ ] **CRITICAL 问题**：标杆数量 < 2、或任一标杆未完成三层剖析、或存在需求视角混入 → 必须修正，阻塞进入第 2 阶段
- [ ] **HIGH 问题**：分析深度不足、关键维度遗漏 → 默认接受修正
- [ ] **MEDIUM/LOW 问题**：记录但不阻塞推进，纳入后续迭代跟踪

**评审通过标准**：
- "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正
- "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
- "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题

**第 1 阶段置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 研究深度 | 25% | 完成原理层→设计层→优化层三层剖析 |
| 视角正确性 | 25% | 无需求视角混入，纯架构技术视角 |
| 覆盖完整性 | 20% | 至少 2-3 个标杆，每个都有适用性评估 |
| 洞察质量 | 20% | 有明确的"对我方架构的启示"和 trade-off 分析 |
| 可复用性 | 10% | 结论能直接指导第 2 阶段方案发散 |

**进入第 2 阶段门槛：≥ 0.80**

**低于门槛处理规则**：
- 置信度 < 0.70：等待人工决策，禁止进入第 2 阶段
- 置信度 0.70-0.80：明确反馈 researcher 补充分析，执行评审修正循环
- 置信度 ≥ 0.80：自动进入第 2 阶段

**自动推进规则**：
- 置信度 ≥ 0.80 且 **2 个 architect-reviewer 均给出"通过"或"有条件通过"** → **确认后推进**（向用户汇报结果摘要，建议进入第 2 阶段）

| 量化推导质量 | 25% | 对比矩阵有量化指标和明确取舍分析，非纯定性描述 |
| 风险识别完整性 | 20% | 每个候选方案有明确风险及缓解策略 |
| 假设验证充分性 | 15% | 关键技术假设已显式列出，有验证路径 |
| 决策可执行性 | 15% | 团队技术栈与能力匹配，落地路径清晰 |

**门槛：≥ 0.85**（ADR 涉及不可逆决策和一致性关键路径，门槛 ≥ 0.85）

**低于门槛处理规则**：
- 置信度 < 0.70：等待人工决策，禁止产出总纲和子系统
- 置信度 0.70-0.80：仅更新决策文档，可产出总纲（标注"有条件"），禁止产出子系统
- 置信度 0.80-0.85：可产出总纲，禁止产出子系统，需在后续迭代中补充验证
- 置信度 > 0.85：可产出总纲 + 对应子系统

**自动推进规则**：
- 循环终止条件达成（所有 CRITICAL 已修正 + 双 reviewer "通过"/"有条件通过"）→ 自动进入 Step 2
- 置信度 < 0.70 或任一 reviewer "不通过" → 等待人工决策
- 3 次评审修正循环后仍未达标 → 降级置信度并标注"待后续补充"，或等待人工决策

#### Step 2: 系统架构总纲定稿（全自动，评审修正循环）

**输入**：ADR 终稿 + `design.md` 初稿（第 0 阶段产出，经第 1-2 阶段迭代）

> **主会话角色铁律**：同 Step 1，禁止直接撰写或评审总纲内容。

**评审修正循环**（复杂度：极高，产出阶段 ≥3 agent，评审阶段 ≥2 agent）：
1. **多 architect 并行发散** — 并行启动 **3 个 architect agent**，各自独立读取 `arch-system.md` 模板 → 从不同架构范式出发发散 ≥2 个总纲结构备选
   - agent-A：数据流与通信范式视角
   - agent-B：状态管理与一致性范式视角
   - agent-C：集成与扩展性范式视角
   - 每个 agent 独立产出，互不参考对方结论
   - 主会话整合多 agent 产出，选择最优结构或融合不同方案的优点
2. **多 architect-reviewer 并行质疑** — 并行派遣 **2 个 architect-reviewer agent**，视角差异化：
   - reviewer-A：评审子系统边界清晰度与接口契约一致性（ADR 约束是否被正确映射到子系统边界）
   - reviewer-B：评审数据流完整性与总纲-ADR 一致性（数据流图中是否存在单点瓶颈、未定义错误传播路径、低置信度 ADR 的处理方式）
   → 各自读取模板 → 至少 5 个质疑点
3. **多 architect 并行修正定稿** — 将质疑点分发给多个 architect agent 并行修正 → 主会话整合 → `design.md` 终稿
   - 修正工作量判断：
     - 少量修正（<5 处）：单 agent 执行
     - 中量修正（5-15 处）：**2 个 architect agent 分块并行 + 主会话整合**
       - architect-A：修正子系统边界、接口契约、ADR 约束映射等技术架构内容
       - architect-B：修正数据流完整性、错误传播路径、低置信度 ADR 处理方式等全局一致性内容
     - 大量修正（>15 处）：**回退到 Step 1 重新发散**

**评审修正循环上限**：最多 3 次

**循环终止条件**：
- ✅ 所有 CRITICAL 问题已修正 + 2 个 architect-reviewer 均给出"通过"或"有条件通过" → Step 2 完成，自动进入 Step 3
- ⚠️ 3 次循环后仍未达标 → 降级置信度并标注"待后续补充"，或等待人工决策
- ❌ 置信度 < 0.85 或任一 reviewer "不通过" → 等待人工决策

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正
- [ ] 2 个 architect-reviewer 均给出"通过"或"有条件通过"
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] 子系统边界清晰度达标（接口契约完整、数据流无断点）
- [ ] ADR 约束已正确映射到子系统边界
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 3 阶段 Step 2 完成（总纲已定稿）"

**第 3 阶段 Step 2 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 子系统边界清晰度 | 25% | 各子系统职责明确，无重叠或遗漏 |
| 接口契约一致性 | 25% | 跨子系统接口定义完整，与 ADR 一致 |
| 数据流完整性 | 20% | 正常路径 + 错误传播路径均已定义 |
| ADR-总纲一致性 | 20% | 所有 ADR 决策正确映射到总纲结构 |
| 可演进性 | 10% | 预留扩展点，关键接口可向后兼容 |

**门槛：≥ 0.85**（系统级架构设计，涉及不可逆决策）

**关键纪律**：`design.md` 未产出前，禁止产出任何 `<subsystem>/design.md`

#### Step 3: 子系统架构定稿（每个子系统独立执行评审修正循环，全自动）

**输入**：ADR 终稿 + 系统架构总纲（`design.md`）

**双重约束**：ADR 技术约束 + 总纲接口契约。若发现冲突，优先暂停子系统设计，修正总纲或 ADR 后再继续。

**重要约束**：
- 禁止主会话一次性批量产出多个子系统文档
- 每个子系统必须独立经过"发散→质疑→修正"评审修正循环
- 只有对应维度 ADR 达到高置信度(>0.80)时，才启动该子系统
- **无依赖关系的子系统，并行启动多个 architect agent 同时设计**

**评审修正循环**（每个 `docs/architecture/<subsystem>/design.md` 独立执行，复杂度：高）：
1. **多 architect 并行发散** — 并行启动 **2 个 architect agent**，各自从不同设计侧重出发，读取 `arch-subsystem.md` 模板 + ADR + 总纲 → 各自产出 ≥2 个候选方案
   - agent-A：侧重核心功能路径与数据流设计
   - agent-B：侧重故障处理、边界条件与运维接口设计
   - 主会话整合两个 agent 的产出，形成统一的子系统初稿
2. **多 architect-reviewer 并行质疑** — 并行派遣 **2 个 architect-reviewer agent**（reviewer-A 评审功能完整性与接口契约，reviewer-B 评审边界条件与可运维性）→ 各自至少 5 个质疑点（模板合规性、边界、推导过程、rationale、一致性、可演进性）
3. **修正定稿** — 根据修正工作量选择 agent 配置：
   - 少量修正（<5 处）：1 个 architect agent 执行修正
   - 中量修正（5-15 处）：**2 个 architect agent 分块并行修正** + 主会话整合
     - architect-A：修正核心功能路径、数据流设计、接口契约定义等正向设计内容
     - architect-B：修正故障处理策略、边界条件覆盖、运维接口设计、与总纲/ADR 一致性等完整性内容
   - 大量修正（>15 处）：**回退到 Step 1 重新执行多 architect 并行发散**
   - 回应所有质疑 → 子系统终稿

**评审修正循环上限**：每个子系统最多 3 次

**循环终止条件**：
- ✅ 所有 CRITICAL 问题已修正 + 2 个 architect-reviewer 均给出"通过"或"有条件通过" → 该子系统定稿
- ⚠️ 3 次循环后仍未达标 → 降级置信度并标注"待后续补充"，或等待人工决策
- ❌ 与总纲/ADR 存在冲突 → 优先暂停该子系统，修正总纲或 ADR 后再继续

**子系统间推进规则**：无依赖关系的子系统并行启动评审修正循环；每个子系统循环终止后汇报结果；全部子系统定稿后进入第 4 阶段

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正
- [ ] 2 个 architect-reviewer 均给出"通过"或"有条件通过"
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] 核心功能路径数据流完整定义
- [ ] 故障处理策略与边界条件覆盖
- [ ] 与总纲/ADR 无冲突
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 3 阶段 Step 3 完成（子系统已定稿）"

**第 3 阶段 Step 3 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 功能路径完整性 | 25% | 核心数据流、状态转换、接口契约完整 |
| 故障处理覆盖度 | 25% | 故障模式、降级策略、恢复流程已定义 |
| 边界条件充分性 | 20% | 输入边界、资源边界、并发边界已覆盖 |
| 与总纲/ADR 一致性 | 20% | 不偏离总纲接口契约和 ADR 技术决策 |
| 可运维性 | 10% | 运维接口、监控点、日志规范已定义 |

**门槛：≥ 0.80**

**执行回溯检查**（每个子系统完成评审修正循环后强制执行）：
- [ ] architect 执行记录（发散 + 修正定稿）
- [ ] architect-reviewer 执行记录（独立质疑）
- [ ] 主会话是否仅搬运 agent 产出，未自行撰写/评审
- 缺失记录 → 强制回退到对应环节

**主会话写入前的强制声明**：
> "执行记录：✅ architect(发散): [子系统名] by [agent-id] ✅ reviewer(质疑): [子系统名] by [agent-id] ✅ architect(修正): [子系统名] by [agent-id] ✅ 写入：<subsystem>/design.md（主会话仅搬运）"

### 第 4 阶段：架构维护与扩展（维护模式）

**核心定位**：前 3 阶段完成后，架构文档已进入"可用状态"。第 4 阶段是**维护模式**——按需调用 `/df:design` 来修正、补充或调整。

**触发原因**：实现验证后发现设计缺陷 / 需求或范围变更 / 补充新标杆研究 / 整合实现中发现的不一致

**下一步判定表**（按优先级顺序检查）：

| 当前状态 | 强制下一步 | 原因 |
|---------|---------|------|
| `adr.md` 不存在 | 第 3 阶段 Step 1：ADR 收敛 | ADR 是总纲和子系统的前置 |
| `design.md` 未定稿 | 第 3 阶段 Step 2：系统总纲定稿 | 总纲是子系统的前置 |
| 子系统文档未完成评审修正循环 | 第 3 阶段 Step 3：补全子系统 | 必须经过完整评审修正循环 |
| 所有文件已定稿 | 等待用户具体指令 | 此时才进入自由指令模式 |

**禁止在第 4 阶段生成多选项列表**。非定稿状态下下一步是唯一的。

**快速通道**（小修不走完整阶段）：文档一致性修正 / 假设验证状态更新 / 单篇标杆补充

> **豁免说明**：快速通道属于维护模式单点修正，适用豁免场景。不走完整出口标准和评审修正循环，但修正后仍需至少 1 个 reviewer 快速复核。

**快速通道质量门禁**：
- 修正完成后，必须派遣 ≥1 个 architect-reviewer agent 快速复核
- 复核重点：修正内容是否与现有文档一致、是否引入新冲突
- reviewer 给出"通过"后方可落盘；发现问题 → 回退到对应完整阶段重新执行

**快速通道出口标准**：
- [ ] 修正范围仅限于声明的修改点（单文件单章节或单篇标杆）
- [ ] 修正后与关联文档无冲突
- [ ] 至少 1 个 architect-reviewer 快速复核通过
- [ ] 文档已写入并验证存在

**快速通道置信度检查**：
- 修正范围是否仅限于声明的修改点？（是/否）
- 修正后是否与关联文档一致？（是/否）
- reviewer 快速复核是否通过？（是/否）
- 全部"是" → 自动落盘；任一"否" → 回退到对应完整阶段

**回退机制**（大变更回到对应子流程）：
- 新增关键技术维度 → 回退到第 2 阶段
- 推翻已有 ADR → 回退到第 3 阶段 Step 1
- 新增子系统 → 回退到第 3 阶段 Step 2-3

## 四级问题分级定义

所有评审环节统一使用四级问题分级：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **CRITICAL** | 阻塞性问题，会导致方向错误 | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响质量 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | 可延期 |

**应用规则**：
- 第 2 阶段 Step 7 出口评审：CRITICAL 问题未修正 → 禁止进入第 3 阶段
- 第 3 阶段评审修正循环质疑回应：接受（修改方案）/ 反驳（说明理由）/ 讨论（标注待澄清，不阻塞）
- 置信度 < 0.70 → 等待人工决策；0.70-0.80 → 确认后推进（标注"有条件"）；> 0.80 → 自动推进

## 红旗信号（自查清单）

以下情况出现时，**立即停止，回到研究阶段**：

- 🚩 标杆研究文档 < 2 个
- 🚩 候选方案只有 1 个
- 🚩 方案描述中代码细节 > 设计 rationale
- 🚩 单次对话内完成"研究→方案→文档"
- 🚩 没有明确的 trade-off 分析
- 🚩 Agent 产出内容未写入项目文件
- 🚩 向用户汇报"本阶段完成"但文件未落盘
- 🚩 主会话一次性批量产出多个子系统架构文档
- 🚩 子系统架构文档未经 architect-reviewer 质疑直接定稿
- 🚩 `design.md` 总纲未产出前就产子系统文档
- 🚩 主会话直接撰写 ADR/总纲/子系统设计文档内容（应由 architect agent 产出）
- 🚩 主会话直接评审文档（应由 architect-reviewer agent 产出）
- 🚩 主会话写入文档前未声明 agent 执行记录
- 🚩 正式文档产出前未读取对应模板
- 🚩 目录归属混淆：将 `docs/requirements/reference/` 下的文件误报为本 skill 产出
- 🚩 生成多选项列表：违反"下一步是唯一的"原则

## 与用户协作模式

**三种协作节奏统一使用**：
1. **自动推进**：阶段内产出→独立评审→评审修正循环自动执行，循环结束后汇报结果
2. **确认后推进**：每阶段/每 Step 完成后汇报，询问"是否进入下一阶段/下一 Step？"
3. **等待人工决策**：置信度 < 0.70、信息缺失或方向选择时停止，等待用户决策

**第 0 阶段前置调研**：交互式提问理解系统背景（唯一全程需要用户参与的环节）
