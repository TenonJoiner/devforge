# product/define — 定义性思考

## 概述

需求定义是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续多个阶段的思考-研究-发散-评审-完善循环，产出高质量的需求规格文档。

> **内容规范强牵引**：需求交付件的所有内容结构、章节要求、自检清单**以 `.claude/templates/*.md` 为唯一准绳**。本 skill 只负责定义**努力程度、写作风格、质量门槛和流程控制**，不再重复模板中已有的内容要求。

**核心原则**：
1. **模板牵引**：所有产出必须严格遵循对应模板，写前先读模板是铁律
2. **标杆先行**：不研究业界同类产品的需求规格，不允许定义自研 Feature
   > **与 `/df:design` 的关系**：本 skill 的标杆研究聚焦**需求视角**（用户场景、功能边界、验收标准、商业模式），与 design 的**技术架构视角**互补。两者研究对象可重叠，但分析维度不同，最终选择的标杆不必一致。
3. **Actor 驱动**：先识别所有交互角色，再围绕角色定义场景
4. **Feature 为主**：正常场景是需求的核心，故障/异常/运维场景从完整性维度补充，不应喧宾夺主
5. **分层产出**：先产出 Actor-Feature 清单并评审定稿，再进入 Scenario 挖掘，避免一次性产出过大文档无法评审
6. **可验收性**：每个 Scenario 可独立验证，验收标准必须量化
7. **长时间迭代**：需求文档需要 3-10 个阶段才能定稿，禁止一次对话定需求

**协作 Agent**：`product-manager`、`researcher`、`pm-reviewer`、`architect-reviewer`、`architect`（详见对应 Agent.md）

## 复杂度判定

本 skill 各阶段的内容类型与基准复杂度判定如下（详见 CLAUDE.md「复杂度驱动原则」）：

| 阶段 | 内容类型 | 基准复杂度 | 判定依据 |
|------|---------|-----------|---------|
| 第 0 阶段：产品定位确认 | 交互式信息收集 | 低 | 引导式问答，无深度分析产出 |
| 第 1 阶段：标杆研究 | 调研报告 | **高** | 多视角深度分析（功能特性 vs 商业模式），产出直接指导 Feature 设计 |
| 第 2 阶段：Actor-Feature 识别 | 需求核心 Feature / Actor 定义 | **极高** | 定义产品边界、Actor 权责、验收标准 |
| 第 3 阶段：Scenario 挖掘 | Scenario / 用例设计 | **高**（默认）/ 中等（降级） | 基于已确定 Feature 的展开；分布式存储 domain 默认涉及跨子系统边界 + 性能/一致性关键路径 |
| 第 4 阶段：维护模式 | 单点修正 / 补充 | 低 | 结构固定、内容基于已有信息组合 |

**决策属性上调**（辅助判定）：
- 第 1 阶段若发现「无现成方案」（需原创设计），复杂度上调一级
- 第 3 阶段基准复杂度「中等」，但 teamskills 为分布式存储系统，domain 默认涉及「跨子系统边界」+「性能/一致性关键路径」，复杂度上调至「高」，默认配置 2 个 agent
- 仅当 domain 为极简单场景（纯 UI 展示、单 Actor 无交互）时才允许降级为「中等」，需明确说明降级理由

**复杂度与 agent 配置对照**：

| 阶段 | 复杂度 | 并行产出 agent | 独立评审 agent |
|------|--------|---------------|---------------|
| 第 1 阶段 | 高 | 每个标杆 2 个 researcher | product-manager + pm-reviewer 独立评审 |
| 第 2 阶段 | 极高 | ≥3 个 product-manager | ≥2 个 reviewer |
| 第 3 阶段 | 高 | 每个 domain 2 个 product-manager | ≥2 个 reviewer |
| 第 3 阶段（降级） | 中等 | 每个 domain 1 个 product-manager | 1-2 个 reviewer |
| 第 4 阶段 | 低 | 1 个对应角色 agent | 按需 |

**与特性级的衔接**：
- `/df:plan` 产出的 `docs/iteration-plan.md` 中的 proposal 清单，是 `/opsx:new` 的输入
- `/opsx:new` 时拷贝对应 proposal 作为变更单元
- 写 spec 时参考 `docs/requirements/<feature-domain>.md` 中的具体需求定义

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

原因：需求定义是长期迭代过程，跨越多次对话。如果产出仅存在于对话中，下次对话将丢失所有中间成果，导致重复劳动或方向偏移。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **写入后必须验证文件存在**（`ls` 确认文件已落盘）

## 主会话职责边界

主会话在多 agent 协作中只承担以下职责，**超出即越界**：

1. **调度**：决定哪个阶段派哪些 agent、什么时机并行
2. **搬运**：将 agent 产出写入文件（仅搬运，不编辑内容）
3. **整合**：当多个 agent 产出需要合并时，选择/拼接/去重（不改写核心结论）
4. **门禁**：检查出口标准、置信度阈值、agent 执行记录完整性

**主会话禁止行为**：
- 直接产出需要深度分析的文档内容
- 对 agent 产出进行"润色"后写入（只能原样搬运或明确标注修改）
- 在没有 agent 执行记录的情况下声称某阶段已完成
- 用一个 agent "内部发散多个方案"替代多个 agent 独立产出

## 启动检测（以文件系统为真实来源）

**核心原则**：状态汇报必须以 `ls`/`wc`/`Read` 扫描文件系统的**实时结果**为准，禁止依赖任何独立维护的状态副本。

```bash
ls docs/requirements/ 2>/dev/null || echo "无现有需求文档"
```

**检测清单**：
1. 执行 `ls -la docs/requirements/`、`ls -la docs/requirements/reference/` 和 `ls -la docs/architecture/reference/` **三个命令**
2. 对声称"已完成"的文档，必须用 `ls` 验证文件存在、用 `wc -l` 验证非空
3. 汇报每个文件时必须使用**完整绝对路径**
4. 若两个目录下存在**同名或类似名文件**，必须在汇报中明确标注归属
5. **禁止行为**：
   - 只检测一个目录就声称"标杆研究已完成"
   - 将 `docs/architecture/reference/` 下的文件计入本 skill 的产出

**状态汇报格式**（强制实时生成，禁止读取缓存）：
```
=== 状态汇报 ===
文档检测结果（实时）：
- product-spec.md: [存在/不存在], X 行
- reference/product-*.md: N 份
- <feature-domain>.md: N 份
当前可判断阶段：[第 X 阶段]
当前置信度：[基于已产出文档质量判断]
```

**如果已有文档**：
> 检测到现有需求文档：
> 1. **新增 Feature** —— 扩展现有文档
> 2. **完善现有定义** —— 迭代更新特定章节
> 3. **重构结构** —— 调整组织方式

## 执行流程

### 第 0 阶段：产品定位确认（强制）

**准入条件**：无
**产出文件**：`docs/requirements/product-spec.md`（初始定位草稿，后续阶段持续丰富）

在启动标杆研究之前，必须先完成轻量级产品定位确认。如果产品边界不清，标杆研究可能选错对象，导致后续全部返工。

**核心原则**：
- 通过交互式提问逐层深入理解产品定位，禁止一次性抛出所有问题
- 追问策略和交互技巧详见 `product-manager.md`

**执行方式（交互式引导，共 7 个 Step）**：

#### Step 1: 破冰（1 分钟）
简要说明前置定位的目的和流程（7 个 Step，每个 Step 1-2 次对话），让用户建立预期。

#### Step 2: 用户动机（最关键，2-3 次对话）
**核心问题**："这个产品/子系统，用户最痛的点是什么？如果没有它，用户现在怎么 workaround？"
**收敛标准**：能用一句话描述"用户在什么场景下，因为什么限制，无法达成什么目标"

#### Step 3: 目标用户/Actor（1-2 次对话）
**核心问题**："谁会直接使用这个系统，或者它的输出？"
**收敛标准**：明确列出所有 Actor 类型，区分人类用户和外部系统

#### Step 4: 部署环境（1 次对话）
**核心问题**："系统最终跑在什么环境里？"
**收敛标准**：明确部署拓扑和规模量级

#### Step 5: 关键约束（1-2 次对话）
**核心问题**："有什么条件是必须满足的，否则这个产品就算失败？"
**收敛标准**：至少明确 2-3 个关键约束，能用具体数字或明确等级描述

#### Step 6: 明确边界（1 次对话）
**核心问题**："什么是明确不在本产品范围内的？"
**收敛标准**：明确列出 2-3 个"明确不做"的事项

#### Step 7: 汇总确认
将上述 Step 收集到的信息整理为定位摘要，向用户确认：

```
=== 产品定位摘要 ===
核心痛点：...
目标用户：...
部署环境：...
关键约束：...
明确边界：...

确认无误？（是/需要调整 X）
```

确认后：**读取 `.claude/templates/req-product-spec.md` 模板**，将定位结论整理为 `product-spec.md` 初始草稿并落盘。

> **product-spec.md 的定位**：从第 0 阶段开始，它就是**持续迭代的活文档**。第 1 阶段标杆研究可能反过来修正产品定位，此时需**回溯修正** product-spec.md 中的定位章节。

> **豁免说明**：第 0 阶段为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。以准入检查清单确认信息充分性。

**第 0 阶段准入检查清单**（写入 `product-spec.md` 初稿前必须确认）：
- [ ] 核心痛点已收敛到一句话描述
- [ ] 所有 Actor 类型已明确列出（区分人类用户和外部系统）
- [ ] 部署拓扑和规模量级已确认
- [ ] 至少 2-3 个关键约束已明确（能用数字或等级描述）
- [ ] 至少 2-3 个"明确不做"的事项已列出
- [ ] 用户已确认定位摘要无误

**第 0 阶段置信度评分**（信息充分性评估）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 痛点收敛度 | 40% | 能用一句话描述核心痛点 |
| Actor 完整性 | 30% | 所有关键 Actor 已识别，无遗漏 |
| 约束明确度 | 30% | 至少 2-3 个可量化的关键约束 |

**进入第 1 阶段门槛：≥ 0.75**（低复杂度门槛可适当降低）
**低于门槛处理**：标记信息缺口清单，继续交互式提问补充，禁止进入第 1 阶段

**自动推进规则**：准入检查清单全部通过 + 置信度 ≥ 0.75 → 自动进入第 1 阶段；否则继续交互式提问

### 第 1 阶段：标杆研究（强制，不可跳过）

**准入条件**：第 0 阶段产品定位确认已完成，且 `product-spec.md` 初始草稿已落盘
**产出文件**：`docs/requirements/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
**模板**：`.claude/templates/ref-requirements.md`

**执行方式**：
1. Master 识别候选标杆列表，向用户确认标杆选择
2. **每个标杆并行启动 2 个 researcher agent**（需求模式），各自从不同需求视角独立研究同一标杆
   - agent-A 侧重：功能特性组织方式、Scenario 策略、用户旅程分析
   - agent-B 侧重：商业模式、定价策略、用户体验设计与量化边界
   - 产出目录：`docs/requirements/reference/`
   - 分析视角：**需求/用户/产品视角**，禁止技术分析
   - 模板约束：使用 `.claude/templates/ref-requirements.md`
   - **禁止**两个 researcher 使用同一 prompt 模板、同一分析框架
3. researcher 完成后，**主会话将内容写入文件**
4. **并行启动 product-manager 和 pm-reviewer 执行独立评审**
   - `product-manager`：验收研究深度，检查是否覆盖 Feature 组织方式、Scenario 策略、量化边界（依据模板自检清单）
   - `pm-reviewer`：评审研究视角完整性，检查是否遗漏关键需求维度、标杆选择是否有偏见、洞察质量是否达标
5. **主会话验收**：检查文档中是否出现技术实现细节，如有则**强制打回重写**
6. 不达标时反馈 researcher 补充分析，**评审修正循环**直到通过

**出口标准**：
- [ ] 每个标杆完成"功能特性组织方式、Scenario 策略、量化边界"的分析
- [ ] 无技术实现细节混入（纯需求/产品视角）
- [ ] product-manager 和 pm-reviewer 均给出"通过"或"有条件通过"
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] 所有标杆文件已落盘并验证存在
- [ ] 文档状态标记为"第 1 阶段完成"
- [ ] **CRITICAL 问题**：标杆数量 < 2、或任一标杆未完成三维度分析、或存在技术实现细节混入 → 必须修正，阻塞进入第 2 阶段
- [ ] **HIGH 问题**：分析深度不足、关键维度遗漏 → 默认接受修正
- [ ] **MEDIUM/LOW 问题**：记录但不阻塞推进，纳入后续迭代跟踪

**评审修正循环**：
1. 主会话汇总 product-manager 和 pm-reviewer 的评审结论
2. 修正阶段 agent 配置（根据修正工作量选择）：
   - 少量修正（<5 处）：1 个 researcher 补充分析
   - 中量修正（5-15 处）：**2 个 researcher 分块并行补充** + 主会话整合
     - researcher-A：补充功能特性组织方式、Scenario 策略、用户旅程分析（与初始 agent-A 视角一致）
     - researcher-B：补充商业模式、定价策略、用户体验设计与量化边界分析（与初始 agent-B 视角一致）
   - 大量修正（>15 处）：**回退到重新选择标杆或重新执行多 researcher 并行研究**
3. **修正完成后，重新启动独立评审**（复核）
4. **评审修正循环上限**：最多 3 次；3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或等待人工决策
5. HIGH/MEDIUM/LOW 级别问题记录但不阻塞

**第 1 阶段置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 研究深度 | 25% | 有 Feature 组织方式、Scenario 策略、量化边界的分析 |
| 视角正确性 | 25% | 无技术实现细节，纯需求视角 |
| 覆盖完整性 | 20% | 至少 2-3 个标杆，每个都有适用性评估 |
| 洞察质量 | 20% | 有明确的"对我方需求的启示" |
| 可复用性 | 10% | 结论能直接指导第 2 阶段 Actor-Feature 设计 |

**进入第 2 阶段门槛：≥ 0.80**

**低于门槛处理规则**：
- 置信度 0.75-0.79 → 自动再修正一次（执行评审修正循环）
- 置信度 < 0.75 → 等待人工决策，禁止进入第 2 阶段

**自动推进规则**：
- 置信度 ≥ 0.80 且双评审均"通过"或"有条件通过" → **确认后推进**（向用户汇报结果摘要，建议进入第 2 阶段）
- 置信度 < 0.80 或存在未修正 CRITICAL 问题 → 等待人工决策或执行评审修正循环

### 第 2 阶段：Actor-Feature 识别与定稿（强制）

**准入条件**：第 1 阶段标杆研究完成并通过 product-manager 验收（≥ 0.80）

#### Step 1：Actor 识别

**产出文件**：`docs/requirements/product-spec.md#actors`

1. **product-manager** 基于产品定位和标杆研究洞察，识别所有 Actor
2. **并行启动 researcher** 补充竞品 Actor 映射表
3. 每个 Actor 明确：类型、职责、交互方式、关注点
4. **写入文件**

**Step 1 写入前检查清单**：
- [ ] Actor 列表非空，至少识别出 1 个核心 Actor
- [ ] 每个 Actor 已明确：类型、职责、交互方式、关注点
- [ ] 文档已按模板结构写入 `product-spec.md#actors`
- [ ] 文件已验证存在（`ls`/`wc` 确认）

#### Plan 阶段（必须，第 2 阶段开始前执行）

**准入条件**：Actor 识别已完成，`product-spec.md#actors` 已落盘

第 2 阶段为**极高复杂度**（核心 Feature / Actor 定义），必须先解构再发散。

**派遣 1 个 architect agent 执行 Plan 阶段的任务解构**（主会话只做调度，不产出分析结论）：

1. **确认切分维度**：按 Actor 分组（每个 agent 负责一组 Actor 的 Feature 推导）
2. **分配视角约束**（每个 agent 独立，互不参考）：
   - agent-A：核心业务流程场景与主线功能（正常路径视角）
   - agent-B：边缘场景、异常流程与运维场景（异常路径视角）
   - agent-C：跨 Actor 交互场景、非功能性需求（性能、安全、可观测性）视角
3. **列出关键假设**（需在产出中显式质疑或验证）：
   - 假设：Step 1 识别的 Actor 列表完整，无遗漏隐性 Actor
   - 假设：标杆研究的洞察可直接映射到本产品的 Feature 设计
   - 假设：Feature 间依赖关系可在整合阶段梳理清楚
4. **识别评审视角缺口**：
   - 产出 agent 已覆盖：正常路径（agent-A）、异常路径（agent-B）、跨域非功能（agent-C）
   - 产出 agent 未覆盖：产品完整性视角（Feature 与产品目标的对齐度）、架构可行性视角（Feature 实现的技术风险）
   - 评审 agent 针对性配置：pm-reviewer 聚焦产品完整性 + Actor 遗漏；architect-reviewer 聚焦架构可行性 + 异常场景边界条件深度

#### Step 2：Feature 识别（多 agent 并行）

**产出文件**：`docs/requirements/product-spec.md`（草稿状态）

1. **并行启动 3 个 product-manager agent**，各自从不同用户场景角度围绕 Actor 核心目标独立推导 Feature 列表
   - 禁止用一个 agent "内部发散多个方案"替代多个 agent 独立产出
   - **agent-A：核心业务流程视角**（正常路径、主线功能、核心用户旅程）
     - 范围：仅限正常业务流转，不涉及故障/异常/运维
     - 禁止：不分析故障恢复、不分析性能指标、不分析跨 Actor 契约细节
   - **agent-B：故障与运维视角**（异常路径、故障模式、运维场景）
     - 范围：仅限非正常运行状态（故障、降级、维护、扩容）
     - 禁止：不推导正常路径功能、不分析性能基准、不分析安全合规
   - **agent-C：系统属性视角**（跨 Actor 交互契约、非功能性需求、安全/合规/可观测性）
     - 范围：聚焦系统级属性（接口契约、性能边界、安全约束、可观测性要求）
     - 禁止：不推导具体业务功能、不分析单一 Actor 内部流程
   - **互斥原则**：三个 agent 的产出范围在 Plan 阶段由主会话明确切分，确保无重叠
   - 每个 agent 独立产出，互不参考对方结论
2. 主会话整合三个 agent 的 Feature 列表，去重并客观记录冲突（仅记录差异，不做价值判断或合并决策，冲突解决交由评审修正循环处理）
3. 每个 Feature 记录：一句话价值摘要、涉及 Actor、归属特性域、优先级
4. 产出 Feature 间依赖关系矩阵
5. **并行启动 architect** 评估各 Feature 实现复杂度
6. Feature 价值论证必须是推导式，禁止"矩阵后直接给结论"
7. 多特性域时按域并行启动多个 product-manager agent（每域仍遵循极高复杂度配置：≥3 个 agent）
8. **读取 `.claude/templates/req-product-spec.md` 模板**，写入 `product-spec.md`

**Step 2 写入前检查清单**：
- [ ] Feature 列表非空，覆盖所有已识别 Actor 的核心目标
- [ ] 每个 Feature 已记录：一句话价值摘要、涉及 Actor、归属特性域、优先级
- [ ] Feature 间依赖关系矩阵已产出
- [ ] 价值论证为推导式，非直接给结论
- [ ] 文档已按模板结构写入 `product-spec.md`
- [ ] 文件已验证存在（`ls`/`wc` 确认）

#### Step 3：独立评审（评审修正循环起点）

**准入条件**：`product-spec.md` 草稿已写入文件

1. **并行启动 `pm-reviewer` 和 `architect-reviewer`** 执行**独立评审**
   - `pm-reviewer` 侧重：Feature 覆盖完整性、产品目标对齐、验收标准清晰度、Actor 权责边界
   - `architect-reviewer` 侧重：架构可行性、Feature 间依赖合理性、异常/运维场景覆盖度、边界条件完整性、可演进性
2. 质疑点数量要求：按特性域复杂度（简单≥3、中等≥5、复杂≥7）

**Step 3 出口标准**：
- [ ] pm-reviewer 和 architect-reviewer 均完成独立评审
- [ ] 质疑点数量达标（简单≥3、中等≥5、复杂≥7）
- [ ] 所有质疑点已按 CRITICAL/HIGH/MEDIUM/LOW 分级标注
- [ ] 评审意见已写入文件或评审记录章节

**评审通过标准**：
- "通过"：无 CRITICAL 问题，HIGH ≤ 2 个
- "有条件通过"：存在 CRITICAL 问题但已明确修正方案，或 HIGH > 2 但无 CRITICAL
- "不通过"：存在未明确修正方案的 CRITICAL 问题

**自动推进**：双评审均"通过"或"有条件通过" → 自动进入 Step 4；任一评审"不通过" → 等待人工决策或**进入评审修正循环**

> **评审修正循环规则**：详见 Step 4。循环上限：最多 3 次。

#### Step 4：评审修正定稿

评审完成后**不等待用户确认**，由 product-manager 自动决策并执行修正。

**修正阶段 agent 配置**（根据修正工作量选择）：
- 少量修正（＜5 处，单点调整）：1 个 product-manager agent 执行修正
- 中量修正（5-15 处，涉及多章节或多维度）：**2 个 product-manager agent 分块并行修正** + 主会话整合
  - product-manager-A：修正核心业务流程视角内容（正常路径 Feature 价值论证、Actor 权责边界、主线功能依赖关系）
  - product-manager-B：修正异常/运维/系统属性视角内容（故障场景覆盖、非功能需求边界、跨 Actor 契约、验收标准可验证性）
- 大量修正（＞15 处，接近重写）：**回退到 Step 2 重新执行多 agent 并行发散**

**修正后复核**：修正完成后，**重新启动独立评审**（pm-reviewer + architect-reviewer 复核），确认问题已修正且无引入新问题。

**评审修正循环上限**：最多 3 次。3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或等待人工决策。

**循环终止条件**：所有 CRITICAL 问题已修正 + 所有 HIGH 问题已修正或有明确处理方案 + pm-reviewer 和 architect-reviewer 均给出"通过"或"有条件通过（CRITICAL 已修正）" + 置信度 ≥ 0.85。

**自动决策规则**：CRITICAL 必须修正 → HIGH 默认修正 → MEDIUM 评估后决定 → LOW 可延期

**自动推进**：≥ 0.85 自动进入第 3 阶段 | 0.75-0.84 自动再修正一次 | < 0.75 **等待人工决策**

**第 2 阶段置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| Actor 完整性 | 25% | 无遗漏隐性 Actor |
| Feature 价值论证 | 25% | 每个 Feature 有"做什么+为什么" |
| 依赖关系清晰度 | 20% | Feature 间依赖矩阵完整 |
| 双评审质量 | 20% | 质疑都有明确回应 |
| 与标杆对齐度 | 10% | Feature 设计体现标杆研究洞察 |

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正
- [ ] pm-reviewer 和 architect-reviewer 均给出"通过"或"有条件通过（CRITICAL 已修正）"
- [ ] 各特性域置信度 ≥ 0.85
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 2 阶段完成（Actor-Feature 已定稿）"

### 第 3 阶段：Scenario 挖掘与特性域定稿（强制）

**准入条件**：第 2 阶段 Actor-Feature 已定稿（≥ 0.85）

#### Plan 阶段（建议，第 3 阶段开始前执行）

第 3 阶段为**高复杂度**（Scenario / 用例设计，默认涉及跨子系统边界 + 性能/一致性关键路径），建议先解构再发散。

**主会话基于上下文自评估，或派遣 1 个 architect agent 快速确认**：
1. **确认切分维度**：按特性域拆分（每个 agent 负责一个 domain 的 Scenario 挖掘）
2. **分配视角约束**（每个 domain 内 2 个 agent 独立，互不参考）：
   - agent-A：正常路径 + 核心业务流程 Scenario 挖掘
   - agent-B：故障模式 + 异常路径 + 运维场景 + 跨域集成 Scenario 挖掘
3. **列出关键假设**（需在产出中显式质疑或验证）：
   - 假设：Actor-Feature 终稿已覆盖所有核心场景（待验证：是否存在未覆盖的隐性场景）
   - 假设：各特性域间 Scenario 无重叠（待验证：跨域场景是否被重复定义或遗漏）
4. **识别评审视角缺口**：
   - 产出 agent 已覆盖：正常路径（agent-A）、异常路径（agent-B）
   - 产出 agent 未覆盖：产品目标对齐视角（Scenario 与 product-spec 的一致性）、架构可行性视角（Scenario 实现的技术风险）
   - 评审 agent 针对性配置：pm-reviewer 聚焦 Scenario 覆盖完整性 + 验收标准可验证性；architect-reviewer 聚焦异常路径覆盖度 + 边界条件完整性

#### Step 1: Scenario 挖掘与特性域文档编写（分域并行）

**产出文件**：各 `docs/requirements/<feature-domain>.md` 初稿

1. 主会话读取 `product-spec.md`（Actor-Feature 终稿）作为统一输入源
2. 按 domain 拆分，无依赖的 domain 并行处理
3. **每个 domain 的 agent 配置**（Scenario 挖掘 = **高复杂度**，默认按分析类型切分，配置 2 个 agent）：
   - **agent-A**：负责正常路径 + 核心业务流程的 Scenario 挖掘
   - **agent-B**：负责故障模式 + 异常路径 + 运维场景 + 跨域集成的 Scenario 挖掘
   - 主会话整合 agent 产出
4. **并行启动多个 `product-manager` agent**，每个负责一个 domain（或一个 domain 内的一个视角）

> **降级条件**：仅当 domain 为极简单场景（如纯 UI 展示、单 Actor 无交互）时，允许降级为 1 个 agent，需在 Plan 阶段明确说明降级理由。
5. 每个 agent 基于定稿的 Actor-Feature 挖掘 Scenario
6. **所有内容结构严格遵循 `.claude/templates/req-feature.md` 模板**
7. 量化非功能需求
8. 所有 domain 初稿完成后，统一进入 Step 2

**Step 1 写入前检查清单**（每个 domain 文档）：
- [ ] 文档已按 `req-feature.md` 模板结构组织（Feature → Scenario → 验收标准）
- [ ] 每个 Scenario 有明确的触发条件、执行步骤、预期结果
- [ ] 每个 Scenario 的验收标准可量化、可独立验证
- [ ] 非功能需求有量化指标（具体数字 + 验收方法）
- [ ] 文档已写入 `docs/requirements/<feature-domain>.md`
- [ ] 文件已验证存在（`ls`/`wc` 确认）

#### Step 2: 独立评审（评审修正循环起点）

**准入条件**：特性域文档初稿已写入文件

1. **并行启动 `pm-reviewer` 和 `architect-reviewer`** 执行**独立评审**
   - `pm-reviewer` 侧重：Scenario 覆盖完整性、验收标准可验证性、产品目标对齐
   - `architect-reviewer` 侧重：异常路径覆盖度、边界条件完整性、非功能需求量化、跨域一致性
2. 质疑点数量要求与第 2 阶段 Step 3 相同

**Step 2 出口标准**：
- [ ] pm-reviewer 和 architect-reviewer 均完成独立评审
- [ ] 质疑点数量达标（简单≥3、中等≥5、复杂≥7）
- [ ] 所有质疑点已按 CRITICAL/HIGH/MEDIUM/LOW 分级标注
- [ ] 评审意见已写入评审记录章节

**评审通过标准**：
- "通过"：无 CRITICAL 问题，HIGH ≤ 2 个
- "有条件通过"：存在 CRITICAL 问题但已明确修正方案，或 HIGH > 2 但无 CRITICAL
- "不通过"：存在未明确修正方案的 CRITICAL 问题

**自动推进**：双评审均"通过"或"有条件通过" → 自动进入 Step 3；任一评审"不通过" → 等待人工决策或直接进入修正

> **评审修正循环规则**：详见 Step 3（与第 2 阶段 Step 4 一致的修正阶段 agent 配置规则）。循环上限：最多 3 次。

#### Step 3: 评审修正定稿

按 domain 分别评分。修正阶段 agent 配置规则与第 2 阶段 Step 4 一致。

**同角色多实例区分（第 3 阶段中量修正时）**：
- product-manager-A：修正正常路径与核心业务流程 Scenario（验收标准可验证性、产品目标对齐、用户旅程完整性）
- product-manager-B：修正故障/异常/运维场景 Scenario（边界条件覆盖、非功能需求量化、跨域一致性）

**修正后复核**：修正完成后，**重新启动独立评审**（pm-reviewer + architect-reviewer 复核），确认问题已修正且无引入新问题。

**评审修正循环上限**：最多 3 次。3 次后仍有未达标项 → 降级置信度并标注"待后续补充"，或等待人工决策。

**循环终止条件**：所有 CRITICAL 问题已修正 + 所有 HIGH 问题已修正或有明确处理方案 + pm-reviewer 和 architect-reviewer 均给出"通过"或"有条件通过（CRITICAL 已修正）" + 各特性域置信度 ≥ 0.85。

**自动推进规则**：
- 各特性域置信度 ≥ 0.85 且双评审均"通过"或"有条件通过" → 自动进入第 4 阶段（维护模式）
- 置信度 0.75-0.84 → 自动再修正一次
- 置信度 < 0.75 → 等待人工决策

**第 3 阶段置信度评分**（按 domain）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 场景覆盖度 | 25% | 正常路径充分 + 至少 4 种故障模式 |
| 验收标准可验证性 | 25% | 每个 Scenario 有可量化验收标准 |
| 非功能需求量化 | 20% | 有具体数字和验收方法 |
| 双评审质量 | 20% | 所有质疑点被回应 |
| 命名空间纪律 | 10% | 无子系统内部模块名，纯用户视角 |

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正
- [ ] pm-reviewer 和 architect-reviewer 均给出"通过"或"有条件通过"
- [ ] 各特性域置信度 ≥ 0.85
- [ ] 验收标准可量化、可独立验证
- [ ] 非功能需求有量化指标
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 3 阶段完成（Scenario 已定稿）"

### 第 4 阶段：需求维护与扩展（维护模式）

**核心定位**：前 3 阶段完成后，需求文档已进入"可用状态"。第 4 阶段是**维护模式**——按需调用 `/df:define` 来修正、补充或调整需求定义。

**快速通道**（小修不走完整阶段）：

> **豁免说明**：快速通道属于维护模式单点修正，适用豁免场景。不走完整出口标准和评审修正循环，但修正后仍需至少 1 个 reviewer 快速复核。

流程：
- 文档一致性修正：直接读取 → 定位差异 → 确认 → 写入 → **1 个 pm-reviewer 快速复核**（1 轮）
- 单 Feature 补充：直接补充 Scenario → **1 个 pm-reviewer 快速复核**（1 轮）
- 单篇标杆补充：researcher 产出 → product-manager 验收 → 落盘 → **1 个 pm-reviewer 快速复核**（1 轮）

**快速通道出口标准**：
- [ ] 修正范围仅限于声明的修改点（单文件单章节或单篇标杆）
- [ ] 修正后与关联文档无冲突
- [ ] 至少 1 个 pm-reviewer 快速复核通过
- [ ] 文档已写入并验证存在

**快速通道置信度检查**：
- 修正范围是否仅限于声明的修改点？（是/否）
- 修正后是否与关联文档一致？（是/否）
- reviewer 快速复核是否通过？（是/否）
- 全部"是" → 自动落盘；任一"否" → 回退到对应完整阶段

**回退机制**（大变更回退到对应子流程重新执行）：
- 新增 Feature → 回退到第 2 阶段 Step 2 → Step 3 → Step 4 → 第 3 阶段
- 推翻已有 Scenario → 回退到第 3 阶段 Step 1 起
- 新增特性域 → 回退到第 3 阶段 Step 1 起

## 常见借口与强制回应

| 用户说 | 必须回应 |
|--------|----------|
| "先快速出一版需求" | 拒绝：快速产出的需求后续返工成本更高 |
| "这个需求 obvious" | 质疑：obvious 的需求往往有最多隐藏假设 |
| "就按你说的做" | 警告：当前置信度 X%，建议完成 Y 调研后再定 |

## 与用户协作模式

**三种协作节奏统一使用**：
- **自动推进**：阶段内产出 → 独立评审 → 评审修正循环自动执行，循环结束后汇报结果
- **确认后推进**：阶段/Step 完成后汇报，等待用户确认再进入下一阶段/Step（第 0 阶段产品定位确认全程需要用户参与）
- **等待人工决策**：信息缺失、方向选择、置信度 < 0.75 时停止，等待用户决策

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要

**三种必须用户介入的情况**：信息缺失 | 方向选择 | 置信度 < 0.75（**等待人工决策**）
