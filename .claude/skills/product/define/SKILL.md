# product/define — 定义性思考

## 概述

需求定义是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续数轮甚至数十轮的思考-研究-发散-评审-完善循环，产出高质量的需求规格文档。

> **内容规范强牵引**：需求交付件的所有内容结构、章节要求、自检清单**以 `.claude/templates/*.md` 为唯一准绳**。本 skill 和 agent 文件只负责定义**努力程度、写作风格、质量门槛和流程控制**，不再重复模板中已有的内容要求。

**核心原则**：
1. **模板牵引**：所有产出必须严格遵循对应模板，写前先读模板是铁律
2. **标杆先行**：不研究业界同类产品的需求规格，不允许定义自研 Feature
3. **Actor 驱动**：先识别所有交互角色，再围绕角色定义场景
4. **Feature 为主**：正常场景是需求的核心，故障/异常/运维场景从完整性维度补充，不应喧宾夺主
5. **分层产出**：先产出 Actor-Feature 清单并评审定稿，再进入 Scenario 挖掘，避免一次性产出过大文档无法评审
6. **可验收性**：每个 Scenario 可独立验证，验收标准必须量化
7. **长时间迭代**：需求文档需要 3-10 轮迭代才能定稿，禁止一次对话定需求
8. **多 agent 协作**：`product-manager` 主导需求定义，`researcher` 负责标杆研究，`pm-reviewer` 负责产品视角独立质疑，`architect-reviewer` 负责技术视角独立质疑，`architect` 按需评估技术可行性

**与特性级的衔接**：
- `/ky:plan` 产出的 `docs/iteration-plan.md` 中的 proposal 清单，是 `/opsx:new` 的输入
- `/opsx:new` 时拷贝对应 proposal 作为变更单元
- 写 spec 时参考 `docs/requirements/<feature-domain>.md` 中的具体需求定义

### Agent 职责边界

| Agent | 职责 | 关键约束 |
|-------|------|----------|
| **researcher** | 标杆研究（客观）<br>• 深入研究竞品/标杆产品的**需求规格、用户场景、功能边界、验收标准**<br>• 分析 Feature 组织方式和场景覆盖策略<br>• 产出 `docs/requirements/reference/*.md` | **只做客观分析，不做主观适配决策**<br>**禁止输出技术架构分析（核心原理、算法、源码级设计）**<br>**必须使用需求视角（用户场景、功能边界、量化策略）** |
| **product-manager** | 需求定义主角（主观）<br>• 识别 Actor 和 Feature<br>• 挖掘 Scenario（正常为主，故障/异常/运维补充）<br>• 制定量化验收标准<br>• 产出需求规格文档 | 承担需求定义的<br>主体责任 |
| **pm-reviewer** | 产品视角独立质疑<br>• 质疑用户/市场假设<br>• 检查商业盲点和 Actor 遗漏<br>• 发现需求冲突和范围蔓延<br>• 调整置信度评估 | 只对评审质量负责<br>按复杂度产出质疑点 |
| **architect-reviewer** | 技术视角独立质疑<br>• 验证技术可行性<br>• 检查验收标准的工程可验证性<br>• 质疑工程假设<br>• 补充技术盲点 | 只对评审质量负责<br>按复杂度产出质疑点 |
| **architect** | 技术可行性顾问<br>• 评估实现复杂度<br>• 标记高风险 Feature<br>• 评估非功能指标的技术可行性 | **在 define skill 中不承担需求定义主角**<br>仅在需要时评估技术可行性 |

### pm-reviewer 的现实映射与核心价值

**现实对应角色**：
- 产品总监 / Staff PM：从更高业务视角审视需求完整性
- 客户成功 / 售前架构师：理解客户真实采购动机和未被言明的诉求
- 解决方案架构师（业务侧）：打通产品能力与业务场景之间的映射

**核心价值**：
- **打破"我们以为用户需要"的幻觉**：强制暴露未被验证的用户假设
- **发现隐性 Actor**：安全审计、合规检查、运维值班、第三方集成方往往被忽略
- **拦截需求蔓延**：检查 Feature 之间是否存在冲突、是否有过度设计
- **商业价值验证**：Feature 是否真的有"为什么"论证，还是技术自嗨

**与 architect-reviewer 的分工边界**：
- `pm-reviewer` 只质疑**产品/用户/商业层面**的问题，不质疑技术实现细节
- `architect-reviewer` 只质疑**技术/工程/验证层面**的问题，不质疑商业价值

## 文档写入铁律

**每一轮迭代的产出必须写入项目文件，禁止仅在对话中输出。**

原因：需求定义是长期迭代过程，跨越多次对话。如果产出仅存在于对话中，下次对话将丢失所有中间成果，导致重复劳动或方向偏移。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本轮完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. 每次写入正式文档后，同步更新 `docs/requirements/README.md` 看板（轮次状态、文档清单、最新评审摘要）

## 执行流程（强制多轮迭代）

### 前置步骤：产品定位确认（强制）

**准入条件**：无
**产出文件**：`docs/requirements/overview.md`（初始定位草稿，后续轮次持续丰富）

在启动标杆研究之前，必须先完成轻量级产品定位确认。如果产品边界不清，标杆研究可能选错对象，导致后续全部返工。

**执行方式**：
1. **product-manager 通过启发式提问与用户确认**：目标用户/主要 Actor、核心痛点、部署环境、关键约束、明确边界、优先级排序
2. **读取 `.claude/templates/req-overview.md` 模板**，将定位结论整理为 `overview.md` 的初始草稿（覆盖模板要求的前置章节即可，无需追求完美）
3. **主会话将初始定位写入 `docs/requirements/overview.md`**
4. 向用户汇报定位摘要，确认无异议后进入第一轮

> **overview.md 的定位**：从前置步骤开始，它就是**持续迭代的活文档**。第一轮标杆研究可能反过来修正产品定位（例如发现用户描述的痛点在标杆产品中并非主流场景），此时需**回溯修正** overview.md 中的定位章节。

### 第一轮：标杆研究（强制，不可跳过）

**准入条件**：前置定位确认已完成，且 `overview.md` 初始草稿已落盘
**产出文件**：`docs/requirements/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
**深度研究要求**（拒绝清单式罗列，强制需求视角）：
- **写作风格**：拒绝翻译式罗列，必须有"对我方需求的启示"；所有分析从用户/产品视角出发，严禁技术实现细节
- **模板约束**：产出前必须读取 `.claude/templates/ref-requirements.md`，严格按模板章节填充
- **核心验收点**：product-manager 依据模板自检清单验收研究深度，重点检查是否有差异化策略分析和适用性评估

**执行方式**：
1. Master 识别需要研究的标杆列表（基于**前置步骤已确认的产品定位**和用户要定义的产品/子系统）
2. **并行启动 researcher 时，必须明确传递需求模式指令**：
   - 产出目录：`docs/requirements/reference/`
   - 分析视角：**需求/用户/产品视角**，禁止技术分析
   - 模板约束：使用 `.claude/templates/ref-requirements.md`
   - 禁止内容：核心原理、算法、数据结构、源码级设计、性能瓶颈、架构决策
   - 强制内容：用户场景、功能边界、需求组织方式、验收标准量化策略
3. researcher 完成研究后，**主会话将研究内容写入** `docs/requirements/reference/<product>.md`
4. **product-manager 验收研究深度**，依据 `.claude/templates/ref-requirements.md` 中的自检清单逐项确认，重点检查：
   - 是否有**洞察**（不是简单翻译竞品文档）？
   - 是否识别了标杆的**差异化需求策略**？
   - 是否有明确的**适用性评估**？
5. **不达标时明确反馈 researcher 补充分析**，循环优化直到通过
6. **主会话验收**：检查文档中是否出现了技术实现细节（如模块名、算法名、接口函数名），如有则判定为视角错误，**强制打回重写**
7. **第一轮置信度评分**（主会话或 evaluator agent 执行）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 研究深度 | 25% | 不是清单罗列，有 Feature 组织方式、Scenario 策略、量化边界的分析 |
| 视角正确性 | 25% | 无技术实现细节，纯需求视角 |
| 覆盖完整性 | 20% | 至少 2-3 个标杆，每个都有适用性评估和差异化分析 |
| 洞察质量 | 20% | 有明确的"对我方需求的启示" |
| 可复用性 | 10% | 结论能直接指导第二轮 Actor-Feature 设计 |

**进入第二轮门槛：≥ 0.80**
- ≥ 0.80：✅ 允许进入第二轮
- 0.70-0.79：⚠️ 补充指定标杆的薄弱维度后重评
- < 0.70：❌ 重新研究

8. **确认所有文件已落盘**，评分达标后，进入下一轮

**product-manager 验收检查清单**（以 `.claude/templates/ref-requirements.md` 自检清单为准，重点核查以下质量信号）：
- [ ] 不是清单式罗列，有"对我方需求的启示"？
- [ ] 识别了标杆的 **差异化需求策略**（主动放弃什么场景）？
- [ ] 是否有明确的 **适用性评估**（✅适合 / ⚠️需调整 / ❌不适用）？
- [ ] **未通过验收的研究报告，禁止进入第 2 轮。**

### 第二轮：Actor-Feature 识别与定稿（总体层，强制）

**准入条件**：标杆研究完成并通过 product-manager 验收

本轮完成**总体层的完整闭环**：先写 Actor-Feature 草稿，再评审质疑，最后修正定稿。只有 Actor-Feature 框架定稿后，才进入第三轮的细节挖掘。

#### Step 2.1：Actor 识别

**产出文件**：`docs/requirements/overview.md#actors`

1. **product-manager** 基于**前置步骤写入 `overview.md` 的产品定位结论**和**标杆研究洞察**，识别所有与系统交互的 Actor
2. **并行启动 researcher** 补充多个竞品的 Actor 映射表，帮助发现遗漏角色。每个 researcher 聚焦一个标杆产品
3. 每个 Actor 必须明确：类型（人类/外部系统/内部组件）、职责、交互方式、关注点
4. **写入文件**并汇报

#### Step 2.2：Feature 识别

**产出文件**：`docs/requirements/overview.md`（草稿状态，定稿后移除草稿标记）

1. **product-manager** 围绕 Actor 的核心目标识别 Feature 列表
2. 每个 Feature 记录：**一句话价值摘要**、**涉及 Actor**、**归属特性域**、**优先级**
3. 产出 **Feature 间依赖关系矩阵**（全局视图，放在 overview.md）
4. **并行启动 architect** 评估各 Feature 的实现复杂度，标注高风险项
5. **禁止快速列出 Feature 名称**，每个 Feature 必须有价值论证
6. **Feature 价值论证必须是推导式，禁止"矩阵后直接给结论"**：
   - 每个 Feature 的优先级必须有"痛点频率 × 影响程度"矩阵推导
   - Feature 归属特性域的拆分必须有 rationale（为什么这两个 Feature 属于同一 domain）
   - 禁止简单罗列 Feature 名称
7. **当存在多个特性域时，可并行启动多个 product-manager agent**（每个负责一个 domain 的 Actor-Feature 映射），禁止单 agent 串行产出所有特性域内容
8. **读取 `.claude/templates/req-overview.md` 模板**，将 Actor-Feature 写入 `overview.md`

> **内容边界**：`overview.md` 中的 Feature 价值只需**一句话摘要**（做什么+为什么），详细的价值论证留给第三轮的特性域文档（参见 `req-feature.md` 模板）展开。

#### Step 2.3：Actor-Feature 独立质疑

**准入条件**：`overview.md` 草稿已写入文件（含 Actor 定义和 Feature 总览）

1. **并行启动 `pm-reviewer` 和 `architect-reviewer`** 进行独立评审
3. 两位 reviewer **独立阅读** Actor-Feature 材料（不要先看 Product Manager 结论）
4. 按特性域复杂度产出质疑点：

| 特性域复杂度 | 质疑点数量 |
|-------------|-----------|
| 简单（单 Actor、单接口） | 至少 3 个 |
| 中等（多 Actor、跨子系统） | 至少 5 个 |
| 复杂（涉及一致性、故障恢复、编排调度） | 至少 7 个 |

5. **质疑分布要求**（强制）：除满足数量要求外，双评审合计必须覆盖以下维度，每个维度至少 1 个质疑：
   - **Actor 遗漏**：是否遗漏了安全审计、合规检查、运维值班等隐性 Actor？
   - **Feature 价值**：某个 Feature 是否缺少"做什么+为什么"的价值论证？
   - **依赖关系**：Feature 间依赖矩阵中的强/弱依赖是否合理？
   - **优先级推导**：Feature 优先级是否基于"痛点频率 × 影响程度"矩阵，而非主观判断？
   - **核心表格针对性质疑**：权力/利益矩阵、Feature 依赖关系矩阵等核心决策表格，每个表格至少 1 个针对性质疑

6. **pm-reviewer 质疑维度**：
   - **用户假设质疑**：这个 Feature 解决的是真实用户痛点，还是团队的主观臆断？
   - **Actor 边界质疑**：是否遗漏了安全审计、合规检查、运维值班等隐性 Actor？
   - **商业价值质疑**：Feature 之间的优先级是否合理？是否存在需求蔓延？
   - **Feature 完整性质疑**：核心能力是否被遗漏？Feature 粒度是否合适？

7. **architect-reviewer 质疑维度**：
   - **技术可行性质疑**：这个需求在现有架构约束下是否可实现？
   - **风险质疑**：Feature 之间是否存在技术冲突？是否明确了技术取舍？
   - **非功能维度质疑**：是否缺少关键的非功能需求方向？

#### Step 2.4：Actor-Feature 修正定稿

1. **product-manager** 阅读两位 Reviewer 的质疑摘要
2. **必须回应每个质疑**（接受/反驳/讨论）
3. 修正 `overview.md`，移除草稿标记，产出 Actor-Feature 终稿：
   - Actor 定义
   - Feature 总览（含一句话价值摘要、涉及 Actor、归属特性域、优先级）
   - Feature 间依赖关系矩阵
   - 全局非功能需求
4. 质疑摘要直接追加到 `overview.md` 末尾（200 字以内）
5. **第二轮置信度评分**（主会话或 evaluator agent 执行）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| Actor 完整性 | 25% | 无遗漏隐性 Actor（审计、运维、合规、第三方集成方） |
| Feature 价值论证 | 25% | 每个 Feature 都有"做什么+为什么"的一句话价值摘要 |
| 依赖关系清晰度 | 20% | Feature 间依赖矩阵完整，强/弱依赖区分合理 |
| 双评审质量 | 20% | pm-reviewer 和 architect-reviewer 的质疑都有明确回应 |
| 与标杆对齐度 | 10% | Feature 设计体现标杆研究洞察，无选择性偏见 |

**进入第三轮门槛：≥ 0.85**
- ≥ 0.85：✅ 允许进入第三轮
- 0.75-0.84：⚠️ 修正后重评
- < 0.75：❌ 回到 Step 2.2 重构

6. **只有 Actor-Feature 定稿且评分达标后，才允许进入第三轮**

### 第三轮：Scenario 挖掘与特性域定稿（细节层，强制）

**准入条件**：Actor-Feature 已在第二轮定稿

本轮完成**细节层的完整闭环**：基于定稿的 Actor-Feature 框架，挖掘 Scenario、编写特性域文档、评审定稿。

```
┌─────────────────────────────────────────────────────────────────┐
│                     第三阶段：细节层定稿                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Phase A: Scenario 挖掘与特性域文档编写                           │
│  ├── product-manager 基于定稿的 Actor-Feature                   │
│  ├── 挖掘 Scenario（正常为主，故障/异常/运维补充）               │
│  ├── 量化非功能需求                                             │
│  └── 产出各特性域文档初稿                                       │
│                          │                                      │
│                          ▼                                      │
│  Phase B: Feature-Scenario 独立质疑                              │
│  ├── 并行启动 pm-reviewer + architect-reviewer                 │
│  ├── 按复杂度产出质疑点                                         │
│  └── 产出质疑摘要                                               │
│                          │                                      │
│                          ▼                                      │
│  Phase C: Feature-Scenario 修正定稿                              │
│  ├── 回应质疑、补充遗漏                                         │
│  ├── 修正置信度评估                                             │
│  └── 产出需求规格终稿                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Phase A: Scenario 挖掘与特性域文档编写（分域并行）

**产出文件**：各 `docs/requirements/<feature-domain>.md` 初稿

**执行策略**：
1. **主会话读取 `overview.md`（Actor-Feature 终稿）**，作为所有 agent 的统一输入源
2. **主会话根据 Feature 归属，将 domain 列表拆分**。若 domain 间存在强依赖（如 domain A 的 scenario 是 domain B 的前置条件），将强依赖链串行处理；无依赖的 domain 并行处理
3. **并行启动多个 `product-manager` agent**，每个 agent 严格限定：
   - 你只能写 `docs/requirements/<feature-domain>.md` 这一个文档
   - 输入上下文：完整的 `overview.md` + 本 domain 涉及的 Feature 列表 + 相关标杆研究报告
   - 禁止写其他 domain 的内容
4. 每个 product-manager agent 基于定稿的 Actor-Feature，为本 domain 的每个 Feature 挖掘完整 Scenario 列表
5. **写作原则**：
   - **正常路径是核心**，必须充分展开
   - **故障/异常/运维场景从完整性角度补充**，关键分布式 Feature 至少覆盖正常路径 + 4 种故障模式
   - **每个 Scenario 必须指定具体 Actor**
   - **所有内容结构严格遵循 `.claude/templates/req-feature.md` 模板**
6. **非功能需求量化**（性能、可用性、容量、扩展性、一致性、安全性、可观测性）
7. **写入各特性域文档初稿**
8. **所有 domain 初稿完成后，统一进入 Phase B（双评审）**

#### Phase B: Feature-Scenario 独立质疑

**准入条件**：特性域文档初稿已写入文件

1. **并行启动 `pm-reviewer` 和 `architect-reviewer`** 进行独立评审
2. 质疑点数量要求与第二轮 Step 2.3 相同

3. **pm-reviewer 质疑维度**：
   - **场景覆盖质疑**：是否遗漏了关键的故障模式？运维场景是否完整？
   - **验收标准质疑**：每个 Scenario 的验收标准是否可独立验证？是否存在模糊描述？
   - **标杆质疑**：是否有选择性借鉴标杆的偏见？
   - **正常场景充分性质疑**：核心使用路径是否描述足够清晰？

4. **architect-reviewer 质疑维度**：
   - **验收标准质疑**：验收标准是否具备工程可验证性？测试是否可自动化？
   - **接口质疑**：涉及子系统交互的需求，接口边界是否完备？
   - **非功能需求质疑**：量化指标是否足够？是否具体可测？

#### Phase C: Feature-Scenario 修正定稿

1. **product-manager** 阅读质疑摘要
2. **必须回应每个质疑**（接受/反驳/讨论）
3. 修正置信度评估
4. 补充遗漏的 Scenario 和 trade-off
5. 更新为需求规格终稿：`docs/requirements/<feature-domain>.md`
6. **Reviewer 质疑摘要记录方式**：直接追加到对应特性域文档末尾，简短凝练（200 字以内）
7. **第三轮置信度评分**（主会话或 evaluator agent 执行，按 domain 分别评分）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 场景覆盖度 | 25% | 正常路径充分展开 + 至少 4 种故障模式（分布式场景） |
| 验收标准可验证性 | 25% | 每个 Scenario 有可量化的验收标准和明确的验证方法 |
| 非功能需求量化 | 20% | 性能/可用性/一致性等指标有具体数字和验收方法 |
| 双评审质量 | 20% | 所有质疑点被回应，补充内容已融入文档 |
| 命名空间纪律 | 10% | 无子系统内部模块名/类名/函数名，纯用户视角 |

**特性域定稿门槛：≥ 0.85**
- ≥ 0.85：✅ 标记为该 domain 终稿
- 0.75-0.84：⚠️ 进入 Phase C 修正后重评
- < 0.75：❌ 回到 Phase A 重写

#### 验收标准

- [ ] `overview.md` 草稿（含 Actor 定义和 Feature 总览）**已写入文件**
- [ ] **pm-reviewer Actor-Feature 独立质疑已完成**
- [ ] **architect-reviewer Actor-Feature 独立质疑已完成**
- [ ] **所有 Actor-Feature 质疑点已回应（接受/反驳/讨论）**
- [ ] Actor-Feature 终稿**已写入 `overview.md`**
- [ ] 特性域文档初稿**已写入文件**
- [ ] **pm-reviewer 特性域独立质疑已完成**
- [ ] **architect-reviewer 特性域独立质疑已完成**
- [ ] **所有特性域质疑点已回应（接受/反驳/讨论）**
- [ ] **隐性假设和遗漏场景已补充**
- [ ] 每个 Scenario 有明确的验收标准
- [ ] 非功能需求已量化
- [ ] **评审摘要已追加到各文档末尾**

### 第四轮：需求维护与扩展（用户驱动）

**核心定位**：前 3 轮完成后，需求文档已进入"可用状态"。第 4 轮不是自动运行的后台任务，而是**用户在后续开发过程中，按需调用 `/ky:define` 来修正、补充或调整需求定义**。

**触发原因**：
1. **实现验证后发现需求缺陷**（原型测试、用户反馈推翻了原假设）
2. **需求或范围变更**（新增 feature domain、裁剪 Feature、调整 Actor 边界）
3. **补充新的标杆研究**（竞品发布新功能、团队对某领域认知加深）
4. **特性级开发中的反馈**（`/opsx:apply` 中发现需求不清、验收标准不可验证）

**交互模式**：

```
用户调用 /ky:define
    ↓
状态自检（只呈现信息，不替用户做决定）
    ↓
用户明确指定任务
    ↓
执行（支持快速通道和第 1/2/3 轮局部子流程）
```

**快速通道**（小修不走完整轮次）：
- **文档一致性修正**：直接读取相关文档 → 定位差异 → 与用户确认 → 写入修正
- **单 Feature 补充**：直接补充某个 Feature 的 Scenario，无需重新走完整流程
- **单篇标杆补充**：researcher 产出 → product-manager 验收 → 落盘，不涉及后续轮次

**大变更回退到局部子流程**：
- 新增 Feature → 进入第二轮 Step 2.2（Feature 识别写入 overview.md）→ Step 2.3（Actor-Feature 评审）→ Step 2.4（修正定稿）→ 第三轮 Phase A（Scenario 挖掘）→ Phase B（评审）→ Phase C（定稿）
- 推翻已有 Scenario 假设 → 进入第三轮 Phase A（Scenario 挖掘）→ Phase B（评审）→ Phase C（修正定稿）
- 新增特性域 → 第三轮 Phase A（产出新特性域文档）→ Phase B（评审）→ Phase C（定稿）

## 输出格式

所有需求交付件的内容结构、章节要求、自检清单**以模板为唯一规范来源**：

- **全局文档**：`docs/requirements/overview.md` → 严格遵循 `.claude/templates/req-overview.md`
- **特性域文档**：`docs/requirements/<feature-domain>.md` → 严格遵循 `.claude/templates/req-feature.md`
- **标杆分析文档**：`docs/requirements/reference/<product>.md` → 严格遵循 `.claude/templates/ref-requirements.md`

本 Skill 和 Agent 只负责**流程控制、努力程度、写作风格和质量门槛**。写文档前**必须先读对应模板**，产出必须覆盖模板 `mandatory-sections` 中列出的所有章节。

## 输出位置

- `docs/requirements/README.md` — 目录导航 + 当前轮次看板（轮次状态、文档冻结状态、最新评审摘要）
- `docs/requirements/overview.md` — 全局需求总纲（含 Actor 定义、Feature 总览、Feature 间依赖关系、全局非功能需求）
- `docs/requirements/<feature-domain>.md` — 按特性域组织的需求规格（含 Feature 详细价值论证、Scenario、域级验收标准、域级非功能需求）
- `docs/requirements/reference/<product>.md` — 标杆需求分析（至少 2-3 个）

> **评审记录不再独立成文**，Reviewer 质疑摘要直接追加到对应文档末尾，简短凝练（150 字以内）。

## 红旗信号（自查清单）

以下情况出现时，**立即停止，回到研究阶段**：

- 🚩 未做产品定位确认就进入标杆研究
- 🚩 标杆研究文档 < 2 个
- 🚩 Feature 只有名称没有"为什么"论证
- 🚩 Scenario 未覆盖正常路径（只有故障场景）
- 🚩 验收标准无法量化（"高性能"、"高可用"、"稳定"）
- 🚩 单次对话内完成"研究→Feature→Scenario→文档"
- 🚩 **Agent 产出内容未写入项目文件**（仅存在于对话中）
- 🚩 **向用户汇报"本轮完成"但文件未落盘**
- 🚩 **需求文档未经 pm-reviewer 和 architect-reviewer 双质疑直接定稿**
- 🚩 Actor-Feature 未定稿就进入 Scenario 挖掘

## 常见借口与强制回应

| 用户说 | 必须回应 |
|--------|----------|
| "先快速出一版需求" | 拒绝：快速产出的需求后续返工成本更高，必须完成标杆研究 |
| "这个需求 obvious，不用分析" | 质疑：obvious 的需求往往有最多隐藏的假设，必须显式定义 |
| "就按你说的做" | 警告：当前置信度 X%，建议完成 Y 调研后再定 |

## 与用户协作模式

每次交互：

1. **汇报当前状态**：
   - 已完成哪些标杆研究
   - 当前在哪一轮迭代
   - 当前置信度评估

2. **提出本次重点**：
   - 本轮要解决的问题
   - 需要用户提供的输入
   - 预计产出

3. **确认下一步**：
   - 用户确认后再继续
   - 低置信度时主动建议"是否需要补充调研 X"
