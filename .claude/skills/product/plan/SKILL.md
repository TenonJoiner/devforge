# product/plan — 规划性思考

## 概述

里程碑 → 迭代主题 → 执行计划 → **三重评审定稿**。从粗到细，从近到远。

**核心原则**：
- **两层递进**：规划（里程碑/主题/Backlog）→ 执行（选择/Wave/就绪检查）
- **三重评审**：`architect`（技术可行）+ `product-manager`（需求对齐）+ `project-manager`（可交付性）独立评审
- **近详远略**：当前里程碑详细规划，后续里程碑粗略规划
- **每层确认**：规划后询问确认，再进入下一层
- **计划是活的**：滚动更新，而非一次定死

## 何时使用

- 制定/更新产品迭代计划（iteration-plan.md）
- 分解需求为 Proposal Backlog
- 协调多子系统并行开发
- 调整计划应对变更

**不适用**：技术方案设计（用 architect）、需求定义（用 define）、单个 proposal 任务分解（特性级处理）

## 启动检测

检测 `docs/iteration-plan.md` 是否存在：
- **不存在** → 模式 A（初始规划）
- **已存在** → 询问：滚动更新(B) / 快速调整(C) / 里程碑调整(D)

## 核心流程

### 第一层：里程碑/迭代规划

一次性完成：里程碑划分 + 迭代主题分解 + Proposal Backlog 建立。

**步骤**：

**1. 里程碑划分**：确定 3-5 个里程碑，每个有达成标准和关键交付物。

**2. 迭代主题分解**（仅当前里程碑，从里程碑目标向下分解）：
- 每个迭代确定**主题**（本轮要达成的能力）和**容量预算**
- 主题应支撑里程碑目标，被依赖的能力优先安排
- 单迭代容量 ≤ Velocity × 1.2

**3. Proposal Backlog 建立**（仅当前里程碑详细，后续粗略）：
- 单一职责、可独立交付、粒度 1-4 周
- 命名：`<子系统>-<动作>-<对象>`（kebab-case）
- 每个 proposal 标注所属迭代主题
- **集成测试必须独立立项**：每个子系统或跨子系统交互场景需有对应的集成测试 proposal（命名：`<子系统>-integration-test-<场景>`），不可合并到特性 proposal 中

**复杂度估算**（基于 AI 辅助编码范式，人工编码需相应上调）：

| 档位 | 故事点 | 工作量 | 判断标准 |
|------|--------|--------|----------|
| S | 1-2 | 1-2天 | ≤3文件，无外部依赖 |
| M | 3-5 | 3-5天 | 单子系统，依赖清晰 |
| L | 8-13 | 1-2周 | 跨文件单特性，需接口定义 |
| XL | 21+ | 必须拆解 | 多子系统或技术不确定（特殊情况需审批+拆分计划+检查点+Spike验证） |

**4. 关键路径与风险识别**：
- 跨里程碑关键路径（最长依赖链）
- 🔴 高风险任务占比 ≤ 40-50%（分布式存储系统阈值）
- 关键路径上的任务有充足资源

**输出（渐进式）**：

| 范围 | 粒度 | 内容 |
|------|------|------|
| 当前里程碑 | 详细 | 迭代主题 + 容量预算 + Proposal Backlog |
| 下一里程碑 | 中等 | 关键 proposal + 接口依赖 |
| 更远里程碑 | 粗略 | 仅关键交付物 + 大致点数范围 |

**确认点**：里程碑划分？迭代主题？Backlog 估算？→ 确认后进入第二层。

---

### 第二层：迭代执行规划

迭代启动前，从 Backlog 选择 proposal，制定 Wave 分组。

**步骤**：

**1. 选择 Proposal**：
- 总点数 ≤ 迭代预算
- 必须包含关键路径任务
- 优先选择依赖已就绪的
- 单迭代 L 数量上限取决于团队规模（参考：5-8 人 ≤ 2-3，10-15 人 ≤ 5-6，20+ 人 ≤ 8-10），XL = 0（特殊审批除外）

**2. Wave 分组**：
- Wave 0：无依赖（立即启动）
- Wave 1：仅依赖 Wave 0
- Wave N：仅依赖 Wave 0..N-1

**3. 风险与缓冲**：

| 风险类型 | 标记 | 应对 |
|----------|------|------|
| 技术可行性 | 🔴 | Spike（1-3天）+ 50-100% 缓冲 |
| 性能达标 | 🔴 | 原型验证 + 性能基线 |
| 外部依赖 | 🟡 | 接口冻结点 + fallback |

**4. 并行就绪检查**：
- [ ] 接口契约已文档化
- [ ] Mock/桩机制已就绪
- [ ] 外部依赖已登记交付时间

**5. 测试 Proposal**：
- 单元测试包含在各特性 proposal 中，不单独立项
- 集成测试、故障测试作为独立 proposal（每迭代点数占比 ≥ 15%）
- 功能 proposal 无对应单元测试 = 未完成

**合理性检查清单**：
- [ ] 总点数 ≤ Velocity × 1.2
- [ ] 🔴 风险任务 ≤ 40-50%，均有应对方案
- [ ] 关键路径已识别，无单点瓶颈
- [ ] 跨子系统契约已冻结，Mock 就绪
- [ ] 外部依赖已确认交付时间

---

### 第三层：三重评审（强制）

**准入条件**：第二层迭代执行规划已产出并写入 `docs/iteration-plan.md`

计划在技术、需求、可交付性三个维度上天然存在三角博弈。必须通过**三重独立评审**后才能视为可执行计划。

```
                    iteration-plan.md
                           │
            ┌──────────────┼──────────────┐
            ↓              ↓              ↓
    ┌───────────────┐ ┌─────────────┐ ┌───────────────┐
    │   architect   │ │product-manager│ │ project-manager│
    │  技术可行性评审  │ │  需求对齐评审  │ │  可交付性评审  │
    └───────┬───────┘ └──────┬──────┘ └───────┬───────┘
            │                │                │
            └────────────────┼────────────────┘
                             ↓
                     汇总修正 → 更新 iteration-plan.md
```

#### Step 1: architect 技术可行性评审

**并行启动 `architect`**，针对 iteration-plan.md 从以下维度发表独立评审意见（150 字以内/里程碑）：

- **技术风险质疑**：高复杂度/高不确定性 proposal 的占比是否合理？风险应对方案是否充分？
- **关键路径质疑**：技术依赖链识别是否准确？是否存在被低估的串行阻塞？
- **接口就绪质疑**：跨子系统契约冻结点和 Mock 就绪检查是否符合实际工作量？
- **性能/一致性质疑**：里程碑验收目标中的非功能指标（延迟/吞吐/一致性）是否可达？
- **里程碑技术假设质疑**：每个里程碑是否有未验证的关键技术假设？标注哪些需要前置 Spike。

**产出**：`docs/iteration-plan.md#评审记录` 中追加 architect 评审摘要。

#### Step 2: product-manager 需求对齐评审

**并行启动 `product-manager`**，针对 iteration-plan.md 从以下维度发表独立评审意见（150 字以内/里程碑）：

- **需求优先级质疑**：当前迭代选择的 proposal 是否覆盖了优先级最高的 Feature？
- **需求完整性质疑**：是否有需求被遗漏或未对应到 proposal？
- **迭代主题对齐质疑**：每个迭代的主题是否与产品里程碑目标一致？
- **验收标准质疑**：里程碑中的交付物是否对应了需求文档中的明确验收标准？
- **范围质疑**：是否存在需求蔓延或迭代范围过大的问题？

**产出**：`docs/iteration-plan.md#评审记录` 中追加 product-manager 评审摘要。

#### Step 3: project-manager 可交付性评审

**由 `project-manager` 自身执行可交付性检查**，同时作为三重评审的一环进行独立审视（150 字以内/里程碑）：

- **资源质疑**：团队容量是否支撑当前计划？缓冲是否充足？
- **Wave/并行质疑**：Wave 分组是否真正最大化了并行度？隐藏阻塞点是否已识别？
- **可交付性质疑**：里程碑目标在给定资源和约束下是否可达？
- **风险与缓冲质疑**：高风险任务占比是否超过 40-50%？风险应对方案是否可行？
- **外部依赖质疑**：接口冻结点、外部 SDK/库 的交付时间是否已确认并留有缓冲？

**产出**：`docs/iteration-plan.md#评审记录` 中追加 project-manager 自评审摘要。

#### Step 4: 汇总修正与定稿

1. `project-manager`（主会话汇总）阅读三个评审摘要
2. **必须回应每个评审意见**：
   - **接受**：修正 iteration-plan.md（调整 Wave/裁剪 scope/增加缓冲）
   - **反驳**：在评审记录中简要说明反驳理由
   - **讨论**：标注为待用户确认项
3. 修正后的 iteration-plan.md 再次确认是否写入文件
4. 经三重评审通过的计划才可进入执行阶段

#### 验收标准

- [ ]  milestone/迭代/Wave 规划**已写入** `docs/iteration-plan.md`
- [ ] **architect 技术可行性评审已完成**
- [ ] **product-manager 需求对齐评审已完成**
- [ ] **project-manager 可交付性评审已完成**
- [ ] **评审摘要已追加到 iteration-plan.md**
- [ ] **三方意见均已回应（接受/反驳/讨论）**
- [ ] 经修正后的计划已落盘

## 运行模式

### 模式 A：初始规划

新项目或重大方向调整。执行第一层 → 第二层 → 第三层完整流程。

### 模式 B：滚动更新

基于进展调整：
1. 估算校准（实际 vs 估算偏差率，>30% 调整基准）
2. 延期根因分析（需求变更/技术难点/估算偏低/资源问题/依赖延迟）
3. 调整当前迭代（移除非关键 proposal、更新 Wave）
4. 更新 Velocity 和风险状态

滚动更新若涉及**里程碑目标调整或新增 XL 级任务**，必须重新触发第三层三重评审。

### 模式 C：快速调整

应对突发：插入紧急需求、资源变动、范围裁剪。评估影响 → 调整 Wave → 更新文档。
快速调整幅度较小（不涉及里程碑变更）时，可由 project-manager 自行判断是否简化为单维度评审。

### 模式 D：里程碑调整

重大变更时：重新划分里程碑/迭代分配，仅保留最近迭代详情，后续清空待重新规划。
必须重新完整执行第一层 → 第二层 → 第三层。

## 铁律检查

- 两层递进：规划 → 执行，每层确认后再进入下一层
- 每个 proposal 必须是可独立交付的最小有价值单元
- 关键路径上的 proposal 必须识别并优先安排
- **计划必须经过三重评审才能视为可执行**

## 红旗信号

- 🚩 跳过里程碑规划直接进入 Wave 分组
- 🚩 一次性输出全部迭代的详细 plan
- 🚩 所有 proposal 都在 Wave 0（未识别依赖）
- 🚩 单个 proposal 超过 4 周
- 🚩 无独立的集成测试 proposal（集成测试不可合并在特性 proposal 中）
- 🚩 计划过于乐观（无缓冲）
- 🚩 **iteration-plan.md 未经三重评审直接推进执行**

## 输出位置

- `docs/iteration-plan.md` — 主计划文件（含可选的评审记录追加区）
