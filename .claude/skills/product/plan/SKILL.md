# product/plan — 规划性思考

## 核心理念

**两阶段规划，按顺序执行；每阶段完成前，不展开下一阶段。**

两阶段规划顺序：

1. **第 1 阶段**：里程碑规划 + Backlog 清单（milestone-plan.md）
2. **第 2 阶段**：迭代执行计划（iteration-m*-i*.md）— **第 1 阶段完成后才能进入**，一次一迭代

独立评审：每阶段完成后，`architect-reviewer` + `product-manager` + `project-manager` 三个 Agent 分别执行**独立评审**，评审通过后进入下一阶段。

**核心原则**：
- **线性推进**：按第 1 阶段 → 第 2 阶段的顺序执行，不跳跃
- **一次一文件**：每次执行只产出一个计划文件
- **出口标准驱动**：每阶段有明确的**出口标准**，达成后**确认后推进**进入下一阶段
- **计划是活的**：评审通过后如需调整，执行滚动更新

### 两阶段分工与近详远略

**设计背景**：v5.0.0 之前是三层（里程碑 → 每里程碑一个文件 → 迭代执行计划），中间层的 Backlog 清单与里程碑计划天然是同一决策过程的产出，单独拆文件增加维护成本和文档间同步负担。v5.0.0 将中间层拆除，内容分别向上下合并。

**两阶段各管什么**：

| 维度 | 第 1 阶段 milestone-plan.md | 第 2 阶段 iteration-m*-i*.md |
|------|------------------------|--------------------------|
| 核心问题 | **有什么活要干** | **这两周具体怎么干** |
| 里程碑划分 | ✅ 全部里程碑（M1-M4） | — |
| Backlog 清单 | ✅ 当前里程碑详细，后续里程碑粗略 | — |
| 迭代总览 | ✅ 仅主题 + 容量预算 | — |
| Proposal → 迭代分配 | ❌ 不分配 | ✅ 从 Backlog 选入本迭代 |
| Wave 并行编排 | ❌ 不涉及 | ✅ 迭代内 proposal 间并行分组 |
| 执行人分配 | ❌ 不涉及 | ✅ 可选 |

**近详远略原则**：

milestone-plan.md 是一个**活文档**，只有当前里程碑展开详细 Backlog 和迭代总览，后续里程碑仅粗略描述目标和预估规模。随项目推进滚动更新：

```
项目启动时：M1 详细 Backlog + 迭代总览 | M2/M3/M4 粗略
M1-I5 左右：M1 保持 | M2 展开详细 Backlog + 迭代总览 | M3/M4 粗略
M2-I4 左右：M1/M2 保持 | M3 展开详细 | M4 粗略
```

**关键规则**：后续里程碑的 Backlog 展开是**更新 milestone-plan.md**，不是创建新文件。一个项目始终只有一个 milestone-plan.md。

## 复杂度判定

本 skill 各阶段的内容类型与基准复杂度判定如下（详见 CLAUDE.md「复杂度驱动原则」）：

| 阶段 | 内容类型 | 基准复杂度 | 判定依据 |
|------|---------|-----------|---------|
| 第 1 阶段：里程碑规划 | 结构化规划 | **高** | 基于已有需求/架构文档进行结构化组合与编排；涉及跨子系统依赖分析 |
| 第 2 阶段：迭代执行计划 | 模板化文档 | **中等** | 基于里程碑计划按模板分解迭代任务；涉及关键路径编排 |

**决策属性上调**（辅助判定）：
- 第 1 阶段涉及「跨子系统边界」的 proposal 依赖分析（通常都有），复杂度从「中等」上调至「高」
- 第 1 阶段涉及「不可逆决策」（里程碑划分影响后续全部迭代），但该属性已内化在跨子系统依赖分析的编排复杂度中，不重复上调
- 第 2 阶段涉及「关键路径」上的 proposal 编排，复杂度从「低-中等」上调至「中等」

**复杂度与 agent 配置对照**：

| 阶段 | 复杂度 | 并行产出 agent | 独立评审 agent |
|------|--------|---------------|---------------|
| 第 1 阶段 | 高 | ≥2 个 agent（不同角色） | 3 个 reviewer（覆盖技术/产品/项目管理） |
| 第 2 阶段 | 高（决策属性上调：关键路径编排 + 多角色交叉验证） | 3 个 agent（不同角色） | 3 个 reviewer |
| 维护模式 | 低 | 1 个对应角色 agent | 按需 |

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

原因：迭代计划是长期迭代过程，跨越多次对话。如果产出仅存在于对话中，下次对话将丢失所有中间成果，导致重复劳动或方向偏移。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **写入后必须验证文件存在**（`ls` 确认文件已落盘）

## 启动检测

`/df:plan` 启动时，执行**启动检测**判断当前所处阶段，**只提示唯一下一步动作**：

| 检测结果 | 当前阶段 | 唯一下一步动作 |
|---------|---------|--------------|
| `milestone-plan.md` 不存在 | **第 1 阶段** | 开始里程碑规划 |
| `milestone-plan.md` 存在（含 Backlog），第一个未规划的迭代（如 M1-I1）的 `iteration-m1-i1.md` 不存在 | **第 2 阶段** | 开始 M1-I1 的迭代执行计划（一次只做这一个迭代） |
| 所有当前目标已完成评审 | **下一目标** | 继续执行 `/df:plan`，自动进入下一个迭代的规划 |
| 当前里程碑所有迭代已规划完毕，下一个里程碑 Backlog 尚未展开 | **里程碑推进** | 更新 milestone-plan.md，展开下一个里程碑的详细 Backlog + 迭代总览（滚动更新） |
| 全部迭代均已规划完毕 | **计划完备** | 建议执行 `/opsx:new` 启动特性开发，或滚动更新 |

**不适用**：技术方案设计（用 architect）、需求定义（用 define）、单个 proposal 任务分解（特性级处理）

### 豁免场景

以下阶段/Step 无正式文档产出，不适用出口标准与评审修正循环，但需有轻量级准入检查清单确认信息充分性：

- **启动检测**：文件系统扫描判断阶段，无产出
- **Step 1（项目背景收集 / 迭代上下文收集）**：纯交互式信息收集，产出为用户回答的上下文信息，不写入正式文档。准入检查清单：确认所有必收集项已回答或已显式标注为假设值
- **Step 2 前置（Plan 阶段）**：轻量级任务解构，不产出正式文档，不适用出口标准与评审修正循环

## 执行流程

> **铁律：主会话禁止直接产出文档内容。** 主会话负责流程调度（启动检测、上下文收集、Agent 调度、评审协调、评审修正循环），文档编写委托给 `project-manager` 或 `architect` Agent（按 Step 中的角色分配）。

### 第 1 阶段：里程碑规划 + Backlog

**目标**：建立里程碑计划文件 `docs/iteration-plan/milestone-plan.md`。

**Step 1：项目背景收集（强制，不可跳过）**

在开始规划前，必须通过交互式提问向用户收集以下信息。**禁止自行假设任何团队/时间参数**：

| 必收集项 | 说明 | 示例 |
|---------|------|------|
| 团队规模 | 参与开发的人数、技能分布 | "3 名 C 语言工程师 + 1 名 Python 工程师" |
| 全职/兼职 | 团队成员是否全职投入本项目 | "2 人全职，2 人兼职 50%" |
| 预期时间跨度 | 项目整体交付周期预期 | "6 个月内完成 MVP" |
| 迭代周期偏好 | 每个迭代的时长 | "2 周一个迭代" |
| 历史 Velocity | 如有历史数据，团队每迭代可完成的故事点 | "无历史数据" 或 "约 15 SP/迭代" |
| 外部约束 | 是否有硬性截止日期、外部依赖、发布窗口等 | "Q3 前需要 Demo 给客户" |

收集方式：一次性提出所有问题，等待用户回答后再开始规划。如果用户明确表示"你自己定"或给出部分信息，则仅对未提供的项做合理假设并**在执行摘要中显式标注为假设值**。

> **豁免说明**：本 Step 为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。

**准入检查清单**：
- [ ] 团队规模已确认或已标注为假设值
- [ ] 全职/兼职比例已确认或已标注为假设值
- [ ] 预期时间跨度已确认或已标注为假设值
- [ ] 迭代周期偏好已确认或已标注为假设值
- [ ] 历史 Velocity 已确认或已标注为假设值
- [ ] 外部约束已确认或已标注为假设值

**Step 2 前置：Plan 阶段（建议）**

第 1 阶段复杂度为「高」（结构化规划 + 跨子系统依赖 + 不可逆决策），建议先解构再发散。主会话基于项目背景和需求/架构文档，完成轻量级任务解构：

1. **确认切分维度**：按职能侧重（Backlog 结构与时间规划 vs 技术依赖与风险）
2. **分配视角约束**：
   - project-manager 视角：里程碑划分、Backlog 清单、时间估算、容量规划
   - architect 视角：Proposal 间技术依赖、并行空间、风险识别
3. **列出关键假设**（列出方向，不预填具体假设）：规划阶段需验证的核心假设类型（如需求完整性、Velocity 可信度、依赖可识别性），具体假设由产出 agent 在发散过程中识别和验证
4. **识别评审视角缺口**（方向性说明，不做具体分析）：确保评审 agent 覆盖产出 agent 未涉及的视角（如产品完整性、可交付性），具体评审配置在 Step 3 独立评审阶段确定

Plan 阶段由主会话基于上下文直接完成（信息整理与视角分配，不做分析结论），不派遣额外 agent。

> **豁免说明**：本阶段为轻量级任务解构，不产出正式文档，不适用出口标准与评审修正循环。

**完成检查清单**：
- [ ] 切分维度已确认（按职能侧重）
- [ ] 视角约束已分配（project-manager + architect）
- [ ] 关键假设类型已列出（方向性，不预填具体值）
- [ ] 评审视角缺口已识别（方向性说明）

**Step 2：Agent 调度（多 agent 并行产出）**

主会话收集完项目背景后，**并行启动 2 个不同角色的 Agent**，各自侧重不同维度独立产出：

| Agent | 侧重维度 | 产出内容 |
|-------|---------|---------|
| `project-manager` | Backlog 结构、里程碑划分、时间估算、容量规划 | 里程碑划分、Backlog 清单（当前详细/后续粗略）、故事点估算、Velocity 校准、迭代总览、容量预算 |
| `architect` | 依赖关系、风险分析、并行空间评估 | Proposal 间依赖关系、技术风险识别、并行空间评估、关键路径识别、风险缓解策略 |

每个 agent 独立读取：
- 用户提供的项目背景参数
- 输入文档路径（需求文档、架构文档、ADR 等）
- 内容模板路径：`.claude/templates/plan-milestone.md`

主会话整合两个 agent 的产出，形成统一的 `docs/iteration-plan/milestone-plan.md`，写入文件。

**Step 2 写入前检查清单**：
- [ ] 里程碑划分已按技术依赖顺序排列
- [ ] Backlog 清单覆盖所有已知 Feature（当前迭代详细，后续迭代可粗略）
- [ ] 每个 Backlog 项有故事点估算
- [ ] Velocity 已校准（有历史数据时基于历史，无历史数据时给出合理假设并标注）
- [ ] 文档已按 `plan-milestone.md` 模板结构写入
- [ ] 文件已验证存在（`ls`/`wc` 确认）

**Step 3：独立评审（产出后自动执行）**

里程碑计划文件产出后，主会话**立即并行启动**三个评审 Agent 执行**独立评审**：
- **architect-reviewer**：技术风险、关键技术假设、依赖链、并行空间
- **product-manager**：Feature 覆盖完整性、里程碑与产品目标对齐、验收标准、MVP 边界
- **project-manager**：时间估算合理性、点数与 Velocity 匹配、里程碑间均衡、风险评估

> **评审视角独立性说明**：project-manager 虽参与了 Step 2 产出，但在评审阶段视角独立——重点评审 architect 产出的技术内容（依赖关系、风险识别、关键路径），不评审自己产出的计划编排部分；计划编排部分的评审由 product-manager 覆盖。

**评审修正循环**：
1. 主会话汇总三个评审 Agent 的结论
2. 修正阶段 agent 配置（根据修正工作量选择）：
   - 少量修正（＜5 处，单点调整）：1 个 `project-manager` Agent 执行修正
   - 中量修正（5-15 处，涉及多章节或多维度）：**2 个不同角色 Agent 分块并行修正** + 主会话整合
     - `project-manager`：修正时间估算、容量规划、Wave 分组、里程碑划分等计划编排内容
     - `architect`：修正依赖关系、技术风险、关键路径、接口冻结点等技术内容
   - 大量修正（＞15 处，接近重写）：**回退到 Step 2 重新执行多 agent 并行发散**
3. **修正完成后，重新启动独立评审**（复核）
4. **循环终止条件**：所有 CRITICAL 问题已修正，且所有评审 Agent 均给出"通过"或"有条件通过"

5. **评审修正循环上限**：最多 3 次
6. **降级处理规则**：3 次循环后仍有未达标项 → 降级置信度至 0.75-0.79 区间，在文件头部状态标注为 CONDITIONAL（有条件推进），允许进入下一阶段但需在下次迭代前补充修正；或**等待人工决策**
7. HIGH/MEDIUM/LOW 级别问题记录但不阻塞

> 阶段间推进：评审修正循环结束后，向用户汇报结果摘要，由用户确认后进入下一阶段

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正（阻塞性问题必须修正，否则不得推进）
- [ ] 所有 HIGH 问题已评估：接受修正 / 接受延期（需记录理由）/ 拒绝（需说明原因）
- [ ] MEDIUM/LOW 问题记录但不阻塞推进，纳入后续迭代跟踪
- [ ] 三个评审 Agent 均给出"通过"或"有条件通过"
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] 文档版本已更新（在文件头部标注版本号和最后更新时间）

**第 1 阶段置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 里程碑合理性 | 25% | 里程碑划分符合技术依赖顺序和产品目标 |
| Backlog 完整性 | 25% | 覆盖所有已知 Feature，无遗漏关键路径 |
| 时间估算合理性 | 20% | 故事点与团队 Velocity 匹配 |
| 独立评审质量 | 20% | 所有 CRITICAL/HIGH 问题有明确回应 |
| 风险识别充分性 | 10% | 已标注关键风险及缓解策略 |

**进入门槛：≥ 0.85**（里程碑划分为不可逆决策，一旦确定难以回退，门槛 ≥ 0.85）

**低于门槛处理规则**：
- 置信度 0.80-0.84 → 自动再修正一次（执行评审修正循环）
- 置信度 < 0.80 → 等待人工决策，禁止进入第 2 阶段

**自动推进规则**：
- 置信度 ≥ 0.85 且所有评审 Agent 给出"通过"或"有条件通过" → **确认后推进**（向用户汇报结果摘要，建议执行 `/df:plan` 进入第 2 阶段）
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 或存在评审 Agent 给出"不通过" → **等待人工决策**（向用户汇报未达标项，等待用户确认是否降级推进或补充修正）

**下一步**：评审通过后，请执行 `/df:plan` 进入第 2 阶段。

---

### 第 2 阶段：迭代执行计划

**目标**：为当前迭代（如 M1-I1）建立 `iteration-m1-i1.md`

> **一次一迭代**：本次执行只产出当前迭代的执行计划文件。下一迭代在本迭代评审通过后，下次执行 `/df:plan` 时再生成。

**Step 1：迭代上下文收集（强制，不可跳过）**

每个迭代的投入和约束可能不同，开始规划前必须向用户确认本迭代的实际情况。**禁止直接沿用 milestone-plan.md 的默认值**：

| 必收集项 | 说明 | 默认值来源 |
|---------|------|-----------|
| 本迭代可用人力 | 实际参与人数和投入比例 | milestone-plan.md 执行摘要 |
| 本迭代周期 | 是否与默认迭代周期一致 | milestone-plan.md 迭代周期偏好 |
| 上次遗留 | 上一迭代未完成的 proposal（仅 I2 起适用） | 无 |
| 容量调整 | 基于实际 Velocity 校准（仅有历史数据时适用） | milestone-plan.md 容量预算 |
| 本迭代特殊约束 | 外部依赖交付时间、硬性截止日期等 | 无 |

收集方式：列出上述问题并附上 milestone-plan.md 中的默认值，用户可逐项确认或修改。若用户回复"按默认"，则沿用默认值并在迭代概览中标注。

> **豁免说明**：本 Step 为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。

**准入检查清单**：
- [ ] 本迭代可用人力已确认或已标注为沿用默认值
- [ ] 本迭代周期已确认或已标注为沿用默认值
- [ ] 上次遗留已确认（I2 起适用）或标注为无
- [ ] 容量调整已确认或已标注为沿用默认值
- [ ] 本迭代特殊约束已确认或已标注为无

**Step 2：Agent 调度（多 agent 并行产出）**

主会话收集完迭代上下文后，**并行启动 3 个不同角色的 Agent**，各自侧重不同维度独立产出：

| Agent | 侧重维度 | 产出内容 |
|-------|---------|---------|
| `project-manager` | Wave 分组、容量均衡、风险缓冲、执行人分配 | Proposal 选择、Wave 分组、容量分配、缓冲容量计算（≥20%）、高风险任务占比控制、风险应对策略、执行人分配 |
| `architect` | 并行就绪检查、接口契约完整性、组内依赖关系 | 并行就绪检查、接口契约完整性、组内技术依赖关系、技术风险评估 |
| `product-manager` | 优先级排序、验收标准与迭代目标对齐、MVP 边界确认 | 迭代内 proposal 优先级排序、验收标准清晰度检查、MVP 边界与里程碑目标对齐 |

每个 agent 独立读取：
- 用户确认的迭代参数
- milestone-plan.md 路径（Backlog 来源）
- 上一迭代计划路径（如有）
- 内容模板路径：`.claude/templates/plan-iteration.md`

主会话整合三个 agent 的产出，形成统一的 `docs/iteration-plan/iteration-m<x>-i<y>.md`，写入文件。整合规则：
- `project-manager` 的 Wave 分组和容量分配作为计划骨架
- `architect` 的依赖关系作为 Wave 分组校验依据，冲突时以依赖关系为准调整分组
- `product-manager` 的优先级排序作为 proposal 选择依据，冲突时由主会话标记待人工确认项

**Step 2 写入前检查清单**：
- [ ] 迭代内 proposal 已按优先级排序，MVP 边界与里程碑目标对齐
- [ ] Wave 分组反映真实技术依赖关系（以 architect 的依赖分析为准）
- [ ] 总点数 ≤ 迭代预算，缓冲容量 ≥ 20%
- [ ] 高风险任务占比 ≤ 50%，每个高风险任务有风险应对策略
- [ ] 文档已按 `plan-iteration.md` 模板结构写入
- [ ] 文件已验证存在（`ls`/`wc` 确认）

**Step 3：独立评审（产出后自动执行）**

迭代执行计划文件产出后，主会话**立即并行启动**三个评审 Agent 执行**独立评审**：
- **architect-reviewer**：Wave 依赖关系、接口契约完整性、技术风险识别
- **product-manager**：优先级覆盖、验收标准清晰度
- **project-manager**：容量均衡、并行度、缓冲充足、风险应对

> **评审视角独立性说明**：product-manager 和 project-manager 虽参与了 Step 2 产出，但在评审阶段视角独立——各自评审其他 agent 产出的内容，不评审自己产出的部分。product-manager 重点评审 architect 产出的技术内容和 project-manager 产出的计划编排内容；project-manager 重点评审 architect 产出的技术内容和 product-manager 产出的优先级内容。

**评审修正循环**：
1. 主会话汇总三个评审 Agent 的结论
2. 修正阶段 agent 配置（根据修正工作量选择）：
   - 少量修正（＜5 处，单点调整）：1 个 `project-manager` Agent 执行修正
   - 中量修正（5-15 处，涉及多章节或多维度）：**2 个不同角色 Agent 分块并行修正** + 主会话整合
     - `project-manager`：修正 Wave 分组、容量分配、缓冲容量、风险应对等计划编排内容
     - `architect`：修正依赖关系、接口契约、技术风险、并行就绪检查等技术内容
   - 大量修正（＞15 处，接近重写）：**回退到 Step 2 重新执行多 agent 并行发散**
3. **修正完成后，重新启动独立评审**（复核）
4. **循环终止条件**：所有 CRITICAL 问题已修正，且所有评审 Agent 均给出"通过"或"有条件通过"

5. **评审修正循环上限**：最多 3 次
6. **降级处理规则**：3 次循环后仍有未达标项 → 降级置信度至 0.75-0.79 区间，在文件头部状态标注为 CONDITIONAL（有条件推进），允许进入下一阶段但需在下次迭代前补充修正；或**等待人工决策**
7. HIGH/MEDIUM/LOW 级别问题记录但不阻塞

**出口标准**：
- [ ] 所有 CRITICAL 问题已修正（阻塞性问题必须修正，否则不得推进）
- [ ] 所有 HIGH 问题已评估：接受修正 / 接受延期（需记录理由）/ 拒绝（需说明原因）
- [ ] MEDIUM/LOW 问题记录但不阻塞推进，纳入后续迭代跟踪
- [ ] 三个评审 Agent 均给出"通过"或"有条件通过"
  - "通过"：无 CRITICAL/HIGH 问题，或所有问题已修正/已评估
  - "有条件通过"：存在已修正的 CRITICAL 问题，或存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
  - "不通过"：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题
- [ ] Wave 分组正确反映迭代内 proposal 间的依赖关系
- [ ] 总点数 ≤ 迭代预算，🔴 高风险任务 ≤ 50%
- [ ] 文档版本已更新（在文件头部标注版本号和最后更新时间）

**第 2 阶段置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 容量均衡 | 25% | 总点数 ≤ 迭代预算，任务分布合理 |
| Wave 依赖正确性 | 25% | 并行分组反映真实依赖关系 |
| 风险缓冲充足 | 20% | 预留 ≥ 20% 缓冲容量 |
| 独立评审质量 | 20% | 所有 CRITICAL/HIGH 问题有明确回应 |
| 可追溯性 | 10% | 每个 proposal 可追溯到 milestone-plan.md Backlog |

**进入门槛：≥ 0.85**（关键路径编排涉及一致性关键路径，一旦执行难以回退，门槛 ≥ 0.85）

**低于门槛处理规则**：
- 置信度 0.80-0.84 → 自动再修正一次（执行评审修正循环）
- 置信度 < 0.80 → 等待人工决策，禁止执行 `/opsx:new`

**自动推进规则**：
- 置信度 ≥ 0.85 且所有评审 Agent 给出"通过"或"有条件通过" → **确认后推进**（向用户汇报结果摘要，建议执行 `/opsx:new` 启动特性开发）
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 或存在评审 Agent 给出"不通过" → **等待人工决策**（向用户汇报未达标项，等待用户确认是否降级推进或补充修正）

**下一步**：评审通过后，请执行 `/opsx:new` 从 Backlog 中选择第一个 proposal 启动特性开发。

## 维护模式：计划维护与调整

**核心定位**：第 1-2 阶段完成后，迭代计划已进入"可用状态"。维护模式按需调用 `/df:plan` 来滚动更新、修正或调整计划。

**触发原因**：迭代进度偏差 / 需求或范围变更 / Velocity 校准 / 里程碑推进

### 快速通道（小修不走完整阶段）

适用场景：迭代内微调、文档一致性修正、单点数据更新。

> **豁免说明**：本通道属于维护模式单点修正，适用豁免场景。不走完整出口标准和评审修正循环，但修正后仍需至少 1 个 reviewer 快速复核。

流程：
- **迭代内调整**：评估进展 → 识别变更 → 更新 iteration 文件 → 必要时更新 milestone-plan.md Backlog → **1 个对应角色 reviewer 快速复核**（如 project-manager 复核计划调整，architect 复核技术依赖变更）
- **文档一致性修正**：直接读取 → 定位差异 → 确认 → 写入 → **无需额外评审**（纯格式/数据同步修正）

**快速通道出口标准**：
- [ ] 变更范围已明确（仅影响单个文件的单点内容）
- [ ] 修正后至少 1 个 reviewer 快速复核通过（迭代内调整适用）
- [ ] 文档已写入并验证存在

**快速通道问题分级**（轻量级，快速判定）：
- **阻塞**：变更引入新冲突或超出声明范围 → 必须修正
- **可接受**：无冲突且在声明范围内 → 通过

### 回退机制（大变更回退到对应子流程重新执行）

- **里程碑推进**：当前里程碑接近尾声时，在 milestone-plan.md 中将下一个里程碑从粗略描述展开为详细 Backlog + 迭代总览 → 基于实际 Velocity 校准 → **执行完整独立评审（3 个 reviewer）** → 评审修正循环（最多 3 次）→ 评审通过后开始新里程碑的第 2 阶段
  > **不创建新文件**：Backlog 展开是更新 milestone-plan.md，不是创建新文件。
  > 本场景**完整适用第 1 阶段的出口标准和评审修正循环**（含置信度评分、门槛 ≥ 0.85、3 次循环上限、降级处理规则）。
- **重大变更（里程碑目标变化）**：涉及里程碑目标变化的重大变更 → **回退到第 1 阶段**重新开始，后续阶段同步更新
  > 本场景**完整适用第 1 阶段的出口标准和评审修正循环**（含置信度评分、门槛 ≥ 0.85、3 次循环上限、降级处理规则）。

## 与用户协作模式

**三种协作节奏统一使用**：
- **自动推进**：阶段内产出 → 独立评审 → 评审修正循环自动执行，循环结束后汇报结果
- **确认后推进**：阶段/Step 完成后汇报，等待用户确认再进入下一阶段/Step
- **等待人工决策**：信息缺失、方向选择、置信度低于阈值时停止，等待用户决策

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要

**必须用户介入的情况**：项目背景信息缺失 | 迭代上下文需确认 | 置信度低于阈值

## 铁律检查

- **每阶段产出后，主会话立即并行启动三个评审 Agent 执行独立评审**（不等待用户确认），未经评审修正循环通过的计划不得推进执行
- Backlog 中每个 proposal 必须是可独立交付的最小有价值单元
- 关键路径上的 proposal 必须优先安排
- **集成测试必须独立立项**，不可合并到特性 proposal 中
- 单个 proposal 不得超过 4 周（XL 必须拆解）
- 高风险任务占比 ≤ 40-50%
- **Wave 是迭代内部的并行编排单位**，不用于跨迭代编排
- **第 1 阶段 Backlog 只列清单，不分配到具体迭代**（迭代分配在第 2 阶段完成）

## 红旗信号

- 主会话直接编写计划文件内容（应委托 project-manager Agent）
- 跳过里程碑规划直接进入迭代执行计划
- 第 1 阶段 Backlog 中已将 proposal 分配到具体迭代（应在第 2 阶段完成）
- Wave 分组出现在跨迭代维度（Wave 只用于迭代内部 proposal 间并行）
- 所有 proposal 都在 Wave 0（未识别迭代内依赖关系）
- 无独立的集成测试 proposal
- 单个 proposal 超过 4 周未拆解
- 评审未通过就推进到下一阶段
- CRITICAL 问题修正后未重新评审
- 评审记录未遵守模板中 200 字/角色的限制
- 未收集项目背景信息就开始里程碑规划
- 未确认迭代上下文就开始迭代执行计划
