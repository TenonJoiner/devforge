# product/plan — 规划性思考

## 核心理念

**两阶段规划，按顺序执行；每阶段完成前，不展开下一���段。**

两阶段规划顺序：

1. **第 1 阶段**：里程碑规划 + Backlog 清单（milestone-plan.md）
2. **第 2 阶段**：迭代执行计划（iteration-m*-i*.md）— **第 1 阶段完成后才能进入**，一次一迭代

独立评审：每阶段完成后，`architect-reviewer` + `pm-reviewer` + `project-manager-reviewer` 三个 Agent 分别执行**独立评审**，评审通过后进入下一阶段。

**核心原则**：
- **线性推进**：按第 1 阶段 → 第 2 阶段的顺序执行，不跳跃
- **一次一文件**：每次执行只产出一个计划文件
- **出口���准驱动**：每阶段有明确的**出口标准**，达成后**确认后推进**进入下一阶段
- **计划是活的**：评审通过后如需调整，执行滚动更新

### 两阶段分工

- 第 1 阶段：里程碑规划 + Backlog 清单（有什么活要干）
- 第 2 阶段：迭代执行计划（这两周具体怎么干）

**关键规则**：后续里程碑的 Backlog 展开是**更新 milestone-plan.md**，不是创建新文件。一个项目始终只有一个 milestone-plan.md。两阶段各管什么详见 `plan-milestone.md` 模板前言。

## 复杂度判定

本 skill 各阶段的内容类型与基准复杂度判定如下：

| 阶段 | 内容类型 | 基准复杂度 | 判定依据 |
|------|---------|-----------|---------|
| 第 1 阶段：里程碑规划 | 结构化规划 | **高** → 上调至 **极高** | 基于已有需求/架构文档进行里程碑结构化编排；基准为「高」（结构化规划+跨子系统依赖分析），上调属性：跨子系统边界+不可逆决策 |
| 第 2 阶段：迭代执行计划 | 结构化规划 | **高** | 基于里程碑计划分解迭代任务；涉及关键路径编排 + 多角色交叉验证 |

**决策属性上调**（辅助判定）：
- 第 1 阶段基准复杂度为「高」（结构化规划 + 跨子系统依赖分析）
- 上调属性 1：「跨子系统边界」（proposal 依赖分析通常都有），复杂度 +1 级
- 上调属性 2：「不可逆决策」（里程碑划分影响后续全部迭代），复杂度 +1 级
- 综合判定：高 + 跨子系统 + 不可逆 = **极高**
- 第 2 阶段涉及「关键路径」上的 proposal 编排（性能 / 一致性关键路径，复杂度 +1 级），且需要多角色交叉验证（product-manager / architect / project-manager 各自侧重容量均衡、技术依赖、执行可行性），复杂度上调至「高」

**篇幅异常检测**（辅助复核）：
- 高/极高复杂度文档篇幅 <100 行 → 红旗，需复核分析深度是否不足
- 低复杂度文档篇幅 >400 行 → 红旗，需复核是否存在冗余或信息密度过低
- 篇幅与同类文档历史均值偏差 >50% → 提示复核，确认是合理差异还是质量问题

**复杂度与 agent 配置对照**：

| 阶段 | 复杂度 | 并行产出 agent | 独立评审 agent |
|------|--------|---------------|---------------|
| 第 1 阶段 | 极高 | ≥5 个 agent（不同角色，详见阶段文件） | ≥3 个 reviewer（覆盖技术/产品/项目管理） |
| 第 2 阶段 | 高（关键路径编排 + 多角色交叉验证） | ≥3 个 agent（不同角色，详见阶段文件） | ≥3 个 reviewer（architect-reviewer + pm-reviewer + project-manager-reviewer） |
| 维护模式 | 低 | 1 个对应角色 agent | 按需 |

## 启动检测

`/df:plan` 启动时，执行**启动检测**判断当前所处阶段，**只提示唯一下一步动作**：

| 检测结果 | 当前阶段 | 唯一下一步动作 |
|---------|---------|--------------|
| `milestone-plan.md` 不存在 | **第 1 阶段** | 开始里程碑规划 |
| `milestone-plan.md` 存在（含 Backlog），第一个未规划的迭代（如 M1-I1）的 `iteration-m1-i1.md` 不存在 | **第 2 阶段** | 开始 M1-I1 的迭代执行计划（一次只做这一个迭代） |
| 所有当前目标已完成评审 | **下一目标** | 继续执行 `/df:plan`，自动进入下一个迭代的规划 |
| 当前里程碑所有迭代已规划完毕，下一个里程碑 Backlog 尚未展开 | **里程碑推进** | 更新 milestone-plan.md，展开下一个里程碑的详细 Backlog + 迭代总览（滚动更新） |
| 全部迭代均已规划完毕 | **计划完备** | 建议执行 `/opsx:new` 启动特性开发，或滚动更新 |

## 红旗信号

- 主会话直接编写计划文件内容（应委托 project-manager Agent）
- 跳过里程碑规划直接进入迭代执行计划
- 所有 proposal 都在 Wave 0（未识别迭代内依赖关系）
- 评审未通过就推进到下一阶段
- CRITICAL 问题修正后未重新评审
- 评审记录章节篇幅超过 200 字
- 未收集项目背景信息就开始里程碑规划
- 未确认迭代上下文就开始迭代执行计划
- 在无 agent 执行记录的情况下声称某阶段/Step 已完成

## 与用户协作模式

本 skill 采用三种协作节奏（详见 `common.md`）：

| 模式 | 说明 |
|------|------|
| **自动推进** | 阶段内产出→评审→修正自动执行，循环结束后汇报结果 |
| **确认后推进** | 交付件/阶段完成后汇报，等待用户确认再进入下一阶段/Step |
| **等待人工决策** | 置信度低于阈值或信息缺失时停止，等待用户决策 |
