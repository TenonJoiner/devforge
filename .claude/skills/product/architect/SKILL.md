# product/architect — 架构探索与决策

## 概述

架构探索是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续数轮甚至数十轮的思考-研究-对比-完善循环，产出高质量的架构设计。

**核心原则**：
1. **标杆先行**：不研究业界方案，不允许提出自研方案
2. **深度对比**：至少 3 个候选方案，每个方案必须有明确的取舍分析
3. **长时间迭代**：架构文档需要 5-20 轮迭代才能定稿，禁止一次对话定架构
4. **方案 > 代码**：聚焦设计决策和 rationale，不讨论具体代码实现
5. **多 agent 协作**：researcher 负责标杆研究，architect 负责方案发散与决策，architect-reviewer 负责独立质疑验证
6. **模板即法源**：所有正式交付文档的内容结构、必填章节、自检清单以 `.claude/templates/*.md` 为唯一标准

## 多 Agent 协作模式

```
└─────────────────────────────────┬───────────────────────────────────────┘

        ↓                         ↓                         ↓
┌───────────────┐       ┌─────────────────┐       ┌─────────────────┐
```

### Agent 职责边界

| Agent | 职责 | 关键约束 |
|-------|------|----------|
| **researcher** | 标杆研究（客观）<br>• 深入研究标杆产品架构<br>• 精读学术论文<br>• 产出 reference/*.md | 只做客观技术分析<br>不做主观适配决策 |
| **architect** | 架构决策（主观）<br>• 发散候选方案<br>• 五维质量属性评估<br>• 识别 trade-off 并决策<br>• 产出 ADR 和架构文档 | 承担架构设计<br>的主体责任；**写文档前必须先读模板** |
| **architect-reviewer** | 独立质疑（验证）<br>• 强制质疑假设<br>• 检查盲点和偏见<br>• 调整置信度评估<br>• 补充遗漏 trade-off | 不对方案负责<br>只对评审质量负责；**评审前先读模板** |

## 文档写入铁律

**每一轮迭代的产出必须写入项目文件，禁止仅在对话中输出。**

原因：架构探索是长期迭代过程，跨越多次对话。如果产出仅存在于对话中，下次对话将丢失所有中间成果，导致重复劳动或方向偏移。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本轮完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **正式文档写入前，必须确认对应模板已被读取**

## Template-first 内容规范

所有正式交付文档的内容结构、章节深度、自检清单，**不再重复定义于本 skill 中**，统一由以下模板规定：

| 文档类型 | 模板路径 | 说明 |
|----------|----------|------|
| 标杆分析 | `.claude/templates/ref-architecture.md` | reference/*.md |
| ADR | `.claude/templates/arch-adr.md` | adr.md |
| 系统架构总纲 | `.claude/templates/arch-system.md` | design.md |
| 子系统架构 | `.claude/templates/arch-subsystem.md` | `<subsystem>/design.md` |

**Skill 只保留以下流程和深度要求**（不重复模板中的章节清单）：
- 标杆分析：每个技术点必须完成"原理层→设计层→优化层"三层剖析
- 方案发散：至少 3 个候选方案（ADR 维度）、至少 2 个候选+1 个被排除（子系统维度）
- 决策 rationale：必须有量化推导，禁止"矩阵后直接给结论"
- 文档产出：必须先读模板，再按模板结构填充，缺失任何 `mandatory-sections` 视为不达标

## 执行流程（强制多轮迭代）

### 第 1 轮：标杆研究（强制，不可跳过）

**准入条件**：无
**产出文件**：`docs/architecture/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
**模板**：`.claude/templates/ref-architecture.md`

**执行方式**：
1. Master 识别需要研究的标杆列表
2. **主会话读取 `ref-architecture.md` 模板**，确认深度要求
3. 并行启动 researcher，每个研究一个标杆
4. researcher 完成研究后，**主会话将研究内容写入** `docs/architecture/reference/<product>.md`
5. **architect 验收研究深度**，依据模板末尾自检清单逐项确认
6. **不达标时明确反馈 researcher 补充分析**，循环优化直到通过
7. **确认所有文件已落盘**，全部验收通过后，进入下一轮

### 第 2 轮：方案发散（强制，分层展开）

**准入条件**：标杆研究完成并通过 architect 验收

方案发散不是只选一个整体架构风格。必须**从整体到关键技术逐层展开**：

**Step 1：整体架构发散**
- 系统整体架构风格的候选方案对比
- 产出文件：`docs/architecture/decisions/decision-overall.md`
- 与用户确认整体方向偏好

**Step 2：关键技术维度确认 + 并行发散**
- architect 基于整体架构方案和标杆研究，列出需要独立探索的关键技术维度
- 说明每个维度为什么需要独立探索（标杆分歧/影响大/多方案可选）
- **与用户一次性确认**：探索哪些维度、每个维度的问题边界
- 确认后，**并行启动 researcher**：每个关键技术维度分配一个 researcher 做深度调研
- researcher 完成调研后，**主会话将调研内容写入** `docs/architecture/decisions/decision-<维度>.research.md`（可选，内容丰富时独立存档）
- architect 基于 researcher 调研结果，设计≥3个候选方案，产出方案对比矩阵
- **主会话将方案内容写入** `docs/architecture/decisions/decision-<维度>.md`
- **每个文件确认已落盘**

**并行纪律**（强制）：
- ❌ **禁止单个 architect agent 串行产出多个关键技术维度的决策文档**
- ✅ **每个关键技术维度独立启动一个 architect agent**，每个 agent 只聚焦该维度的候选方案设计与 trade-off 分析
- 主会话负责检查各维度间的一致性和关联影响，发现冲突时回退到对应维度重新启动 architect 修正

**典型关键技术维度示例**（根据具体项目调整）：

| 维度 | 探索什么 | 示例候选方案 |
|------|---------|-------------|
| 存储单元设计 | 数据如何组织和寻址 | Chunk-based / Block-based / Slab-based |
| 传输层设计 | 数据如何跨节点传输 | RDMA / TCP / 混合；单边 / 双边 |
| 调度策略 | 如何分配和迁移数据 | 集中式 / 分布式；局部性 / 负载均衡 / 成本优先 |
| 元数据管理 | 索引和位置追踪 | 集中式 / 分布式；哈希 / 范围分片 |
| 淘汰策略 | 多层缓存的替换策略 | LRU / LFU / ARC；多层联动机制 |
| 共享机制 | 跨实例数据复用 | COW / Prefix Tree / Hash 去重 |

```
│  • 调度 Agent、验收产出、写入文件                             │
└──────────────────────────┬──────────────────────────────────┘
                           │
           ┌───────────────┴───────────────┐
           ↓                               ↓
 ┌─────────────────────┐       ┌─────────────────────┐
 │   researcher        │       │   architect       │
 │  （技术深度调研）     │       │  （方案设计决策）     │
 ├─────────────────────┤       ├─────────────────────┤
 │ • 针对特定技术维度   │       │ • 基于调研结果       │
 │   做深度对比调研     │       │   设计候选方案       │
 │ • 收集业界实践数据   │       │ • 分析 trade-off    │

```

#### 验收标准

- [ ] 整体架构探索笔记**已写入文件**
- [ ] 关键技术维度已识别并与用户确认
- [ ] 每个关键技术维度的探索笔记**已写入文件**
- [ ] 每个探索笔记：≥3个候选方案、有参考来源、有优缺点分析、有可量化对比维度

### 第 3 轮：评估收敛与文档定稿（强制，逐维度收敛）

**准入条件**：所有探索笔记已写入文件

**收敛顺序**：先 ADR 决策 → 再系统总纲 → 最后子系统文档

#### Step 1: ADR 收敛（三阶段协作）

```
┌─────────────────────────────────────────────────────────────────┐
│                     Step 1: ADR 收敛                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Phase A: architect 初评                                         │
│  ├── 读取 `.claude/templates/arch-adr.md`                       │
│  ├── 读取 decision-<维度>.md                                    │
│  ├── 执行五维质量属性评估                                        │
│  ├── 判断置信度，识别 trade-off                                 │
│  └── 产出 ADR 初稿 → docs/architecture/adr.md#草稿区            │
│                          │                                      │
│                          ▼                                      │
│  Phase B: architect-reviewer 独立质疑                             │
│  ├── 读取 `.claude/templates/arch-adr.md`                       │
│  ├── 独立阅读材料（不带预设）                                     │
│  ├── 对每个 ADR 提出质疑（按复杂度分级，单一技术 ADR ≥5 个，    │
│  │    演进式复杂 ADR ≥8 个）                                     │
│  ├── 识别隐性假设和遗漏 trade-off                               │
│  ├── 调整置信度评估                                               │
│  └── 产出 Reviewer 质疑摘要（融入 adr.md）                       │
│                          │                                      │
│                          ▼                                      │
│  Phase C: architect 修正定稿                                     │
│  ├── 回应所有质疑（接受/反驳/讨论）                               │
│  ├── 修正置信度评估                                              │
│  ├── 补充遗漏的 trade-off 和风险                                 │
│  └── 更新 ADR 终稿 → docs/architecture/adr.md                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**质疑点数量要求**（不固定"3个"，按复杂度分级）：

| ADR 类型 | 最低质疑点数量 | 质疑分布要求 |
|---------|---------------|-------------|
| 单一技术选型 ADR（如索引机制） | 至少 **5 个** | 至少1个隐性假设、1个 trade-off、1个实现风险、1个量化依据、1个边界场景 |
| 复杂演进式 ADR（如整体架构） | 至少 **8 个** | 在上述基础上，额外覆盖阶段切换条件、技术债务、迁移成本、标杆偏差 |

同时要求：**每个核心决策表格/矩阵都必须有至少 1 个针对性质疑。**

**Reviewer 质疑摘要记录方式**：直接融入 `docs/architecture/adr.md` 对应章节，不写成独立大文档。

**置信度门槛**：
| 置信度 | 可产出文档 |
|--------|-----------|
| < 70% | 仅更新决策文档，**禁止产出总纲和子系统** |
| 70%-80% | 可产出总纲（标注"有条件"），**禁止产出子系统** |
| > 80% | 可产出总纲 + 对应子系统 |

#### Step 2: 系统架构总纲定稿（强制三阶段）

**输入**：ADR 终稿。

**执行流程**：
1. **Phase A：architect 发散** — 读取 `arch-system.md` 模板 → 基于 ADR 决策发散 ≥2 个总纲结构备选，产出 `docs/architecture/design.md` 初稿
2. **Phase B：architect-reviewer 独立质疑** — 读取 `arch-system.md` 模板 → 至少 5 个质疑点，重点检查子系统边界、接口一致性、低置信度 ADR 的预留策略
3. **Phase C：architect 修正定稿** — 回应质疑后产出 `design.md` 终稿

**总纲必须回答的问题**：
- 有哪些子系统？每个子系统的职责边界是什么？
- 子系统之间如何交互？接口契约是什么？
- 端到端数据流和控制流是什么？
- 错误如何传播？故障如何隔离？
- 哪些 ADR 决策是"高置信度纳入"，哪些是"中/低置信度预留"？

**验收标准**：
- [ ] 子系统划分清晰，与 ADR 决策一致
- [ ] 包含模块关系图/数据流图/时序图（ASCII）
- [ ] 接口契约有高层抽象定义
- [ ] 对低/中置信度 ADR 有明确标注和预留策略
- [ ] **architect-reviewer 已审阅并确认无结构性矛盾**

**关键纪律**：**`design.md` 未产出前，禁止产出任何 `<subsystem>/design.md`**

#### Step 3: 子系统架构定稿（强制，多 Agent 协作）

**输入**：ADR 终稿 + 系统架构总纲（`design.md`）。

**双重约束**：
1. **ADR 技术约束**：该子系统涉及的关键技术决策；
2. **总纲接口契约**：`design.md` 中定义的子系统边界、交互协议、数据格式、错误语义。

若发现 ADR 与总纲存在冲突，**优先暂停子系统设计，修正总纲或 ADR 后再继续**。

**重要约束**：
- ❌ **禁止主会话一次性批量产出多个子系统文档**
- ✅ **每个子系统文档必须独立经过"发散→质疑→修正"三阶段**
- 只有对应维度的 ADR 达到"高置信度"(>80%) 时，才启动该子系统的详细架构文档写作

**执行流程（每个 `docs/architecture/<subsystem>/` 下的设计文档独立执行）**：

```
┌─────────────────────────────────────────────────────────────────┐

│                                                                 │
│  Phase A: Architect 子系统方案发散                               │
│  ├── 读取 `.claude/templates/arch-subsystem.md`                 │
│  ├── 基于已收敛的 ADR 和 design.md，识别该子系统 ≥2 个候选实现方案 │
│  ├── 分析各方案与周边子系统的边界、接口、数据流                   │
│  └── 产出子系统架构初稿 → docs/architecture/<subsystem>/design.md │
│                          │                                      │

│  ├── 检查是否混入代码细节、是否缺少设计 rationale                │

│                          ▼                                      │

│  ├── 根据质疑直接修正 design.md 正文                             │
│  ├── 清理代码细节，强化设计 rationale                            │
│  └── 在文档末尾用 3-5 句话记录修正摘要                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Reviewer 对子系统架构的质疑维度**：
- **模板合规性**：是否缺失 `mandatory-sections` 中的章节？自检清单是否通过？
- **边界质疑**：该子系统与周边子系统的职责划分是否清晰？接口是否完备？
- **推导过程质疑**：方案对比是否只是"矩阵后直接给结论"，缺少"约束 → 适配度分析 → 取舍决策"的推导链条？
- **rationale 质疑**：是否回答了"为什么不是其他方案""最优/最差场景是什么""未来脆弱点在哪里"？
- **一致性质疑**：子系统设计与 ADR 决策、总纲接口契约是否存在矛盾？
- **可演进性质疑**：当前方案是否为未来演进埋下了高额重构成本？耦合度是否过高？
- **形式质疑**：是否混入了代码细节？是否缺少动态视图？

#### 产出文件

- 低置信度：更新各 `decision-<维度>.md`（追加评估章节）
- 中置信度：`docs/architecture/adr.md`（ADR 草稿区，标注待验证假设）
- 高置信度：
  - `docs/architecture/adr.md`（正式 ADR）
  - `docs/architecture/design.md`（系统架构总纲）
  - `docs/architecture/<subsystem>/design.md`（子系统架构主文档，**每个子系统独立经过 Step 3 三阶段后产出**）

#### 验收标准

ADR 部分：
- [ ] 整体架构选型已收敛
- [ ] 每个关键技术维度完成五维质量属性评估
- [ ] 关键 trade-off 识别
- [ ] 每个维度有独立的置信度评估（低/中/高）
- [ ] **architect-reviewer 独立质疑已完成**
- [ ] **所有质疑点已回应（接受/反驳/讨论）**
- [ ] **隐性假设和遗漏 trade-off 已补充**
- [ ] 明确下一步（继续探索/实验验证/收敛定稿）
- [ ] **评估结果已写入对应文件**

子系统架构文档部分（每个 `docs/architecture/<subsystem>/design.md` 独立验收）：
- [ ] **该子系统对应维度的 ADR 置信度 > 80%（高）**
- [ ] **文档符合 `.claude/templates/arch-subsystem.md` 的模板要求**
- [ ] **文档经过"发散→质疑→修正"三阶段**
- [ ] 包含 ≥2 个备选实现方案的**推导式对比**（非简单矩阵后直接结论）
- [ ] **包含模块关系图/数据流图/时序图等动态视图（ASCII 形式）**
- [ ] 设计 rationale 清晰（"为什么选这个方案"有论证）
- [ ] 与周边子系统的边界和接口定义清晰
- [ ] **architect-reviewer 对子系统初稿的独立质疑已完成**
- [ ] **至少 5 个子系统层面的质疑点已回应**
- [ ] 文中无具体代码实现（类定义/函数签名/可编译语法）
- [ ] **子系统架构终稿已写入项目文件**

#### 禁止事项

- ❌ **主会话一次性批量产出多个子系统文档**
- ❌ **未经 reviewer 质疑直接定稿子系统架构文档**
- ❌ **在低置信度 ADR 上直接写作高置信度子系统文档**
- ❌ **`design.md` 总纲未产出前直接产子系统文档**

### 第 4 轮：架构维护与扩展（用户驱动）

**核心定位**：前 3 轮完成后，架构文档已进入"可用状态"。第 4 轮不是自动运行的后台任务，而是**用户在后续开发过程中，按需调用 `/ky:arch` 来修正、补充或调整架构设计**。

**触发原因**：
1. **实现验证后发现设计缺陷**（原型测试、workload 分析、硬件兼容性审计等推翻了原假设）
2. **需求或范围变更**（新增 feature domain、裁剪子系统、调整接口边界）
3. **补充新的标杆研究或技术调研**（新技术出现、友商发布新方案、团队对某领域认知加深）
4. **整合子系统实现中发现的不一致**（代码实现与架构文档偏离，需要统一源头）

**交互模式**：

```
用户调用 /ky:arch
    ↓
状态自检（只呈现信息，不替用户做决定）
    ↓
用户明确指定任务
    ↓
执行（支持快速通道和第 1/2/3 轮局部子流程）
```

**快速通道**（小修不走完整轮次）：
- **文档一致性修正**：直接读取相关文档 → 定位差异 → 与用户确认 → 写入修正
- **假设验证状态更新**：直接更新跟踪表和 ADR 对应章节
- **单篇标杆补充**：researcher 产出 → architect 验收 → 落盘，不涉及后续轮次

**大变更回退到局部子流程**：
- 新增关键技术维度 → 进入第 2 轮（该维度发散）→ 第 3 轮（ADR + 总纲更新）
- 推翻已有 ADR 假设 → 进入第 3 轮 Step 1（ADR 修正）→ 回检总纲和子系统
- 新增子系统 → 第 3 轮 Step 2（总纲更新）→ Step 3（新子系统三阶段定稿）

### 禁止 vs 允许的技术表达边界

| ❌ 禁止（实现细节） | ✅ 允许（设计思想层面） |
|---------------------|------------------------|
| C++/Python 具体语法（`struct`、`class`、`std::unordered_map`） | 伪代码描述核心字段和关系 |
| CUDA/OpenMP/MPI 具体 API（`cudaMemcpyAsync`、`ibv_qp`） | 高层抽象"异步DMA搬运"、"RDMA双边操作" |
| 可编译的函数签名和返回值 | 算法流程的文字+ASCII流程图描述 |
| 具体配置文件字段和默认值 | 配置策略的分类和抽象原则 |
| 未经推导直接给出的性能数字 | 基于公式或标杆数据的量级估算 |

### 文档长度与拆分原则（软约束）

- **子系统主文档（design.md）建议长度**：400–800 行
- **模块拆分文档（module-*.md）建议长度**：200–600 行
- **拆分信号**：主文档 >800 行时，将独立模块提取为 module-*.md

## 红旗信号（自查清单）

以下情况出现时，**立即停止，回到研究阶段**：

- 🚩 标杆研究文档 < 2 个
- 🚩 候选方案只有 1 个
- 🚩 方案描述中代码细节 > 设计 rationale
- 🚩 单次对话内完成"研究→方案→文档"
- 🚩 没有明确的 trade-off 分析
- 🚩 用户反馈"思考深度不够"
- 🚩 **Agent 产出内容未写入项目文件**（仅存在于对话中）
- 🚩 **向用户汇报"本轮完成"但文件未落盘**
- 🚩 **主会话一次性批量产出多个子系统架构文档**
- 🚩 **子系统架构文档未经 architect-reviewer 质疑直接定稿**
- 🚩 **`design.md` 总纲未产出前就产子系统文档**
- 🚩 **正式文档产出前未读取对应模板**

## 常见借口与强制回应

| 用户说 | 必须回应 |
|--------|----------|
| "先快速出一版" | 拒绝：快速产出的架构后续返工成本更高，必须完成标杆研究 |
| "这个方案业界都在用" | 质疑：我们的场景和业界是否相同？记录适配成本评估 |
| "就按你说的做" | 警告：当前置信度 X%，建议完成 Y 调研后再定 |

## 与用户协作模式

每次交互：

1. **汇报当前状态**：
   - 已完成哪些标杆研究
   - 当前在哪一轮迭代
   - 当前置信度评估

2. **提出本次重点**：
   - 本轮要解决的问题
   - 需要用户提供的输入
   - 预计产出

3. **确认下一步**：
   - 用户确认后再继续
   - 低置信度时主动建议"是否需要补充调研 X"
