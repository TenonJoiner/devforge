# 通用规范

本文件定义 design skill 的通用规范，所有阶段共享。

---

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

原因：架构设计是长期迭代过程，跨越多次对话。如果产出仅存在于对话中，下次对话将丢失所有中间成果，导致重复劳动或方向偏移。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **写入后必须验证文件存在**（`ls` 确认文件已落盘）

---

## 评审与修正记录规范

写入正式交付文档的评审记录和修正记录必须简略，避免喧宾夺主。

- **评审记录章节 ≤ 200 字**：文档中「评审记录」章节总篇幅不超过 200 字
- **聚焦结论**：仅保留各评审 agent 名称、通过结论（通过/有条件通过）、遗留问题数及简要分类（如"2 MEDIUM：容量估算、风险缓冲"）
- **详细评审意见不写入文档**：具体质疑点、分析过程留存于对话上下文，不写入正式交付文档正文
- **修正记录同样简略**：聚焦"改了什么"，控制在 200 字以内，不展开论证过程

## 通用术语

### 问题分级

**核心原则**：问题级别基于对整个产品系统级的影响判定，而非对单个文档所描述的局部影响判定。

| 级别 | 含义 | 判定基准 | 处理方式 |
|------|------|---------|----------|
| **CRITICAL** | 阻塞性问题，会导致系统方向错误 | **系统级影响** | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响局部质量 | **文档/模块级影响** | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | **优化建议** | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | **非功能性** | 可延期 |

**CRITICAL 判定标准**（必须同时满足 3 条）：
1. 影响整个产品系统的核心决策或整体方向
2. 不修正会导致后续工作推倒重来
3. 涉及不可逆或高成本变更

**HIGH 判定标准**：显著影响局部设计或实现质量，但不改变系统整体方向

**MEDIUM 判定标准**：优化建议，有改善空间，不影响功能正确性

**LOW 判定标准**：格式、命名、注释等非功能性问题

**判定时必须考虑的系统上下文**：
- 该文档在整个产品系统中的位置和重要性
- 该文档所描述的组件/模块的可替换性
- 错误的影响范围（仅限当前模块 vs 影响多个子系统 vs 影响整体架构）
- 修正成本（局部调整 vs 推倒重来）

**常见误判纠正示例**：
1. 子系统设计文档内部结构混乱 → **HIGH**（非 CRITICAL）。理由：该子系统可独立重构，不影响其他子系统
2. Feature Scenario 定义不完整 → **HIGH**（非 CRITICAL）。理由：该 Feature 可后续补充，不影响产品整体方向
3. 核心一致性协议选型错误 → **CRITICAL**。理由：系统级方向性决策，影响多个子系统且难以回退
4. 整体架构分层错误 → **CRITICAL**。理由：影响所有子系统，不修正导致后续工作推倒重来

### 评审结论三态

- **"通过"**：无未处理的 CRITICAL/HIGH 问题，或所有问题已修正/已评估
- **"有条件通过"**：存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
- **"不通过"**：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题

### 修正工作量分级

| 级别 | 范围 | agent 配置 |
|------|------|-----------|
| 少量修正 | < 10 个问题（CRITICAL+HIGH+MEDIUM） | 1 个对应角色 agent |
| 中量修正 | 10-30 个问题 | 2 个 agent 分块并行修正 + 主会话整合 |
| 大量修正 | > 30 个问题 | 评估：分批修正 or 回退重做 |

## 标准评审修正循环

> **独立评审**：一个或多个 Agent 独立产出评审意见，强调多 agent 并行、独立产出，而非一个 agent 产出后另一个复核。
> **评审修正循环**：发现问题 → 回退判断 → 修正 → 复核的完整迭代闭环。
> **出口标准**：用 checkbox 列表明确定义通过条件的结构化门禁，包含结构性 checklist + 量化置信度评分 + 自动推进规则。

1. **独立评审**：并行启动 ≥2 个 reviewer，按模板自检清单提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **回退判断**：根据 CRITICAL 数量和框架性错误判断是否回退重做。**框架性错误**指：整体方向错误、核心假设错误、架构选型根本不适用等

   | 条件 | 决策 |
   |------|------|
   | CRITICAL ≥3 **或** (CRITICAL 1-2 且为框架性错误) | **回退重做** |
   | CRITICAL 1-2 且非框架性错误 | 进入修正 |
   | CRITICAL = 0 | 进入修正 |

3. **修正**：按修正工作量分级选择 agent 配置执行修正
4. **复核**：修正完成后重新启动独立评审
5. **循环上限**：最多 3 次
6. **终止条件**：所有 CRITICAL 问题已修正 + 所有 reviewer 均给出"通过"或"有条件通过"
7. **降级规则**：3 次循环后仍未达标 → 降级置信度并标注"待后续补充"，或等待人工决策

## 主会话职责边界

主会话在多 agent 协作中只承担以下职责，**超出即越界**：

1. **调度**：决定哪个阶段派哪些 agent、什么时机并行
2. **搬运**：将 agent 产出搬运到目标位置（仅搬运，不编辑内容）。**目标位置由 agent 类型决定**：
   - **产出型 agent**（设计文档、决策文档、研究报告等）→ 写入对应文件
   - **交互型 agent**（交互式调研、提问、用户反馈、状态汇报等）→ **完整搬运到主对话正文**
   > **关键事实**：用户**看不到** Agent 工具的返回内容，只能看到主会话输出的文本。如果主会话不把交互型 agent 的产出搬运到对话正文，用户侧将完全看不到 agent 的工作成果，等同于该轮 agent 调用不存在。
3. **整合**：当多个 agent 产出需要合并时，选择/拼接/去重（不改写核心结论）
4. **门禁**：检查出口标准、置信度阈值、agent 执行记录完整性

**主会话禁止行为**：
- 直接产出需要深度分析的文档内容
- 对 agent 产出进行实质性内容修改后写入（整合仅限选择、拼接、去重；格式统一不影响语义）
- 在没有 agent 执行记录的情况下声称某阶段已完成
- 用一个 agent "内部发散多个方案"替代多个 agent 独立产出
- **用"等用户回答""请用户决策"等简短摘要替代交互型 agent 的实际产出**（问题列表、选项、反馈点必须完整呈现）
- 对交互型 agent 提问做语义重排或裁剪后呈现（格式统一、添加澄清前缀不算改写）

## 多轮交互的轮次编号规范

交互式调研、产品定位、项目背景收集等**多轮交互场景**，问题编号**必须包含轮次标识**，禁止跨轮重复使用裸编号。

- ✅ **正确**：`R1-Q1`、`R1-Q2`（第 1 轮）；`R2-Q1`、`R2-Q2`（第 2 轮）
- ❌ **错误**：每轮都用 `Q1`、`Q2`（跨轮编号重复，用户难以区分本轮新问题与上一轮已答问题）

**配套规则**：
- 主会话搬运 agent 提问到对话正文时，**必须保留 / 显式标注轮次编号**
- 如果 agent 输出仍使用裸编号（Q1/Q2），主会话搬运时**统一替换为 R{n}-Q{m} 格式**（这属于格式调整，不影响语义）
- 交互式 agent 的 prompt 中应**显式要求 agent 使用轮次编号**，避免每次都需要主会话替换
