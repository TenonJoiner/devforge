# devforge-design — 架构探索与决策

## 概述

架构探索是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续多个阶段的思考-研究-发散-评审-收敛循环，产出高质量的架构设计文档。

> **内容规范强牵引**：架构交付件的所有内容结构、章节要求、自检清单**以 `.claude/templates/*.md` 为唯一准绳**。本 skill 只负责定义**努力程度、写作风格、质量门槛和流程控制**，不再重复模板中已有的内容要求。

**核心原则**：
1. **模板牵引**：所有产出必须严格遵循对应模板，写前先读模板是铁律
2. **标杆先行**：不研究业界同类产品的架构，不允许提出自研方案
3. **深度对比**：每个维度至少 ≥2 个候选方案，每个方案必须有明确的 Trade-off 分析
4. **方案 > 代码**：聚焦设计决策和 rationale，不讨论具体代码实现
5. **长时间迭代**：架构文档需要 5-20 个阶段才能定稿，禁止一次对话定架构

## 通用规范

### 通用术语

#### 问题分级

问题级别基于对**整个产品系统级**的影响判定，不按文档局部影响放大。

| 级别 | 含义 | 判定基准 | 处理方式 |
|------|------|---------|----------|
| **CRITICAL** | 阻塞性问题，会导致系统方向错误 | **系统级影响**（同时满足：影响核心决策/方向 + 不修正会推倒重来 + 不可逆或高成本） | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响局部质量 | **文档/模块级影响**，不改变系统整体方向 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | **优化建议**，不影响功能正确性 | 评估后决定 |
| **LOW** | 轻微问题 | 格式、命名、注释等非功能性 | 可延期 |

> 常见误判：子系统内部结构问题、单 ADR 论证不足 → 归 HIGH（局部问题，可独立修复）；核心一致性协议选型、整体架构分层错误 → 归 CRITICAL（系统级方向性决策）。reviewer agent 在评审 prompt 中需明确这个判定边界。

#### 评审结论

Reviewer 产出：
- 问题列表（每个问题标注级别 CRITICAL/HIGH/MEDIUM/LOW 和所属维度）
- 缺陷密度（自动计算：所有问题分数之和 / 评估对象数）

系统自动判断：
- 基于缺陷密度和 CRITICAL 问题数量自动判断是否通过
- 不需要 reviewer 主观给出"通过/不通过"结论

### 标准评审修正循环

各阶段遵循统一的评审修正循环。具体门槛值（缺陷密度阈值、评估对象数计算方式）见各阶段规则。

1. **独立评审**：并行启动 ≥2 个 reviewer，按模板自检清单提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **计算缺陷密度**：所有问题分数之和 / 评估对象数

   问题分值：**CRITICAL=10分, HIGH=3分, MEDIUM=1分, LOW=0.1分**

3. **判断退出条件**（两步判定）：
   - **先判断 CRITICAL 问题**（一票否决）：有 CRITICAL → 进入修正循环；无 CRITICAL → 继续
   - **再判断缺陷密度**：达标 → 成功退出；超标 → 进入修正循环
4. **修正策略**（此为唯一定义，各阶段文件不得重复）

   基于问题数量决定 agent 配置：
   - < 10 处：1 个 agent 读取文件 + review 文件后 Edit 修正
   - 10-30 处：2 个 agent 按产出对象/维度分块并行修正，各自 Edit 对应部分
   - \> 30 处：按各阶段规则判断是否回退到对应步骤重新执行多 agent 并行发散（回退规则见各阶段，各阶段不重复定义此处 3 档策略本身）
   - 分工原则：问题集中在某个产出对象 → 针对该对象；问题分散 → 按产出对象并行；问题集中在某个维度 → 按维度分工
   - 修正 agent 直接 Edit 文件，主会话只确认修正完成状态，不中转修正内容

5. **复核**：修正完成后回到步骤 1（独立评审）
6. **循环上限**：最多 3 次
7. **降级规则**：3 次循环后仍未达标 → 进入人工决策（见步骤 8）
8. **人工决策返回路径**：用户对 3 次循环仍未达标的产出可选择三条路径之一，主会话据此恢复流程：
   - **接受现状**：放行该交付件，写入 `**评审状态**: ⚠️ ACCEPTED-WITH-DEBT（缺陷密度 X，残留 N 个 HIGH）` 标记，记录残留问题到「评审记录」章节，进入下一阶段
   - **重启循环**：根据用户提示（如调整视角切分、扩大 reviewer 角色、补充上下文）重新发起独立评审，循环计数从 1 开始
   - **回退重做**：按各阶段定义的「>30 处回退目标」回退到对应步骤重新执行多 agent 并行产出
9. **评审状态标记契约**（机器可解析，启动检测依此判定阶段过渡）：

   ```markdown
   **评审状态**: ✅ PASS（缺陷密度 X/评估对象，无 CRITICAL，循环 N/3）
   ```

   - 仅当无 CRITICAL + 缺陷密度达标时写入 `✅ PASS`；评审中或未通过写入 `❌ FAIL` 或不写
   - 接受带债放行写入 `⚠️ ACCEPTED-WITH-DEBT`（不视为 PASS，但允许阶段过渡）
   - 启动检测搜索 `**评审状态**:` 行；无 PASS / ACCEPTED-WITH-DEBT 标记 = 该交付件未完成

**自动决策规则**：CRITICAL 必须修正 → HIGH 默认修正 → MEDIUM 评估后决定 → LOW 可延期

### 缺陷密度门槛标定依据

> 各阶段门槛的设定逻辑，供主会话和 reviewer 参考。门槛随阶段对"不可逆性"的敏感度升级：

| 阶段 | 门槛 | 估算逻辑 |
|------|------|---------|
| 第 1 阶段（标杆研究） | 2.0 分/标杆 | 研究报告信息不完整（outside-in），适度容忍在研 gap |
| 第 2 阶段步骤 1（整体架构发散） | 2.0 分/方案 | 极高复杂度但发散阶段仍有探索性 |
| 第 2 阶段步骤 3（维度方案发散） | 1.5 分/方案 | 维度边界已定义，方案应更收敛 |
| 第 3 阶段步骤 1（ADR 收敛） | **0.5 分/ADR** | 涉及不可逆决策，最严门槛。N=10 累计 ≤ 5 分 |
| 第 3 阶段步骤 2（系统总纲） | 1.5 分/子系统 | 已基于 ADR 收敛，重点验证一致性 |
| 第 3 阶段步骤 3（子系统设计） | 1.5 分/章节 | 章节包含功能流/故障/接口/运维等 |
| 第 4 阶段（维护快速通道） | 3.0 分/修正点 | 范围有限（单文件单章节），N=1 时 ≤ 3 分 |

### 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

- **原始 agent**（researcher / architect）：将完整产出写入 `-draft-` 标记的中间文件，向主会话返回轻量摘要（≤5 行：关键数字、覆盖率、异常点）
- **合并 agent**（同类型新实例）：读取 draft 文件，执行合并/去重，写入最终文件，返回轻量摘要
- **Reviewer agent**：将评审意见追加到 `-review.md` 文件，返回数字摘要（issues 数、缺陷密度、CRITICAL 数）
- 所有文件写入后用 `ls` 验证存在，完成后才能汇报"本阶段完成"

**文件命名规范**：
- 所有非最终产出统一用 `-draft-` 标记，如 `decision-overall-draft-{view}.md`、`reference/<product>-draft-{view}.md`
- 评审意见统一追加到单 `-review.md` 文件，如 `decision-overall-review.md`、`reference/<product>-review.md`
- 最终文件不含 `-draft-` 和 `-review` 标记
- **子系统目录命名**：使用 kebab-case 英文标识，如 `docs/architecture/storage-engine/`，禁止使用中文字符
- 中文子系统名称仅用于文档内部标题和阅读显示

**例外**：步骤内部的工作指令（如视角切分理由说明）属于执行中间件，可按对应阶段规则豁免落盘。

### 主会话职责边界

**主会话只负责**（必须由主会话亲自执行）：
- 调度 agent（决定派遣哪些 agent、视角切分、prompt 内容、并发节奏）
- 状态跟踪（记录各 agent 完成状态、文件路径、轻量摘要数字）
- 检查出口标准（文件存在性、✅ PASS 标记、数量门槛）
- 缺陷密度判定（从 reviewer 返回的数字摘要中读取，做通过/修正/回退决策）
- 与用户交互（确认、汇报、等待人工决策）

**已下放给 agent 的职责**（主会话不再执行）：
- 文件写入：agent 自行 Write/Edit 到 draft/review/最终文件
- 内容整合：由**同类型新 agent 实例**执行合并去重（researcher 合并 researcher 的 draft，architect 合并 architect 的 draft）
- 评审记录：reviewer 追加到 `-review.md`，合并 agent 将 PASS 标记写入最终文件

**典型场景对照**（消除"整合 vs 深度分析"的边界歧义）：

| 场景 | 正确做法 | 错误做法 |
|------|---------|---------|
| researcher 产出标杆报告 | researcher 自行 Write 到 `reference/<product>-draft-{view}.md`，返回 ≤5 行摘要 | 返回完整报告给主会话中转写入 |
| 多路 architect 产出后合并 | 派**新 architect agent**读取 draft，合并后 Write 最终文件 | 主会话读完整内容做合并并写入 |
| reviewer 提出修正意见 | 派 architect agent 读取文件 + review 文件后 Edit 修正 | 主会话直接改写文件内容 |

**关键判定**：原始产出 → 原始 agent 写 `-draft-`；合并去重 → 同类型新 agent 实例（保持领域一致性）；评审意见 → reviewer 追加到 `-review.md`；最终文件 → 合并 agent 写。

**边界警戒**（这些行为会破坏 agent 协作的质量保障机制）：
- 主会话读取 agent 完整产出——会撑爆上下文窗口，只取 ≤5 行轻量摘要
- 主会话实质性修改 agent 产出的核心内容——agent 产出的价值在于独立视角，主会话修改会引入偏见
- 无 agent 执行记录就声称完成——违反可追溯性
- 单 agent 内部发散替代多 agent 独立产出——多 agent 的价值在于视角独立性，"内部发散"无法复现
- 合并 agent 与原始 agent 类型不一致（如用 researcher 合并 architect 的 draft）——领域知识不一致会导致合并失真
- 委托 plan agent 生成调度方案——调度判断无中间产物、无产品依赖，由主会话直接决策更准确
- 决策树式"伪动态"——只预定义下游真正依赖的契约，其余由主会话根据目标动态生成

### Agent 并发控制

为防止同时启动过多 agent 触发模型服务端限流，主会话在派遣 agent 时必须遵守以下并发限制。

**核心参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `agent.max_concurrent` | **5** | 同时运行的 agent 上限。单一 `Agent` 工具调用计为 1 个并发单元 |

**滑动窗口策略**（细粒度补位，非批次等待）：

1. 主会话统计当前步骤需要启动的 agent 总数 N，维护一个待启动队列
2. **初始启动**：取队列前 `agent.max_concurrent` 个 agent，全部使用 `run_in_background` 并行启动
3. **补位规则**：每当任意一个 agent 完成（收到完成通知），**立即**从队列中取出下一个 agent 启动，始终保持运行中的 agent 数 = min(剩余待启动, `agent.max_concurrent`)
4. **终止条件**：队列为空且所有已启动 agent 均已完成
5. 有依赖关系的 agent 不能同时存在于队列中，必须等上游 agent 完成后才能入队

**执行约束**：
- 各阶段文件中的"并行启动"指令，均受此并发上限约束
- 主会话在派遣前必须统计 agent 总数，若超限则明确写出滑动窗口方案（窗口大小、总 agent 数、预计启动节奏）
- 禁止以"并发限制"为由将本应多 agent 独立产出的步骤压缩为单 agent 执行

### 评审纪要写作约束

> 此节是写作风格约束，**不计入评审判定项**。状态标记契约见上方「标准评审修正循环」步骤 9。

独立评审完成后，评审结果按以下方式处理：

- **Reviewer agent**：将逐条评审意见追加到 `-review.md` 文件（如 `reference/<product>-review.md`、`decision-overall-review.md`）
- **合并 agent**：评审通过后，将「评审记录」章节写入最终文件末尾
- **主会话**：只读取 reviewer 返回的数字摘要（issues 数、缺陷密度、CRITICAL 数）做通过/修正/回退判定

**写作要求**：
- 写入评审记录前先读对应模板（`ref-architecture.md` / `arch-system.md` / `arch-adr.md` / `arch-subsystem.md`）的「评审记录」示例，模板定义了表格列、日期格式等具体要求
- 日期格式精确到分钟，如 `2026-05-09 14:30`
- 评审记录表格的「缺陷密度」列每一轮（含修正后的复核轮次）都必须有数值，不可留空
- 评审记录章节 ≤ 200 字
- 详细评审意见（逐条分析）留存对话上下文，不写入正式文档

### 视角切分原则

多 agent 并行发散时，每个 agent 从独立、互补的视角产出。视角由主会话基于**本次产品的具体上下文**动态生成，在派遣 prompt 中说明切分理由。

**视角质量标准**：互补（不重叠）、独立（任一视角缺失会导致分析盲区）、有明确命名和切分理由。

**反面约束**：不得套用固定视角清单（如对所有产品都用"数据流/状态/性能/故障/演进"五视角）。视角服务于产品决策——不同产品的决策关注点不同，视角应反映这种差异。各阶段文件引用本原则，不再重复定义。

### 多轮交互的轮次编号规范

交互式调研、产品定位、项目背景收集等**多轮交互场景**，问题编号**必须包含轮次标识**，禁止跨轮重复使用裸编号。

- ✅ **正确**：`R1-Q1`、`R1-Q2`（第 1 轮）；`R2-Q1`、`R2-Q2`（第 2 轮）
- ❌ **错误**：每轮都用 `Q1`、`Q2`（跨轮编号重复，用户难以区分本轮新问题与上一轮已答问题）

**配套规则**：
- 主会话搬运 agent 提问到对话正文时，**必须保留 / 显式标注轮次编号**
- 如果 agent 输出仍使用裸编号（Q1/Q2），主会话搬运时**统一替换为 R{n}-Q{m} 格式**（这属于格式调整，不影响语义）
- 交互式 agent 的 prompt 中应**显式要求 agent 使用轮次编号**，避免每次都需要主会话替换

## 复杂度判定

| 阶段 | 内容类型 | 复杂度 | agent 配置 |
|------|---------|-------|-----------|
| 第 0 阶段 | 交互式信息收集 | 低 | 1 个 architect（交互式，豁免评审循环） |
| 第 1 阶段 | 标杆研究（架构调研报告） | 高 | 每个标杆 ≥2 个 researcher + 整体 ≥2 个 reviewer |
| 第 2 阶段 步骤 1 | 整体架构发散（系统级方案） | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 2 阶段 步骤 3 | 维度方案发散 | 高 | 每维度 ≥3 个 architect + ≥2 个 reviewer |
| 第 3 阶段 步骤 1 | ADR 收敛（不可逆决策） | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 3 阶段 步骤 2 | 系统架构总纲定稿 | **极高** | 阶段总 architect ≥5 + ≥3 个 reviewer |
| 第 3 阶段 步骤 3 | 子系统架构定稿 | 高 | 每子系统 ≥3 个 architect + ≥2 个 reviewer |
| 第 4 阶段 | 维护模式快速通道 | 低 | 1 个 architect-reviewer 快速复核 |

**说明**：
- 极高/高复杂度的 agent 数下限是**阶段或步骤的总 agent 数**，不要求每个子单元独立达标
- 视角切分由主会话基于产品上下文动态决定，禁止套用固定视角清单
- **决策属性上调**：涉及「跨子系统边界 / 不可逆决策 / 性能或一致性关键路径 / 无现成方案 / 安全合规敏感」的文档，复杂度在基准上 +1 级（不超过"极高"）

## 启动检测

**核心原则**：状态判定以文件系统实时扫描（`ls`/`wc`/`Read`）为准。文件存在性是必要条件，**`✅ PASS` / `⚠️ ACCEPTED-WITH-DEBT` 评审状态标记是阶段过渡的充分条件**。

**检测命令**：
```bash
ls -la .claude/domain-config.yaml
ls -la docs/architecture/
ls -la docs/architecture/reference/
ls -la docs/architecture/decisions/
# 若 design.md 已定稿（含 PASS），扫描子系统目录
# ls -la docs/architecture/<subsystem-en>/
```

**状态机**（按优先级从上到下匹配，命中即停）：

| # | 当前文件状态 | 下一步动作 |
|---|------------|-----------|
| 1 | `domain-config.yaml` 不存在 或 `design.md` 不存在 | → 第 0 阶段 |
| 2 | `reference/` 下标杆文件 < 2 个 | → 第 1 阶段步骤 1-2 |

## 执行流程

> 主会话进入每个阶段时，必须先读取对应的详细规则文件（`0X-*.md`）。

### 第 0 阶段：前置调研与定位确认

- **准入**：无
- **产出**：`.claude/domain-config.yaml` + `docs/architecture/design.md`（初稿/WIP）
- **规则**：`00-research-prep.md`

### 第 1 阶段：标杆研究

- **准入**：第 0 阶段完成（design.md 初稿已落盘 + domain-config.yaml 已生成）
- **产出**：`docs/architecture/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
- **规则**：`01-benchmarking.md`

### 第 2 阶段：方案发散

- **准入**：第 1 阶段所有标杆文件通过评审（无 CRITICAL + 缺陷密度达标）
- **产出**：`docs/architecture/decisions/decision-overall.md` + `docs/architecture/decisions/decision-<维度>.md`
- **规则**：`02-divergence.md`

### 第 3 阶段：评估收敛与文档定稿

- **准入**：第 2 阶段所有决策文档通过评审（无 CRITICAL + 缺陷密度达标）
- **产出**：`docs/architecture/adr.md` + `docs/architecture/design.md`（定稿）+ `docs/architecture/<subsystem-en>/design.md`
- **规则**：`03-convergence.md`

### 第 4 阶段：架构维护与扩展

- **准入**：第 3 阶段所有架构文档通过评审（无 CRITICAL + 缺陷密度达标）
- **产出**：按需修正现有文档
- **规则**：`04-maintenance.md`

## 红旗信号

- 未做前置调研就进入标杆研究
- 标杆研究文档 < 2 个
- 候选方案只有 1 个（每维度 ≥2 个备选方案）
- 方案描述中代码细节 > 设计 rationale
- 单次对话内完成"研究→方案→文档"
- Agent 产出内容未写入项目文件
- 架构文档无评审记录即直接定稿
- design.md 总纲未定稿前就产子系统文档
- 高/极高复杂度文档篇幅 < 100 行（可能分析深度不足）
- 低复杂度文档篇幅 > 400 行（可能存在冗余）
- 主会话直接产出原本应由 agent 产出的内容（如自己写架构方案、自己整合 ADR）

- 阶段 N 结束汇报中预演或提前执行了阶段 N+1 的步骤

## 与用户协作模式

**核心边界**：自动推进与人工确认的切换点按**交付件是否完成闭环**划分，不按"阶段内/阶段间"简单切分。

**三种协作节奏**：
1. **自动推进**：交付件内部，产出→评审→修正循环全自动闭环，不中途询问用户
2. **确认后推进**：交付件闭环后，向用户汇报产出概况（产出文件、缺陷密度、关键决策点），等待确认后再进入下一交付件
3. **等待人工决策**：缺陷密度超过门槛、CRITICAL 未修正、信息缺失、或需用户做不可逆决策时停止

**阶段衔接规则**（确认后推进的具体约束）：

阶段 N 结束汇报时：
- **必须**：汇报阶段 N 的产出摘要（仅当前阶段的交付件成果）
- **必须**：告知下一阶段名称（如"下一步：进入第 1 阶段（标杆研究）"），等待用户确认推进
- **禁止**：预演下一阶段的具体步骤、工具、候选对象或中间产物
- **禁止**：提前执行下一阶段的任何步骤（包括搜索、筛选、候选列举）
- **禁止**：将下一阶段才应让用户确认的内容提前到当前阶段汇报中展示

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要
