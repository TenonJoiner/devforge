# devforge-design — 架构探索与决策

## 概述

架构探索是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 通过持续多个阶段的思考-研究-对比-完善循环，产出高质量的架构设计。

**核心原则**：
1. **标杆先行**：不研究业界方案，不允许提出自研方案
2. **深度对比**：至少 3 个候选方案，每个方案必须有明确的取舍分析
3. **长时间迭代**：架构文档需要 5-20 个阶段才能定稿，禁止一次对话定架构
4. **方案 > 代码**：聚焦设计决策和 rationale，不讨论具体代码实现
5. **模板即法源**：所有正式交付文档的内容结构、必填章节、自检清单以 `.claude/templates/*.md` 为唯一标准

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

详细规则见 `common.md`。

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **写入后必须验证文件存在**（`ls` 确认文件已落盘）

## 复杂度判定

> **判定前必须读取对应 Template**：下表中各阶段的复杂度是基于当前项目 template 的内容要求评估的。主会话在判定复杂度前，必须先读取对应阶段的 template，了解其章节结构和内容要求。若更换项目 template，复杂度可能随之变化。

| 阶段 | 内容类型 | 基准复杂度 | agent 配置 |
|------|---------|-----------|-----------|
| 第 0 阶段 | 交互式信息收集 | 低 | 1 个 architect |
| 第 1 阶段 | 标杆研究 | 中等 | 每个标杆 2 个 researcher + 2 个 reviewer |
| 第 2 阶段 | 方案发散 | 步骤 1: 极高（系统级架构）; 步骤 4: 高（按维度） | 步骤 1: ≥5 个 architect; 步骤 4: 每个维度 ≥3 个 architect |
| 第 3 阶段 | 评估收敛 | 步骤 1 和 步骤 2: 极高（ADR/系统总纲）; 步骤 3: 高（子系统） | 步骤 1 和 步骤 2: ≥5 个 architect; 步骤 3: 每个子系统 ≥3 个 architect |
| 第 4 阶段 | 维护模式 | 低 | 1 个对应角色 agent |

**决策属性上调**：
- 涉及「跨子系统边界」→ 复杂度 +1 级
- 涉及「不可逆决策」→ 复杂度 +1 级
- 涉及「性能 / 一致性关键路径」→ 复杂度 +1 级
- 「无现成方案」（需原创设计）→ 复杂度 +1 级
- 涉及「安全 / 合规敏感」→ 复杂度 +1 级

**上调法边界**：
- 基准为"极高"的文档，即使命中上调属性，复杂度仍保持"极高"（不存在超极高级别），上调属性仅作为对极高复杂度的佐证
- 上调结果不超过"极高"级别

**上调法示例验证**：
- 80 行 ADR（如一致性协议选型）——基准极高，即使短小仍为极高 → 正确
- 300 行调研报告（如含跨子系统+不可逆分析）——基准中等，命中上调属性 → 中等→高 → 正确

## 启动检测

**核心原则**：状态汇报必须以 `ls`/`wc`/`Read` 扫描文件系统的**实时结果**为准。

**检测命令**：
```bash
ls -la docs/architecture/
ls -la docs/architecture/decisions/
ls -la docs/architecture/reference/
```

**状态判定**：
- design.md 不存在 → 第 0 阶段
- design.md 存在但 reference/ 下无标杆研究 → 第 1 阶段
- reference/ 有文件但 decisions/ 下无决策文档 → 第 2 阶段
- decisions/ 有文件但 design.md 未定稿 → 第 3 阶段
- 全部文档已定稿 → 第 4 阶段（维护模式）

**行为规则**：
- 主流程未完成时，根据文档状态直接判定并提示唯一的下一步动作，不罗列多个选项让用户选择
- 主流程已完成后（维护模式），允许给出多个改进方向供用户选择（如新增/完善/重构）
- **禁止跳过启动检测直接假设从零开始**

> 主会话进入本阶段时，必须先读取 `01-benchmarking.md` 获取详细执行规则。

**准入条件**：第 1 阶段完成，标杆研究通过评审（≥ 0.80）
**产出文件**：`docs/architecture/decisions/decision-overall.md` + `docs/architecture/decisions/decision-<维度>.md`
**详细规则**：📖 读取 `02-divergence.md`

> 主会话进入本阶段时，必须先读取 `02-divergence.md` 获取详细执行规则。

### 第 3 阶段：评估收敛与文档定稿

**准入条件**：第 2 阶段完成，所有决策文档通过评审（≥ 0.80）

**详细规则**：📖 读取 `03-convergence.md`

> 主会话进入本阶段时，必须先读取 `04-maintenance.md` 获取详细执行规则。

## 红旗信号

- 未做前置调研就进入标杆研究
- 标杆研究文档 < 2 个
- 候选方案只有 1 个
- 方案描述中代码细节 > 设计 rationale
- 单次对话内完成"研究→方案→文档"
- Agent 产出内容未写入项目文件
- 架构文档未经独立评审直接定稿
- design.md 总纲未产出前就产子系统文档
- 高/极高复杂度文档篇幅 < 100 行（可能分析深度不足，需检查是否遗漏关键决策依据）
- 低复杂度文档篇幅 > 400 行（可能存在冗余或信息密度过低）
- 篇幅与同类文档历史均值偏差 > 50%（需复核确认合理差异）

## 与用户协作模式

**核心边界**：自动推进与人工确认的切换点按**交付件是否完成闭环**划分，不按"阶段内/阶段间"简单切分。

**三种协作节奏**：
1. **自动推进**：交付件内部，产出→评审→修正循环全自动闭环，不中途询问用户
2. **确认后推进**：交付件闭环后，向用户汇报产出概况，等待确认后再进入下一交付件
3. **等待人工决策**：置信度低于当前阶段门槛（各阶段见对应子阶段文件中的置信度评分表）、信息缺失、或需用户做不可逆决策时停止

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要
