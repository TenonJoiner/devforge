# 第 3 阶段：评估收敛与文档定稿

**准入条件**：第 2 阶段完成，所有决策文档通过评审（≥ 0.80）
**产出文件**：`docs/architecture/adr.md` + `docs/architecture/design.md`（定稿）+ `docs/architecture/<subsystem>/design.md`

---

## 核心流程

第 3 阶段包含 3 个 步骤，按顺序执行：

1. **步骤 1**：ADR 收敛（≥5 个 architect 并行）
2. **步骤 2**：系统架构总纲定稿（≥5 个 architect 并行）
3. **步骤 3**：子系统架构定稿（每个子系统 ≥3 个 architect 并行）

---

## 步骤 1：ADR 收敛

> **Plan 阶段内联**：第 3 阶段为极高复杂度（ADR 收敛），必须先解构再发散。

**任务解构**（Plan agent 工作流程）：
1. 派遣 **1 个 architect agent** 读取 `.claude/templates/arch-adr.md`
2. 分析系统上下文（第 2 阶段决策文档、系统总纲当前状态、各维度分歧点）
3. 决定 agent 数量（基于复杂度与 template 要求，极高复杂度 ≥5）
4. 设计切分维度（按决策域/按子系统/按分析类型）
5. 产出调度方案，包含三个完整部分：
   - (1) 并行产出 agent 配置（角色 + 视角约束 + 负责维度 + 输入材料）
   - (2) 评审 agent 配置（基于盲区分析，识别产出 agent 未覆盖的视角，配置对应的 reviewer）
   - (3) 整合策略（合并规则 + 冲突处理）
6. 产出以结构化文本形式在对话中输出

**Plan 禁止事项**：
- 不产出深度分析内容
- 不预设最终结论或倾向性方案

**主会话职责**：原样执行 architect 的调度方案，不做修改。

**ADR 发散**：

**产出文件**：`docs/architecture/adr.md`

1. **并行启动 ≥5 个 architect agent**，各自从不同视角独立收敛 ADR
2. 每个 architect 基于第 2 阶段的决策文档，提炼关键架构决策
3. 主会话整合为统一的 ADR 索引
4. 写入 `docs/architecture/adr.md`

### 评审修正循环

按 `common.md` 标准评审修正循环执行，本 步骤 特有配置：

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（ADR 索引，记录所有关键架构决策及其依据）、重要性（ADR 是架构决策的最终记录，影响所有子系统的实现方向）、可替换性（修正成本评估，部分 ADR 涉及不可逆决策）。

- **独立评审**：基于 Plan agent 识别的盲区，并行启动 ≥3 个 reviewer（极高复杂度下限），角色互不相同：
  - 评审视角差异化：至少覆盖技术可行性、一致性、完整性三个维度
  - 质疑点数量要求：≥5 个（ADR 为极高复杂度）
  - **评审配置推导规则**：每个 Plan agent 识别的视角盲区必须有对应的 reviewer 覆盖
  
- **修正 agent 配置**（按 `common.md` 修正工作量分级）：
  - 少量修正：1 个 architect
  - 中量修正：2 个 architect 分块并行修正 + 主会话整合
  - 大量修正：评估：分批修正 or 回退到本步骤开头重新执行 ≥5 个 architect 并行发散

**步骤 1 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| ADR 完整性 | 30% | 所有关键决策均已记录 |
| 决策 rationale 清晰度 | 30% | 每个 ADR 有明确的"为什么" |
| 与决策文档一致性 | 40% | ADR 与第 2 阶段决策文档无冲突 |

**进入门槛：≥ 0.85**

## 步骤 2：系统架构总纲定稿

> **Plan 阶段内联**：步骤 2 为极高复杂度（系统级架构设计定稿），必须先解构再发散。

**产出文件**：`docs/architecture/design.md`（定稿）

**任务解构**（Plan agent 工作流程）：
1. 派遣 **1 个 architect agent** 读取 `.claude/templates/arch-system.md`
2. 分析系统上下文（第 2 阶段决策文档、ADR 索引、各子系统设计状态）
3. 决定 agent 数量（基于复杂度与 template 要求，极高复杂度 ≥5）
4. 设计切分维度（按分析类型：子系统边界/数据流完整性/故障处理覆盖/性能约束一致性/可演进性）
5. 产出调度方案，包含三个完整部分：
   - (1) 并行产出 agent 配置（角色 + 视角约束 + 负责维度 + 输入材料）
   - (2) 评审 agent 配置（基于盲区分析，识别产出 agent 未覆盖的视角，配置对应的 reviewer）
   - (3) 整合策略（合并规则 + 冲突处理）
6. 产出以结构化文本形式在对话中输出

**Plan 禁止事项**：
- 不产出深度分析内容
- 不预设最终结论或倾向性方案

**主会话职责**：原样执行 architect 的调度方案，不做修改。若对方案有疑问，与用户确认或重新派遣 plan agent。

1. **并行启动 ≥5 个 architect agent**，各自从不同视角独立定稿系统总纲
2. 主会话整合为统一的 design.md 定稿
3. 写入 `docs/architecture/design.md`

### 评审修正循环

按 `common.md` 标准评审修正循环执行，本 步骤 特有配置：

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（系统架构总纲，定义全局子系统划分、接口契约和数据流）、重要性（是后续所有子系统设计和实现的最高层级参照）、可替换性（修正成本评估，架构总纲变更影响所有子系统）。

- **独立评审**：基于 Plan agent 识别的盲区，并行启动 ≥3 个 reviewer（极高复杂度下限），角色互不相同：
  - 评审视角差异化：至少覆盖子系统边界、数据流、故障处理三个维度
  - 质疑点数量要求：≥5 个（系统架构总纲为极高复杂度）
  - **评审配置推导规则**：每个 Plan agent 识别的视角盲区必须有对应的 reviewer 覆盖
  
- **修正 agent 配置**（按 `common.md` 修正工作量分级）：
  - 少量修正：1 个 architect
  - 中量修正：2 个 architect 分块并行修正 + 主会话整合
  - 大量修正：评估：分批修正 or 回退到本步骤开头重新执行 ≥5 个 architect 并行发散

**步骤 2 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 子系统边界清晰度 | 25% | 各子系统职责明确，无重叠或遗漏 |
| 数据流完整性 | 25% | 正常路径 + 错误传播路径均已定义 |
| 与 ADR 一致性 | 25% | 与 adr.md 无冲突 |
| 故障处理覆盖度 | 25% | 故障模式、降级策略、恢复流程已定义 |

**进入门槛：≥ 0.85**

## 步骤 3：子系统架构定稿

**产出文件**：`docs/architecture/<subsystem>/design.md`（各子系统文档）

1. 按子系统拆分，无依赖的子系统并行处理
2. **每个子系统并行启动 ≥3 个 architect agent**
3. 主会话整合为统一的子系统架构文档
4. 写入 `docs/architecture/<subsystem>/design.md`

### 评审修正循环

按 `common.md` 标准评审修正循环执行，本 步骤 特有配置：

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（子系统架构文档，定义单一子系统内部设计）、重要性（子系统可独立重构，但接口契约变更可能影响其他子系统）、可替换性（修正成本评估，子系统内部设计修正成本相对可控）。

- **独立评审**：并行启动 ≥2 个 reviewer（角色互不相同：architect-reviewer + product-reviewer）
  - 评审视角差异化：至少覆盖功能完整性、故障处理两个维度
  - 质疑点数量要求：≥3 个（子系统架构为高复杂度）
  
- **修正 agent 配置**：
  - 少量修正：1 个 architect
  - 中量修正：2 个 architect 分块并行修正 + 主会话整合
  - 大量修正：回退到本 步骤 开头重新执行 ≥3 个 architect 并行发散

**步骤 3 置信度评分**（按子系统）：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 功能路径完整性 | 25% | 核心数据流、状态转换、接口契约完整 |
| 故障处理覆盖度 | 25% | 故障模式、降级策略、恢复流程已定义 |
| 边界条件充分性 | 20% | 输入边界、资源边界、并发边界已覆盖 |
| 与总纲/ADR 一致性 | 20% | 不偏离总纲接口契约和 ADR 技术决策 |
| 可运维性 | 10% | 运维接口、监控点、日志规范已定义 |

**进入门槛：≥ 0.80**
