# 第 0 阶段：前置调研与定位确认

**准入条件**：无
**产出文件**：`docs/architecture/design.md`（初稿/WIP 状态）

---

## 核心原则

不调研清楚背景和约束就开始标杆研究，等同于盲人摸象。必须与用户通过**多次交互式提问**逐层深入，禁止一次性抛出所有问题。

与 `/df:define` 第 0 阶段保持一致的逐步展开式提问：每轮聚焦 1-2 个主题，从通用到具体逐层深入，根据用户回答动态调整后续问题方向。

---

## 执行方式

交互式前置调研由**派遣 architect agent 执行**，主会话负责搬运产出到对话正文。

> **关键事实**：用户看不到 Agent 工具的返回内容，只能看到主会话输出的文本。主会话必须将 architect agent 的交互式提问**完整搬运到对话正文**，用户才能看到问题并回答。

**主会话职责**：
1. 派遣 architect agent 执行交互式提问
2. 将 agent 产出的问题列表完整搬运到对话正文（不改写、不裁剪，仅格式统一）
3. 收集用户回答后，将回答传递给 agent 继续下一轮提问
4. 最终将 agent 产出的 design.md 初稿写入文件

### 步骤 1：检查 domain-config.yaml

主会话首先检查 `.claude/domain-config.yaml` 是否存在：

**场景 A：文件不存在**（首次执行，/df:define 未执行）
- 执行完整的领域信息收集（包括 /df:define 负责的部分）
- 生成新的 domain-config.yaml

**场景 B：文件已存在**（/df:define 已执行过）
- 读取现有配置，向用户展示
- 询问："这些信息是否仍然准确？"
- 如果准确：跳过已有字段的提问，只收集 design 负责的字段
- 如果需要修正：逐项确认需要修改的字段

### 步骤 2：领域信息收集（交互式提问）

主会话以 architect 视角进行多轮交互式提问：

**基础信息确认**（如果 domain-config.yaml 不存在）

如果文件不存在，先收集基础信息（同 /df:define 阶段 0）：
- 系统类型（primary_type + sub_type）
- 系统规模（complexity + team_size + codebase_size）
- 质量属性优先级（1-5 排序）

**编程语言确认**

```
问题："主要使用什么编程语言？"
选项：
- C
- C++
- Rust
- Go
- Python
- Java
- 其他（请在备注中说明）
```

追问：
- "语言版本和标准？"（如 C11, C++17, Rust 2021）
- "工具链？"（如 gcc/clang, rustc, go）
- 如果多语言："哪个是主语言？次要语言用于什么场景？"

**标杆产品清单**

```
问题："你希望研究哪些标杆产品作为参考？"

主会话建议："根据你描述的 <系统类型>，我建议研究以下标杆产品：
1. <标杆 1> - <简短描述>
2. <标杆 2> - <简短描述>
3. <标杆 3> - <简短描述>
是否确认？"

用户可以：
- 确认
- 修正（"改为研究 XXX"）
- 补充（"再加上 YYY"）
```

**架构约束深化**

```
问题："一致性模型的具体实现？"
选项：
- 强一致性（Raft / Paxos / 2PC）
- 最终一致性（Gossip / Vector Clock）
- 因果一致性
- 其他（请说明）
```

```
问题："并发模型？"
选项：
- 单线程
- 多线程
- 事件驱动
- Actor 模型
- 其他（请说明）
```

```
问题："内存管理策略？"
选项：
- 手动管理
- GC
- 混合（部分手动 + 部分 GC）
```

**性能基线量化**

```
问题："延迟目标？"
- P50: <输入>
- P99: <输入>
- P99.9: <输入>
```

```
问题："吞吐目标？"
- 单节点: <输入> (MB/s 或 OPS)
- 集群: <输入> (MB/s 或 OPS)
```

```
问题："扩展性目标？"
- 节点数范围: <输入>
- 是否要求线性扩展: <是/否>
```

### 步骤 3：系统背景与约束确认（原有流程）

主会话继续进行系统背景与约束的交互式提问：
- 系统目标
- 核心约束
- 部署拓扑和规模量级
- 明确不做的事项

### 步骤 4：生成配置文件和文档

**生成/更新 domain-config.yaml**

读取 `.claude/templates/domain-config.yaml` 模板，填充收集到的信息：

```yaml
# 如果是场景 A（文件不存在），填充完整信息
# 如果是场景 B（文件已存在），只更新 design 负责的字段

languages:  # 从编程语言确认获取
  primary:
    - name: "<从编程语言确认获取>"
      version: "<从编程语言确认获取>"
      toolchain: [<从编程语言确认获取>]

benchmarks:  # 从标杆产品清单获取
  - <标杆 1>
  - <标杆 2>
  - <标杆 3>

quality_attributes:
  targets:  # 从性能基线量化获取
    consistency_model: "<从架构约束深化获取>"
    latency_p99: "<从性能基线量化获取>"
    throughput: "<从性能基线量化获取>"

architecture:  # 从架构约束深化获取
  distributed:
    enabled: true/false
    consensus_protocol: "<从架构约束深化获取>"
  concurrency:
    model: "<从架构约束深化获取>"
  memory:
    strategy: "<从架构约束深化获取>"

metadata:
  created_at: "<当前日期>"  # 如果是新建
  created_by: "/df:design"  # 如果是新建
  last_updated: "<当前日期>"  # 如果是更新
  updated_by: "/df:design"  # 如果是更新
```

写入 `.claude/domain-config.yaml`。

**4.2 生成 design.md**

读取 `.claude/templates/arch-system.md` 模板，将调研结论整理为 `design.md` 初始草稿并落盘。

> **豁免说明**：第 0 阶段适用豁免场景：纯交互式信息收集阶段。不执行完整多 agent 评审修正循环，但需通过准入检查清单 + 置信度评分确认信息充分性。

---

## 出口标准

### 准入检查清单

- [ ] **domain-config.yaml 已存在或已生成**
- [ ] **languages 字段已填充**（至少一个主语言，含版本和工具链）
- [ ] **benchmarks 列表已填充**（至少 2-3 个标杆产品）
- [ ] **architecture 字段已填充**（如果适用）
- [ ] **quality_attributes.targets 已填充**（至少 3 个量化指标）
- [ ] 系统目标已明确（解决什么问题）
- [ ] 核心约束已确认（性能/一致性/可用性/成本）
- [ ] 部署拓扑和规模量级已确认
- [ ] 至少 2-3 个"明确不做"的事项已列出
- [ ] 用户已确认定位摘要无误

### 置信度评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 目标清晰度 | 25% | 能用一句话描述系统目标 |
| 语言栈明确度 | 25% | 主语言已确认，版本和工具链已明确 |
| 架构约束明确度 | 25% | 一致性模型、并发模型、内存管理策略已明确 |
| 性能基线量化 | 15% | 至少 3 个可量化的性能指标 |
| 边界清晰度 | 10% | 明确不做什么 |

**进入第 1 阶段门槛：≥ 0.80**

---

## 自动推进规则

准入检查清单全部通过 + 置信度 ≥ 0.80 → 自动进入第 1 阶段

否则继续交互式提问
