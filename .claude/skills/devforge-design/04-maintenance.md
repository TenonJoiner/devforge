# 第 4 阶段：架构维护与扩展

**准入条件**：第 3 阶段完成（所有架构文档已定稿，含 ✅ PASS 标记）
**产出文件**：按需修正现有文档

---

## 核心定位

前 3 阶段完成后，架构文档已进入"可用状态"。第 4 阶段是**维护模式**——按需调用 `/df:design` 来修正、补充或调整架构设计。

---

## 快速通道（小修不走完整阶段）

> **豁免说明**：快速通道属于维护模式单点修正，适用豁免场景。不走完整出口标准和评审修正循环，但修正后仍需至少 1 个 reviewer 快速复核。

**适用场景**：
- 文档一致性修正（如交叉引用错误、术语不统一）
- 单个 ADR 补充（论证细化、追溯关系补充）
- 单篇标杆补充（新增标杆或视角）
- 单个子系统局部调整（不涉及接口契约变更）

**流程**：
1. 派 architect agent 执行修正（直接 Edit 对应文件）
2. 派 1 个 architect-reviewer 快速复核
3. reviewer 将复核意见追加到 `-review.md`，返回数字摘要
4. 主会话基于缺陷密度判定通过/再修正

**评审参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| 评估对象数 | 修正点数 | 本次维护涉及的具体修正点（章节级或 ADR 级） |
| 缺陷密度门槛 | ≤ 3.0 分/修正点 | 见 SKILL.md「缺陷密度门槛标定依据」（N=1 时累计 ≤ 3 分，1 个 HIGH 到顶） |
| 复核 agent | architect-reviewer × 1 | 单 reviewer 快速复核 |
| 修正循环上限 | 1 次（小修原则上一次到位） | 超过 1 次说明应转回退机制 |

**成功退出条件**：
- 无 CRITICAL 问题
- 缺陷密度 ≤ 3.0 分/修正点
- 修正后的文档末尾保持 `**评审状态**: ✅ PASS` 标记（按 SKILL.md「评审状态标记契约」格式，更新评审日期）

---

## 回退机制（大变更回退到对应子流程）

| 变更类型 | 回退目标 |
|---------|---------|
| 新增子系统 | 回退到第 2 阶段步骤 1（整体架构发散重新评估） |
| 推翻已有 ADR | 回退到第 2 阶段步骤 3（对应维度重新发散） |
| 新增技术维度 | 回退到第 2 阶段步骤 2-3（维度划分 + 维度方案发散） |
| 子系统边界调整 | 回退到第 3 阶段步骤 2（系统总纲重新定稿） |
| 子系统内部重大重构 | 回退到第 3 阶段步骤 3（该子系统重新定稿） |

回退到完整阶段后，按对应阶段的完整流程执行，包含完整的多 agent 并行产出 + 独立评审 + 评审修正循环。

---

## 执行规则

1. **判断变更类型**：快速通道 vs 回退到完整阶段
   - 单文件单章节、不涉及跨子系统影响 → 快速通道
   - 涉及多文档、跨子系统、不可逆决策 → 回退到完整阶段
2. **快速通道**：
   - 执行修正（architect agent Edit 对应文件）
   - 1 个 architect-reviewer 快速复核
   - 通过 → 落盘；不通过 → 再修正一次或回退到完整阶段
3. **回退到完整阶段**：
   - 按对应阶段的完整流程执行
   - 包含完整的多 agent 并行产出 + 独立评审 + 评审修正循环
   - 阶段完成后更新对应文档的 `**评审状态**` 标记
