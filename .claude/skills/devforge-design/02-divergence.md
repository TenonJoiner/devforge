# 第 2 阶段：方案发散

**准入条件**：第 1 阶段完成，标杆研究通过评审（≥ 0.80）
**产出文件**：`docs/architecture/decisions/decision-overall.md` + `docs/architecture/decisions/decision-<维度>.md`

---

## 核心流程

第 2 阶段包含以下步骤，按顺序执行：

1. **步骤 1**：任务解构与整体架构发散（≥5 个 architect 并行）→ **独立评审**（≥3 个 reviewer）
2. **步骤 2**：维度确认（主会话整合）
3. **步骤 3**：并行 researcher 调研（每个维度 2 个 researcher）
4. **步骤 4**：并行 architect 发散（每个维度 ≥3 个 architect）→ **独立评审**（每个维度 ≥2 个 reviewer）
5. **步骤 5**：同步迭代 design.md（主会话整合）
6. **步骤 6**：执行回溯检查（验证 agent 执行记录）
7. **步骤 7**：阶段级一致性检查 → **独立评审**（≥2 个 reviewer）

---

## 步骤 1：任务解构与整体架构发散

> **Plan 阶段内联**：第 2 阶段为极高复杂度（系统级架构设计），必须先解构再发散。

**任务解构**（Plan agent 工作流程）：
1. 派遣 **1 个 architect agent** 读取 `.claude/templates/arch-system.md`
2. 分析系统上下文（系统类型、技术栈、业务目标、约束条件）
3. 决定 agent 数量（基于复杂度与 template 要求，极高复杂度 ≥5）
4. 设计切分维度（按子系统/按分析类型/按职能侧重/按范围深度）
5. 产出调度方案，包含三个完整部分：
   - (1) 并行产出 agent 配置（角色 + 视角约束 + 负责维度 + 输入材料）
   - (2) 评审 agent 配置（基于盲区分析，识别产出 agent 未覆盖的视角，配置对应的 reviewer）
   - (3) 整合策略（合并规则 + 冲突处理）
6. 产出以结构化文本形式在对话中输出，不写入文件

**Plan 禁止事项**：
- 不产出深度分析内容
- 不预设最终结论或倾向性方案

**主会话职责**：原样执行 architect 的调度方案，不做修改。若对方案有疑问，与用户确认或重新派遣 plan agent。

> **整合成本评估**：步骤 1 中 ≥5 个 architect 并行产出同一份 `decision-overall.md`，虽然整合多个独立视角需要合并/去重/冲突处理，但整体架构方案的系统级决策权重极高，单一视角的结构性盲区可能导致后续所有维度方案推倒重来。并行收益（5 个独立视角覆盖数据流/状态管理/性能/故障/可演进性）远超整合开销（主会话按 Plan agent 的整合策略合并）。

**整体架构发散**：
1. **并行启动 ≥5 个 architect agent**，各自从不同架构视角独立发散
   - **5 个 architect 的视角区分**（通过 prompt 明确指定）：
     - architect（关注数据流与通信范式）：分析数据如何在子系统间流动、选择同步/异步通信模式、定义消息格式与序列化方案
     - architect（关注状态管理与一致性范式）：分析状态存储位置、一致性协议选择、分布式事务处理方式
     - architect（关注性能约束倒推）：从性能目标倒推架构约束、识别性能瓶颈点、设计缓存与优化策略
     - architect（关注故障处理与降级策略）：分析故障模式、设计降级方案、定义恢复流程与数据修复机制
     - architect（关注可演进性与扩展性）：分析未来扩展点、设计插件机制、定义接口演进策略
2. 每个 architect 产出 ≥2 个整体架构备选方案
3. 主会话整合为统一的整体架构初稿
4. 写入 `docs/architecture/decisions/decision-overall.md`

**步骤 1 出口标准**（写入后）：
- [ ] 整体架构备选方案 ≥2 个
- [ ] 各子系统职责边界已定义
- [ ] 数据流正常路径 + 错误传播路径已定义
- [ ] CRITICAL 问题已修正
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 2 阶段 步骤 1 完成（待评审）"

**步骤 1 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 方案充分性 | 30% | 整体架构备选方案 ≥2 个 |
| 子系统边界清晰度 | 30% | 各子系统职责明确，无重叠或遗漏 |
| 数据流完整性 | 40% | 正常路径 + 错误传播路径均已定义 |

**进入门槛：≥ 0.80**

### 步骤 1 评审修正循环

产出写入文件后，立即进入独立评审：

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（整体架构方案文档，定义子系统边界和全局数据流）、重要性（影响后续所有维度的方案发散和子系统设计）、可替换性（修正成本评估）。

1. **独立评审**：基于 Plan agent 识别的盲区，并行启动 ≥3 个 reviewer（极高复杂度下限），角色互不相同
   - 评审视角差异化：至少覆盖架构合理性、需求对齐、可实现性三个维度
   - 质疑点数量要求：≥5 个（整体架构方案为极高复杂度）
2. **回退判断**：按 common.md 回退决策表执行
3. **修正**：按 common.md 修正工作量分级配置 agent
4. **复核**：修正后重新评审
5. **循环上限**：最多 3 次
6. **终止条件**：所有 CRITICAL 已修正 + 所有 reviewer 给出"通过"或"有条件通过"

---

## 步骤 2：维度确认

主会话整合步骤 1 的多个 architect 产出及评审意见，识别收敛和分歧点，确认架构决策维度清单（如数据流、一致性、存储、网络、故障处理等），形成维度划分方案。

**步骤 2 出口标准**（维度划分方案形成后）：
- [ ] 维度清单已明确（≥3 个维度，覆盖数据流/一致性/存储/网络/故障处理等）
- [ ] 每个维度有明确的决策边界说明（不与其他维度重叠）
- [ ] 各维度与 decision-overall.md 的子系统划分一致
- [ ] 维度划分方案已向用户确认
- [ ] 文档状态标记为"第 2 阶段 步骤 2 完成（维度已确认）"

**步骤 2 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 维度覆盖完整性 | 40% | 所有子系统关键决策点均有对应维度覆盖 |
| 维度边界清晰度 | 30% | 各维度间无重叠或遗漏 |
| 与整体架构一致性 | 30% | 维度划分与 decision-overall.md 对齐 |

**进入门槛：≥ 0.80**

## 步骤 3：并行 researcher 调研（按维度）

每个维度并行启动 2 个 researcher agent，从不同视角研究标杆方案在该维度的设计选择。产出各维度的调研摘要。

## 步骤 4：并行 architect 发散（按维度）

每个维度并行启动 ≥3 个 architect agent，基于调研摘要进行该维度的架构方案发散。每个 architect 产出 ≥2 个备选方案。各维度独立并行执行，无依赖关系。

**步骤 3-4 出口标准**（步骤 3 为 researcher 调研，步骤 4 为 architect 发散，共享出口标准）（写入后）：
- [ ] 每个维度有 ≥2 个备选方案
- [ ] 每个方案有明确的 Trade-off 分析
- [ ] 文档已写入 `docs/architecture/decisions/decision-<维度>.md`
- [ ] CRITICAL 问题已修正
- [ ] 文档状态标记为"第 2 阶段 步骤 4 完成（待评审）"

**步骤 3-4 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 方案多样性 | 40% | 每个维度 ≥2 个独立方案 |
| Trade-off 清晰度 | 30% | 各方案取舍分析明确 |
| 与整体架构一致性 | 30% | 各维度方案与 decision-overall.md 不冲突 |

**进入门槛：≥ 0.80**

### 步骤 4 评审修正循环

每个维度的 decision-<维度>.md 写入后，立即进入独立评审：

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（维度决策文档，聚焦单一技术维度的方案对比）、重要性（影响该维度的最终技术选型）、可替换性（修正成本评估）。

1. **独立评审**：每个维度并行启动 ≥2 个 reviewer（高复杂度下限），角色互不相同：
   - `architect-reviewer`（关注技术正确性）：质疑方案的技术可行性、与标杆的差异理由
   - `product-reviewer`（关注需求对齐）：质疑维度方案与产品需求的匹配度、Actor 覆盖完整性
2. **回退判断** → **修正** → **复核**（按 common.md 标准流程）
3. **循环上限**：最多 3 次；终止条件：所有 CRITICAL 已修正 + 所有 reviewer 给出"通过"或"有条件通过"

## 步骤 5：同步迭代 design.md

主会话整合各维度决策文档及评审意见，更新 `docs/architecture/design.md`，确保各维度方案间无冲突、子系统边界一致。

**步骤 5 出口标准**（design.md 更新后）：
- [ ] design.md 已反映所有维度决策的最新结论
- [ ] 各子系统接口契约无冲突
- [ ] 跨维度依赖关系已明确标注
- [ ] 文档已写入并验证存在
- [ ] 文档状态标记为"第 2 阶段 步骤 5 完成（design.md 已同步）"

**步骤 5 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 维度整合完整性 | 40% | 所有维度决策均已映射到 design.md 对应章节 |
| 子系统一致性 | 30% | 各子系统接口契约无冲突 |
| 跨维度依赖清晰度 | 30% | 跨维度依赖关系已明确标注 |

**进入门槛：≥ 0.80**

## 步骤 6：执行回溯检查

验证每个 agent 的执行记录完整性，确认所有并行产出的文件已落盘且通过评审。

**步骤 6 出口标准**（回溯检查完成后）：
- [ ] 所有 agent 执行记录已核实（派遣数量、产出文件、评审结论）
- [ ] 所有并行产出文件已落盘（`docs/architecture/decisions/decision-*.md`）
- [ ] 各步骤独立评审均已通过（所有 reviewer 给出"通过"或"有条件通过"）
- [ ] 未发现遗漏的 agent 产出（对话中有产出但未写入文件）
- [ ] 文档状态标记为"第 2 阶段 步骤 6 完成（回溯检查通过）"

**步骤 6 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 执行记录完整性 | 50% | 所有 agent 执行记录已核实，无遗漏 |
| 文件落盘完整性 | 30% | 所有产出文件已确认存在 |
| 评审闭环完整性 | 20% | 各步骤评审修正循环均已闭环 |

**进入门槛：≥ 0.80**

## 步骤 7：阶段级一致性检查

验证各维度决策文档之间的一致性，检查跨维度冲突、接口契约不一致等问题。

**步骤 7 出口标准**（一致性检查完成后）：
- [ ] 各维度决策文档间无跨维度冲突（如数据流方向与接口契约一致）
- [ ] 各子系统接口契约版本一致（同一接口在不同维度文档中定义一致）
- [ ] 跨维度依赖关系已核实（上游输出与下游输入匹配）
- [ ] CRITICAL 问题已修正
- [ ] 文档状态标记为"第 2 阶段 步骤 7 完成（一致性检查通过）"

**步骤 7 置信度评分**：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 跨维度无冲突 | 40% | 各维度决策文档间无逻辑冲突 |
| 接口契约一致性 | 30% | 同一接口在各维度文档中定义一致 |
| 跨维度依赖匹配 | 30% | 上游输出类型/格式与下游输入类型/格式匹配 |

**进入门槛：≥ 0.80**

### 步骤 7 评审修正循环

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在架构体系中的位置（阶段级一致性检查，验证各维度决策文档间的跨维度无冲突）、重要性（确保整体架构的内部一致性，发现跨维度矛盾）、可替换性（修正成本评估）。

1. **独立评审**：并行启动 ≥2 个 reviewer（关注跨维度一致性）：
   - `architect-reviewer`（关注跨维度技术一致性）
   - `project-reviewer`（关注跨维度依赖合理性）
2. 按 common.md 标准评审修正循环执行回退判断 → 修正 → 复核
