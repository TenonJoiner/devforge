# devforge-tdd-workflow — TDD 铁律开发工作流

## 概述

RED → GREEN → REFACTOR。没有先失败的测试，决不写生产代码。

**核心原则**：
- **测试先行**：每个功能增量必须有测试失败作为起点
- **最小步长**：每个步骤只解决当前测试失败，不超前实现
- **绿色保障**：REFACTOR 阶段必须保持测试常绿

## 何时使用

- 实现单个 OpenSpec task（尤其是 tasks.md 中的步骤 N.M）
- 为现有函数补充缺失单元测试
- 重构思路上需要测试保驾护航的代码修改
- **修复 Bug**：先写失败测试复现 bug，再修复，确保回归保护

### Bug 修复的 TDD 流程

1. **RED**：写测试复现 bug（测试必须失败，证明 bug 存在）
2. **Verify RED**：确认失败信息与 bug 现象一致
3. **GREEN**：修复 bug 使测试通过
4. **Verify GREEN**：确认所有测试通过（包括修复前已有的测试）
5. **REFACTOR**：在测试保护下清理修复代码

**禁止不写测试直接修 bug**。没有回归测试的 bug 修复只是临时止血，不是真正修复。

## 覆盖范围

一次 `/df:tdd` 调用完成**单个 Scenario** 的完整 RED → GREEN → REFACTOR 循环。

**循环粒度是"行为"，不是"接口"**：一个 Scenario 是一个行为，可能需要多个接口协作实现。底层接口的拆分是实现细节，不是 TDD 循环的边界。

### 多接口 Scenario 的执行步骤

当一个 scenario 涉及多个接口时，按以下步骤展开：

**步骤 1：写 Scenario 测试（顶层 RED）**

假设所有需要的接口都已存在，写出验证完整行为的测试。运行测试，记录失败模式。

**步骤 2：根据失败模式分支**

- **编译失败（接口不存在）** → 进入步骤 3
- **编译通过、运行失败（断言不通过）** → 跳到步骤 4

**步骤 3：补齐接口存根（让测试可编译）**

为每个缺失的接口写最小空实现：返回默认值（NULL/0/Err）即可，不实现真实逻辑。

> 这不违反"先有失败测试"——这些存根的失败测试就是顶层 scenario 测试本身。空存根让 scenario 从"编译失败"变成"运行失败"，进入步骤 4。

**步骤 4：定位失败的具体接口**

读 scenario 测试的失败信息（哪个 assert、期望值 vs 实际值），定位是哪个接口的行为缺失或错误。

**步骤 5：进入子 TDD 循环（嵌套 RED-GREEN-REFACTOR）**

为该接口启动子循环：
- 子 RED：写该接口的**单元测试**（不依赖其他未实现的接口，必要时 mock 或用空存根）
- 子 GREEN：实现该接口的最小真实逻辑使单元测试通过
- 子 REFACTOR：清理该接口内部代码

**步骤 6：回到 scenario 测试**

运行顶层 scenario 测试：
- 仍失败 → 回到步骤 4，处理下一个失败的接口
- 通过 → 进入步骤 7

**步骤 7：顶层 REFACTOR**

在 scenario 测试和所有单元测试保护下：去重、改名、整理接口边界、清理临时存根。

### 简单 Scenario 的捷径

如果 scenario 只涉及 1-2 个简单接口，可以跳过嵌套：直接在 scenario 的 GREEN 阶段写完整实现，REFACTOR 时再决定是否拆分接口。

判断标准：**写实现时如果觉得需要先想清楚某个接口的契约，就停下来用嵌套循环**。

## 核心流程

**适用范围**：以下 RED → GREEN → REFACTOR 是**单次循环的标准动作**，顶层 scenario 循环和嵌套子接口循环都遵循它。

### 阶段 1：RED（写一个失败的测试）

**成功标准**：编译通过但测试失败（断言失败或预期错误）。

**步骤**：
1. 读取当前 task 的目标和 specs/design 约束
2. 确定本次要暴露的行为缺口
3. 在测试文件中编写最小测试用例
4. 运行测试，确认失败，记录失败信息

**红旗自查**：
- 测试一开始就通过了（说明测试没测到目标行为）
- 测试写了太多断言（应只有一个核心断言暴露缺口）
- 直接复制了生产代码到测试里（测试成了实现的影子）

### 阶段 2：GREEN（用最快速度让测试通过）

**成功标准**：所有测试通过（返回码 0）。

**步骤**：
1. 写最小量生产代码使测试通过
2. 运行测试确认通过

**阶段性容忍（必须在 REFACTOR 阶段清理）**：
- 直接返回硬编码值
- 复制粘贴重复逻辑
- 用最直白的方式实现条件分支

**边界说明**：
这些临时写法仅在当前测试与下一次 REFACTOR 之间短暂存在，**不得在未重构的情况下直接提交**。如果因为时间或其他原因跳过 REFACTOR，这些临时代码必须被标记为 TODO/FIXME 并在后续 task 中清理。

**禁止的操作**：
- 提前实现未被测试覆盖的行为
- 引入复杂抽象（提前设计）
- 优化性能（留在 REFACTOR 阶段）
- 以"GREEN 允许丑代码"为借口跳过 REFACTOR

### 阶段 3：REFACTOR（scenario 级别即时清理）

**成功标准**：测试保持绿色，代码质量提升。

**步骤**：
1. 识别代码异味：重复、过长函数、魔法数字、不清晰命名
2. 执行安全重构（提取函数、重命名变量、消除重复）
3. 运行测试确认绿色

**REFACTOR 期间必须检查**：
- [ ] 所有测试通过
- [ ] 圈复杂度未显著上升

## 铁律检查

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

**违反规则的字面意思就是违反规则的精神。**

### 违反铁律的处理

如果你在测试之前写了生产代码：

1. **删除它**。不是注释掉，不是保留作为"参考"，是**删除**
2. **不要看它**。不要"改编"它。不要在脑子里记着它
3. **从测试重新开始**。从零实现

**没有例外**：
- ❌ "保留作为参考" → 你会改编它，那就是测试后写
- ❌ "只是调整一下" → 那就是测试后写
- ❌ "已经花了 X 小时" → 沉没成本谬误。保留无法信任的代码才是技术债
- ❌ "保持为'探索代码'" → 探索可以，探索完必须删除，用 TDD 重新实现

### 为什么顺序重要

测试先写回答"应该做什么"，测试后写回答"做了什么"。测试后写被实现偏见污染——你测试的是你构建了什么，而非需要什么。30 分钟事后补的测试 ≠ TDD：你得到了覆盖率，失去了测试有效的证据。

## 常见借口与回应

**所有这些借口都意味着：删除代码，用 TDD 重新开始。**

| 借口 | 现实 |
|------|------|
| "测试框架还没搭好" | 先花 10 分钟写一个最小可运行的测试桩。测试框架是第一优先级 |
| "这太简单了，边写边测就行" | 简单代码也会出错。写测试只需 30 秒 |
| "TDD 不适合系统编程语言" | 这是本团队的工作方式。C/Rust/Go 都有成熟的测试框架和工具链 |
| **"我会在写完后补测试"** | **测试后写立即通过，证明不了任何事。你在测试"做了什么"而非"应该做什么"** |
| **"保留作为参考，然后先写测试"** | **你会改编它。那就是测试后写。删除就是删除** |
| **"需要先探索一下"** | **可以。探索完扔掉探索代码，用 TDD 重新实现** |
| **"测试难写说明设计不清晰"** | **听测试的反馈。难测试 = 难使用。简化设计** |
| **"TDD 会拖慢我"** | **TDD 比调试快。务实 = 测试先行** |
| **"已经手动测试过所有边界情况"** | **手动测试是临时的：无记录、无法重跑、每次修改都要重测** |
| **"已经花了 X 小时，删掉太浪费"** | **沉没成本谬误。保留无法信任的代码才是技术债** |
| **"测试后写能达到同样目的"** | **不能。测试后写回答"做了什么"，测试先写回答"应该做什么"。测试后写被实现偏见污染** |
| **"TDD 太教条，务实意味着灵活"** | **TDD 就是务实：提前发现 bug、防止回归、文档化行为、支持重构** |
| **"这次情况不同，因为..."** | **不，不是。删除代码，用 TDD 重新开始** |

## 输出位置

- 测试文件：对应 rules `testing` 中定义的命名规范
- 生产代码：对应 task 指定的源文件

## 红旗 - 停止并重新开始

以下任何情况都意味着：**删除代码，用 TDD 重新开始**

- 测试之前写了代码
- 实现之后补测试
- 测试立即通过（没看到失败）
- 无法解释测试为什么失败
- 测试"稍后"补
- 合理化"就这一次"
- "我已经手动测试过了"
- "测试后写能达到同样目的"
- "重要的是精神不是仪式"
- "保留作为参考"或"改编现有代码"
- "已经花了 X 小时，删除太浪费"
- "TDD 太教条，我在务实"
- "这次情况不同，因为..."

**所有这些都意味着：删除代码，用 TDD 重新开始。**

## 测试反模式

添加 mock 或测试工具时，阅读 @testing-anti-patterns.md 避免常见陷阱：
- 测试 mock 行为而非真实行为
- 在生产类中添加测试专用方法
- 不理解依赖就 mock
- 不完整的 mock
