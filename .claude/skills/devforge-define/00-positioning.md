# 第 0 阶段：产品定位确认

**准入条件**：无
**产出文件**：`.claude/domain-config.yaml`

---

## 核心原则

在启动标杆研究之前，必须先完成轻量级产品定位确认。如果产品边界不清，标杆研究可能选错对象，导致后续全部返工。

**交互原则**：
- 通过交互式提问逐层深入理解产品定位，禁止一次性抛出所有问题
- 每个问题等待用户回答后再提出下一个问题
- 根据用户回答动态调整后续问题

---

## 执行方式

第 0 阶段为交互式产品定位确认，由**主会话直接执行**。

### 步骤 1：检查 domain-config.yaml

主会话首先检查 `.claude/domain-config.yaml` 是否存在：

**场景 A：文件不存在**（首次执行）
- 执行完整的领域信息收集（步骤 2）
- 生成新的 domain-config.yaml

**场景 B：文件已存在**（/df:design 或 /df:define 先前已执行过）
- 读取现有配置，向用户展示
- 询问："这些信息是否仍然准确？需求视角下是否有需要调整的？"
- 如果准确：跳过步骤 2 中已有字段的提问
- 如果需要修正：逐项确认需要修改的字段

### 步骤 2：产品定位信息收集（交互式提问）

主会话进行多轮交互式提问，收集产品定位信息：

**2.1 系统类型识别**

通过 AskUserQuestion 确认：

```
问题："你正在开发什么类型的系统？"
选项：
- 分布式系统（存储/计算/消息队列/数据库）
- 操作系统/内核
- 编译器/运行时
- 网络协议栈
- 虚拟化/容器
- 高性能计算
- 复杂嵌入式系统
- 其他（请在备注中说明）
```

如果选择"分布式系统"，追问子类型：
```
问题："具体是哪种分布式系统？"
选项：
- 分布式存储（对象存储/块存储/文件系统）
- 分布式计算（批处理/流处理）
- 消息队列/流处理系统
- 数据库系统
```

**2.2 核心痛点与目标用户**

主会话逐个提问（每个问题等待用户回答后再提下一个）：

1. "这个系统要解决什么核心痛点？（用一句话描述）"
   - 根据回答追问：如果提到性能 → "具体的性能目标是什么？"；如果提到一致性 → "需要什么级别的一致性保证？"

2. "主要面向哪类用户？（如：开发者、运维人员、最终用户、外部系统）"
   - 根据回答追问：如果是开发者 → "什么语言/技术栈的开发者？"；如果是运维 → "主要关注哪些运维场景？"

3. "典型使用场景是什么？（请列举 2-3 个主力场景）"
   - 根据回答追问：对每个场景问"这个场景的频率和重要性如何？"

4. "明确不做的事项有哪些？（请列举 2-3 个边界）"

**2.3 质量属性优先级**

主会话以文本方式询问用户对以下质量属性的优先级排序：

```
请对以下质量属性按优先级排序（从高到低）：
- 一致性（数据一致性、并发正确性）
- 性能（高吞吐、低延迟）
- 可用性（高可用、容错）
- 可维护性（代码质量、可读性）
- 成本（开发成本、运维成本）

请直接以文本回复排序结果，例如："一致性 > 性能 > 可用性 > 可维护性 > 成本"
```

主会话从用户文本回复中解析排序结果，并追问："为什么这样排序？有具体的量化目标吗？"

**2.4 系统规模与部署场景**

```
问题："预期的系统规模和部署场景？"
- 部署规模：<选项：单机 / 小集群（<10节点）/ 中等集群（10-100节点）/ 大规模集群（>100节点）>
- 数据规模：<选项：GB级 / TB级 / PB级 / EB级>
- 并发规模：<选项：<100 / 100-1k / 1k-10k / 10k-100k / >100k>
```

根据用户选择追问：
- 如果选择"大规模集群" → "预期的节点数量级？集群管理的主要挑战是什么？"
- 如果选择"PB级/EB级" → "数据增长速度如何？是否需要数据分层存储？"
- 如果选择">10k 并发" → "并发模式是什么？（长连接/短连接/混合）"

### 步骤 3：生成 domain-config.yaml

读取 `.claude/templates/domain-config.yaml` 模板，填充收集到的信息：

```yaml
system:
  primary_type: "<从步骤 2.1 获取>"
  sub_type: "<从步骤 2.1 获取>"
  scale:
    deployment: "<从步骤 2.4 获取：单机/小集群/中等集群/大规模集群>"
    data_scale: "<从步骤 2.4 获取：GB级/TB级/PB级/EB级>"
    concurrency: "<从步骤 2.4 获取：并发规模>"
  description: "<从步骤 2.2 的核心痛点 + 目标用户 + 主力场景 + 不做事项总结>"

# languages 字段由 /df:design 第 0 阶段填充

quality_attributes:
  priorities:
    consistency: <从步骤 2.2 获取排序>
    performance: <从步骤 2.2 获取排序>
    availability: <从步骤 2.2 获取排序>
    maintainability: <从步骤 2.2 获取排序>
    cost: <从步骤 2.2 获取排序>

metadata:
  created_at: "<当前日期>"
  contributors: ["/df:define"]
```

写入 `.claude/domain-config.yaml`。

> **注意**：本文件由 /df:define 和 /df:design 共同维护。define 只填充 system 和 quality_attributes，其他字段（languages, benchmarks 等）由 design 填充。标杆研究后可能需回溯修正 description 字段。

---

## 出口标准

### 准入检查清单

写入 `domain-config.yaml` 前必须确认：

- [ ] **系统类型已确认**（primary_type + sub_type）
- [ ] **系统规模已确认**（deployment + data_scale + concurrency）
- [ ] **质量属性优先级已排序**（1-5，并追问了原因和量化目标）
- [ ] **核心痛点已收敛**到一句话描述
- [ ] **核心用户类型已确认**（仅大类，如管理员/开发者/外部系统）
- [ ] **典型使用场景已列举**（2-3 个主力场景）
- [ ] **明确不做的事项已列出**（2-3 个边界）

写入后必须确认：

- [ ] **domain-config.yaml 已生成并落盘**（用 `ls` 验证文件存在）

**进入第 1 阶段条件**：准入检查清单全部通过
