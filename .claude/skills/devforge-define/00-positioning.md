# 第 0 阶段：产品定位确认

**准入条件**：无
**产出文件**：`docs/requirements/product-spec.md`（初始定位草稿，后续阶段持续丰富）

---

## 核心原则

在启动标杆研究之前，必须先完成轻量级产品定位确认。如果产品边界不清，标杆研究可能选错对象，导致后续全部返工。

- 通过交互式提问逐层深入理解产品定位，禁止一次性抛出所有问题
- 追问策略和交互技巧详见 `product.md`

---

## 执行方式

第 0 阶段为交互式产品定位确认，由**主会话直接执行**。

### 步骤 1：检查 domain-config.yaml

主会话首先检查 `.claude/domain-config.yaml` 是否存在：

**场景 A：文件不存在**（首次执行）
- 执行完整的领域信息收集（步骤 2）
- 生成新的 domain-config.yaml

**场景 B：文件已存在**（/df:design 已执行过）
- 读取现有配置，向用户展示
- 询问："这些信息是否仍然准确？"
- 如果准确：跳过步骤 2 中已有字段的提问
- 如果需要修正：逐项确认需要修改的字段

### 步骤 2：领域信息收集（交互式提问）

主会话以 product 视角进行多轮交互式提问：

**2.1 系统类型识别**

通过 AskUserQuestion 确认：

```
问题："你正在开发什么类型的系统？"
选项：
- 分布式系统（存储/计算/消息队列/数据库）
- 操作系统/内核
- 编译器/运行时
- 网络协议栈
- 虚拟化/容器
- 高性能计算
- 复杂嵌入式系统
- 其他（请在备注中说明）
```

如果选择"分布式系统"，追问子类型：
```
问题："具体是哪种分布式系统？"
选项：
- 分布式存储（对象存储/块存储/文件系统）
- 分布式计算（批处理/流处理）
- 消息队列/流处理系统
- 数据库系统
```

**2.2 质量属性优先级**

```
问题："请对以下质量属性排序（1 最高，5 最低）"
选项（multiSelect: false，需要排序）：
- 一致性（数据一致性、并发正确性）
- 性能（高吞吐、低延迟）
- 可用性（高可用、容错）
- 可维护性（代码质量、可读性）
- 成本（开发成本、运维成本）
```

追问："为什么这样排序？有具体的量化目标吗？"

**2.3 系统规模**

```
问题："预期的系统规模？"
- 团队规模：<输入>
- 代码行数范围：<选项：<10k / 10k-50k / 50k-100k / 100k-500k / >500k>
- 复杂度：<选项：small / medium / large / very-large>
```

### 步骤 3：产品定位确认（原有流程）

主会话继续进行产品定位确认的交互式提问：
- 核心痛点
- 目标用户（Actor 大类）
- 主力场景
- 明确不做的事项

### 步骤 4：生成配置文件和文档

**4.1 生成/更新 domain-config.yaml**

读取 `.claude/templates/domain-config.yaml` 模板，填充收集到的信息：

```yaml
system:
  primary_type: "<从步骤 2.1 获取>"
  sub_type: "<从步骤 2.1 获取>"
  scale:
    complexity: "<从步骤 2.3 获取>"
    team_size: "<从步骤 2.3 获取>"
    codebase_size: "<从步骤 2.3 获取>"
  description: "<从步骤 3 的核心痛点总结>"

quality_attributes:
  priorities:
    consistency: <从步骤 2.2 获取排序>
    performance: <从步骤 2.2 获取排序>
    availability: <从步骤 2.2 获取排序>
    maintainability: <从步骤 2.2 获取排序>
    cost: <从步骤 2.2 获取排序>

metadata:
  created_at: "<当前日期>"
  created_by: "/df:define"
```

写入 `.claude/domain-config.yaml`。

**4.2 生成 product-spec.md**

读取 `.claude/templates/req-product-spec.md` 模板，将定位结论整理为 `product-spec.md` 初始草稿并落盘。

> **product-spec.md 的定位**：从第 0 阶段开始，它就是**持续迭代的活文档**。第 1 阶段标杆研究可能反过来修正产品定位，此时需**回溯修正** product-spec.md 中的定位章节。

> **豁免说明**：第 0 阶段适用豁免场景：纯交互式信息收集阶段。不执行完整多 agent 评审修正循环，但需通过准入检查清单 + 置信度评分确认信息充分性。

---

## 出口标准

### 准入检查清单

写入 `product-spec.md` 初稿前必须确认：

- [ ] **系统类型已确认**（primary_type + sub_type）
- [ ] **系统规模已确认**（complexity + team_size + codebase_size）
- [ ] **质量属性优先级已排序**（1-5）
- [ ] **domain-config.yaml 已生成并落盘**
- [ ] 核心痛点已收敛到一句话描述
- [ ] 核心用户类型已确认（仅确认大类：如管理员/开发者/外部系统，不深入枚举具体 Actor）
  > Actor 的完整识别和深度建模是第 2 阶段 agent 的职责，第 0 阶段只需确认产品面向的用户大类，禁止要求用户逐一列举角色
- [ ] 部署拓扑和规模量级已确认
- [ ] 至少 2-3 个关键约束已明确（能用数字或等级描述）
- [ ] 至少 2-3 个"明确不做"的事项已列出
- [ ] 用户已确认定位摘要无误

### 置信度评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 痛点收敛度 | 30% | 能用一句话描述核心痛点 |
| 系统类型明确度 | 25% | primary_type 和 scale 已确认 |
| 质量属性明确度 | 20% | 质量属性优先级已排序 |
| Actor 完整性 | 15% | 所有关键 Actor 大类已识别 |
| 约束明确度 | 10% | 至少 2-3 个可量化的关键约束 |

**进入第 1 阶段门槛：≥ 0.75**

> 门槛说明：第 0 阶段为信息收集阶段，尚未产生正式设计决策，门槛适当降低至 0.75。后续阶段逐步提高（第 1 阶段 0.80，第 2/3 阶段 0.85）。

**低于门槛处理**：标记信息缺口清单，继续交互式提问补充，禁止进入第 1 阶段

---

## 自动推进规则

准入检查清单全部通过 + 置信度 ≥ 0.75 → 自动进入第 1 阶段

否则继续交互式提问
