# devforge-define — 定义性思考

## 概述

需求定义是**长期、深度、迭代**的过程，不是单次对话能完成的。本 skill 采用**多 agent 协作模式**，通过持续多个阶段的思考-研究-发散-评审-完善循环，产出高质量的需求规格文档。

> **内容规范强牵引**：需求交付件的所有内容结构、章节要求、自检清单**以 `.claude/templates/*.md` 为唯一准绳**。本 skill 只负责定义**努力程度、写作风格、质量门槛和流程控制**，不再重复模板中已有的内容要求。

**核心原则**：
1. **模板牵引**：所有产出必须严格遵循对应模板，写前先读模板是铁律
2. **标杆先行**：不研究业界同类产品的需求规格，不允许定义自研 Feature
3. **Actor 驱动**：先识别所有交互角色，再围绕角色定义场景
4. **分层产出**：先产出 Actor-Feature 清单并评审定稿，再进入 Scenario 挖掘
5. **可验收性**：每个 Scenario 可独立验证，验收标准必须量化
6. **长时间迭代**：需求文档需要 3-10 个阶段才能定稿，禁止一次对话定需求

## 通用规范

### 通用术语

#### 问题分级

**核心原则**：问题级别基于对整个产品系统级的影响判定，而非对单个文档所描述的局部影响判定。

| 级别 | 含义 | 判定基准 | 处理方式 |
|------|------|---------|----------|
| **CRITICAL** | 阻塞性问题，会导致系统方向错误 | **系统级影响** | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响局部质量 | **文档/模块级影响** | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | **优化建议** | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | **非功能性** | 可延期 |

**CRITICAL 判定标准**（必须同时满足 3 条）：
1. 影响整个产品系统的核心决策或整体方向
2. 不修正会导致后续工作推倒重来
3. 涉及不可逆或高成本变更

**HIGH/MEDIUM/LOW 判定标准**：
- **HIGH**：影响局部设计或实现质量，但不改变系统整体方向
- **MEDIUM**：优化建议，不影响功能正确性
- **LOW**：格式、命名、注释等非功能性问题

**常见误判纠正示例**：
1. 子系统设计文档内部结构混乱 → **HIGH**（非 CRITICAL）。理由：该子系统可独立重构，不影响其他子系统
2. Feature Scenario 定义不完整 → **HIGH**（非 CRITICAL）。理由：该 Feature 可后续补充，不影响产品整体方向
3. 核心一致性协议选型错误 → **CRITICAL**。理由：系统级方向性决策，影响多个子系统且难以回退

#### 评审结论

Reviewer 产出：
- 问题列表（每个问题标注级别 CRITICAL/HIGH/MEDIUM/LOW 和所属维度）
- 缺陷密度（自动计算：所有问题分数之和 / 标杆数量）

系统自动判断：
- 基于缺陷密度和 CRITICAL 问题数量自动判断是否通过
- 不需要 reviewer 主观给出"通过/不通过"结论

### 标准评审修正循环

各阶段遵循统一的评审修正循环。具体门槛值（缺陷密度阈值、评估对象数计算方式）见各阶段规则。

1. **独立评审**：并行启动 ≥2 个 reviewer，按模板自检清单提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **计算缺陷密度**：所有问题分数之和 / 评估对象数

   问题分值：**CRITICAL=10分, HIGH=3分, MEDIUM=1分, LOW=0.1分**

3. **判断退出条件**（两步判定）：
   - **先判断 CRITICAL 问题**（一票否决）：有 CRITICAL → 进入修正循环；无 CRITICAL → 继续
   - **再判断缺陷密度**：达标 → 成功退出；超标 → 进入修正循环
4. **修正策略**（此为唯一定义，各阶段文件不得重复）

   基于问题数量决定 agent 配置：
   - < 10 处：1 个 agent 串行修正
   - 10-30 处：2 个 agent 分块并行修正 + 主会话整合
   - \> 30 处：按各阶段规则判断是否回退到对应步骤重新执行多 agent 并行发散（回退规则见各阶段，各阶段不重复定义此处 3 档策略本身）
   - 分工原则：问题集中在某个产出对象 → 针对该对象；问题分散 → 按产出对象并行；问题集中在某个维度 → 按维度分工

5. **复核**：修正完成后回到步骤 1（独立评审）
6. **循环上限**：最多 3 次
7. **降级规则**：3 次循环后仍未达标 → 等待人工决策

**自动决策规则**：CRITICAL 必须修正 → HIGH 默认修正 → MEDIUM 评估后决定 → LOW 可延期

### 缺陷密度门槛标定依据

> 此节记录各阶段缺陷密度门槛的设定逻辑，供后续校准和质疑参考。数字不是经验拍脑袋的值，是基于以下假设：

| 阶段 | 门槛 | 估算逻辑 |
|------|------|---------|
| 第 1 阶段（标杆研究） | 2.0 分/标杆 | 总分 ÷ 标杆数 ≤ 2.0。研究报告信息不完整（outside-in），适度容忍在研 gap。例 N=3 累计 ≤ 6 分（1 个 HIGH + 3 个 MEDIUM 已逼近），CRITICAL 单个否决 |
| 第 2 阶段步骤 1（Actor） | 1.5 分/Actor | 总分 ÷ Actor 数 ≤ 1.5。Actor 内容简单（4 字段）、列表短（5-15），且是 Feature 的基础——需要比原 2.0 更严。例 N=10 累计 ≤ 15 分 |
| 第 2 阶段步骤 2（Feature） | 1.5 分/Feature | 总分 ÷ Feature 数 ≤ 1.5。与 Actor 持平：Feature 内容更复杂但对象数更多（15-50），统计上均衡 |
| 第 3 阶段（Scenario） | 1.5 分/**Feature** | 总分 ÷ Feature 数（非 Scenario 数）≤ 1.5。Scenario 粒度更细但评审维度收窄，与 Feature 门槛对齐 |
| 第 4 阶段（维护模式） | 3.0 分/修正点 | 总分 ÷ 修正点数 ≤ 3.0。快速通道范围有限（单文件单章节），N=1 时 ≤ 3 分（1 个 HIGH 到顶，不可叠加 MEDIUM） |

**校准规则**：
- 门槛值被持续验证 10 轮后若无误判（假阳性/假阴性），可定为稳定值
- 任一阶段连续 2 轮以底线通过（刚好不超标，修正 1 次即通过），应考虑下调门槛
- 任一阶段连续 2 轮在第一轮通过（无任何 HIGH 问题），可讨论该阶段是否仍需评审循环

### 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。** Agent 产出后主会话立即写入文件，写入后用 `ls` 验证存在，完成后才能汇报"本阶段完成"。

**例外**：步骤内部的工作指令（如调度方案、盲区分析）属于执行中间件，可按对应阶段规则豁免落盘。

### 主会话职责边界

主会话只负责：调度 agent、搬运产出到文件、整合多 agent 产出（选择/拼接/去重）、检查出口标准。

**禁止**：直接产出深度分析内容、实质性修改 agent 产出、无 agent 执行记录就声称完成、用单 agent 内部发散替代多 agent 独立产出。

**agent 不可用时的降级约束**：当派遣的 agent 因基础设施故障（非超时、非内容纠偏）而不可用时，禁止主会话降级为替代执行研究或评审分析。处理方式：(a) 尝试修复派遣参数（视角、prompt、model 等）重发，(b) 仍不可用则暂停当前交付件并向用户报告阻塞原因，等待人工决策后方可调整执行方案。

### Agent 并发控制

为防止同时启动过多 agent 触发模型服务端限流，主会话在派遣 agent 时必须遵守以下并发限制。

**核心参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `agent.max_concurrent` | **5** | 同时运行的 agent 上限。单一 `Agent` 工具调用计为 1 个并发单元 |

**滑动窗口策略**（细粒度补位，非批次等待）：

1. 主会话统计当前步骤需要启动的 agent 总数 N，维护一个待启动队列
2. **初始启动**：取队列前 `agent.max_concurrent` 个 agent，全部使用 `run_in_background` 并行启动
3. **补位规则**：每当任意一个 agent 完成（收到完成通知），**立即**从队列中取出下一个 agent 启动，始终保持运行中的 agent 数 = min(剩余待启动, `agent.max_concurrent`)
4. **终止条件**：队列为空且所有已启动 agent 均已完成
5. 有依赖关系的 agent 不能同时存在于队列中，必须等上游 agent 完成后才能入队

**执行约束**：
- 各阶段文件中的"并行启动"指令，均受此并发上限约束
- 主会话在派遣前必须统计 agent 总数，若超限则明确写出滑动窗口方案（窗口大小、总 agent 数、预计启动节奏）
- 禁止以"并发限制"为由将本应多 agent 独立产出的步骤压缩为单 agent 执行

### 评审纪要写作规范（由主会话执行）

独立评审完成后，主会话负责将评审结果写入文档末尾的「评审记录」章节，须遵守以下约束：

- **评审状态标记行**（机器可解析，用于启动检测判定阶段过渡）：
  ```markdown
  **评审状态**: ✅ PASS（缺陷密度 ≤ X/评估对象，无 CRITICAL，循环 N/3）
  ```
  仅当评审真正通过（无 CRITICAL + 缺陷密度达标）时才写入 `✅ PASS`。评审未通过或仍在循环中时写入 `❌ FAIL` 或不写标记。
- 评审记录章节 ≤ 200 字
- 仅保留 reviewer 名称、结论、遗留问题数及分类
- 详细评审意见留存对话上下文，不写入正式文档

> 此规范为写作约束，不计入评审判定项。
> **启动检测唯一依据**：`✅ PASS` 标记是阶段过渡判定的唯一可信信号。文件存在但无 PASS 标记 = 该阶段未完成。

## 复杂度判定

| 阶段 | 内容类型 | 复杂度 | agent 配置 |
|------|---------|-------|-----------|
| 第 0 阶段 | 交互式信息收集 | 低 | 主会话直接执行（豁免） |
| 第 1 阶段 | 标杆研究（调研报告） | 高 | 每个标杆 2-3 个 researcher + 整体 2 个 reviewer |
| 第 2 阶段 | Actor-Feature 定义 | 极高 | 多 agent 分步协作（详见 `02-actor-feature.md`） |
| 第 3 阶段 | Scenario 挖掘 | 中等（基准） | 每个 domain 独立判定（综合 Actor 数量、依赖密度、规模与并发、质量关键路径等因素，非二维布尔，详见 `03-scenario.md`）：高复杂度 → ≥3 个 product + ≥2 个 reviewer；中等 → 1-2 个 product + 1-2 个 reviewer |
| 第 4 阶段 | 维护模式 | 低 | 1 个对应角色 agent |

**说明**：
- 第 1 阶段统一采用高复杂度标准，符合 devforge 定位（大型复杂基础软件）
- 第 3 阶段基准为「中等」（Scenario / 用例设计），由主会话基于 domain 多因素综合判断是否上调至「高」（详见 `03-scenario.md`）

## 启动检测

**核心原则**：状态汇报必须以 `ls`/`wc`/`Read` 扫描文件系统的**实时结果**为准。

**检测命令**：
```bash
ls -la .claude/domain-config.yaml
ls -la docs/requirements/
ls -la docs/requirements/reference/
# 若 product-spec.md 存在，读取 Feature 总览表的「归属特性域」列，
# 逐个检查 docs/requirements/<特性域名>.md 是否存在
```

**状态判定**（按执行流程顺序）：

> **判定原则**：文件存在性只是必要条件，**`✅ PASS` 评审状态标记才是阶段过渡的充分条件**。

1. domain-config.yaml 不存在 → 第 0 阶段
2. domain-config.yaml 存在，但 reference/ 下无标杆研究文件（或 < 2 个）→ 第 1 阶段（步骤1-2：标杆选择+调研产出）
3. reference/ 下标杆文件数 ≥ 2：
   - **逐个读取**每个标杆文件末尾，搜索 `**评审状态**:` 标记
   - 任一标杆文件无 `✅ PASS` 标记（或无标记 / `❌ FAIL`）→ **第 1 阶段步骤3（评审修正循环）**
   - 所有标杆文件均有 `✅ PASS` → 第 2 阶段准入通过
4. product-spec.md 不存在 → 第 2 阶段（步骤1-2：Actor 识别+Feature 定义产出）
5. product-spec.md 存在：
   - 读取文件末尾评审状态，无 `✅ PASS` 标记 → **第 2 阶段步骤3（评审修正循环）**
   - 有 `✅ PASS` 标记 → 进入特性域文档检查
6. 逐个检查各特性域文档（名单来自 product-spec.md#Feature 总览的「归属特性域」列）：
   - 文档不存在 → 第 3 阶段（该特性域 Scenario 挖掘）
   - 文档存在但无 `✅ PASS` 标记 → **第 3 阶段步骤2（该特性域评审修正循环）**
   - 所有特性域文档均有 `✅ PASS` 标记 → 第 4 阶段（维护模式）

**行为规则**：主流程未完成时，根据文档状态直接判定并提示唯一的下一步动作，不罗列多个选项让用户选择；主流程已完成后（维护模式），允许给出多个改进方向供用户选择。

## 执行流程

> 主会话进入每个阶段时，必须先读取对应的详细规则文件（`0X-*.md`）。

### 第 0 阶段：产品定位确认

- **准入**：无
- **产出**：`.claude/domain-config.yaml`
- **规则**：`00-positioning.md`

### 第 1 阶段：标杆研究

- **准入**：第 0 阶段完成
- **产出**：`docs/requirements/reference/<product>.md`（至少 2-3 个）
- **规则**：`01-benchmarking.md`

### 第 2 阶段：Actor-Feature 识别与定稿

- **准入**：标杆研究通过评审（无 CRITICAL 问题 + 缺陷密度达标，门槛见「缺陷密度门槛标定依据」）
- **产出**：`docs/requirements/product-spec.md`（Actor-Feature 章节定稿）
- **规则**：`02-actor-feature.md`

### 第 3 阶段：Scenario 挖掘与特性域定稿

- **准入**：Actor-Feature 已定稿（无 CRITICAL 问题 + 缺陷密度达标，门槛见「缺陷密度门槛标定依据」）
- **产出**：`docs/requirements/<feature-domain>.md`
- **规则**：`03-scenario.md`

### 第 4 阶段：需求维护与扩展

- **准入**：所有特性域文档已定稿（无 CRITICAL 问题 + 缺陷密度达标，门槛见「缺陷密度门槛标定依据」）
- **产出**：按需修正现有文档
- **规则**：`04-maintenance.md`

## 红旗信号

- 未做产品定位确认就进入标杆研究
- 标杆研究文档 < 2 个
- Feature 只有名称没有"为什么"论证
- 验收标准无法量化
- 产出文档无评审记录即直接定稿
- Agent 产出内容未写入项目文件
- 需求文档未经独立评审直接定稿
- 高/极高复杂度文档篇幅 < 100 行或低复杂度 > 400 行
- 阶段 N 结束汇报中预演或提前执行了阶段 N+1 的步骤（如阶段 0 出口处列出标杆候选）

## 与用户协作模式

**核心边界**：自动推进与人工确认的切换点按**交付件是否完成闭环**划分，不按"阶段内/阶段间"简单切分。

**三种协作节奏**：
1. **自动推进**：交付件内部，产出→评审→修正循环全自动闭环，不中途询问用户
2. **确认后推进**：交付件闭环后，向用户汇报产出概况，等待确认后再进入下一交付件
3. **等待人工决策**：缺陷密度超过门槛、信息缺失、或需用户做不可逆决策时停止

**阶段衔接规则**（确认后推进的具体约束）：

阶段 N 结束汇报时：
- **必须**：汇报阶段 N 的产出摘要（仅当前阶段的交付件成果）
- **必须**：告知下一阶段名称（如"下一步：进入第 1 阶段（标杆研究）"），等待用户确认推进
- **禁止**：预演下一阶段的具体步骤、工具、候选对象或中间产物
- **禁止**：提前执行下一阶段的任何步骤（包括搜索、筛选、候选列举）
- **禁止**：将下一阶段才应让用户确认的内容提前到当前阶段汇报中展示

> 例如：阶段 0 结束只能汇报 domain-config.yaml 已生成 + 下一步进入阶段 1，不可列出候选标杆或让用户确认标杆选择。

**正常流程**：汇报状态 → 提出执行计划 → 自动执行 → 汇报结果摘要
