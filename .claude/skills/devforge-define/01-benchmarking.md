# 第 1 阶段：标杆研究

**准入条件**：第 0 阶段完成，`.claude/domain-config.yaml` 已生成
**产出文件**：`docs/requirements/reference/<product>.md`（每个标杆一个文件，至少 2-3 个）
**模板**：`.claude/templates/ref-requirements.md`

---

## 执行方式

### 标杆选择（主会话统一处理）

**选择逻辑**：
1. 主会话读取 `domain-config.yaml` 的 `system.primary_type` 和 `system.sub_type`
2. **构造搜索查询并执行网络搜索**：
   - 搜索查询示例：
     - `"{primary_type} {sub_type} benchmark products 2026"`
     - `"best {primary_type} {sub_type} solutions market leaders"`
     - `"{primary_type} open source vs commercial comparison"`
   - 使用 WebSearch 工具搜索，获取最新的行业标杆产品
   - 从搜索结果中提取候选标杆（关注：市场份额、技术成熟度、开源/商业、架构特点）
3. 基于 `quality_attributes.priorities` 调整选择倾向：
   - 一致性优先 → 优先选择强一致性的标杆
   - 性能优先 → 优先选择高吞吐/低延迟的标杆
   - 可用性优先 → 优先选择高可用架构的标杆
4. 向用户展示候选标杆列表（2-5 个），说明选择理由，确认最终选择
   - 基准：2-3 个标杆
   - 复杂系统：3-5 个标杆
   - 确保覆盖不同架构流派或技术路线

### Plan 阶段（任务解构）

> **核心原则**：标杆研究是深度思考过程，统一采用多 agent 协作模式，确保研究质量。

**任务解构**：主会话派遣 **1 个 product agent** 执行任务解构：

1. 读取 `.claude/domain-config.yaml`，理解系统类型、质量属性优先级、架构特征
2. 读取 `.claude/templates/ref-requirements.md`，识别各章节的决策权重分布
3. 读取已确认的标杆列表
4. **制定研究策略**：
   - 基于质量属性确定重点对比维度（如一致性优先 → 重点研究一致性协议的权衡）
   - 基于系统复杂度确定研究深度（如 very-large → 需要研究可扩展性设计）
   - 识别需要被显式质疑的关键假设
5. **决定 agent 配置**：
   - **每个标杆的 researcher 数量**：2-3 个（根据标杆复杂度和系统规模调整）
   - **每个标杆的研究视角切分**：如何让多个 researcher 从不同角度研究同一标杆？
     - 按用户角色切分（如：管理员视角 vs 开发者视角）
     - 按使用阶段切分（如：部署配置 vs 日常运维 vs 故障处理）
     - 按分析维度切分（如：功能完整性 vs 性能特性 vs 运维复杂度）
   - **整体的 reviewer 数量**：2 个（product-reviewer + architect-reviewer）
6. 产出调度方案（并行产出 agent 配置 + 评审 agent 配置 + 整合策略）
7. **额外研究规划约束**：确保分析维度完整（至少覆盖功能与场景、质量属性、运维与集成三个维度）、对比视角独立

主会话原样执行 product 的调度方案。

### 并行产出

1. **每个标杆并行启动 2-3 个 researcher agent**：
   - 各自从不同需求视角独立研究同一标杆（视角由 Plan 阶段确定）
   - 产出目录：`docs/requirements/reference/`
   - 分析视角：**需求/用户/产品视角**，禁止技术分析
   - 模板约束：使用 `.claude/templates/ref-requirements.md`
   - **禁止**多个 researcher 使用同一 prompt 模板、同一分析框架

2. researcher 完成后，**主会话将内容写入文件**

3. **并行启动 2 个 reviewer agent 执行独立评审**（评审所有标杆的研究报告）：
   
   > **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——标杆研究报告在需求体系中的位置（作为 Actor-Feature 定义的输入依据）、重要性（错误标杆选择或分析偏差会导致后续需求定义偏离方向）、可替换性（修正成本评估，标杆研究阶段的修正成本远低于 Feature 定义阶段）。
   
   - `product-reviewer` 重点评审需求视角正确性、三维度分析完整性（功能与场景、质量属性、运维与集成）、量化指标合理性
   - `architect-reviewer` 重点评审研究深度、方法论严谨性、结论与产品定位的匹配度
   - reviewer 需要横向对比多个标杆，确保分析深度和质量标准一致

4. **主会话收集评审结果**：
   - 收集问题列表（每个问题标注级别和所属维度）
   - 计算缺陷密度
   - 进入"出口标准"验证流程

---

## 评审修正循环及出口标准

> **前置条件**：并行产出章节的第 3-4 步已完成（reviewer 已执行评审，主会话已收集评审结果）

### 验证流程

**第 1 步：形式完整性检查**（主会话执行）
- [ ] 标杆数量 ≥ 2
- [ ] 所有标杆文件已落盘并验证存在（用 `ls` 验证）
- [ ] 每个标杆文档末尾添加状态记录（见下方格式）

**不满足 → 直接打回，不进入后续验证**

---

**第 2 步：计算缺陷密度**（主会话执行）

基于并行产出章节收集的评审结果，计算缺陷密度。

**缺陷密度计算**：
```
缺陷密度 = 所有问题的分数之和 / 标杆数量
问题分值：CRITICAL=10分, HIGH=3分, MEDIUM=1分, LOW=0.1分
```

**示例**：
- 3 个标杆，发现问题：
  - 标杆 A：1 个 HIGH-深度 (3分) + 1 个 MEDIUM-广度 (1分)
  - 标杆 B：2 个 MEDIUM-完整性 (2分)
  - 标杆 C：1 个 HIGH-可用性 (3分) + 3 个 LOW (0.3分)
- 总分 = 3 + 1 + 2 + 3 + 0.3 = 9.3 分
- 缺陷密度 = 9.3 / 3 = 3.1 分/标杆

---

**第 3 步：判断退出条件与修正循环**

1. **先判断 CRITICAL 问题**（一票否决）：
   - 有 CRITICAL 问题 → **进入修正循环**
   - 无 CRITICAL 问题 → 继续判断缺陷密度

2. **再判断缺陷密度**：
   - 缺陷密度 ≤ 2.0 分/标杆 → **成功退出**（进入第 2 阶段）
   - 缺陷密度 > 2.0 分/标杆 → **进入修正循环**

3. **修正循环**（自动执行）：
   
   **修正策略**（基于问题数量）：
   - < 5 个问题：1 个 researcher 串行修正
   - 5-15 个问题：2 个 researcher 并行修正
   - 15-30 个问题：3-4 个 researcher 并行修正
   - \> 30 个问题：评估是否回退重做
   
   **分工原则**：
   - 问题集中在某个标杆 → 针对该标杆分配 researcher
   - 问题分散在多个标杆 → 按标杆并行修正
   - 问题集中在某个维度（如深度不足）→ 按维度分工
   
   **修正流程**：
   - 修正 agent（原 researcher）修正问题并更新文件
   - 重新执行评审：回到「并行产出」第 3 步
   - 重新验证：回到本章节第 2 步（跳过第 1 步）
   - 循环上限：最多 3 次

4. **循环失败处理**：
   - 修正 3 次后仍不达标 → **等待人工决策**（决定是否继续修正、回退重做或降级标注）

**成功退出条件**（同时满足）：
- ✅ 无 CRITICAL 问题
- ✅ 缺陷密度 ≤ 2.0 分/标杆

```markdown

- **完成时间**：YYYY-MM-DD

- **遗留问题**：[无 / 列出问题]
```
