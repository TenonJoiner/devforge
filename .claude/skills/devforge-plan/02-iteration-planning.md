# 第 2 阶段：迭代执行计划

**目标**：为当前迭代（如 M1-I1）建立 `iteration-m1-i1.md`

> **一次一迭代**：本次执行只产出当前迭代的执行计划文件。下一迭代在本迭代评审通过后，下次执行 `/df:plan` 时再生成。

## Step 1：迭代上下文收集（强制，不可跳过）

派遣 `project-manager` 向用户确认本迭代的实际投入和约束。**禁止直接沿用 milestone-plan.md 的默认值**。收集完成后，主会话执行信息充分性检查清单确认信息充分性。

> **豁免说明**：本 Step 为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。

**信息充分性检查清单**：

| 必收集项 | 说明 |
|---------|------|
| 本迭代可用人力 | 实际投入人数 |
| 本迭代周期 | 实际周期长度 |
| 上次遗留任务 | I2 起适用 |
| 容量调整 | 如有 |
| 本迭代特殊约束 | 如假期、外部依赖等 |

## Step 2：Agent 调度（多 agent 并行产出）

### Plan 阶段（任务解构）

第 2 阶段复杂度为「高」（关键路径编排 + 多角色交叉验证），建议先解构再发散。在 Agent 调度前，主会话执行轻量级任务解构：

1. **读取 `.claude/templates/plan-iteration.md`**，识别各章节的决策权重分布（迭代目标、任务编排、风险管��各占比）
2. 基于迭代上下文和 milestone-plan.md，确认切分维度、分配视角约束、列出关键假设类型（如迭代预算假设、依赖就绪假设、外部交付假设）、识别评审视角缺口

Plan 阶段由主会话基于上下文直接完成，不派遣额外 agent。

**产出形式**：主会话以结构化文本形式在对话中输出任务解构结果，用户可见，用于透明化调度决策。该输出作为 Step 2 调度 agent 时 prompt 的组成部分。

### 并行产出

主会话收集完迭代上下文并完成 Plan 阶段任务解构后，**并行启动 3 个不同角色的 Agent**，各自侧重不同维度独立产出：

- **project-manager**（关注容量均衡与执行可行性）：侧重 Wave 分组、迭代容量均衡、执行可行性验证
- **architect**（关注技术依赖与关键路径）：侧重技术依赖分析、关键路径识别、并行就绪检查
- **product-manager**（关注优先级与价值对齐）：侧重优先级排序、价值对齐验证、验收标准映射

主会话整合三个 agent 的产出，形成统一的 `docs/iteration-plan/iteration-m<x>-i<y>.md`，写入文件。

### 整合规则（主会话执行）

1. 三个 agent 产出按章节拼接：project-manager 产出里程碑/迭代结构 → architect 产出技术依赖/Wave 分组 → product-manager 产出优先级/价值论证
2. 冲突裁决：技术依赖冲突以 architect 为准；优先级冲突以 product-manager 为准；容量冲突以 project-manager 为准
3. 去重：三个 agent 可能重叠描述同一 proposal，主会话合并描述、保留最完整版本
4. 写入前确认文档按 `.claude/templates/plan-iteration.md` 模板结构组织

## Step 2 出口标准

- [ ] 文档已按 `.claude/templates/plan-iteration.md` 模板结构写入、文件已验证存在
- [ ] 总点数 ≤ 迭代预算，任务分布合理
- [ ] Wave 并行分组反映真实依赖关系
- [ ] 预留 ≥ 20% 缓冲容量
- [ ] 每个 proposal 可追溯到 milestone-plan.md Backlog
- [ ] **CRITICAL 问题处理规则**：产出阶段 CRITICAL 问题由主会话退回 agent 重新产出，阻塞进入 Step 3
- [ ] **评审 agent 判定标准**：3 个评审 agent 均给出"通过"或"有条件通过"（定义见 `common.md` 评审结论三态）
- [ ] **文档版本/状态要求**：文档头部已标注状态为"待评审"

## Step 2 置信度评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 容量均衡 | 30% | 总点数 ≤ 迭代预算，任务分布合理 |
| Wave 依赖正确性 | 30% | 并行分组反映真实依赖关系 |
| 风险缓冲充足 | 20% | 预留 ≥ 20% 缓冲容量 |
| 可追溯性 | 20% | 每个 proposal 可追溯到 milestone-plan.md Backlog |

**进入门槛：≥ 0.85**（迭代执行计划涉及关键路径编排，直接影响开发执行顺序，为不可逆决策）

**低于门槛处理规则**：
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → 等待人工决策

**自动推进规则**：
- 出口标准全部通过 + 置信度 ≥ 0.85 → 自动进入 Step 3（独立评审）
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → 等待人工决策

## Step 3：独立评审（产出后自动执行）

迭代执行计划文件产出后，主会话**立即并行启动** `architect-reviewer` + `pm-reviewer` + `project-manager-reviewer` 执行**独立评审**。

> **评审视角**：三个评审 agent 覆盖技术/产品/项目管理三个独立视角：
- `architect-reviewer`：重点评审 Wave 分组、技术依赖、关键路径
- `pm-reviewer`：重点评审优先级与价值对齐、验收标准映射
- `project-manager-reviewer`：重点评审容量均衡、执行可行性、迭代预算合理性

### 评审修正循环

1. 主会话汇总三个评审 Agent 的结论
2. 修正阶段 agent 配置（根据修正工作量选择）：
   - 少量修正（＜5 处，单点调整）：1 个 `project-manager` Agent 执行修正
   - 中量修正（5-15 处）：**2 个不同角色 Agent 分块并行修正** + 主会话整合（architect 负责技术类修正：依赖关系、接口契约、技术风险、并行就绪检查；project-manager 负责计划编排类修正：Wave 分组、容量分配、缓冲容量、风险应对）
   - 大量修正（＞15 处，接近重写）：**回退到 Step 2 重新执行多 agent 并行发散**
3. **修正完成后，重新启动独立评审**（复核）
4. **循环终止条件**：所有 CRITICAL 和 HIGH 问题已修正，且所有评审 Agent 均给出"通过"或"有条件通过"
5. **评审修正循环上限**：最多 3 次
6. **降级处理规则**：3 次循环后仍有未达标项 → 降级置信度并标注"待后续补充"，或**等待人工决策**
7. HIGH/MEDIUM/LOW 级别问题记录但不阻塞

## Step 3 出口标准

- [ ] 所有 CRITICAL 问题已修正（阻塞性问题必须修正，否则不得推进）
- [ ] 所有 HIGH 问题已评估：接受修正 / 接受延期（需记录理由）/ 拒绝（需说明原因）
- [ ] MEDIUM/LOW 问题记录但不阻塞推进，纳入后续迭代跟踪
- [ ] `architect-reviewer`、`pm-reviewer` 和 `project-manager-reviewer` 均给出"通过"或"有条件通过"（定义见 `common.md`）
- [ ] Wave 分组正确反映迭代内 proposal 间的依赖关系
- [ ] 总点数 ≤ 迭代预算，高风险任务 ≤ 50%
- [ ] 文档版本已更新（在文件头部标注版本号和最后更新时间）
