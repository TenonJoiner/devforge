# 通用规范

本文件定义 product/plan skill 各阶段共享的通用规范。

## 通用术语

以下术语在本 skill 各阶段中统一使用，不重复定义。

### 问题分级

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **CRITICAL** | 阻塞性问题，会导致方向错误 | 必须修正，阻塞推进 |
| **HIGH** | 重要缺陷，显著影响质量 | 默认接受修正 |
| **MEDIUM** | 一般问题，有改善空间 | 评估后决定 |
| **LOW** | 轻微问题，可延后处理 | 可延期 |

### 评审结论三态

- **"通过"**：无未处��的 CRITICAL/HIGH 问题，或所有问题已修正/已评估
- **"有条件通过"**：存在已评估并记录处理方案的 HIGH 问题，无未处理的 CRITICAL 问题
- **"不通过"**：存在未处理的 CRITICAL 问题，或存在未评估的 HIGH 问题

### 修正工作量分级

| 级别 | 范围 | agent 配置 |
|------|------|-----------|
| 少量修正 | < 5 处，单点调整 | 1 个对应角色 agent |
| 中量修正 | 5-15 处，结构性调整 | 2 个 agent 分块并行修正 + 主会话整合 |
| 大量修正 | > 15 处或接近重写 | 回退到对应产出阶段重新执行多 agent 并行发散 |

## 文档写入铁律

**每一阶段迭代的产出必须写入项目文件，禁止仅在对话中输出。**

**执行规则**：
1. Agent 产出内容后，**主会话必须立即将内容写入对应文件**
2. 写入完成后才能向用户汇报"本阶段完成"
3. 验收检查必须包含"文件是否已写入"这一项
4. **写入后必须验证文件存在**（`ls` 确认文件已落盘）

## 评审与修正记录规范

写入正式交付文档的评审记录和修正记录必须简略，避免喧宾夺主。

- **评审记录章节 ≤ 200 字**：文档中「评审记录」章节总篇幅不超过 200 字，避免喧宾夺主
- **聚焦结论**：仅保留各评审 agent 名称、通过结论（通过/有条件通过）、遗留问题数及简要分类（如"2 MEDIUM：容量估算、风险缓冲"）
- **详细评审意见不写入文档**：具体质疑点、分析过程留存于对话上下文，不写入正式交付文档正文
- **修正记录同样简略**：聚焦"改了什么"，控制在 200 字以内，不展开论证过程

## 标准评审修正循环骨架

1. **独立评审**：并行启动 ≥2 个 reviewer，视角差异化，按模板自检清单提出 CRITICAL/HIGH/MEDIUM/LOW 分级问题
2. **修正**：按修正工作量分级选择 agent 配置执行修正
3. **复核**：修正完成后重新启动独立评审
4. **循环上限**：最多 3 次
5. **终止条件**：所有 CRITICAL 和 HIGH 问题已修正 + 所有 reviewer 均给出"通过"或"有条件通过"
6. **降级规则**：3 次循环后仍未达标 → 降级置信度并标注"待后续补充"，或等待人工决策

## 主会话职责边界

主会话在多 agent 协作中只承担以下职责，**超出即越界**：

1. **调度**：决定哪个阶段派哪些 agent、什么时机并行
2. **搬运**：将 agent 产出写入文件（仅搬运，不编辑内容）
3. **整合**：当多个 agent 产出需要合并时，选择/拼接/去重（不改写核心结论）
4. **门禁**：检查出口标准、置信度阈值、agent 执行记录完整性

### 主会话禁止行为

- 直接编写计划文件内容（应委托 project-manager / architect / product-manager Agent）
- 跳过里程碑规划直接进入迭代执行计划
- 评审未通过就推进到下一阶段
- CRITICAL 问题修正后未重新评审
- 未收集项目背景信息就开始里程碑规划
- 未确认迭代上下文就开始迭代执行计划
- 在无 agent 执行记录的情况下声称某阶段/Step 已完成

## 用户协作节奏

| 模式 | 说明 |
|------|------|
| **自动推进** | 阶段内产出→评审→修正自动执行，循环结束后汇报结果 |
| **确认后推进** | 交付件/阶段完成后汇报，等待用户确认再进入下一阶段/Step |
| **等待人工决策** | 置信度低于阈值或信息缺失时停止，等待用户决策 |

## 产出质量标准

- **每阶段产出后，主会话立即并行启动评审 Agent（`architect-reviewer` + `pm-reviewer` + `project-manager-reviewer`）执行独立评审**（不等待用户确认），未经评审修正循环通过的计划不得推进执行
- Backlog 中每个 proposal 必须是可独立交付的最小有价值单元
- 关键路径上的 proposal 必须优先安排
- **集成测试必须独立立项**，不可合并到特性 proposal 中
- 单个 proposal 不得超过 4 周（XL 必须拆解）
- 高风险任务占比 ≤ 40-50%
- **Wave 是迭代内部的并行编排单位**，不用于跨迭代编排
- **第 1 阶段 Backlog 只列清单，不分配到具体迭代**（迭代分配在第 2 阶段完成）
