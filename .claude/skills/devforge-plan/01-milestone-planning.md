# 第 1 阶段：里程碑规划 + Backlog

**目标**：建立里程碑计划文件 `docs/iteration-plan/milestone-plan.md`。

## 步骤 1：项目背景收集（强制，不可跳过）

派遣 `project` 通过交互式提问向用户收集项目背景信息。**禁止自行假设任何团队/时间参数**。收集完成后，主会话执行信息充分性检查清单确认信息充分性。

> **豁免说明**：本步骤为纯交互式信息收集阶段，无正式文档产出，不适用出口标准与评审修正循环。

**信息充分性检查清单**（确认所有必收集项已回答或已显式标注为假设值）：

| 必收集项 | 说明 |
|---------|------|
| 团队规模 | 全职/兼职人数 |
| 全职/兼职比例 | 实际可用人力 |
| 预期时间跨度 | 项目总周期 |
| 迭代周期偏好 | 1 周 / 2 周 / 其他 |
| 历史 Velocity | 如有历史数据 |
| 外部约束 | 依赖、假期、里程碑等 |

## 步骤 2：Agent 调度（多 agent 并行产出）

### Plan 阶段（任务解构）

第 1 阶段复杂度为「极高」（结构化规划 + 跨子系统依赖 + 不可逆决策），必须先解构再发散。

**任务解构**（Plan agent 工作流程）：
1. 派遣 **1 个 project agent** 读取 `.claude/templates/plan-milestone.md`
2. 识别各章节的决策权重分布（里程碑划分、依赖分析、Backlog 组织各占比）
3. 基于项目背景和需求/架构文档，分析系统上下文
4. 决定 agent 数量（基于极高复杂度下限，≥5）
5. 设计切分维度（按职能侧重：结构/依赖/价值/数据/容量），分配视角约束
6. 列出关键假设类型（团队容量与 Velocity 假设、跨子系统接口就绪假设、外部依赖交付假设）
7. 识别评审视角缺口
8. 产出调度方案，包含三个完整部分：
   - (1) 并行产出 agent 配置（角色 + 视角约束 + 负责维度 + 输入材料）
   - (2) 评审 agent 配置（基于盲区分析）
   - (3) 整合策略（合并规则 + 冲突处理）
9. 产出以结构化文本形式在对话中输出

**Plan 禁止事项**：
- 不产出深度分析内容
- 不预设最终结论或倾向性方案

**主会话职责**：原样执行 project agent 的调度方案，不做修改。若对方案有疑问，与用户确认或重新派遣 plan agent。

**完成检查清单**（由 plan agent 确认）：
- [ ] 模板已读取并识别决策权重分布
- [ ] 切分维度已确认
- [ ] 视角约束已确认
- [ ] 关键假设类型已确认
- [ ] 评审视角缺口已确认（方向性，不预填具体值）

### 并行产出

主会话收集完项目背景并完成 Plan 阶段任务解构后，**并行启动 5 个不同角色的 Agent**，各自侧重不同维度独立产出：

- **project**（关注里程碑结构与 Backlog 组织）：侧重里程碑划分、proposal 清单、优先级排序、Backlog 结构设计
- **architect**（关注技术依赖与风险识别）：侧重技术依赖分析、关键路径识别、技术风险评估、接口冻结点确定
- **product**（关注价值对齐与优先级）：侧重价值对齐、优先级排序、验收标准映射、用户价值论证
- **researcher**（关注历史数据与标杆参考）：侧重历史 Velocity 数据分析、标杆项目规模参考、估算基准建立
- **project**（关注容量与风险管控）：侧重容量预算、缓冲策略、风险应对预案、资源分配合理性验证

主会话整合五个 agent 的产出，形成统一的 `docs/iteration-plan/milestone-plan.md`，写入文件。

> **整合成本评估**：5 个 agent 产出同一文档，各 agent 按职能侧重分块（project 负责结构/Backlog，architect 负责依赖/风险，product 负责价值/优先级，researcher 负责数据/基准），产出内容按章节天然分区，冲突主要集中在交叉评估项（如估算数值不一致）。冲突按职能裁决：技术依赖以 architect 为准，优先级以 product 为准，容量估算以 project 为准。并行收益（5 个独立视角 vs 单一视角的盲区减少）显著大于整合开销（去重 + 按职能裁决冲突）。

## 步骤 2 出口标准

- [ ] 文档已按 `.claude/templates/plan-milestone.md` 模板结构写入、文件已验证存在
- [ ] 里程碑划分符合技术依赖顺序和产品目标
- [ ] Backlog 覆盖所有已知 Feature，无遗漏关键路径
- [ ] 故事点与团队 Velocity 匹配
- [ ] 已标注关键风险及缓解策略
- [ ] **CRITICAL 问题处理规则**：产出阶段 CRITICAL 问题由主会话退回 agent 重新产出，阻塞进入 步骤 3
- [ ] **评审 agent 判定标准**：3 个评审 agent 均给出"通过"或"有条件通过"（定义见 `common.md` 评审结论三态）
- [ ] **文档版本/状态要求**：文档头部已标注状态为"待评审"

## 步骤 2 置信度评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 里程碑合理性 | 25% | 里程碑划分符合技术依赖顺序和产品目标 |
| Backlog 完整性 | 25% | 覆盖所有已知 Feature，无遗漏关键路径 |
| 时间估算合理性 | 20% | 故事点与团队 Velocity 匹配 |
| 风险识别充分性 | 20% | 已标注关键风险及缓解策略 |
| 文档结构合规性 | 10% | 已按模板结构写入 |

**进入门槛：≥ 0.85**（里程碑划分为不可逆决策，门槛 ≥ 0.85）

**低于门槛处理规则**：
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → 等待人工决策，禁止进入 步骤 3

**自动推进规则**：
- 出口标准全部通过 + 置信度 ≥ 0.85 → 自动进入 步骤 3（独立评审）
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → 等待人工决策

## 步骤 3：独立评审（产出后自动执行）

里程碑计划文件产出后，主会话**立即并行启动** `architect-reviewer` + `product-reviewer` + `project-reviewer` 执行**独立评审**。

> **主会话职责**：在派遣评审 agent 的 prompt 中必须注入文档的系统上下文——本文档在项目规划中的位置（里程碑计划，定义整体迭代节奏和 Backlog 优先级）、重要性（影响后续所有迭代的执行计划、资源分配和团队承诺）、可替换性（修正成本评估）。

> **评审视角**：三个评审 agent 覆盖技术/产品/项目管理三个独立视角：
- `architect-reviewer`：重点评审技术依赖、关键路径、风险识别
- `product-reviewer`：重点评审价值对齐、优先级、验收标准映射
- `project-reviewer`：重点评审计划编排、里程碑划分、团队容量匹配、Velocity 合理性、可交付性

### 评审修正循环

1. 主会话汇总三个评审 Agent 的结论
2. **回退判断**（按 common.md 标准评审修正循环中的回退决策表执行）：CRITICAL ≥3 或 (CRITICAL 1-2 且为框架性错误) → 回退重做；否则进入修正
3. 修正阶段 agent 配置（根据修正工作量选择）：
   - 少量修正（<10 处，单点调整）：1 个 `project` Agent 执行修正
   - 中量修正（10-30 处）：**3 个不同角色 Agent 分块并行修正** + 主会话整合（architect 负责技术类修正：依赖关系、技术风险、关键路径、接口冻结点；project 负责计划编排类修正：时间估算、容量规划、Wave 分组、里程碑划分；researcher 负责数据类修正：Velocity 校准、历史数据引用、估算基准调整）
   - 大量修正（>30 处，接近重写）：**回退到 步骤 2 重新执行多 agent 并行发散**（5 个 agent）
4. **修正完成后，重新启动独立评审**（复核）
5. **循环终止条件**：所有 CRITICAL 和 HIGH 问题已修正，且所有评审 Agent 均给出"通过"或"有条件通过"
6. **评审修正循环上限**：最多 3 次
7. **降级处理规则**：3 次循环后仍有未达标项 → 降级置信度并标注"待后续补充"，或**等待人工决策**
8. HIGH/MEDIUM/LOW 级别问题记录但不阻塞

## 步骤 3 出口标准

- [ ] 所有 CRITICAL 问题已修正（阻塞性问题必须修正，否则不得推进）
- [ ] 所有 HIGH 问题已评估：接受修正 / 接受延期（需记录理由）/ 拒绝（需说明原因）
- [ ] MEDIUM/LOW 问题记录但不阻塞推进，纳入后续迭代跟踪
- [ ] `architect-reviewer`、`product-reviewer` 和 `project-reviewer` 均给出"通过"或"有条件通过"（定义见 `common.md`）
- [ ] 文档版本已更新（在文件头部标注版本号和最后更新时间）

## 步骤 3 置信度评分

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 独立评审质量 | 40% | 所有 CRITICAL/HIGH 问题有明确回应 |
| 修正完整性 | 30% | 修正后无新增冲突或遗漏 |
| 评审结论一致性 | 30% | 三个评审结论不矛盾 |

**进入门槛：≥ 0.85**

**低于门槛处理规则**：
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → 等待人工决策

**自动推进规则**：
- 步骤 2 和 步骤 3 出口标准全部通过 + 置信度 ≥ 0.85 → **确认后推进**（向用户汇报结果摘要，建议执行 `/df:plan` 进入第 2 阶段）
- 置信度 0.80-0.84 → 自动再修正一次
- 置信度 < 0.80 → **等待人工决策**

**下一步**：评审通过后，请执行 `/df:plan` 进入第 2 阶段。
