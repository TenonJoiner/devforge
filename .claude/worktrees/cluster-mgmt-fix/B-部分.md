## Feature: 提供滚动升级与版本灰度发布

### Feature 价值论证

**用户痛点**：
2-5 人运维团队管理数十到数百个边缘+云端节点，时序数据库承载 7×24 高频写入，几乎不存在停机维护窗口；版本一旦上线便要求"在不停写、不丢点的前提下"完成升级。同时新版本若存在缺陷（写入路径回归、内存泄漏、协议兼容性问题），全量升级会让缺陷在 100% 节点同时暴发。

**现有方案不足**：
- **手动逐节点升级**（自建脚本 / Ansible playbook）：耗时长（百节点级数小时），人工切流易出错，缺乏自动副本检查，曾出现"升级中两个副本同时离线"的事故；
- **InfluxDB Enterprise** 的滚动升级仅在管理面层提供，节点状态机和副本切换需 DBA 手动协调，且不内置灰度比例；
- **TimescaleDB（多节点）** 升级需要先断开 access node 与 data node 的连接，业务可见中断；
- **K8s StatefulSet 默认滚动升级**：按 Pod 顺序替换，未感知数据库副本健康度，可能在副本未追赶完成前就升级下一个，触发可用性事故。

**本 Feature 价值**：
将"升级"建模为**受副本健康度门控**的状态机：每个节点升级前先确认其分片在其他副本上完整可用，再进入"流量摘除→优雅关闭→镜像替换→数据恢复→健康验证→重新挂载流量"的子流程；同时支持**按比例灰度**，将新版本承担的读流量限制在配置比例内，作为缺陷爆炸半径控制手段。结合 Feature 1 的故障检测和 Feature 4 的优雅关闭信号，将"高风险大动作"拆为"可暂停、可观察、可回滚"的若干小步。

**成功指标**（SMART）：
- 滚动升级期间集群写入成功率 ≥ 99.9%（按 5 分钟滑动窗口统计）；
- 单节点升级总耗时 ≤ 120s（含 ≤ 60s 优雅关闭 + ≤ 60s 启动恢复+ readiness 切换）；
- 升级期间任意时刻在线副本数 ≥ min(配置副本数, 集群可用副本数 - 1)，即不会让任一分片副本数因升级动作降至最小可用值以下；
- 灰度节点在 5 分钟滑动窗口内接收的读请求比例 ≤ 配置比例 + 5% 偏差。

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 运维工程师在 3 副本集群上触发全量滚动升级，系统按节点逐一替换 | 升级期间集群写入成功率 ≥ 99.9%（5 分钟滑动窗口）；任意时刻每个分片在线副本数 ≥ 2 | 自动化 |
| 2 | 运维工程师配置灰度比例 20%，仅 1 个节点升级到新版本承担读流量 | 灰度节点在 5 分钟滑动窗口内接收的读请求比例 ≤ 20% + 5% 偏差；写入路径不参与灰度（仍以分片主副本为准） | 自动化 |
| 3 | 运维工程师观察灰度节点 24 小时无回归后扩大至 100% 完成全量升级 | 全量完成后所有节点版本号一致；升级总时长 ≤ 节点数 × 120s + 调度间隔 | 自动化 |
| 4 | 单个节点升级中启动失败（新版本进程退出码非 0 / readiness 长时间未达 ready），系统自动回滚该节点 | 故障节点 ≤ 180s 内回滚到旧版本镜像并恢复服务；其余节点保持原计划 | 自动化 |
| 5 | 升级预检发现新旧版本数据格式或协议不兼容（如 WAL 编码版本跳跃） | 预检在停止任何旧版本节点前完成；输出不兼容项清单（字段级）并阻止继续；运维工程师确认后才能进入"强制迁移模式" | 自动化 |
| 6 | 升级进行到一半时另一个未升级节点意外宕机（命中 Feature 1 故障检测） | 升级流程在 ≤ 5s 内自动暂停（≤ 2.5 个心跳周期，默认心跳 2s）；待 Feature 1 完成副本切换且集群恢复健康后，运维工程师可一键续作或终止 | 自动化 |
| 7 | 升级期间集群发生网络分区，少数派侧的待升级节点未能与多数派达成升级协调一致 | 少数派侧不启动镜像替换；分区恢复后由多数派重新下发升级指令；任意时刻不出现"分区两侧同时升级同一分片所有副本"的情况 | 自动化 |
| 8 | 节点升级中 WAL 数据损坏（启动时 CRC 校验失败） | 节点不进入 ready 状态；系统标记为"需要从健康副本恢复"，不影响已升级节点；告警上报到告警规则引擎 | 自动化 |
| 9 | 优雅关闭期间已确认写入数据完整性 | 升级关闭窗口（默认 terminationGracePeriodSeconds=60s）内已确认写入在 WAL 中持久化率 100%；新主副本上读取一致性等同 Feature 1 副本切换语义 | 自动化 |

## Feature: 控制节点级资源配额与过载保护

### Feature 价值论证

**用户痛点**：
边缘节点资源严重受限（典型 4C8G + 100GB SSD），常与采集进程、推理进程共部署。一旦现场某个异常数据源（如 PLC 故障产生数据风暴、设备时间错乱重传历史数据）触发流量尖峰，单节点很快耗尽内存或磁盘，OS 级 OOM Kill 把整个数据库进程杀掉，**该节点本来还正常的设备的数据也全部写入失败**。运维团队事后抢救时发现 WAL 已被截断，丢点。

**现有方案不足**：
- **Linux cgroup**：只能限制 OS 级 CPU/内存/IO，对数据库无业务感知，触发硬限时通常已经晚了（无降级窗口，直接 kill）；
- **InfluxDB enterprise** 提供 `max-concurrent-write-limit`，但只是单一维度且限速后客户端无明确背压协议，客户端只能通过 5xx 推断；
- **TimescaleDB** 依赖 PostgreSQL 连接池和 OS 限制，对时序写入毛刺不敏感；
- 多数现有方案在过载触发时**没有给写入方明确的退避指引**，写入方继续重试反而加剧雪崩。

**本 Feature 价值**：
在数据库层面建立"业务感知 + 多维水位 + 主动背压 + 客户端退避协议"四位一体的过载保护机制：
1. 配额维度覆盖内存、磁盘、并发查询、WAL 积压；
2. 接近阈值（如 80%）即触发**软背压**（写入方收到带退避建议的响应），到硬阈值（如 95%）才进入**拒绝模式**；
3. 通过响应字段告知写入方"建议退避时长 + 是否可重试"，让 IoT 设备群 / 应用服务集群在客户端 SDK 层做指数退避，而非盲目重试；
4. 配额可在线动态调整，配置变更复用 Feature 7 下发链路。

**成功指标**（SMART）：
- 过载保护触发后，**核心写入链路**（已建立时间线的设备）在 5 分钟内的可用性 ≥ 99%（不被新流量冲击挤兑）；
- 从指标越过阈值到背压信号到达写入方 ≤ 5s（含 ≤ 2s 检测 + ≤ 3s 信号传播）；
- 背压触发后节点持续稳定运行，进程不被 OS OOM Kill，核心写入链路不中断（72h 压力测试观察）；
- 过载解除后，写入方在 ≤ 60s 内自动回归正常速率（无需人工介入）。

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|
| 1 | 运维工程师配置内存 80% 软水位 / 95% 硬水位，写入压力增长触发软背压 | 越过 80% 后 ≤ 5s 内写入响应携带退避建议字段；越过 95% 后新写入收到拒绝响应；背压触发后节点持续稳定运行，核心写入链路不中断 | 自动化 |
| 2 | 运维工程师配置磁盘配额 90%，达到阈值后系统拒绝新建时间线 | 新建时间线请求返回明确错误码（指示"配额超限-不可重试"）；已有时间线的写入不受影响（成功率与水位前一致） | 自动化 |
| 3 | 运维工程师配置并发查询上限，超额查询被快速拒绝 | 超出配额的查询在 1s 内收到拒绝响应，响应明确指示可重试（含建议退避窗口） | 自动化 |
| 4 | 写入速率突增导致 WAL 积压超过配额，背压向写入方传播 | 背压信号 ≤ 5s 内通过响应到达写入方（IoT 设备群 / 应用服务集群） | 自动化 |
| 5 | 过载触发后写入方（IoT 设备群 / 应用服务集群）依据响应进行指数退避，过载解除后自动回归正常速率 | 客户端 SDK 在收到退避建议后实际下一次重试间隔 ≥ 建议退避时长；过载解除后 ≤ 60s 内写入速率恢复到背压前 90% 以上；恢复过程不需要人工介入 | 自动化 |
| 6 | 配额值被设置为不合理的极低值（如内存配额 1MB） | 节点启动时校验拦截并拒绝；继续使用上一次有效配置；告警上报"配置非法" | 自动化 |
| 7 | 过载保护触发期间运维工程师动态调高配额（复用 Feature 7 下发链路） | 配额变更 ≤ 10s 内在该节点生效；批量场景下 ≤ 30s 内全部目标节点生效（与 Feature 7 一致）；背压随即解除 | 自动化 |
| 8 | 节点故障导致分片副本数下降，存活副本承担额外负载触发过载 | 系统优先保障已确认写入的可用性；仅对新增写入进行背压；不影响 Feature 1 的副本切换流程 | 自动化 |
| 9 | 网络分区期间，少数派侧节点因无法与多数派同步导致 WAL 持续积压 | 积压超过配额时少数派侧主动停止接收写入并向写入方返回"分区中-请重试到其他可达节点"；分区恢复后 ≤ 60s 内自动恢复接收 | 自动化 |
| 10 | 升级回滚（Feature 5 #4）期间节点资源占用瞬间升高 | 配额阈值不被升级动作豁免；若触发软水位，按正常背压流程处理；升级回滚不会因为配额限制而失败（升级流量走管理通道，不计入业务配额） | 自动化 |
| 11 | 节点 WAL 文件损坏导致背压组件无法读取积压指标 | 背压组件降级为按写入速率估算积压；告警上报"指标降级"；不会因为指标缺失而完全关闭背压（fail-safe 默认开启保护） | 自动化 |

## Feature: 提供集群配置的集中管理与动态下发

### Feature 价值论证

**用户痛点**：
数十到数百个边缘节点分散部署在工厂车间、变电站、风场，运维工程师无法逐一 SSH 改配置；现场常见"同样的配置参数，A 站点改了 B 站点忘了改"的不一致问题；某些参数希望调整后立即生效（写入批量大小、采样压缩级别），但传统方案要求重启服务，重启等于"该节点写入中断 30s+"，不可接受。

**现有方案不足**：
- **Ansible/Salt**：本质是命令分发器，需要节点常驻 SSH，弱网下成功率低；不支持"动态生效"，多数场景需重启服务；缺少配置版本化和灰度回滚；
- **etcd/Consul** 自建配置中心：通用配置中心不感知数据库语义，下发非法配置会让节点崩溃；无内置审计；
- **InfluxDB / TimescaleDB**：配置需要 reload SIGHUP 或重启，且不支持按节点标签做差异化下发；
- **缺乏防篡改链路**：配置在传输和落盘中可能被中间环节修改，安全审计员无法追溯到"谁在什么时间下发了什么"。

**本 Feature 价值**：
建立**版本化 + 标签化 + 校验前置 + 签名传输 + 动态生效 + 审计留痕**的配置下发体系：
1. 云端配置中心持有所有版本，每次提交生成新版本号，支持 diff 对比；
2. 按节点标签（如 region=east, role=edge-leader）做差异化下发；
3. 下发前在云端做 schema 校验，节点收到后做二次校验，非法配置不会被加载；
4. 下发负载由 KMS 签发签名链，节点验签通过后才生效，防止中间人篡改；
5. 大部分参数支持热加载（不重启），少数参数显式标记"需重启"并由调度系统选择安全窗口；
6. 一键回滚到任一历史版本，所有操作记录推送到安全审计员。

**成功指标**（SMART）：
- 配置变更从云端提交到 100 个目标节点生效 ≤ 30s（P95）；
- 配置下发成功率 ≥ 99.5%（含弱网节点的合理重试）；
- 非法配置阻断率 = 100%（任何一道校验失败都不会加载）；
- 所有下发操作 100% 可在审计日志中追溯到"谁、何时、下发到哪些节点、配置 diff、是否回滚"。

### Scenario 清单

| # | 场景 | 验收标准 | 可测试性 |
|---|------|---------|---------|

## Feature: 支持边缘节点批量部署与初始化

### Feature 价值论证

**用户痛点**：
新项目（如新工厂上线、新风场投产）需要一次性部署数十个边缘节点，每个节点都要经历"OS 准备→数据库二进制安装→配置文件生成→密钥下发→启动→注册→健康验证"七步。手动操作每节点 30-60 分钟，50 节点 = 25-50 人时；操作过程中漏配某个参数会让节点上线后表现异常，需要返工。

**现有方案不足**：
- **通用部署工具（Ansible/Terraform）**：缺少数据库特定的健康验证逻辑（"启动 ≠ 真正可用"），无法判定 WAL 是否健康、注册是否成功、首批数据写入是否通畅；
- **自建脚本**：通常只覆盖正常路径，遇到 OS 版本不达标、磁盘格式错误、网络丢包等异常时无标准处理流程；
- **部署完成后缺乏"批量健康基线"**，运维工程师只能逐节点 ssh 检查，规模化时不可行；
- **凭证签发与节点身份绑定** 通常通过预共享密钥手动配置，签发流程不可审计。

**本 Feature 价值**：
提供**声明式部署清单 + 预检前置 + 自动化串联 + 健康基线验证 + 失败重试 + 审计留痕**的端到端能力：
1. 运维工程师提交 YAML 部署清单（节点列表、配置模板、差异化覆盖），其余流程自动化；
2. 预检阶段在不修改目标节点状态的前提下做 OS、磁盘、网络、时间同步检查，输出不达标清单；
3. 自动串联安装→配置→KMS 签发凭证→启动→注册→首条写入验证→首条读取验证；
4. 失败节点自动重试（指数退避，最多 N 次），仍失败的节点标记 failed 并输出原因；
5. 部署后输出"健康基线报告"（含所有节点的版本、配置 hash、初始指标）。

**成功指标**（SMART）：
- 50 节点批量部署总耗时 ≤ 30 分钟（P95）；
- 首次部署成功率 ≥ 95%，自动重试后 ≥ 99%；
- 部署后健康基线报告覆盖率 = 100%（每个节点都有报告）；
- 失败节点输出的原因可被运维工程师据其直接定位问题，不需要再 ssh 排查（运维测可用率 ≥ 90%）。
