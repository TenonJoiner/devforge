## 设计原则

<定义指导系统设计的顶层原则，所有子系统设计与技术选型必须服从以下原则>

| 原则 | 内涵 | 在分布式存储场景中的体现 |
|------|------|------------------------|
| 数据一致性优先 | 一致性是存储系统的生命线，任何以牺牲一致性换取性能或可用性的决策都必须经过显式评审 | 写操作在返回成功前，必须确保数据已持久化到法定副本数；不允许在故障场景下出现静默数据丢失 |
| 故障常态假设 | 任何组件随时可能故障，设计必须面向故障恢复而非故障避免 | 所有网络调用必须有超时、重试和降级策略；单节点故障不得影响集群可用性 |
| 性能可预测 | 追求稳定且可预期的尾延迟，而非单纯峰值吞吐 | P99 延迟是核心验收指标；热路径上禁止全局锁和动态内存分配 |
| 不重复造轮子 | 优先使用经过生产验证的成熟方案、开源组件或学术成果，只在有明确差异化优势时自研 | 一致性协议优先参考 Raft/Paxos 成熟实现；序列化优先采用 protobuf/flatbuffers |
| 可演进性 | 架构必须支持在线升级、向后兼容和灰度扩展，避免因版本升级导致长期停服 | 协议和持久化格式必须设计版本字段；支持滚动升级和回滚 |

### 故障假设

<明确定义系统在设计和测试时必须容忍的故障类型与范围>

| 故障类型 | 假设 | 系统级应对策略 |
|----------|------|---------------|
| Crash-stop | 任意进程随时崩溃，重启后数据可能丢失或损坏 | WAL + 定期 checkpoint；重启后从 WAL 恢复状态 |
| Network partition | 网络延迟、丢包、分区不可避免，无法区分节点崩溃与网络中断 | 基于心跳的超时检测 + 租约机制；分区时遵循预设的 CP/AP 策略 |
| Disk corruption | 磁盘可能因硬件故障产生位翻转或扇区损坏 | 端到端校验和（CRC32C / xxHash64）；定期 scrub 扫描 |
| Silent data corruption | 数据在传输或存储过程中可能静默损坏而不触发 IO 错误 | 每次网络传输和落盘必须携带校验和；读取时验证校验和 |
| Byzantine fault | `□ 假设存在 / □ 不假设存在`，若存在需说明阈值（如 f < N/3） | <若考虑拜占庭故障，说明对应的容错机制，如 BFT 变体> |
