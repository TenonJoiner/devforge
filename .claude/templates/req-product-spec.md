## Actors

> **写什么**：系统涉及的所有 Actor 定义。Actor = 产品黑盒视角下与系统交互的**外部**实体，含 3 类：人类 / 外部系统 / 时间触发源。每个 Actor 定义类型、职责、交互方式、关注点。通过权力/利益矩阵指导需求优先级。
> **不写什么**：**禁止**列出内部组件（写入网关、查询协调器、存储引擎等架构内部模块），这些归 `docs/architecture/.../design.md`。不在此处定义具体 Feature 或 Scenario（留给特性域文档）。
> **最低深度**：至少定义 2 个 Actor（至少含 1 个人类角色和 1 个外部系统角色）；权力/利益矩阵覆盖所有关键 Actor。

### Actor 严格定义与判定决策树

**Actor = 产品黑盒视角下，与**运行中的系统**发生**持续、直接交互**的外部实体。** 类型限定为 3 类：

| 类型 | 判定条件 | 典型示例 |
|------|---------|---------|
| **人类** | 物理存在的人 | 数据分析师、SRE、DBA、合规审计员 |
| **外部系统** | 系统边界外、独立运行的另一个系统/设备 | Grafana、传感设备、KMS、K8s |
| **时间触发源** | 时间事件 + **驱动了一个有用户价值的独立用例** | TTL 触发自动清理、定时降采样、定时备份 |

**判定决策树**（每个候选必须走完）：

```
候选 X 是 Actor 吗？
  ├─ 是物理存在的人？ → 进入独立性判定
  ├─ 是系统边界外的另一系统/设备？ → 进入独立性判定
  ├─ 是时间事件 + 驱动了独立用例（有用户价值）？ → 进入独立性判定
  ├─ 是内部模块的执行步骤（如查询协调器分发子查询）？ → ❌ 不是 Actor，归 design.md
  ├─ 是状态驱动的内部组件（如选主器、副本管理器）？ → ❌ 不是 Actor，归 design.md
  └─ 是内部协议适配层（如 MQTT broker、HTTP server）？ → ❌ 不是 Actor，归 design.md

独立性判定（类型通过后必须继续）：
  ├─ 与运行中的系统有持续、直接的交互？（一次性行为如选型评估 → ❌）
  ├─ 能驱动至少 1 个其他 Actor 无法覆盖的独立 Feature？（关注点已被完全覆盖 → ❌）
  └─ 全部通过 → ✅ 独立 Actor
```

### Actor 反例（**不应**作为 Actor）

| 候选 | 判定 | 理由 | 归属 |
|------|------|------|------|
| 写入网关 / 查询协调器 / 存储引擎 | ❌ | 内部组件，承担系统内部分工 | `design.md` |
| 副本管理器 / 选主切主器 / 故障检测器 | ❌ | 状态驱动的内部组件，非时间驱动 | `design.md` |
| 内嵌 MQTT Broker / HTTP server | ❌ | 内部协议适配层 | `design.md` |
| 平台架构师 / 技术选型评估者 | ❌ | 一次性行为（选型评估），非持续交互 | 不记录 |
| 边缘运维人员（已有 SRE） | ❌ | 同一技能栈在不同环境下操作，应合并到 SRE | 合并 |
| TTL 调度 / 定时降采样调度 | ✅ 时间触发源 | 驱动了独立用例（清理/物化视图刷新） | 本文档 |
| 外部 Prometheus / Grafana / KMS | ✅ 外部系统 | 系统边界外的另一系统 | 本文档 |

对每个关键 Actor 进行权力/利益分析，以指导需求优先级和交互设计：

| Actor | 权力（高/低） | 利益（高/低） | 策略 | 需求关注点 |
|-------|---------------|---------------|------|------------|
| | | | | |

矩阵策略说明：
- **高权力+高利益**：重点满足，保持密切协作
- **高权力+低利益**：令其满意，避免反对
- **低权力+高利益**：持续告知，争取支持
- **低权力+低利益**：最小投入，监控即可
