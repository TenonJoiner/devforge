## 迭代概览

| 项目 | 内容 |
|------|------|
| 里程碑 | M1: MVP 端到端可运行 |
| 迭代编号 | M1-I3 |
| 主题 | 传输层核心实现 + 索引增强 + 存储引擎补全 |
| 容量预算 | 60-80 SP（原始 30-40 SP/2 周，经变更延长至 4 周，容量相应翻倍。变更依据见下方说明） |
| 实际选入 | 63 SP（9 个 proposal） |
| 团队 | 4-6 名 C 语言工程师 + AI 辅助（沿用里程碑默认参数） |
| 迭代周期 | 4 周 |
| 前置依赖 | M1-I1 全部完成（P01-P08）；M1-I2 完成 P09/P14/P15，P10 未完成延入本迭代 |

**选入策略**：M1-I3 承担三个核心职责——（1）完成传输层全栈实现（RDMA 数据面 + TCP 控制面 + 协议封装），为 M1-I4 的调度器和连接器提供传输基础；（2）完成 Prefix Hash Chain 索引和 Raft 核心集成，补全元数据服务的关键能力；（3）消化 I2 遗留的 P10（DRAM 缓存）并推进存储引擎的分层和校验能力。

**周期变更依据**：milestone-plan.md 原始规划 M1-I3 为 2 周/30-40 SP。经评估后延长至 4 周/60-80 SP，原因如下：（1）P10（DRAM 缓存，8 SP）从 M1-I2 裁剪遗留，需额外消化；（2）传输层（P19 RDMA，13 SP，高风险）和 Raft（P17a+P17b，13 SP，高风险）两个高风险 proposal 同时推进，需要更充裕的缓冲以应对技术不确定性；（3）用户主动要求延长至 4 周以确保关键路径上的高风险任务有足够的降级决策窗口。

**P10 遗留处理**：P10（storage-impl-dram-cache，8 SP）从 M1-I2 裁剪延入。P10 无迭代内前置依赖（仅依赖 P03 接口定义，I1 已完成），可在 Wave 0 立即启动。P10 是 P11（数据分层）的前置依赖，P11 安排在 Wave 1，确保依赖链正确。4 周周期下容量充足（63 SP < 80 SP 上限），无需裁剪其他 proposal 为 P10 腾出空间。

**裁剪优先级**（若容量不足时的降级顺序）：
1. P13（storage-impl-checksum，3 SP）可延至 M1-I4。CRC32C 校验是数据完整性保障，但不在关键路径上，延后不影响端到端闭环。裁剪后降至 60 SP。
2. P17b（metadata-impl-raft-snapshot，5 SP）可延至 M1-I4。Raft 快照是 Raft 核心的增强能力，P17a 完成后 Raft 基础功能已可用。裁剪后降至 55 SP。**跨迭代影响**：P17b 裁剪后 Raft 日志无法截断，长时间运行将导致日志无限增长，影响 M1-I4 的 P18（分区管理）启动条件——P18 需要 Raft 集群在多分区场景下稳定运行，日志膨胀可能导致重放时间过长。建议：若裁剪 P17b，需在 I4 最高优先级补回，且 P17a 须在 I3 结束前产出日志截断的最小桩实现（仅按固定条目数截断，不含快照传输）以缓解日志增长。
3. P11（storage-impl-tiering，8 SP）可延至 M1-I4。数据分层是存储引擎增强，MVP 阶段 DRAM 缓存 + SSD 持久化已可工作。裁剪后降至 47 SP。

**硬承诺**（不可裁剪）：P10(8) + P19(13) + P20(5) + P21(5) + P16(8) + P17a(8) = 47 SP。这 6 个 proposal 位于关键路径上或是后续迭代的硬性前置条件。

> Wave 0 内的 proposal 仅依赖 M1-I1/I2 已完成的产出，可由不同开发者并行开发。

| P19 | `transport-impl-rdma` | 13 SP | L | 功能 | **高风险**。RDMA 双边操作数据面：连接管理、Send/Recv 封装、CQ 轮询（SPDK Reactor 原生）、零拷贝传输；基于 P05 Spike 确认的线程模型方案 | -- |

| P16 | `metadata-impl-prefix-index` | 8 SP | L | 功能 | Prefix Hash Chain 索引：token 序列前缀匹配、索引构建/查询/失效传播；基于 P14（I2 已完成）的内存哈希表 | -- |
| P17a | `metadata-impl-raft-core` | 8 SP | L | 功能 | **高风险**。Raft 核心集成：C 语言 Raft 库绑定、日志存储适配（基于 P15 WAL）、Leader 选举、日志复制；基于 P07 Spike 确认的 Raft 库选型。**P17a 需在 Day 14 前产出快照接口桩（stub），使 P17b 可在 Day 12 左右提前开始接口层开发** | -- |

**Wave 0 合计**：45 SP，6 个 proposal

**Wave 0 内部并行度**：6 个 proposal 之间无互相依赖，可全部并行。按子系统分组：
- 存储引擎组：P10（DRAM 缓存）+ P13（校验），两者操作不同层，无资源冲突
- 传输层组：P19（RDMA）+ P20（TCP），两者是独立的传输协议实现，可完全并行
- 元数据组：P16（Prefix 索引）+ P17a（Raft 核心），P16 操作索引层，P17a 操作一致性层，无冲突

**人员分配建议**（6 人团队）：
- 工程师 A：P19（RDMA，13 SP，最大单体，需全程投入）
- 工程师 B：P17a（Raft 核心，8 SP，高风险，需专注）
- 工程师 C：P10（DRAM 缓存，8 SP）→ Wave 1 接手 P11（数据分层）
- 工程师 D：P16（Prefix 索引，8 SP）
- 工程师 E：P20（TCP 控制面，5 SP）→ Wave 1 接手 P21（协议封装）
- 工程师 F：P13（校验，3 SP）→ Wave 1 接手 P17b（Raft 快照）

**P19/P20 传输层资源隔离约定**：P19（RDMA）和 P20（TCP）虽属同一子系统但操作完全不同的网络栈。并行开发约定：（1）P19 和 P20 各自定义独立的连接管理和消息收发接口，Day 5 前产出各自的内部 API；（2）P21（协议封装）在 Wave 1 统一两者的消息头格式和请求/响应匹配机制；（3）Day 7 进行一次传输层接口对齐检查，确认 P21 的统一封装可行。

**P17a/P15 WAL 集成约定**：P17a（Raft 核心）依赖 P15（WAL，I2 已完成）提供的日志存储层。约定：（1）P17a 通过 P15 暴露的 WAL 追加写入接口存储 Raft 日志条目；（2）P17a 在 Day 3 前完成 Raft 库绑定骨架，Day 5 前完成与 WAL 的集成联调。

## 关键路径

> 从 Wave 0 到最后一个 Wave 的最长依赖链（迭代内部 + 跨迭代影响）。

### 迭代内部关键路径

```
关键路径 1（传输层全栈）：
P19(13 SP, W0) ──┐
                  ├──→ P21(5 SP, W1)    总计 18 SP，约 3-3.5 周
P20(5 SP, W0)  ──┘

关键路径 2（存储引擎分层）：
P10(8 SP, W0) → P11(8 SP, W1)           总计 16 SP，约 3 周

关键路径 3（Raft 全栈）：
P17a(8 SP, W0) → P17b(5 SP, W1)         总计 13 SP，约 2.5-3 周
```

**迭代内最长路径**：关键路径 1（P19 → P21），总计 18 SP，约 3-3.5 周。P19 是本���代最大的单体 Proposal（13 SP）且为高风险，位于关键路径起点。

### 跨迭代关键路径影响

```
传输层链：
P19(I3) + P20(I3) → P21(I3) → P24(SDK, I4) → P25(Prefix API, I4) → P34(vLLM集成, I5/I6)
                              → P30(传输集成测试, I5)

元数据链：
P16(I3) → P25(Prefix API, I4) → P32(Prefix集成, I5) → P34(vLLM集成, I5/I6)
P17a(I3) → P17b(I3) → P18(Partition, I4) → P22(Routing, I4) → P33(4节点验证, I5/I6)

存储引擎链：
P10(I3) → P11(I3) → P12(Replication, I4) → P31(复制集成测试, I5)
P13(I3) → P12(Replication, I4) [校验和用于副本数据完整性验证]
```

**瓶颈分析**：

1. **P19（RDMA 双边操作，13 SP，高风险）** 是本迭代和 M1 整体的关键瓶颈。P19 延期将直接阻塞 P21（协议封装）→ P24（Connector SDK）→ P34（vLLM 集成），影响 M1 端到端闭环。缓解措施：（1）P19 内部按"连接管理→Send/Recv 封装→CQ 轮询→零拷贝→单元测试"分阶段推进，Day 10 前产出核心 API 供 P21 使用；（2）工程师 A 全程投入 P19，不分配其他任务；（3）P05 Spike 已验证基础可行性，降低技术不确定性。

2. **P17a（Raft 核心，8 SP，高风险）** 是元数据一致性链的关键节点。P17a 延期将阻塞 P17b → P18 → P22，影响调度器和 4 节点集群验证。缓解措施：（1）P07 Spike 已验证 Raft 库集成可行性；（2）P17a 优先完成 Leader 选举和日志复制核心功能，快照（P17b）作为增强可裁剪。

3. **P10 → P11 依赖链**：P10 是 I2 遗留，若 P10 再次延期，P11 将无法启动，进而影响 P12（数据复制）。缓解措施：P10 在 Wave 0 最高优先级启动，工程师 C 专注完成。

**风险统计**：2 个 🔴 高风险（P19 RDMA + P17a Raft）+ 5 个 🟡 中风险。高风险任务占比：P19(13) + P17a(8) = 21 SP / 63 SP = 33%，在 40-50% 阈值范围内（P17b 从高风险降为中风险，因 Raft 核心已由 P17a 验证，P17b 是增强功能且可提前启动接口层开发）。

**P19 降级预案**（Day 14 决策门）：若 Day 14（第 2 周末）P19 核心 API 仍未就绪：
- P21 切换为 TCP-only 协议封装（仅封装 P20 的 TCP 传输），RDMA 协议封装延至 I4
- P19 继续开发但不阻塞 P21，P19 在 I3 结束时产出 RDMA 独立模块，I4 再与 P21 集成
- 传输层接口冻结从 Day 14 推迟至 Day 17，冻结内容调整为 TCP-only 传输接口 + RDMA 接口标记为草案（不作为 I4 的硬性依赖）
- 影响评估：M1-I4 的 P24（SDK）可先基于 TCP-only 传输层开发，RDMA 数据面在 I4/I5 补全
