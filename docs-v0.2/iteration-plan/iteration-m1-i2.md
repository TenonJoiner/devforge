**选入策略**：M1-I2 聚焦两个核心子系统的基础实现——存储引擎（SPDK Blobstore 封装 + DRAM 缓存）和元数据服务（内存哈希表索引 + 异步 WAL）。这 4 个 proposal 是 M1 关键路径上的核心节点，后续迭代的分层（P11）、Prefix 索引（P16）、Raft（P17a）、数据复制（P12）均依赖本迭代产出。

**前置条件（M1-I1 Spike 结论依赖）**：
- P09 依赖 P05（SPDK-RDMA Spike）通过 → 确认 SPDK Reactor 线程模型可用
- P14 依赖 P06（分布式索引 Spike）通过 → 确认分片方案或采用备选方案
- P15 依赖 P08（WAL checkpoint Spike）通过 → 确认增量 checkpoint 方案或降级方案

**M1-I1 延期降级预案**（分级启动策略）：

| I1 完成状态 | I2 启动策略 |
|------------|-----------|
| 全部完成 | 按计划启动 4 个 proposal |
| P08 未完成（其余完成） | P09/P10/P14 按计划启动；P15 进入设计阶段（接口定义+模块骨架），WAL 具体方案待 P08 结论后确定，估算不变 |
| P05 未完成（其余完成） | P10/P14/P15 按计划启动；P09 延至 P05 完成后启动（最多延迟 3-5 天），I2 容量降至 24 SP |
| P06 未完成（其余完成） | P09/P10/P15 按计划启动；P14 按单节点方案启动（ADR-003 备选），待 P06 结论后决定是否调整 |
| 2+ Spike 未完成 | 触发 milestone-plan.md 决策门，I2 范围和启动时间需重新评估 |

**裁剪优先级**（若容量不足时的降级顺序）：P10（DRAM 缓存 8 SP）可延至 M1-I3。P10 的下游依赖是 P11（数据分层）和 P35（雪崩防护），均排在 M1-I3 之后。裁剪后硬承诺降至 29 SP（P09 + P14 + P15）。P09/P14/P15 不可延后，因为它们直接阻塞关键路径。

**I3 容量吸收方案**（若 P10 裁剪延至 I3）：milestone-plan.md 中 M1-I3 预算 30-40 SP，主题为"传输层 + 索引增强"。若 P10（8 SP）带入 I3，I3 候选总量将达 ~38-46 SP。吸收策略：（1）P20（TCP 控制面 5 SP）可延至 I4，因 MVP 阶段 TCP 仅用于控制面，不在关键路径上；（2）P10 在 I3 中优先级高于 P20，因为 P11（分层）直接依赖 P10。

| P15 | `metadata-impl-wal` | 8 SP | L | 功能 | 异步 WAL：追加写入、批量刷盘、崩溃恢复重放、增量 checkpoint（或降级方案，取决于 P08 Spike 结论） | — |

**Wave 0 合计**：35-37 SP（P14 双路径估算），4 个 proposal，全部可并行

**Wave 0 内部并行度**：P09/P10/P14/P15 之间无互相依赖，可全部并行。P14 和 P15 属于元数据子系统但操作不同组件（索引 vs WAL）。4 名工程师可各认领一个 proposal。

**P09/P10 SPDK Reactor 资源隔离约定**：P09（Blobstore）和 P10（DRAM 缓存）虽操作不同层，但共享 SPDK Reactor 资源和 NVMe 设备句柄。并行开发约定：（1）P09 负责定义 IO channel 管理的公共接口（Day 3 前产出），P10 基于该接口实现 DRAM 层的内存分配；（2）两者各自使用独立的 SPDK thread，通过 spdk_thread_send_msg 跨线程通信，避免直接共享可变状态；（3）Day 5 进行一次 15 分钟的接口对齐检查，确认 Reactor 资源分配无冲突。

**人员交叉熟悉机制**：为缓解单点人员风险，采用"主负责+副了解"模式：P09/P10 的负责人互为副手（同属存储引擎子系统），P14/P15 的负责人互为副手（同属元数据子系统）。副手在 Day 3 和 Day 7 各花 1 小时 review 主负责人的代码，确保任一人卡住时副手可接手核心模块。

> **注意**：本迭代所有 proposal 均在 Wave 0，这是因为它们的依赖全部来自 M1-I1（已完成），迭代内部无互相依赖。这不是红旗信号——红旗是"未识别迭代内依赖关系"，而本迭代确实不存在迭代内依赖。

```
本迭代所有 proposal 均在 Wave 0，无迭代内依赖链。

**瓶颈分析**：P09（SPDK Blobstore 封装）是本迭代最大的单体 Proposal（13 SP），也是后续迭代最多下游依赖的节点（P11 分层、P12 复制、P13 校验均依赖 P09）。缓解措施：P09 内部按"Blobstore API 封装→IO channel 管理→Reactor 集成→单元测试"分阶段推进，第 7 天完成核心 API 封装，使后续迭代的接口对接可提前准备。

```

P15(本迭代) → P17a(Raft核心,I3) [P15 提供 WAL 存储层供 Raft 日志使用]

## 风险与缓冲

| # | 风险 | 标记 | 影响 | 缓解 |
|---|------|------|------|------|
| 1 | SPDK Blobstore API 学习曲线超预期 | 🟡 | P09 延期，阻塞 M1-I3 的 P11/P12 | P05 Spike 已验证基础可行性；参考 InfiniStore 的 Blobstore 封装实现；前 3 天聚焦 API 熟悉 |
| 2 | P06 Spike 结论为"分布式分片不可行"，P14 需切换单节点方案 | 🟡 | P14 实现方案变更，估算可能偏差 ±2 SP | 单节点方案更简单（估算下调），但需在 P14 设计中预留分布式扩展接口 |
| 3 | P08 Spike 结论为"增量 checkpoint 不达标"，P15 需采用降级方案 | 🟡 | P15 WAL 设计需调整为"缩小单 WAL 文件 + 全量重建" | 降级方案实现复杂度更低，但需在设计中明确重建时间的 SLA 降级 |
| 4 | 4 个 L 级 proposal 同时推进，代码评审瓶颈 | 🟡 | 评审排队导致合并延迟 | 每个 proposal 按子模块分批提交评审，不等全部完成再评审；交叉评审（存储工程师评审元数据，反之亦然） |
| 5 | P09/P10 SPDK Reactor 资源竞争 | 🟡 | IO channel 或内存分配冲突导致运行时错误 | Day 3 P09 产出 IO channel 公共接口，Day 5 接口对齐检查；各自使用独立 SPDK thread |

**风险统计**：0 个 🔴 高风险 + 5 个 🟡 中风险。本迭代无技术可行性风险（Spike 已在 M1-I1 完成），风险主要来自 Spike 结论对实现方案的影响、资源隔离和工程执行效率。
