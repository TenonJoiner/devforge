## 迭代概览

| 项目 | 内容 |
|------|------|
| 里程碑 | M1: MVP 端到端可运行 |
| 迭代编号 | M1-I1 |
| 主题 | 基础设施搭建 + 高风险 ADR Spike 验证 |
| 容量预算 | 25-35 SP（来自 milestone-plan.md 迭代总览） |
| 实际选入 | 34 SP（8 个 proposal） |
| 团队 | 4-6 名 C 语言工程师 + AI 辅助 |
| 迭代周期 | 2 周 |
| 前置依赖 | 无（M1 首个迭代） |

**选入策略**：M1-I1 承担两个关键职责——（1）搭建全项目构建/测试/开发基础设施，为后续所有迭代提供基础；（2）集中验证 4 个高风险 ADR 假设（ADR-003/004/006/009），在 M1 早期暴露技术风险。实际选入 34 SP 略超预算上限（35 SP），因为 4 个 Spike 验证是 M1 决策门的硬性前置条件，不可延后。

**裁剪优先级**（若容量不足时的降级顺序）：P08（WAL checkpoint Spike）可延至 M1-I2，将硬承诺降至 29 SP。P08 的结论影响 P15（WAL 实现），而 P15 排在 M1-I2，延后 P08 不影响关键路径。其余 3 个 Spike（P05/P06/P07）不可延后，因为它们直接阻塞 M1-I2/I3 的核心实现。

## 关键路径

> 从 Wave 0 到最后一个 Wave 的最长依赖链（迭代内部）。

```
P01(5 SP) + P02(3 SP) → P05(8 SP)    总计 16 SP，约 1.5 周
P01(5 SP) → P06/P07/P08(5 SP each)   总计 10 SP，约 1 周
```

**关键路径**：P01 + P02 → P05（SPDK-RDMA Spike），总计 16 SP

**瓶颈分析**：P05 是本迭代最大的单体 Proposal（8 SP），且位于关键路径上。P05 的结论直接决定 M1 后续迭代的 RDMA 数据面方案（P19）。缓解措施：P01 和 P02 优先启动，确保 P05 在迭代第 4-5 天即可开始；P05 内部按"环境搭建→原型编码→性能测试→结论报告"分阶段推进，第 7 天产出初步结论。

## 风险与缓冲

| # | 风险 | 标记 | 影响 | 缓解 |
|---|------|------|------|------|
| 1 | SPDK Reactor 与 RDMA CQ 轮询不兼容 | 🔴 | P19（RDMA 实现）方案需切换为独立线程，影响 M1-I3 | P05 Spike 第 7 天产出初步结论；备选方案已在 ADR-006 中定义 |
| 2 | 分布式索引分片延迟不达标 | 🔴 | P14（哈希表索引）需简化为单节点方案 | P06 Spike 验证；备选方案：单节点索引 + 后续迭代再分布式化 |
| 3 | Raft 库 API 不稳定或性能不足 | 🔴 | P17a/P17b 需更换 Raft 库或先用单节点元数据 | P07 Spike 同时评估 willemt/raft 和 canonical/raft |
| 4 | WAL checkpoint 重建时间远超目标 | 🔴 | 需重新设计 checkpoint 策略 | P08 Spike 验证增量方案；备选：缩小单 WAL 文件大小 |
| 5 | SPDK/DPDK 依赖编译环境搭建耗时超预期 | 🟡 | Wave 1 全部延迟启动 | P02 Docker 镜像预装依赖；P01 优先验证编译链 |

**风险统计**：4 个 🔴 高风险（全部是 Spike 验证，这是本迭代的核心目标）+ 1 个 🟡 外部依赖风险

**决策门（分级预案）**：

| Spike 失败数 | 应对动作 |
|-------------|---------|
| 0 | 按计划推进 M1-I2 |
| 1（P05/P06/P07 之一） | 启用该 Spike 对应的 ADR 备选方案，受影响的下游 proposal 估算上调 20%，在 M1-I2 规划时吸收 |
| 1（P08） | WAL checkpoint 方案降级为"缩小单 WAL 文件 + 全量重建"，P15 估算上调但不影响关键路径 |
| 2 | 触发架构方向局部重评估：评估两个失败 Spike 的交叉影响，确认备选方案组合是否可行；M1 总估算可能上调 15-20% |
| ≥ 3 | 触发架构方向全面重评估（参见 milestone-plan.md 风险 R6），M1 范围和周期需重新规划 |
