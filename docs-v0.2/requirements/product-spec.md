## 业务上下文

### 行业背景

- **所属行业领域**：AI 基础设施 / 企业级分布式存储
- **当前行业核心趋势与挑战**：
  - LLM 推理服务规模化部署成为刚需，GPU 显存（HBM）成本高昂且容量有限
  - KV Cache 是推理服务的核心资源，其存储与管理直接影响 TTFT（首 token 延迟）和吞吐
  - 现有方案（MoonCake、LMCache）验证了 KV Cache 卸载到外部存储的可行性，但缺乏企业级能力（多租户隔离、QoS、合规审计、故障自愈）
  - 通用对象存储（MinIO、Ceph）缺乏原生 KV Cache 语义支持，与推理引擎集成成本高
- **技术演进方向对本品的影响**：
  - PD 分离（Prefill-Decode Disaggregation）成为主流架构，KV Cache 跨节点传输需求激增
  - RDMA 网络在 GPU 集群中普及，但 TCP 回退能力仍不可少
  - 长上下文（128k+ tokens）和多轮对话场景成为标配，对 KV Cache 的容量和索引效率提出更高要求

### 竞争格局

| 竞品/方案 | 定位 | 核心优势 | 主要短板 | 与本品关系 |
|-----------|------|----------|----------|------------|
| **MoonCake** | 开源 KV Cache 卸载框架 | PD 分离架构验证充分，论文级别优化 | 开源版功能有限，运维/监控能力弱，企业级能力缺失 | 功能对标，本品补齐企业级能力 |
| **LMCache** | 开源 KV Cache 缓存层 | 多后端支持（Redis/Ceph/本地），社区活跃 | 后端扩展依赖社区，缺乏统一存储抽象，企业级能力不足 | 功能对标，本品提供统一企业级存储层 |
| **VAST Datastore** | 全闪存统一存储平台 | 多协议统一（S3/NFS/GPUDirect），EB 级扩展 | 定位通用存储，非 KV Cache 专用，软硬件一体化成本高 | 架构参考，本品聚焦 KV Cache 场景 |
| **MinIO** | 开源 S3 兼容对象存储 | 100% S3 API 兼容，高性能 | 开源收缩策略风险，无 KV Cache 语义，企业版功能剥离 | 部分能力对标，本品差异化在 KV Cache 原生支持 |
| **Ceph** | 开源统一存储平台 | 统一块/文件/对象存储，生态成熟 | 架构复杂运维成本高，无原生 KV Cache 语义 | 部分能力对标，本品简化运维+KV Cache 专用优化 |
| **裸 HBM 扩容** | 直接购买更多 GPU 显存 | 延迟最低，无需额外系统 | 成本极高（HBM 单价是 DRAM 的 10x+），容量天花板明显 | 本品替代方案，以存储换计算降低成本 |

### 商业模式假设

- **目标客户群体**：
  - 拥有私有化 GPU 集群的企业（互联网大厂、AI 独角兽、金融/政务云服务商）
  - MaaS（Model-as-a-Service）平台运营方
  - 提供 AI 推理基础设施的 IDC/云服务提供商
- **价值主张**：
  - 以低成本外部存储（DRAM/SSD）替代昂贵 HBM，降低 LLM 推理基础设施的 TCO
  - 原生 KV Cache 语义支持，与 vLLM/SGLang 等推理框架零摩擦集成
  - 企业级多租户隔离、QoS、审计合规，满足生产环境要求
- **收入模式假设**：
  - 核心存储引擎开源（建立生态），企业级功能（多租户、QoS、审计、高级监控）闭源订阅
  - 按节点/容量订阅，支持年度授权和按需扩容
- **关键成功因素**：
  - 与主流推理框架（vLLM、SGLang、NVIDIA Dynamo）的深度集成
  - 性能指标（TTFT、TBT、吞吐）达到或接近 HBM 方案的水平
  - 企业级可靠性（数据耐久性、故障自愈、在线扩容）达到金融级标准

## 约束与假设

### 技术约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 部署环境 | 私有化数据中心，客户裸金属服务器，Linux 环境 | 部署方案、网络配置、硬件适配 |
| 容器编排 | 产品本身部署于 K8s + 容器，需支持 CSI 集成 | 部署架构、存储接口设计 |
| 网络栈 | 支持 RDMA（RoCE v2/InfiniBand）+ TCP 双栈，RDMA 优先 | 传输层设计、故障回退策略 |
| 语言生态 | 主要面向 C 语言推理引擎集成（vLLM/SGLang 核心为 C++/Python） | SDK/Connector 设计语言 |
| 存储介质 | HBM（推理引擎本地）→ DRAM（节点本地）→ SSD/NVMe（节点本地）→ 网络存储（远端） | 数据分层策略、热冷数据迁移 |

### 法规与合规约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 数据安全 | 企业客户要求数据加密（传输+静态）、访问审计、密钥管理 | 安全模块设计、审计日志 |
| 多租户隔离 | 多客户/多模型共享集群时，数据不可跨租户访问 | 权限模型、命名空间隔离 |
| 合规认证 | 金融/政务客户可能要求等保、SOC2 等认证（远期） | 安全架构设计预留 |

### 用户能力约束

| 约束项 | 内容 | 影响范围 |
|--------|------|----------|
| 运维团队规模 | 目标客户运维团队通常 3-10 人，需自动化运维 | 运维界面设计、告警自动化 |
| 推理框架多样性 | 客户同时使用多种推理框架（vLLM、SGLang、TGI 等） | Connector 抽象层设计 |
| 存储知识门槛 | 推理工程师通常不熟悉分布式存储运维 | 简化部署、一键配置模板 |

## 使用场景与上下文

**产品定位**：面向 LLM 推理场景的企业级分布式对象存储，在通用对象存储能力基础上原生支持 KV Cache 语义（chunk 级索引、prefix 匹配、PD 分离传输），以低成本外部存储替代昂贵 HBM，同时满足企业级多租户隔离、QoS、审计合规要求。

**部署环境**：
- 私有化数据中心 GPU 集群
- 节点配置：GPU 节点（HBM）+ 存储节点（DRAM/SSD）
- 网络：RDMA（RoCE v2/IB）+ 管理网络（TCP）
- 编排：K8s + 容器部署，支持 CSI 挂载

**典型使用流程**：

```
[推理引擎启动] → [通过 Connector 注册到存储集群]
    ↓
[Prefill 阶段] → [计算 KV Cache] → [异步写入存储（HBM→DRAM→SSD 分层）]
    ↓
[Decode 阶段] → [从存储加载 KV Cache] → [连续生成 Token]
    ↓
[多轮对话] → [Prefix 匹配命中缓存] → [跳过重复计算，直接复用]
    ↓
[集群扩缩容] → [存储层自动数据重平衡] → [推理引擎无感知]
    ↓
[运维监控] → [Prometheus 指标 + 可视化 Dashboard] → [告警/故障自愈]
```

### 详细定义

#### Actor: 推理服务开发者

- **类型**：人类用户
- **职责**：负责将存储系统通过 Connector/SDK 集成到推理服务中，配置缓存策略和性能参数，监控集成后的推理性能指标（TTFT、TBT、命中率）。是推理引擎与存储系统之间的"技术桥梁"。
- **交互方式**：
  - 主要：SDK API（C/C++ / Python）、配置文件（YAML/JSON）
  - 次要：CLI 工具（调试、状态查询）、文档/示例代码
- **关注点**：
  1. 集成复杂度（是否能在 1 小时内完成基础集成）
  2. 与主流推理框架的兼容性（vLLM、SGLang、Dynamo 版本覆盖）
  3. 性能可预测性（延迟分布稳定，无异常抖动）
  4. 调试便利性（详细的错误信息、连接状态可见）
  5. 文档质量（API 参考、最佳实践、故障排查指南）
- **使用频率**：集成阶段高频（每日），上线后中频（每周监控）
- **痛点摘要**：
  - MoonCake/LMCache 的配置参数多达 40-50 项，缺乏场景化模板，需反复试验
  - 版本兼容性要求高（vLLM v1 支持仍在演进），升级时易出兼容性问题
  - 故障时缺乏清晰的诊断路径，日志分散难以定位根因
  - 不同推理框架的 Connector 接口不统一，需为每种框架单独适配

> **标杆映射**：对应 MoonCake "推理服务开发者"、LMCache "推理服务开发者"、MinIO "应用开发者"
> **差异化启示**：提供场景化配置模板（RAG/多轮对话/长上下文），标准化 Connector 接口降低多框架适配成本

#### Actor: 集群运维工程师

- **类型**：人类用户
- **职责**：负责存储集群的部署、日常监控、容量规划、故障处理和扩缩容操作。确保存储层 7x24 稳定运行，是系统可用性的第一责任人。
- **交互方式**：
  - 主要：CLI 工具（部署、运维命令）、Web Dashboard（监控可视化）
  - 次要：Prometheus/Grafana（指标查看）、告警通知（PagerDuty/钉钉/企业微信）
  - 自动化：K8s Operator（声明式部署）、Ansible/Terraform（基础设施即代码）
- **关注点**：
  1. 部署简易性（能否在 15 分钟内完成单节点部署，1 小时内完成多节点集群）
  2. 运维自动化（故障自愈、自动重建、自动扩缩容）
  3. 监控可观测性（关键指标全覆盖、告警准确无噪音）
  4. 容量管理（使用率预测、扩容操作无停机）
  5. 故障恢复时间（磁盘/节点故障后的自动恢复速度和数据完整性）
- **使用频率**：日常（每日监控巡检）
- **痛点摘要**：
  - Ceph 架构复杂，生产级调优需专业知识，小团队难以驾驭
  - MoonCake 监控指标不够丰富，缺少可视化 Dashboard
  - LMCache 故障恢复机制不完善，Controller 故障需手动恢复
  - MinIO 开源版 Web 控制台功能大幅削减，基础管理被迫使用 CLI
  - 扩容时数据再平衡影响业务性能，缺乏精细控制

> **标杆映射**：对应 MoonCake "集群运维工程师"、LMCache "集群运维工程师"、MinIO "平台工程师"、Ceph "存储管理员"
> **差异化启示**：开箱即用的告警模板、引导式故障诊断、一键扩缩容、在线再平衡性能控制

#### Actor: MaaS 平台运营者

- **类型**：人类用户
- **职责**：负责多租户环境下的资源分配、QoS 策略制定、成本优化和合规审计。关注整体平台的有效吞吐（goodput）、资源利用率和租户满意度。
- **交互方式**：
  - 主要：Web 管理控制台（租户管理、配额配置、策略设置）
  - 次要：API（自动化租户开通、批量配置）、报表导出（成本分析、审计报告）
- **关注点**：
  1. 多租户隔离强度（数据不可跨租户访问、性能互不干扰）
  2. QoS 策略灵活性（按租户/模型/时段分配 IOPS/带宽/延迟 SLA）
  3. 成本可视性（每租户存储成本、缓存命中率与成本关系）
  4. 审计合规（操作日志完整、报表生成便捷、满足等保/SOC2 要求）
  5. 租户自助服务（租户自主管理子资源、查看自身配额使用率）
- **使用频率**：持续（运营决策），日常（租户管理），阶段性（月度/季度审计）
- **痛点摘要**：
  - MoonCake 仅通过 tenant_id 做基础隔离，无 RBAC、无细粒度权限控制
  - LMCache 无多租户公平调度，缺乏租户级配额管理
  - 现有开源方案缺少租户级成本分摊和计费数据
  - 审计日志格式不统一，SIEM 集成困难
  - 合规报告生成需手动汇总，耗时耗力

> **标杆映射**：对应 MoonCake "MaaS 平台运营者"、LMCache "ML 平台工程师"、MinIO "存储管理员"
> **差异化启示**：企业级多租户隔离（RBAC+命名空间+配额）、内置成本分析报表、自动化合规报告生成

#### Actor: 安全合规官

- **类型**：人类用户
- **职责**：负责制定安全策略、审查审计日志、验证合规状态、管理加密密钥策略、执行安全认证。是企业级存储产品从"开源工具"升级为"生产系统"的关键把关人。与 MaaS 平台运营者的核心差异：安全合规官关注**合规证明**（证据链、认证、报告），平台运营者关注**运营效率**（配额、性能、成本）。
- **交互方式**：
  - 主要：Web 管理控制台（加密策略配置、审计日志审查、合规报表生成）
  - 次要：SIEM 系统对接（审计日志实时推送）、合规报告导出（PDF/Excel 格式）
- **关注点**：
  1. 数据加密完备性（传输加密 + 静态加密 + 密钥隔离，满足等保/SOC2 要求）
  2. 审计日志完整性（所有数据访问和管理操作不可篡改、可追溯）
  3. 合规认证可达性（产品是否具备通过等保 2.0 三级、SOC2 Type II 的条件）
  4. 访问控制强度（RBAC/ABAC 策略的粒度和不可绕过性）
  5. 安全事件响应（异常访问检测、告警时效、事件调查支持）
- **使用频率**：阶段性（安全策略制定、月度/季度审计、合规检查期高频）
- **痛点摘要**：
  - MoonCake/LMCache 无企业级权限管控，仅有基础 tenant_id 隔离
  - 审计日志格式不统一，SIEM 集成困难，合规报告需手动汇总
  - 开源方案缺乏加密设计，金融/政务客户无法通过准入审查
  - 等保 2.0 三级要求"安全管理中心"独立部署，安全模块与管理模块需物理隔离

> **标杆映射**：对应 MinIO "安全合规官"、Ceph "安全合规官"、LMCache "企业安全/合规团队"
> **差异化启示**：安全模块与管理模块架构级隔离、审计日志不可篡改（WORM 模式）、开箱即用的合规报告模板

#### Actor: 应用开发者

- **类型**：人类用户
- **职责**：负责通过 S3 API 将存储系统集成到通用应用中（非推理场景）。与推理服务开发者的核心差异：推理服务开发者聚焦**推理引擎与 KV Cache 的专用集成**（chunk 级索引、prefix 匹配、PD 传输），应用开发者聚焦**通用对象存储的标准接入**（S3 API、K8s CSI、大数据框架集成）。
- **交互方式**：
  - 主要：S3 API/SDK（AWS CLI、 boto3、Terraform）
  - 次要：K8s PVC（通过 CSI 驱动声明式使用存储）
- **关注点**：
  1. S3 API 兼容性（100% 兼容 AWS S3 核心操作子集，零改造迁移）
  2. SDK 易用性（多语言 SDK、与现有工具链无缝集成）
  3. K8s 集成便利性（CSI 驱动稳定、StorageClass 配置简单）
  4. 认证方式灵活性（支持 AWS Signature V4、临时凭证 STS）
- **使用频率**：集成阶段高频（每日），上线后低频（按需扩展）
- **痛点摘要**：
  - 通用对象存储（MinIO/Ceph）与推理引擎集成成本高，需自行开发适配层
  - 推理专用存储（MoonCake/LMCache）缺乏 S3 API 兼容，无法服务通用场景
  - K8s CSI 驱动配置复杂，StorageClass 参数难以理解

> **标杆映射**：对应 MinIO "应用开发者"、Ceph "应用开发者"
> **差异化启示**：统一存储引擎同时服务 KV Cache 专用语义和 S3 通用语义，一套系统满足两种场景

#### Actor: AI 基础设施架构师

- **类型**：人类用户
- **职责**：负责企业级 AI 推理基础设施的技术方案评估、架构标准制定和长期演进规划。是技术选型的关键决策影响者。
- **交互方式**：
  - 主要：技术文档（架构白皮书、性能基准报告）、概念验证（POC）环境
  - 次要：与厂商技术交流、同行案例参考、社区讨论
- **关注点**：
  1. 架构可扩展性（能否从 3 节点平滑扩展到 100+ 节点）
  2. 与现有系统兼容性（是否兼容现有对象存储工具链、K8s 生态）
  3. 长期演进路线（产品路线图清晰度、社区活跃度、供应商锁定风险）
  4. 性能基准（在目标负载下的 TTFT、TBT、吞吐量化数据）
  5. TCO 模型（硬件成本、运维人力、许可费用的综合成本）
- **使用频率**：阶段性（架构评审、技术选型期高频，上线后低频）
- **痛点摘要**：
  - 开源方案（MoonCake/LMCache）企业级能力缺失，生产环境风险高
  - 通用存储（MinIO/Ceph）缺乏 KV Cache 语义，与推理引擎集成成本高
  - MinIO 开源收缩策略导致供应商锁定风险担忧
  - Ceph 架构复杂，小团队运维成本高，与 AI 场景匹配度低
  - 缺乏面向 AI 推理场景的专用存储基准测试数据

> **标杆映射**：对应 MoonCake "AI 基础设施架构师"、Ceph "基础设施架构师"、MinIO "企业采购决策者"（技术评估部分）
> **差异化启示**：提供面向 AI 推理场景的专用基准测试套件、清晰的开放治理承诺、小规模优化降低准入门槛

#### Actor: 推理引擎（vLLM/SGLang/Dynamo）

- **类型**：外部系统
- **职责**：通过标准化 Connector 接口向存储系统发起 KV Cache 的读写请求，在 Prefill 阶段异步写入新生成的 KV Cache，在 Decode 阶段按需加载历史 KV Cache，在多轮对话中利用 Prefix 匹配跳过重复计算。
- **交互方式**：
  - 主要：Connector API（C/C++ 库调用，异步回调机制）
  - 次要：gRPC/HTTP（配置同步、心跳保活）
- **关注点**：
  1. 读写延迟（Prefill 阶段写入不阻塞计算、Decode 阶段读取延迟可预测）
  2. 接口稳定性（API 版本兼容性、升级时无破坏性变更）
  3. 异步非阻塞（I/O 操作不阻塞推理主线程）
  4. 错误处理（网络/存储故障时的优雅降级、自动重试）
  5. 资源开销（Connector 客户端的 CPU/内存占用低）
- **使用频率**：持续（每次推理请求都触发读写）
- **痛点摘要**：
  - 现有方案与推理引擎紧耦合，升级任一方都可能导致兼容性问题
  - RDMA 故障时回退 TCP 的延迟抖动大，影响推理 SLO
  - 长上下文场景下 KV Cache 加载延迟不可预测，TTFT 超标
  - 多实例间缓存不共享，每个实例独立缓存导致重复计算

> **标杆映射**：对应 MoonCake "vLLM/SGLang"、LMCache "LLM 推理引擎"
> **差异化启示**：松耦合标准化 Connector、RDMA/TCP 双栈智能切换、全局缓存池跨实例共享

#### Actor: 监控系统（Prometheus/Grafana）

- **类型**：外部系统
- **职责**：通过拉取（Pull）或接收推送（Push）方式采集存储集群的运行指标，提供可视化 Dashboard 和告警触发能力。
- **交互方式**：
  - 主要：Prometheus `/metrics` HTTP 端点（拉取模式）
  - 次要：OpenTelemetry（分布式追踪数据推送）、Webhook（告警推送）
- **关注点**：
  1. 指标覆盖度（存储层所有关键操作都有对应指标）
  2. 指标规范性（命名遵循 Prometheus 最佳实践，有完整 HELP 说明）
  3. 采集性能（指标端点响应快，不影响存储服务）
  4. 标签维度（支持按租户/节点/存储层/操作类型多维过滤）
  5. 告警规则模板（开箱即用的告警规则，降低配置门槛）
- **使用频率**：持续（按 scrape interval 周期性采集，通常 15-30s）
- **痛点摘要**：
  - LMCache 可观测性建设中，指标不够丰富
  - MoonCake 缺少可视化 Dashboard，运维人员需自行搭建
  - 告警规则需从零配置，缺乏行业最佳实践参考
  - 分布式追踪覆盖不足，跨组件故障定位困难

> **标杆映射**：对应 MoonCake/LMCache/MinIO/Ceph 的 Prometheus 集成
> **差异化启示**：开箱即用的 Grafana Dashboard 模板、预置告警规则、分布式追踪原生支持

#### Actor: 外部身份源（LDAP/SSO/OIDC）

- **类型**：外部系统
- **职责**：提供统一身份认证服务，存储系统通过标准协议（LDAP/OIDC/SAML）与之对接，实现与企业现有身份体系的单点登录（SSO）。
- **交互方式**：
  - 主要：LDAP 协议（用户/组查询、绑定认证）、OIDC/OAuth2（令牌交换）
  - 次要：SCIM（用户/组自动同步）
- **关注点**：
  1. 协议兼容性（支持 LDAP v3、OIDC、SAML 2.0 主流协议）
  2. 认证延迟（身份验证响应快，不阻塞存储请求）
  3. 高可用对接（身份源故障时的降级策略，如本地缓存、备用源）
  4. 用户/组同步（自动同步企业组织架构，减少手动维护）
  5. 令牌安全（Token 有效期管理、刷新机制、撤销能力）
- **使用频率**：持续（每次需要身份验证时触发）
- **痛点摘要**：
  - 存储系统与身份源对接配置复杂，证书/域名/端口问题频发
  - 身份源故障时存储系统无法降级，导致服务中断
  - 用户同步延迟，新入职员工无法即时获得存储访问权限
  - 组嵌套结构复杂，权限继承规则难以正确映射

> **标杆映射**：对应 MinIO "外部身份源"、Ceph "外部身份提供商"
> **差异化启示**：简化对接配置（向导式配置）、身份源故障时本地缓存降级、支持多身份源冗余

#### Actor: 外部 KMS

- **类型**：外部系统
- **职责**：管理加密密钥的全生命周期（生成、存储、轮换、销毁），存储系统通过标准接口（KMIP、PKCS#11、HashiCorp Vault API）与之对接，实现数据加密密钥的托管。
- **交互方式**：
  - 主要：KMIP 协议（密钥创建、获取、销毁）、Vault API（密钥版本管理）
  - 次要：HSM 集成（FIPS 140-2 合规场景）
- **关注点**：
  1. 密钥隔离（不同租户使用不同密钥，密钥不可跨租户复用）
  2. 密钥轮换（支持自动轮换策略，旧密钥解密→新密钥加密的平滑过渡）
  3. 高可用对接（KMS 故障时的降级策略，如本地密钥缓存、只读模式）
  4. 合规审计（密钥操作完整审计日志，满足监管要求）
  5. 性能影响（密钥获取/解密操作不成为 I/O 瓶颈）
- **使用频率**：低频（密钥获取后缓存，仅在轮换或故障时高频触发）
- **痛点摘要**：
  - KMS 与存储系统集成复杂，证书配置易出错
  - KMS 故障时存储系统无法读写，缺乏优雅降级
  - 密钥轮换过程中数据访问中断
  - 多租户场景下密钥管理混乱，缺乏清晰的隔离策略

> **标杆映射**：对应 MinIO "KMS 服务"、Ceph "外部密钥管理系统"
> **差异化启示**：KMS 故障时本地缓存密钥的只读降级、自动化密钥轮换、租户级密钥隔离

### 隐性 Actor 处理

基于标杆研究（MinIO 识别了 7 个人类角色 + 4 个外部系统，Ceph 识别了 8 个人类角色），以下候选 Actor 经过评估：

| 候选 Actor | 决策 | 理由 |
|-----------|------|------|
| **终端用户**（使用推理服务的人） | **不纳入主 Actor 列表** | 终端用户与存储系统无直接交互，其体验通过"推理服务开发者"和"推理引擎"间接体现。但在权力/利益矩阵中作为间接利益相关者标注。 |
| **安全合规官** | **✅ 纳入为独立 Actor** | 标杆研究显示 6 个产品中有 5 个将安全合规作为独立角色。安全合规的专业性（加密算法选择、合规认证、审计策略）与平台运营的业务性（配额、计费、租户管理）有本质差异。等保/SOC2 认证要求安全模块与管理模块物理隔离，合并会导致架构重构风险。 |
| **应用开发者**（非推理类，通过 S3 API 使用） | **✅ 纳入为独立 Actor** | S3 API 兼容被列为 P0，其价值主张面向通用对象存储用户（AWS CLI/SDK/Spark 等）。推理服务开发者聚焦推理引擎集成，两者目标用户不同。忽略应用开发者会导致 S3 API 兼容的 P0 优先级缺乏用户支撑。 |
| **数据工程师** | **不纳入，职责由应用开发者覆盖** | 数据工程师关注数据管道、ETL、生命周期策略配置。在我方目标客户的组织架构中（3-10 人运维团队），数据工程师与应用开发者通常为同一角色，或数据生命周期策略由集群运维工程师配置。不单独拆分以避免角色过度细分。 |
| **企业采购决策者** | **不纳入主 Actor 列表，在权力/利益矩阵中高亮标注** | 采购决策者（CTO/CIO）是重要利益相关者，但不直接操作系统。其关注点（TCO、供应商稳定性、支持 SLA、开源治理）在 AI 基础设施架构师的职责中已覆盖，并作为高权力间接利益相关者在矩阵中标注。 |
| **云管理员** | **不单独纳入，能力合并到集群运维工程师** | 云管理员在 Ceph 中是独立角色（负责私有云/云平台存储后端），但我方部署于 K8s 环境，K8s 层面的管理由集群运维工程师覆盖。 |

**结论**：主 Actor 从 8 个扩展为 **10 个**（新增安全合规官、应用开发者）。隐性 Actor 的能力需求覆盖方式：
- 安全合规官 → **独立 Actor**（安全策略、合规认证、审计审查）
- 应用开发者 → **独立 Actor**（S3 API 应用集成、K8s CSI 使用）
- 云管理员 → 集群运维工程师（K8s 部署运维）
- 数据工程师 → 应用开发者 / 集群运维工程师（生命周期策略配置）
- 终端用户/采购决策者 → 权力/利益矩阵中的间接利益相关者

## Feature 总览

> **轮次标注**：本章节第 1 轮基于标杆研究列出初步 Feature 名称和价值摘要。第 2 轮确认：主 Actor、归属特性域、优先级（P0/P1/P2）、Feature 间依赖关系。

### Feature 清单（按特性域分组）

#### 特性域：cache-offloading（KV Cache 核心卸载能力）

> 本特性域覆盖 KV Cache 的写入、读取、Prefix 匹配、PD 分离传输等核心 offloading 能力。是产品区别于通用对象存储的核心差异化域。

| Feature | 价值摘要 | 主 Actor | 归属特性域 | 优先级 |
|---------|----------|----------|------------|--------|
| **KV Cache 写入存储** | 将推理引擎生成的 KV Cache 异步写入外部存储，释放 HBM 压力，使单 GPU 可服务更长上下文 | 推理引擎、推理服务开发者 | cache-offloading | **P0** |
| **KV Cache 读取加载** | 从外部存储按需加载 KV Cache 到推理引擎，支撑长上下文和多轮对话场景 | 推理引擎、推理服务开发者 | cache-offloading | **P0** |
| **Prefix Cache 匹配** | 基于 token 序列前缀匹配命中历史缓存，避免重复计算，TTFT 可降低 50-80% | 推理引擎、推理服务开发者 | cache-offloading | **P0** |
| **PD 分离传输** | 支持 Prefill 节点到 Decode 节点的 KV Cache 高效传输，使 PD 分离架构可行 | 推理引擎、AI 基础设施架构师 | cache-offloading | **P1** |
| **热点数据多副本** | 对频繁访问的 KV Cache 块自动多副本复制，避免单节点网络拥塞 | 推理引擎、集群运维工程师 | cache-offloading | **P1** |

**价值论证（P0 Feature）**：

- **KV Cache 写入存储**：标杆研究（MoonCake、LMCache）均将此列为最核心的基础能力。没有写入能力，产品无法释放 HBM 压力，也就失去了存在的意义。MoonCake 生产环境验证：有效请求容量提升 59%-498%。
- **KV Cache 读取加载**：与写入配对形成完整读写闭环。LMCache 数据显示：多轮对话场景 TTFT 从 28s 降至 3.7s（7.7x 提升）。读取延迟直接决定用户体验。
- **Prefix Cache 匹配**：MoonCake 数据显示命中率 30%-50% 时 TTFT 降低 84%。LMCache 将此列为核心场景（RAG、多轮对话）的前提。没有 Prefix 匹配，缓存的价值大打折扣。
- **PD 分离传输**：PD 分离已成为主流推理优化架构（vLLM v1、SGLang 均支持）。MoonCake 的 Transfer Engine 实现单节点 190GB/s 传输。没有高效传输，PD 分离架构无法落地。

---

#### 特性域：data-lifecycle（数据生命周期管理）

> 本特性域覆盖数据分层存储、生命周期管理、压缩、快照克隆、跨区域复制。是产品作为"企业级存储"而非"临时缓存"的核心能力域。

| Feature | 价值摘要 | 主 Actor | 归属特性域 | 优先级 |
|---------|----------|----------|------------|--------|
| **数据分层存储** | 按热度自动在 HBM/DRAM/SSD 间迁移数据，平衡性能与成本 | 推理引擎、集群运维工程师 | data-lifecycle | **P0** |
| **数据冗余与自愈** | 多副本保护，磁盘/节点故障自动重建，保证数据不丢 | 集群运维工程师、MaaS 平台运营者 | data-lifecycle | **P0** |
| **数据压缩** | 对 KV Cache 数据进行压缩存储，降低存储成本 30-50% | 集群运维工程师、MaaS 平台运营者 | data-lifecycle | **P1** |
| **快照与克隆** | 支持 KV Cache 数据的时间点快照和快速克隆，支撑模型版本回滚 | 推理服务开发者、集群运维工程师 | data-lifecycle | **P1** |
| **跨区域复制** | 跨数据中心复制 KV Cache 数据，支撑异地多活和灾备 | MaaS 平台运营者、AI 基础设施架构师 | data-lifecycle | **P2** |

**价值论证（P0 Feature）**：

- **数据分层存储**：标杆共识（MoonCake、LMCache 均实现 GPU→CPU→磁盘→远程分层）。用廉价存储扩展缓存容量是"以存储换计算"的核心逻辑。MinIO AIStor 的 DRAM 缓存面向训练场景，我方需面向推理场景优化。
- **数据冗余与自愈**：MinIO 和 Ceph 均将数据保护列为 P0。MoonCake/LMCache 缺失持久化保证是其最大短板。作为存储系统（非缓存系统），数据不丢是底线要求。Ceph 的 11 个 9 耐久性是行业标准参考。

#### 特性域：cluster-management（集群管理）

> 本特性域覆盖集群扩缩容、数据重平衡、容量预测告警、配置模板。是运维效率和生产可用性的保障域。

| Feature | 价值摘要 | 主 Actor | 归属特性域 | 优先级 |
|---------|----------|----------|------------|--------|
| **集群扩缩容** | 在线添加/移除存储节点，数据自动重平衡，业务无感知 | 集群运维工程师、AI 基础设施架构师 | cluster-management | **P0** |
| **可观测性** | Prometheus 指标、结构化日志、分布式追踪，支撑故障诊断和性能调优 | 集群运维工程师、推理服务开发者 | cluster-management | **P1** |
| **容量预测与告警** | 基于历史趋势预测存储容量需求，提前告警避免耗尽 | 集群运维工程师、MaaS 平台运营者 | cluster-management | **P1** |
| **配置模板化** | 提供场景化配置模板（高吞吐/低延迟/均衡），降低调优门槛 | 推理服务开发者、集群运维工程师 | cluster-management | **P1** |

**价值论证（P0 Feature）**：

- **集群扩缩容**：Ceph 的在线扩容零停机是核心卖点。MoonCake 支持动态节点加入（80s 内完成）。作为企业级存储，在线扩缩容是基本能力。MinIO 开源版 2025-12 起停止支持集群扩展是反面教材。
- **可观测性**：LMCache 的可观测性建设滞后（v0.4.3 才引入 EventBus + OTel），导致运维困难。MoonCake 监控指标不够丰富。我方应在设计初期就纳入完整的可观测性体系。Prometheus 是云原生监控事实标准。

#### 特性域：security-governance（安全与治理）

> 本特性域覆盖多租户隔离、QoS、数据加密、审计日志、身份认证。是产品从"开源工具"升级为"企业级产品"的关键能力域，也是与 MoonCake/LMCache 最大的差异化域。

| Feature | 价值摘要 | 主 Actor | 归属特性域 | 优先级 |
|---------|----------|----------|------------|--------|
| **多租户隔离** | 租户间数据不可见，资源配额独立，满足共享集群安全要求。包含三个子维度：**数据隔离**（命名空间/RBAC）、**性能隔离**（QoS 限流）、**审计证明**（操作审计日志） | MaaS 平台运营者、AI 基础设施架构师 | security-governance | **P0** |
| **数据加密** | 传输加密（TLS 1.3）+ 静态加密（AES-256-GCM），密钥由外部 KMS 管理 | 安全合规官、外部 KMS | security-governance | **P1** |
| **QoS 性能保障** | 为不同租户/模型分配 IOPS/带宽/延迟 SLA，防止 noisy neighbor。作为多租户隔离的子维度在 P0 交付基础限流，P1 阶段完善细粒度 SLA | MaaS 平台运营者、推理引擎 | security-governance | **P1** |
| **审计日志** | 记录所有数据访问和管理操作，满足合规审计要求。作为多租户隔离的子维度在 P0 交付基础日志，P1 阶段完善 SIEM 集成和合规报表 | 安全合规官、SIEM 系统 | security-governance | **P1** |

**价值论证（P0 Feature）**：

- **多租户隔离**：MoonCake 仅 tenant_id 基础隔离，LMCache 仅 cache_salt 简单隔离，均无法满足企业级多租户需求。MinIO IAM + Ceph 账户级硬隔离是参考标准。MaaS 平台运营者将此列为最高优先级关注点。
- **数据加密**：MinIO 和 Ceph 均将传输+静态加密列为 P0。MoonCake/LMCache 无原生加密是明确短板。金融/政务客户将数据加密列为高优先级需求，但不是一票否决项（可通过外部 KMS 对接或客户自管理密钥满足合规）。

#### 特性域：engine-integration（引擎集成）

> 本特性域覆盖推理引擎 Connector、S3 API 兼容。是产品融入生态、降低接入门槛的关键域。

| Feature | 价值摘要 | 主 Actor | 归属特性域 | 优先级 |
|---------|----------|----------|------------|--------|
| **推理引擎 Connector** | 提供 vLLM/SGLang/Dynamo 等框架的即插即用集成 | 推理服务开发者、推理引擎 | engine-integration | **P0** |
| **S3 API 兼容** | 支持主流对象存储操作（PUT/GET/LIST/DELETE），兼容现有工具链 | 应用开发者、AI 基础设施架构师 | engine-integration | **P1** |

**价值论证（P0 Feature）**：

- **推理引擎 Connector**：LMCache 与 vLLM 团队共同维护的标准化 Connector 是其生态成功的关键。MoonCake 的 vLLM/SGLang 官方支持也是核心优势。没有 Connector，存储系统无法被推理引擎使用。
- **S3 API 兼容**：MinIO 将 100% S3 兼容作为核心卖点，Ceph 也将 S3 兼容列为 P0。S3 API 已成为对象存储的事实标准。兼容 S3 意味着兼容现有工具链（AWS CLI/SDK、Terraform、Spark 等），大幅降低迁移成本。

### 优先级统计

| 优先级 | 数量 | Feature 列表 |
|--------|------|-------------|
| **P0** | 8 | KV Cache 写入存储、KV Cache 读取加载、Prefix Cache 匹配、数据分层存储、数据冗余与自愈、集群扩缩容、多租户隔离、推理引擎 Connector |
| **P1** | 11 | PD 分离传输、热点数据多副本、数据压缩、快照与克隆、容量预测与告警、配置模板化、QoS 性能保障、审计日志、数据加密、可观测性、S3 API 兼容 |
| **P2** | 1 | 跨区域复制 |

> **P0 压缩说明**：基于双评审意见，P0 从 12 个压缩到 8 个：
> 1. **保留 P0**（8 个）：核心存储底座（数据分层、数据冗余、集群扩缩容、多租户隔离）+ 核心 AI 能力（KV Cache 读写、Prefix 匹配、Connector）
> 2. **降级 P1**（4 个）：PD 分离传输（实现复杂度高，P0 稳定后接入）、S3 API 兼容（通用能力，存储底座稳定后接入）、数据加密（私有化场景非所有客户刚需，P1 覆盖）、可观测性（MVP 基础指标即可，P1 完善）
> 3. **多租户隔离 P0 包含三个子维度**：数据隔离（命名空间/RBAC）+ 性能隔离（QoS 基础限流）+ 审计证明（基础审计日志）。QoS 性能保障和审计日志作为独立 Feature 保留在 P1，用于完善细粒度 SLA 和 SIEM 集成
> 4. **开发顺序**：通用存储底座先行（P0 中 4 个通用能力）→ AI 专用能力叠加（P0 中 4 个 AI 能力）→ 增强能力完善（P1 阶段）

### 隐含基础设施

> 以下能力虽未被列为独立 Feature，但所有 P0 Feature 均依赖其存在。架构设计时必须将元数据服务作为首个 ADR 决策，因其设计选择（集中式/分布式、强一致/最终一致）将直接影响所有子系统的设计。

| 基础设施 | 被依赖的 P0 Feature | 设计约束 |
|----------|---------------------|----------|
| **元数据服务** | 多租户隔离（租户元数据）、Prefix Cache 匹配（token→存储位置索引）、数据分层存储（热度元数据）、数据冗余与自愈（副本放置元数据）、集群扩缩容（节点拓扑/分片映射） | 必须支持租户级命名空间隔离，为数据加密的密钥隔离预留扩展点 |
| **身份认证服务** | 多租户隔离（租户识别）、审计日志（操作者身份） | 支持 LDAP/OIDC/SAML，故障时本地缓存降级 |
| **I/O 调度层** | 数据分层存储（本地迁移）、数据冗余与自愈（副本重建）、集群扩缩容（重平衡） | 三个调度器独立：本地迁移调度器（毫秒级）、副本重建调度器（秒级）、重平衡调度器（分钟级），禁止合并为单一模块 |

**架构风险预警**：
- 元数据服务若采用集中式设计（如单节点 etcd），会成为单点瓶颈；若采用分布式设计（如基于 Raft），会增加复杂度但提高可用性。此决策必须在架构设计阶段作为首个 ADR 产出。
- 数据分层存储（垂直数据流：HBM→DRAM→SSD）与集群扩缩容（水平数据流：节点间重平衡）是两种不同的数据流动模式，架构设计时禁止混为一谈。

## 全局非功能需求（第 3 轮量化）

> **轮次标注**：本章节第 1 轮仅列出目标维度框架，第 3 轮完成具体目标值量化。
> **量化依据**：基于 8 份标杆研究的实际数据（MoonCake、LMCache、MinIO、Ceph、VAST Datastore、Azure Blob Storage）+ 行业标准参考。

### 性能需求

#### KV Cache 读写性能

| 指标 | 目标值 | 边界条件 | 验收方法 |
|------|--------|----------|----------|
| **HBM 层写入延迟** | P99 ≤ 50 μs | 单次写入 ≤ 1MB（典型 KV Cache chunk），无并发拥塞 | 基准测试：单连接顺序写入 10 万次，测量 P99 延迟 |
| **DRAM 层写入延迟** | P99 ≤ 200 μs | 单次写入 ≤ 1MB，节点内本地写入 | 基准测试：单连接顺序写入 10 万次，测量 P99 延迟 |
| **SSD 层写入延迟** | P99 ≤ 2 ms | NVMe SSD，队列深度 32，单次写入 ≤ 1MB | 基准测试：fio 顺序写入测量 |
| **HBM 层读取延迟** | P99 ≤ 30 μs | 单次读取 ≤ 1MB，缓存命中 | 基准测试：随机读取 10 万次，测量 P99 延迟 |
| **DRAM 层读取延迟** | P99 ≤ 150 μs | 单次读取 ≤ 1MB，缓存命中 | 基准测试：随机读取 10 万次，测量 P99 延迟 |
| **SSD 层读取延迟** | P99 ≤ 1.5 ms | NVMe SSD，直接读取无压缩 | 基准测试：fio 随机读取测量 |
| **跨节点 RDMA 读取** | P99 ≤ 500 μs | 同一 RoCE v2 fabric 内，MTU 4096 | 基准测试：跨节点 RDMA 读取 10 万次 |
| **PD 分离传输吞吐** | ≥ 10 GB/s/节点 | MoonCake 基准：190 GB/s（单机），我方多节点集群目标按 5% 收敛系数 | 基准测试：Prefill 节点到 Decode 节点 1GB KV Cache 传输 |
| **Prefix 匹配命中率** | 30%-50%（典型工作负载） | RAG/多轮对话场景，公共前缀占比 20%-40% | 实际工作负载 trace 回放，统计命中率分布 |

**分层存储层级定义**：

| 层级 | 介质 | 典型容量/节点 | 典型带宽 | 典型 IOPS | 适用场景 |
|------|------|--------------|----------|-----------|----------|
| **L0（热层）** | HBM（GPU 显存） | 256 GB/卡 | 3.5 TB/s | — | 推理引擎本地热缓存（**推理引擎管理**，存储系统通过 Connector 感知容量状态） |
| **L1（温层）** | DRAM DDR5 | 256-512 GB/节点 | 200-400 GB/s | 2-5M | 热点 KV Cache 块（**存储系统管理**） |
| **L2（凉层）** | NVMe SSD | 4-16 TB/节点 | 14 GB/s（读）/ 7 GB/s（写） | 500K-1M | 中频访问数据（**存储系统管理**） |
| **L3（冷层）** | 网络远端存储 | PB 级集群 | 受网络拓扑限制 | 受网络拓扑限制 | 归档数据、跨区域复制目标（**存储系统管理**） |

> **职责边界说明**：L0（HBM）由推理引擎管理，存储系统的数据分层策略覆盖 L1-L3 层。各特性域文档中的分层描述统一使用此 4 层体系，L0 层标注为推理引擎管理。
>
> **标杆参考**：MoonCake 分层（HBM→CPU→磁盘），LMCache 分层（GPU→CPU→后端存储），MinIO AIStor DRAM 缓存（用于训练场景）。

| 指标 | 目标值 | 边界条件 | 验收方法 |

### 一致性需求

| 指标 | 目标值 | 边界条件 | 验收方法 |
|------|--------|----------|----------|
| **写入全局可见延迟** | ≤ 10 ms（同步复制） | 3 副本全写入后返回，集群内同机房 | 压测工具：写入后立即读取，验证一致性 |
| **副本一致性** | 强一致性（同步复制模式） | 副本数 ≥ 3，同步写入半数以上节点 | 故障演练：写入后故障部分节点，验证数据完整性 |
| **故障恢复一致性** | 无脏读、无幻读 | 故障切换后，旧主节点数据不泄露 | 故障演练：主节点故障，验证新主节点数据无异常 |
| **多客户端并发一致性** | 读-your-writes | 同一客户端写入后立即读取，必须返回最新值 | 并发压测：多客户端读写，验证读-your-writes |
| **分布式事务隔离级别** | 支持 Read Committed（默认） | 跨节点事务，租户内操作 | 并发压测：验证无脏读、无不可重复读（可配置升级到 Repeatable Read） |

> **标杆参考**：Ceph 强一致性（RADOS），MinIO 最终一致性（配置可升为强一致），MoonCake/LMCache 无一致性保证（缓存语义）。

### 安全性需求

#### 加密标准

| 指标 | 目标值 | 边界条件 | 验收方法 |
|------|--------|----------|----------|
| **传输加密** | TLS 1.3（默认），TLS 1.2（兼容模式） | 所有客户端-服务器通信，RDMA over Converged Ethernet (RoCE) 使用 MACsec | 安全审计：抓包分析，验证无明文传输 |
| **静态加密** | AES-256-GCM（数据）、AES-256-XTS（磁盘） | 数据落盘前加密，密钥由外部 KMS 管理 | 安全审计：dd if=/dev/urandom 读取磁盘，验证无明文 |
| **密钥管理** | 支持 KMIP 1.4+、HashiCorp Vault API | 外部 KMS 对接，本地密钥缓存降级 | 安全审计：KMS 故障时验证降级策略 |
| **租户级密钥隔离** | 不同租户使用不同 DEK（数据加密密钥） | KEK（密钥加密密钥）由 KMS 管理 | 安全审计：验证跨租户密钥隔离 |

#### 密钥生命周期

| 指标 | 目标值 | 边界条件 | 验收方法 |
|------|--------|----------|----------|
| **密钥轮换** | 支持自动轮换，周期可配置（默认 90 天） | DEK 轮换时旧数据自动重新加密 | 密钥轮换演练：触发轮换，验证新旧数据均可用 |
| **密钥缓存有效期** | ≤ 1 小时（本地缓存密钥） | KMS 故障时降级使用缓存密钥 | 故障演练：KMS 不可达，验证缓存降级 |
| **密钥撤销** | 支持即时撤销，撤销后 ≤ 1 分钟生效 | KMS 撤销命令到达存储节点 | 安全审计：撤销密钥后，验证旧密钥无法解密新数据 |

> **标杆参考**：MinIO AES-256-GCM + KMS，Azure Blob Storage AES-256 + Azure Key Vault，Ceph dm-crypt/LUKS + Barbican。

## 术语表

| 术语 | 定义 | 英文全称/别名 | 备注 |
|------|------|---------------|------|
| KV Cache | 键值缓存，LLM 推理过程中生成的键值对张量，用于加速自注意力计算 | Key-Value Cache | 也称 KV 缓存、KV 张量 |
| HBM | 高带宽内存，GPU 显存 | High Bandwidth Memory | 如 HBM2e、HBM3 |
| TTFT | 首 token 时间，从请求发起到第一个生成 token 返回的延迟 | Time To First Token | 推理核心性能指标 |
| TBT | 词间时间，连续生成 token 的平均间隔延迟 | Time Between Tokens | 也称 inter-token latency |
| PD 分离 | 将 LLM 推理的 Prefill 阶段（计算密集型）和 Decode 阶段（内存密集型）分到不同 GPU 执行 | Prefill-Decode Disaggregation | 主流推理优化架构 |
| Prefix Cache | 基于输入 token 序列前缀匹配的缓存机制，命中时可跳过重复计算 | Prefix Caching | MoonCake 称 Prefix KV Cache |
| RDMA | 远程直接内存访问，绕过 CPU 直接在网络设备间传输数据 | Remote Direct Memory Access | 本系统优先传输方式 |
| RoCE v2 | 基于以太网的 RDMA 协议 | RDMA over Converged Ethernet v2 | 私有化部署常用 |
| 纠删码 | 数据冗余技术，通过数学编码将数据分片存储，容忍部分节点故障 | Erasure Coding | — |
| MaaS | 模型即服务，对外提供 LLM API 服务的平台 | Model-as-a-Service | 目标客户类型之一 |
| TCO | 总拥有成本，包含硬件、软件、运维、电力等全部成本 | Total Cost of Ownership | 本系统核心价值主张 |
| CSI | 容器存储接口，K8s 中存储插件的标准接口 | Container Storage Interface | 本系统部署集成方式 |
| KMS | 密钥管理服务，管理加密密钥的生命周期 | Key Management Service | 外部系统，本系统需对接 |
| QoS | 服务质量，为不同用户/流量分配资源保障 | Quality of Service | 防止 noisy neighbor |
| RBAC | 基于角色的访问控制 | Role-Based Access Control | 权限模型之一 |
| ABAC | 基于属性的访问控制 | Attribute-Based Access Control | 权限模型之一，更细粒度 |
| Noisy Neighbor | 相邻租户/流量占用过多资源，影响其他用户 | — | QoS 需解决的问题 |
| Goodput | 有效吞吐，单位时间内成功处理的请求数（排除超时/失败） | — | 推理服务核心指标 |
| SIEM | 安全信息与事件管理，收集和分析安全日志的系统 | Security Information and Event Management | 审计日志的下游消费方 |
| KMIP | 密钥管理互操作协议，存储系统与 KMS 对接的标准协议 | Key Management Interoperability Protocol | 加密密钥管理 |
| SCIM | 跨域身份管理系统，用于用户/组自动同步 | System for Cross-domain Identity Management | 身份源同步协议 |
