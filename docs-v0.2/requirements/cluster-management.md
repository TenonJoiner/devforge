## 概述

本特性域覆盖集群**扩缩容**、**可观测性**、**容量预测**、**配置模板化**。是运维效率和生产可用性的保障域。

## Feature 价值论证

### F1: 集群扩缩容（P0）

#### 痛点

存储集群的容量需求随业务增长动态变化：
- 新租户接入、推理请求量增长时，需要快速扩容避免容量告急
- 业务低谷期需要缩容降低成本
- 扩缩容期间业务不能中断

标杆数据：
- Ceph：在线扩容零停机是核心卖点
- MoonCake：支持动态节点加入（80s 内完成）
- MinIO 开源版 2025-12 起停止支持集群扩展（反面教材）

#### 当前方案不足

- **MoonCake/LMCache 扩缩容能力弱**：缺乏自动化扩缩容和在线重平衡
- **重平衡影响业务**：扩缩容时数据迁移占用网络/磁盘带宽，影响读写性能
- **缩容风险高**：缩容时数据迁出若不慎，可能导致数据丢失

#### 用户价值

在线添加/移除存储节点，数据自动重平衡，**业务无感知**：
- 扩容期间吞吐下降 ≤ 20%，延迟增加 ≤ 2x
- 缩容有安全确认流程，避免数据丢失
- 扩缩容全程可监控进度

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 扩容期间业务影响 | 吞吐下降 ≤ 20%，延迟增加 ≤ 2x | 扩容演练：在线扩容，测量业务指标 |
| 缩容安全确认 | 缩容前显示预估影响，运维确认后执行 | 缩容演练：验证安全确认流程 |
| 重平衡速率控制 | 可配置最大迁移速率（默认 20MB/s/节点） | 配置测试：调整速率，验证生效 |
| 扩缩容失败回滚 | 失败时自动回滚，不留中间状态 | 故障演练：扩缩容中断，验证回滚 |
| 扩缩容进度可见 | 进度百分比和 ETA 可通过 API/CLI 查询 | API 测试：查询扩缩容状态 |

### F2: 可观测性（P1）

#### 痛点

存储集群的运维依赖于全面的监控和诊断能力：
- LMCache v0.4.3 才引入 EventBus + OpenTelemetry，监控能力建设滞后
- MoonCake 监控指标不够丰富，运维人员需自行搭建
- 分布式追踪覆盖不足，跨组件故障定位困难

标杆数据：
- Prometheus 是云原生监控事实标准
- MinIO/Ceph 提供全面的指标覆盖
- 告警规则模板是降低运维门槛的关键

#### 当前方案不足

- **指标覆盖不全**：KV Cache 专用指标（如命中率、分层分布）缺失
- **告警规则从零配置**：缺乏开箱即用的告警模板
- **分布式追踪缺失**：跨组件故障时难以定位根因

#### 用户价值

Prometheus 指标、结构化日志、分布式追踪，**支撑故障诊断和性能调优**：
- 关键指标全覆盖（8 类指标，100% 覆盖）
- 开箱即用的 Grafana Dashboard
- 告警规则模板（覆盖 20+ 常见故障场景）

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 指标端点响应时间 | P99 ≤ 10ms（/metrics 端点） | 基准测试：100 并发抓取，测量延迟 |
| 指标覆盖率 | 100%（8 类关键指标全覆盖） | 代码审计：对照指标清单逐项检查 |
| Dashboard 模板 | 提供 ≥ 3 个场景化 Dashboard（集群总览/租户分析/性能诊断） | UI 测试：Dashboard 功能完整性 |
| 告警规则模板 | 提供 ≥ 20 个常见故障告警规则 | 配置测试：导入规则，验证触发 |
| 追踪 Span 覆盖率 | 关键路径 Span 覆盖率 ≥ 95% | 代码审计：追踪关键调用路径 |

### F3: 容量预测与告警（P1）

#### 痛点

存储容量耗尽是生产环境的重大风险：
- 容量耗尽前往往没有预警，突然发生时影响业务
- 扩容规划需要提前数周准备硬件和资源
- 多租户环境下，租户级容量管理更加复杂

#### 当前方案不足

- **无容量预测**：仅支持阈值告警，无法预测耗尽时间
- **租户级容量管理缺失**：只能看全局容量，无法区分租户
- **扩容建议缺失**：无法给出最优扩容方案

#### 用户价值

基于历史趋势预测存储容量需求，**提前告警避免耗尽**：
- 容量预测提前 7-30 天预警（可配置）
- 预测模型考虑季节性和增长趋势
- 租户级容量配额告警

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 预测准确性 | 7 天预测误差 ≤ 15% | 历史数据验证：对比历史预测与实际 |
| 预测提前量 | 至少提前 7 天预警 | 演练：注入容量增长，验证预警时间 |
| 告警分级 | 支持 INFO/WARN/CRITICAL 三级 | 配置测试：触发各级告警 |
| 预测模型可配置 | 支持线性/指数/季节性模型 | 配置测试：切换模型，验证生效 |
| 租户级预测 | 支持按租户维度预测 | 数据验证：多租户场景验证准确性 |

### F4: 配置模板化（P1）

#### 痛点

MoonCake/LMCache 的配置参数多达 40-50 项，缺乏场景化模板：
- 推理工程师难以选择正确的配置组合
- 不同场景（高吞吐/低延迟/均衡）需要不同配置
- 配置错误导致性能下降或资源浪费

标杆数据：
- MoonCake 配置复杂，运维人员需深入理解内部机制
- MinIO 提供场景化配置模板（如"高性能"、"高可用"）

#### 当前方案不足

- **配置参数繁多**：40-50 项参数，理解和调优成本高
- **缺乏场景化模板**：无法快速选择适合场景的配置
- **配置验证缺失**：错误的配置组合可能引发问题

#### 用户价值

提供场景化配置模板（高吞吐/低延迟/均衡），**降低调优门槛**：
- 开箱即用场景模板（3+ 个场景）
- 模板参数有解释和推荐值
- 配置变更可预览影响

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 模板数量 | ≥ 3 个场景模板（高吞吐/低延迟/均衡） | UI 测试：验证模板存在和选择 |
| 模板覆盖场景 | 覆盖 ≥ 80% 的常见使用场景 | 用户调研：对比实际场景与模板 |
| 配置预览 | 变更前可预览参数差异 | UI 测试：配置变更预览功能 |
| 配置验证 | 应用配置前进行语法和合理性验证 | 配置测试：注入错误配置，验证被拒绝 |

## 场景（Scenario）

### S1: 在线扩容（添加存储节点）

**类型**：运维场景
**Feature**：集群扩缩容（F1）

#### 前置条件

- 现有集群 5 节点，存储使用率 75%
- 新节点硬件已就绪，网络连通正常
- K8s 环境正常

#### 用户旅程

```
T+0min   运维工程师创建新节点配置
    ↓
T+1min   K8s 调度新 Pod 启动，新节点注册到元数据服务
    ↓
T+2min   新节点状态：JOINING，接收元数据同步
    ↓
T+5min   元数据同步完成，新节点标记为 ACTIVE
    ↓
T+5min   重平衡调度器开始迁移数据（限速 20MB/s/节点）
    ↓
T+60min  数据迁移完成（1TB，限速 20MB/s ≈ 50min）
    ↓
T+60min  集群存储使用率从 75% 降至 62%
    ↓
T+60min  运维工程师验证集群健康，扩容完成
```

#### 验收标准

- [ ] 新节点加入期间读写请求正常
- [ ] 扩容期间吞吐下降 ≤ 20%
- [ ] 扩容期间延迟增加 ≤ 2x
- [ ] 扩容完成后数据完整性校验通过
- [ ] 扩容进度可通过 API 查询

### S2: 在线缩容（移除存储节点）

**类型**：运维场景
**Feature**：集群扩缩容（F1）

#### 前置条件

- 现有集群 5 节点，存储使用率 40%（可安全缩容）
- 缩容节点数据可完整迁移到剩余节点

#### 用户旅程

```
T+0min   运维工程师执行缩容计划（移除节点 D）
    ↓
T+1min   系统显示缩容影响评估：
         - 需要迁移数据：500GB
         - 迁移时间预估：25 分钟（20MB/s）
         - 缩容后剩余节点使用率：50%
    ↓
T+2min   运维工程师确认缩容
    ↓
T+2min   系统进入缩容准备阶段（禁止新数据写入节点 D）
    ↓
T+5min   数据迁移开始（从节点 D 到剩余 4 个节点）
    ↓
T+30min  数据迁移完成，节点 D 清空
    ↓
T+31min  K8s 驱逐节点 D 的 Pod
    ↓
T+32min  缩容完成，剩余 4 节点集群运行正常
```

#### 验收标准

- [ ] 缩容前显示预估影响（数据量、时间、剩余节点容量）
- [ ] 缩容前需要运维确认（安全门控）
- [ ] 缩容期间数据不丢失
- [ ] 缩容完成后集群健康状态正常
- [ ] 缩容失败时可回滚

### S3: 磁盘故障触发自动修复

**类型**：故障场景
**Feature**：数据冗余与自愈（关联 data-lifecycle 域）

#### 故障注入

```
故障注入：节点 C 的 NVMe SSD 出现只读错误（smartctl 标记为 failing）
```

#### 系统响应

```
T+0s     存储层检测到磁盘只读错误
    ↓
T+5s     故障确认，节点 C 标记为 DEGRADED
    ↓
T+5s     告警：节点 C 磁盘故障，集群降级运行
    ↓
T+5s     调度器识别节点 C 上的数据副本需要重建
    ↓
T+10s    副本重建启动（从健康节点重建到备用节点）
    ↓
T+5min   500GB 数据重建完成
    ↓
T+5min   集群恢复到 HEALTHY 状态
    ↓
T+1min   运维工程师收到告警，开始处理磁盘更换
    ↓
T+30min  磁盘更换完成，节点 C 恢复正常
```

#### 验收标准

- [ ] 磁盘故障后 ≤ 30 秒触发告警
- [ ] 故障期间集群降级运行但不中断服务
- [ ] 副本重建自动启动，数据完整性保证
- [ ] 重建完成后集群恢复到 HEALTHY
- [ ] 故障详情可通过 Dashboard 查看

### S8: 指标端点高负载

```
触发条件：Prometheus 以 100 并发抓取 /metrics 端点
```

```
T+0ms    100 并发请求到达 /metrics 端点
    ↓

```

#### 验收标准

- [ ] 指标端点限流生效（超出并发的请求排队）
- [ ] 正常负载下 P99 ≤ 10ms
- [ ] 高负载下 P99 ≤ 500ms（排队延迟可接受）
- [ ] 限流有日志记录

### S11: 多租户并发运行（正常场景）

**类型**：正常场景
**Feature**：可观测性（F2）
**Actor**：集群运维工程师、MaaS 平台运营者

#### 前置条件

- 存储集群正常运行（5 节点）
- 已有 3 个租户（tenant-a、tenant-b、tenant-c）
- 各租户有不同配额和 QoS 配置

#### 用户旅程

```
T=0      tenant-a（高优先级）发送高吞吐请求
    ↓
T=0      tenant-b（标准优先级）发送常规请求
    ↓
T=0      tenant-c（低优先级）发送批量请求
    ↓
T=0      三个租户的请求并发处理，互不干扰
    ↓
T=1min   Grafana Dashboard 查看多租户指标：
         - tenant-a：吞吐量 5K ops/s，延迟 P99 150μs（QoS 保障满足）
         - tenant-b：吞吐量 2K ops/s，延迟 P99 200μs（QoS 满足）
         - tenant-c：吞吐量 1K ops/s，延迟 P99 500μs（尽力而为）
    ↓
T=1min   各租户配额使用率分开统计
    ↓
T=1min   集群运维工程师确认：多租户资源隔离正常，无 noisy neighbor
```

#### 验收标准

- [ ] 多租户并发请求被分别计费和统计
- [ ] Dashboard 支持按租户维度过滤指标
- [ ] 高优先级租户的 QoS 保障不被低优先级租户影响
- [ ] 租户间数据完全隔离（不可跨租户访问）
