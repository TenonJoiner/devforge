# 特性域：security-governance（安全与治理）

## 概述

本特性域覆盖**多租户隔离**、**数据加密**、**QoS 性能保障**、**审计日志**。是产品从"开源工具"升级为"企业级产品"的关键能力域，也是与 MoonCake/LMCache 最大的差异化域。

**与架构子系统的追溯关系**：
- 多租户隔离实现 → `docs/architecture/multi-tenant/design.md`

- 审计日志系统 → `docs/architecture/audit/design.md`

## Feature 价值论证

### F1: 多租户隔离（P0）

#### 痛点

MaaS 平台和企业 AI 基础设施需要多租户共享集群：
- 不同租户的数据必须完全隔离（金融合规要求）
- 租户间性能不能相互影响（noisy neighbor 问题）
- 租户管理员需要自助管理子资源

标杆数据：
- MinIO IAM + Ceph 账户级硬隔离是参考标准
- MoonCake 仅通过 tenant_id 基础隔离（不安全）
- LMCache 仅 cache_salt 简单隔离（无法防止恶意访问）

#### 当前方案不足

- **MoonCake/LMCache 无多租户能力**：仅 tenant_id 字段，无 RBAC、无细粒度权限
- **隔离强度不足**：恶意租户可能通过构造 tenant_id 访问其他租户数据
- **配额管理缺失**：租户可无限制占用资源

#### 用户价值

租户间数据不可见，资源配额独立，满足共享集群安全要求：
- 三个子维度：数据隔离（命名空间/RBAC）+ 性能隔离（QoS 限流）+ 审计证明（操作审计日志）
- P0 交付：基础命名空间隔离 + 基础 RBAC + 基础配额
- P1 完善：细粒度 RBAC + 细粒度 QoS SLA + SIEM 集成

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 数据隔离强度 | 100%（跨租户数据访问企图 100% 被拒绝） | 安全测试：构造跨租户访问请求，验证拒绝 |
| 配额强制 | 租户配额超限时写入被拒绝（硬限制） | 配额演练：租户达到配额上限 |
| 权限控制粒度 | 支持 ≥ 4 种预定义角色（Admin/Developer/Operator/Viewer） | 配置测试：验证角色存在和权限差异 |
| 认证协议 | LDAP v3 + OIDC（默认） | 集成测试：对接 LDAP/OIDC |
| 权限验证延迟 | ≤ 10ms（每次请求的权限检查） | 性能测试：权限检查延迟分布 |

### F2: 数据加密（P1）

#### 痛点

企业客户对数据安全有严格要求：
- 金融/政务客户要求静态数据加密（防止物理磁盘泄露）
- 传输加密防止网络窃听
- 密钥管理需要符合合规要求

标杆数据：
- MinIO：AES-256-GCM + KMS
- Azure Blob Storage：AES-256 + Azure Key Vault
- Ceph：dm-crypt/LUKS + Barbican

#### 当前方案不足

- **MoonCake/LMCache 无加密**：数据明文存储，无法满足企业需求
- **加密密钥管理缺失**：无外部 KMS 对接能力
- **加密对性能影响未知**：加密/解密开销需要评估

#### 用户价值

传输加密（TLS 1.3）+ 静态加密（AES-256-GCM），密钥由外部 KMS 管理：
- 传输加密默认启用，TLS 1.3
- 静态加密可选启用，AES-256-GCM
- 密钥由外部 KMS（KMIP/Vault）管理，支持自动轮换

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 传输加密 | TLS 1.3（默认），TLS 1.2（兼容） | 安全审计：抓包验证无明文 |
| 静态加密 | AES-256-GCM，数据落盘前加密 | 安全审计：磁盘读取验证无明文 |
| 密钥隔离 | 不同租户使用不同 DEK | 配置测试：验证租户密钥隔离 |
| 密钥轮换 | 自动轮换（默认 90 天），零停机 | 密钥演练：触发轮换，验证无中断 |
| 加密性能影响 | 加密后写入延迟 P99 增加 ≤ 15% | 对比测试：加密开/关的写入延迟 |
| KMS 故障降级 | KMS 不可达时使用本地缓存密钥（只读降级） | 故障演练：KMS 不可达，验证降级 |

### F3: QoS 性能保障（P1）

#### 痛点

多租户共享集群时，noisy neighbor 是最大风险：
- 一个租户的高负载可能拖垮其他租户的延迟 SLO
- 不同租户/模型可能有不同的优先级需求
- 需要按租户/命名空间/QoS 类别分配 IOPS/带宽/延迟保障

标杆数据：
- Ceph：支持 QoS（通过 mClock 调度器）
- MinIO：企业版支持 IOPS/带宽限制
- Kubernetes：支持按 QoS 类别（Guaranteed/Burstable/BestEffort）调度

#### 当前方案不足

- **MoonCake/LMCache 无 QoS**：租户间无性能隔离
- **LMCache 无公平调度**：多租户场景下资源竞争严重
- **缺乏延迟保障**：仅有吞吐限制，无延迟 SLA

#### 用户价值

为不同租户/模型分配 IOPS/带宽/延迟 SLA，防止 noisy neighbor：
- P0 基础限流：租户级 IOPS/带宽硬限制
- P1 细粒度 SLA：可配置保障值和最大值的 SLA 级别

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 租户限流精度 | 实际 IOPS 与配置值误差 ≤ 5% | 压测：租户达到上限，测量实际 IOPS |
| 限流响应时间 | 限流触发 ≤ 100ms（令牌桶刷新） | 压测：突发流量，测量限流生效时间 |
| 多租户公平性 | 共享资源时，租户间延迟差异 ≤ 2x | 压测：多租户并发，测量延迟分布 |
| 延迟 SLA 保障 | P99 延迟 ≤ 保障值（满足 SLA 的租户） | SLA 演练：配置延迟 SLA，注入负载 |
| 限流规则可动态更新 | 运行时修改限流参数，≤ 1s 生效 | 配置测试：修改限额，验证生效时间 |

### F4: 审计日志（P1）

#### 痛点

合规审计要求存储系统记录所有数据访问和管理操作：
- 金融/政务客户需要完整的操作审计（等保 2.0 要求）
- 审计日志需要不可篡改（WORM 模式）
- 审计日志需要支持 SIEM 集成

标杆数据：
- MinIO：完整的审计日志（每请求记录）
- Ceph：审计日志支持写入 CephFS 或外部 Syslog
- 等保 2.0 三级：要求记录所有管理员操作和数据访问

#### 当前方案不足

- **MoonCake/LMCache 无审计**：无操作日志记录
- **日志格式不统一**：难以对接 SIEM 系统
- **日志可篡改**：无 WORM 保护

#### 用户价值

记录所有数据访问和管理操作，满足合规审计要求：
- P0 基础审计日志：所有读写/管理操作
- P1 SIEM 集成：Syslog + OpenTelemetry 推送，开箱即用合规报告模板

#### 成功指标

| 指标 | 目标 | 验收方法 |
|------|------|----------|
| 审计日志完整性 | 100%（所有操作有对应记录） | 安全审计：随机抽样验证 |
| 审计日志不可篡改 | WORM 模式，一次写入不可修改/删除 | 安全测试：尝试修改日志 |
| 日志格式 | JSON 结构化，字段完整（时间/用户/操作/资源/结果/租户） | 日志审计：验证格式规范 |
| 日志查询性能 | 1 亿条日志内，复杂查询 ≤ 10s | 性能测试：大规模日志查询 |
| SIEM 推送延迟 | ≤ 5s（操作发生到 SIEM 收到） | 集成测试：推送延迟测量 |

## 场景（Scenario）

### S1: 租户开通与隔离验证

**类型**：正常场景
**Feature**：多租户隔离（F1）

#### 前置条件

- 管理员已完成身份认证（LDAP/OIDC）
- 存储集群运行正常

#### 用户旅程

```
T+0min   MaaS 平台运营者创建新租户
    ↓
T+1min   填写租户信息：
         - 租户名称：tenant-alice
         - 配额：DRAM 1TB，SSD 10TB
         - 管理员：alice@example.com
    ↓
T+2min   租户创建完成，alice@example.com 收到邀请
    ↓
T+3min   alice@example.com 登录，获得 tenant-alice 的 Admin 角色
    ↓
T+5min   alice 上传 KV Cache 数据（属于 tenant-alice）
    ↓
T+10min  bob@example.com（另一个租户）尝试访问 tenant-alice 的数据
    ↓
T+10min  访问被拒绝：403 Forbidden（跨租户隔离）
```

#### 验收标准

- [ ] 租户创建后，其他租户无法访问其数据
- [ ] 跨租户访问企图被 100% 拒绝
- [ ] 拒绝事件有审计日志记录
- [ ] 租户管理员可以管理本租户资源

### S2: 租户配额超限处理

**类型**：正常场景
**Feature**：多租户隔离（F1）

#### 触发条件

```
触发条件：租户 DRAM 配额 = 1TB，当前使用 = 1TB，新请求需要 100GB
```

#### 系统响应

```
T+0ms    租户发起新写入请求（100GB KV Cache）
    ↓

    ↓
T+1ms    决策：尝试 LRU 驱逐租户内最冷数据

T+10ms   LRU 驱逐 100GB，释放配额
    ↓
T+10ms   写入请求继续执行
    ↓
T+20ms   写入完成，配额使用 = 1TB
    ↓
T+30ms   告警：租户配额使用率达到 100%，建议扩容
```

#### 验收标准

- [ ] 配额超限时 LRU 驱逐生效
- [ ] 驱逐后写入继续执行
- [ ] 配额超限有告警通知
- [ ] 配额使用量指标准确

### S3: 跨租户数据访问企图（安全测试）

**类型**：异常场景
**Feature**：多租户隔离（F1）

#### 触发条件

```
触发条件：攻击者（租户 A）通过构造恶意 tenant_id 尝试访问租户 B 的数据
```

#### 系统响应

```
T+0ms    攻击者发送请求：kv_cache_read(chunk_id=123, tenant_id=tenant-b)
    ↓
T+1ms    鉴权层验证：请求的 tenant_id（tenant-b）与认证 Token 的 tenant_id（tenant-a）不一致
    ↓
T+2ms    请求被拒绝：403 Forbidden（租户 ID 伪造检测）
    ↓
T+2ms    安全事件 +1
    ↓
T+10ms   审计日志记录：
         - 时间：T+0ms
         - 源 IP：10.0.0.5
         - 租户：tenant-a
         - 操作：kv_cache_read
         - 资源：chunk_id=123
         - 拒绝原因：租户 ID 不匹配
         - 结果：DENIED
    ↓
T+1min   如果同一租户在 1 分钟内出现 ≥ 5 次拒绝，触发安全告警
```

#### 验收标准

- [ ] 租户 ID 伪造请求被 100% 拒绝
- [ ] 拒绝事件有详细审计日志
- [ ] 异常频率超阈值时有安全告警
- [ ] 拒绝不影响正常租户的请求

### S4: 外部 KMS 故障时的加密降级

**类型**：故障场景
**Feature**：数据加密（F2）

#### 故障注入

```
故障注入：外部 KMS 服务不可达（iptables DROP 所有到 KMS 的流量）
```

#### 系统响应

```
T+0s     存储节点请求 KMS 获取 DEK（数据加密密钥）
    ↓
T+1s     请求超时（KMS 不可达）
    ↓
T+1s     系统检测：KMS 不可达，切换到本地缓存 DEK
    ↓
T+2s     缓存 DEK 有效（缓存有效期 1 小时），读取继续
    ↓
T+2s     告警：KMS 连接失败，使用缓存 DEK 降级运行
    ↓
T+2s     写入操作降级：新数据使用缓存 DEK 加密（而非 KMS 新生成的 DEK）
    ↓
T+30min  KMS 恢复连接
    ↓
T+31min  系统自动切回 KMS 管理模式
    ↓
T+31min  KMS 连接恢复正常告警
```

#### 验收标准

- [ ] KMS 故障时 ≤ 5s 检测并降级
- [ ] 降级期间读取服务不中断（使用缓存 DEK）
- [ ] 降级期间写入继续（新数据用缓存 DEK）
- [ ] KMS 恢复后自动切回，无数据丢失
- [ ] 降级事件有告警和日志

### S5: 密钥轮换（日常维护）

**类型**：运维场景
**Feature**：数据加密（F2）

#### 前置条件

- 数据加密已启用
- KMS 连接正常
- 密钥轮换策略：90 天自动轮换

#### 用户旅程

```
T+0s     KMS 触发密钥轮换事件（到期前 7 天通知）
    ↓
T+0s     存储系统收到轮换通知
    ↓
T+0s     决策：旧 DEK 保持可解密，新 DEK 用于新数据
    ↓
T+0s     新数据使用新 DEK 加密
    ↓

    ↓
T+2hour  旧 DEK 标记为废弃（仅用于解密历史数据）
    ↓
T+2hour  审计日志记录：密钥轮换完成，影响数据量 X GB
```

#### 验收标准

- [ ] 密钥轮换期间服务不中断
- [ ] 轮换后新旧数据均可正常读取
- [ ] 轮换过程有日志记录
- [ ] 轮换进度可通过指标监控

### S6: Noisy Neighbor 限流

**类型**：正常场景
**Feature**：QoS 性能保障（F3）

#### 触发条件

```
触发条件：租户 A 配置 IOPS 上限 10000，突发流量达到 15000 IOPS
```

#### 系统响应

```
T+0ms    租户 A 突发流量 15000 IOPS
    ↓
T+0ms    令牌桶限流器检查：当前令牌不足
    ↓
T+0ms    限流器排队（最大队列 100 请求）
    ↓
T+10ms   令牌补充，队列中的请求开始处理
    ↓
T+100ms  10000 IOPS 正常通过，5000 IOPS 被限流
    ↓
T+100ms  租户 A 收到限流响应：429 Too Many Requests
    ↓
T+200ms  租户 A 降低发送速率，限流解除
    ↓
T+200ms  租户 A 恢复正常 8000 IOPS
```

#### 验收标准

- [ ] 租户 IOPS 超限时精确限流（误差 ≤ 5%）
- [ ] 被限流请求收到 429 响应
- [ ] 限流期间其他租户服务不受影响
- [ ] 限流规则可动态更新

### S9: 合规审计准备（等保 2.0）

**类型**：运维场景
**Feature**：审计日志（F4）、多租户隔离（F1）

#### 前置条件

- 系统已运行 1 年
- 需要通过等保 2.0 三级审计

#### 用户旅程

```
T+0hour  安全合规官启动合规审计准备
    ↓
T+1hour  系统导出合规报告：
         - 用户清单及权限（10 个租户，50 个用户）
         - 访问控制策略（RBAC 规则）
         - 审计日志保留（365 天内所有操作）
```

### S10: 多租户公平调度（正常场景）

**类型**：正常场景
**Feature**：QoS 性能保障（F3）
**Actor**：MaaS 平台运营者

#### 前置条件

- 存储集群正常运行
- 两个租户已配置不同 SLA：
  - tenant-a：低延迟 SLA，P99 ≤ 500 μs（高级）
  - tenant-b：标准 SLA，P99 ≤ 2000 μs（标准）

#### 用户旅程

```
T=0      两个租户同时以满负载运行
    ↓
T=0      共享存储层资源（DRAM L1 层）
    ↓
T=0      QoS 调度器按 SLA 优先级分配资源：
         - tenant-a 的请求优先处理
         - tenant-b 的请求在资源充足时处理
    ↓
T=1min   监控数据：
         - tenant-a：P99 = 480 μs ✓（满足 SLA）
         - tenant-b：P99 = 1800 μs ✓（满足 SLA）
    ↓
T=5min   MaaS 平台运营者在 Dashboard 查看 SLA 达成率：
         - tenant-a：99.5%（SLA 达成）
         - tenant-b：99.8%（SLA 达成）
```

#### 验收标准

- [ ] 不同 SLA 级别的租户均满足各自 SLA（P99 ≤ 保障值）
- [ ] 高优先级租户不受低优先级租户影响
- [ ] Dashboard 显示各租户 SLA 达成率
- [ ] SLA 违规时有告警通知

### S11: 安全合规官—审计日志月度审查（正常场景）

**类型**：正常场景
**Feature**：审计日志（F4）
**Actor**：安全合规官

#### 前置条件

- 存储集群已运行 30 天
- 审计日志保留期配置为 365 天
- 安全合规官持有合规官角色

#### 用户旅程

```
T=0      安全合规官登录管理控制台
    ↓
T=1min   导航到「安全审计」→「审计日志」页面
    ↓
T=1min   设置查询条件：
         - 时间范围：过去 30 天
         - 操作类型：全部（读/写/删/管理）
         - 结果：全部（成功/失败）
    ↓
T=2min   查询返回结果（1 亿条日志，P99 ≤ 10s）
    ↓
T=2min   识别异常事件：
         - tenant-a 在 2026-04-15 01:00 出现 23 次认证失败
         - tenant-b 在 2026-04-20 14:30 出现了 5 次跨租户访问尝试（均被拒绝）
    ↓
T=10min  安全合规官生成月度审计报告（PDF）
    ↓
T=10min  报告包含：操作统计、异常事件摘要、安全建议
```

#### 验收标准

- [ ] 日志查询在 1 亿条规模下 P99 ≤ 10s
- [ ] 查询结果支持多条件过滤（时间/操作类型/结果/租户）
- [ ] 异常事件可视化高亮
- [ ] 审计报告可导出（PDF/Excel）
- [ ] 报告包含合规所需的全部要素（用户/时间/操作/结果）
