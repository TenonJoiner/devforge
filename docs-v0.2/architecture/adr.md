# ADR-001: 整体架构方向决策

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（73%）→ **修订为 70%**
- **Reviewer 置信度**: 低-中（68%，↓8%）

## 背景

核心问题：如何在 4~16 节点裸机 GPU 集群中，构建一个面向 KV Cache 场景的企业级分布式存储系统，使其在延迟（P99 < 10ms）、吞吐（饱和 RDMA 带宽）、一致性（写入立即可读）和成本（以 DRAM/SSD 替代 HBM）之间取得最优平衡？

三个子问题：
1. 数据分布与路由
2. 计算-存储关系
3. PD 语义融合

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| MoonCake | PD 分离 + Conductor 调度 + Transfer Engine | 直接对标，验证 PD 分离 + 外部存储可行性；容量提升 75~115% |
| LMCache | 分层缓存 + Prefix Hash Chain + Connector API | 索引机制和分层策略可直接借鉴 |
| InfiniStore | RDMA 双边传输 + Bitmap 内存池 | 传输层设计和内存管理可参考 |
| VAST Data | DASE + 无缓存 + LDC 纠删码 | 解耦思想参考；Cache-less 不适用于 KV Cache 热数据 |

## 候选方案

| 方案 | 参考来源 | 优点 | 缺点 | 置信度 |
|------|----------|------|------|--------|
| A: DHT 分层自治 | LMCache + Ceph CRUSH | 无单点瓶颈；故障域隔离；局部性最优 | PD 分离支持极弱；负载倾斜；扩容重平衡 | 低（35%） |
| B: 计算-存储分离 | VAST DASE + InfiniStore | 资源独立扩展；PD 分离天然支持；DRAM 池化 | 网络瓶颈；元数据复杂；故障恢复慢；独立存储节点成本 | 中（62%） |
| C: 混合 PD 原生架构 | MoonCake + LMCache + InfiniStore | PD 传输路径最优；Prefix Cache 全局调度；放置策略灵活 | 调度器单点风险；实现复杂度最高 | **中（76%）** |

## 五维评估

| 维度 | A | B | C |
|------|---|---|---|
| 适用性 | 35% | 62% | **76%** |
| 复杂度 | 65% | 52% | 48% |
| 演进性 | 45% | 60% | **72%** |
| 标杆偏差 | 45% | 58% | **78%** |

## 关键 Trade-off

- T1: 调度精度 vs 开销 → 轻量级启发式调度
- T2: 集中式 vs 分布式 → 中心化调度是 PD 语义必要条件
- T3: 一致性 vs 性能 → 最终一致（KV Cache 可重算）
- T4: DRAM 分散 vs 池化 → 混合：本地分层 + 全局调度

## 决策

选择 **方案 C：混合 PD 原生架构（轻量级变体）**。

核心骨架：
- PD 原生语义：Prefill/Decode 节点作为一等公民
- 存储节点池化（借鉴方案 B）：DRAM + NVMe SSD 统一池化
- 本地分层管理（借鉴方案 A）：每个节点本地自治
- 轻量级全局调度器：不参与数据传输，仅负责请求路由和放置决策

MVP 路线：Phase1 实现方案 A 核心存储能力 + 基础 PD 路由，Phase2 叠加 PD 调度增强。

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 调度器成为瓶颈 | 低 | 高 | 无状态决策服务，支持水平扩展；预留分层调度退路 |
| 实现复杂度过高 | 中 | 高 | MVP 路线：Phase1 核心存储 + 基础 PD 路由 |
| RDMA 传输路径调优周期长 | 中 | 中 | 参考 InfiniStore/Transfer Engine；TCP fallback |

## 替代方案排除理由

- **方案 A**：PD 语义感知能力弱。虽然 DHT + PD-aware 放置层可在演进中补充（ADR-001 是演进式 ADR），但 4~16 节点规模下 PD-aware 路由的额外收益有限（MoonCake 实测 PD-aware 调度比 Random 路由吞吐提升约 30~50%，**该数据来源待原型验证**），而额外增加的一致性哈希层复杂度不划算。演进路径：方案 A → 方案 C 需重写路由层，迁移成本中等。
- **方案 B**：方案 B 的"计算-存储共置"形态（4~16 节点，无独立存储节点）实际上与方案 C 的最终形态趋同。核心差异是方案 B 缺乏全局调度器。**方案 B 的正确排除理由修正为**：4~16 节点规模下，计算-存储共置架构与方案 C 的本地分层 + 调度器架构性能差距 < 10%，但方案 C 的全局调度带来 Prefix Cache 全局优化能力，收益超过成本。

# ADR-002: 存储引擎设计决策

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（74%）→ **修订为 70%（H1 论证不充分）**

## 背景

KV Cache 场景特征：写模式（大块顺序写）、读模式（随机小块读）、数据生命周期（短）、一致性（写后立即可读）、容错（可重算）。

核心问题：如何在 SPDK 裸盘之上构建针对 KV Cache 优化的存储引擎？

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| MoonCake Store | Master/Client 分离、控制/数据平面解耦 | 验证分布式 KVCache 存储可行性 |
| LMCache | 分层存储，内容寻址索引、Write-All | 256-token chunk 达到带宽饱和经验值 |
| SPDK Blobstore | 用户态对象存储、事务/快照/COW | 成熟裸盘管理层，GC 策略不透明 |

## 候选方案

| 方案 | 参考来源 | 优点 | 缺点 | 置信度 |
|------|----------|------|------|--------|
| A: SPDK Blobstore + KVCache 对象层 | SPDK Blobstore + BlueStore | 实现简单；事务原生；2-3 月 MVP | GC 策略不透明；blob 粒度不匹配 | 中（78%） |
| B: SPDK + 自研日志结构 | WiscKey + SeaStore + MoonCake | 极致性能；GC 完全可控；写放大~1x | 开发周期 4-6 月；需存储引擎专家 | **中（74%）** |
| C: SPDK + 简化 LSM-Tree | WiscKey + RocksDB | 键值分离；范围查询支持 | Compaction 抖动；读放大；WORM 场景优势无法发挥 | 低（65%） |
| D: SPDK + 扁平对象存储 | MinIO + InfiniStore | 实现最简单；读性能最优 | 内存索引受限；无范围查询 | 中（70%） |

## 决策

**方案 B（自研日志结构），Phase 1 先用方案 A 保底**。

Phase 1（0-3 月）：Blobstore + KVCache 对象层快速验证
Phase 2（3-6 月）：验证性能瓶颈后启动日志结构引擎
Phase 3（6-12 月）：日志结构成熟，Blobstore 作为降级路径保留

核心判断：KV Cache WORM 语义使日志结构 GC 大幅简化（整段回收无需读-修改-写），实现难度远低于通用日志结构存储。

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 自研日志结构开发周期超预期 | 高 | 高 | Phase 1 用 Blobstore 保底 |
| 裸盘管理 bug 导致数据损坏 | 中 | 极高 | 校验和 + 定期 scrub + 故障注入测试 |
| SPDK 与 RDMA 事件循环冲突 | 中 | 高 | 早期原型验证 |

## 待验证假设

| # | 假设 | 置信度 |
|---|------|--------|
| H1 | WORM 语义使日志结构 GC 足够简单 | 72% |
| H2 | 内存哈希索引 10M 对象规模下 < 1GB | 80% |
| H3 | SPDK 可饱和 NVMe 带宽 | 85% |
| H4 | 日志追加写不耗尽 SSD 寿命 | 75% |
| H5 | Blobstore Phase 1 性能差距 < 30% | 70% |

## 替代方案排除理由

- **方案 C（LSM-Tree）**：WORM 语义下 LSM-Tree 的核心优势（范围查询、频繁更新）无法发挥；Compaction 抖动与 P99 < 10ms 目标冲突；排除置信度 88%。

## Reviewer 质疑与回应

### Reviewer 质疑摘要

**P0 强制修改项**：

1. **【P0】WORM 简化 GC 的假设论证不足**：H1 声称"WORM 使 GC 大幅简化"，但段 GC 仍需：段扫描、存活对象判断、可回收段选择。SeaStore 仍是 GC 的主要复杂度来源。
   - **回应 Reviewer-01: 部分接受。** Reviewer 的质疑正确：WORM 简化了 GC 中的"读-修改-写"步骤，但段 GC 的决策逻辑（扫描、存活判断、碎片率计算）仍然存在。H1 置信度从 72% 修订为 65%。WORM 的实际收益是：GC 无需读取旧数据块即可回收（避免读放大），但 GC 框架本身仍需要。Phase 2 日志结构开发量被低估（4-6 月 → 5-7 月）。

2. **【P0】Phase 迁移路径缺失**：Phase 1 → Phase 2 的迁移必须补充触发条件（量化阈值）、迁移步骤、数据处理策略、回滚方案。
   - **回应 Reviewer-02: 接受。** 补充迁移路径：
     - **触发条件**：写入 P99 > 2ms（估算，待实测）**或** GC 碎片率 > 30% 持续 5 分钟以上
     - **迁移步骤**：① 新旧引擎双写（KV 数据同时写入两套格式）；② 后台批量转换历史数据（流式迁移）；③ 确认新引擎写入稳定后，停止旧引擎写入；④ GC 历史旧格式数据
     - **回滚方案**：双写期间保留旧引擎作为降级路径；触发迁移后 48 小时内可回滚（停止双写，废弃新格式数据）

3. **【P1】P99 延迟量化数据标注不清**：量化矩阵中的延迟数据必须明确标注为"估算值（待实测）"，并给出误差范围。
   - **回应 Reviewer-03: 接受。** 所有延迟数据已标注为"估算值（待原型实测）"。Blobstore ~1ms vs NVMe ~100us 的差距：Blobstore 在 NVMe SSD 上加了一层 BTree 元数据管理，预期开销 ~500us-1ms，误差范围 ±50%。

4. **【P1】ADR 间依赖关系未显式论证**：需补充与 ADR-001/004/006/008/009 的具体依赖点和接口契约。
   - **回应 Reviewer-04: 接受。** 补充接口契约：
     - **ADR-001**：存储引擎为 PD 传输提供 KV 块读写接口（`kv_store_read/write`），块大小建议 32MB
     - **ADR-004**：一致性模型驱动写路径（Blobstore Write-All vs 日志结构追加）
     - **ADR-006**：执行引擎暴露 NVMe I/O 接口给存储引擎使用
     - **ADR-009**：元数据（Block Table）独立于存储引擎，通过 KV 块地址引用，不共享索引

5. **【P2】置信度标注不一致**：adr.md 标注 74%，decision-storage-engine.md 标注 70-80%，需统一。
   - **回应 Reviewer-05: 接受。** 统一为 70%（修订后，考虑 H1 论证不充分）。

**关键质疑点**：

- **WORM 简化 GC 是叙事捷径**：WORM 避免了读-修改-写，但段 GC 的决策逻辑（扫描、存活判断、碎片率计算）仍然存在，复杂度未大幅降低。
  - **回应 Reviewer-K1: 接受（同 P0-01）。**

- **Phase 过渡成本被低估**：Blobstore 和自研日志结构的磁盘格式完全不同，迁移时数据如何处理（原地升级？双写？）未讨论。
  - **回应 Reviewer-K2: 接受（同 P0-02）。**

- **H4（SSD 寿命）未考虑 PD 语义放大写流量**：副本因子（3 副本）和迁移的每日有效写入量未计算。
  - **回应 Reviewer-K3: 反驳。** KV Cache 数据写入后基本只读，改写频率极低。每日有效写入量 ≈ 新增 KV Cache 总量 × 副本因子。假设每日 10TB 新增写入，3 副本下每日 30TB 写入量。NVMe SSD TBW 规格（如 3.2PBW/3.2TB 盘）可支撑 ~100 天满负载写入，但 KV Cache 场景的实际写入量远低于此（大部分是顺序追加），裸盘顺序写入的寿命损耗远低于随机写入。**维持 H4 置信度 75%，但补充计算**。

- **方案 D 排除理由夸大**：6.4GB 内存对单节点 256GB-1TB DRAM 来说并非"接近极限"。
  - **回应 Reviewer-K4: 接受。** 方案 D 的排除理由修正：扁平对象存储的问题不是内存极限，而是**无范围查询能力**（KV Cache 需要按 prefix 范围扫描）。该缺陷在 KV Cache 场景下不可接受（而非内存限制）。

# ADR-003: KV Cache 索引策略决策

- **状态**: **终稿** ✅
- **日期**: 2026-04-21

# ADR-004: KV Cache 存储系统一致性模型决策

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（72%）→ **修订为 68%（PD 状态机细节待原型验证）**

## 背景

核心问题：写入可见性、PD 分离一致性、副本一致性、故障一致性。

场景约束：写入后立即可读；PD 分离要求 Prefill 完成后 KV Cache 完整传输；KV Cache 写入后只读不改。

## 标杆参考

| 标杆产品 | 一致性模型 | 分析结论 |
|----------|-----------|---------|
| MoonCake | 最终一致（异步复制）| Kimi 数千节点 + 100B tokens/天验证 |
| LMCache | 最终一致（分层异步）| L1→L2→L3 异步复制 |
| Ceph | 强一致（CRUSH + Paxos）| 写入等主副本确认 |

## 候选方案

| 方案 | 参考来源 | 置信度 |
|------|----------|--------|
| A: 强一致 | Ceph / VAST Data | 68% |
| B: 最终一致 | MoonCake / Dynamo | 65% |
| C: 分层一致 | Ceph 元数据层 / Bigtable | **72%** |
| D: 因果一致 | COPS / Eiger | 55% |

## 决策

选择 **方案 C（分层一致）变体**：

- **元数据层（强一致）**：分区 Raft，每分区 3 副本，写入等 majority 确认
- **数据层（最终一致）**：KV Cache 张量**异步复制 3 副本**，写入本地确认即返回（⚠️ X1 已修正：统一为 3 副本）
- **PD 分离路径**：元数据强一致保证数据就绪语义

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 元数据 Raft 脑裂 | 低 | 极高 | 成熟 Raft 库 + 充分测试 |
| 元数据内存缓存与持久化不一致 | 中 | 高 | 写 Raft log 再更新内存 |
| 数据副本不一致导致读取错误 | 中 | 高 | 校验和 + 定期 scrub |

## 待验证假设

H1: 百万级 chunk 元数据内存命中率 > 99%
H2: 分区 Raft 在 4~16 节点下 QPS > 10K/s
H3: 异步复制后台流量不干扰前台 I/O

## 替代方案排除理由

- **方案 A**：写延迟增加 5~10ms，KV Cache 只读特性使强一致收益不足
- **方案 B**：元数据（chunk 位置、引用计数）错误后果严重，必须强一致
- **方案 D**：无成熟开源参考，实现复杂度极高

# ADR-005: 数据冗余策略决策

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（76%）→ **修订为 72%（经济性边界待量化）**

## 背景

KV Cache 数据是"可重算但代价高"的数据（7B 模型单次 Prefill 约 5s GPU 时间）。冗余是必须的，但需在可靠性和效率间权衡。

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| VAST Data LDC | 局部可修复码（LRC）| 146+4 配置，小集群不可用 |
| Ceph | 多副本 + 纠删码 | 3 副本 + k+m 纠删码成熟 |
| MoonCake | 尽力而为复制 | KVCache 可重算，容忍丢失 |

## 候选方案

| 方案 | 参考来源 | 置信度 |
|------|----------|--------|
| A: 3 副本 | Ceph | 83% |
| B: 纠删码（k+m）| Reed-Solomon | 70% |
| C: 混合（热 3 副 + 冷纠删）| Ceph + VAST | **72%** |
| D: LRC 局部可修复码 | VAST LDC | 63% |

## 决策

**分阶段混合策略**：
- MVP（Phase 1）：**3 副本**起步（⚠️ X1 已解决：与 ADR-004 统一）
- 演进（Phase 2+）：热数据 3 副本 + 冷数据纠删码（触发条件：节点数 ≥ 6 **且**存储利用率 > 60%，两者 AND）

## 关键 Trade-off

T1: 存储效率 vs 可靠性 → 3 副本存储开销 3x，但可靠性最高
T2: 重建时间 vs 存储效率 → 纠删码存储效率高但重建慢

## 待验证假设

H1: 存储成本比可计算（副本 vs 纠删码经济性）
H2: 重建限速阈值可配置
H3: 冷热数据分布可观测

## 替代方案排除理由

- **方案 D（LRC）**：4~16 节点下 LRC 配置（12+2+2）不可行，最小需 12 节点
- **方案 B（纯纠删码）**：4~5 节点下无法配置（最小 k+m=4+2=6）

## Reviewer 质疑与回应

### Reviewer 质疑摘要

**P0 强制修改项**：

1. **ADR-004 与 ADR-005 副本数矛盾（致命）**：ADR-004 选择"2 副本"，ADR-005 选择"3 副本起步"，直接冲突。
   - **回应 Reviewer-01: ✅ X1 已解决。** 决策：**Phase 1 数据层统一采用 3 副本**。理由：3 副本满足 Raft majority（3 节点 Raft 组），本地确认即返回（异步复制），与 ADR-004 的分层一致模型兼容。数据 durability 由本地 NVMe 持久化保证（写入 NVMe 后即可确认）。

2. **补充"可重算 vs 3 副本"经济性分析**：3x 存储成本 vs 5s GPU 重计算成本，在 100TB 规模下的绝对经济性对比是什么？
   - **回应 Reviewer-02: 接受。** 经济性分析：
     - **3 副本存储成本**：假设 NVMe SSD $0.1/GB，100TB × $0.1/GB × 3 副本 = $30,000（一次性）
     - **5s GPU 重计算成本**：假设 A100 GPU $3/小时，7B 模型 Prefill 每次 5s → $0.0042/次。若每天 10,000 次缓存未命中 → $42/天 = $15,330/年
     - **结论**：在高频缓存未命中场景下（>400 次/天），3 副本存储成本低于重计算成本。**H1 置信度修订为 75%**，经济性分析待原型验证后修正。

3. **补充 LRC (4+2+1) 可行性分析**：(4+2+1) LRC 只需 7 节点，存储开销 ~83%，重建只需局部组内 ~3 块。
   - **回应 Reviewer-03: 接受。** LRC (4+2+1) 可行性补充：
     - **存储效率**：LRC (4+2+1) 存储开销 ~71%（4+2+1=7 块数据，存储 7+1+1=9 块），比 3 副本（33% 效率）高
     - **最小节点需求**：LRC (4+2+1) 最小需要 7 节点，4~6 节点集群不可用
     - **决策**：LRC 作为 Phase 2 演进候选方案，**Phase 1 维持 3 副本**，Phase 2 在节点数 ≥ 7 且存储利用率 > 60% 时评估 LRC vs 纠删码

4. **补充触发条件推导**："节点数 ≥ 6 且存储利用率 > 60%"的阈值从何而来？60% 是 AND 还是 OR 关系？
   - **回应 Reviewer-04: 接受。** 触发条件明确为 **AND（两者同时满足）**：
     - **节点数 ≥ 6**：纠删码最小配置 k+m=4+2=6，触发前需满足最小节点数
     - **存储利用率 > 60%**：利用率超过 60% 时，存储效率问题变得突出，此时引入纠删码可节省存储成本
     - 该阈值基于 Ceph 生产经验（利用率 > 70% 时纠删码收益显著），保守取 60%

**关键质疑点**：

- **"可重算"论点自毁冗余需求**：如果可重算且代价低，为何还要 3 副本？
  - **回应 Reviewer-K1: 讨论。** "可重算"论点有边界条件：只有在缓存未命中时才有重计算成本。如果命中率 > 90%，每天实际重计算成本极低。因此 3 副本的真正价值不是"防止丢失"，而是**保证高命中率场景下的 P99 延迟稳定**（副本保证读取路径有数据）。该洞察已补充到 H1 分析中。

- **冷/热数据边界未定义**：谁决策数据何时从"热"变"冷"？
  - **回应 Reviewer-K2: 接受。** 冷/热数据边界由存储系统**自动判断**：DRAM 层 LRU 淘汰时，被淘汰的数据自动降级为 SSD 层（热→冷），由分层策略（ADR-008）驱动。手动 pin 接口可阻止降级（pin 的数据不参与 LRU 淘汰）。

- **H4（副本数与 Raft majority）未声明**：Phase 1 数据层"写入本地确认即返回"——是否意味着没有等 majority？
  - **回应 Reviewer-K3: 接受。** 数据层写入**不等 Raft majority**（本地 NVMe 持久化即可确认），Raft majority 仅用于**元数据**（Block Table 位置、COMMITTED 状态）确认。数据 durability 由本地 NVMe 写入保证，与 Raft majority 独立。

# ADR-006: 执行引擎/调度模型决策

## 背景

核心挑战：同时处理 NVMe SSD I/O（SPDK）和 RDMA 网络 I/O（libibverbs），需统一调度、NUMA 亲和、无锁热路径、C 语言约束。

- **执行引擎层（ADR-006）**：RDMA CQ 使用忙轮询（~3μs 延迟），保证最低延迟
- **传输层（ADR-007）**：封装执行引擎能力，对外暴露**统一事件接口**（epoll fd）

| libibverbs 与 SPDK reactor 兼容性 | 中 | 高 | Phase 1 原型验证 |
| 回调模型代码难以维护 | 中 | 中 | 状态机框架封装 |

H2: spdk_poller_register 可安全注册 RDMA comp_channel fd

# ADR-007: KV Cache Offloading 传输层架构

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（75%）→ **修订为 70%（WR 批处理和降级路径待原型验证）**

## 背景

核心问题：RDMA 与 TCP 协作模式、双边 vs 单边 RDMA、控制平面与数据平面分离、事件驱动 vs 忙轮询 CQ、WR 批处理参数。

部署环境：InfiniBand/RoCE v2 为主，TCP 管理/兼容回退。P99 < 10ms，单次传输 32MB~100MB+。

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| InfiniStore | RDMA 双边 + TCP 控制面分离 + WR 批处理 | 直接借鉴：双边 RDMA 服务端主导 |
| VAST Data | NVMe-oF over RDMA/TCP | RDMA 优先原则借鉴 |
| MoonCake | Transfer Engine 多协议抽象 | 多协议后端扩展思想借鉴 |

## 候选方案

| 方案 | 参考来源 | 置信度 |
|------|----------|--------|
| A: RDMA 独占 + TCP 仅管理面 | InfiniStore | 65% |
| B: RDMA+TCP 双栈自动切换 | MoonCake | 70% |
| C: RDMA 为主 + TCP 应急 | InfiniStore TCP 回退 | **75%** |

## 决策

选择 **方案 C（RDMA 为主 + TCP 应急）**（⚠️ X2 已对齐 ADR-006）：

- RDMA 双边操作（服务端主导）
- TCP 控制平面 + RDMA 数据平面
- 静态绑定 + 单向故障降级
- WR 批处理 MAX_WR_BATCH=32（⚠️ 敏感性待分析）
- **对外暴露 epoll fd 统一事件接口**（封装执行引擎的忙轮询 RDMA CQ）
- **降级触发：请求级超时**（⚠️ 修正为请求级，非 4.3s 全局超时）

架构对齐说明（X2 已解决）：传输层对内调用执行引擎的忙轮询 CQ，对外暴露 epoll fd，传输层是执行引擎的用户，不存在 CQ 模式冲突。

## 关键 Trade-off

- T1: 极致性能 vs 故障弹性 → 单向降级，平衡常态性能和故障弹性
- T2: 双边 vs 单边 RDMA → 双边（服务端控制内存生命周期）
- T3: 事件驱动 vs 忙轮询 → 传输层对外统一事件接口（epoll fd），内部根据 Phase 调用执行引擎忙轮询或事件驱动

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| RDMA 网卡/驱动兼容性 | 中 | 高 | 提供兼容性列表；启动时 RDMA selftest |
| TCP 应急路径未充分测试 | 中 | 高 | CI 每晚跑 TCP-only 测试 |
| RoCE 网络配置复杂 | 高 | 中 | 网络配置检查脚本 + 文档明确要求 |

## 待验证假设

H1: 事件驱动 CQ 在 1000+ 并发连接下延迟增加 < 10μs
H2: WR 批处理 32/批可饱和 100Gbps RDMA 带宽
H3: TCP 应急路径可被 CI 充分测试

## 替代方案排除理由

- **方案 A**：违反"故障常态假设"，无降级路径
- **方案 B**：过度工程化，双路径 + 链路健康监测 + 路径选择状态，团队首次接触 RDMA 时风险高

## Reviewer 质疑与回应

### Reviewer 质疑摘要

**P0 强制修改项**：

1. **ADR-006 与 ADR-007 架构对齐（立即，X2 P0）**：两者无法同时成立。
   - **回应 Reviewer-01: ✅ X2 已解决。** 统一方案：传输层对外暴露 epoll fd，对内调用执行引擎的忙轮询 CQ。执行引擎负责 RDMA CQ 处理，传输层负责协议逻辑，两者协同无冲突。

2. **补充 WR 批处理敏感性分析**：MAX_WR_BATCH=32 来自 InfiniStore，KV Cache 场景不同。
   - **回应 Reviewer-02: 接受。** 补充敏感性分析：
     - KV Cache 块大小：32MB~100MB+
     - 100Gbps RDMA：32MB 传输时间 ≈ 2.5ms；64MB ≈ 5ms；MAX_WR_BATCH=32 时，batch 内所有 WR 共享握手开销，batch 越大效率越高
     - **敏感区间**：batch 8~64，低于 8 则 WR 握手开销占比高；高于 64 则并行度过高可能导致 CQ 拥塞
     - **决策**：Phase 1 使用 MAX_WR_BATCH=32 作为默认值，可通过配置动态调整

3. **补充降级触发阈值 4.3s 设计依据**：推理请求超时时不可接受，应改为请求级超时。
   - **回应 Reviewer-03: 接受。** 降级触发修正为**请求级超时**：
     - **触发条件**：单个 KV Cache 请求在 RDMA 路径上超过 **100ms**（P99 预算的 10%）
     - 阈值推导：10ms P99 目标 → 正常路径 P99 < 5ms → 降级阈值设为 100ms（10x 余量），避免误触发
     - 降级后：SLO 放宽至 100ms（降级期间降级路径 SLO = 100ms，而非 10ms）

4. **正视 InfiniStore TCP bug**：降级路径依赖有 bug 的参考实现。
   - **回应 Reviewer-04: 接受。** 缓解策略：① Phase 1 自行实现 TCP 降级路径，不依赖 InfiniStore 参考实现；② TCP 降级是 Phase 2 特性，Phase 1 降级到错误日志而非 TCP；③ 降级路径有独立的集成测试覆盖。

**关键质疑点**：

- **"故障常态假设"与 MoonCake 自相矛盾**：MoonCake 也是 RDMA 为主无 TCP 降级。
  - **回应 Reviewer-K1: 讨论。** MoonCake 的部署环境（Kimi 内部）与本系统不同，本系统面向多租户和多用户环境，网络配置不一致性更高，因此保留 TCP 降级路径。MoonCake 的选择是"默认 RDMA"，本系统的选择是"默认 RDMA + 降级到错误"（Phase 1），逐步演进。

- **静默降级未被正视**：RoCE 配置错误导致 RDMA 降质不自知。
  - **回应 Reviewer-K2: 接受。** 补充：RDMA 静默降质检测通过启动时 RDMA selftest + 定期 ping 验证实现。检测到静默降质时触发告警，强制降级。

- **TCP 降级后 P99 必然超标**：降级后 SLO 必然失败。
  - **回应 Reviewer-K3: 接受。** 已在上方决策中修正：降级期间 SLO 放宽至 100ms（降级路径），正常路径维持 10ms。降级触发时通知运维系统。

- **部分降级场景未处理**：静态绑定设计无法处理部分降级。
  - **回应 Reviewer-K4: 接受。** 补充：检测到部分节点静默降质时，触发全局告警 + 路由层将该节点标记为不可用，新请求不再路由到该节点。存量请求在超时时自动重试到其他节点。

# ADR-008: KV Cache 数据分层策略

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（72%）→ **修订为 65%（H1/H2/H3 均低于整体值，整体置信度取下界）**

## 背景

核心问题：分层粒度、迁移触发、DRAM 角色、HBM 管理归属。

约束：4~16 节点，100TB~10PB，C 语言，PD 分离架构兼容。

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| LMCache | L1 GPU→L2 CPU→L3 Disk 三层 | Write-All 策略简化一致性 |
| MoonCake | DRAM 为主 + SSD 为辅 | +75~115% 容量验证 |
| InfiniStore | 纯 DRAM | 不适合持久化需求 |

## 候选方案

| 方案 | 参考来源 | 置信度 |
|------|----------|--------|
| A: Chunk 级自动分层 | LMCache 分层管理器 | 低（45%） |
| B: 节点级粗粒度分层 | Ceph CRUSH 设备类 | 中（65%） |
| C: 混合分层（本地自治 + 热点提升）| LMCache + InfiniStore | **高（75%）** |

## 决策

选择 **方案 C（混合分层）**（⚠️ X4 已解决：补充 Connector API 设计草案）：

| 层级 | 介质 | 管理方 | 职责 |
|------|------|--------|------|
| L0 | GPU HBM | 推理引擎 | 当前 batch 活跃 KV Cache |
| L1 | 节点 DRAM | 存储系统 | 温数据、高频复用 KV |
| L2 | NVMe SSD | 存储系统 | 冷数据、历史积累 |

- HBM←→DRAM：通过 **Connector API** 由推理引擎控制（⚠️ 设计草案见下方）
- DRAM←→SSD：本地自治，LRU 淘汰
- 全局数据分布：一致性哈希，局部性优先

### Connector API 设计草案（X4 已解决）

> `docs/architecture/connector/design.md` 已删除，此处补充最小化设计草案：

**核心接口**（Phase 1 MVP）：
```
connector_hint_valloc(ctx, prompt_prefix, len, target_layer);  // 建议 KV Cache 在指定层级分配
connector_hint_pin(ctx, key, key_len, ttl_sec);              // 建议保持 pin 状态
connector_notify_evict(ctx, key, key_len);                      // 通知推理引擎已被淘汰
```

**H3 置信度修订**：Connector API 是 Phase 2 特性。Phase 1 MVP 由存储系统自主管理全部分层，Connector API 在 Phase 2 再与 vLLM/SGLang 集成验证。

### DRAM 满边界条件（Admission Control）

DRAM 满时：**拒绝新写入（返回错误）**，推理引擎收到错误后降级到 NVMe SSD 路径。

**理由**：存储系统不能单方面降级 SSD（推理引擎可能已为 KV Cache 预留了 HBM→DRAM 的传输路径）。正确的处理是通知推理引擎（via Connector API），由推理引擎决策降级路径。

### pin/unpin 接口语义

- **pin**：推理引擎通过 `connector_hint_pin` 请求保持数据在 DRAM 层，不参与 LRU 淘汰
- **谁有权 pin**：仅推理引擎（Connector 客户端）
- **释放机制**：TTL 超时自动 unpin，或推理引擎显式调用 unpin
- **并发安全**：pin/unpin 操作通过 Raft 协调（ADR-004 元数据层）

## 关键 Trade-off

- T1: 精细优化 vs 复杂度 → 本地自治为主，跨节点 chunk 级迁移收益不覆盖复杂度
- T2: 缓存模式 vs 存储模式 → **存储模式**（KV Cache 重计算成本远高于存储成本）
- T3: HBM 管理归属 → **Phase 1 存储系统自主管理；Phase 2 集成 Connector API**

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| DRAM 容量不足，命中率过低 | 中 | 高 | 容量规划工具；DRAM:SSD 比例可配置 |
| LRU 不适应长周期 KV Cache | 中 | 中 | 支持 pin/unpin 接口 |
| 异步迁移影响前台延迟 | 低 | 中 | 限速 + 低优先级 IO |

## 待验证假设

H1: 1TB DRAM/节点覆盖 80% 访问热数据（置信度 65%）
H2: 本地 LRU + 一致性哈希在 4~16 节点下命中率 > 70%（置信度 60%）
H3: Connector API hints 机制有效指导放置（置信度 55%）

## 替代方案排除理由

- **方案 A**：元数据开销在百万 chunk 规模下不可控；抖动风险不可接受
- **方案 B**：粒度过粗；与 PD 分离架构不兼容

## Reviewer 质疑与回应

### Reviewer 质疑摘要

**P0 强制修改项**：

1. **补充 Connector API 设计草案（X4 P0）**：`docs/architecture/connector/design.md` 已被删除，Connector API 设计不存在。
   - **回应 Reviewer-01: ✅ X4 已解决。** 已在 ADR 决策章节补充 Connector API 最小化设计草案，包含 3 个核心接口（H3 置信度修订为 Phase 2 验证）。

# ADR-009: 元数据存储引擎选型

- **状态**: **终稿** ✅
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（72%）→ **修订为 62%（重建算法和循环依赖待原型验证）**

## 背景

核心挑战：
- 极端读多写少（50:1~500:1）
- 范围查询需求（按 prefix hash）
- 引用计数更新（写放大敏感）
- 规模：单节点 1000 万条目约 2-5 GB，16 节点约 32 GB

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|---------|
| LMCache | 内存前缀哈希链索引 | 验证元数据完全驻留内存的可行性 |
| etcd | BoltDB (B+tree) | 验证 B+tree 在元数据场景的稳定性 |
| TiKV | RocksDB | 写密集型 KV，非读多写少优化 |

## 候选方案

| 方案 | 参考来源 | P99 读延迟 | 置信度 |
|------|----------|-----------|--------|
| A: RocksDB | Facebook/TiKV | 50-200 us | 低（45%） |
| B: LMDB | Symas/OpenLDAP | 10-50 us | 中（70%） |
| C: 内存 Hash + WAL | Redis/LMCache/uthash | 1-5 us | **中（72%）** |
| D: 自研 B+tree | SQLite/LMDB | 10-50 us | 低（35%） |

## 决策

选择 **方案 C（内存哈希表 + 异步 WAL）首选，方案 B（LMDB）备选**。

核心理由：
1. 读性能最优（1-5 us），比 RocksDB 快 10-100x
2. 引用计数零开销（内存原地修改）
3. 实现极简（uthash ~2000 行，纯 C）
4. LMCache 生产验证

### WAL 重建算法（P0-1/P0-2 已解决）

> WAL 丢失后，如何重建 chunk 位置和引用计数？

**重建过程**：
1. **扫描 NVMe SSD**：遍历所有 KV Cache 块文件，提取每个块的元数据头（chunk_id、version、引用计数）
2. **构建临时哈希表**：将扫描结果存入临时哈希表（chunk_id → {location, refcount}）
3. **恢复内存状态**：用临时哈希表重建内存哈希索引
4. **重建期间策略**：系统以只读模式运行（KV Cache 写入暂停），等待重建完成
5. **重建时间估算**：SSD 顺序扫描速度约 3GB/s，100TB 数据扫描约 9.3 小时（⚠️ 该估算基于顺序扫描，SSD 实际 IOPS 可能限制吞吐量）

**与 ADR-004 Raft 的关系（X3 已解决）**：
- **本地 WAL**：记录本地内存哈希表的变更（用于本地节点重启后恢复）
- **Raft 日志**：记录跨节点协调操作（如副本同步、COMMITTED 状态更新）
- **两者关系**：本地 WAL 是 Raft 日志的必要补充——Raft 只复制协调操作，本地 WAL 记录所有本地变更。重启时，先从 Raft 日志恢复协调状态，再从本地 WAL 恢复本地状态。

### 跳表实现明确（P0-3）

跳表是**自研实现**（~500 行 C 代码），不依赖外部库。理由：Redis 和 leveldb 的跳表实现可参考，但本系统场景简单（仅需按 score 排序的 kv 对），自研工程量可控。

## 关键 Trade-off

- T1: 读性能 vs 持久化 → 接受异步 WAL 故障窗口（chunk 位置可重建）
- T2: LSM-Tree 适配性 → 读多写少场景下 LSM-Tree 读放大固有缺陷无法消除
- T3: 开发成本 vs 成熟度 → uthash + WAL 约 3000-5000 行，远低于自研 B+tree

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 元数据规模超过内存容量 | 中 | 高 | 精确建模；超阈值切换 LMDB；支持分片 |
| 异步 WAL 故障窗口丢失 | 中 | 高 | 配置 fsync 策略；关键操作强制同步 |
| 无内置范围查询 | 低 | 中 | 额外维护跳表二级索引 |

## 待验证假设

H1: 单节点元数据峰值 < 5GB
H2: 异步 WAL 丢失窗口可接受
H3: 内存跳表二级索引满足 Prefix Hash 范围查询性能

## 替代方案排除理由

- **方案 A（RocksDB）**：LSM-Tree 读放大（3-10x）与读多写少场景根本错配；TiKV 类比不成立
- **方案 D（自研 B+tree）**：开发成本 5000-10000 行，风险集中，违背不重复造轮子原则
