## 问题定义

KV Cache 存储系统的一致性模型需要回答的核心问题：

1. **写入可见性**：KV Cache 写入后，多快对其他节点/请求可见？
2. **PD 分离一致性**：Prefill 节点写入的 KV Cache，Decode 节点读取时是否保证看到完整数据？
3. **副本一致性**：多副本场景下，写入是否需要等所有副本确认？
4. **故障一致性**：节点故障恢复后，数据是否与故障前一致？

**场景约束**：
- 推理引擎要求：KV Cache 写入后需**立即可读**（否则 TTFT 会增加）
- PD 分离要求：Prefill 完成后的 KV Cache 必须完整传输到 Decode 节点
- 数据特征：KV Cache 写入后**只读不改**（除非删除/淘汰），无并发写冲突

## 标杆一致性方案提炼

| 标杆 | 一致性模型 | 关键特征 | 与本系统的关联 |
|------|-----------|---------|--------------|
| **MoonCake** | **最终一致**（异步传输）| KV Cache 异步写入全局存储池；短暂不一致窗口可容忍；推理场景可接受 | 验证了最终一致在推理场景的可行性 |
| **VAST Data** | **强一致**（SCM 原子操作）| 无缓存，所有元数据持久化在 SCM；本地事务提交；无需分布式共识 | 强一致的极致实现，但依赖 SCM 硬件 |
| **LMCache** | **最终一致**（分层异步）| L1→L2→L3 异步复制；各层独立淘汰；不保证全局一致 | 分层存储天然最终一致 |
| **InfiniStore** | **强一致**（单节点内存）| 单节点内存存储，无分布式一致性问题 | 单节点简化了一致性，但本系统需分布式 |
| **Ceph** | **强一致**（CRUSH + Paxos）| 写入等主副本确认；元数据通过 Paxos 强一致 | 通用存储的强一致参考 |

**关键洞察**：
- KV Cache 数据具有**写入后只读**的特性，天然避免了最复杂的一致性场景（并发写冲突）
- 推理场景对"短暂不一致"的容忍度较高：如果 Prefix Cache 命中时数据还未全局同步，最坏情况是重新计算（性能降级，非正确性错误）
- 但 PD 分离场景要求** Prefill→Decode 的传输必须是强一致的**（Decode 不能读到不完整的 KV Cache）

## 候选方案

### 方案 A：强一致（写入等副本确认）

**核心思路**：所有 KV Cache 写入必须等到指定数量的副本（如 majority）确认后才返回成功。读取时从主副本读取，保证读到最新数据。

**参考来源**：Ceph CRUSH + Paxos / VAST Data SCM 原子事务

**优点**：
1. **正确性最强**：写入后立即可读，无不一致窗口
2. **PD 分离天然支持**：Prefill 写入确认后，Decode 读取一定完整
3. **推理引擎集成简单**：Connector 无需处理"数据未就绪"的复杂逻辑
4. **故障恢复快**：任意副本可用于恢复，数据不丢

**缺点**：
1. **写延迟高**：需等待跨网络副本确认，写延迟 = 网络 RTT × 2（假设 majority=2/3）
2. **写吞吐下降**：同步复制限制了写入吞吐
3. **实现复杂**：需要分布式共识协议（Raft/Paxos）或主从复制管理
4. **可用性降低**： minority 节点故障时写入阻塞

**适用场景**：对数据正确性要求极高、可接受写延迟增加的场景。

---

### 方案 B：最终一致（异步复制）

**核心思路**：写入本地确认后立即返回，后台异步复制到其他节点/副本。读取时可能读到旧版本，但通过版本号检测可发现不一致。

**参考来源**：MoonCake 异步传输 + LMCache 分层异步 + Dynamo/Cassandra

**优点**：
1. **写延迟最低**：本地确认即可返回，无需等待网络复制
2. **写吞吐最高**：不受复制延迟限制
3. **可用性最高**：节点故障不影响写入（只要本地节点存活）
4. **实现简单**：无需分布式共识协议

**缺点**：
1. **读取可能不一致**：读取时可能读到旧版本或未同步的数据
2. **PD 分离需额外保证**：Prefill→Decode 传输不能依赖异步复制，需要点对点确认
3. **冲突解决复杂**：虽然 KV Cache 只读，但元数据（如引用计数）更新可能有冲突
4. **故障恢复复杂**：需要处理"某些副本有数据，某些没有"的部分故障场景

**适用场景**：对写延迟敏感、可容忍短暂不一致、有成熟冲突处理机制的场景。

---

### 方案 C：分层一致（元数据强一致 + 数据最终一致）

**核心思路**：区分元数据和数据的一致性要求。元数据（chunk 位置、引用计数、生命周期状态）使用强一致（Raft/Paxos），数据（KV Cache 张量本身）使用最终一致（异步复制）。

**参考来源**：Ceph 的元数据强一致 + 数据异步复制 + Bigtable/HBase 设计思想

**优点**：
1. **延迟与正确性平衡**：元数据强一致保证系统状态正确，数据异步复制保证低延迟
2. **PD 分离可支持**：Prefill 写完数据后更新元数据（强一致），Decode 通过元数据定位数据
3. **故障恢复可控**：元数据强一致保证不会"丢失数据的位置信息"
4. **扩展性良好**：数据复制与元数据管理解耦，可独立扩展

**缺点**：
1. **实现复杂度最高**：需要同时维护两套一致性机制
2. **元数据可能成为瓶颈**：强一致元数据服务的 QPS 有限
3. **边界情况复杂**：元数据已更新但数据未复制完成时，如何处理读取请求？

**适用场景**：大规模系统、元数据量相对小但关键、数据量大且可容忍异步复制的场景。

---

### 方案 D：因果一致 + 显式屏障（KV Cache 专用优化）

**核心思路**：基于因果一致性模型，利用 KV Cache 的"写入后只读"特性做专用优化。引入显式屏障（barrier）机制：PD 分离传输时在 Prefill→Decode 路径插入屏障，确保 Decode 看到完整数据。

**参考来源**：COPS/Eiger 因果一致性系统 + MoonCake 的传输确认机制

**优点**：
1. **专为 KV Cache 设计**：利用"无并发写"特性简化因果一致性实现
2. **PD 分离语义原生支持**：Prefill→Decode 的 barrier 机制保证传输完整性
3. **延迟接近最终一致**：正常写入无需等待，仅在 PD 传输时需要 barrier
4. **Prefix Cache 友好**：前缀匹配的读取不需要强一致，因果一致足够

**缺点**：
1. **理论复杂度高**：因果一致性需要维护依赖图（happens-before 关系）
2. **实现无成熟参考**：需要大量自研代码，无开源参考实现
3. **验证困难**：因果一致性的正确性验证比强/最终一致更复杂

**适用场景**：愿意投入大量研发资源、追求理论与工程最优平衡的场景。

## 关键 Trade-off 分析

### T1：写延迟 vs 正确性
- **方案 A** 选择正确性，接受写延迟增加
- **方案 B** 选择低延迟，接受短暂不一致
- **本系统选择**：偏向方案 C（分层一致），原因：
  - KV Cache 的"写入后只读"特性使得数据层面的最终一致风险很低
  - 元数据（位置、引用计数）的错误会导致严重后果（数据丢失、泄漏），必须强一致
  - 用户要求"写入后立即可读"可以通过元数据强一致 + 本地数据读取来实现

### T2：实现复杂度 vs 专用优化
- **方案 D** 提供最优的 KV Cache 专用一致性，但实现复杂度极高
- **方案 C** 是成熟的工程折中，有大量参考实现
- **本系统选择**：方案 C，原因：
  - 4~16 节点规模下，方案 C 的元数据服务不会成为瓶颈
  - 方案 D 的自研成本过高，风险不可控
  - 可在方案 C 基础上，对 PD 分离路径做专用优化（类似 barrier）

### T3：元数据强一致的实现方式
- **Raft  per 分区**：每个数据分区一个 Raft 组，扩展性好但资源开销大
- **集中式 Raft**：全局一个 Raft 组，简单但扩展性受限
- **本系统选择**：分区 Raft（类似 TiKV / Ceph OSD Map），4~16 节点下每个分区约 3~5 个副本

## 初步倾向

**倾向方案 C（分层一致）的变体**：

- **元数据**：强一致，基于分区 Raft（每分区 3 副本，写入等 majority 确认）
  - 元数据包括：chunk 位置、引用计数、生命周期状态、模型版本映射
  - 元数据量估计：百万级 chunk × 每条 100B ≈ 100MB，可完全缓存在内存中
- **数据（KV Cache 张量）**：最终一致，异步复制到 2 个副本（共 3 份）
  - 写入本地确认即可返回，后台异步复制
  - 读取从本地副本读取（通过元数据定位）
- **PD 分离路径**：Prefill→存储节点写入后，元数据更新为"可用"状态（强一致），Decode 读取时通过元数据确认数据可用性
- **故障恢复**：元数据 Raft 保证状态一致，数据副本通过后台校验和修复

**关键设计决策**：
1. 元数据与数据解耦，独立一致性策略
2. 元数据使用内存缓存 + Raft 持久化，保证低延迟
3. PD 分离的"可用性"通过元数据状态机管理（WRITING → COMMITTED → AVAILABLE）

## 待验证假设

| # | 假设 | 验证方法 | 不成立的后果 |
|---|------|---------|------------|
| H1 | 百万级 chunk 的元数据在内存中缓存的命中率 > 99% | 原型测试（模拟元数据访问模式）| 元数据读取延迟增加，需优化缓存策略 |
| H2 | 分区 Raft 在 4~16 节点下不会成为瓶颈（QPS > 10K/s）| Raft 原型压测 | 需减少分区数或改用集中式 |
| H3 | 数据异步复制的后台流量不会干扰前台 I/O | 混合负载压测 | 需限速后台复制或调整优先级 |

## 风险矩阵

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 元数据 Raft 实现 bug 导致脑裂 | 低 | 极高 | 使用成熟 Raft 库（如 etcd/raft）；充分测试分区场景 |
| 元数据内存缓存与持久化不一致 | 中 | 高 | 写操作先写 Raft log，再更新内存缓存；重启时从 log 重建 |
| 数据副本不一致导致读取错误 | 中 | 高 | 每个 chunk 带校验和；定期 scrub；读取时可选校验 |
| 分区 Raft 组数过多导致资源开销 | 中 | 中 | 合理设置分区数（如每节点 1~4 个分区）；监控内存/CPU |
