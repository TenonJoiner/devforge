## 问题定义

KV Cache 存储系统的数据分层策略需要解决的核心问题：

1. **分层粒度**：数据在 HBM→DRAM→SSD 之间迁移的粒度是什么？（chunk 级、节点级、还是全局统一？）
2. **迁移触发条件**：什么条件下触发数据迁移？（访问频率、年龄、容量压力、还是显式指令？）
3. **迁移控制方式**：自动还是手动？如果是自动，策略如何制定？
4. **分层对延迟的影响**：跨层访问的延迟增加是否可接受？

**场景约束**：
- 三层介质：HBM（GPU 显存，~80GB/卡，延迟 ~10ns）→ DRAM（节点内存，~1TB/节点，延迟 ~100ns）→ NVMe SSD（~10TB/节点，延迟 ~10μs）
- 数据特征：KV Cache 块大小 ~32MB（256 tokens × Llama-3.1-8B），写入后高频读取一段时间，然后逐渐变冷
- 成本差异：HBM 成本是 DRAM 的 10x+，DRAM 成本是 SSD 的 5x+

## 标杆分层方案提炼

| 标杆 | 分层策略 | 关键特征 | 与本系统的关联 |
|------|---------|---------|--------------|
| **MoonCake** | **两层：HBM → CPU DRAM** | CPU DRAM 为主存储；SSD 为辅（未详述）；异步传输隐藏延迟 | 验证了 HBM→DRAM 卸载的可行性 |
| **LMCache** | **三层：L1 GPU → L2 CPU → L3 Remote/Disk** | 分层存储管理器；异步流水线；内容寻址索引；各层独立淘汰 | 分层管理器的抽象设计可直接借鉴 |
| **InfiniStore** | **单层：纯 CPU DRAM** | 无持久化层；纯内存存储；LRU 淘汰 | 本系统需要持久化，不能仅内存 |
| **vLLM PagedAttention** | **单层：GPU HBM 内部管理** | 块级分配器；LRU 淘汰；无跨层迁移 | GPU 内部缓存管理参考 |

## 候选方案

### 方案 A：Chunk 级自动分层（细粒度自动迁移）

**核心思路**：以 chunk（~32MB）为粒度，根据访问频率和年龄自动在 HBM→DRAM→SSD 之间迁移。每个 chunk 独立管理其所在层级。

**参考来源**：LMCache 分层存储管理器 + Linux page cache 分层思想

**架构草图**：

```
┌─────────────────────────────────────────┐
│            全局分层管理器                 │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐  │
│   │  HBM    │ │  DRAM   │ │  SSD    │  │
│   │ (热)    │ │ (温)    │ │ (冷)    │  │
│   └────┬────┘ └────┬────┘ └────┬────┘  │
│        │           │           │       │
│        └───────────┼───────────┘       │
│                    │                   │
│              迁移决策引擎               │
│         (访问频率 + 年龄 + 容量压力)      │
└─────────────────────────────────────────┘
```

**优点**：
1. **粒度最细**：每个 chunk 独立管理，最大化存储效率
2. **自动化**：无需人工干预，系统根据策略自动优化
3. **局部性最优**：热数据保持在最快介质，冷数据自动降级
4. **延迟隐藏**：异步迁移，前台 I/O 不受影响

**缺点**：
1. **实现最复杂**：需要维护每个 chunk 的状态、位置、迁移历史
2. **元数据开销大**：百万级 chunk × 分层状态 = 大量元数据
3. **抖动风险**：频繁的分层迁移（thrashing）导致性能波动
4. **策略调参复杂**：阈值设置不当可能导致过早迁移或过度滞留

**适用场景**：数据冷热分布不均匀、追求极致存储效率的场景。

---

### 方案 B：节点级分层（粗粒度本地自治）

**核心思路**：每个存储节点本地管理三层介质，但不对单个 chunk 做跨层迁移。而是将节点角色区分为：GPU 节点（含 HBM）、内存节点（纯 DRAM）、SSD 节点（纯 SSD）。数据通过一致性哈希分布在不同角色的节点上。

**参考来源**：Ceph 的 CRUSH 设备类 + Alluxio 的分层存储

**架构草图**：

```
┌─────────────────────────────────────────┐
│           GPU 节点（HBM + DRAM）          │
│   最热数据，Prefill/Decode 直接访问        │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│         内存节点（纯 DRAM）               │
│        温数据，高频复用的 KV Cache        │
└─────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────┐
│         SSD 节点（纯 NVMe）               │
│        冷数据，历史积累的 KV Cache        │
└─────────────────────────────────────────┘
```

**优点**：
1. **实现简单**：无需跨层迁移逻辑，节点角色固定
2. **无抖动风险**：数据位置稳定，不会频繁迁移
3. **故障域清晰**：某类节点故障只影响该层数据
4. **资源规划简单**：每种节点配置固定，易于容量规划

**缺点**：
1. **粒度太粗**：冷热数据混合在同一节点，无法精细优化
2. **资源利用率低**：GPU 节点的 DRAM 可能被冷数据占用，内存节点的 DRAM 可能不足
3. **扩容不灵活**：增加某层容量必须增加该类型节点
4. **成本可能更高**：需要专门的内存节点和 SSD 节点

**适用场景**：硬件资源类型分明、冷热分布相对均匀、追求简单运维的场景。

---

### 方案 C：混合分层（节点本地自治 + 热点提升）

**核心思路**：每个存储节点本地管理 DRAM→SSD 分层（节点自治），全局层面支持热点 chunk 的跨节点 DRAM 缓存（类似 CDN 的回源缓存）。HBM 层由推理引擎自主管理（非存储系统职责）。

**参考来源**：LMCache 分层 + InfiniStore 内存池 + 分布式缓存系统（如 Memcached/Redis Cluster）

**架构草图**：

```
┌─────────────────────────────────────────┐
│    HBM 层（推理引擎自主管理，非存储职责）   │
│        vLLM PagedAttention 块分配器       │
└─────────────────────────────────────────┘
                    │ Connector API
┌─────────────────────────────────────────┐
│    存储节点集群（每节点 DRAM + SSD）       │
│                                         │
│   ┌─────────────┐    ┌─────────────┐   │
│   │   DRAM      │◀──▶│   NVMe SSD  │   │
│   │ (本地热数据) │    │ (本地冷数据) │   │
│   └─────────────┘    └─────────────┘   │
│        │                                │
│   本地 LRU/LFU 淘汰                       │
│        │                                │
│   热点 chunk 可全局多副本缓存（可选）        │
└─────────────────────────────────────────┘
```

**优点**：
1. **职责清晰**：HBM 管理交给推理引擎（vLLM 已做得很好），存储系统专注 DRAM→SSD
2. **局部性优先**：大部分访问在单节点内完成（DRAM hit），无需跨网络
3. **可渐进实现**：先实现单节点分层，再叠加全局热点缓存
4. **与现有架构兼容**：每个存储节点本地自治，与 DHT/分离式架构都兼容

**缺点**：
1. **全局热点需要额外机制**：跨节点热点缓存需要独立的复制/同步机制
2. **HBM 与 DRAM 的边界模糊**：Connector 需要管理 HBM 和 DRAM 之间的数据流动
3. **容量规划复杂**：每节点的 DRAM/SSD 比例需要仔细设计

**适用场景**：HBM 管理已由推理引擎处理、存储系统专注持久化和温数据管理的场景。

## 关键 Trade-off 分析

### T1：精细优化 vs 实现复杂度
- **方案 A** 追求最精细的优化，但实现复杂
- **方案 C** 在精细度和复杂度之间取得平衡
- **本系统选择**：方案 C（混合分层），原因：
  - HBM 管理已由推理引擎（vLLM/SGLang）处理，存储系统无需重复造轮子
  - 4~16 节点规模下，节点本地自治 + 全局热点缓存已足够
  - chunk 级跨层迁移的元数据开销在百万级规模下不可忽视

### T2：自动分层 vs 显式控制
- **自动分层**：系统根据策略自动迁移，对用户透明
- **显式控制**：用户/推理引擎显式指定数据放置层级
- **本系统选择**：以自动分层为主，显式控制为辅（Connector API 提供 hints），原因：
  - 推理引擎更了解数据的访问模式（Prefill 阶段知道即将生成多少 token）
  - 自动分层处理无法预测的情况（如突发热点）

### T3：DRAM 作为缓存 vs 存储层
- **缓存模式**：DRAM 是 SSD 的前端缓存，数据最终都在 SSD 上
- **存储模式**：DRAM 和 SSD 都是持久化层，数据有选择地放在某一层
- **本系统选择**：存储模式（DRAM 和 SSD 都持久化），原因：
  - KV Cache 的重新计算成本远高于存储成本，不能随意淘汰
  - 缓存模式在 DRAM 不足时大量淘汰，导致命中率下降
  - 存储模式允许数据长期驻留在 DRAM（只要容量允许）

## 初步倾向

**倾向方案 C（混合分层）**：

### 三层定义

| 层级 | 介质 | 管理方 | 职责 | 容量 |
|------|------|--------|------|------|
| **L0：HBM** | GPU 显存 | 推理引擎（vLLM/SGLang）| 当前 batch 的活跃 KV Cache | ~80GB/GPU |
| **L1：DRAM** | 节点内存 | 存储系统 | 温数据、高频复用的 KV Cache | ~1TB/节点 |
| **L2：SSD** | NVMe SSD | 存储系统 | 冷数据、历史积累的 KV Cache | ~10TB/节点 |

### 关键设计决策

1. **HBM（L0）←→ DRAM（L1）**：通过 Connector API 由推理引擎显式控制
   - Prefill 阶段：生成 KV Cache → 异步写入 DRAM
   - Decode 阶段：从 DRAM 加载 KV Cache → 注入 HBM
   - Connector 提供 `put_kv()` / `get_kv()` / `prefetch_kv()` 接口

2. **DRAM（L1）←→ SSD（L2）**：存储系统自动管理
   - 每个存储节点本地自治管理 DRAM→SSD 分层
   - 采用 LRU/LFU 混合淘汰策略
   - DRAM 满时，最冷的 chunk 异步写入 SSD
   - SSD 读取命中时，chunk 可提升回 DRAM（可选）

3. **全局热点缓存（可选）**：
   - 高频访问的 chunk 可在多个节点的 DRAM 中缓存多副本
   - 通过全局调度器或一致性哈希的虚拟节点实现
   - MVP 阶段暂不实现，后续迭代

4. **分层策略参数**：
   - DRAM 淘汰阈值：DRAM 使用率 > 85% 时触发淘汰
   - SSD 预读：顺序读取时预读相邻 chunk
   - 异步迁移：所有跨层迁移均为后台异步，不阻塞前台 I/O

## 风险矩阵

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| DRAM 容量不足导致频繁 SSD 访问 | 中 | 高 | 容量规划工具；监控告警；自动扩容提示 |
| 分层迁移逻辑 bug 导致数据丢失 | 低 | 极高 | 迁移操作原子化（先写目标层，再删源层）；校验和验证 |
| 冷热判定策略不当导致性能波动 | 中 | 中 | 可配置策略参数；A/B 测试；灰度调整 |
| SSD 寿命耗尽（大量写入）| 中 | 高 | 监控 SSD 健康状态；磨损均衡；预留替换容量 |
