## 问题定义

KV Cache 存储系统的传输层需要解决的核心问题：

1. **多协议支持**：GPU 集群通常同时存在 RDMA（InfiniBand/RoCE）和 TCP 网络，如何统一抽象？
2. **零拷贝传输**：KV Cache 数据块大（32MB+），必须避免 CPU 拷贝开销
3. **控制平面与数据平面分离**：控制消息（连接管理、元数据查询）需要可靠性，数据消息需要高吞吐
4. **故障回退**：RDMA 连接故障时如何优雅降级到 TCP？
5. **PD 分离传输优化**：Prefill 节点到 Decode 节点的 KV Cache 传输是核心路径，需优化

**场景约束**：
- 部署环境：RDMA（InfiniBand / RoCE v2）为主网络，TCP 为管理网络/兼容回退
- 数据特征：大块连续内存（32MB~100MB+），适合 RDMA 的零拷贝传输
- 延迟要求：P99 < 10ms（含网络传输）

## 候选方案

### 方案 A：RDMA 独占 + TCP 仅管理面

**核心思路**：数据传输完全走 RDMA（双边操作），TCP 仅用于管理面（节点发现、心跳、配置同步）。RDMA 连接建立后，所有 KV Cache 读写均通过 RDMA READ/WRITE 完成。

**参考来源**：InfiniStore 服务端主导模式 + MoonCake Transfer Engine 的 RDMA 后端

**架构草图**：

```
┌─────────────────────────────────────────┐
│           应用层（KV Cache 读写 API）      │
├─────────────────────────────────────────┤
│           传输抽象层                      │
│  ┌─────────┐              ┌─────────┐  │
│  │  RDMA   │  ←────────→  │   TCP   │  │
│  │(数据面) │   无数据流    │(管理面) │  │
│  └────┬────┘              └────┬────┘  │
└───────┼────────────────────────┼───────┘
        │                        │
   ┌────┴────┐              ┌────┴────┐
   │ libibverbs│             │  socket  │
   │ (IB/RoCE)│             │ (TCP/IP) │
   └─────────┘              └─────────┘
```

**优点**：
1. **实现简单**：只处理一种数据协议，代码路径单一，调试简单
2. **性能最优**：无 TCP 协议栈开销，无数据面协议切换逻辑
3. **延迟最低**：RDMA 双边操作 ~2μs，无 TCP 回退的决策延迟
4. **测试覆盖简单**：只需验证 RDMA 路径

**缺点**：
1. **硬件锁定**：无 RDMA 网卡的节点完全无法使用，失去 TCP 兼容性
2. **故障恢复慢**：RDMA 连接故障时只能重连，无降级路径
3. **网络配置要求高**：需要无损以太网（PFC/ECN）或 InfiniBand，部署门槛高
4. **运维困难**：RDMA 问题诊断比 TCP 困难（内核旁路导致常规网络工具失效）

**适用场景**：所有节点均有 RDMA 网卡、有专业运维团队、追求极致性能的内部集群。

---

### 方案 B：RDMA+TCP 双栈（数据面自动切换）

**核心思路**：数据传输同时支持 RDMA 和 TCP，运行时根据链路健康状态自动选择。RDMA 优先，TCP 作为回退。控制平面统一走 TCP（或 RDMA SEND/RECV）。

**参考来源**：MoonCake Transfer Engine 多协议后端 + InfiniStore TCP 回退模式

**架构草图**：

```
┌─────────────────────────────────────────┐
│           应用层（KV Cache 读写 API）      │
├─────────────────────────────────────────┤
│           传输抽象层                      │
│  ┌─────────────────────────────────┐   │
│  │      路径选择器（拓扑感知）        │   │
│  │   ┌─────────┐    ┌─────────┐   │   │
│  │   │  RDMA   │ ←→ │   TCP   │   │   │
│  │   │(优先)   │    │(回退)   │   │   │
│  │   └────┬────┘    └────┬────┘   │   │
│  └────────┼──────────────┼────────┘   │
├───────────┼──────────────┼────────────┤
│      链路健康监测（RTT/丢包/带宽）        │
└───────────┼──────────────┼────────────┘
       ┌────┴────┐    ┌────┴────┐
       │ libibverbs│    │  socket  │
       └─────────┘    └─────────┘
```

**优点**：
1. **兼容性最好**：支持纯 RDMA、纯 TCP、混合部署三种模式
2. **故障弹性**：RDMA 链路故障时自动切换到 TCP，业务不中断
3. **渐进部署**：新节点可先通过 TCP 接入，再升级到 RDMA
4. **运维友好**：TCP 路径可用于问题诊断和降级运维

**缺点**：
1. **实现复杂度高**：需要维护两套数据路径，路径选择逻辑增加代码复杂度
2. **性能不一致**：TCP 回退时性能下降 10~100 倍，可能触发级联故障（更多请求走 TCP → 更慢 → 更多超时）
3. **状态同步开销**：需要持续监测链路健康，维护路径选择状态
4. **测试矩阵翻倍**：需要验证 RDMA→RDMA、RDMA→TCP、TCP→TCP、TCP→RDMA 四种组合

**适用场景**：混合硬件环境、需要高可用保障、有逐步升级网络基础设施计划的场景。

---

### 方案 C：RDMA 为主 + TCP 应急（静态绑定 + 动态降级）

**核心思路**：初始化时静态绑定 RDMA 为主路径，仅在 RDMA 连接故障时动态降级到 TCP。降级是单向的（RDMA → TCP），恢复后需人工或定时任务重新升级到 RDMA。

**参考来源**：InfiniStore 的 TCP 回退设计（但 InfiniStore 当前 TCP 模式有 bug，Issue #187）

**架构草图**：

```
┌─────────────────────────────────────────┐
│           应用层（KV Cache 读写 API）      │
├─────────────────────────────────────────┤
│           传输抽象层                      │
│  ┌─────────────────────────────────┐   │
│  │      静态绑定 + 故障降级         │   │
│  │   ┌─────────┐    ┌─────────┐   │   │
│  │   │  RDMA   │ ──→│   TCP   │   │   │
│  │   │(主路径) │故障 │(应急)   │   │   │
│  │   └────┬────┘    └─────────┘   │   │
│  └────────┼────────────────────────┘   │
│      故障检测（超时/心跳丢失）            │
└───────────┼────────────────────────────┘
       ┌────┴────┐
       │ libibverbs│
       └─────────┘
```

**优点**：
1. **常态性能最优**：正常情况全部走 RDMA，无 TCP 路径选择开销
2. **故障有兜底**：RDMA 故障时不会完全不可用，可降级到 TCP
3. **实现复杂度适中**：只需在连接层做故障检测和切换，无需运行时路径选择
4. **状态简单**：每个连接只有"RDMA"或"TCP"两种状态

**缺点**：
1. **降级后恢复困难**：从 TCP 恢复回 RDMA 需要重新建立连接，可能中断 inflight 请求
2. **降级触发条件难调**：超时阈值设太短导致频繁降级，设太长导致故障响应慢
3. **TCP 路径可能未经充分测试**：因为平时不走 TCP，TCP 路径的 bug 可能在紧急时才暴露

**适用场景**：RDMA 网络稳定、故障率低、可接受短暂降级的场景。

## 关键 Trade-off 分析

### T1：极致性能 vs 兼容性
- **方案 A** 选择极致性能，放弃兼容性
- **方案 B** 选择兼容性，接受实现复杂度和性能不一致
- **本系统选择**：偏向方案 C（RDMA 为主 + TCP 应急），原因：
  - 用户约束中"RDMA 为主，TCP 兼容"已经明确
  - 4~16 节点规模下，RDMA 网络相对容易保障稳定
  - 但完全放弃 TCP 不可接受（运维、诊断、渐进部署都需要 TCP）

### T2：双边 vs 单边 RDMA
- **双边 RDMA**（服务端发起 READ/WRITE）：延迟 +1μs，但服务端完全控制内存生命周期，适合 KV Cache 的 LRU 管理
- **单边 RDMA**（客户端直接访问服务端内存）：延迟最低，但服务端需暴露内存地址和 rkey，内存管理复杂
- **本系统选择**：双边 RDMA，参考 InfiniStore 的选择理由

### T3：WR 批处理 vs 逐条提交
- **批处理**（32 WR/批）：减少门铃 ringing 频率，提升吞吐，但增加首条 WR 的延迟
- **逐条提交**：延迟最低，但吞吐受限
- **本系统选择**：批处理（参考 InfiniStore MAX_WR_BATCH=32），KV Cache 场景是大块传输，批处理收益明显

## 初步倾向

**倾向方案 C（RDMA 为主 + TCP 应急）的变体**：

- **数据面**：RDMA 双边操作为主路径，初始化时静态绑定
- **控制面**：TCP 统一处理控制消息（连接建立、元数据查询、错误通知）
- **故障降级**：RDMA 连接超时（可配置，默认 ~4.3s）时降级到 TCP
- **恢复策略**：定时尝试重建 RDMA 连接，成功后新请求走 RDMA，inflight 请求仍走 TCP
- **WR 批处理**：MAX_WR_BATCH=32，与 InfiniStore 一致

**关键设计决策**：
1. 采用 **InfiniStore 式的双边 RDMA**（服务端主导），而非单边 RDMA
2. 采用 **TCP 控制平面 + RDMA 数据平面** 的分离设计
3. 采用 **事件驱动 CQ 处理**（libuv/epoll），而非忙轮询，以支持高并发连接
