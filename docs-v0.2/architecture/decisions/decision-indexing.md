# ADR-003: KV Cache 索引策略决策

- **状态**: 草稿
- **日期**: 2026-04-21
- **决策人**: architect
- **置信度**: 中（70-80%）

---

## 背景

### 问题定义

KV Cache 分布式存储系统的核心索引需求是：**在 100TB~10PB 数据规模、百万级~千万级 chunk 量级下，如何高效支持以下操作？**

| 操作 | 描述 | 性能要求 |
|------|------|----------|
| **Prefix Match** | 给定 token 序列，找到最长匹配前缀的 KV Cache | P99 < 5ms（调度器决策路径） |
| **Content-Addressable Lookup** | 给定内容哈希，定位存储位置 | P99 < 1ms |
| **Deduplication** | 相同 token 序列的 KV 只存一份 | 写入路径自动去重 |
| **Non-Prefix Reuse** | RAG 场景下非前缀位置的 KV 复用 | 可选，Phase 2 支持 |
| **Multi-Version Model** | 不同模型/版本生成的 KV 不可混用 | 模型名+版本作为索引维度 |
| **Distributed Scale-Out** | 索引随节点数水平扩展 | 16 节点下无单点瓶颈 |

### 关键约束

1. **数据规模**：100TB~10PB，对应约 100 万~1000 万个 chunk（按 256-token chunk、~32MB/chunk 估算）
2. **访问模式**：写入后高频复用，读多写少（典型比例 10:1 ~ 100:1）
3. **生命周期**：KV Cache 生命周期为小时~天级，远长于 MoonCake 的秒~分钟级
4. **一致性**：写入后立即可读（强一致），因为调度器依赖索引做路由决策
5. **语言约束**：C 语言实现，无 GC，需手动管理内存

## 标杆参考

| 标杆产品 | 相关技术 | 分析结论 |
|----------|----------|----------|
| **LMCache** | Prefix Hash Chain + ChunkedTokenDatabase | 直接借鉴。内容寻址索引设计成熟，但 Python 实现需迁移到 C；分布式查找依赖 ZMQ，需替换 |
| **MoonCake** | PrefixHash 算法 + Block Table | 直接借鉴。块级哈希链 + Block Table 设计简洁，生产验证（Kimi）；但 Conductor 未开源，调度器集成需自研 |
| **SGLang** | RadixAttention（Radix Tree） | 参考其树形前缀匹配思想。单节点设计优雅，但分布式扩展需重新设计；非前缀复用天然不支持 |
| **vLLM** | PagedAttention（块哈希表） | 参考其块级管理思想。仅支持前缀匹配，粒度太细（16 token/页），不适合分布式存储场景 |
| **IPFS** | Merkle DAG + DHT | 参考其内容寻址和分布式路由思想。Merkle Tree 结构与本场景的 prefix chain 有相似性 |

### 标杆关键洞察

1. **Prefix Hash Chain 是行业共识**：LMCache、MoonCake 均采用链式哈希实现前缀匹配，验证了这一方向的可行性
2. **Radix Tree 仅适用于单节点**：SGLang 的 RadixAttention 在单节点 GPU HBM 中表现优异，但无分布式扩展方案
3. **内容寻址去重是必选项**：所有标杆均支持基于内容哈希的去重，这是 KV Cache 确定性计算特性的自然选择
4. **块大小是核心调参点**：LMCache 256-token（~32MB）vs MoonCake 512-token（~64MB），需在带宽饱和与复用粒度间权衡

## 候选方案

### 方案 A：Prefix Hash Chain + Distributed Block Table（LMCache/MoonCake 路线扩展）

**核心思路**：
将 token 序列按固定大小切分为 chunk，每个 chunk 计算内容哈希，相邻 chunk 的哈希通过链式依赖形成前缀哈希链。前缀匹配时顺序比较哈希链，直到第一个不匹配的 chunk。Block Table 维护 hash -> (node_id, storage_location, ref_cnt) 的映射，按一致性哈希分片到多个索引节点。

**参考来源**：LMCache ChunkedTokenDatabase + MoonCake PrefixHash + Block Table

**架构草图**：

```
Token Sequence: [t0,t1,...t255] [t256,...t511] [t512,...t767] ...
                │  Chunk 0   │  │  Chunk 1   │  │  Chunk 2   │
                │   Hash0    │  │   Hash1    │  │   Hash2    │
                └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
                      │               │               │
                      ▼               ▼               ▼
                Prefix Hash Chain: H0 -> H1 -> H2 -> ...
                      │
                      ▼
    ┌─────────────────────────────────────────────────────────────┐
    │              Distributed Block Table (分片)                  │
    │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
    │  │ Shard 0     │  │ Shard 1     │  │ Shard N     │         │
    │  │ (hash%N)    │  │ (hash%N)    │  │ (hash%N)    │         │
    │  │ H0->loc0    │  │ H1->loc1    │  │ Hk->lock    │         │
    │  └─────────────┘  └─────────────┘  └─────────────┘         │
    └─────────────────────────────────────────────────────────────┘
```

**优点**：
1. **前缀匹配逻辑简单**：顺序比较哈希链即可，无需复杂树结构遍历
2. **内容寻址去重天然支持**：相同 chunk 内容必然产生相同哈希，Block Table 自动去重
3. **增量计算友好**：新增 token 只需计算最后一个 chunk 的哈希，无需全量重算
4. **分布式扩展直接**：按 chunk hash 分片，天然负载均衡，无热点集中
5. **生产验证充分**：LMCache 和 MoonCake 均采用类似设计，风险低

**缺点**：
1. **前缀匹配需顺序遍历**：最长前缀匹配需逐个 chunk 查询，O(N_chunks) 延迟随序列长度线性增长
2. **非前缀复用不支持**：只能匹配连续前缀，RAG 场景的动态拼接无法直接支持
3. **哈希计算开销**：每个 chunk 需计算哈希（SHA-256 或 xxHash），长序列时累积不可忽视

**适用场景**：对话系统、代码补全等前缀复用率高的场景；数据规模可达 10PB 级。

**理论复杂度分析**：

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|-----------|-----------|------|
| 插入（单 chunk） | O(1) 哈希计算 + O(1) 表插入 | O(1) 每 chunk | 哈希计算可并行 |
| 前缀匹配（完整序列） | O(L/B * DHT_lookup) | O(1) | L=token 数, B=chunk 大小；DHT_lookup ~ O(1) |
| 精确查找（by hash） | O(1) | O(1) | 直接哈希定位 |
| 去重检查 | O(1) | O(1) | 写入时先查 Block Table |
| 全量索引内存 | — | O(N_chunks) | N_chunks = 总 chunk 数 |

> 假设 chunk 大小 B=256 tokens，序列长度 L=128k tokens，则 N_chunks=500。单次前缀匹配需 500 次 DHT 查找。若 DHT 查找平均 100us，总延迟约 50ms——**这是关键瓶颈**。

### 方案 B：Distributed Radix Tree（SGLang 路线扩展）

**核心思路**：
将 SGLang 的 RadixAttention 从单节点扩展到分布式。Radix Tree 的每个节点代表一个 token，边代表 token 值。前缀匹配即从根节点开始逐 token 遍历树。分布式版本中，按 token 范围或子树分片到不同索引节点。

**参考来源**：SGLang RadixAttention + 分布式 B-tree/Radix Tree 研究（如 Apache Ignite、CockroachDB 的分布式索引）

**架构草图**：

```
                    [Root]
                      │
        ┌─────────────┼─────────────┐
        │             │             │
     [t=1]         [t=2]         [t=3]
        │             │             │
    ┌───┴───┐     ┌───┴───┐     ┌───┴───┐
   [t=5] [t=6]   [t=7] [t=8]   [t=9] [t=10]
    │       │     │       │     │       │
   ...     ...   ...     ...   ...     ...

Distributed Sharding (by subtree):
  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
  │  Index Node 0   │  │  Index Node 1   │  │  Index Node 2   │
  │  (subtree 0-3)  │  │  (subtree 4-7)  │  │  (subtree 8-11) │
  │                 │  │                 │  │                 │
  │  + Local Radix  │  │  + Local Radix  │  │  + Local Radix  │
  │    Tree         │  │    Tree         │  │    Tree         │
  └─────────────────┘  └─────────────────┘  └─────────────────┘
```

**优点**：
1. **前缀匹配效率最优**：Radix Tree 的压缩节点（radix=token 值范围）可将多层压缩为一层，平均遍历深度 O(log V) ~ O(1)，V 为词汇表大小
2. **细粒度前缀共享**：相同前缀的 token 序列共享树路径，内存效率高
3. **单节点性能优异**：SGLang 实测单节点前缀匹配延迟 <100us
4. **支持最长前缀匹配的增量更新**：新序列插入只需遍历/创建新路径，无需全量重算

**缺点**：
1. **分布式扩展复杂**：Radix Tree 是有序结构，跨节点分片需要维护子树边界，分裂/合并代价高
2. **树结构内存开销大**：每个 token 一个树节点，百万级 chunk 时节点数可达数十亿，内存占用高
3. **内容寻址去重不直接**：Radix Tree 按 token 值组织，而非内容哈希，去重需额外哈希表
4. **无生产级分布式实现**：SGLang 仅单节点实现，分布式 Radix Tree 在 KV Cache 场景无先例

**适用场景**：单节点或少量节点（<4）的 prefix cache 场景；对前缀匹配延迟要求极致（<1ms）的场景。

**理论复杂度分析**：

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|-----------|-----------|------|
| 插入（单 token） | O(L) 树遍历 + O(1) 节点创建 | O(L) 每序列 | 需遍历到叶子 |
| 前缀匹配 | O(L * shard_lookup) | O(1) | 最坏情况跨多个分片 |
| 精确查找 | O(L) | O(1) | 必须遍历树 |
| 去重检查 | O(L) + O(1) 哈希 | O(1) | 需额外哈希表 |
| 全量索引内存 | — | O(N_tokens) | 每个 token 一个树节点 |

> 假设词汇表 V=128k（LLaMA3），Radix Tree 的压缩因子约 10-100，则 128k token 序列的遍历深度约 1000-10000。但分布式分片后，跨分片跳转增加网络 RTT（~2us RDMA），累积延迟不可忽视。

### 方案 C：Hybrid Index（内容哈希 + B+Tree 范围索引）

**核心思路**：
双层索引结构：
- **L1（内容寻址层）**：哈希表维护 chunk_hash -> chunk_metadata，负责去重和精确查找
- **L2（前缀匹配层）**：B+Tree 维护 (prefix_signature, chunk_seq_id) -> chunk_list，负责前缀匹配。prefix_signature 是前 k 个 chunk 的聚合哈希（如 Merkle Tree 的根哈希）

前缀匹配时，先在 L2 用 B+Tree 找到候选前缀范围，再在 L1 验证精确匹配。

**参考来源**：LMCache Prefix Hash Chain + Ceph CRUSH 的层级哈希 + B+Tree 范围查询（经典数据库索引）

**架构草图**：

```
Token Sequence: [chunk0] [chunk1] [chunk2] [chunk3] ...
                    │      │      │      │
                    ▼      ▼      ▼      ▼
              ┌─────────────────────────────────────┐
              │  L1: Content Hash Table (去重+定位)  │
              │  hash(chunk0) -> metadata0           │
              │  hash(chunk1) -> metadata1           │
              │  hash(chunk2) -> metadata2           │
              └─────────────────────────────────────┘
                           ▲
                           │ 验证
              ┌────────────┴─────────────────────────┐
              │  L2: Prefix B+Tree (范围索引)         │
              │                                      │
              │  Key: (merkle_root, seq_id)          │
              │  Value: [chunk_hash0, chunk_hash1,..]│
              │                                      │
              │  B+Tree 按 merkle_root 排序，支持     │
              │  范围查询：找具有相同前缀的序列        │
              └─────────────────────────────────────┘
```

**优点**：
1. **前缀匹配效率提升**：B+Tree 范围查询将 O(N_chunks) 的顺序遍历降为 O(log N_sequences + M)，M 为候选序列数
2. **内容寻址与前缀解耦**：L1 专注去重和定位，L2 专注前缀匹配，职责清晰
3. **B+Tree 分布式成熟**：B+Tree 的分片策略（如范围分片）在分布式数据库中已充分验证
4. **支持更灵活的查询**：B+Tree 可扩展支持"相似前缀"查询（通过前缀签名距离）

**缺点**：
1. **实现复杂度高**：需要维护两层索引的一致性，插入操作需更新两个结构
2. **Merkle Tree 计算开销**：prefix_signature 需要计算 Merkle Tree，增加 CPU 开销
3. **两层索引的内存占用**：相比单一结构，内存占用增加约 1.5-2x
4. **无直接标杆参考**：该混合结构在 KV Cache 场景无先例，需自行验证

**适用场景**：数据规模大（>1PB）、前缀匹配延迟要求严格（<5ms）、团队有分布式数据库索引经验的场景。

**理论复杂度分析**：

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|-----------|-----------|------|
| 插入 | O(1) L1 + O(log N) L2 | O(1) L1 + O(1) L2 | 需更新两层 |
| 前缀匹配 | O(log N_seq + M * k) | O(M) | M=候选序列数, k=验证 chunk 数 |
| 精确查找 | O(1) | O(1) | 直接 L1 哈希 |
| 去重检查 | O(1) | O(1) | L1 哈希查找 |
| 全量索引内存 | — | O(N_chunks * 1.5) | 两层结构额外开销 |

> 假设 N_sequences = 1000 万，B+Tree 高度约 4-5。前缀匹配时 L2 查询 O(5)，候选序列 M 通常 <10（相同前缀的序列不多），验证 O(10 * 500) = 5000 chunk 比较——**实际可能不如方案 A 简单直接**。

## 关键 Trade-off

### 1. 前缀匹配延迟 vs 实现复杂度

- **方案 A**：O(N_chunks) 顺序查找，128k tokens 时 ~50ms。但实现简单，风险低。
- **方案 B**：O(tree_depth) 遍历，理论上更快，但分布式扩展引入的跨节点跳转可能抵消优势。
- **结论**：在 4~16 节点规模下，方案 A 的 50ms 前缀匹配延迟对于调度器决策是可接受的（调度器本身有其他开销如队列估计、负载检查）。若延迟成为瓶颈，可通过**批量并行查找**优化（同时发起所有 chunk 的 DHT 查询）。

### 2. 内容寻址去重 vs 前缀匹配效率

- 方案 A 将两者统一在哈希链中，去重和前缀匹配共享同一数据结构。
- 方案 C 将两者解耦，但引入两层一致性维护成本。
- **结论**：KV Cache 的确定性计算特性使得内容寻址是"免费"的（相同输入必然产生相同哈希）。方案 A 的统一设计更符合这一特性。

### 3. 生产验证度 vs 理论最优

- 方案 A 有 LMCache（多公司生产部署）和 MoonCake（Kimi 数千节点）的双重验证。
- 方案 B 和 C 在分布式 KV Cache 场景无生产先例。
- **结论**：在置信度 < 高的情况下，优先选择有生产验证的方案。

### 4. 内存占用 vs 查询效率

- 方案 B 的 Radix Tree 在单节点下内存效率高（共享前缀路径），但分布式后每个节点需维护部分树，总内存可能不降反升。
- 方案 A 的哈希表内存占用与数据量线性相关，但可预测、易规划。
- **结论**：方案 A 的内存模型更简单，适合容量规划。

## 决策

**选择方案 A（Prefix Hash Chain + Distributed Block Table），但进行以下关键改进：**

### 核心决策

采用 **Prefix Hash Chain** 作为核心索引结构，**Distributed Block Table** 按一致性哈希分片，同时引入以下优化：

1. **并行批量查找**：前缀匹配时并行发起所有 chunk 的 DHT 查询（而非顺序），将 O(N) 降为 O(1) 网络往返 + O(N) 本地比较
2. **本地 Prefix Cache**：每个存储节点维护本地 LRU 缓存最近匹配的 prefix 结果，减少重复 DHT 查询
3. **分层 Chunk Size**：L1（热数据）使用 128-token chunk 提高命中率，L2/L3 使用 512-token chunk 减少元数据
4. **非前缀复用预留**：Block Table 的 metadata 中预留 `blend_compatible` 标记位，为 Phase 2 的 CacheBlend 扩展做准备

|------|--------|--------|--------|------|

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| 前缀匹配延迟在超长上下文（1M+ tokens）下超标 | 中 | 高 | 1) 并行批量查找降低网络往返；2) 本地 prefix cache 缓存热点；3) 若仍超标，可退化为方案 C 的 Hybrid 结构 |
| Block Table 单分片成为热点（某些 chunk 被大量引用）| 中 | 高 | 1) 一致性哈希使用虚拟节点（vnodes）分散热点；2) 热点 chunk 的 metadata 缓存在所有节点；3) 监控分片负载，自动重平衡 |
| 哈希碰撞导致错误去重 | 低 | 高 | 1) 使用 SHA-256（碰撞概率 2^-256，可忽略）；2) 碰撞时回退到逐字节比较；3) 模型名+版本作为索引维度，隔离不同模型 |
| 索引节点故障导致元数据丢失 | 中 | 高 | 1) Block Table 每个分片 3 副本（Raft 共识）；2) 定期快照 + WAL 日志；3) 故障时从其他副本恢复 |
| 非前缀复用（RAG）需求迫切，但方案 A 不支持 | 中 | 中 | 1) Phase 1 预留 `blend_compatible` 标记位；2) Phase 2 引入 CacheBlend 的选择性重计算机制；3) 评估是否需独立索引结构 |
| 千万级 chunk 下内存占用超标 | 中 | 中 | 1) Block Table entry 紧凑编码（<128 bytes/entry）；2) 冷数据 metadata 下沉到 SSD；3) 分层索引（热数据内存 resident，冷数据按需加载）|

## 替代方案排除理由

- **方案 B（Distributed Radix Tree）**：
  1. 无分布式 KV Cache 场景的生产验证，技术风险高
  2. Radix Tree 的有序结构在分布式分片时面临复杂的子树分裂/合并问题
  3. 内容寻址去重需要额外哈希表，与树结构重复
  4. C 语言实现 Radix Tree 的分布式版本工程量大，性价比低

- **方案 C（Hybrid Hash + B+Tree）**：
  1. 实现复杂度显著高于方案 A，但前缀匹配延迟改善有限（估算从 50ms 降至 20ms）
  2. 两层索引的一致性维护增加故障场景
  3. 无标杆参考，所有优化需自行验证
  4. 若方案 A 的并行批量查找优化后延迟可接受，则 Hybrid 的额外复杂度不划算
