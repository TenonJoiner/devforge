## 问题定义

基于第 0 步前置调研和第 1 轮标杆研究，本系统需解决的核心架构问题是：

> **如何在 4~16 节点裸机 GPU 集群中，构建一个面向 KV Cache 场景的企业级分布式存储系统，使其在延迟（P99 < 10ms）、吞吐（饱和 RDMA 带宽）、一致性（写入立即可读）和成本（以 DRAM/SSD 替代 HBM）之间取得最优平衡？**

该问题可分解为三个子问题：
1. **数据分布与路由**：KV Cache 数据如何在节点间分布？如何定位数据？
2. **计算-存储关系**：GPU 计算节点与存储节点是耦合还是分离？分离粒度如何？
3. **PD 语义融合**：Prefill-Decode 分离的传输语义如何融入存储架构？

## 标杆架构模式提炼

| 标杆 | 核心架构模式 | 关键特征 | 与本系统的关联 |
|------|-------------|---------|--------------|
| **MoonCake** | **三层解耦**（Prefill / KVCachePool / Decode）+ 全局调度 | Conductor 全局调度；CPU DRAM 为主存储；RDMA 传输；最终一致 | 直接对标，验证 PD 分离 + 外部存储的可行性 |
| **LMCache** | **分层缓存层**（L1 GPU → L2 CPU → L3 Remote）+ 内容寻址 | 多后端适配（14+）；Prefix Hash Chain 索引；异步流水线 | 索引机制和分层策略可直接借鉴 |
| **InfiniStore** | **单节点内存池** + RDMA 双边传输 | 纯内存存储；服务端主导传输；bitmap 分配器；无原生集群 | 传输层设计和内存管理可参考，集群能力需自建 |
| **VAST Data** | **DASE**（计算/存储完全解耦，共享一切）| 无缓存（SCM 持久化元数据）；无数据所有权；NVMe-oF；宽条带纠删码 | 解耦思想参考，但 Cache-less 设计不适用于 KV Cache 热数据场景 |

**关键洞察**：
- 四个标杆代表了两种根本不同的哲学：**功能分层**（LMCache/MoonCake）vs **物理解耦**（VAST Data/InfiniStore）
- MoonCake 和 LMCache 验证了一个核心假设：**KV Cache 从 HBM 卸载到 CPU DRAM/SSD 在工程上是可行的，且收益显著（容量提升 75~115%）**
- InfiniStore 的局限（无原生集群、纯内存）恰好是本系统需要补齐的
- VAST Data 的 DASE 过于"通用存储"导向，其 Cache-less 设计牺牲了热数据低延迟，与 KV Cache 场景冲突

## 候选方案

### 方案 A：DHT 分层自治（Distributed Hash Table + Local Tiering）

**核心思路**：数据通过一致性哈希分布到各存储节点，每个节点本地自治管理 HBM→DRAM→SSD 分层。无中心调度器，请求通过哈希直接路由到目标节点。

**参考来源**：LMCache 多后端 + Ceph CRUSH + Cassandra Partitioner

**架构草图**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     推理引擎客户端（vLLM/SGLang）                  │
│                         Connector 层                             │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 哈希路由（一致性哈希环）
                            ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Storage    │    │  Storage    │    │  Storage    │    │  Storage    │
│  Node 1     │    │  Node 2     │    │  Node 3     │    │  Node N     │
│ ┌─────────┐ │    │ ┌─────────┐ │    │ ┌─────────┐ │    │ ┌─────────┐ │
│ │  HBM    │ │    │ │  HBM    │ │    │ │  HBM    │ │    │ │  HBM    │ │
│ ├─────────┤ │    │ ├─────────┤ │    │ ├─────────┤ │    │ ├─────────┤ │
│ │  DRAM   │ │    │ │  DRAM   │ │    │ │  DRAM   │ │    │ │  DRAM   │ │
│ ├─────────┤ │    │ ├─────────┤ │    │ ├─────────┤ │    │ ├─────────┤ │
│ │  NVMe   │ │    │ │  NVMe   │ │    │ │  NVMe   │ │    │ │  NVMe   │ │
│ └─────────┘ │    │ └─────────┘ │    │ └─────────┘ │    │ └─────────┘ │
│  本地分层管理  │    │  本地分层管理  │    │  本地分层管理  │    │  本地分层管理  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                            │
                    节点间 RDMA 互连（环形/全连接）
```

**优点**：
1. **无单点瓶颈**：无中心调度器/元数据服务器，每个节点自治，天然水平扩展
2. **故障域隔离**：单节点故障只影响该节点上的数据分区，不影响全局服务
3. **局部性最优**：数据哈希到固定节点，同节点内的 HBM→DRAM→SSD 分层无需跨网络
4. **实现简洁**：无需全局状态同步，Gossip 协议维护拓扑即可

**缺点**：
1. **PD 分离支持弱**：Prefill 和 Decode 可能落在不同节点，KV Cache 需跨节点传输，但 DHT 路由不感知 PD 语义
2. **负载均衡挑战**：热点前缀可能导致某些节点成为热点（负载倾斜），需要虚拟节点或重新哈希
3. **扩容重平衡**：新增节点时需要迁移数据分区，产生大量后台流量
4. **跨节点读取延迟**：Prefix Cache 命中时，如果目标数据在远程节点，需一次网络 RTT

**适用场景**：数据分布均匀、无强烈 PD 分离需求、追求极致去中心化的场景。

### 方案 B：计算-存储分离（Compute-Storage Disaggregation）

**核心思路**：GPU 计算节点（运行推理引擎）与存储节点（CPU + DRAM + NVMe）物理分离。计算节点通过 RDMA/NVMe-oF 直接访问存储节点的内存池和 SSD。存储节点之间通过一致性协议同步元数据。

**参考来源**：VAST Data DASE + InfiniStore 内存池 + MoonCake 的存储解耦思想

**架构草图**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      GPU 计算节点集群（Prefill + Decode）              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                 │
│  │ vLLM/SGLang │  │ vLLM/SGLang │  │ vLLM/SGLang │  ...            │
│  │  + HBM缓存   │  │  + HBM缓存   │  │  + HBM缓存   │                 │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘                 │
│         │                │                │                         │
│         └────────────────┴────────────────┘                         │
│                    RDMA / NVMe-oF（零拷贝）                          │
└─────────────────────────────────────────────────────────────────────┘
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      存储节点集群（CPU + DRAM + NVMe）                 │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐     │
│  │  DRAM Pool      │  │  DRAM Pool      │  │  DRAM Pool      │     │
│  │  (热数据缓存)    │  │  (热数据缓存)    │  │  (热数据缓存)    │     │
│  ├─────────────────┤  ├─────────────────┤  ├─────────────────┤     │
│  │  NVMe SSD       │  │  NVMe SSD       │  │  NVMe SSD       │     │
│  │  (持久化层)      │  │  (持久化层)      │  │  (持久化层)      │     │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘     │
│         元数据分片管理（Raft/Paxos per shard）                        │
└─────────────────────────────────────────────────────────────────────┘
```

**优点**：
1. **资源独立扩展**：计算节点和存储节点可独立扩缩容，GPU 升级和存储扩容解耦
2. **PD 分离天然支持**：Prefill 节点写入存储池，Decode 节点从存储池读取，传输路径清晰
3. **内存池化效率高**：存储节点的 DRAM 可被所有 GPU 节点共享，利用率高于分散到各节点
4. **运维简化**：存储节点可标准化（同构硬件），计算节点可按需配置不同 GPU 型号

**缺点**：
1. **网络成为瓶颈**：所有数据访问必须跨网络，即使同节点内的数据也需走 RDMA，延迟基线增加 ~2μs
2. **元数据管理复杂**：需要分布式元数据服务（如基于 Raft 的分片），引入一致性协议开销
3. **故障恢复慢**：存储节点故障时，该节点上的所有数据需从冗余副本重建，影响面大
4. **成本增加**：需要独立的存储节点（CPU + DRAM + NVMe），而非复用 GPU 节点的空闲资源

**适用场景**：大规模集群（16+ 节点）、计算/存储资源需求比例差异大、有独立运维团队。

### 方案 C：混合 PD 原生架构（Hybrid PD-Native）

**核心思路**：深度融合 Prefill-Decode 分离语义到存储架构中。Prefill 节点和 Decode 节点作为一等公民，存储层为 PD 传输优化。全局调度器（类似 MoonCake Conductor）统一决策请求路由和 KV Cache 放置。存储节点兼具"传输中继"和"持久化缓存"双重角色。

**参考来源**：MoonCake 三层解耦 + LMCache 分层存储 + InfiniStore RDMA 传输

**架构草图**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    全局调度器（Conductor 类似）                        │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐             │
│   │ Prefix Match │→│  Placement   │→│   Transfer   │             │
│   │   Engine     │  │  Decision    │  │  Scheduling  │             │
│   └──────────────┘  └──────────────┘  └──────────────┘             │
└─────────────────────────────────────────────────────────────────────┘
           │                    │                    │
           ▼                    ▼                    ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   Prefill Pool  │◀──▶│  Unified Store  │◀──▶│   Decode Pool   │
│  (GPU + HBM)    │    │ (DRAM + NVMe)   │    │  (GPU + HBM)    │
│                 │    │                 │    │                 │
│  ┌───────────┐  │    │  ┌───────────┐  │    │  ┌───────────┐  │
│  │ 计算 KV   │──┼───▶│  │ 热数据缓存 │  │◀───┼──│ 加载 KV   │  │
│  └───────────┘  │    │  └───────────┘  │    │  └───────────┘  │
│                 │    │  ┌───────────┐  │    │                 │
│                 │    │  │ 持久化层  │  │    │                 │
│                 │    │  └───────────┘  │    │                 │
└─────────────────┘    └─────────────────┘    └─────────────────┘
         │                      │                      │
         └──────────────────────┴──────────────────────┘
                        RDMA 传输网络
```

**优点**：
1. **PD 传输最优**：存储层aware of PD 语义，可优化 Prefill→Decode 的直接传输路径（绕过存储节点，或作为中继加速）
2. **Prefix Cache 全局最优**：全局调度器可基于前缀匹配结果，将请求路由到已有缓存的节点，最大化命中率
3. **放置策略灵活**：调度器可决策"热数据多副本放置""就近放置"等策略，DHT 和分离式架构难以做到
4. **渐进式部署**：可先作为纯缓存层部署（类似 LMCache），再逐步承担持久化职责

**缺点**：
1. **调度器单点风险**：全局调度器是潜在瓶颈和故障点，需要高可用设计（主备/多活）
2. **实现复杂度高**：需要同时管理 PD 传输调度、数据放置、一致性、故障恢复，代码复杂度最高
3. **扩展性上限**：调度器的决策能力随节点数增长而下降，16+ 节点时可能需要分层调度
4. **状态同步开销**：调度器需要维护全局视图（节点状态、缓存分布、负载），状态同步引入开销

**适用场景**：PD 分离是核心需求、追求极致 Prefix Cache 命中率、可接受更高实现复杂度的场景。

| **参考来源** | LMCache + Ceph | VAST DASE + InfiniStore | MoonCake + LMCache |

| **Prefix Cache 效率** | ⭐⭐ 局部最优 | ⭐⭐⭐ 全局共享 | ⭐⭐⭐⭐⭐ 全局最优调度 |

| **资源利用率** | ⭐⭐⭐ DRAM 分散 | ⭐⭐⭐⭐ DRAM 池化 | ⭐⭐⭐⭐ 调度优化后高 |

## 初步倾向与待验证假设

### 当前初步倾向

基于用户约束（4~16 节点、裸机、C 语言优先、替代现有对象存储）和标杆验证，**当前倾向方案 C（混合 PD 原生）的变体**：

- **核心骨架**：采用方案 C 的 PD 原生语义和全局调度思想
- **存储层**：融合方案 B 的存储节点池化 + 方案 A 的本地分层管理（每个存储节点本地自治分层）
- **调度器**：轻量级全局调度（非 MoonCake Conductor 的重量级调度），仅负责请求路由和放置决策，不参与数据传输

**理由**：
1. 用户明确提到"替代现有对象存储"，意味着需要完整的存储能力（持久化、冗余、扩展），不仅是缓存层
2. PD 分离是行业趋势（MoonCake 论文验证），不支持 PD 原生语义将失去核心竞争力
3. 4~16 节点规模下，轻量级全局调度的开销可控

### 待验证假设

| # | 假设 | 验证方法 | 不成立的后果 |
|---|------|---------|------------|
| H1 | 轻量级全局调度器在 16 节点下不会成为瓶颈 | 原型压测（模拟调度决策 QPS）| 需退化为方案 A 或分层调度 |
| H2 | 每个存储节点本地自治分层（HBM→DRAM→SSD）的延迟可接受 | SPDK + RDMA 原型测试 | 需改为方案 B 的纯分离式 |
| H3 | Prefix Hash Chain 索引在 10PB 规模下性能可接受 | 索引原型（百万级 chunk 压测）| 需改用 B-tree 或 LSM-Tree 变体 |
| H4 | SPDK reactor 模型可满足 I/O 调度 + 网络事件处理的双重需求 | SPDK 原型（NVMe I/O + RDMA CQ 轮询）| 需引入 Seastar/DAOS 协程框架 |
