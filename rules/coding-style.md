# coding-style — 通用编码规范

跨语言通用编码原则，作为所有语言特定规范的父文档。语言特定规范：C、[Go](coding-style-go.md)、[Python](coding-style-python.md)、[Java](coding-style-java.md)。冲突时以语言特定规范为准。

## 1. 命名

- 禁止单字母变量（循环变量 `i`/`j`/`k` 除外）
- 禁止匈牙利命名法、无意义缩写（领域通用缩写除外：`id`、`url`、`buf`、`len`、`ctx`、`cfg`、`err`、`fd`）
- 布尔变量使用 `is`/`has`/`can`/`should`/`needs` 前缀
- 常量/枚举使用 `SCREAMING_SNAKE_CASE`（或语言惯用等价形式）
- 集合变量使用复数名词（`users` 而非 `userList`）
- 错误变量/类型以 `Err`/`Error` 开头或结尾（语言惯用形式）

## 2. 错误处理

- 禁止静默吞错（空的 catch/except/错误检查分支）
- 错误信息包含上下文：操作名、参数值、失败原因、错误码
- 区分可恢复错误（外部依赖失败）与不可恢复错误（编程错误/前置条件违反）
- 错误传播优先于就地处理；边界处（API 入口、用户交互层）转换为友好信息
- 底层错误向上传播时保留原始错误信息（cause/chain/wrapping）

## 3. 内存安全

- 分配与释放必须配对，禁止跨层泄漏
- 禁止悬垂指针/引用、数据竞争、缓冲区溢出
- 资源获取即初始化：支持 RAII 的语言使用作用域绑定；不支持的语言使用 `defer`/`finally`/`with`/`try-with-resources`
- 涉及分配大小的整数运算必须检查溢出

## 4. 并发安全

- 共享状态必须有同步保护
- 锁的获取和释放必须配对，禁止跨函数边界持有锁
- 避免死锁：定义全局锁顺序、使用超时/trylock、禁止在持有锁时调用可能再次获取锁的外部回调
- 最小化临界区：只保护必要的数据访问，禁止在临界区内执行阻塞 IO、长时间计算、格式化日志输出
- 优先使用语言级并发原语（channel、协程、线程池）而非裸锁

## 5. 测试

- 核心模块覆盖率 ≥ 85%，新增代码覆盖率 ≥ 95%
- 测试必须可重复、可隔离：不依赖外部服务状态、不依赖执行顺序
- 每个测试只验证一个概念
- 纯逻辑单元测试执行时间 < 1 秒/个；涉及 IO、超时、故障注入的测试标记为慢测试，单独分组执行

## 6. 代码组织

- 一个文件/模块只负责一个概念
- 公开接口与实现细节分离：接口声明暴露最小必要集合
- 内部函数/方法标记为私有/静态/非导出
- 依赖按层次组织：系统库 → 第三方库 → 项目公共模块 → 当前模块内部
- 禁止循环依赖
- 公共接口变更必须评估对调用方的影响

## 7. 日志与可观测性

- 日志级别分层：`ERROR`（需人工介入）、`WARN`（可自动恢复但需关注）、`INFO`（关键业务流程节点）、`DEBUG`（开发调试细节）、`TRACE`（最细粒度数据流）
- 使用参数化/占位符形式输出日志，禁止运行时字符串拼接
- 禁止在日志中输出敏感信息（密码、Token、个人身份信息）
- 错误路径必须有日志记录或错误码传播，禁止静默失败

## 8. 断言与不变量

- 断言只用于检测程序内部不变量（不可能发生的情况）
- 禁止用断言处理用户输入、网络数据、磁盘 IO 等外部错误
- 断言失败后的代码不能依赖断言的副作用（发布版本中断言可能被关闭）
- 外部错误必须通过显式错误处理路径返回错误码或抛出异常

## 9. 安全

- 字符串/缓冲区操作必须使用长度限制版本，禁止无界拷贝
- 格式化字符串必须是字符串字面量或经过严格校验，禁止用户输入直接作为 format 参数
- 整数运算涉及分配大小时必须检查溢出
- 输入数据必须校验（长度、范围、格式）后再使用

## 10. 工具链

具体格式化、静态分析、内存检查、竞态检测工具配置见各语言 `coding-style-<lang>.md`。

## 代码审查清单（通用项）

- [ ] 命名是否自解释？是否存在无意义缩写？
- [ ] 错误处理是否完备？是否有静默吞错？可恢复与不可恢复错误是否区分正确？
- [ ] 是否有内存泄漏风险？分配与释放是否配对？资源在错误路径下是否释放？
- [ ] 是否有并发安全问题？锁是否配对？临界区是否最小化？是否定义了锁顺序？
- [ ] 测试覆盖率是否达标（核心模块 ≥ 85%，新增代码 ≥ 95%）？
- [ ] 断言是否只用于内部不变量？外部错误是否通过显式错误处理路径处理？
- [ ] 是否有安全漏洞（无界拷贝、格式化字符串注入、整数溢出、未校验输入）？
- [ ] 代码是否符合对应语言的特定规范？
- [ ] 是否有循环依赖或过度耦合？
- [ ] 日志是否使用参数化形式？是否泄露敏感信息？
