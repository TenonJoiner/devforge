# product/define — 定义性思考

## 概述

定义性思考 Skill —— 场景驱动 → 精确定义 → 场景独立可验收。

**核心原则**：
- **Actor 驱动**：先识别所有交互角色，再围绕角色定义场景
- **可验收性**：每个 Scenario 可独立验证，且明确由哪个 Actor 触发
- **完整性**：接口包含错误处理契约
- **渐进明细**：从粗粒度到细粒度逐步完善

## 何时使用

- 定义产品需求规格（requirements/）
- 制定功能验收标准
- 量化非功能需求（性能/可用性/容量）
- **继续完善**：已有需求文档需要迭代更新

**不适用场景**：
- 架构方向探索（使用 architect）
- 子系统接口设计（使用 architect）
- 迭代计划制定（使用 plan）
- 具体代码实现

## 启动检测

```bash
ls docs/requirements/ 2>/dev/null || echo "无现有需求文档"
```

**如果已有文档**：
> 检测到现有需求文档：
> 1. **新增 Feature** —— 扩展现有文档
> 2. **完善现有定义** —— 迭代更新特定章节
> 3. **重构结构** —— 调整组织方式

## 核心流程（灵活选用）

### 模式 A：从零定义

#### 阶段 0：使用场景与上下文

**目标**：明确产品在什么环境下被谁使用，建立共识基线。

**产出**：在需求文档开头写明以下内容：

```markdown
## 使用场景与上下文

**产品定位**：一句话说明产品是什么、解决什么核心问题

**部署环境**：产品运行的典型环境（如：3-5 节点集群、Linux x86_64、万兆网络）

**典型使用流程**：用户如何与产品交互的端到端概述（1-3 段）
```

#### 阶段 1：Actor 识别与定义

**目标**：识别所有与产品存在交互的角色，明确每个角色的职责边界。

Actor 不仅是"用户"，也包括外部系统、内部组件、运维工具等一切与产品产生交互的实体。

**Actor 定义模板**：

```markdown
## Actors

### Actor: <角色名称>

**类型**：人类 / 外部系统 / 内部组件

**职责**：该角色承担什么职责（1-2 句）

**交互方式**：通过什么途径与产品交互（CLI/API/SDK/内部调用）

**关注点**：该角色最关心什么（如：延迟、数据安全、易用性）
```

**常见 Actor 类型**（按需选取）：

| 类型 | 示例 | 说明 |
|------|------|------|
| 应用客户端 | SDK 调用方、API 消费者 | 读写数据的主要使用方 |
| 管理员 | 集群管理员、DBA | 部署、配置、扩缩容、权限管理 |
| 运维系统 | 监控平台、告警系统 | 健康检查、指标采集、故障响应 |
| 内部组件 | 副本节点、元数据服务 | 系统内部的自治交互 |
| 外部依赖 | 时钟服务、配置中心 | 产品依赖的外部系统 |

**检查清单**：
- [ ] 是否遗漏了运维/监控类 Actor？
- [ ] 内部组件间的交互是否也识别为 Actor？
- [ ] 每个 Actor 的职责边界是否清晰、无重叠？

#### 阶段 2：Feature 识别

**围绕 Actor 分组**：
- 按 Actor 的核心目标识别 Feature
- 同一 Feature 可能涉及多个 Actor（标注主 Actor 和协作 Actor）
- 检查与现有 Feature 的关系

### Feature-Scenario 结构

```markdown
## Feature: <特性名称>

**做什么**：该特性提供的能力和行为

**为什么**：解决什么问题或带来什么价值

**涉及 Actor**：<主 Actor>、<协作 Actor>（如有）

### Scenario: <场景名称>

**Actor**：<执行该场景的具体 Actor>

**前置条件**：系统处于什么状态

**触发动作**：<Actor> 执行什么操作

**预期行为**：系统应产生什么结果

**验证方法**：具体的测试手段（可独立编写测试用例验收）
```

#### 阶段 3：Scenario 挖掘

**每个 Scenario 必须指定 Actor**：明确是哪个 Actor 在什么上下文下触发该场景。

**必须覆盖的场景类型**：

| 类型 | 说明 | 示例（含 Actor） |
|------|------|------|
| 正常路径 | 标准使用流程 | **应用客户端**写入数据成功 |
| 节点故障 | 单节点宕机 | **应用客户端**写入时**副本节点**宕机，数据不丢失 |
| 网络分区 | 网络隔离/脑裂 | **应用客户端**在网络分区期间写入，恢复后数据一致 |
| 并发冲突 | 多客户端竞争 | 多个**应用客户端**并发写入同一 key，按版本号解决冲突 |
| 资源耗尽 | IO 错误/磁盘满/内存不足 | **应用客户端**写入时磁盘满，系统拒绝并返回明确错误码 |
| 运维操作 | 管理维护动作 | **管理员**执行在线扩容，**应用客户端**无感知 |

**分布式系统强制要求**：涉及分布式交互的 Feature 至少覆盖 4 种故障模式。

**多 Actor 交互场景**：当 Scenario 涉及多个 Actor 时，标注主 Actor（触发方）和协作 Actor（响应方）。

#### 阶段 4：非功能需求量化

| 类型 | 量化指标 | 验收标准示例 |
|------|---------|-------------|
| 性能 | 吞吐、延迟 | P99 写入延迟 < 10ms，单节点 QPS > 10K |
| 可用性 | SLA、MTTR | 年度可用性 ≥ 99.99%，MTTR < 5分钟 |
| 容量 | 数据量、连接数 | 单节点支持 10TB 数据，10000 并发连接 |
| 扩展性 | 扩容粒度 | 支持在线扩容，单步增加 1 节点 |
| 一致性 | 一致性级别 | 线性一致 / 顺序一致 / 最终一致 |

### 模式 B：迭代完善

**常见迭代场景**：
1. **补充 Scenario**：已有 Feature 增加边界情况
2. **细化需求**：补充边界条件、错误场景
3. **量化指标**：将"高性能"转化为具体数字
4. **追溯架构变更**：架构调整后同步更新需求

## 铁律检查

```
每个 Scenario 必须指定具体的 Actor
每个 Scenario 必须可独立编写测试用例验收
```

## 红旗信号

- 🚩 Scenario 未指定 Actor（不知道谁触发了这个场景）
- 🚩 需求文档缺少 Actor 定义（角色模糊，职责不清）
- 🚩 需求文档缺少使用场景上下文（不知道产品在什么环境下使用）
- 🚩 Scenario 无法独立验收（依赖其他 Scenario 的状态）
- 🚩 分布式场景未覆盖故障模式（只有 happy path）
- 🚩 非功能需求无具体数字（"高性能"、"高可用"）
