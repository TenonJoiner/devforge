# product/architect — 架构探索与决策

## 概述

架构探索与决策 Skill —— 发散思考 → 多方案对比 → 收敛记录。

**核心原则**：
- **质量优于流程**：输出质量比严格执行步骤更重要
- **螺旋式迭代**：可反复进入，持续完善同一文档
- **长期协作**：设计讨论可能持续数月，保持上下文连贯

## 何时使用

- 系统架构设计（子系统分解、技术选型）
- 竞品分析和方案对比
- 重大技术决策（一致性模型、存储引擎、协议选择）
- 架构重构或技术债务清理评估
- **继续完善**：已有架构文档需要迭代更新

**不适用场景**：
- 具体特性实现细节（使用代码级 Skill）
- 代码级 bug 修复
- 已有明确方案的简单增量开发

## 启动检测

每次执行时，首先检查现有文档状态：

```bash
ls docs/architecture/ docs/adr.md 2>/dev/null || echo "无现有文档"
```

**如果已有文档**：
> 检测到现有架构文档：
> 1. **继续完善** [xxx.md] —— 迭代更新特定章节
> 2. **探索新方向** —— 启动新的架构讨论
> 3. **对比回顾** —— 检视现有决策是否仍适用

## 核心流程（灵活选用）

### 模式 A：从零探索（Greenfield）

适用于全新系统或重大架构调整。

#### 阶段 1：发散思考

**目标**：尽可能全面地收集信息、生成候选方案。

**启发方法**：
- **竞品分析**：技术方案对比矩阵（功能/性能/一致性/可运维性/社区活跃度）
- **多方案探索**：强制生成至少 3 个候选方案
  - 保守方案（基于现有技术栈）
  - 激进方案（采用新技术/新架构）
  - 折中方案（平衡风险与收益）

**输出**：候选方案列表（名称、核心思路、适用场景、主要风险）

#### 阶段 2：多方案对比

**质量属性五维评估**：

| 维度 | 评估要点 | 关键指标 |
|------|---------|---------|
| 性能 | 吞吐、延迟、扩展性 | QPS、P99 延迟、水平扩展能力 |
| 一致性 | 数据一致性模型、故障后完整性 | 强一致/最终一致、副本一致性 |
| 可靠性 | 故障恢复、容错能力、可用性 | MTTR、MTBF、多副本策略 |
| 可维护性 | 代码复杂度、运维难度、监控能力 | 圈复杂度、故障定位时间 |
| 扩展性 | 功能扩展、容量扩展、接口兼容 | 插件机制、平滑扩容 |

**Trade-off 分析**：
- 各维度得分（高/中/低）
- 明确取舍点（如：高一致性 vs 高性能）
- 识别不可接受的缺陷

#### 阶段 3：收敛记录

根据讨论深度，选择产出物：

| 置信度 | 产出形式 | 下一步 |
|--------|---------|--------|
| 低（<60%） | 探索笔记（docs/architecture/exploration-xxx.md） | 补充调研后再次讨论 |
| 中（60-85%） | ADR 草稿（docs/adr.md#草稿区） | 继续完善或实验验证 |
| 高（>85%） | 正式 ADR + 架构文档 | 进入详细设计 |

**注意**：置信度是主观评估，不强制门槛。

### 模式 B：迭代完善（Iteration）

适用于已有文档需要更新。

**流程**：
1. 读取现有文档
2. 询问用户本次迭代重点
3. 针对性更新特定章节
4. 记录变更理由（在文档中保留修订历史）

### 模式 C：快速记录（Quick Capture）

适用于讨论中产生的灵感或临时决策。

**直接产出**：
- 在 docs/architecture/notes/ 下创建时间戳笔记
- 或在 docs/adr.md 中追加「待整理决策」

## ADR 格式

```markdown
## ADR-XXX: <决策标题>

- **状态**: 草稿/已接受/已替代
- **日期**: YYYY-MM-DD
- **决策人**: <姓名>

### 背景

<问题描述和上下文>

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: xxx | ... | ... |
| B: xxx | ... | ... |
| C: xxx | ... | ... |

### 决策

选择 **方案 X**，因为...

### 影响

- 正面影响：...
- 负面影响：...
- 风险：...

### 修订历史

- YYYY-MM-DD: <变更内容>（记录后续的架构调整）
```

## 红旗信号（自查清单）

- 🚩 只有一个候选方案（无比较即无决策）
- 🚩 方案描述只有优点没有缺点
- 🚩 未考虑团队技术栈和运维能力
- 🚩 重大决策无记录（不可追溯）

## 常见借口与回应

| 借口 | 回应 |
|------|------|
| "时间紧，先做一个再说" | 记录为「临时决策」，标注技术债务风险 |
| "业界都用这个" | 你的场景可能与业界不同，记录适配成本评估 |
| "先上线再优化" | 标注架构缺陷可能无法后期修补的风险 |

## 协作建议

1. **跨会话保持**：每次结束时总结当前状态，便于下次继续
2. **增量产出**：不必一次完成整份文档，可分批交付
3. **反向验证**：定期回顾旧决策，确认是否仍适用

## 输出位置

- `docs/architecture/<subsystem>.md` — 子系统架构
- `docs/architecture/exploration/` — 探索笔记
- `docs/adr.md` — 架构决策记录
